## Сравнение v4.2 с v4.1

### Что сохранено

v4.2 сохраняет главную структуру v4.1:

- AI-driven SDLC как жизненный цикл программного изменения;
- SPDD как глубокий опорный случай;
- Gas Town / Beads как глубокий опорный случай;
- защищённые методологические профили;
- сравнительные главы;
- финальную карту выбора режима.

Сохранены сравнительные узлы:

```text
SWE-chat / Programming by Chat / How Coding Agents Fail
specification / ADR / contract
SPDD / Spec Kit / Kiro / TDAD / Constitutional SDD
Spec Kit / GSD / BMAD / Reversa / OpenSpec / AgentSPEX / Gas Town
Harness / Sandvault / platform agents
architecture fitness / contract tests / test data
SASE / open-source policies / CODEOWNERS
incident / stale ADR / context cleanup
```

### Что усилено

ADR теперь имеет статус защищённого профиля архитектурного решения внутри Части III. Это не создаёт новую deep-case главу, но требует раскрыть Nygard, MADR, статусы, reconstructed ADR, операционную проекцию и подтверждение решения.

В Части VI добавлен persistent work graph как понятие, не заменяющее Gas Town.

В Части VIII добавлен workflow runner / workflow engine как подузел исполнения, наблюдаемости и воспроизводимости.

В Части X усилен слой подтверждений: API compatibility, performance / SLO и rollout / runtime confirmation.

В Частях XI–XII добавлены governance-сигналы и более широкий хвост жизненного цикла: stale specification, stale rules / skills и work graph cleanup.

### Что ограничено

Истории остаются первым разделом сайта. В теории допустимы ссылки и короткие фактические якоря, но недопустим пересказ истории внутри теоретической главы.

Технический атлас ограничен выбранными тяжёлыми досье:

```text
SPDD
Spec Kit
Kiro Specs
Constitutional SDD
TDAD
GSD / Open GSD
BMAD Method
Gas Town / Beads
ADR
```

Он хранит техническую фактуру и не должен становиться концептуальным дублёром теории.

## Почему не создан отдельный пакет “будущих документов”

Пользователь уточнил, что будущие документы ещё не написаны и будут создаваться по скелетону. Поэтому в v4.2 удалена формулировка, похожая на требование иметь набор будущих документов перед письмом. Вместо неё введён блок “Порядок написания и готовность материала”.

Практическое правило:

```text
если материала достаточно — писать часть по скелетону;
если материала недостаточно — сначала расширить досье или источниковый слой;
не закрывать слабую часть гладкой теоретической прозой.
```

## Что не менялось

Публичные файлы сайта в `/content` не менялись.

Старые story documents не менялись.

SPDD и Gas Town не были переписаны как текст будущей главы; обновлён только скелетон.

Предыдущий несанкционированный вариант с широким “concept evidence atlas” не включён.

## Риски после патча

1. Часть III может распухнуть из-за ADR. При написании нужно держать ADR как protected decision profile, а не как третий deep anchor.
2. Технический атлас может начать дублировать теорию. Для него нужен отдельный boundary audit.
3. Короткие ссылки на истории могут незаметно превратиться в пересказ историй. Для теории нужен story-boundary audit.
