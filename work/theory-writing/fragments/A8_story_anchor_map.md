# A8 story anchor map

| Якорь | Роль в A8 | Факт, который вошёл в текст | Граница / что не утверждается |
|---|---|---|---|
| Stripe Minions | Корпоративный якорь для различия между подготовкой артефакта для ревью и правом принятия. | Агент берёт задачу, пишет код, проходит проверки и готовит ветку; если результат выглядит пригодным, инженер открывает PR и запрашивает ревью, а Stripe описывает такие PR как human-reviewed. | Не утверждается, что Minions отменили ревью, что throughput-метрики измеряют acceptance без человека или что детали devbox/Toolshed нужны для этого аргумента. |
| Jökull Sólberg `/babysit-pr` | Якорь PR-lifecycle: автоматизация ожидания, проверки и исправления без передачи полного права решения. | `/babysit-pr` ждёт CI/Greptile, запускает Codex review, классифицирует замечания как Fix/Dismiss/Escalate, исправляет валидные пункты, повторяет цикл до зелёного CI, отсутствия нерешённых замечаний и лимита итераций; Escalate требует человеческой оценки. | `ready to merge` трактуется как подготовленное состояние и пакет доказательств, а не как право агента присвоить себе merge authority. |
| Mike McQuaid / Sandvault / worktrees | Якорь локального исполнения: безопасная способность действовать через изоляцию и отдельные ветки. | Sandvault использует macOS sandboxing и пользователя без админ-прав; worktrees дают параллельные ветки; агент может запускаться в sandboxed worktree. | Sandbox и worktree не трактуются как разрешение на публикацию. Они уменьшают риск действия, но не отменяют ревью. |
| Mike McQuaid local review | Якорь личного review-boundary после локального исполнения. | McQuaid локально проверяет работу агента перед тем, как поделиться ею: читает diff, правит, запускает тесты, использует ревью другой AI-системой и учитывает сбои CI. | Не превращается в общую норму для всех команд; используется как пример явного разрыва между локальной способностью и внешним sharing. |
| Homebrew AI/LLM contribution rules | Policy-якорь рядом с Mike story. | Автор AI-assisted contribution должен раскрыть tool/model, проверить сгенерированный материал, отвечать на комментарии и, если он не maintainer, не держать больше одного AI-generated PR одновременно. | Не обобщается на все проекты; используется как пример, где авторская ответственность отделена от генерации patch. |
| Zig no-AI policy / Contributor Poker | Жёсткая participation boundary: проект может отказать AI-origin work до технического обсуждения diff. | Zig policy имеет ограниченную область действия и запрещает LLM-generated content, paraphrasing/editing/translation/brainstorming/bug finding в рабочих пространствах проекта; Contributor Poker объясняет ревью maintainer как scarce investment в ответственного человека. | Не превращено в этическую главу и не расширено на весь Zig ecosystem; отдельные PR-инциденты не пересказываются. |
| CODEOWNERS | Механизм владения и owner-review. | Path ownership, base-branch CODEOWNERS, требование write access и защита ветки могут требовать code-owner approval до merge. | CODEOWNERS не представлен как полная архитектурная власть; это нижний уровень ownership/approval. |
| ADR status | Архитектурный governance-якорь. | ADR со статусом proposed не равен accepted; архитектурное ограничение появляется после принятия, а не после генерации текста. | Формат ADR сам по себе не решает governance. Нужен процесс принятия. |
| ADR review/adoption process | Процессный якорь для архитектурного acceptance. | AWS process показывает Proposed → review/action items/reject/approve → Accepted, а затем использование ADR в ревью кода и архитектуры. | Не превращается в AWS-specific governance prescription для всех команд; используется как конкретная модель process boundary. |
| Beads `bd gate` / PWG | Механизм durable acceptance boundary. | Human review, GitHub run, GitHub PR и другие ожидания можно моделировать как gate, а не как свободный текст в prompt. | Beads не объявлен единственной реализацией PWG; используется как практический пример durable gate. |
| Linux / LLVM / tiangolo-FastAPI / QEMU policy cluster | Набор разных границ политик участия вокруг human responsibility, maintainer load и provenance/DCO. | Linux удерживает DCO/human sign-off; LLVM соединяет прозрачность, human approval и рамку extractive contribution; tiangolo/FastAPI требует meaningful human intervention и предупреждает о human effort DoS; QEMU отклоняет AI-generated/derived contributions из-за DCO/provenance-рисков. | Не делается полная сравнительная таблица policies; QEMU не называется мягким вариантом; кластер нужен только для инварианта: генерация diff не равна праву принятия, подтверждения provenance или расходования ревью-ёмкости. |
