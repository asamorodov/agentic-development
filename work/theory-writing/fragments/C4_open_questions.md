# C4 open questions

Статус: `ready_with_known_debts` для первого writing-пакета C4. Фрагмент годится как мост от execution runtime к PWG, но перед публикационной сборкой остаются долги.

1. Нужен отдельный source-pass по официальным страницам Claude Code / Codex Worktrees, если в финальной версии останутся утверждения не только об изоляции worktree, но и о конкретных механиках locks, cleanup или handoff.
2. Devbox-детали Stripe в C4 частично опираются на курируемую стенограмму/заметки Mark Doyle и ChatPRD страницу рабочего процесса. Перед финальной публикацией желательно проверить, какие из этих деталей можно оставить как утверждения из первичных Stripe-источников, а какие должны быть помечены как публичная реконструкция/вторичный материал.
3. Блок durable execution сейчас достаточно силён для теоретического различения, но технический атлас должен отдельно раскрыть LangGraph / Temporal / DBOS / Restate, чтобы C4 не перегружался деталями API.
4. Визуальный слой намеренно ограничен двумя собственными схемами. Реальные скриншоты из Stripe/Shopify/Mike не вставлены: в текущем архиве нет локальных `content/assets/story-images/...`, нужны asset-pass, проверка прав/статуса и решение, не дублируют ли они уже A6/story-страницы.
5. Companion-файлы созданы как пакет свидетельств для C4; `C4_degradation_and_duplication_audit.md` добавлен после прохода на состояние запуска / состояние работы. Перед публикационной сборкой файл нужно перечитать вместе с A6/A4, чтобы не закрепить дублирование соседних узлов.
6. В финальной сборке главы проверить дублирование с A6: C4 должен оставаться мостом runtime → PWG, а не повторной мини-главой о слоях среды исполнения.

7. В третьем проходе добавлен Beads Dependencies как источник readiness/work item. Перед техническим атласом стоит решить, оставлять ли Beads в C4 как единственный конкретный PWG-источник или добавить более нейтральную модель work item без продуктового примера.
8. После четвёртого прохода во вступление добавлено явное различение состояния запуска / состояния работы и восстановления выполнения / восстановления работы. Оставшийся долг — не дать этой границе разрастись в повтор A6 при финальной сборке главы.
9. Source/depth-проход добавил два официальных Stripe-источника: Sessions 2026 Developer Keynote для сцены “запуск завершился, PR ещё сливает человек” и бенчмарк Stripe integration для более строгой формы пакета свидетельств. Долг: не расширять C4 дальше в обзор Stripe agent experience; дополнительные материалы про направление агентов лучше оставить A6, story или технический атлас.
10. После композиционного выравнивания основной текст больше не говорит “это пример для C4”, а ведёт единый аргумент: запуск оставляет след, PWG решает право продолжения и принятия. Оставшийся долг — при сборке главы проверить, не стоит ли вынести часть плотной Stripe/Roast/worktree-фактуры в technical atlas, если A6 уже покроет runtime-слой.

11. Адресный проход по состоянию запуска / состоянию работы усилен без новых источников: поля состояния запуска теперь явно ограничены командой/рабочим процессом, thread, worktree, devbox, браузером, песочницей, checkpoint и логом. Оставшийся долг для финальной сборки — не дать этой полезной формуле превратиться в повтор A6 и сохранить C4 как мост к PWG.
12. Сценарий продолжения добавлен как аргумент, а не как инструкция. При финальной сборке проверить, не нужно ли связать его с соседним C3/evidence и A9/lifecycle repair, не расширяя C4 в отдельный playbook восстановления.
13. Первый общий редакторский проход снял локальную структурную проблему: durable execution больше не собран в один перегруженный абзац, а source usage больше не дублирует Roast tutorial. Долг остаётся тем же: поздняя сборка должна решить, сколько runtime-фактуры оставить в C4 после сопоставления с A6.
14. Второй общий редакторский проход не нашёл блокеров: исправлены только плотность worktree-абзаца и повтор после сценария продолжения. Долг по финальному объёму runtime-фактуры остаётся открытым до сборки рядом с A6.
15. Третий общий редакторский проход не нашёл содержательных блокеров и не менял аргумент. Исправлена только классификация inline-схем в markup: обе теперь имеют `class="synthetic-figure"`. Статус остаётся `ready_with_known_debts` из-за поздней сверки с A6/A4/C3/A9 и нерешённого asset-pass по реальным изображениям.
