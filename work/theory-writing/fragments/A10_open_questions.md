## Открытые вопросы

1. **Стабильность публичных story anchors.** В A10 ссылки ведут на `#story-...` anchors из публичной сборки корпуса историй. Перед финальной сборкой сайта нужно проверить, что эти anchors действительно существуют или заменить их на стабильные внутренние ссылки главы.

2. **Нужны ли внешние источники в самом A10.** Edge/source-pass добавил пять внешних источников только для policy-bound contribution: Zig Code of Conduct, текст Mike McQuaid на сайте Open Source Initiative, Linux Foundation guidance, EFF policy и OpenInfra policy. Остальная карта остаётся композиционной и не требует самостоятельного внешнего source-pass. Если финальное заключение должно иметь внешние citations про AI-driven SDLC, governance, security или review effort шире этого edge case, нужен отдельный source-pass с открытием первоисточников из source map.

3. **Термин `Persistent Work Graph`.** В A10 он используется как уже принятый термин. После фактической сверки с A4/B2 формулировки считаются рабочими, но перед финальной публикационной сборкой стоит ещё раз проверить, не расширяет ли A10 PWG в сторону process profile, execution runtime или Gas Town.

4. **Граница process / PWG / Gas Town.** Основной текст различает процесс как ответ на повторяемый сбой, PWG как долговечное состояние конкретной работы и Gas Town как организационную среду. Оставшийся долг — финальная композиционная сверка после C5/atlas-routing, чтобы атласные статьи не начали обратно размывать эту границу.

5. **Policy-bound contribution cluster.** A10 использует policy anchors только для демонстрации режима, где возможность агента сделать патч не равна праву участвовать во внешнем проекте. Для A10 этого достаточно; если в A8, Handbook или atlas появится отдельный open-source policy section, туда можно вынести более широкий обзор.

6. **Handbook / Fieldbook projection.** Risk radar, review package checklist, refusal branch, mode-choice worksheet и edge-case mini-matrix оставлены вне основного A10. Нужно решить позднее, какие из них попадут в Handbook / Fieldbook как рабочие инструменты.

7. **Visual asset pass.** В текущем листе нет разрешённых локальных изображений, специфичных для A10. Если при финальной сборке появятся новые A10-relevant assets из A/B/C или C5, их нужно сохранить как реальные изображения, а не заменять синтетическими схемами.

8. **Закрытый долг: фактическая сверка A/B/C.** Первичный result ошибочно оставил этот пункт как открытый. При post-import integration фактические A1–A9, B1–B3 и C1–C4 были открыты и сверены; этот долг больше не является blocker-ом A10.

## Readiness status

`ready_with_known_debts`: A10 можно использовать как поздний синтетический фрагмент и вход для C5/atlas sync. Основная функция выполнена: карта выбирает минимально достаточную внешнюю структуру для конкретного изменения, допускает комбинации режимов и деэскалацию, не превращается в оглавление A/B/C и не делает maturity ladder. Механические долги результата закрыты post-import repair; оставшиеся вопросы относятся к публикационной сборке, anchors, optional external source-pass и будущей синхронизации с C5/атласом.
