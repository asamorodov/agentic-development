# A2 degradation and duplication audit

## Проверка несущего фрагмента после выравнивания

Фрагмент был сжат из более справочного варианта в один аргумент: agentic workflow создаёт три опасные подмены — specification как accepted architecture, contract gate как architecture rationale, ADR как agent instruction. Это лучше соответствует скелетону: A2 остаётся сравнительным узлом Part III, а не каталогом specification methodologies или отдельным deep case по ADR.

## Что убрано из основного текста как лишнее разрастание

- Полный пример operational projection card оставлен в сжатом виде. Детальная карточка остаётся в figure candidates / technical atlas.
- Подробная reconstructed ADR schema сокращена до поля-перечня и принципа `origin ≠ status`. Полный schema лучше вынести в ADR technical atlas.
- Истории Product Migration сокращены до функции: Winder.AI показывает oracle gap, DEV story показывает setup/review discipline. Подробный пересказ должен жить не в A2, а в migration/evidence section.
- TDAD оставлен только как предупреждение про visible tests / hidden tests / mutation и gaming risk. Полный TDAD workflow относится к будущему evidence/oracle узлу.

## Риск дублирования с будущими разделами

- A3 может подробно сравнивать SPDD, Spec Kit, Kiro и SDD spectrum. A2 использует их только для доказательства роли specification.
- Узел про architecture fitness / contract tests / test data может раскрыть Pact, oasdiff, can-i-deploy, TDAD и migration parity глубже. A2 использует их только для различения contract/oracle и ADR.
- Lifecycle repair / stale ADR section должен продолжить мысль о post-merge обновлении specification, contracts, ADR and projections. A2 только задаёт мост к этому разделу.

## Оставшаяся проверка качества

Текущий фрагмент не должен дальше расти количеством источников. Новые источники можно добавлять только если они усиливают одну из трёх подмен или помогают точнее описать boundary между intent, evidence and decision memory.

## Остаточная починка c59bb38f83

План остаточной починки:

2. Смягчить сухой переход от contract/oracle к ADR через MADR `Confirmation`, не добавляя новых источников.

- `A2_open_questions.md` дополнен вопросом о том, нужно ли в финальной терминологии держать `contract/oracle` как связку или разделить contract и oracle.

- Короткий объём Winder.AI и DEV migration story.
- Разделение `status` и `origin` для generated/reconstructed ADR.
