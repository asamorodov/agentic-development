# GSD / Open GSD

## Черновое назначение

Документ создан режимом свежего сбора для накопления материала из первоисточников. Это рабочее методологическое досье по GSD / Open GSD: не финальный раздел сайта и не краткая карточка, а буфер фактуры о том, как методология устроена как процесс разработки с ИИ-агентами.

Текущее состояние источников: девятый содержательный проход, 2026-06-09. Основной корпус фактуры взят из актуальных первичных источников Open GSD: сайт, документация `gsd-core`, документация `gsd-pi`, GitHub-репозитории, дорожная карта, страницы `Promise` и `Origin Credit`. Второй проход углубил внутреннюю механику: архитектура/конфигурация `gsd-core`, команды/конфигурация/диагностика проблем/миграция `gsd-pi`, восстановление и диагностику состояния. Третий проход добавил низкоуровневые контракты агентских ролей `gsd-core`: `gsd-planner`, `gsd-plan-checker` и `gsd-executor`. Четвёртый проход сверил эти контракты с текущими файлами `open-gsd/gsd-core/main/agents/*` и уточнил, что это уже не только происхождение из `get-shit-done-redux`, а актуальная поверхность `gsd-core`. Пятый проход добавил детали наблюдаемости, журнала аудита, границы безопасности для ключей, команд `headless`, MCP и forensics в `gsd-pi`, хуки/рабочие процессы и контекст среды выполнения. Шестой проход добавил текущие детали `gsd-core` из `docs/v1/user-guide`, `docs/v1/configuration`, `docs/INVENTORY.md` и `docs/issue-driven-orchestration.md`: маршрутизацию пространств имён, валидацию Nyquist, защиту от дрейфа плана, ревью кода, backlog/seeds/threads/workstreams, усиление безопасности, оркестрацию от issue, синхронизацию состояния и контроль стоимости/моделей. Седьмой проход добавил низкоуровневые справочные файлы `gsd-core` (`checkpoints.md`, `gates.md`, `context-budget.md`) и первичную документацию `gsd-browser`, чтобы точнее описать человеческие остановки, типы контрольных остановок, бюджет контекста и браузерные свидетельства как сопутствующий слой проверки. Восьмой проход добавил детали `verification-patterns.md`, `artifact-types.md` и `model-profiles.md`: четыре уровня проверки реализации, жизненный цикл и потребителей артефактов, `HANDOFF.json` / `.continue-here.md`, маршрутизацию профилей моделей, `inherit` для сред выполнения не от Anthropic и динамическую эскалацию маршрутизации. Девятый проход уточнил текущую `gsd-core`-конфигурацию и команды: фазовые `models`, `model_profile`, `fast_mode`, `effort`, `agent_skills`, `global_learnings`, `graphify`, `ingest_docs`, `parallelization`, написание команд Codex, `worktree-path-safety` и обновлённый контекст релизов и версий [источники: `gsd-core` документация по конфигурации, https://www.opengsd.net/docs/v1/configuration; `gsd-core` документация по командам, https://www.opengsd.net/docs/v1/commands; GitHub-релизы `open-gsd/gsd-core`, https://github.com/open-gsd/gsd-core/releases].

## Короткая ориентация

GSD в текущем Open GSD-контуре означает `Git. Ship. Done.` и описывает дисциплинированный цикл работы с ИИ-агентами для разработки: обсудить решение, спланировать, выполнить, проверить и довести до поставки. В старой и разговорной традиции название также раскрывалось как `Get Shit Done`; Open GSD сохраняет эту линию, но в продуктовой поверхности использует более нейтральное `Git. Ship. Done.` [источники: GitHub `open-gsd/gsd-core`, https://github.com/open-gsd/gsd-core; главная страница Open GSD, https://www.opengsd.net/; Origin Credit, https://opengsd.net/origin].

В Open GSD есть несколько поверхностей, которые важно не смешивать:

- `gsd-core` — фреймворк на основе промптов / набор команд, навыков и агентов, который встраивает GSD-цикл в существующие среды выполнения ИИ-агентов для разработки: Claude Code, Codex, Gemini CLI, OpenCode, Copilot, Cursor, Windsurf и другие. Он хранит рабочее состояние в `.planning/` и ведёт разработку фазами [источники: страница продукта `gsd-core`, https://opengsd.net/products/gsd-core; GitHub `open-gsd/gsd-core`, https://github.com/open-gsd/gsd-core].
- `gsd-pi` — локальный агент разработки и оркестратор, работающий напрямую в терминале, который сам ведёт этапы, срезы и задачи, хранит состояние среды выполнения в SQLite-базе проекта и проецирует состояние в `.gsd/` markdown-файлы для ревью, контекста промптов и истории, удобной для git [источники: GitHub `open-gsd/gsd-pi`, https://github.com/open-gsd/gsd-pi; документация Auto Mode, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].
- `gsd-browser` — отдельный сопутствующий продукт для браузерного доказательства: детерминированное управление Chrome, MCP-инструменты, проверки-утверждения, трассы, снимки экрана, записи и передача управления человеку. Дорожная карта явно говорит, что это не поверхность GSD, а продукт рядом с GSD, который может подставлять свидетельства в цикл проверки [источник: дорожная карта, https://opengsd.net/roadmap].
- `gsd-workbench` и `gsd-cloud` заявлены как будущие поверхности: настольная рабочая среда и размещённое облако для проектного состояния и удалённого ведения этапов [источник: дорожная карта, https://opengsd.net/roadmap].

Для будущей теории важно писать не про один инструмент, а про семейство поверхностей вокруг одного рабочего цикла. `gsd-core` показывает GSD как методологию поверх уже выбранного агента. `gsd-pi` показывает тот же мотив как самостоятельную среду выполнения / оркестратор с базой состояния, супервизорной логикой, worktrees, восстановлением и контролем стоимости.

Уточнение после открытия источников: текущая документация `gsd-core` также показывает сильный ход в сторону экономии токенов. v1.40 поставляет шесть метанавыков пространств имён (`/gsd-workflow`, `/gsd-project`, `/gsd-quality`, `/gsd-context`, `/gsd-manage`, `/gsd-ideate`) так что системный промпт может показывать примерно 120 токенов маршрутизаторов вместо плоского списка из 86 навыков примерно на 2 150 токенов; конкретные команды остаются доступными для прямого вызова [GSD User Guide](https://www.opengsd.net/docs/v1/user-guide). Это не просто аккуратность UI. Это методологический ответ на ту же проблему, которую мы видели в многопроходном документном процессе: поверхности инструментов и процессов становятся слишком большими для дешёвого показа в промпте, поэтому им нужна иерархическая маршрутизация.

## Роль в ИИ-ориентированном SDLC

GSD занимает место между свободной разработкой через чат и тяжёлой корпоративной SDLC. Его задача — сделать работу ИИ-агентов для разработки проверяемой и возобновляемой, когда фича уже слишком сложна для одного короткого промпта, но команда ещё не хочет заводить отдельную бюрократическую систему планирования.

Главный тезис Open GSD: инженерия с ИИ деградирует, когда вся работа копится в одном длинном контексте. Сайт формулирует это через `context bloat`, `explicit plans`, `clean execution contexts`, `real verification` и историю git, которая отражает реальную работу [источник: главная страница Open GSD, https://www.opengsd.net/]. В `gsd-core` это раскрыто как `context rot`: модель не ломается резко, но начинает хуже удерживать ранние решения, стиль, ограничения, имена файлов и сигнатуры функций по мере заполнения окна контекста [источник: `docs/explanation/context-engineering.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/explanation/context-engineering.md].

GSD отвечает на это не одним большим файлом спецификации, а разбиением разработки на ограниченные фазы / срезы и переносом состояния в долговечные артефакты. Для `gsd-core` это `.planning/` с markdown/json-артефактами. Для `gsd-pi` это `.gsd/` слой проекций плюс SQLite-база данных как авторитетное состояние среды выполнения. В обоих вариантах один артефакт становится входом для следующего шага, а агенту не нужно держать весь ход работы в памяти одной беседы.

Архитектура `gsd-core` прямо описывает его как фреймворк метапромптинга между пользователем и ИИ-агентами для разработки. Внутри есть слой команд (`commands/gsd/*.md`), слой рабочих процессов (`get-shit-done/workflows/*.md`), слой агентов со свежим контекстом, слой CLI-инструментов (`gsd-tools.cjs` и узел маршрутизации команд) и файловый слой `.planning/`. Это важно для будущей теории: GSD не только набор красивых команд, а система маршрутизации промптовых рабочих процессов, где тонкий оркестратор загружает контекст, запускает специализированных агентов, собирает результаты и обновляет состояние между шагами [источник: `gsd-core` architecture, https://www.opengsd.net/docs/v1/architecture].

Текущая `gsd-core`-документация показывает ещё один методологический мотив: сама поверхность команд стала достаточно большой, чтобы требовать слой маршрутизации. `User Guide` описывает шесть метанавыков пространств имён: `/gsd-workflow`, `/gsd-project`, `/gsd-quality`, `/gsd-context`, `/gsd-manage`, `/gsd-ideate`. Они нужны не как новые ручные команды, а как первый слой маршрутизации, чтобы модель видела короткий список маршрутизаторов вместо плоского списка из десятков навыков; конкретные команды вроде `/gsd-plan-phase` всё равно можно вызывать напрямую [источник: `gsd-core` User Guide, https://www.opengsd.net/docs/v1/user-guide].

## Проблема, которую решает методология

### `context rot` и деградация длинной сессии

Документация `gsd-core` описывает context rot как снижение качества ответов в длинной сессии разработки: модель начинает противоречить ранним решениям, уходить от стиля проекта, игнорировать требования, которые были заданы раньше, или выдумывать имена файлов и сигнатуры, которые ранее называла правильно [источник: `context-engineering.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/explanation/context-engineering.md].

Простое `/clear` решает проблему шума, но теряет непрерывность. GSD предлагает другой ход: основной оркестратор остаётся тонким, тяжёлую работу выполняют субагенты со свежим контекстом, а результаты фиксируются в файлах. В `gsd-core` оркестратор не должен сам лезть в исходные файлы; он запускает агентов, собирает результаты, обновляет общее состояние и маршрутизирует следующий шаг [источник: `context-engineering.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/explanation/context-engineering.md].

### Непроверенная реализация

GSD исходит из того, что "код написан" не равно "фаза завершена". В `gsd-core` шаг Verify сверяет цель фазы, решения из `CONTEXT.md`, планы и сводки выполнения; он проверяет покрытие требований, покрытие решений и согласованность с целью фазы. Если есть расхождения, создаются целевые планы исправлений [источник: `the-phase-loop.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/explanation/the-phase-loop.md].

В `gsd-pi` эта идея доведена до механики среды выполнения: после единицы работы GSD проверяет наличие ожидаемого артефакта на диске; для `run-uat` файл оценки должен иметь канонический вердикт, например `PASS`, `FAIL` или `PARTIAL`. Повторные проверки артефакта ограничены тремя попытками; после этого автоматический режим ставится на паузу вместо бесконечного цикла [источник: документация Auto Mode, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

### Потеря состояния между сессиями

GSD фиксирует решения, требования, дорожную карту, планы, сводки, проверку и передачу контекста в файловых или артефактах, поддержанных БД. В `gsd-core` `STATE.md` — навигационный слой, который показывает активный этап, фазу, завершённые/ожидающие планы и блокеры; рабочие процессы читают его при старте [источник: `the-phase-loop.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/explanation/the-phase-loop.md].

В `gsd-pi` SQLite-база данных является источником истины для этапов, срезов, задач, требований, сводок и статуса завершения. Markdown-файлы `.gsd/` — проекции для ревью, промптов и истории, удобной для git; редактирование проекций само по себе не перезаписывает состояние базы данных, если команда GSD явно не импортирует или не сохраняет изменения [источник: документация Auto Mode, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

## Рабочий процесс: gsd-core

`gsd-core` организует работу этап за этапом, фазу за фазой. Центральная модель: `Discuss -> (UI design) -> Plan -> Execute -> Verify -> Ship` [источник: `the-phase-loop.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/explanation/the-phase-loop.md].

### 1. Инициализация / новый проект

Стартовая команда:

```text
/gsd-new-project
```

Она задаёт вопросы, может запускать исследование, формирует требования и дорожную карту. В руководстве по первому учебному проекту результатом становятся `.planning/PROJECT.md`, `.planning/REQUIREMENTS.md`, `.planning/ROADMAP.md`, `.planning/STATE.md` и `.planning/config.json` [источники: `your-first-project.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/tutorials/your-first-project.md; `COMMANDS.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/COMMANDS.md].

Для существующего репозитория README рекомендует сначала выполнить `/gsd-map-codebase`, чтобы переиндексировать кодовую базу, а затем `/gsd-new-project` для восстановления контекста планирования [источник: GitHub `open-gsd/gsd-core`, https://github.com/open-gsd/gsd-core].

### 2. Фаза Discuss

Команда:

```text
/gsd-discuss-phase [N]
```

Фаза Discuss фиксирует решения реализации до планирования: библиотеки, обработку ошибок, форму API, структуры данных, краевые случаи, поведение UI. Документация подчёркивает, что цель фазы в `ROADMAP.md` говорит, что надо получить, а Discuss нужен, чтобы решить, как именно это строить. Иначе планировщик будет сам делать правдоподобные, но потенциально неверные предположения [источник: `the-phase-loop.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/explanation/the-phase-loop.md].

Выход: `CONTEXT.md` в каталоге фазы и `DISCUSSION-LOG.md` как журнал аудита. Учебное руководство показывает файл `.planning/phases/01-core-cli/CONTEXT.md` с `## Implementation Decisions`; планировщик затем читает этот файл, поэтому решения пользователя доходят до планов задач [источники: `COMMANDS.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/COMMANDS.md; `your-first-project.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/tutorials/your-first-project.md].

### 3. Необязательное проектирование UI

Для фронтенд/UI-фазы есть отдельный шаг:

```text
/gsd-ui-phase [N]
```

Он создаёт `{phase}-UI-SPEC.md` / `UI-SPEC.md`: контракт дизайна для макета, взаимодействия и визуального поведения до написания кода. Документация считает этот шаг полезным, когда UI достаточно сложен, чтобы неоднозначность дизайна привела к расходящимся реализациям [источники: `the-phase-loop.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/explanation/the-phase-loop.md; `COMMANDS.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/COMMANDS.md].

### 4. Фаза Plan

Команда:

```text
/gsd-plan-phase [N]
```

Фаза Plan выполняет исследование, декомпозицию и проверку плана. В описании фазового цикла участвуют субагенты со свежим контекстом: исследователь пишет `RESEARCH.md`, планировщик читает `RESEARCH.md` и `CONTEXT.md`, создаёт файлы `PLAN.md`, а plan-checker проверяет полноту, согласованность и область [источник: `the-phase-loop.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/explanation/the-phase-loop.md].

Каждый `PLAN.md` должен описывать ограниченную единицу работы: затрагиваемые файлы, конкретные изменения, критерии приёмки. Планы упорядочиваются в волны зависимостей, чтобы параллельное исполнение было безопасным: исполнители в одной волне не должны трогать пересекающиеся области ответственности [источник: `the-phase-loop.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/explanation/the-phase-loop.md].

Команда поддерживает флаги, которые показывают зрелость рабочего процесса: `--research`, `--skip-research`, `--validate`, `--prd`, `--ingest`, `--reviews`, `--bounce`, `--mvp`, `--tdd`. `--mvp` организует задачи как вертикальные срезы и может породить `SKELETON.md` для Walking Skeleton; `--tdd` помечает подходящие задачи, добавляющие поведение как начинающиеся с падающего теста [источник: `COMMANDS.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/COMMANDS.md].

Отдельно важна контрольная остановка легитимности пакетов: если исследователь рекомендует внешние пакеты, `slopcheck install <package> --json` пишет таблицу аудита в `RESEARCH.md`; `[SLOP]` удаляется, `[SUS]` и WebSearch `[ASSUMED]` получают `checkpoint:human-verify` перед задачей установки. Если `slopcheck` недоступен, все пакеты считаются `[ASSUMED]` и контрольная остановка включается [источник: `COMMANDS.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/COMMANDS.md].

Документация по архитектуре уточняет, что проверка пакета встроена в цепочку исследователь -> планировщик -> исполнитель: исследователь запускает проверки по конкретному реестру, планировщик добавляет `checkpoint:human-verify` и строку цепочки поставки для планов с установкой пакетов, исполнитель не делает тихие подстановки при провалившейся установке. В той же цепочке есть проверка покрытия решений: на фазе планирования она блокирующая, а на фазе проверки валидация уже неблокирующая; это разделяет "не планировать без переноса решений" и "зафиксировать расхождение после реализации" [источник: `gsd-core` architecture, https://www.opengsd.net/docs/v1/architecture].

В текущем `User Guide` поверх прежних контрольных остановок явно вынесена `предварительная проверка Nyquist`. Она срабатывает перед выполнением планов и проверяет планы, ссылаемые файлы, критерии приёмки и граф зависимостей до того, как агент начнёт менять код. Если находка исправима автоматически, GSD пишет `PLAN-REPAIRS.md` и пересобирает план; если нужен человек, создаётся `CHECKPOINT-NYQUIST.md`; если всё чисто, появляется `NYQUIST-PASS.md`. Для методологии это важная деталь: часть проверки сдвинута в момент перед отправкой на исполнение, где ещё дёшево остановить плохой план [источник: `gsd-core` User Guide, https://www.opengsd.net/docs/v1/user-guide].

Низкоуровневый `gsd-planner` показывает, что `PLAN.md` в GSD задуман как исполняемый запрос для отдельного агента-исполнителя, а не как предварительная заметка. Планировщик обязан сначала разобрать зафиксированные решения из `CONTEXT.md`, ссылаться на `D-NN` в действиях задачи, не переносить отложенные идеи в план и не заменять пользовательские решения формулировками вроде `v1`, `static for now`, `future enhancement`, `placeholder` или `basic version`. Если весь объём не помещается в безопасный контекст, планировщик должен предложить разделение фазы, а не молча урезать требования [источник: `agents/gsd-planner.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-planner.md].

У `gsd-planner` есть обязательный аудит покрытия нескольких источников по четырём типам входа: `GOAL` из `ROADMAP.md`, `REQ` из `REQUIREMENTS.md`, `RESEARCH` из `RESEARCH.md` и решения `CONTEXT`. Каждый элемент должен быть покрыт планом; если что-то не покрыто, планировщик возвращает предупреждение о незапланированных пунктах с вариантами: добавить план, разделить фазу или отложить только после подтверждения разработчика. Это важная деталь для будущей теории: GSD пытается не только разбивать работу, но и формально защищать перенос решений из предыдущих артефактов в исполняемую работу [источник: `agents/gsd-planner.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-planner.md].

Анатомия задачи у `gsd-planner` жёсткая: каждая задача должна иметь точные пути файлов, конкретные действия, команду проверки и измеримые критерии готовности. Проверка должна быть выполнимой быстро; если теста нет, план обязан явно сказать, что волна 0 создаёт тестовый каркас. В `must_haves` планировщик выводит наблюдаемые факты, обязательные артефакты и ключевые ссылки: что пользователь должен уметь сделать, какие файлы или объекты базы данных должны существовать, какие соединения между компонентами, API и базе данных особенно вероятно сломаются [источник: `agents/gsd-planner.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-planner.md].

Проход 04 сверил те же контракты агентских ролей в текущем `open-gsd/gsd-core`. Это закрывает часть осторожности из прохода 03: `gsd-planner`, `gsd-plan-checker` и `gsd-executor` лежат прямо в актуальном `gsd-core`, поэтому их можно использовать как первичный источник для текущей методологии, а не только как исторический источник. При этом старое название репозиторий всё ещё полезен как происхождение для линии наследования, но финальный текст может ссылаться на текущие файлы `open-gsd/gsd-core/main/agents/*` [источники: `agents/gsd-planner.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-planner.md; `agents/gsd-plan-checker.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-plan-checker.md; `agents/gsd-executor.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-executor.md].

Из текущего `gsd-planner` для будущего Handbook важна ещё одна рабочая деталь: планирование не должно выдавать задачи без явной модели владения файлами. Для параллельного выполнения `PLAN.md` должен делать конфликтующие пути видимыми заранее; если две задачи трогают один и тот же файл или одну и ту же область состояния, они не должны попадать в одну волну. Это связывает качество плана с будущей безопасностью волны исполнителей, а не только с полнотой текста плана [источник: `agents/gsd-planner.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-planner.md].

Планировщик также переводит размер работы в бюджет контекста, а не в человеко-часы. План должен обычно завершаться примерно в пределах 50% контекста; одна задача целится в 10-30% контекста, TDD-план — около 40%, потому что `RED -> GREEN -> REFACTOR` требует дополнительного чтения, запусков и анализа вывода. Это не только оптимизация: в логике GSD качество падает при росте контекста, поэтому планирование управляет размером задач через ожидаемое потребление контекста [источник: `agents/gsd-planner.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-planner.md].

Для `--mvp` планировщик требует вертикальные срезы функции: после каждой задачи реальный пользователь должен получить новую наблюдаемую возможность. Горизонтальная инфраструктурная задача допускается только как Walking Skeleton, где создаётся `SKELETON.md` с framework, DB, auth, deployment и решениями по структуре каталогов; последующие фазы не должны заново обсуждать эти решения. Если одновременно включён TDD-режим, первая задача становится падающим end-to-end-тестом для успешного сценария [источник: `agents/gsd-planner.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-planner.md].

Новый `Plan Drift Guard` закрывает ещё один слабый участок: план может быть хорошим в момент создания, но устареть после изменения кода, `CONTEXT.md`, `REQUIREMENTS.md`, `ROADMAP.md` или исследовательских файлов. `User Guide` описывает проверку `/gsd-plan-status`, которая сравнивает заголовки плана, хэши зависимостей, `source_sha` и `HEAD`; статусы делятся на `valid`, `stale`, `orphaned`, `dirty` и `expired`. Если план устарел, GSD предлагает обновить, синхронизировать, перепланировать или отдать ситуацию на ручное ревью. Это отличает "план есть" от "план всё ещё применим к текущему репозиторию" [источник: `gsd-core` User Guide, https://www.opengsd.net/docs/v1/user-guide].

### 5. Фаза Execute

Команда:

```text
/gsd-execute-phase <N>
```

Фаза Execute запускает планы, часто параллельными волнами. Каждый исполнитель получает свежее окно контекста со сводкой проекта, контекстом фазы, исследованием и конкретным `PLAN.md`. Исполнитель пишет код и делает атомарный коммит; один коммит соответствует завершённой задаче в плане [источники: `the-phase-loop.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/explanation/the-phase-loop.md; `COMMANDS.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/COMMANDS.md].

`COMMANDS.md` уточняет, что команда создаёт сводки по планам, git-коммиты и артефакт проверки фазы после полного завершения. Если шаг установки падает из-за пакета, исполнитель выводит `checkpoint:human-verify` и останавливается; он не должен молча устанавливать альтернативу с похожим именем, потому что это риск slopsquatting [источник: `COMMANDS.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/COMMANDS.md].

`gsd-executor` исполняет `PLAN.md` атомарно: читает frontmatter, цель, ссылки на контекст, задачи, проверку и критерии успеха; для `type="auto"` выполняет задачу, проверяет критерии готовности, делает коммит и фиксирует завершение с хешем коммита для `SUMMARY.md`. Если план содержит checkpoint, исполнитель выполняет работу до checkpoint и останавливается со структурированным checkpoint-сообщением; продолжение должно запускаться свежим агентом, который сверяет предыдущие коммиты и возобновляется с указанной точки [источник: `agents/gsd-executor.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-executor.md].

Правила отклонений у исполнителя показывают, где GSD проводит границу автономности. Сбои, недостающая критическая функциональность и блокеры, напрямую вызванные текущей задачей, исправляются автоматически с тестами/проверками и записываются в сводку. Но архитектурное изменение — новая таблица, серьёзная схема, новый сервисный слой, смена библиотеки, подход к аутентификации, инфраструктура или ломающий API — требует остановки и checkpoint с описанием влияния и альтернатив. Сбой установки через менеджер пакетов тоже не допустимый для автоматического исправления: исполнитель не ищет похожий пакет и не повторяет установку под другим именем, а требует `checkpoint:human-verify` [источник: `agents/gsd-executor.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-executor.md].

Контракт исполнителя содержит явный сигнал застревания: если во время выполнения задачи агент сделал пять и более подряд `Read`/`Grep`/`Glob` без `Edit`/`Write`/`Bash`, он должен остановиться, коротко объяснить, почему ещё ничего не написал, и либо начать действие, либо признать конкретный блокер. Это неудобная, но ценная деталь: GSD пытается обнаружить бесконечное чтение и аналитический паралич внутри самой агентской роли, а не только на уровне внешнего таймера [источник: `agents/gsd-executor.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-executor.md].

Между выполнением и UAT появился отдельный слой ревью: `/gsd-code-review-phase`, `/gsd-code-review-plan`, `/gsd-code-review-file`, `/gsd-code-review-task`, `/gsd-code-review-changes`, `/gsd-code-review-diff`, `/gsd-code-review-commit`, `/gsd-code-review-branch` и `/gsd-code-review-pr`. Документация позиционирует его как проверку с новой агентской перспективы до поставки: корректность, сопровождаемость, безопасность, тесты и согласованность с планом. Важная граница: ревью не заменяет `/gsd-verify-work`, потому что он оценивает изменение как код и процесс, а проверка проверяет наблюдаемое поведение и критерии приёмки [источник: `gsd-core` User Guide, https://www.opengsd.net/docs/v1/user-guide].

### 6. Проверка работы

Команда:

```text
/gsd-verify-work [N]
```

Это пользовательская приёмочная проверка с автоматической диагностикой. В учебном руководстве GSD пошагово спрашивает пользователя, проходят ли наблюдаемые проверки (`node todo.js add`, `todo list`, `todo done`), записывает результаты в `UAT.md`, а при провале диагностирует причину и создаёт план исправления [источник: `your-first-project.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/tutorials/your-first-project.md].

Для браузерного UAT текущий сопутствующий продукт Open GSD — `gsd-browser`, который даёт детерминированную навигацию, версионированные ссылки, проверки-утверждения, снимки экрана, визуальные сравнения, записи и передачу управления человеку. Устаревшие серверы Playwright MCP остаются пригодными, если уже настроены [источник: `COMMANDS.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/COMMANDS.md].

`gsd-browser` стоит описывать как бэкенд свидетельств рядом с GSD, а не как ещё один вариант `gsd-core`. Его страница продукта и документация показывают отдельный операционный цикл: установить CLI, запустить `gsd-browser mcp`, навигировать, делать снимки с версионированными ссылками вроде `@v1:e1`, выполнять действия, явно проверять результаты через проверки-утверждения и сохранять доказательства. В доказательства входят снимки экрана, записи, трассы, HAR, PDF-экспорты, аннотации, debug bundles и сессии live viewer, которые человек может проверить или взять под управление [источники: страница продукта `gsd-browser`, https://opengsd.net/products/gsd-browser; `gsd-browser` CLI & MCP Commands, https://opengsd.net/docs/v5/commands].

### 7. Ship

Команда:

```text
/gsd-ship [N]
```

Ship создаёт PR из проверенной работы фазы, требует пройденный `/gsd-verify-work`, установленный и аутентифицированный `gh` CLI. Тело PR включает цель фазы из `ROADMAP.md`, сводку изменений из `SUMMARY.md`, покрытые REQ-IDs, статус проверки, ключевые решения и необязательные настроенные секции в PRD-стиле из `ship.pr_body_sections`. `STATE.md` обновляется [источник: `COMMANDS.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/COMMANDS.md].

### Навигация и лёгкие пути

`/gsd-progress --next` автоматически определяет и запускает следующий логический шаг рабочего процесса: нет проекта -> `/gsd-new-project`, фаза требует обсуждения -> `/gsd-discuss-phase`, требует планирования -> `/gsd-plan-phase`, требует выполнения -> `/gsd-execute-phase`, требует проверки -> `/gsd-verify-work`, все фазы завершены -> предлагает `/gsd-complete-milestone` [источник: `COMMANDS.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/COMMANDS.md].

Для задач ниже порога полного фазового цикла есть `/gsd-quick` и `/gsd-fast`. Документация прямо говорит, что фазовый цикл чрезмерен для переименования переменной, исправления опечатки или недостающего импорта; правило: если задача полностью задаётся коротким промптом и может быть выполнена за один агентский ход без уточнения, лучше пропустить фазовый цикл [источник: `context-engineering.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/explanation/context-engineering.md].

Командная поверхность `gsd-core` показывает, что "лёгкий путь" не сводится к одному быстрому промпту. `/gsd-manager` умеет ставить фоновые задачи по чек-листу, хранить их в `.planning/todos/`, показывать heartbeat/status и откатывать состояние задачи; `/gsd-ingest-docs` принимает внешние документы в `.planning/`, классифицирует их как `prd`, `api-docs`, `tech-spec`, `design`, `requirements` или `notes`, дедуплицирует и пишет конфликты в `INGEST-CONFLICTS.md`; `/gsd-graphify` строит граф зависимостей из `ROADMAP.md`/requirements/issues/plans/карты кода в `.planning/graphs/` и затем может выводить критический путь или заблокированную работу. Это важный источник фактуры: вокруг фазового цикла есть слой управления входящим материалом и очередью работы, чтобы не превращать каждый небольшой запрос в полный этап [источник: `gsd-core` документация по командам, https://www.opengsd.net/docs/v1/commands].

### Внутренняя механика `gsd-core`

`gsd-core` держит отдельные справочные файлы для контрольных остановок, checkpoints, профилей моделей, паттернов проверки, конфигурации планирования, git-интеграции, бюджета контекста, формата продолжения, domain probes, revision loop, anti-patterns и типов артефактов. Это объясняет, почему методология выглядит как "команды", но ведёт себя как повторяемая система: команда рабочего процесса не должна заново изобретать правила проверки, ветвления или продолжения; она подключает общий справочный слой [источник: `gsd-core` architecture, https://www.opengsd.net/docs/v1/architecture].

Справочный слой полезен ещё и тем, что фиксирует правила, которые легко потерять в высокоуровневом описании. `context-budget.md` запрещает оркестратору читать `agents/*.md`, потому что `subagent_type` сам загружает описание агента; запрещает встраивать большие файлы в промпты субагентов; для окна контекста меньше 500K tokens рекомендует читать только frontmatter/status/summaries у тяжёлых артефактов, а полные тела `SUMMARY.md`, `VERIFICATION.md` и `RESEARCH.md` оставлять для больших окон и реальной необходимости. Это делает "тонкий оркестратор" не метафорой, а конкретной дисциплиной чтения [источник: `context-budget.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/get-shit-done/references/context-budget.md].

Тот же файл вводит уровни деградации: при 0-30% контекста возможны полные операции, при 30-50% нужно предпочитать frontmatter и делегирование, при 50-70% экономить и предупреждать пользователя, при 70%+ сразу зафиксировать прогресс. Набор предупреждающих признаков практичен: тихое частичное завершение, возрастающая расплывчатость формулировок и пропуск шагов протокола. Важная оговорка источника: оркестратор не может семантически доказать качество результата субагента, он проверяет прежде всего структурную полноту, поэтому нужны `must_haves` и точечная проверка [источник: `context-budget.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/get-shit-done/references/context-budget.md].

`model-profiles.md` показывает, что управление стоимостью в GSD встроено не только в размер контекста, но и в маршрутизацию моделей. Базовые профили `quality`, `balanced`, `budget`, `adaptive` и `inherit` задают разные модели для ролей: планировщик и отладчик получают более сильные модели там, где важны архитектурные решения и диагностика; картограф кодовой базы, синтезатор, проверяющий и аудитор могут уходить на более дешёвые модели, если задача структурная и массовая. Это методологически важно: GSD не предполагает, что каждый агент должен работать на максимальной модели, а связывает качество модели с ценой ошибки в конкретной фазе [источник: `model-profiles.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/get-shit-done/references/model-profiles.md].

В том же reference-файле есть два неудобных операционных правила. Для сред выполнения не от Claude установщик выставляет `resolve_model_ids: "omit"` в `~/.gsd/defaults.json`, чтобы субагенты использовали модель по умолчанию выбранной среды выполнения; если человек хочет разные модели по агентам, он задаёт `model_overrides` именами, которые понимает эта среда выполнения, например `o3` или `o4-mini`. Для Claude Code с OpenRouter или локальным провайдером нужен профиль `inherit`; иначе профиль `balanced` по умолчанию может порождать алиасы моделей Anthropic и привести к неожиданным расходам. Динамическая маршрутизация, если включена, берёт уровень агента по умолчанию (`light`, `standard`, `heavy`) и при мягком сбое может подняться на следующий уровень, но жёсткие сбои должны всплывать сразу [источник: `model-profiles.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/get-shit-done/references/model-profiles.md].

Текущая страница конфигурации раскрывает тот же механизм уже как пользовательскую конфигурацию. `models.planning`, `models.execution`, `models.verification`, `models.research`, `models.code_review`, `models.ui_review`, `models.debug`, `models.issue_triage`, `models.graph_analysis` задают фазовые модели; `model_overrides.<agent-name>` перекрывает их для конкретных ролей; `runtime: "claude" | "codex" | "gemini" | "opencode" | "github-copilot" | "cursor" | "windsurf" | "kilo"` включает настройки с учётом среды выполнения по умолчанию. `model_profile: "quality" | "balanced" | "budget" | "adaptive"` остаётся более крупной ручкой, а `fast_mode` и `effort` дают быстрый путь к дешёвым/скоростным настройкам без ручного заполнения всех переопределений агентов. Для методологии это означает, что выбор модели становится частью политики рабочего процесса, а не личной привычкой оператора [источник: `gsd-core` документация по конфигурации, https://www.opengsd.net/docs/v1/configuration].

В той же конфигурации появились `agent_skills` и `global_learnings`, которые точнее отвечают на вопрос "где живёт контекст". `agent_skills` управляет тем, какие проектные skills подмешиваются к субагентам: можно задать глобальные skills для всех агентов, skills по имени агента и skills по типу фазы. `global_learnings` задаёт файл, куда пишутся повторно используемые паттерны, предпочтения и уроки; это не артефакт фазы, а слой памяти поверх отдельных фаз. Такая настройка опасна при переносе между проектами: неправильно подобранные skills или глобальные learnings могут незаметно изменить поведение планировщика/исполнителя сильнее, чем сама команда `/gsd-plan-phase` [источник: `gsd-core` документация по конфигурации, https://www.opengsd.net/docs/v1/configuration].

В `execute-phase` планы группируются в волны зависимостей. Каждый исполнитель получает свежий контекст, конкретный `PLAN.md`, контекст проекта из `PROJECT.md`/`STATE.md` и контекст фазы из `CONTEXT.md`/`RESEARCH.md`. Для больших окон контекста, начиная примерно с 500K токенов, включается адаптивное обогащение контекста: исполнители получают больше предыдущих сводок, а верификатор получает все `PLAN.md`, `SUMMARY.md`, `CONTEXT.md` и `REQUIREMENTS.md`. Для безопасности параллельных коммитов используются `--no-verify`-коммиты внутри волны с последующим запуском pre-commit-хуков один раз после волны, а запись `STATE.md` защищается lockfile-механикой [источник: `gsd-core` architecture, https://www.opengsd.net/docs/v1/architecture].

Файловая модель `.planning/` шире, чем базовые `PROJECT.md`/`REQUIREMENTS.md`/`ROADMAP.md`: документация по архитектуре перечисляет `research/`, `codebase/`, `quick/`, `todos/`, `threads/`, `seeds/`, `debug/`, `ui-reviews/` и `continue-here.md`. Для существующих проектов особенно важна `.planning/codebase/`: после последней волны выполнения неблокирующий `codebase_drift_gate` сравнивает diff от `last_mapped_commit` с `.planning/codebase/STRUCTURE.md` и, если появились новые структурные элементы, предупреждает или запускает ограниченный remap. Это не блокирует проверку, но показывает, что GSD пытается поддерживать карту кода после изменений [источник: `gsd-core` architecture, https://www.opengsd.net/docs/v1/architecture].

`User Guide` уточняет назначение трёх долго живущих поверхностей рядом с фазовым циклом. Backlog (`/gsd-capture`, `/gsd-todo-*`) хранит идеи, ошибки и будущую работу в `.planning/backlog/` до превращения в пункт дорожной карты. Seeds (`/gsd-seed`, `/gsd-seed-harvest`, `/gsd-seed-promote`) сохраняют архитектурные подсказки и спекулятивные идеи в `.planning/seeds/` и позволяют позже повысить их до требований или решений. Threads (`/gsd-thread`, `/gsd-thread-continue`, `/gsd-thread-status`, `/gsd-thread-close`) дают постоянный контекст исследования или обсуждения в `.planning/threads/`, чтобы длительное расследование не смешивалось с исполнением фазы. Это важно для переноса методологии: не весь контекст должен становиться контекстом фазы; часть контекста живёт как ожидание, гипотеза или отдельная нить исследования [источник: `gsd-core` User Guide, https://www.opengsd.net/docs/v1/user-guide].

Workstreams (`/gsd-workstream-create`, `/gsd-workstream-branch`, `/gsd-workstream-switch`, `/gsd-workstream-checkpoint`, `/gsd-workstream-merge`, `/gsd-workstream-status`, `/gsd-workstream-list`, `/gsd-workstream-close`) добавляют ещё один уровень изоляции: параллельные направления работы получают собственные ветки и состояние планирования. Это не та же вещь, что волны зависимостей внутри одной фазы: волны параллелят проверенные задачи плана, а workstreams разделяют более крупные направления, которые могут жить дольше одной фазы и позже сходиться через merge/checkpoint [источник: `gsd-core` User Guide, https://www.opengsd.net/docs/v1/user-guide].

На уровне контрактов агентских ролей эта механика становится ещё жёстче: `gsd-plan-checker` работает как предварительный Revision Gate. Он не исправляет планы молча и не запускает реализацию; его результат — `PASS`, `NEEDS_REVISION` или `BLOCKED_BY_MISSING_INPUT`. При `NEEDS_REVISION` checker должен назвать нужные конкретные изменения и ожидаемые обновления плана и не давать вердикт о готовности к исполнению. Это делает ревью плана самостоятельной фазой, а не украшением вокруг результата планировщика [источник: `agents/gsd-plan-checker.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-plan-checker.md].

`gsd-plan-checker` проверяет не только полноту списка задач. Его чек-лист включает покрытие цели, требований, исследования, явных решений из `CONTEXT.md`, отсутствие незапланированных TODO, порядок зависимостей, отсутствие пересечений у параллельных задач, наблюдаемые критерии приёмки, команды проверки, путь отката/восстановления, бюджет контекста и наличие ключевых ссылок. Отдельная проверка ловит планы "только для успешного сценария": если план описывает реализацию без пути ошибок, пустые состояния, аутентификацию/права, безопасность миграции/данных или видимые пользователю сбойные режимы, он не должен проходить как готовый к исполнению [источник: `agents/gsd-plan-checker.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-plan-checker.md].

Текущая сверка с `open-gsd/gsd-core` усилила важное различение: plan-checker не является вторым planner. Его работа — вернуть вердикт и объяснить, какие именно изменения должны появиться в плане. Это удерживает цепочку ответственности: planner исправляет артефакт плана, checker повторно оценивает, исполнитель получает только проверенный вход. Если checker начнёт "чинить по дороге", GSD потеряет проверяемую границу между ревью плана и исполнением [источник: `agents/gsd-plan-checker.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-plan-checker.md].

Наблюдаемость в `gsd-core` встроена в слой маршрутизации команд, но по умолчанию не шумит. При сбое routing hub пишет одну JSON-строку в stderr с `kind`, `traceId`, `command`, `timestamp` и `message`; `kind` различает `UnknownCommand`, `InvalidArgs`, `HandlerRefusal` и `HandlerFailure`. Полный журнал аудита включается явно через `GSD_AUDIT=1` или `config.audit.enabled`; тогда события dispatch пишутся в `.planning/.gsd-trace.jsonl`, один JSON object на строку, с `traceId` и, когда он передан, `parentTraceId`. Файл только дописываемый, добавлен в `.gitignore` и не ротируется автоматически, поэтому это операторский диагностический след, а не пользовательский артефакт сайта или документации [источник: `gsd-core` документация по конфигурации, https://www.opengsd.net/docs/v1/configuration].

У журнала аудита есть важная граница приватности. Аргументы команд по умолчанию не попадают ни в события stderr, ни в файл аудита; для записи аргументов нужен явный `GSD_AUDIT_ARGS=1`. В той же документации по конфигурации API-ключи для Brave/Firecrawl/Exa маскируются в UI, таблицах подтверждения, журналах и выводе `config-set`, но открытый текст хранится в `.planning/config.json`; документация прямо называет этот файл границей безопасности. Для будущего текста это неудобная, но важная деталь: GSD делает операции наблюдаемыми, но безопасность зависит от того, как команда обращается с `.planning/config.json` и журналами аудита [источник: `gsd-core` документация по конфигурации, https://www.opengsd.net/docs/v1/configuration].

Текущая документация по конфигурации добавляет `security_hardening`: атомарные записи с резервными файлами, проверку `.planning/` в `.gitignore`, проверку целостности установщика, принудительное применение политики для опасных git-команд и явные контрольные остановки разрешений для инструментов. Отдельный `safety.dangerous_git_policy` может быть `block`, `confirm` или `allow`; значение по умолчанию `confirm` требует явного согласия перед разрушительными git-операциями. В `tools.allowed`, `tools.denied` и `tools.confirm_required` можно задать политику для команд и MCP-инструментов. Это показывает, что GSD постепенно переносит безопасность из общих советов в конфигурационный слой процесса [источник: `gsd-core` документация по конфигурации, https://www.opengsd.net/docs/v1/configuration].

`parallelization` и безопасность worktree показывают другой край этой конфигурации. В `gsd-core` можно управлять `max_agents`, `conflict_strategy`, `auto_commit`, `branch_naming`, `wave_strategy`, `checkpoint_after_wave`, `checkpoint_after_task`, `require_clean_worktree`, `protect_branches` и `worktree_path`. Отдельный справочник `worktree-path-safety` описывает типовые сбои: HEAD остаётся на `main`, а ветка этапа создаётся рядом; агент стартует не из ожидаемого worktree; путь worktree выходит за допустимую базу; защищённая ветка попадает под запись. Это не второстепенная реализационная деталь: параллельное исполнение в GSD безопасно только при явной политике веток, путей и checkpoint-ритма [источники: `gsd-core` документация по конфигурации, https://www.opengsd.net/docs/v1/configuration; `gsd-core` документация по командам, https://www.opengsd.net/docs/v1/commands].

Отдельная деталь бюджета контекста относится к MCP как к внешнему ограничителю метода. `context-budget.md` подчёркивает, что каждый включённый MCP-сервер добавляет схему в каждый ход, даже если инструменты не вызываются; тяжёлые серверы могут стоить десятки тысяч токенов на ход. Это не настраивается через `.planning/config.json`, а живёт в `.claude/settings.json` через `enabledMcpjsonServers` / `disabledMcpjsonServers`. Для GSD это важный сбойный режим: методология может аккуратно выбирать `model_profile`, но потерять выигрыш из-за неаудированного MCP-набора перед длинной фазой [источник: `context-budget.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/get-shit-done/references/context-budget.md].

## Рабочий процесс: gsd-pi

`gsd-pi` переносит тот же мотив в самостоятельный локально-ориентированный агент. Установка:

```text
npx @opengsd/gsd-pi@latest
npm install -g @opengsd/gsd-pi@latest
```

Минимальные требования в Getting Started: Node.js 22.0.0, npm, Git 2.20 и учётные данные одного поддерживаемого провайдера моделей. Прямая глобальная установка и неинтерактивная установка `--yes` тоже описаны [источник: Getting Started, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/getting-started.md].

### Сессия и команды

Базовый запуск:

```text
gsd
```

Распространённые slash-команды внутри сессии:

```text
/gsd config
/gsd auto
/gsd quick "Describe the task"
/gsd status
```

`gsd-pi` хранит планирование проекта и состояние выполнения в `.gsd/` [источник: GitHub `open-gsd/gsd-pi`, https://github.com/open-gsd/gsd-pi].

Справочник команд показывает, что `gsd-pi` вырос из одного `/gsd auto` в полноценную рабочую оболочку. Базовая системная поверхность включает `/gsd prefs`, `/gsd model`, `/gsd mode`, `/gsd config`, `/gsd keys`, `/gsd doctor`, `/gsd inspect`, `/gsd show-config`, `/gsd init`, `/gsd mcp`, `/gsd context`, `/gsd skill-health`, `/gsd hooks`, `/gsd migrate`, `/gsd recover --confirm`, `/gsd rebuild markdown` и `/gsd rebuild database`. Важно, что `rebuild markdown` пересобирает проекции из канонической базы данных, а устаревшие проекции завершения карантинятся и не импортируются; `rebuild database` зарезервирован для пересборок, нативных для БД, и не импортирует markdown-проекции. Это усиливает общий принцип `gsd-pi`: markdown удобен для чтения, но не должен молча подменять истину среды выполнения [источник: `gsd-pi` commands, https://www.opengsd.net/docs/v2/commands].

Операционная поверхность тоже широкая: `/gsd queue` управляет будущими этапами, `/gsd debug` и `/gsd debug continue` ведут длительные debug-сессии, `/gsd dispatch` запускает конкретную фазу напрямую, `/gsd verdict` переопределяет вердикт проверки этапа с обоснованием, `/gsd history` и `/gsd usage` показывают историю и расход, `/gsd forensics` запускает debugger с полным доступом для разборов сбоев автоматического режима, `/gsd closeout` чинит провалившиеся git-действия закрытия, `/gsd worktree` управляет worktrees, `/gsd brief` создаёт автономные визуальные HTML-брифы, `/gsd extract-learnings` сохраняет Decisions/Lessons/Patterns/Surprises после этапа [источник: `gsd-pi` commands, https://www.opengsd.net/docs/v2/commands].

Для headless/CI-сценариев есть `gsd headless`: он запускает автоматический режим без TUI, умеет `next`, `query`, `dispatch`, `new-milestone --context`, `new-milestone --context-text`, принимает timeout и `--max-restarts`, а `gsd headless query` возвращает быстрый JSON-снимок с активным этапом/срезом/задачей, блокерами, следующим действием и стоимостью. Коды выхода различают завершение, ошибку/тайм-аут и блокировку. `gsd headless recover` даёт non-TTY-эквивалент `/gsd recover --confirm` и после него документация рекомендует проверять восстановленное состояние через `gsd headless query`. Это делает `gsd-pi` не только интерактивным терминальным агентом, но и потенциальной внутренней службой для внешнего оркестратора [источник: `gsd-pi` commands, https://www.opengsd.net/docs/v2/commands].

Ещё одна текущая поверхность — режим MCP-сервера. `gsd --mode mcp` запускает GSD как MCP-сервер через stdin/stdout и открывает инструменты из агентской сессии внешним ИИ-клиентам. Отдельно описан Cloud MCP Gateway Runtime: `gsd-cloud-mcp-gateway` поднимает HTTP-шлюз, а `gsd-daemon cloud` сопрягает и подключает локальную среду выполнения к этому шлюзу. Для методологического досье это важно как расширение границы GSD: методология может быть не только локальной TUI-сессией, но и инструментальным бэкендом для внешних клиентов и удалённых оркестраторов [источник: `gsd-pi` commands, https://www.opengsd.net/docs/v2/commands].

### Автоматический режим как машина состояний

Ключевой режим:

```text
/gsd auto
```

Автоматический режим описан как движок автономного исполнения: он берёт следующую единицу работы из авторитетного SQLite-состояния в корне проекта, создаёт свежую агентскую сессию, вставляет сфокусированный промпт с заранее встроенным релевантным контекстом, даёт LLM выполнить единицу работы, сохраняет результат в базе данных, обновляет markdown-проекции вроде `STATE.md` и отправляет на исполнение следующую единицу [источник: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

Основной цикл по срезу:

```text
Планирование (со встроенным исследованием)
-> Исполнение (по каждой задаче)
-> Завершение
-> Переоценка дорожной карты
-> Следующий срез
-> Проверка этапа
-> Завершение этапа
```

Планирование изучает кодовую базу, исследует релевантную документацию и декомпозирует срез на задачи с `must_haves`. Исполнение запускает каждую задачу в свежем окне контекста. Завершение пишет сводку, сценарий UAT, отмечает дорожную карту и делает коммиты. Переоценка проверяет, сохраняет ли дорожная карта смысл. Проверка этапа сравнивает критерии успеха дорожной карты с фактическими результатами перед закрытием этапа [источник: документация Auto Mode, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

Постепенное планирование: GSD может полностью спланировать первый срез, а последующие срезы оставить набросками с меткой `[sketch]` в `M###-ROADMAP.md`. Перед исполнением автоматический режим запускает `refine-slice`, чтобы превратить набросок в полный план среза на основании текущей кодовой базы и предыдущих сводок срезов [источник: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

### Режим глубокого планирования

Для проектов, где нужно предварительное обследование, включается:

```yaml
planning_depth: deep
```

или запуск:

```text
/gsd new-project --deep
/gsd new-milestone --deep
```

Глубокий режим добавляет поэтапный поток предварительного исследования перед планированием уровня этапа: предпочтения рабочего процесса -> контекст проекта -> требования -> решение об исследовании -> необязательное исследование проекта -> контекст/дорожная карта этапа. Артефакты: `.gsd/PREFERENCES.md`, `.gsd/PROJECT.md`, `.gsd/REQUIREMENTS.md`, `.gsd/runtime/research-decision.json`, `.gsd/research/STACK.md`, `FEATURES.md`, `ARCHITECTURE.md`, `PITFALLS.md`, затем `.gsd/этапами/<...>/M###-CONTEXT.md` и `M###-ROADMAP.md` [источник: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

Важная оговорка: исследование проекта носит информационный, а не обязательный характер. Если исследование выявило новое обязательство, его нужно добавить в `.gsd/REQUIREMENTS.md`, прежде чем планирование будет на него полагаться [источник: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

В документации v2 это ограничение сформулировано жёстче: `REQUIREMENTS.md` рендерится из требований, сохранённых в базе данных; агенты должны сохранять отдельные требования через `gsd_requirement_save`, а финальный `gsd_summary_save` для `REQUIREMENTS` проваливается, если активных строк требований нет. Это защита от ситуации, где markdown выглядит как контракт, но состояние среды выполнения не знает о требованиях [источник: Auto Mode docs, https://www.opengsd.net/docs/v2/auto-mode].

### Политика инструментов и границы действий

Каждая единица автоматического режима имеет `UnitContextManifest` с `ToolsPolicy`; GSD применяет политику до выполнения вызовов инструментов. Единицы исполнения используют режим `all` и могут редактировать файлы проекта, запускать shell-команды и отправлять субагентов. Единицы планирования/обсуждения обычно работают в режиме `planning`: широкие чтения, записи только в артефакты планирования внутри `.gsd/`, shell-команды только на чтение, без отправки субагентов. `planning-dispatch` разрешает отправку субагентов для изолированной разведки/планирования/ревью. Режим `docs` дополнительно разрешает явные glob-шаблоны документации вроде `docs/**`, верхнеуровневых `README*.md`, `CHANGELOG.md`, верхнеуровневых `*.md` [источник: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

Эта деталь важна для теории: в `gsd-pi` безопасность рабочего процесса не только соглашение промпта, но политика среды выполнения. Запрещённые записи, небезопасный bash и неправильная отправка субагентов должны блокироваться жёсткой ошибкой политики, а не оставаться "на совести" LLM [источник: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

### Предзагрузка контекста и режим контекста

Промпт dispatch встраивает план задачи, план среза, предыдущие сводки задач, сводки зависимостей, выдержку из дорожной карты и реестр решений. Профиль токенов управляет объёмом: `budget` встраивает минимальный контекст, `quality` встраивает всё [источник: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

Режим Context Mode включён по умолчанию. Единицы получают указание: использовать `gsd_exec` для шумных сканирований, сборок, тестов и диагностики; `gsd_exec_search` перед повтором предыдущего запуска в песочнице; `gsd_resume` после сжатия контекста/возобновления сессии для чтения `.gsd/last-snapshot.md`. `gsd_exec` пишет ограниченные stdout/stderr и метаданные в `.gsd/exec/`, а агент получает короткую сводку. В режиме worktree этапа `gsd_exec` отклоняет скрипты, нацеленные на исходный корень проекта через `cd ../../..` и похожие шаблоны выхода вверх по дереву каталогов [источник: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

### Git-изоляция

`gsd-pi` поддерживает `git.isolation`:

- `none` по умолчанию — работа идёт прямо в текущей ветке, без milestone branch/worktree. Полезно для рабочих процессов с горячей перезагрузкой, где изоляция файлов ломает инструменты разработки.
- `worktree` — каждый этап работает в `.gsd/worktrees/<MID>/` на ветке `milestone/<MID>`. Требует хотя бы один commit; в репозитории без коммитов временно работает как `none`.
- `branch` — работа идёт в корне проекта на ветке `milestone/<MID>`, полезно для репозиториев с большим количеством submodule.

В worktree/branch-режиме этап объединяется в `main` через squash merge как один чистый коммит. Коммиты внутри этапа последовательны и используют формат conventional commit с GSD metadata trailers вроде `GSD-Task: M001/S01/T01` [источники: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md; Git Strategy, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/git-strategy.md].

### Параллельное исполнение и ограничение одного хоста

Автоматический режим и параллельная координация опираются на SQLite-базу корня проекта в WAL-режиме на локальном диске. Это поддерживает несколько терминалов / рабочих процессов на одной машине, но совместное использование `.gsd/gsd.db*` между машинами или через сетевые файловые системы не поддерживается. Если нужна межхостовая оркестрация, нужен внешний координатор [источник: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

Параллельная оркестрация: независимые этапы могут выполняться одновременно; каждый этап получает собственный рабочий процесс и worktree, а общая среда выполнения SQLite/WAL в корне проекта координирует heartbeat рабочих процессов, аренды этапов, владение dispatch, окна повторов и команды управления [источник: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

Реактивное исполнение задач включено по умолчанию: GSD выводит граф зависимостей из IO-аннотаций в планах задач. Если минимум три готовые задачи можно считать безопасными, не конфликтующие задачи отправляются параллельно через субагентов; зависимые задачи ждут предшественников. Если граф неоднозначен или готовых задач меньше порога, автоматический режим откатывается к обычному последовательному исполнителю. Названа реализация: `reactive-graph.ts`, интеграция в `auto-dispatch.ts` и `auto-prompts.ts` [источник: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

### Восстановление и надзор

`gsd-pi` явно проектируется для длительной автономной работы:

- Восстановление после сбоя: состояние рабочего процесса, состояние dispatch-единиц и метаданные поставленной на паузу сессии сохраняются в SQLite; следующий `/gsd auto` восстанавливает прерванную единицу, читает сохранившийся файл сессии и синтезирует брифинг восстановления из вызовов инструментов на диске [источник: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].
- Автоперезапуск в headless-режиме: `gsd headless auto` перезапускается после сбоев с экспоненциальной задержкой 5s -> 10s -> 30s до верхнего предела, по умолчанию 3 попытки, настраивается через `--max-restarts N`; SIGINT/SIGTERM обходят перезапуск [источник: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].
- Восстановление после ошибок провайдера: ограничения частоты автоматически возобновляются после retry-after или 60s; серверные ошибки автоматически возобновляются после 30s; постоянные ошибки вроде unauthorized, invalid key и billing ставят работу на бессрочную паузу и требуют ручного возобновления [источник: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].
- Монитор давления контекста: при 70% использования контекста GSD отправляет сигнал завершения, чтобы агент успел зафиксировать долговечный вывод до заполнения окна контекста [источник: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].
- Обнаружение застревания: анализ скользящего окна обнаруживает повторяющиеся паттерны dispatch, включая A -> B -> A -> B; делает одну повторную попытку с диагностическим промптом, затем останавливается для вмешательства человека [источник: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].
- Надзор за тайм-аутами: мягкий предел 20 минут, простой 10 минут, жёсткий предел 30 минут по умолчанию; среда выполнения записывает фазу тайм-аута, timeoutAt, lastProgressAt, lastProgressKind, recoveryAttempts, lastRecoveryReason [источник: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

### Принудительная проверка

`gsd-pi` позволяет задавать контролируемые команды проверки:

```yaml
verification_commands:
  - npm run lint
  - npm run test
verification_auto_fix: true
verification_max_retries: 2
```

Команды должны быть прямо запускаемыми проверками. Один shell pipeline с `|` разрешён, но `||`, перенаправления, точки с запятой, обратные кавычки и command substitution отвергаются, потому что проверка выполняется как контролируемый список команд, а не как произвольная shell-программа. Если команды не заданы и план задачи не даёт `verify`, GSD пытается обнаружить проект: сначала scripts в `package.json`, затем Python pytest markers [источник: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

### Управление автоматическим режимом

Команды управления:

```text
/gsd auto
Escape
/gsd stop
/gsd steer
/gsd capture "add rate limiting to API endpoints"
/gsd visualize
/gsd status
```

Escape ставит паузу после текущей единицы с сохранением беседы. `/gsd auto` возобновляет работу из последнего состояния БД. `/gsd steer` позволяет жёстко направлять плановые документы во время исполнения; изменения подхватываются на следующей границе фазы. `/gsd capture` сохраняет мысль в режиме "запусти и забудь", после чего она автоматически сортируется между задачами. Панель через `Ctrl+Alt+G` или `/gsd status` показывает текущий milestone/slice/task, прошедшее время, разбивку стоимости/токенов по единицам, прогнозы стоимости, завершённые/активные единицы, ожидающие captures и статус параллельных workers [источник: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

Удалённое управление через Telegram: `/pause`, `/resume`, `/status`, `/progress`, `/budget`, `/log [n]`; GSD опрашивает входящие команды Telegram примерно каждые 5 секунд во время активных сессий автоматического режима [источник: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

### Конфигурация и внешние интеграции `gsd-pi`

Предпочтения живут в `~/.gsd/PREFERENCES.md` и `.gsd/PREFERENCES.md`; локальные значения проекта либо переопределяют глобальные, либо объединяются с ними по типу поля. Через `/gsd prefs` оператор управляет моделями, тайм-аутами, потолком бюджета, профилем токенов и поэтапным предварительным обследованием. API-ключи инструментов хранятся глобально в `~/.gsd/agent/auth.json` и подставляются во все проекты; переменные окружения имеют приоритет над сохранёнными ключами [источник: `gsd-pi` configuration, https://www.opengsd.net/docs/v2/configuration].

Конфигурации MCP-серверов читаются из `.mcp.json` и `.gsd/mcp.json`. Первый путь подходит для MCP-конфигурации, общей для репозитория, второй — для локальных для машины путей, локальных секретов и личных сервисов. Документация рекомендует проверять серверы через `mcp_servers`, `mcp_discover` и `mcp_call`, а для `stdio` использовать абсолютные пути и `env` внутри config, а не полагаться на интерактивный shell-профиль [источник: `gsd-pi` configuration, https://www.opengsd.net/docs/v2/configuration].

`token_profile` координирует выбор модели, пропуск фаз и сжатие контекста: `budget` пропускает research/reassessment и выбирает более дешёвые модели, `balanced` выполняет стандартный цикл, `quality` предпочитает более качественные модели. Дополнительные переключатели фаз позволяют явно управлять `skip_research`, `skip_reassess`, `skip_slice_research`, `reassess_after_slice` и `require_slice_discussion` [источник: `gsd-pi` configuration, https://www.opengsd.net/docs/v2/configuration].

Для многорепозиторной работы есть `workspace.mode: parent`: `plan-slice` проверяет `targetRepositories` по объявленным ID репозиториев, пути задач ограничиваются объявленными корнями репозиториев, а конфигурация по репозиторию может задавать роль, команды проверки и политику коммитов. Это важная граница: GSD может вести родительское рабочее пространство, но только если репозитории явно объявлены и пути не выходят за их корни [источник: `gsd-pi` configuration, https://www.opengsd.net/docs/v2/configuration].

Конфигурация `gsd-pi` также показывает, что рабочий процесс расширяется через хуки, шаблоны workflow, маршрутизацию skills и контекст среды выполнения. `post_unit_hooks` запускаются после известных типов единиц вроде `execute-task`, `complete-slice`, `reassess-roadmap` и `run-uat`; `pre_dispatch_hooks` могут изменять, пропускать или заменять промпт единицы до отправки. `always_use_skills`, `prefer_skills`, `avoid_skills` и `skill_rules` управляют выбором skills по ситуации. `.gsd/RUNTIME.md` встраивается в промпты исполнения задач и хранит сведения о среде выполнения: API endpoints, service ports, deployment targets, environment-specific configuration. Это важное уточнение к "где живёт контекст": не только в roadmap/decisions/knowledge, но и в отдельном runtime-файле, который предотвращает выдумывание путей, URL и окружений во время исполнения [источник: конфигурация `gsd-pi`, https://www.opengsd.net/docs/v2/configuration].

Уточнение после открытия источников по конфигурации `gsd-core`: `.planning/config.json` теперь раскрывает много переключателей рабочего процесса, которые существенно меняют метод, а не только предпочтения. Схема включает `workflow.nyquist_validation`, `workflow.human_verify_mode`, `workflow.plan_bounce`, `workflow.code_review`, `workflow.security_enforcement`, `parallelization.max_concurrent_agents`, `git.branching_strategy`, `gates.*`, `agent_skills`, `response_language`, `global_learnings` и `review.models.<cli>` для маршрутизации ревью в инструменты вроде Codex/Gemini/OpenCode [GSD Configuration](https://www.opengsd.net/docs/v1/configuration). Для теории это важно: GSD является конфигурируемой процессной машиной; две команды, которые говорят «мы используем GSD», на деле могут запускать разные политики проверки, ревью, параллельности и безопасности.

Проход с открытием источников нашёл конкретный механизм тайм-аутов/надзора, важный для дизайна нашего собственного цикла. Документация команд GSD упоминает checkpoint heartbeats для фоновых запусков `execute-phase`: они испускают маркеры `[checkpoint]` на каждой границе волны и плана, чтобы Claude API SSE stream не простаивал достаточно долго для срабатывания `Stream idle timeout - partial response received` в фазах с несколькими планами [GSD Commands](https://www.opengsd.net/docs/v1/commands). Это полезный паттерн для многопроходного документного процесса: длительная работа должна не только писать файлы, но и производить дешёвые сигналы живучести/прогресса на смысловых границах.

## Артефакты

### `gsd-core` / `.planning/`

Низкоуровневый `artifact-types.md` задаёт важное правило: форматированный файл сам по себе ещё не рабочий артефакт. У каждого типа артефакта должны быть форма, жизненный цикл, расположение и механизм потребления; артефакт, который ни один рабочий процесс не читает, методологически инертен. Поэтому для GSD важно не только перечислять `.planning/*.md`, но и показывать, какая команда или агент потребляет файл на следующем шаге [источник: `artifact-types.md`, https://github.com/open-gsd/get-shit-done-redux/blob/main/get-shit-done/references/artifact-types.md].

Основные артефакты:

- `.planning/PROJECT.md` — описание проекта и требования из начального обсуждения [источник: `your-first-project.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/tutorials/your-first-project.md].
- `.planning/REQUIREMENTS.md` — REQ-IDs для возможностей [источник: `your-first-project.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/tutorials/your-first-project.md].
- `.planning/ROADMAP.md` — фазы, цели, статус, критерии успеха [источник: `your-first-project.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/tutorials/your-first-project.md].
- `.planning/STATE.md` — текущий milestone, прогресс фазы, активные решения, блокеры, метрики прогресса; рабочие процессы ориентируются по нему [источник: `the-phase-loop.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/explanation/the-phase-loop.md].
- `.planning/config.json` — настройки рабочего процесса, профиля моделей, контрольных остановок, git-стратегии, безопасности, хуков, переопределений моделей и переключателей рабочего процесса [источник: документация по конфигурации, https://opengsd.net/docs/v1/configuration].

## Контекст и передача состояния

GSD полезен именно потому, что контекст не живёт только в окне беседы. В `gsd-core` `.planning/` делает результаты шагов читаемыми между сессиями: `CONTEXT.md` сохраняется до запуска planner; `PLAN.md` сохраняется до запуска executor; `VERIFICATION.md` сохраняется до review; `STATE.md` сообщает любому агенту, где сейчас находится проект [источник: `the-phase-loop.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/explanation/the-phase-loop.md].

`artifact-types.md` уточняет эту передачу состояния через потребителей. `ROADMAP.md` читают `plan-phase`, `discuss-phase`, `execute-phase`, `progress` и state-команды; `STATE.md` читают все оркестрационные workflows и resume/progress/next; `CONTEXT.md` сначала потребляет планировщик, затем исполнитель; `SUMMARY.md` после завершения плана читают оркестратор, планировщик будущих планов и сводка этапа; `HANDOFF.json` и `.continue-here.md` потребляет `resume-project`. Для будущей теории это сильная деталь: долговечный контекст работает потому, что у каждого файла есть следующий читатель, а не потому, что он лежит в репозитории [источник: `artifact-types.md`, https://github.com/open-gsd/get-shit-done-redux/blob/main/get-shit-done/references/artifact-types.md].

В `gsd-pi` перенос состояния ещё жёстче: база данных является авторитетной, а markdown-проекции остаются поверхностью ревью/промптов/истории. Это снижает риск, что агент восстановит состояние среды выполнения из устаревшего markdown, но создаёт другое операционное требование: нельзя делить SQLite/WAL runtime через сетевую файловую систему и нельзя считать ручную правку проекции достаточной для изменения состояния [источник: Auto Mode docs, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

Для будущей теории это хороший пример различия между "файл как состояние" и "БД как состояние с markdown-проекцией". В `gsd-core` plain-text-артефакты сами являются рабочей памятью. В `gsd-pi` plain-text-артефакты остаются проверяемыми, но истина среды выполнения находится в БД.

Миграция из v1 `.planning` в `.gsd` показывает, насколько серьёзно `gsd-pi` относится к авторитетности состояния. `/gsd migrate` парсит старые `PROJECT.md`, `ROADMAP.md`, `REQUIREMENTS.md`, каталоги фаз, планы, сводки, исследования, верхнеуровневые `decisions/` и `seeds/`, мапит phases -> slices и plans -> tasks, пишет иерархию в базу данных и только затем рендерит markdown-проекции. Если уже есть `.gsd/`, она бэкапится в `.gsd-backups/migrate-YYYYMMDD-HHMMSS/`; полный legacy-вход архивируется в `.gsd/migration/legacy/` [источник: Migration from v1, https://www.opengsd.net/docs/v2/migration].

Если база данных отсутствует или повреждена, `gsd-pi` не должен молча восстанавливать состояние среды выполнения из markdown. Документация предлагает явную операцию `/gsd recover`: она очищает сохранённую иерархию плюс состояние, связанное с валидацией, и реконструирует иерархию milestone/slice/task из отрендеренного markdown. Это разрушительное восстановление/импорт, а не обычный путь чтения состояния [источник: Migration from v1, https://www.opengsd.net/docs/v2/migration].

В `gsd-core` похожая проблема решается легче, потому что `.planning/` остаётся файловым состоянием, но текущая документация всё равно добавляет команды синхронизации состояния: `/gsd-state-sync`, `/gsd-state-recover`, `/gsd-recover-context`, `/gsd-resolve-execution` и `/gsd-plan-status`. Они нужны, когда состояние планирования, git-история, артефакты фазы или прерванное исполнение перестают согласовываться. Для методологии это важная оговорка: долговечные артефакты помогают пережить потерю контекста, но сами нуждаются в проверке свежести и восстановлении после прерванной работы [источник: `gsd-core` User Guide, https://www.opengsd.net/docs/v1/user-guide].

Проход с открытием источников добавил деталь адаптивного контекста. Во время `execute-phase` планы группируются в волны зависимостей; каждый исполнитель получает свежее окно контекста на 200K или до 1M для моделей, которые это поддерживают. Если окно контекста составляет 500K+ токенов, GSD обогащает промпты субагентов сводками предыдущих волн и контекстом фазы; для стандартных окон на 200K промпты используют сокращённые версии с порядком, удобным для кэша [GSD Core Architecture](https://www.opengsd.net/docs/v1/architecture). Это конкретный пример бюджета контекста как политики, зависящей от среды выполнения, а не фиксированного текстового правила.

## Роли / агенты / участники

В `gsd-core` явные агентские роли появляются в workflow:

- researcher / `gsd-phase-researcher`, `gsd-project-researcher`, `gsd-research-synthesizer`, `gsd-codebase-mapper`, `gsd-ui-researcher`;
- planner / `gsd-planner`, `gsd-roadmapper`, `gsd-pattern-mapper`;
- checker / `gsd-plan-checker`, `gsd-integration-checker`, `gsd-nyquist-auditor`, `gsd-ui-checker`;
- executor / `gsd-executor`, `gsd-debugger`, `gsd-doc-writer`;
- verifier / `gsd-verifier`, `gsd-doc-verifier`, `gsd-ui-auditor`.

Документация по конфигурации показывает сопоставление phase type -> agent и переопределения модели/профиля по отдельным агентам; например `models.execution`, `models.verification`, `model_overrides.gsd-planner`, `runtime: "codex"` [источник: документация по конфигурации, https://opengsd.net/docs/v1/configuration].

Документация по архитектуре даёт более низкоуровневую картину: описания агентов лежат в `agents/*.md` с frontmatter `name`, `description`, `tools`, `color`, а общий список на момент документации — 33 агента. Это не только роли в тексте workflow, но и артефакты установки с разрешёнными инструментами (`Read`, `Write`, `Edit`, `Bash`, `Grep`, `Glob`, `WebSearch` и т.д.) [источник: `gsd-core` architecture, https://www.opengsd.net/docs/v1/architecture].

`agent_skills` добавляет ещё один слой роли: один и тот же `gsd-executor` может получить разные дополнительные проектные навыки в зависимости от имени агента или типа фазы. Это полезно, если команда хочет всегда давать frontend-фазам design-system skill, а backend-фазам API/security skill. Но это также усложняет происхождение поведения: чтобы объяснить, почему агент выбрал конкретный паттерн, уже недостаточно смотреть только на `agents/gsd-executor.md` и `PLAN.md`; нужно проверить `.planning/config.json` и источник skill, который был применён к этому запуску [источник: `gsd-core` документация по конфигурации, https://www.opengsd.net/docs/v1/configuration].

Низкоуровневые контракты агентских ролей уточняют ответственность. `gsd-planner` отвечает за перевод цели roadmap, requirements, research и решений фазы в исполнимый `PLAN.md`, но не должен исполнять код. `gsd-plan-checker` отвечает за блокирующую проверку плана до исполнения и не должен сам чинить реализацию. `gsd-executor` отвечает за атомарное выполнение task plan, проверку критериев готовности, коммит и сводку, но не должен принимать архитектурные решения, если они выходят за рамки плана. Проход 04 сверил, что эти файлы доступны в текущем `open-gsd/gsd-core`, поэтому для будущего текста их лучше цитировать через `gsd-core/main/agents/*`, а `get-shit-done-redux` оставить только как исторический контекст lineage. Это разделение важно: GSD удерживает качество не только формой фазового цикла, но и запретом каждой роли брать чужую власть [источники: `agents/gsd-planner.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-planner.md; `agents/gsd-plan-checker.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-plan-checker.md; `agents/gsd-executor.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-executor.md].

Человек участвует не как постоянный микроменеджер, а как владелец неоднозначности и приёмки: отвечает на вопросы Discuss, утверждает roadmap, проверяет UAT, разрешает `checkpoint:human-verify`, вмешивается при постоянных ошибках провайдера, merge conflicts, блокировках stale worker/invalid worktree и застрявших циклах.

## Человеческие подтверждения и контрольные остановки

`gsd-core` по умолчанию интерактивен. `mode: interactive` подтверждает каждый шаг; `mode: yolo` автоматически одобряет решения. Настройки контрольных остановок включают `confirm_project`, `confirm_phases`, `confirm_roadmap`, `confirm_breakdown`, `confirm_plan`, `execute_next_plan`, `issues_review`, `confirm_transition`. Настройки безопасности включают `always_confirm_destructive` и `always_confirm_external_services` [источник: документация по конфигурации, https://opengsd.net/docs/v1/configuration].

В фазовом цикле человеческий ввод особенно виден здесь:

- Утвердить предложенный roadmap после `/gsd-new-project` [источник: `your-first-project.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/tutorials/your-first-project.md].
- Ответить на вопросы о предпочтениях реализации в `/gsd-discuss-phase`; эти ответы становятся `CONTEXT.md` [источник: `your-first-project.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/tutorials/your-first-project.md].
- Запустить `/gsd-verify-work` и ответить на наблюдаемые критерии успеха; сбой создаёт планы исправления [источник: `your-first-project.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/tutorials/your-first-project.md].
- Разрешить `checkpoint:human-verify` перед установкой подозрительного или предполагаемого пакета [источник: `COMMANDS.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/COMMANDS.md].
- Разрешить `CHECKPOINT-NYQUIST.md`, если предварительная проверка нашла проблему, которую нельзя безопасно исправить автоматически. Это человеческая точка решения ещё до исполнения, а не после того, как код уже изменён [источник: `gsd-core` User Guide, https://www.opengsd.net/docs/v1/user-guide].
- Разрешить результаты Revision Gate до исполнения: если `gsd-plan-checker` возвращает `NEEDS_REVISION`, план должен быть исправлен и проверен заново, а не передан исполнителю как "достаточно хороший" [источник: `agents/gsd-plan-checker.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-plan-checker.md].
- Принять решение по архитектурным отклонениям, поднятым исполнителем: новая схема данных, новый сервисный слой, смена библиотеки, подход к auth, инфраструктура или breaking API требуют checkpoint, потому что исполнитель не должен сам расширять архитектурную область [источник: `agents/gsd-executor.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-executor.md].
- Checkpoint работает как структурированная граница продолжения: исполнитель выполняет работу до checkpoint, пишет, что уже сделано, что осталось проверить человеком и с какой точки должен стартовать следующий свежий агент. Это не просто просьба "посмотреть"; это механизм сохранения состояния между остановкой и возобновлением [источник: `agents/gsd-executor.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-executor.md].
- Оркестрация от issue добавляет ещё одну человеческую границу: GSD может импортировать GitHub issues в `.planning/issues/`, классифицировать их как `feature`, `bug`, `enhancement`, `documentation`, `technical-debt`, `research`, `question` или `invalid`, а затем человек подтверждает, что входит в этап/область текущей фазы. Документация подчёркивает ревью с участием человека для планов issue, потому что issue tracker часто содержит шум, дубликаты и недоописанные пожелания [источник: `gsd-core` issue-driven orchestration, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/issue-driven-orchestration.md].

Справочник `checkpoints.md` добавляет экономическую оговорку: человеческая проверка имеет холодный старт, потому что человеку нужно восстановить контекст, прочитать дифф и проверить работу. Поэтому GSD по умолчанию предлагает `human_verify_mode: "end-of-phase"`: несколько волн выполнения можно делать подряд, а затем запускать одну консолидированную человеческую проверку. Это снижает стоимость переключения внимания, но повышает риск позднего обнаружения ошибки; режимы `after-each-wave` и `after-each-task` есть как более строгие альтернативы для рискованной работы [источник: `checkpoints.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/get-shit-done/references/checkpoints.md].

Тот же справочник различает human verification, human decision и human action. Решения по архитектуре, легитимности пакетов и области не объединяются ради экономии: они должны немедленно останавливать работу, потому что продолжение без ответа меняет смысл плана. Human action checkpoint нужен для того, что агент не может сделать сам: OAuth/browser login, ручная настройка DNS, настройка платёжного провайдера, ротация credentials, App Store approval. В оговорке для Claude Code источник допускает автоматическое выполнение настройки CLI/API, если ключи уже настроены; человеческое действие требуется только там, где нет API или нужна интерактивная auth [источник: `checkpoints.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/get-shit-done/references/checkpoints.md].

В `gsd-pi` человеческие контрольные остановки выражены как средства управления автономным исполнением:

- Escape ставит активный автоматический режим на паузу, сохраняя беседу [источник: документация Auto Mode, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].
- `/gsd steer` изменяет плановые документы во время исполнения; изменения подхватываются на следующей границе фазы [источник: документация Auto Mode, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].
- `require_slice_discussion: true` ставит паузу перед каждым срезом и показывает контекст среза для ревью; после подтверждения исполнение продолжается [источник: документация Auto Mode, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].
- Постоянные ошибки провайдера ставят процесс на бессрочную паузу и требуют ручного возобновления [источник: документация Auto Mode, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].
- Git-конфликты и грязные пересечения требуют ручной очистки перед повторным запуском [источник: документация Auto Mode, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].
- `/gsd verdict <pass|needs-attention|needs-remediation>` может вручную переопределить записанный вердикт проверки этапа с явным обоснованием; это человеческая точка ответственности после автоматической оценки [источник: команды `gsd-pi`, https://www.opengsd.net/docs/v2/commands].
- `/gsd doctor` и `/gsd doctor fix` становятся первым шагом, когда GSD застрял, шумит или ведёт себя неожиданно; doctor проверяет `.gsd/`, ссылки roadmap/slice/task, состояние среды выполнения и здоровье Git worktree [источник: Troubleshooting, https://www.opengsd.net/docs/v2/troubleshooting].

## Проверка результата и качество

GSD проверяет качество на нескольких уровнях:

- Проверка плана до исполнения: plan-checker должен поймать неполные, несогласованные или неоднозначные планы до запуска параллельных исполнителей [источник: `the-phase-loop.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/explanation/the-phase-loop.md].
- предварительная проверка Nyquist: перед исполнением проверяются ссылаемые файлы, критерии приёмки и граф зависимостей; результатом становятся `PLAN-REPAIRS.md`, `CHECKPOINT-NYQUIST.md` или `NYQUIST-PASS.md` [источник: `gsd-core` User Guide, https://www.opengsd.net/docs/v1/user-guide].
- Свежесть плана: `/gsd-plan-status` проверяет `source_sha`, хэши зависимостей и связь артефактов плана с текущим `HEAD`; устаревший или dirty plan не должен восприниматься как готовый к исполнению только потому, что файл существует [источник: `gsd-core` User Guide, https://www.opengsd.net/docs/v1/user-guide].
- Revision Gate: `gsd-plan-checker` возвращает `PASS`, `NEEDS_REVISION` или `BLOCKED_BY_MISSING_INPUT`; при revision он перечисляет конкретные исправления плана, а не разрешает исполнение [источник: `agents/gsd-plan-checker.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-plan-checker.md].
- Покрытие требований: `VERIFY` / verifier смотрит, покрыты ли REQ-IDs [источник: `the-phase-loop.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/explanation/the-phase-loop.md].
- Покрытие решений: решения из `CONTEXT.md` должны попасть в планы и поставленный код; документация по конфигурации описывает контрольные остановки покрытия решений, где plan-phase translation gate блокирующий, а verify-phase validation gate неблокирующий [источник: документация по конфигурации, https://opengsd.net/docs/v1/configuration].
- Аудит покрытия нескольких источников: `gsd-planner` должен явно покрыть `GOAL`, `REQ`, `RESEARCH` и решения `CONTEXT`; missing items становятся warning/action до исполнения, а не тихой потерей области [источник: `agents/gsd-planner.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-planner.md].
- Безопасность волн до исполнения: plan-checker должен увидеть порядок зависимостей и отсутствие пересечений для параллельных задач до запуска волны исполнителей. Если план не показывает, какие файлы и области ответственности разделены, параллельность становится скрытым источником конфликтов, а не ускорением [источник: `agents/gsd-plan-checker.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-plan-checker.md].
- Самопроверка исполнителя: `gsd-executor` проверяет критерии готовности, запускает запланированные команды проверки, коммитит только завершённую задачу и записывает хеш коммита для сводки; при checkpoint продолжение должно стартовать в свежем контексте и сверить предыдущие коммиты [источник: `agents/gsd-executor.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-executor.md].
- Лестница проверки из `verification-patterns.md`: GSD различает четыре уровня проверки результата — файл существует в ожидаемом месте, содержимое является реальной реализацией, реализация подключена к остальной системе, поведение фактически работает при вызове. Первые три уровня часто можно проверить программно, а четвёртый нередко требует человеческой проверки или свидетельств из браузера/среды выполнения. Это полезно для теории, потому что "артефакт существует" не должен считаться равным "фича готова" [источник: `verification-patterns.md`, https://github.com/open-gsd/get-shit-done-redux/blob/main/get-shit-done/references/verification-patterns.md].
- Независимое ревью кода: `/gsd-code-review-*` смотрит на корректность, сопровождаемость, безопасность, тесты и согласованность с планом до поставки; это отдельный слой от UAT и критериев приёмки [источник: `gsd-core` User Guide, https://www.opengsd.net/docs/v1/user-guide].
- UAT: `/gsd-verify-work` проходит по критериям успеха и записывает `UAT.md` [источник: `your-first-project.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/tutorials/your-first-project.md].
- Браузерные свидетельства: `gsd-browser` может давать снимки экрана, визуальные сравнения, записи, HAR, трассы и assertions как доказательство в цикле проверки [источники: roadmap, https://opengsd.net/roadmap; `COMMANDS.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/COMMANDS.md].
- Контролируемые команды проверки в `gsd-pi`: lint/test-команды, повторы auto-fix, ограничения синтаксиса команд [источник: документация Auto Mode, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].
- Проверка существования артефакта и вердикта в `gsd-pi`: отсутствующий артефакт вызывает повтор с явным контекстом сбоя; после лимита следует пауза [источник: документация Auto Mode, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].
- Видимость стоимости и токенов: панель и отчёты показывают разбивку стоимости/токенов по единицам, прогнозы и связанное с бюджетом состояние [источник: документация Auto Mode, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].
- Обнаружение дрейфа кодовой базы в `gsd-core`: после execute-phase неблокирующая проверка дрейфа сравнивает новые структурные элементы с документами карты кодовой базы и предлагает `/gsd-map-codebase --paths ...` либо делает auto-remap, если это настроено [источник: `gsd-core` architecture, https://www.opengsd.net/docs/v1/architecture].
- Проверка здоровья среды выполнения в `gsd-pi`: `/gsd doctor` проверяет целостность `.gsd/`, ссылки roadmap/slice/task, состояние среды выполнения и здоровье Git worktree перед возобновлением `/gsd auto` [источник: Troubleshooting, https://www.opengsd.net/docs/v2/troubleshooting].
- Наблюдаемость dispatch в `gsd-core`: событие ошибки в stderr даёт `traceId` даже без полного журнала аудита, а `.planning/.gsd-trace.jsonl` позволяет разбирать успешные и ошибочные dispatches после включения `GSD_AUDIT=1`. Это полезно не для критериев приёмки, а для диагностики самого GSD-процесса: какой handler отказал, какие дочерние dispatches относятся к top-level invocation и где потерялась маршрутизация [источник: документация по конфигурации, https://www.opengsd.net/docs/v1/configuration].

`gates.md` полезен как карта блокировок, потому что в высокоуровневой документации разные остановки легко смешать. Pre-flight gate срабатывает до дорогой работы: например, легитимность пакета или missing API key должны остановить планирование/исполнение до того, как агент потратит много контекста. Revision gate проверяет артефакт после создания: `PLAN.md` может получить `NEEDS_REVISION`, но это ещё не ошибка среды выполнения. Escalation gate нужен во время исполнения, когда появилась неразрешимая неоднозначность: package not found, архитектурное отклонение, ambiguous user intent. Abort gate используется для невосстановимого состояния: corrupted repository, impossible requirements или security breach. Для будущей теории важно, что GSD не просто "спрашивает человека", а выбирает разные типы остановки в зависимости от момента и цены продолжения [источник: `gates.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/get-shit-done/references/gates.md].

`verification-patterns.md` даёт практическую сторону этой проверки: искать universal stub patterns (`TODO`, `FIXME`, `PLACEHOLDER`, `coming soon`, `lorem ipsum`, пустые обработчики, `return null`, hardcoded counts/IDs where dynamic expected), затем проверять подключение и поведение в среде выполнения. Для React/Next.js источник отдельно показывает, что экспорт компонента и JSX недостаточны: нужно увидеть использование props/state, imports, data fetching или handlers, а интерактивные элементы должны реально отвечать на действия. Эти детали лучше уйдут в Handbook, но в теории они нужны как пример того, что GSD проверяет реализацию против наблюдаемого поведения, а не против наличия файлов [источник: `verification-patterns.md`, https://github.com/open-gsd/get-shit-done-redux/blob/main/get-shit-done/references/verification-patterns.md].

`gsd-browser` в проверке качества добавляет не только screenshots, но и проверяемую цепочку состояние браузера -> действие -> утверждение -> свидетельство. Документация CLI/MCP показывает команды и инструменты для snapshot refs, `assert`, `screenshot`, `recording start/stop`, `trace`, `har`, `network`, `debug-bundle`, `timeline`, `session export`, визуальных сравнений и live human takeover. Для GSD это значит, что browser UAT можно превратить из "посмотри глазами" в воспроизводимый артефакт свидетельств, но хранение и публикация этих файлов требуют отдельного правила [источник: `gsd-browser` CLI & MCP Commands, https://opengsd.net/docs/v5/commands].

Уточнение после открытия источников по валидации: User Guide описывает Nyquist validation как предварительное сопоставление автоматизированного покрытия тестами к каждому требованию фазы. Researcher обнаруживает тестовую инфраструктуру, связывает requirements с test commands и создаёт задачи волны 0 для нужного тестового каркаса; plan-checker применяет это как восьмое измерение проверки, отклоняя планы без automated verify commands [GSD User Guide](https://www.opengsd.net/docs/v1/user-guide). Есть также `/gsd-validate-phase` для ретроактивной валидации: она обнаруживает, существуют ли `VALIDATION.md` и `SUMMARY.md`, находит реализацию, связывает requirements с tests, показывает gap plan, запускает auditor и обновляет `VALIDATION.md` [GSD User Guide](https://www.opengsd.net/docs/v1/user-guide).

Документация по архитектуре добавляет codebase drift gate после исполнения. Он сравнивает изменения с момента `last_mapped_commit` с `.planning/codebase/STRUCTURE.md`, считает структурные изменения вроде новых каталогов, barrel exports, migrations и route modules, затем предупреждает или выполняет auto-remap при достижении порога [GSD Core Architecture](https://www.opengsd.net/docs/v1/architecture). Это важно для будущей теории: проверка — это не только «tests pass», но и «карта кодовой базы не стала незаметно устаревшей».

## Хвост жизненного цикла

После реализации GSD не останавливается на commit:

- `gsd-core` запускает `/gsd-verify-work`, затем `/gsd-ship`, создаёт PR и обновляет `STATE.md` [источники: `COMMANDS.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/COMMANDS.md; `your-first-project.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/tutorials/your-first-project.md].
- `/gsd-complete-milestone` архивирует milestone и помечает релиз тегом; `/gsd-new-milestone` начинает следующий цикл версии [источник: `COMMANDS.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/COMMANDS.md].
- Issue-driven orchestration может закрывать GitHub issue после `/gsd-ship`, создавать план из входящего issue и связывать phase work с метаданными issue, но документация оставляет ревью/подтверждение человеком перед включением issue в roadmap/phase [источник: `gsd-core` issue-driven orchestration, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/issue-driven-orchestration.md].
- Закрытие milestone в `gsd-pi` требует последней оценки `milestone-validation` с вердиктом `pass`; fail/partial/absent блокируют закрытие [источник: документация Auto Mode, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].
- `gsd-pi` генерирует HTML-отчёты в `.gsd/reports/` после завершения milestone и может генерировать отчёты вручную через `/gsd report` [источник: документация Auto Mode, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].
- Git strategy в `gsd-pi` может слить milestone branch/worktree в `main` через squash merge как чистый commit и затем удалить worktree/branch [источник: Git Strategy, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/git-strategy.md].
- `git.auto_pr` в `gsd-pi` может отправить `main` и milestone branch, затем создать PR через аутентифицированный `gh pr create`; сбой создания PR не фатален и записывается в журнал [источник: конфигурация `gsd-pi`, https://www.opengsd.net/docs/v2/configuration].
- GitHub sync может сопоставлять этапы, slices и tasks с GitHub Issues, PRs и Milestones, сохранять соответствие в `.gsd/.github-sync.json` и пропускать синхронизацию, когда GitHub API rate limit низкий [источник: конфигурация `gsd-pi`, https://www.opengsd.net/docs/v2/configuration].
- `/gsd report` генерирует HTML-отчёты для всех этапов и открывает индекс отчётов; `/gsd export` остаётся alias. Для внешних систем `gsd headless query` даёт JSON-снимок без LLM session и RPC child, поэтому после реализации есть две разные поверхности чтения результата: человек смотрит HTML/report/dashboard, а автоматизация читает структурированное состояние [источник: команды `gsd-pi`, https://www.opengsd.net/docs/v2/commands].

## Сильные стороны

- GSD даёт понятный операционный цикл для инженерии с ИИ: не "поговорили с моделью", а ограниченные фазы / срезы с долговечными артефактами.
- Гигиена контекста встроена в саму форму работы: тяжёлые исследование, планирование, выполнение и проверка уходят в свежие контексты, а основная сессия остаётся тонкой.
- Решения фиксируются до планирования, поэтому planner не обязан угадывать проектные предпочтения.
- Проверка отделена от выполнения и может создавать планы исправлений вместо принятия "код скомпилировался" как критерий готовности.
- Git-история становится частью метода: атомарные коммиты на план в `gsd-core`, последовательные коммиты задач / squash merge этапа в `gsd-pi`.
- `gsd-pi` добавляет защитные механизмы уровня среды выполнения: политику инструментов, состояние с опорой на БД, восстановление после сбоя, восстановление после ошибок провайдера, обнаружение застревания, надзор за тайм-аутами, повторы проверки артефактов.
- `gsd-core` теперь имеет несколько предварительных защит перед исполнением: валидацию Nyquist, проверки свежести плана, plan checker и слой ревью кода. Это снижает вероятность, что устаревший или недоопределённый план попадёт сразу к исполнителю.
- Backlog, seeds, threads и workstreams дают место для контекста разной зрелости: идея может подождать, исследование может жить отдельно, параллельная линия может иметь собственный checkpoint.
- Таксономия артефактов связывает файлы с потребителями: `CONTEXT.md` нужен планировщику/исполнителю, `SUMMARY.md` нужен последующим планам, `HANDOFF.json` нужен workflow возобновления. Это снижает риск накопления "документов для вида", если команда действительно соблюдает механизм потребления [источник: `artifact-types.md`, https://github.com/open-gsd/get-shit-done-redux/blob/main/get-shit-done/references/artifact-types.md].
- Профили моделей дают экономический слой метода: планирование и отладку можно отправлять на более сильные модели, массовое картирование/проверку — на дешёвые, а среды выполнения не от Anthropic удерживать через `inherit`/`omit`, чтобы workflow не создавал неожиданные расходы на модели [источник: `model-profiles.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/get-shit-done/references/model-profiles.md].
- Существуют более лёгкие примитивы (`/gsd-quick`, `/gsd-fast`), что важно: методология признаёт свои накладные расходы и даёт альтернативу для малых задач.
- Экосистема Open GSD строится с видимой дорожной картой, страницами promise/origin и MIT-licensed репозиториями; это важно после сложной истории доверия вокруг прежнего upstream [источники: Promise, https://opengsd.net/promise; Origin Credit, https://opengsd.net/origin; roadmap, https://opengsd.net/roadmap].

## Сбойные режимы / риски чрезмерного применения

### Накладные расходы и церемониальность

Документация честно фиксирует компромиссы: `/gsd-discuss-phase`, `/gsd-plan-phase` и `/gsd-execute-phase` как отдельные шаги занимают больше времени, чем один обычный промпт; запуск субагентов медленнее одной правки в текущем контексте; для опечатки, переименования или недостающего импорта фазовый цикл избыточен [источник: `context-engineering.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/explanation/context-engineering.md].

Низкоуровневые контракты агентских ролей усиливают этот риск: даже фаза планирования включает извлечение зафиксированных решений, аудит нескольких источников, бюджетирование контекста, optional TDD/MVP shaping и plan-checker Revision Gate. Для маленькой правки эта дисциплина будет дороже результата, поэтому GSD нельзя описывать как универсальный стиль для любого изменения [источники: `agents/gsd-planner.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-planner.md; `agents/gsd-plan-checker.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-plan-checker.md].

### Неверный размер фазы

Слишком большая фаза превращается в исследовательский проект: планировщик с трудом декомпозирует работу, исполнители блокируются, проверка становится полным аудитом. Слишком маленькая фаза создаёт plan-файлы на несколько строк, а накладные расходы превышают пользу. Хорошая фаза имеет цель в одно предложение, ограниченное исследование, несколько непересекающихся планов и ясное проверяемое определение готовности [источник: `the-phase-loop.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/explanation/the-phase-loop.md].

Plan Drift Guard показывает ещё один вариант сбоя: даже правильно выбранная фаза может породить план, который устареет до исполнения. Если между планированием и исполнением изменился код, decisions, requirements или research, stale plan превращается в скрытую инструкцию к старому состоянию репозитория. Поэтому `/gsd-plan-status` и пути refresh/replan нужно описывать как обязательную проверку для отложенных или возобновлённых работ, а не как удобную диагностику [источник: `gsd-core` User Guide, https://www.opengsd.net/docs/v1/user-guide].

### Иллюзия автономности

`/gsd auto` звучит как «запусти и уйди», но документация показывает много условий, где нужна человеческая интервенция: постоянные ошибки провайдера, merge conflicts, dirty overlap, invalid worktree, отсутствующие closeout-инструменты, исчерпание повторов проверки артефакта, застрявшие циклы. Это не слабость, а граница применимости: GSD автоматизирует цикл, но не отменяет ответственность оператора [источник: документация Auto Mode, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

В `gsd-core` похожая граница видна внутри исполнителя: он может автоматически исправлять прямые ошибки текущей задачи, но architectural changes, package substitution и неясные blockers должны превращаться в checkpoint. Если эту границу убрать, GSD легко деградирует в "агент сам решил всё по дороге", а долговечные артефакты начнут фиксировать уже не человеческие решения, а самовольные отклонения исполнителя [источник: `agents/gsd-executor.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-executor.md].

`checkpoints.md` показывает обратный риск: если включить проверку после каждой задачи, человек начнёт постоянно восстанавливать контекст и проверять малые диффы, а стоимость процесса вырастет быстрее, чем качество. Поэтому режим verification checkpoint должен зависеть от риска: end-of-phase для обычной работы, after-each-wave для рискованных волн, after-each-task только для особо чувствительных изменений. Человеческие decision/action checkpoints при этом нельзя откладывать тем же способом, потому что они блокируют смысловое продолжение [источник: `checkpoints.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/get-shit-done/references/checkpoints.md].

### Путаница DB/projection в `gsd-pi`

Ручная правка `.gsd/STATE.md`, `.gsd/DECISIONS.md` или `.gsd/KNOWLEDGE.md` может выглядеть как изменение состояния, но источник истины среды выполнения — SQLite database. Редактирование проекций не переопределяет базу данных, если они не импортированы или не сохранены через GSD. Это потенциальный источник путаницы для пользователей, привыкших к `gsd-core`, где markdown-артефакты сами являются рабочим состоянием [источник: документация Auto Mode, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

При повреждённой базе данных recovery намеренно сделан явным: `/gsd recover` стирает persisted hierarchy and validation-related state before rebuilding from rendered markdown. Это защищает от тихого дрейфа состояния, но создаёт риск для оператора, который воспринимает recover как безобидную "починку" [источник: Migration from v1, https://www.opengsd.net/docs/v2/migration].

### Ограничение одного хоста

Параллельная координация `gsd-pi` зависит от локального диска SQLite/WAL. Попытка растянуть `.gsd/gsd.db*` через сетевую файловую систему или несколько машин не поддерживается. Для распределённой оркестрации нужен внешний координатор [источник: документация Auto Mode, https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md].

### Цепочка поставки и доверие

Open GSD явно возник как поддерживаемое продолжение после опасений о доверии/владении вокруг прежнего upstream. `Origin Credit` признаёт сложную главу; `Promise` делает transparency/community/accountability частью нового проекта. Для методологического текста это нужно подавать аккуратно: не как сплетню, а как операционный урок о доверии к агентским инструментам, поверхностям установщика, package scopes и пути миграции [источники: Promise, https://opengsd.net/promise; Origin Credit, https://opengsd.net/origin; GitHub `open-gsd/gsd-core`, https://github.com/open-gsd/gsd-core].

### Особенности отдельных сред выполнения

`gsd-core` поддерживает несколько сред выполнения, но синтаксис команд различается: Claude Code / Copilot / OpenCode / Kilo используют `/gsd-command-name`, Gemini использует `/gsd:command-name`, Codex использует `$gsd-command-name`. Установщик записывает правильную форму. Прямое копирование файлов из `agents/` или `commands/` в среду выполнения не от Claude пропускает конвертацию и может порождать ошибки schema validation [источники: `COMMANDS.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/COMMANDS.md; GitHub `open-gsd/gsd-core`, https://github.com/open-gsd/gsd-core].

Документация команд уточняет для Codex отдельную операционную деталь: после установки команды запускаются как `$gsd-start`, `$gsd-progress --next`, `$gsd-plan-phase 1`, `$gsd-execute-phase 1` и т.п.; slash form здесь не является переносимой нормой. Это важно для Handbook: если будущий текст будет давать исполняемые примеры для Codex, он должен использовать `$gsd-*`, а не копировать форму `/gsd-*` в стиле Claude [источник: `gsd-core` документация по командам, https://www.opengsd.net/docs/v1/commands].

`model-profiles.md` добавляет ещё один риск, зависящий от среды выполнения: профиль по умолчанию может не соответствовать выбранному стеку провайдера. Для Claude Code поверх OpenRouter/локальным провайдером нужен `inherit`, иначе GSD может пытаться вызывать алиасы Anthropic для субагентов; для Codex/OpenCode/Gemini CLI установщик должен выставить `resolve_model_ids: "omit"`, чтобы среда выполнения сама выбирала модель. Если команда переносит `.planning/config.json` между средами без проверки этих настроек, методология может неожиданно изменить стоимость и качество исполнения [источник: `model-profiles.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/get-shit-done/references/model-profiles.md].

Troubleshooting `gsd-pi` отдельно фиксирует трение путей/пакетов/среды выполнения: старый пакет `gsd-pi` без scope может конфликтовать со scoped `@opengsd/gsd-pi`, npm global bin может отсутствовать в `PATH`, pnpm может ругаться на global bin directory, автоматический режим может писать артефакты не в тот каталог при старой версии, Windows file locks (`EPERM`, `EBUSY`, `EACCES`) могут мешать обновлять файлы `.gsd/` или worktrees [источник: Troubleshooting, https://www.opengsd.net/docs/v2/troubleshooting].

В `gsd-core` похожий дрейф среды выполнения/конфигурации возникает из-за слишком мощной `.planning/config.json`: `fast_mode`, `effort`, `model_profile`, переопределения моделей по агентам, `agent_skills`, `global_learnings`, `search_gitignored`, `parallelization.*` и `safety.*` могут менять поведение процесса без видимого изменения команды. Это удобство для зрелой команды и риск для переноса методологии: два проекта оба "запускают `/gsd-plan-phase`", но фактически используют разные модели, skills, gates, границы поиска и политику параллельности [источник: `gsd-core` документация по конфигурации, https://www.opengsd.net/docs/v1/configuration].

Осторожность прохода 03 по lineage частично закрыта: низкоуровневые agent files, использованные для planner / plan-checker / executor, доступны в текущем `open-gsd/gsd-core`. Риск остаётся не в том, что источник "не тот", а в том, что такие контракты могут меняться быстрее, чем high-level docs. Перед публикацией стоит ещё раз сверить raw-файлы и релизную версию, но досье уже может считать их первичными источниками текущего `gsd-core`, а не только историческим материалом [источники: `agents/gsd-planner.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-planner.md; `agents/gsd-plan-checker.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-plan-checker.md; `agents/gsd-executor.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/agents/gsd-executor.md; GitHub `open-gsd/gsd-core`, https://github.com/open-gsd/gsd-core].

### Инертные артефакты

`artifact-types.md` показывает риск, который легко пропустить при переносе GSD в другую команду: если файл не читает ни один workflow, он не участвует в процессе. Команда может создать красивые `METHODOLOGY.md`, `DISCUSSION-LOG.md`, handoff-файлы или custom notes, но без явной точки потребления они становятся архивом рядом с работой. Для будущего текста это хорошая граница: долговечные артефакты усиливают SDLC только тогда, когда следующий шаг действительно использует их как вход [источник: `artifact-types.md`, https://github.com/open-gsd/get-shit-done-redux/blob/main/get-shit-done/references/artifact-types.md].

### Путаница аудита и приватности

Файл аудита `gsd-core` выглядит как удобное доказательство хода работы, но он специально выключен по умолчанию и добавлен в `.gitignore`. При включении `GSD_AUDIT_ARGS=1` аргументы команд попадают в события дословно; при хранении API-ключей `.planning/config.json` является границей безопасности, потому что plaintext keys лежат именно там, хотя UI и журналы их маскируют. Если команда будет коммитить `.planning/` без явного правила исключений, методология может создать новый риск утечки рядом с тем местом, где она пытается усилить прослеживаемость [источник: документация по конфигурации, https://www.opengsd.net/docs/v1/configuration].

`security_hardening` снижает часть риска через атомарные записи, резервные файлы, проверку `.planning/` в `.gitignore`, dangerous git policy и tool gate разрешений, но не отменяет операторской ответственности. Если команда ослабит `safety.dangerous_git_policy`, добавит широкие `tools.allowed` или будет хранить provider keys в видимом для репозитория `.planning/config.json`, GSD не сможет методологически компенсировать плохую гигиену безопасности [источник: документация по конфигурации, https://www.opengsd.net/docs/v1/configuration].

`gsd-browser` добавляет похожую проблему приватности со стороны свидетельств. Screenshots, HAR, recordings, PDF exports и trace/debug bundles могут содержать пользовательские данные, cookies, tokens, internal URLs или коммерческую информацию. Product/docs позиционируют эти артефакты как доказательство browser behaviour, но досье должно держать границу: свидетельства полезны для проверки, однако не каждый артефакт свидетельств безопасно коммитить или прикладывать к публичному issue/PR [источники: страница продукта `gsd-browser`, https://opengsd.net/products/gsd-browser; `gsd-browser` CLI & MCP Commands, https://opengsd.net/docs/v5/commands].

### Площадь поверхности и сложность маршрутизации

Шесть маршрутизаторов пространств имён помогают скрыть от модели плоский список из десятков навыков, но для человека это создаёт новый слой, который нужно понимать. Ошибка маршрутизации может привести к тому, что агент выберет quick/debug/management path вместо полного фазового цикла или наоборот. `docs/INVENTORY.md` полезен как диагностический источник: он показывает реальную поверхность команд, агентов и рабочие процессы, но сам факт такого inventory означает, что Open GSD уже достаточно велик, чтобы требовать сопровождения собственной карты [источник: `gsd-core` inventory, https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/INVENTORY.md].

Команды вроде `/gsd-ingest-docs`, `/gsd-graphify`, `/gsd-surface`, `/gsd-manager` и `/gsd-docs-*` усиливают метод, но расширяют входные каналы. Теперь ошибочный документ может попасть в состояние планирования через ingest, граф зависимостей может устареть относительно кодовой базы, фоновая задача может жить рядом с фазой без явного пункта дорожной карты, а API-карта поверхностей может быть прочитана как полная истина, хотя endpoint discovery всегда зависит от соглашений фреймворка. Это хорошие инструменты для зрелого процесса, но для теории нужно держать формулировку узкой: GSD работает, когда дополнительные поверхности имеют явных потребителей и проверки свежести [источник: `gsd-core` документация по командам, https://www.opengsd.net/docs/v1/commands].

Есть ещё один скрытый источник цены поверхности: MCP-схемы. `context-budget.md` прямо говорит, что включённые MCP-серверы увеличивают базовую токенную нагрузку каждого хода и не управляются обычной GSD-конфигурацией. Если команда подключила много серверов "на всякий случай", GSD может быстрее входить в degraded context tiers даже при аккуратной `.planning/` дисциплине [источник: `context-budget.md`, https://raw.githubusercontent.com/open-gsd/gsd-core/main/get-shit-done/references/context-budget.md].

Проход с открытием источников уточняет риск конфигурации. Поскольку GSD раскрывает review models, parallelization, gates, git strategy, security enforcement, Nyquist validation и поведение context/window как configuration, метод мощный, но его трудно сравнивать извне. Case study должен фиксировать настройки уровня `config.json`, иначе утверждение «GSD succeeded/failed» недоопределено. Это особенно верно для `parallelization.max_concurrent_agents`, `workflow.nyquist_validation`, `workflow.code_review`, `workflow.security_enforcement` и маршрутизации review model [GSD Configuration](https://www.opengsd.net/docs/v1/configuration).

Паттерн checkpoint heartbeat также даёт урок: долгоживущая оркестрация может падать по причинам транспорта/живучести даже при здоровом underlying plan. Устойчивой agentic SDLC нужны смысловые сигналы прогресса, возобновляемое состояние и артефакты с файловой опорой; иначе тайм-аут выглядит как методологический сбой, хотя на деле это сбой надзора [GSD Commands](https://www.opengsd.net/docs/v1/commands).

## Аналитические заметки предыдущих проходов

### Что должно попасть в теорию

- GSD как пример операционного цикла для ИИ-ориентированного SDLC: Discuss -> Plan -> Execute -> Verify -> Ship, где каждый шаг предотвращает конкретный класс сбоя.
- Context rot как практический драйвер: длинная сессия деградирует без явной стратегии сброса и состояния.
- Субагенты со свежим контекстом + долговечные артефакты как базовый паттерн: контекст укорачивается, но непрерывность сохраняется через файлы/БД.
- Различие `gsd-core` и `gsd-pi`: фреймворк внутри существующей среды выполнения vs самостоятельный локально-ориентированный агент/оркестратор.
- Человеческие точки решения: решения Discuss до планирования, утверждение roadmap/plan, UAT, checkpoints легитимности пакетов, slice discussion gate, вмешательство после блокеров автоматического режима.
- Verification как фаза, производящая артефакты, а не запоздалая мысль: `VERIFICATION.md`, `UAT.md`, браузерные свидетельства, контролируемые lint/test-команды, повторы проверки артефактов.
- Наблюдаемость как отдельный слой качества: `traceId`, `.planning/.gsd-trace.jsonl`, `/gsd doctor`, `/gsd inspect`, `/gsd forensics`, `/gsd report`, `gsd headless query` показывают, что зрелая agentic SDLC методология должна проверять не только код, но и собственное состояние оркестрации.
- Планирование как исполняемая передача: `PLAN.md` должен переносить `GOAL`, `REQ`, `RESEARCH` и решения `CONTEXT` в конкретные пути файлов, действия, команды проверки, критерии готовности, `must_haves` и ключевые ссылки.
- Ревью плана как blocking gate: `gsd-plan-checker` возвращает `NEEDS_REVISION`, если план теряет требования, решения, error paths, порядок зависимостей, verification или бюджет контекста.
- Контракты агентских ролей как источник методологии: текущие `open-gsd/gsd-core/main/agents/*` показывают не только роли, но и границы власти каждого агента, поэтому их можно использовать как материал для теории о границах делегирования.
- Границы полномочий агентов: planner не исполняет, checker не чинит реализацию, executor не меняет архитектуру без checkpoint.
- Модели состояния: `.planning/` как plain-text-состояние в `gsd-core`; авторитетное SQLite-состояние + `.gsd/`-проекции в `gsd-pi`.
- Git как часть метода: атомарные коммиты, изоляция worktree/branch, squash merge, тело PR, сгенерированное из артефактов планирования.
- Граница чрезмерного применения: фазовый цикл полезен, когда задача затрагивает файлы, сессии, исследование или решения; для крошечных изменений предпочтительны quick/fast-примитивы.
- Здоровье/восстановление как полноценный жизненный цикл: `/gsd doctor`, `/gsd forensics`, `/gsd recover`, `/gsd closeout`, headless query/recover и явная миграция показывают, что автономная работа требует отдельной механики диагностики, а не только prompt-дисциплины.
- Граница нескольких репозиториев и внешней интеграции: `workspace.mode: parent`, MCP-конфигурации, GitHub sync и `git.auto_pr` расширяют GSD за пределы одного репозитория, но требуют явного объявления путей, ключей, серверов и аутентификации `gh`.
- Маршрутизация как методология: namespace meta-skills показывают, что зрелая agentic methodology должна управлять не только кодом, но и собственной skill surface, иначе большой набор команд становится контекстным шумом.
- Пригодность до исполнения как отдельная тема: Nyquist validation и Plan Drift Guard проверяют, можно ли вообще передавать план исполнителю до того, как появляется diff.
- Артефакт как передача ответственности: `artifact-types.md` помогает сформулировать правило, что долговечный контекст должен иметь следующего потребителя. Без механизма потребления `.planning/` превращается в архив, а не в рабочую память.
- Многоуровневая проверка реализации: existence -> substance -> wiring -> behavior. Это хорошая рамка для различения "файл создан", "код не stub", "код подключён" и "пользовательское поведение работает".
- Маршрутизация моделей как часть методологии, а не только оптимизация стоимости: разные роли и фазы имеют разную цену ошибки, поэтому GSD связывает model profile с типом работы и средой выполнения.
- Конфигурация как часть метода: `models.*`, `model_overrides`, `fast_mode`, `effort`, `agent_skills`, `global_learnings`, `parallelization.*` и `safety.*` могут менять реальный процесс без изменения видимой команды. В теории нужно показывать GSD не только как цикл, но и как слой политики процесса вокруг цикла.
- Входные каналы рядом с фазовым циклом: ingest, graphify, обнаружение API-поверхности и фоновые задачи помогают управлять рабочим материалом до превращения в фазу, но требуют проверки свежести и явных потребителей.
- Контекст разной зрелости: backlog, seeds, threads и workstreams показывают, что долговечный контекст бывает не только "решение" или "план"; методология должна хранить идеи, гипотезы и параллельные линии без насильственного превращения всего в текущий область текущей фазы.
- Таксономия контрольных остановок: pre-flight, revision, escalation и abort gates показывают, что "остановить агента" бывает разным действием с разной ценой и разным моментом применения.
- Экономика checkpoint: human verification нужно группировать с учётом цены холодного старта, а human decisions/actions нельзя откладывать только ради меньшего числа прерываний.
- Браузерные свидетельства как сопутствующий слой проверки: `gsd-browser` показывает, как acceptance может подкрепляться assertions, recordings, traces, HAR и live takeover, не превращаясь в отдельную GSD-поверхность.
- Бюджет контекста как часть методологии: правила чтения файлов, запрет на чтение `agents/*.md` оркестратором, уровни деградации и стоимость MCP-схемы должны попасть в теорию про гигиену контекста.

### Что лучше уйдёт в Handbook / Fieldbook

- Командная шпаргалка по `gsd-core`: `/gsd-new-project`, `/gsd-discuss-phase`, `/gsd-plan-phase`, `/gsd-execute-phase`, `/gsd-verify-work`, `/gsd-ship`, `/gsd-progress --next`.
- Практическое правило выбора phase size: примеры good/too broad/too small phase.
- Пример структуры `.planning/phases/01-.../`: `CONTEXT.md`, `RESEARCH.md`, `PLAN.md`, `SUMMARY.md`, `VERIFICATION.md`, `UAT.md`.
- Рецепт для подключения существующего проекта: `/gsd-map-codebase` -> `/gsd-new-project`.
- Рецепт для автоматического режима `gsd-pi`: установка, настройка провайдера, `/gsd auto`, пауза/возобновление/остановка/направление/захват/статус.
- Настройка команды проверки и ограничение синтаксиса shell.
- Настройка Git-изоляции: `none`, `worktree`, `branch`; когда горячая перезагрузка ломается из-за worktree.
- Чек-лист диагностики для застрявшего автоматического режима: `/gsd status`, `.gsd/journal/`, `/gsd forensics`, `/gsd doctor`, проверки Git-конфликтов.
- Безопасная работа с checkpoints легитимности пакетов и подозрительными пакетами.
- Рецепт миграции/восстановления: `/gsd migrate` для `.planning` -> `.gsd`, `/gsd doctor` после миграции, `/gsd recover` только как явный разрушительный импорт из markdown.
- Рецепт headless-режима для CI/оркестраторов: `gsd headless query`, `gsd headless dispatch`, `gsd headless new-milestone --context`, коды выхода `0/1/2`.
- Fieldbook-рецепт для внешней автоматизации: `gsd headless query` для чтения состояния, `gsd headless dispatch <phase>` для принудительного шага, `gsd headless recover` как явная команда восстановления, `gsd --mode mcp` только если внешний клиент действительно должен управлять GSD-инструментами.
- Рецепт для нескольких репозиториев: `workspace.mode: parent`, `repositories.<id>.path`, `targetRepositories`, проверка и commit policy по каждому репозиторию.
- Операторская памятка по аудиту и приватности: когда включать `GSD_AUDIT`, когда нельзя включать `GSD_AUDIT_ARGS`, почему `.planning/config.json` и `.planning/.gsd-trace.jsonl` нельзя автоматически считать безопасными артефактами для коммита.
- Шаблон качественного `PLAN.md`: точные пути файлов, действия задачи, критерии приёмки, команду проверки, откат/восстановление, бюджет контекста, `must_haves`, обязательные артефакты, ключевые ссылки.
- Чек-лист для ревью плана перед исполнением по `gsd-plan-checker`: покрытие, порядок зависимостей, пересечения, смещение к счастливому пути, проверяемость, область и недостающий вход.
- Практическая памятка для продолжения свежим агентом после checkpoint: что должно быть в структурированном checkpoint-сообщении, как новый исполнитель сверяет предыдущие коммиты и где человек подтверждает продолжение.
- Операторская инструкция по checkpoints: когда исполнитель может исправить ошибку сам, а когда обязан остановиться из-за архитектурного решения, решения по пакету или блокера.
- Шпаргалка по маршрутизаторам пространств имён `gsd-core`: `/gsd-workflow`, `/gsd-project`, `/gsd-quality`, `/gsd-context`, `/gsd-manage`, `/gsd-ideate`; когда вызывать маршрутизатор, а когда прямую команду.
- Рецепт предварительной проверки перед исполнением: `/gsd-plan-status`, артефакты Nyquist, путь обновления/перепланирования, что делать с устаревшими, осиротевшими, грязными или истёкшими планами.
- Рецепты проверки из `verification-patterns.md`: поиск stub-pattern, проверка соединений, runtime smoke-проверка, браузерно подкреплённые свидетельства и правила, когда программной проверки недостаточно.
- Рецепты жизненного цикла артефактов: для каждого файла указать производителя, потребителя, жизненный цикл, правило замены/архивации и поведение при возобновлении; особенно для `HANDOFF.json`, `.continue-here.md`, `SUMMARY.md`, `VERIFICATION.md`.
- Рецепты профилей моделей: как выбрать `quality` / `balanced` / `budget` / `adaptive` / `inherit`, как настроить `model_overrides`, где нужен `resolve_model_ids: "omit"` и как не получить неожиданные расходы через OpenRouter или локального провайдера.
- Примеры команд именно для Codex: использовать `$gsd-*` (`$gsd-start`, `$gsd-plan-phase 1`, `$gsd-progress --next`), а не `/gsd-*` в стиле Claude, если пример предназначен именно для Codex.
- Практика настройки `agent_skills` и `global_learnings`: где хранить проектные навыки, как не смешать глобальные уроки между проектами и как проверять, какие skills были подмешаны конкретному агенту.
- Практика ingest/graphify/surface: когда запускать `/gsd-ingest-docs`, как читать `INGEST-CONFLICTS.md`, когда обновлять `.planning/graphs/`, как не считать `API-SURFACE.md` исчерпывающей картой без проверки маршрутов конкретного фреймворка.
- Чек-лист безопасности worktree/parallelization: `require_clean_worktree`, защищённые ветки, `worktree_path`, именование веток, checkpoint после волны/задачи и проверка, что агент действительно стартовал из нужного worktree.
- Практика backlog/seeds/threads: как захватывать будущую работу, когда повышать seed до требования/решения, когда заводить отдельный thread вместо обсуждения в активной фазе.
- Рецепт workstream: создание, branch/checkpoint/merge/status/close, отличие workstream от волны зависимостей внутри одной фазы.
- Чек-лист security hardening для `gsd-core`: `.planning/` gitignore validation, `safety.dangerous_git_policy`, `tools.allowed/denied/confirm_required`, атомарные записи/резервные копии, проверки целостности установщика.
- Рецепт issue-driven orchestration: приём issue, классификация, ревью, преобразование в работу roadmap/phase, закрытие после поставки.
- Рецепт политики checkpoint: когда выбирать `end-of-phase`, `after-each-wave`, `after-each-task`, и почему decision/action checkpoints не откладываются как проверка.
- Чек-лист бюджета контекста: что читать полностью, что читать только как frontmatter/summary, когда делать checkpoint, как проверять `must_haves` у результата субагента и как аудитить включённые MCP-серверы.
- Рецепт browser-proof: `gsd-browser mcp`, ссылки на снимки, утверждения, снимки экрана/записи/трассы/HAR, передача управления человеку, хранение артефактов и политика приватности для браузерных свидетельств.

## Локальные различения, чтобы не спутать

- GSD не равен Spec Kit. По имеющимся источникам GSD делает упор на повторяющийся рабочий цикл, субагентов со свежим контекстом, долговечное состояние и проверку; полноценное сравнение со Spec Kit нужно делать в отдельном сравнительном проходе.
- `gsd-core` не равен `gsd-pi`: первый встраивается в существующую среду выполнения и хранит состояние в `.planning/`; второй сам является терминальным агентом/оркестратором и хранит авторитетное состояние среды выполнения в SQLite с `.gsd/`-проекциями.
- `gsd-browser` не является GSD-поверхностью; roadmap называет его сопутствующим продуктом для браузерного доказательства и свидетельств, который подключается к проверке [источник: roadmap, https://opengsd.net/roadmap].
- Open GSD не нужно описывать как чисто новый проект без прошлого: `Origin Credit` прямо признаёт исходный GSD Lex Christopherson / Taches, а `Promise` — восстановление с прозрачностью после ущерба доверию [источники: Origin Credit, https://opengsd.net/origin; Promise, https://opengsd.net/promise].

## Кандидаты на иллюстрации

- Схема фазового цикла для `gsd-core`: `Discuss -> (UI design) -> Plan -> Execute -> Verify -> Ship`; можно собрать свою схему по `the-phase-loop.md`, не копируя изображение [источник: https://raw.githubusercontent.com/open-gsd/gsd-core/main/docs/explanation/the-phase-loop.md]. Статус: кандидат на авторскую схему.
- Поток артефактов для `.planning/`: `PROJECT.md` / `REQUIREMENTS.md` / `ROADMAP.md` / `STATE.md` -> фазовый `CONTEXT.md` -> `RESEARCH.md` -> `PLAN.md` -> `SUMMARY.md` -> `VERIFICATION.md` -> `UAT.md` -> тело PR. Статус: кандидат на авторскую схему.
- Машина состояний автоматического режима `gsd-pi`: планирование -> исполнение -> завершение -> переоценка дорожной карты -> следующий срез -> проверка этапа -> завершение этапа [источник: https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/auto-mode.md]. Статус: кандидат на авторскую схему.
- Схема авторитетности состояния `gsd-pi`: SQLite database как источник истины -> `.gsd/` markdown-проекции -> промпт/контекст dispatch -> результат единицы -> обновление БД. Статус: кандидат на авторскую схему.
- Схема миграции/восстановления: legacy `.planning/` -> preview в `/gsd migrate` -> импорт в БД -> `.gsd/`-проекции -> `.gsd/migration/*`; повреждённая БД -> явный `/gsd recover` -> пересборка из отрендеренного markdown. Статус: кандидат на авторскую схему по документации миграции.
- Слои архитектуры `gsd-core`: команда пользователя -> слой команд -> слой рабочих процессов -> агенты со свежим контекстом -> CLI-инструменты -> `.planning/`. Статус: кандидат на краткую схему, полезную для объяснения "prompt framework vs standalone agent".
- Схема рабочей области с несколькими репозиториями для `gsd-pi`: родительское `.gsd`-состояние -> объявленные идентификаторы репозиториев -> задачи с областью путей -> политика проверки/коммитов по репозиториям. Статус: кандидат на Handbook-схему.
- Схема Git-изоляции: `none`, `worktree`, `branch`; ветка/worktree этапа -> squash merge в `main` [источник: https://raw.githubusercontent.com/open-gsd/gsd-pi/main/docs/user-docs/git-strategy.md]. Статус: кандидат на авторскую схему.
- Карта поверхностей из дорожной карты Open GSD: `gsd-core`, `gsd-pi`, `gsd-workbench`, `gsd-cloud`, сопутствующий `gsd-browser` [источник: https://opengsd.net/roadmap]. Статус: кандидат на краткую карту экосистемы; не брать графику сайта без проверки прав.
- Схема полномочий агентов для `gsd-core`: планировщик читает `ROADMAP.md`/`REQUIREMENTS.md`/`RESEARCH.md`/`CONTEXT.md` -> создаёт исполнимый `PLAN.md`; plan-checker возвращает `PASS`/`NEEDS_REVISION`/`BLOCKED_BY_MISSING_INPUT`; исполнитель выполняет атомарную задачу, checkpoint или коммит. Статус: кандидат на авторскую схему по контрактам агентских ролей.
- Матрица аудита покрытия: строки `GOAL`, `REQ`, `RESEARCH`, `CONTEXT decisions`; столбцы задачи плана / артефакты / проверка / ключевые ссылки. Статус: кандидат на Handbook-схему для объяснения quality gate.
- Схема безопасности волн: задачи `PLAN.md` -> владение файлами/областями ответственности -> волны зависимостей -> контексты исполнителей -> сводки/коммиты. Статус: кандидат на Handbook-схему, чтобы объяснить, почему plan-checker проверяет пересечения до исполнения.
- Карточка продолжения после checkpoint: триггер -> работа, выполненная к этому моменту -> вопрос/проверка для человека -> точка возобновления -> свежий исполнитель читает предыдущие коммиты. Статус: кандидат на Fieldbook-памятку по человеческим контрольным остановкам.
- Схема стека наблюдаемости: `stderr HandlerFailure traceId` -> опциональный `.planning/.gsd-trace.jsonl` -> `/gsd doctor` / `/gsd inspect` / `/gsd forensics` -> `/gsd report` / `gsd headless query`. Статус: кандидат на Handbook-схему по диагностике самого процесса.
- Схема границы безопасности: `.planning/config.json` с ключами в открытом тексте, маскированный вывод UI/журнала, `GSD_AUDIT_ARGS` выключен по умолчанию, `.gsd/mcp.json` для локальных MCP-секретов. Статус: кандидат на Fieldbook-схему по безопасной эксплуатации.
- Поверхность маршрутизации `gsd-core`: намерение пользователя -> маршрутизатор пространства имён (`workflow/project/quality/context/manage/ideate`) -> конкретная команда -> рабочий процесс/агент/справочный слой. Статус: кандидат на схему "как не перегрузить модель списком команд".
- Схема предварительной проверки пригодности перед исполнением: `PLAN.md` + `source_sha` + хэши зависимостей -> `/gsd-plan-status` -> Nyquist -> `PLAN-REPAIRS.md` / `CHECKPOINT-NYQUIST.md` / `NYQUIST-PASS.md` -> отправка исполнителю. Статус: кандидат на Handbook-схему.
- Карта зрелости контекста: пункт backlog -> seed -> thread -> дорожная карта/требование/решение -> фазовые `CONTEXT.md`/`PLAN.md` -> поставленное свидетельство. Статус: кандидат на теоретическую схему про разные состояния контекста.
- Диаграмма workstream vs wave: ветка/checkpoint/merge workstream как долгоживущая линия; dependency wave как параллельное исполнение задач внутри фазы. Статус: кандидат на Fieldbook-схему.
- Диаграмма таксономии контрольных остановок: предварительная проверка до дорогой работы, revision после создания артефакта, escalation во время исполнения, abort для невосстановимого состояния. Статус: кандидат на теоретическую схему по разным типам остановки.
- Схема экономики checkpoint: end-of-phase vs after-each-wave vs after-each-task verification, отдельно decision/action checkpoints. Статус: кандидат на Handbook-схему для выбора режима человеческих контрольных остановок.
- Цикл браузерных свидетельств: ссылки на снимки -> действие -> утверждение -> снимок экрана/трасса/HAR/запись -> свидетельство для UAT/PR, с границей приватности. Статус: кандидат на Fieldbook-схему для браузерного UAT.
- Карта бюджета контекста: чтение оркестратором только frontmatter -> `must_haves` субагента -> уровни деградации -> цена MCP-схемы. Статус: кандидат на теоретическую схему про гигиену контекста.
- Лестница проверки: existence -> substance -> wiring -> behavior, с отдельной отметкой, где достаточно статических проверок/проверок среды выполнения, а где нужен человек или браузерные свидетельства. Статус: кандидат на теоретическую и Handbook-схему.
- Граф потребления артефактов: рабочий процесс-производитель -> форма/расположение артефакта -> рабочий процесс-потребитель; особенно `CONTEXT.md` -> планировщик/исполнитель, `SUMMARY.md` -> будущий планировщик/сводка этапа, `HANDOFF.json` + `.continue-here.md` -> возобновление. Статус: кандидат на схему долговечного контекста.
- Матрица маршрутизации моделей: роль агента / фаза -> уровень по умолчанию (`light`, `standard`, `heavy`) -> профиль (`quality`, `balanced`, `budget`, `adaptive`, `inherit`) -> оговорки среды выполнения (`omit`, OpenRouter/local). Статус: кандидат на Fieldbook-схему по стоимости и качеству.
- Стек политики конфигурации: видимая команда -> `.planning/config.json` -> значения по умолчанию среды выполнения -> `models.*` / `model_overrides` -> `agent_skills` -> `global_learnings` -> контрольные остановки безопасности/параллельности. Статус: кандидат на теоретическую схему о том, что GSD-команда исполняется внутри слоя политики.
- Слой приёма и графа: внешние документы -> `/gsd-ingest-docs` -> `.planning/` + `INGEST-CONFLICTS.md`; дорожная карта/issues/plans/карты кода -> `/gsd-graphify` -> `.planning/graphs/` -> критический путь/заблокированная работа. Статус: кандидат на Handbook-схему для подготовки материала до фазового цикла.
- Схема безопасности worktree: настроенный базовый путь -> ветка/worktree этапа -> защищённые ветки -> проверка `cwd` исполнителя -> ритм checkpoint. Статус: кандидат на Fieldbook-схему по безопасной параллельности.
