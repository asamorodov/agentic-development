# Spec Kit

## Черновое назначение

Это рабочее методологическое досье по Spec Kit. Его задача — сохранить фактуру источников для будущего текста сайта: порядок работы, команды, файлы, артефакты, места человеческого решения, проверки, ограничения и неудобные детали. Досье не является финальной главой и не должно сглаживать Spec Kit до общей формулы “сначала спецификация, потом код”.

## Краткая рабочая карта

Spec Kit в этом досье лучше читать не как один CLI и не как набор слеш-команд, а как несколько слоёв одного SDD-процесса. Нижний слой — стандартная цепочка `specify → plan → tasks → implement`, где каждый шаг производит Markdown-артефакт для следующего шага. Над ней находятся интеграции, которые высаживают этот процесс в разные ИИ-агенты разработки. Ещё выше появляются рабочие процессы, пресеты и расширения: рабочие процессы делают процесс возобновляемой многошаговой оркестрацией с контрольными точками; пресеты меняют шаблоны, команды и терминологию; расширения добавляют предметно-специфичные команды, интеграции внешних инструментов и контрольные точки качества [Workflows](https://github.github.com/spec-kit/reference/workflows.html), [Presets](https://github.github.com/spec-kit/reference/presets.html), [Extensions](https://github.github.com/spec-kit/reference/extensions.html).

Эта карта важна для будущей теории: Spec Kit уже не сводится к «спецификация перед кодом». Его текущая поверхность показывает, как процесс с первичностью спецификации превращается в настраиваемую инфраструктуру разработки: с каталогами, приоритетами разрешения, человеческими контрольными точками, возобновляемыми запусками и риском путаницы в цепочке поставки и переопределениях.

## Роль в ИИ-ориентированном SDLC

Spec Kit позиционирует себя как набор инструментов для Spec-Driven Development: разработка начинается с описания того, что нужно построить, проходит через структурированные фазы и только затем передаётся ИИ-агент разработки для реализации [документация Spec Kit](https://github.github.com/spec-kit/). В терминах будущей теории это соседний спецификационный режим: он переносит центр тяжести из одноразового запроса в цепочку версионируемых Markdown-артефактов.

Официальная документация формулирует базовую цепочку как `Spec -> Plan -> Tasks -> Implement`: каждая фаза создаёт Markdown-артефакт, который становится входом для следующей фазы [документация Spec Kit](https://github.github.com/spec-kit/). Это важнее, чем сам факт наличия CLI. Методология заставляет намерение пройти через несколько документов: сначала поведение и границы, затем технические решения, затем исполнимые задачи, затем реализация.

Spec Kit не является отдельной IDE. Он устанавливает командные файлы, правила контекста и структуру каталогов для выбранного агента через `specify init` и интеграции [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Поэтому его лучше понимать как переносимый слой процесса поверх Copilot, Codex, Claude, Gemini, Kiro, Windsurf и других инструментов, а не как одного исполнителя.

## Проблема, которую решает

Главная проблема в источниках описана как разрыв между намерением, документацией и кодом. В `spec-driven.md` Spec Kit утверждает, что традиционная разработка делает код источником истины, а спецификации быстро отстают; SDD пытается перевернуть это отношение: спецификация и план реализации должны управлять кодом, а не обслуживать его задним числом [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).

В практическом ИИ-контексте проблема ещё острее: агент склонен быстро переходить к коду, подставлять правдоподобные детали и терять связь с исходным намерением. Spec Kit отвечает на это шаблонами, фазами и проверками. В `spec-driven.md` отдельно подчёркнуто, что шаблоны должны сдерживать модель: спецификация должна держаться на `WHAT` и `WHY`, а детали `HOW` уходят в план [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).

Статья Spec Kit Agents добавляет исследовательский угол: даже структурированный SDD не решает проблему “слепота к контексту” в больших репозиториях. Авторы предлагают фазовые хуки привязки к контексту: перед фазами `Specify`, `Plan`, `Tasks`, `Implement` выполняются хуки пробного чтения без записи, а валидационные хуки проверяют промежуточные артефакты и среду [Spec Kit Agents](https://arxiv.org/abs/2604.05278). Для досье это важная оговорка: сама цепочка документов полезна, но без привязки к реальному репозиторию агент может всё равно строить план на устаревших или выдуманных предпосылках.

## Рабочий процесс

### Установка и инициализация

Быстрый старт показывает запуск `uvx --from git+https://github.com/github/spec-kit.git specify init <PROJECT_NAME>` или инициализацию текущего каталога через `specify init .` [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html). Для постоянной установки документация также указывает `pipx install git+https://github.com/github/spec-kit.git`; README репозитория показывает вариант `uv tool install specify-cli --from git+https://github.com/github/spec-kit.git@vX.Y.Z` [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html) [github/spec-kit](https://github.com/github/spec-kit).

Installation Guide добавляет важную оговорку по цепочке поставки: официальные поддерживаемые пакеты Spec Kit идут из репозитория `github/spec-kit`; одноимённые пакеты на PyPI, включая `specify-cli`, не аффилированы с проектом и не поддерживаются сопровождающими [Installation Guide](https://github.github.com/spec-kit/installation.html). Поэтому практическая инструкция по Spec Kit должна фиксировать источник установки, а не просто писать “установите `specify-cli`”. Там же указаны предварительные требования: Linux/macOS или Windows с PowerShell-скриптами без WSL, `uv` или `pipx`, Python 3.11+ и Git [Installation Guide](https://github.github.com/spec-kit/installation.html).

При инициализации у интеграции тоже есть вариант по умолчанию, который может быть неожиданным. В интерактивном терминале CLI предлагает выбрать интеграцию агента разработки, а в неинтерактивных запусках вроде CI или конвейерных запусках по умолчанию выбирается GitHub Copilot, если не передать `--integration` [Installation Guide](https://github.github.com/spec-kit/installation.html). Это маленькая, но важная деталь для автоматизации: запуск `specify init` без явного `--integration codex|claude|gemini|...` может поставить не тот набор команд и правила контекста.

Справочник Core Commands добавляет параметры, которые меняют не содержание методологии, а её техническую посадку в проект: `--script sh|ps` выбирает тип служебных скриптов, `--no-git` отключает инициализацию Git, `--branch-numbering sequential|timestamp` меняет стратегию нумерации веток фич, а `--preset <id>` позволяет установить пресет уже на этапе `specify init` [Core Commands](https://github.github.com/spec-kit/reference/core.html). Там же есть неудобная деталь про поведение по умолчанию: `git` расширение пока включено при `specify init` по умолчанию, но начиная с `v0.10.0` должно требовать явного включения через `specify extension add git` [Core Commands](https://github.github.com/spec-kit/reference/core.html). Для будущего практического текста это значит, что команды и поведение веток нельзя описывать как вечную константу: локальная версия CLI и параметры `init` имеют значение.

Инициализация выбирает интеграцию. Например, документация даёт `specify init my-project --integration copilot`, а для Codex README показывает `specify init . --integration codex --integration-options="--skills"` [документация Spec Kit](https://github.github.com/spec-kit/) [github/spec-kit](https://github.com/github/spec-kit). Для Codex CLI интеграция основана на skills: skills устанавливаются в `.agents/skills`, а команды вызываются как `$speckit-<command>`; большинство других агентов используют слеш-команды вида `/speckit.*` [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html) [github/spec-kit](https://github.com/github/spec-kit).

Справочник по интеграциям показывает, что “поддерживаемый агент” в Spec Kit не означает одинаковое поведение всех поверхностей. Codex CLI может ставиться в режиме skills, Kiro IDE получает Markdown-команды, а Kiro CLI из-за отсутствия динамической подстановки аргументов получает резервные разделы с прямыми вызовами скриптов; Pi поддерживается через Markdown-команды, но документация отдельно предупреждает, что CLI не настраивает MCP-сервер внутри Pi [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Поэтому практическая инструкция должна проверять не только имя интеграции, но и её тип, формат команд, способ запуска в среде агента и примечания по конкретному агенту.

Текущая таблица поддерживаемых интеграций делает это различие ещё шире. В ней рядом стоят IDE-агенты, CLI-агенты, интеграции на основе skills и специальные форматы: `goose` использует YAML-рецепты в `.goose/recipes/`, `devin` устанавливает skills в `.devin/skills/` и вызывает их как `/speckit-<command>`, `hermes` ставит skills глобально в `~/.hermes/skills/`, `rovodev` генерирует `.rovodev/skills/`, обёртки и `prompts.yml` с запуском через `acli rovodev`, а `generic` требует явный `--integration-options="--commands-dir <path>"` для неподдержанного агента [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Это важная фактура для методологии: Spec Kit обещает переносимый SDD-процесс, но переносимость достигается не одинаковой командной поверхностью, а набором адаптеров с разной степенью изоляции и разными способами обнаружения команд.

Реестр Python делает эту неоднородность ещё заметнее. В `src/specify_cli/integrations/__init__.py` зарегистрированы десятки встроенных интеграций, причём имена подпакетов иногда отличаются от пользовательских ключей: например, `kiro_cli` регистрирует ключ `kiro-cli`, а `cursor_agent` — `cursor-agent` [src/specify_cli/integrations/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/__init__.py). Это важно для диагностики: каталог исходного кода, ключ интеграции, имя CLI-бинарника и имя папки, куда ставятся команды, могут быть разными сущностями.

Документация по интеграциям подчёркивает деталь, которую легко потерять: в проекте есть интеграция по умолчанию, но возможна установка нескольких интеграций, если они безопасны для совместной установки или пользователь явно берёт риск через `--force` [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Это превращает Spec Kit в командный слой, где один и тот же набор спецификационных артефактов может обслуживаться разными агентами, но общие шаблоны всё равно выравниваются под интеграцию по умолчанию.

При этом “поддерживается” и “безопасна для совместной установки” — разные статусы. Справочник считает безопасными только интеграции с изолированными каталогами агентов, отдельным файлом контекста, стабильными настройками вызова команд и отдельным установочным манифестом; в список безопасных входят, например, `codex`, `claude`, `cursor-agent`, `gemini`, `qwen`, `kimi`, `windsurf`, `trae`, `tabnine`, `junie`, `roo`, `shai`, `kilocode`, `qodercli`, `iflow`, `codebuddy` и `auggie`, но туда не попадают интеграции с общими файлами контекста, динамическими путями установки или общими настройками инструмента [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Для будущего практического справочника это означает, что команду `specify integration install <key>` нельзя трактовать как безусловно безопасное добавление “ещё одного агента”: сначала нужно смотреть безопасность совместной установки и понимать, какие файлы будут видны какому агенту.

Справочник по интеграциям показывает, что этот слой управляется отдельным жизненным циклом, а не только начальным `specify init`: `specify integration list` показывает доступные интеграции, текущую интеграцию по умолчанию, установленные интеграции, тип CLI/IDE и безопасность совместной установки; `install`, `use`, `switch`, `upgrade` и `uninstall` меняют состояние проекта после инициализации [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Важная операционная деталь: управляемая поддержка совместной установки появилась в Spec Kit `0.8.5`, а одноразовый запуск через `uvx --from git+https://github.com/github/spec-kit.git specify ...` использует временную копию только для этой команды и не обновляет постоянный `specify` на `PATH` [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Поэтому странное поведение интеграций надо проверять не только по документации, но и по локальной версии CLI.

Исходный код Python уточняет, что у слоя интеграций есть свой проверяемый журнал установки. `IntegrationManifest.manifest_path` пишет файл `.specify/integrations/<key>.manifest.json`, а для общих скриптов/шаблонов используется отдельный `speckit.manifest.json` [src/specify_cli/integrations/manifest.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/manifest.py) [src/specify_cli/shared_infra.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/shared_infra.py). В манифест попадают относительные пути и SHA-256-хеши установленных файлов; при удалении файл удаляется только если текущий хеш совпадает с записанным, а изменённые файлы остаются на месте без `force` [src/specify_cli/integrations/manifest.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/manifest.py). Это делает жизненный цикл интеграции не просто удобной командой, а механизмом сохранения пользовательских правок.

`IntegrationBase` показывает отдельный слой запуска команд через CLI агента. Для CLI-интеграций бинарник можно переопределить переменной вида `SPECKIT_INTEGRATION_<KEY>_EXECUTABLE`, а дополнительные флаги передать через `SPECKIT_INTEGRATION_<KEY>_EXTRA_ARGS`; для ключей с дефисом дефис заменяется подчёркиванием, например `kiro-cli` превращается в `SPECKIT_INTEGRATION_KIRO_CLI_EXTRA_ARGS` [src/specify_cli/integrations/base.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/base.py). Поэтому “запустить `/speckit.plan`” в автоматизированном рабочем процессе означает не только наличие командного файла, но и конкретную командную строку агента, модель, формат вывода, режим потокового вывода/захвата и локальные переопределения окружения.

При работе без Git есть отдельная переменная `SPECIFY_FEATURE`: справочник Core Commands описывает её как переопределение обнаружения каталога фичи для репозиториев без Git; её нужно задать в контексте агента до `/speckit.plan` и последующих команд [Core Commands](https://github.github.com/spec-kit/reference/core.html). Это уточняет границу Git-центричного описания Spec Kit: Git-ветка остаётся стандартной опорой, но методология допускает явную адресацию каталога фичи без ветки.

Фактические сгенерированные bash-скрипты делают эту картину ещё более конкретной. `common.sh` ищет корень проекта сначала по `.specify`, а не по Git; Git используется как резервный вариант только если `.specify` не найден [scripts/bash/common.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/common.sh). Это важная защита для монорепозиториев и вложенных проектов: Spec Kit пытается не “уплыть” в родительский Git-репозиторий, если сам набор инструментов инициализирован глубже.

В `common.sh` активная фича определяется несколькими слоями: `SPECIFY_FEATURE_DIRECTORY` имеет приоритет как явное переопределение, затем читается `.specify/feature.json` с ключом `feature_directory`, и только после этого используется поиск по префиксу имени ветки в `specs/` [scripts/bash/common.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/common.sh). Отдельная переменная `SPECIFY_FEATURE` влияет на `get_current_branch`, особенно в режиме без Git; если Git отсутствует, скрипт пытается выбрать последний каталог фичи по последовательному префиксу или префиксу временной метки и только затем падает к `main` [scripts/bash/common.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/common.sh). Поэтому в практическом описании нельзя сводить активную фичу к одной ветке: реальная адресация состоит из переопределений окружения, сохранённого состояния и резервного поиска по `specs/`.

`create-new-feature.sh` показывает, что `/speckit.specify` опирается на отдельный скрипт с собственными гарантиями: он поддерживает `--json`, `--dry-run`, `--allow-existing-branch`, `--short-name`, `--number` и `--timestamp`; последовательная нумерация берёт максимум из Git-веток, удалённых refs и каталогов `specs/`, а режим временной метки создаёт префикс вида `YYYYMMDD-HHMMSS` [scripts/bash/create-new-feature.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/create-new-feature.sh). В обычном режиме скрипт создаёт ветку, каталог фичи и `spec.md` из `spec-template`, а при `--dry-run` вычисляет имена и пути без создания ветки, каталогов и файлов [scripts/bash/create-new-feature.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/create-new-feature.sh). Это место полезно для практического справочника: перед реальным запуском можно проверить в режиме `--dry-run`, какие пути ветки и спецификации получатся.

PowerShell-вариант `create-new-feature.ps1` сохраняет те же пользовательские флаги: `-Json`, `-DryRun`, `-AllowExistingBranch`, `-ShortName`, `-Number`, `-Timestamp`, `-Help` [scripts/powershell/create-new-feature.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/create-new-feature.ps1). Для Windows-проектов это важно не как косметический порт bash-скрипта, а как отдельная исполнимая поверхность с собственными правилами путей, `Get-ChildItem`, `Resolve-Path` и вызова `git`. Текущий `create-new-feature.ps1` уже считает порядковый номер не только по `specs/`, но и по локальным веткам и удалённым refs, а в режиме без побочных изменений использует `git ls-remote` вместо `fetch` [scripts/powershell/create-new-feature.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/create-new-feature.ps1). Исторически именно здесь был сбой: GitHub issue #975 описывал, что старый PowerShell-скрипт видел только `specs/` на текущей ветке и мог повторно создать `001-*`, если существующая ветка фичи ещё не была влита в `main`; PR #986 позже включил исправление [issue #975](https://github.com/github/spec-kit/issues/975) [PR #986](https://github.com/github/spec-kit/pull/986).

### Рекомендуемая цепочка

Quick Start различает быстрый путь для экспериментов и более строгий путь для продуктовых фич. Лёгкая цепочка: `/speckit.specify -> /speckit.plan -> /speckit.tasks -> /speckit.implement`. Для задач с существенной неоднозначностью документация рекомендует регулярные контрольные точки качества: `/speckit.constitution -> /speckit.specify -> /speckit.clarify -> /speckit.checklist -> /speckit.plan -> /speckit.tasks -> /speckit.analyze -> /speckit.implement` [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html).

`/speckit.constitution` создаёт или обновляет управляющие принципы проекта. Quick Start предлагает передать конкретные принципы прямо аргументом команды, например подход `library-first`, строгий TDD и предпочтение паттернов функционального программирования [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html). В README уточняется, что этот шаг создаёт или обновляет `.specify/memory/constitution.md`, к которому агент разработки обращается на следующих фазах [github/spec-kit](https://github.com/github/spec-kit).

Текущий шаблон `/speckit.constitution` показывает, что команда constitution работает не как свободное редактирование одного файла. Он должен загрузить `.specify/memory/constitution.md`, заполнить токены-заполнители, определить `RATIFICATION_DATE`, `LAST_AMENDED_DATE` и повышение `CONSTITUTION_VERSION` по semver: `MAJOR` для несовместимых изменений управления, `MINOR` для новых или существенно расширенных принципов, `PATCH` для уточнений и несемантических правок [constitution.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/constitution.md). После этого команда обязана сверить зависимые артефакты: `.specify/templates/plan-template.md`, `spec-template.md`, `tasks-template.md`, командные файлы и рабочие указания документации, а в начало constitution добавить `Sync Impact Report` с изменением версии, изменёнными принципами, шаблонами со статусами `updated`/`pending` и TODO [constitution.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/constitution.md). Это сильная деталь для методологии: долгосрочные правила проекта должны проталкиваться в шаблоны и команды, а не просто лежать отдельным манифестом.

`/speckit.specify` описывает, что и зачем нужно построить, без выбора стека. В `spec-driven.md` команда описана как механизм, который автоматически нумерует фичу, создаёт ветку, копирует шаблон спецификации и создаёт структуру `specs/[branch-name]/` [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md). Это не просто генерация текста: команда связывает спецификацию с веткой и каталогом фичи.

Текущий командный шаблон уточняет это устройство: создание или переключение Git-ветки вынесено в `before_specify` хук, а сама основная команда всегда создаёт каталог фичи и `spec.md` [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md). Каталог спецификации больше нельзя понимать только как производную от имени ветки: команда может брать `SPECIFY_FEATURE_DIRECTORY`, иначе генерирует каталог под `specs/` по стратегии из `.specify/init-options.json` (`sequential` или `timestamp`) и сохраняет результат в `.specify/feature.json` [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md). Это важная практическая деталь: нижестоящие команды должны находить активную фичу через сохранённый каталог фичи, а не полагаться только на имя ветки.

`/speckit.specify` сам создаёт `checklists/requirements.md` и валидирует начальную спецификацию против критериев качества: отсутствие деталей реализации, фокус на пользовательской ценности, завершённые обязательные секции, тестируемые требования, измеримые критерии успеха, краевые случаи, ограниченный охват, зависимости и предположения [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md). Если остаются `[NEEDS CLARIFICATION]`, команда ограничивает их числом трёх, приоритизирует охват, безопасность/приватность и UX и задаёт пользователю вопросы с вариантами [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md). Это показывает, что первая фаза уже содержит встроенный цикл проверки качества, а не только первичную генерацию `spec.md`.

`/speckit.clarify` предназначена для снятия неоднозначности до планирования. Quick Start рекомендует использовать её перед `/speckit.plan`; ответы записываются в раздел Clarifications, а если пользователь сознательно пропускает уточнение ради исследовательского спайка или прототипа, это нужно явно сказать, чтобы агент не блокировался на недостающих уточнениях [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html) [github/spec-kit](https://github.com/github/spec-kit).

Командный шаблон для `/speckit.clarify` делает эту фазу более строгой, чем обычный чат с вопросами. Команда сканирует текущий `spec.md` по таксономии функционального охвата, доменных данных, UX-потока, нефункциональных качеств, интеграций, краевых случаев, ограничений, терминологии и сигналов завершённости; затем задаёт максимум пять вопросов, причём ровно по одному за раз [clarify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/clarify.md). Принятый ответ сразу записывается в `## Clarifications` / `### Session YYYY-MM-DD`, встраивается в подходящие разделы спецификации, после чего при наличии `checklists/requirements.md` команда пересчитывает чекбоксы требований [clarify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/clarify.md). Для будущего текста это важная механика: уточнение не остаётся в диалоге, а возвращается в `spec.md` как версионируемый источник.

`/speckit.checklist` создаёт чек-лист качества для проверки полноты, ясности и согласованности требований. Quick Start помещает эту команду перед техническим планом, а README называет её необязательной командой для валидации качества требований [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html) [github/spec-kit](https://github.com/github/spec-kit).

В шаблоне `/speckit.checklist` чек-листы названы “unit tests for requirements writing”: они проверяют качество текста требований, а не работоспособность реализации [checklist.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/checklist.md). Команда создаёт или дописывает тематические файлы в `FEATURE_DIR/checklists/`, например `ux.md`, `api.md`, `security.md`; элементы получают идентификаторы `CHK###`, а минимум 80% пунктов должны иметь ссылку трассируемости на раздел spec или явную пометку `[Gap]`, `[Ambiguity]`, `[Conflict]`, `[Assumption]` [checklist.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/checklist.md). Это удобно для теории: Spec Kit проверяет не “кнопка работает”, а “требование о кнопке достаточно ясно, полно и измеримо”.

`/speckit.plan` переводит спецификацию в технический план. `spec-driven.md` перечисляет его работу: анализ требований, пользовательские истории и критерии приёмки; проверка соответствия constitution; перевод бизнес-требований в архитектуру и детали реализации; создание сопроводительных документов для моделей данных, API-контрактов и тестовых сценариев; quickstart-валидация [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).

Командный шаблон для `/speckit.plan` добавляет конкретный порядок: запустить `setup-plan` из корня репозитория, прочитать `FEATURE_SPEC`, `/memory/constitution.md` и шаблон `IMPL_PLAN`, заполнить Technical Context, выполнить Constitution Check, затем пройти Phase 0 research и Phase 1 design/contracts [plan.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/plan.md). В Phase 0 неизвестности превращаются в исследовательские задачи, а результат должен попасть в `research.md` в форме `Decision`, `Rationale`, `Alternatives considered`; в Phase 1 создаются `data-model.md`, `contracts/`, `quickstart.md`, после чего обновляется файл контекста агента между специальными маркерами [plan.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/plan.md). Значит, планирование — это не один документ `plan.md`, а пакет проектных артефактов и обновление контекста агента.

Сгенерированный `setup-plan.sh` уточняет входную контрольную точку для этой фазы. Он получает пути через `get_feature_paths`, а затем проверяет имя ветки через `check_feature_branch`, если только `.specify/feature.json` не указывает на существующий активный каталог фичи [scripts/bash/setup-plan.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/setup-plan.sh). Если `plan.md` уже существует, скрипт не перезаписывает его, а пропускает копирование шаблона; если шаблон не найден, создаёт пустой `plan.md` с предупреждением [scripts/bash/setup-plan.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/setup-plan.sh). Это неудобная, но важная деталь: отсутствие шаблона не обязательно останавливает фазу plan, поэтому качество `plan.md` нельзя проверять только по факту существования файла.

`/speckit.tasks` читает `plan.md` как обязательный вход и, если они есть, `data-model.md`, `contracts/`, `research.md`; затем выводит `tasks.md`, отмечает независимые задачи `[P]` и выделяет безопасные группы параллельного выполнения [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md). Это место, где план становится исполнимым списком работ.

`setup-tasks.sh` ведёт себя строже, чем `setup-plan.sh`: он останавливается, если в активном каталоге фичи нет `plan.md` или `spec.md`, и прямо говорит запустить сначала соответствующие команды [scripts/bash/setup-tasks.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/setup-tasks.sh). Он также собирает список доступных сопроводительных документов (`research.md`, `data-model.md`, `contracts/`, `quickstart.md`) и разрешает `tasks-template` через общий стек переопределений шаблонов; если шаблон не найден ни в переопределениях, пресетах, расширениях, ни в core-шаблонах, команда завершается ошибкой [scripts/bash/setup-tasks.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/setup-tasks.sh). Поэтому `tasks.md` в Spec Kit появляется после более жёсткой проверки входов, чем `plan.md`.

PowerShell `common.ps1` показывает, что Windows-ветка не сводится к переписанным именам файлов. Он ищет `.specify` через `Resolve-Path -LiteralPath`, чтобы корректно обрабатывать пути с wildcard-символами, проверяет Git именно на уровне найденного корня Spec Kit, принимает имена веток с gitflow-префиксом вроде `feat/004-name` через `Get-SpecKitEffectiveBranchName`, и сравнивает `.specify/feature.json` с активным каталогом фичи без учёта регистра только на Windows [scripts/powershell/common.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/common.ps1). Для методологии это добавляет ещё одну границу: если будущая инструкция пишется для Windows, нельзя просто дать bash-команды и считать, что поведение идентично; надо проверять `.ps1`-скрипты, их JSON-вывод и путь к установленным шаблонам.

Шаблон `/speckit.tasks` требует организовывать задачи по пользовательским историям, чтобы каждая история была независимо тестируемым инкрементом [tasks.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/tasks.md). Формат задачи строгий: `- [ ] T001 [P] [US1] Description with file path`; метка истории обязательна только внутри фаз историй, а в фазах `setup`/`foundational`/`polish` её не используют [tasks.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/tasks.md). Команда должна вывести граф зависимостей, примеры параллельного выполнения, стратегию MVP-first и сводку по числу задач на каждую пользовательскую историю [tasks.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/tasks.md). Для будущего текста это помогает показать, как Spec Kit превращает спецификацию в декомпозицию работ, пригодную для агента, а не просто в человеческий TODO.

`/speckit.analyze` проверяет согласованность и покрытие спецификации, плана и задач перед реализацией. Quick Start рекомендует держать первый запуск анализа до `/speckit.implement`, потому что найденные пробелы ещё можно исправить в плане и задачах; повторный анализ после реализации допускается как дополнительная проверка [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html).

Текущий `/speckit.analyze` принципиально работает без записи: он не меняет файлы и только выдаёт структурированный аналитический отчёт по `spec.md`, `plan.md`, `tasks.md` и constitution [analyze.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/analyze.md). Он строит инвентарь требований, историй/действий, покрытие задачами и правила constitution, затем ищет дублирование, неоднозначность, недоопределённость, проблемы выравнивания с constitution, пробелы покрытия и несогласованности; конфликты с constitution автоматически получают критичность `CRITICAL` [analyze.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/analyze.md). Команда ограничивает находки до 50 и в конце предлагает исправляющие правки только как отдельный следующий шаг с явным разрешением пользователя [analyze.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/analyze.md). Это хороший пример человеческой контрольной точки: методика заставляет сначала увидеть карту несогласованностей, а не сразу править всё агентом.

`/speckit.implement` исполняет задачи и строит фичу согласно плану [github/spec-kit](https://github.com/github/spec-kit). В досье важно не изображать эту команду как магическое завершение: она получает смысл только при наличии предыдущих артефактов и проверок.

Шаблон `/speckit.implement` перед чтением контекста сканирует `FEATURE_DIR/checklists/`: если хотя бы один checklist содержит незакрытые пункты, команда должна остановиться и спросить, продолжать ли реализацию [implement.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/implement.md). Затем она читает `tasks.md`, `plan.md`, необязательные `data-model.md`, `contracts/`, `research.md`, `quickstart.md` и constitution, проверяет или создаёт ignore-файлы по фактической технологии проекта, выполняет задачи по фазам, уважает `[P]`, требует тесты до кода при TDD и отмечает выполненные задачи как `[X]` в `tasks.md` [implement.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/implement.md). Так появляется проверяемый след выполнения: не только код изменён, но и список задач отражает состояние работ.

`/speckit.taskstoissues` конвертирует сгенерированный список задач в GitHub issues для отслеживания и исполнения [github/spec-kit](https://github.com/github/spec-kit). Эта команда показывает “хвост” методологии: файл задач может покинуть локальное ИИ-рабочее пространство и стать частью командного трекинга.

Шаблон для `/speckit.taskstoissues` особенно осторожен: команда должна получить Git remote через `git config --get remote.origin.url` и продолжать только если remote является GitHub URL; при создании issues через GitHub MCP сервер нельзя создавать issue в репозитории, который не соответствует remote URL [taskstoissues.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/taskstoissues.md). Это маленькая, но важная деталь управления: перенос задач в GitHub issues является внешним действием и требует проверки адресата.

### Рабочие процессы как оркестрация

Справочник Workflows показывает, что Spec Kit умеет автоматизировать многошаговые SDD-последовательности: рабочие процессы связывают команды, prompt-файлы, shell-шаги и человеческие контрольные точки, поддерживают условную логику, циклы, fan-out/fan-in и приостановку/возобновление (`pause`/`resume`) [Workflows](https://github.github.com/spec-kit/reference/workflows.html). Встроенный пример `Full SDD Cycle` запускает `speckit.specify`, затем контрольную точку `review-spec`, затем `speckit.plan`, контрольную точку `review-plan`, затем `speckit.tasks` и `speckit.implement` [Workflows](https://github.github.com/spec-kit/reference/workflows.html).

Состояние рабочего процесса сохраняется в `.specify/workflows/runs/<run_id>/`: `state.json`, `inputs.json`, `log.jsonl` [Workflows](https://github.github.com/spec-kit/reference/workflows.html). Это важная фактура для будущего текста: контекст живёт не только в `specs/`, но и в состоянии запуска рабочего процесса, если используется движок рабочего процесса.

Справочник Workflows добавляет машинно-ориентированную грань: `specify workflow run <source> --json` печатает на stdout только JSON-объект с `run_id`, `workflow_id`, `status`, `current_step_id` и `current_step_index`, а прогресс шага уходит в stderr [Workflows](https://github.github.com/spec-kit/reference/workflows.html). Это делает движок рабочего процесса пригодным не только для человека в терминале, но и для внешнего оркестратора, который должен надёжно понять, где процесс приостановлен, завершился ошибкой или завершён.

Управление каталогами рабочих процессов тоже влияет на происхождение процесса. Поиск рабочего процесса идёт через каталоги, а порядок разрешения такой: `SPECKIT_WORKFLOW_CATALOG_URL`, затем `.specify/workflow-catalogs.yml`, затем `~/.specify/workflow-catalogs.yml`, затем встроенные значения по умолчанию [Workflows](https://github.github.com/spec-kit/reference/workflows.html). Если будущий текст будет разбирать “какой рабочий процесс реально исполнился”, нужно проверять не только YAML в проекте, но и источник каталога, который его поставил.

Во всех базовых командных шаблонах есть общий механизм хуков расширений: команда ищет `.specify/extensions.yml`, читает ключи вида `hooks.before_specify`, `hooks.after_plan`, `hooks.before_implement`, фильтрует `enabled: false`, различает необязательные и обязательные хуки и для обязательного хука выводит `EXECUTE_COMMAND` [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md) [plan.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/plan.md) [implement.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/implement.md). При этом сами шаблоны не интерпретируют непустые `condition`; условие оставлено реализации HookExecutor [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md). Это важная неудобная деталь: расширяемость есть, но её фактическая надёжность зависит от того, как конкретная интеграция исполняет `EXECUTE_COMMAND` и протокола хуков.

Уточнение прохода source-opening: слой рабочего процесса делает Spec Kit ближе к маленькому оркестратору, а не только к командам. `specify workflow run <source>` может запускать рабочий процесс из id каталога, URL или локального YAML-файла; состояние запуска получает `run_id`, `workflow_id`, `status`, `current_step_id`, а `specify workflow resume <run_id>` продолжает приостановленный/упавший запуск с того места, где он остановился [Workflows](https://github.github.com/spec-kit/reference/workflows.html). Встроенный пример `Full SDD Cycle` прямо содержит контрольные точки проверки после `specify` и `plan`: `review-spec` и `review-plan` могут принять или отклонить продолжение (`approve`/`reject`), а отклонение прерывает рабочий процесс. Это показывает, что человеческая контрольная точка в Spec Kit может быть не только дисциплиной пользователя, но и элементом YAML-рабочего процесса.

Ещё одна важная техническая деталь: `--json` у `workflow run` / `resume` печатает машиночитаемый результат на stdout, а прогресс шагов уходит в stderr. Для будущего практического справочника это означает, что рабочие процессы Spec Kit можно встраивать в автоматизацию без разбора насыщенного терминального текста, но нужно отдельно обрабатывать stderr как операционный журнал [Workflows](https://github.github.com/spec-kit/reference/workflows.html).

## Артефакты и файлы

Основные артефакты:

- `.specify/memory/constitution.md` — проектные принципы и правила управления, созданные или обновлённые `/speckit.constitution`; последующие фазы должны учитывать этот файл [github/spec-kit](https://github.com/github/spec-kit).
- `.specify/feature.json` — текущее состояние фичи, куда `/speckit.specify` записывает `feature_name`, `feature_number`, `spec_file` и `feature_dir`; нижестоящие команды могут использовать этот файл как опору вместо угадывания активной ветки [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md).
- `specs/[branch-name]/spec.md` — спецификация фичи с пользовательскими историями, требованиями и критериями приёмки; создаётся `/speckit.specify` [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).
- `specs/[feature]/checklists/requirements.md` — стартовый чек-лист качества требований, который `/speckit.specify` создаёт и обновляет при уточнениях; пункты проверяют качество требований, а не реализацию [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md) [clarify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/clarify.md).
- `specs/[feature]/checklists/*.md` — тематические checklists, созданные `/speckit.checklist`; каждый файл содержит назначение, дату создания, имя фичи, пункты с `CHK###` и ссылки трассируемости или явные пометки `[Gap]`, `[Ambiguity]`, `[Conflict]`, `[Assumption]` [checklist.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/checklist.md).
- `specs/[branch-name]/plan.md` — технический план реализации, создаваемый `/speckit.plan`; содержит стек, архитектурные решения, обоснование и связи с требованиями [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).
- `specs/[branch-name]/research.md` — исследовательские заметки, например сравнения библиотек; в примере chat-системы туда попадают сравнения WebSocket-библиотек [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).
- `specs/[branch-name]/data-model.md` — модели данных, например схемы Message и User в примере chat-фичи [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).
- `specs/[branch-name]/contracts/` — API-контракты, WebSocket-события или REST endpoints [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).
- `specs/[branch-name]/quickstart.md` — ключевые сценарии валидации, генерируемые планированием [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).
- `specs/[branch-name]/tasks.md` — исполняемый список задач, созданный `/speckit.tasks`; может включать `[P]` для задач, пригодных для параллельного выполнения [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).
- `.specify/templates/`, `.specify/scripts/`, `.specify/memory/` — проектные файлы Spec Kit, которые обновляются при `specify init --here --force`; руководство Upgrade Guide отдельно говорит, что `specs/` и исходный код при обновлении не трогаются [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html).
- `.specify/templates/overrides/` — локальные переопределения проекта, которые README ставит выше пресетов, расширений и core-шаблонов в стеке разрешения; они позволяют изменить шаблон или команду для одного проекта без создания пресета [github/spec-kit](https://github.com/github/spec-kit). Для анализа конкретного проекта это один из первых каталогов, где нужно искать расхождение между “core Spec Kit” и реальным поведением.
- `.specify/integration.json` — состояние слоя интеграций: интеграцию по умолчанию, установленные интеграции, настройки интеграции и схема [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html).
- `.specify/extensions.yml` — проектную конфигурацию расширений и привязки хуков. Core командные шаблоны ищут в ней `before_*` и `after_*` хуки, но исполнение зависит от HookExecutor и конкретной интеграции [Extensions Reference](https://github.github.com/spec-kit/reference/extensions.html) [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md).
- `.specify/extension-catalogs.yml` — проектный источник каталога для поиска и установки расширения. Порядок разрешения: `SPECKIT_CATALOG_URL`, проектная конфигурация, пользовательская конфигурация, встроенные значения по умолчанию [Extensions Reference](https://github.github.com/spec-kit/reference/extensions.html). Для происхождения это отдельный файл: он объясняет, откуда в проект пришло расширение, а не только что оно установлено.
- `.specify/extensions/agent-context/agent-context-config.yml` — конфигурация встроенного расширения `agent-context`. В `AGENTS.md` репозитория Spec Kit описано, что сюда записываются `context_file` и `context_markers` вроде `<!-- SPECKIT START -->` / `<!-- SPECKIT END -->`; слой Python и встроенные bash/PowerShell-скрипты читают эти маркеры из единого источника истины [AGENTS.md](https://github.com/github/spec-kit/blob/main/AGENTS.md).
- `.specify/preset.yml` и `.specify/manifest.lock` — локальные следы слоя пресетов: пресеты могут принести шаблоны, команды, хуки, поправки constitution, определения рабочих процессов и ограничения зависимостей; lock-файл фиксирует разрешённые ссылки каталога [Presets Reference](https://github.github.com/spec-kit/reference/presets.html).
- `.specify/workflow-catalogs.yml` — проектный источник каталога рабочих процессов; он стоит ниже `SPECKIT_WORKFLOW_CATALOG_URL`, но выше пользовательская конфигурация и встроенных значений по умолчанию в порядке разрешения [Workflows](https://github.github.com/spec-kit/reference/workflows.html).
- `.specify/workflows/runs/<run_id>/state.json`, `inputs.json`, `log.jsonl` — состояние запуска рабочего процесса для pause/resume [Workflows](https://github.github.com/spec-kit/reference/workflows.html).
- `.specify/integrations/<key>.manifest.json` — манифест конкретной интеграции: id интеграции, версию CLI, timestamp установки и соответствие относительных путей SHA-256-хешам [src/specify_cli/integrations/manifest.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/manifest.py). Это файл происхождения для команд и скриптов конкретного агента.
- `.specify/integrations/speckit.manifest.json` — манифест общей инфраструктуры: `.specify/scripts/` и `.specify/templates/`, которые зависят от core-пакета/source checkout и от разделителя вызова интеграции по умолчанию [src/specify_cli/shared_infra.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/shared_infra.py) [src/specify_cli/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/__init__.py).
- `.specify/integration-catalogs.yml` и `.specify/integrations/.cache/catalog-*.json` — конфигурация и кеш для каталогов интеграций. Порядок источников: `SPECKIT_INTEGRATION_CATALOG_URL`, проектная конфигурация, пользовательская конфигурация, встроенный каталог по умолчанию и community-каталоги; community-каталог по умолчанию используется только для обнаружения [src/specify_cli/integrations/catalog.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/catalog.py).
- `.devin/skills/`, `.goose/recipes/`, `.rovodev/skills/`, `.rovodev/prompts.yml`, `.acli/rovodev/mcp.json`, `~/.hermes/skills/` и пользовательский каталог интеграции `generic` — примеры мест, специфичных для агента, куда Spec Kit может установить команды вне привычной схемы `.agents/skills` или `.claude/skills` [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Эти пути важны для аудита: отсутствие `/speckit.plan` в одном каталоге ещё не значит, что команда не установлена для другой поверхности агента.
- `.specify/scripts/bash/common.sh` и соответствующие PowerShell-варианты — сгенерированный вспомогательный слой, который определяет корень репозитория, активную фичу, проверку веток, разделитель вызова команд и порядок выбора шаблонов [scripts/bash/common.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/common.sh).
- `.specify/scripts/bash/create-new-feature.sh` — скрипт, который создаёт ветку, каталог фичи и стартовый `spec.md`, либо в `--dry-run` только вычисляет эти значения [scripts/bash/create-new-feature.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/create-new-feature.sh).
- `.specify/scripts/bash/setup-plan.sh` — скрипт подготовки `plan.md`; он разрешает активную фичу, проверяет именование ветки, если не действует `.specify/feature.json`, и копирует `plan-template` только если `plan.md` ещё нет [scripts/bash/setup-plan.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/setup-plan.sh).
- `.specify/scripts/bash/setup-tasks.sh` — скрипт подготовки tasks-фазы; он требует существующие `spec.md` и `plan.md`, перечисляет сопроводительные документы и разрешает `tasks-template` через стек переопределений/пресетов/расширений/core [scripts/bash/setup-tasks.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/setup-tasks.sh).
- `.specify/scripts/powershell/common.ps1`, `create-new-feature.ps1`, `setup-plan.ps1`, `setup-tasks.ps1` — Windows/PowerShell-варианты, которые нужно читать отдельно при диагностике Windows-проектов. Важные детали: `-LiteralPath`, проверка имени ветки с удалением gitflow-префикса, сравнение путей без учёта регистра на Windows, `Get-InvokeSeparator` из `.specify/integration.json`, и разрешение шаблонов через переопределения/пресеты/расширения/core [scripts/powershell/common.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/common.ps1) [scripts/powershell/create-new-feature.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/create-new-feature.ps1).
- `src/specify_cli/integrations/base.py` и `src/specify_cli/integrations/__init__.py` — Python-слой реестра интеграции. `IntegrationBase` задаёт `key`, `config`, `registrar_config`, `context_file`, `invoke_separator`, `multi_install_safe`, построение нативного вызова команды и установку командных файлов; `__init__.py` регистрирует встроенные интеграции в `INTEGRATION_REGISTRY` [src/specify_cli/integrations/base.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/base.py) [AGENTS.md в github/spec-kit](https://raw.githubusercontent.com/github/spec-kit/main/AGENTS.md).

Репозиторий `github/spec-kit` физически показывает устройство инструмента: `integrations`, `templates`, `extensions`, `presets`, `workflows`, `src/specify_cli` [github/spec-kit](https://github.com/github/spec-kit). Для будущего текста это полезно как доказательство, что Spec Kit — не только методологическая статья, но CLI, шаблоны, интеграции, расширения и движок рабочего процесса.

Пресеты и расширения добавляют отдельный слой разрешения, который может менять фактический файл, читаемый командой. В пресетах локальные переопределения проекта `.specify/templates/overrides/` имеют наивысший приоритет, затем идут установленные пресеты, установленные расширения и core-шаблоны `.specify/templates/`; разрешение выполняется для каждого файла, а `specify preset resolve <name>` показывает, какой файл победил [Presets](https://github.github.com/spec-kit/reference/presets.html). Это сильная, но потенциально опасная деталь: два проекта с одинаковой командой `/speckit.plan` могут фактически читать разные `plan-template.md`, если у них разные пресеты или overrides.

Расширения живут в `.specify/extensions/<ext>/`, могут иметь проектную конфигурацию, локальные переопределения и переменные окружения; слияние конфигурации идёт от значений расширения по умолчанию к проектной конфигурации, затем локального переопределения и переменных окружения [Extensions](https://github.github.com/spec-kit/reference/extensions.html). Поэтому расширения не стоит описывать просто как “плагины”: они могут менять команды, шаблоны и контрольные точки качества, а также вводить локальные конфигурационные состояния, которые не очевидны по одному Markdown-файлу спецификации.

## Где живёт контекст

Контекст разделён между несколькими слоями.

Первый слой — `specs/[branch-name]/`: спецификация, план, research, data model, contracts, quickstart и tasks. Quick Start отмечает, что команды автоматически определяют активную фичу по текущей Git-ветке, например `001-feature-name`; чтобы переключиться между спецификациями, достаточно переключить ветку [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html). Это делает Git-ветку частью адресации контекста.

Второй проход по шаблонам уточняет эту картину: ветка остаётся важной, но активная фича также хранится в `.specify/feature.json`, а `SPECIFY_FEATURE_DIRECTORY` может явно указать каталог спецификации [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md). Поэтому будущему тексту лучше не описывать Spec Kit как систему, которая всегда “просто берёт текущую ветку”. Реальный механизм допускает хук ветки, переопределение окружения и сохранённое состояние фичи.

Сгенерированный `common.sh` добавляет ещё одну практическую тонкость: он считает `.specify` главным маркером корня проекта Spec Kit, а Git-корень — запасным вариантом [scripts/bash/common.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/common.sh). Если `.specify` лежит внутри большого репозитория, команды должны работать относительно этого вложенного корня. Для будущего текста это хороший пример того, как методология защищает свой локальный контекст от случайного расширения до всего монорепозитория.

PowerShell `common.ps1` сохраняет тот же приоритет `.specify` над Git, но добавляет Windows-специфичные условия контекста. `Test-HasGit` проверяет наличие `.git` как каталог или файл, что важно для worktree/submodule; `Test-FeatureJsonMatchesFeatureDir` нормализует пути и сравнивает их без учёта регистра только на Windows; `Find-FeatureDirByPrefix` останавливается с ошибкой, если один префикс совпал с несколькими каталогами `specs/` [scripts/powershell/common.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/common.ps1). В досье это закрывает часть прежнего открытого вопроса: PowerShell-вариант не просто “похож” на bash, он явно обрабатывает worktree, чувствительность к регистру и неоднозначные совпадения по префикс.

Второй слой — `.specify/memory/constitution.md`. Конституция задаёт долговременные правила проекта: стандарты качества, тестирования, согласованности UX, требования к производительности и управления техническими решениями [github/spec-kit](https://github.com/github/spec-kit). В отличие от спецификации фичи, это не одноразовый документ конкретной задачи.

После чтения `constitution.md` этот слой лучше понимать как живой артефакт управления. Команда `/speckit.constitution` должна не только переписать текст принципов, но и синхронизировать зависимые шаблоны и указания документации, а в `Sync Impact Report` зафиксировать, какие файлы обновлены и что осталось со статусом pending [constitution.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/constitution.md). Если этот отчёт устарел или отсутствует, нижестоящие команды могут ссылаться на старую версию правил даже при свежей constitution.

Третий слой — файлы интеграций и правила контекста, которые Spec Kit устанавливает под конкретного агента. Для Codex это `.agents/skills` и `AGENTS.md`, для Claude Code `.claude/skills` и `CLAUDE.md`, для некоторых других агентов собственные каталоги команд и файлы правил [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Здесь живёт адаптация методологии к поверхности конкретного инструмента.

Метаданные уровня исходного кода показывают, что агентский контекст — не один универсальный файл. `codex` и `claude` используют каталоги skills, но с разными файлами контекста: `AGENTS.md` и `CLAUDE.md`; `cursor-agent` тоже ставит skills в `.cursor/skills`, а управляемый контекст хранит в `.cursor/rules/specify-rules.mdc`; `kiro-cli` кладёт команды в `.kiro/commands/` и опирается на `AGENTS.md`; `windsurf` кладёт команды в `.windsurf/workflows`, а файл контекста держит в `.windsurf/rules/specify-rules.md`; `gemini` смотрит на `GEMINI.md`. Для `pi` в справочнике важнее не деталь пути, а ограничение: Spec Kit не настраивает MCP-сервер автоматически [src/specify_cli/integrations/codex/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/codex/__init__.py) [src/specify_cli/integrations/claude/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/claude/__init__.py) [src/specify_cli/integrations/cursor_agent/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/cursor_agent/__init__.py) [src/specify_cli/integrations/kiro_cli/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/kiro_cli/__init__.py) [src/specify_cli/integrations/windsurf/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/windsurf/__init__.py) [src/specify_cli/integrations/gemini/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/gemini/__init__.py) [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Практический вывод: при переносе проекта между агентами нужно проверять не только `.specify/integration.json`, но и то, какой файл реально получает управляемая секция от `agent-context`.

Актуальный список поддерживаемых интеграций добавляет несколько краевых случаев для вопроса о том, где живёт контекст. `Hermes` хранит skills глобально в пользовательском домашнем каталоге, а в проекте оставляет локальный маркер `.hermes/skills/`; `RovoDev` использует не только каталог skills, но и обёртки/MCP-конфигурацию; `Goose` получает recipes, а не Markdown слеш-команды; `generic` вообще переносит командные файлы в каталог, который пользователь сам передал через `--commands-dir` [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Поэтому будущий аудит Spec Kit-проекта должен искать команды не только внутри репозитория: часть рабочей поверхности может оказаться в домашнем каталоге, в конфигурации конкретного вендора или в произвольном каталоге.

В этом слое важно различать установленную интеграцию и интеграцию по умолчанию. Справочник говорит, что `specify integration use <key>` меняет интеграцию по умолчанию без удаления остальных установленных интеграций и обновляет управляемые общие шаблоны, чтобы ссылки на команды соответствовали стилю вызова новой интеграции по умолчанию; изменённые или неотслеживаемые общие шаблоны сохраняются, если не передан `--force` [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Значит, фактическая prompt-поверхность проекта определяется не только набором папок `.agents/skills`, `.claude/skills` и т.п., но и тем, какая интеграция сейчас используется по умолчанию в `.specify/integration.json`.

`AGENTS.md` самого репозитория Spec Kit уточняет, что обновление файла контекста теперь централизовано через встроенное расширение `agent-context`. Класс интеграции задаёт `context_file`, а управляемая секция Spec Kit создаётся, обновляется и удаляется с маркерами из `.specify/extensions/agent-context/agent-context-config.yml`; пользователь может полностью отключить этот слой командой `specify extension disable agent-context`, после чего Spec Kit пропускает создание, обновление и удаление файла контекста [AGENTS.md](https://github.com/github/spec-kit/blob/main/AGENTS.md). Для будущей теории это важная грань “где живёт контекст”: часть контекста находится не в `specs/` и не в constitution, а в файле инструкций конкретного агента, причём этот слой можно случайно или сознательно выключить.

Четвёртый слой — конфигурация расширений, пресетов и переопределений. `.specify/templates/overrides/` может перекрыть core-шаблон локально, `.specify/extensions.yml` может привязать хуки к фазам, а `.specify/preset.yml` описывает применённые пресеты, конфигурацию и переопределения; справочник по пресетам отдельно показывает объявления зависимостей и хук `preset.updated` как часть жизненного цикла пресета [github/spec-kit](https://github.com/github/spec-kit) [Extensions Reference](https://github.github.com/spec-kit/reference/extensions.html) [Presets Reference](https://github.github.com/spec-kit/reference/presets.html). Этот слой влияет на команды, но не является содержательным документом фичи.

Пятый слой — установочные файлы манифестов и каталоги. `.specify/integrations/*.manifest.json` хранит хеши файлов, созданных Spec Kit, а `.specify/integration-catalogs.yml` может менять источник интеграций вплоть до пользовательского каталога через переменную окружения [src/specify_cli/integrations/manifest.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/manifest.py) [src/specify_cli/integrations/catalog.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/catalog.py). Это контекст сопровождения, а не контекста feature: по нему видно, какие команды считаются управляемыми, какие источники каталогов использовались и какие файлы нельзя тихо перезаписать.

Шестой слой появляется при использовании рабочего процесса: `.specify/workflows/runs/<run_id>/` хранит состояние выполнения, входы и лог шагов [Workflows](https://github.github.com/spec-kit/reference/workflows.html). Это важно для процессов с шагами-гейтами: рабочий процесс может остановиться на человеческом решении и быть продолжен командой `specify workflow resume <run_id>` [Workflows](https://github.github.com/spec-kit/reference/workflows.html).

## Роли, агенты и участники

Базовый Spec Kit не навязывает сложную систему ролей. Его “участники” — человек, выбранный ИИ-агент разработки, Specify CLI и набор команд/шаблонов. Человек задаёт намерение, принципы, уточнения, стек и принимает решения на фазовых переходах; агент генерирует и изменяет артефакты; CLI устанавливает инфраструктуру и управляет интеграциями, рабочими процессами, пресетами и расширениями.

У выбранного агента есть ещё одна роль: он должен корректно исполнять контракт команд своей интеграции. Для интеграции на Markdown-командах это чтение и выполнение файлов `/speckit.*`; для Codex skills-режима — вызов `$speckit-*`; для Kiro CLI — работа с резервными инструкциями, где аргументы не подставляются динамически, а команда получает явные примеры прямого вызова скриптов [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html) [issue #1926](https://github.com/github/spec-kit/issues/1926). Это не продуктовая роль вроде PM/developer, но для методологии она критична: один и тот же `spec.md` может быть создан по-разному, если поверхность агента иначе понимает параметры команды.

Python-слой интеграций делает роль “выбранного агента” технически конкретной. В `AGENTS.md` репозитория Spec Kit сказано, что каждая интеграция — отдельный подпакет под `src/specify_cli/integrations/<agent>/`, а глобальный `INTEGRATION_REGISTRY` является единым источником сведений о каталогах, форматах, возможностях и файлах контекста [AGENTS.md в github/spec-kit](https://raw.githubusercontent.com/github/spec-kit/main/AGENTS.md). `IntegrationBase` дополнительно показывает, что CLI-интеграция должна иметь `key`, совпадающий с исполняемым инструментом, потому что проверки опираются на `shutil.which`, а для нестандартной установки можно задать `SPECKIT_INTEGRATION_<KEY>_EXECUTABLE` [src/specify_cli/integrations/base.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/base.py). Это важная рабочая деталь: “поддержка агента” в Spec Kit — не только набор Markdown-команд, но и контракт реестра для установки, проверки и возможной диспетчеризации.

Отдельные классы интеграций добавляют специфичную для агента упаковку команд. `CodexIntegration` в режиме skills устанавливает команды как skills и меняет разделитель вызова на `-`, из-за чего `speckit.plan` превращается в `$speckit-plan`; при скриптовой диспетчеризации он строит `codex exec "<prompt>"` и добавляет `--json`, если нужен машиночитаемый вывод [src/specify_cli/integrations/codex/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/codex/__init__.py). `ClaudeIntegration` тоже ставит skills, но дополнительно добавляет `argument-hint`, `user-invocable` и `disable-model-invocation` в frontmatter skill; `CursorAgentIntegration` по умолчанию ставит skills в `.cursor/skills`, а не обычные `.md` слеш-команды [src/specify_cli/integrations/claude/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/claude/__init__.py) [src/specify_cli/integrations/cursor_agent/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/cursor_agent/__init__.py). Поэтому человек, проверяющий результат, должен смотреть не только на содержимое `templates/commands/*.md`, но и на постобработку конкретной интеграции: она может менять имя команды, место хранения, frontmatter и способ неинтерактивной диспетчеризации.

Справочник поддерживаемых интеграций показывает, что у некоторых агентов роль “исполнителя Spec Kit-команды” зависит от внешней среды сильнее, чем у классических слеш-команд. `Hermes` работает через глобальный каталог skills, `RovoDev` через `acli rovodev`, MCP-конфигурацию и сгенерированные скрипты-обёртки, `Pi` требует ручной настройки MCP-сервера, а `generic` интеграция перекладывает ответственность за каталог команд на пользователя [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). В методологическом описании это нужно сохранить как границу ответственности: Spec Kit может поставить командные файлы, но не всегда полностью настраивает агентскую среду, которая должна эти файлы прочитать и исполнить.

Исследовательский угол появляется в `spec-driven.md`: там сказано, что исследовательские агенты могут собирать технический контекст, проверять совместимость библиотек, бенчмарки производительности, последствия для безопасности и организационные ограничения [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md). В официальной quickstart-цепочке это не оформлено как отдельная роль пользователя, но идея исследовательского контекста встроена в plan/research-артефакты.

Spec Kit Agents делает роли явными: многоагентный SDD-конвейер с ролями PM и developer, оркестратором на state machine, хуками обнаружения и валидационными хуками [Spec Kit Agents](https://arxiv.org/abs/2604.05278). Этот источник нельзя автоматически переносить на весь Spec Kit как поставляемое поведение по умолчанию, но он показывает направление усиления методологии: отделить роль product/requirements от роли developer и добавить проверяемые контекстные хуки.

## Человеческие подтверждения

В базовом порядке работы человеческие решения стоят между фазами, даже если не всегда оформлены как отдельные экраны подтверждения.

Перед `/speckit.plan` человек должен проверить, что `spec.md` действительно описывает нужное поведение, а неопределённости либо сняты через `/speckit.clarify`, либо сознательно оставлены для исследовательского прототипа [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html). Перед `/speckit.tasks` нужно проверить технический план: выбор стека, архитектуру, ограничения, quickstart-валидация и связь с требованиями [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md). Перед `/speckit.implement` нужно проверить `tasks.md` и результат `/speckit.analyze`, потому что после начала реализации исправлять пробелы дороже [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html).

Командные шаблоны добавляют ещё одну явно машинно-проверяемую контрольную точку. `/speckit.implement` перед стартом собирает статус всех файлов в `FEATURE_DIR/checklists/`; если есть незакрытые пункты, он выводит таблицу со счётчиками и спрашивает пользователя, продолжать ли несмотря на незавершённые чек-листы [implement.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/implement.md). Это не просто рекомендация “проверить требования”: незакрытый чек-лист становится блокирующим prompt перед кодом.

Движок рабочих процессов делает эти решения явными: встроенный `Full SDD Cycle` включает контрольную точку `review-spec` с сообщением “Review the generated spec before planning” и контрольную точку `review-plan` перед фазой задач; при `reject` рабочий процесс прерывается [Workflows](https://github.github.com/spec-kit/reference/workflows.html). Это хороший материал для будущего текста: Spec Kit можно использовать как последовательность команд вручную, но рабочие процессы превращают фазовые решения в формальные остановки.

Руководство Upgrade Guide добавляет ещё один вид человеческого решения: при `specify init --here --force` обновляются командные файлы, скрипты, шаблоны и memory-файлы, а документация отдельно предупреждает сохранить или закоммитить `.specify/memory/constitution.md`, если он был кастомизирован [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html). Значит, сопровождение Spec Kit-проекта требует проверки не только артефактов фичи, но и самой методологической инфраструктуры.

`/speckit.constitution` добавляет управленческое решение в явной форме: если тип повышения версии неоднозначен, шаблон требует предложить обоснование перед финализацией, а при неизвестной `RATIFICATION_DATE` разрешает оставить `TODO()` с объяснением и включить его в `Sync Impact Report` [constitution.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/constitution.md). Это показывает, что человек принимает решение не только о фиче, но и о статусе проектных правил: это уточнение, новое правило или несовместимая смена принципа.

Слой расширений и пресетов расширяют список человеческих решений. Справочник Extensions Reference говорит, что команды расширений регистрируются в текущей интеграции ИИ-агента разработки, но если команда не появилась, агент может требовать перезапуск [Extensions Reference](https://github.github.com/spec-kit/reference/extensions.html). Справочник Presets Reference даёт `specify preset resolve <name>` для просмотра, какой файл реально победил в стеке разрешения [Presets Reference](https://github.github.com/spec-kit/reference/presets.html). В рабочих терминах это значит: перед важным запуском человеку может понадобиться проверить не только `spec.md`, но и то, какие шаблоны/команды реально используются после пресетов, переопределений и расширений.

Жизненный цикл интеграций добавляет ещё одну человеческую контрольную точку. При `specify integration install <key>` Spec Kit автоматически продолжает только если уже установленная и новая интеграция объявлены безопасными для совместной установки; иначе нужно сознательно выбрать `switch` или передать `--force` [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). При `switch` в уже установленную интеграцию параметр `--integration-options` отклоняется, потому что изменение параметров требует переустановки управляемых файлов; рекомендованный путь — сначала `upgrade <key> --integration-options ...`, затем `use <key>` [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Это маленькое, но важное место решения: человек выбирает между переносимостью команды, чистотой инструкций конкретного агента и риском перезаписи управляемых файлов.

## Проверка результата

Проверка в Spec Kit многоуровневая.

На уровне требований `/speckit.checklist` валидирует полноту, ясность и согласованность требований; `spec-driven.md` описывает checklists как “unit tests” для спецификации: нет ли `[NEEDS CLARIFICATION]`, тестируемы ли требования, измеримы ли критерии успеха [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html) [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).

Командные шаблоны дают более конкретную лестницу проверки требований. `/speckit.specify` создаёт стартовый `requirements.md` и оценивает готовность файла спецификации; `/speckit.clarify` после каждого принятого ответа обновляет статусы пунктов чек-листа; `/speckit.checklist` создаёт дополнительные доменные чек-листы со ссылками трассируемости и issue-маркерами [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md) [clarify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/clarify.md) [checklist.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/checklist.md). Поэтому чек-листы в Spec Kit — не один финальный отчёт, а серия файлов, которые сопровождают спецификацию до реализации.

На уровне плана `spec-driven.md` описывает соответствие constitution: план должен проверять соответствие проектной constitution и архитектурным принципам [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md). В этом же файле есть более жёсткий пример constitutional foundation с контрольными точками вроде Simplicity Gate, Anti-Abstraction Gate и Integration-First Gate; важно воспринимать его как методологический пример из `spec-driven.md`, а не обязательно как неизменный набор статей для каждого современного проекта [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).

На уровне согласованности `/speckit.analyze` проверяет согласованность и покрытие спецификации, плана и задач перед реализацией [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html). Это главный официальный механизм против рассинхронизации артефактов до реализации.

Шаблон `/speckit.analyze` полезен именно как аудит без записи: он строит карту покрытия, помечает требования без задач, задачи без требований, расплывчатые прилагательные, недостающие критерии приёмки, placeholder-маркеры и конфликты с constitution; при этом он не исправляет документы без отдельного разрешения [analyze.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/analyze.md). В досье это стоит хранить как пример правильного разделения: сначала диагностика, затем отдельная правка.

На уровне репозиторных свидетельств Spec Kit Agents предлагает хуки обнаружения и валидационные хуки: хуки обнаружения собирают релевантные файлы, соглашения, зависимости и историю до фаз, валидация проверяет промежуточные артефакты, а после реализации — проектные проверки вроде тестов и линтеров, где они применимы [Spec Kit Agents](https://arxiv.org/abs/2604.05278). Это не заменяет официальный quickstart, но хорошо показывает, где базовому Spec Kit может понадобиться усиление для больших репозиториев.

На уровне установленной среды `specify check` проверяет наличие `git` и CLI-based ИИ-агентов разработки, работает без сети и отдельно советует `specify self check`, если локальная команда ведёт себя как старая версия или нужная CLI-возможность отсутствует [Core Commands](https://github.github.com/spec-kit/reference/core.html). `specify version --features --json` предназначен для скриптов и агентов разработки, которым нужно выбрать рабочий процесс по возможностям установленного CLI [Core Commands](https://github.github.com/spec-kit/reference/core.html). Это отдельный вид проверки: перед методологическим процессом нужно понять, что локальная CLI-поверхность действительно поддерживает ожидаемые команды.

На уровне слоя интеграций проверка начинается с `specify integration list`: он показывает текущую интеграцию/интеграцию по умолчанию, остальные установленные интеграции, необходимость внешнего CLI-инструмента или IDE-режим, а также безопасность совместной установки [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). После обновления CLI сам факт новой версии недостаточен: `specify integration upgrade [<key>]` обновляет установленные файлы интеграций, блокируется на локально изменённых файлах без `--force`, удаляет устаревшие файлы и держит общие шаблоны синхронизированными с интеграцией по умолчанию [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Поэтому проверка установленной среды должна включать и версию CLI, и состояние управляемых файлов интеграций.

Для современных поддерживаемых интеграций проверка должна учитывать тип поверхности. `specify integration list` полезен как общий вход, но для `generic` дополнительно нужен фактический `--commands-dir`, для `Hermes` нужно проверить глобальный `~/.hermes/skills/`, для `RovoDev` — сгенерированные обёртки, `.rovodev/prompts.yml` и MCP-конфигурацию, для `Goose` — recipes, а для `Pi` — ручную настройку MCP-сервера, которую CLI не делает автоматически [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Это не косметическая деталь: команда может считаться установленной, но конкретная поверхность агента всё ещё может не видеть свежие команды или исполнять их через другой механизм запуска.

На уровне сгенерированных скриптов есть дополнительные технические проверки. `check_feature_branch` принимает последовательные имена веток `001-feature-name` и timestamp-имена `YYYYMMDD-HHMMSS-feature-name`, но отклоняет ветку без распознаваемого префикса, если проект находится в Git-репозитории [scripts/bash/common.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/common.sh). `setup-tasks.sh` дополнительно останавливает фазу tasks, если отсутствуют `spec.md`, `plan.md` или обязательный `tasks-template` [scripts/bash/setup-tasks.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/setup-tasks.sh). Эти проверки полезны не как гарантия качества требований, а как защита от запуска фаз в неверном каталоге или в неполной инфраструктуре.

PowerShell-слой добавляет проверки, которые стоит отдельно учитывать в Windows-инструкции. `Get-FeaturePathsEnv` возвращает набор путей `FEATURE_DIR`, `FEATURE_SPEC`, `IMPL_PLAN`, `TASKS`, `RESEARCH`, `DATA_MODEL`, `QUICKSTART`, `CONTRACTS_DIR`; `Get-InvokeSeparator` читает `.specify/integration.json`, чтобы форматировать команду как `/speckit.plan` или `/speckit-plan` в зависимости от настроек интеграции [scripts/powershell/common.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/common.ps1). Ещё одна проверка скрыта в композиции шаблонов: `Resolve-TemplateContent` пытается найти Python 3 и PyYAML для чтения манифесты пресетов; если Python или PyYAML отсутствуют, стратегии композиции могут быть проигнорированы с предупреждением [scripts/powershell/common.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/common.ps1). Поэтому “файлы на месте” не всегда означает “используется тот же стек шаблонов”, особенно в проектах с пресетами.

## Хвост жизненного цикла

Spec Kit не заканчивается на `/speckit.implement`.

Во-первых, `tasks.md` может быть превращён в GitHub issues через `/speckit.taskstoissues` [github/spec-kit](https://github.com/github/spec-kit). Это делает результат планирования пригодным для командного отслеживания и распределения.

Во-вторых, руководство Upgrade Guide показывает сопровождение самой методологической среды: CLI можно обновлять через `specify self upgrade`, проектные файлы — через `specify init --here --force`, а `specs/` при этом считается безопасной пользовательской зоной, которую пакеты шаблонов не трогают [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html).

Руководство Upgrade Guide делает это сопровождение довольно конкретным. `specify self check` без записи проверяет, есть ли новый релиз; `specify self upgrade --dry-run` показывает, что будет запущено; голая команда `specify self upgrade` выполняется сразу и автоматически определяет `uv tool` или `pipx`, а для `uvx`, checkout исходного кода и неподдерживаемой среды выполнения печатает указания без изменения файлов [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html). Для закреплённого обновления тег должен начинаться с `vMAJOR.MINOR.PATCH`; имена веток, hash refs, `latest` и bare versions без `v` отклоняются [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html). Это полезная деталь для автоматизации: обновление Spec Kit имеет собственную проверяемую семантику, а не сводится к “переустановите пакет”.

Обновление проектных файлов имеет отдельный риск. `specify init --here --force` обновляет слеш-командные файлы, `.specify/scripts/`, `.specify/templates/` и `.specify/memory/`, но документация прямо предупреждает, что `.specify/memory/constitution.md` сейчас может быть перезаписан шаблоном по умолчанию, если его заранее не сохранить или не восстановить из Git [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html). То есть `specs/` защищён, а файл проектного управления — нет; это важная граница безопасности обновления.

В-третьих, рабочие процессы поддерживают приостановку/возобновление (`pause`/`resume`) и состояние запуска, поэтому долгий процесс можно продолжать с того места, где он остановился на контрольной точке или ошибке [Workflows](https://github.github.com/spec-kit/reference/workflows.html).

В-четвёртых, расширения и пресеты имеют собственный жизненный цикл. Расширения можно искать, ставить, обновлять, отключать, менять приоритет и каталог; разрешение каталогов идёт через `SPECKIT_CATALOG_URL`, `.specify/extension-catalogs.yml`, пользовательскую конфигурацию и встроенные значения по умолчанию, а слияние конфигурации идёт от значений по умолчанию к проектной конфигурации, локальным переопределениям и переменным окружения `SPECKIT_<EXT>_*` [Extensions Reference](https://github.github.com/spec-kit/reference/extensions.html). Пресеты можно складывать по приоритету, а разрешение файлов идёт через локальные переопределения проекта, установленные пресеты, установленные расширения и базовые шаблоны; `specify preset resolve <name>` показывает, какой файл фактически выбран [Presets Reference](https://github.github.com/spec-kit/reference/presets.html). Это делает жизненный цикл Spec Kit-проекта похожим на сопровождение маленькой методологической платформы, а не только набора документов.

В-пятых, интеграции имеют безопасный путь удаления, переключения (`switch`) и обновления (`upgrade`). Справочник говорит, что Spec Kit отслеживает каждый файл, созданный при установке, с исходным SHA-256-хешем: неизменённые файлы удаляются автоматически, изменённые файлы сохраняются, а `--force` нужен для удаления или перезаписи изменённых файлов [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). При `integration upgrade` команда блокируется на локально изменённых файлах без `--force`, устаревшие файлы удаляются, а общие шаблоны остаются синхронизированными с интеграцией по умолчанию [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Это важный “хвост” метода: перенос между агентами и обновление команд не должны тихо уничтожать локальные правки.

Детали исходного кода делают этот хвост ещё строже. Инсталлятор общей инфраструктуры различает три режима: обычный `install` добавляет только отсутствующие файлы, ``refresh_managed=True`` перезаписывает только те файлы, чей текущий хеш на диске всё ещё совпадает с прежним хешем из манифеста, а `force=True` перезаписывает обычные файлы независимо от хеша; symlink-файлы и родительские symlink-каталоги не перезаписываются, потому что могут уводить запись за пределы корня проекта [src/specify_cli/shared_infra.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/shared_infra.py). При записи манифеста используется временный файл и ``os.replace``, а пути валидируются как относительные к корню проекта [src/specify_cli/integrations/manifest.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/manifest.py). Для практической методологии это означает, что “обновить Spec Kit” — это операция с безопасностью на уровне diff, а не простое копирование свежих шаблонов.

В-шестых, `agent-context` устанавливается как встроенное расширение при `specify init`, если пользователь не передал `--ignore-agent-tools`; слой обновления и справочника отдельно показывает, что проблемы после обновления часто лечатся не повторным `specify`, а перезапуском IDE/редактора и проверкой специфичных для агента каталогов [Installation Guide](https://github.github.com/spec-kit/installation.html) [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html). Для жизненного цикла это означает: после обновления методологической инфраструктуры нужно проверить не только файлы на диске, но и то, что конкретный агент перечитал новые команды/skills/контекст.

Открытый хвост: официальная документация хорошо описывает путь до реализации, но слабее фиксирует обязательное согласование после реализации. В issue #1063 пользователь формулирует отсутствие первоклассного способа согласовать пробелы после реализации после `specify -> plan -> tasks -> implement`, чтобы вносить изменения обратно в `spec.md`, `plan.md`, `tasks.md` [issue #1063](https://github.com/github/spec-kit/issues/1063). Этот issue нельзя использовать как официальный факт о продуктовой дорожной карте, но он точно показывает практический риск: код может уйти вперёд, а спецификационные артефакты снова начнут отставать.

## Сильные стороны

Spec Kit даёт явную цепочку передачи контекста. Вместо одного большого prompt появляются `spec.md`, `plan.md`, сопроводительные документы и `tasks.md`; каждый следующий шаг обязан читать предыдущие артефакты [документация Spec Kit](https://github.github.com/spec-kit/) [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).

Методика хорошо материализована. Есть CLI, интеграции, шаблоны, скрипты, пресеты, расширения и рабочие процессы, а не только статья о SDD [github/spec-kit](https://github.com/github/spec-kit) [CLI Reference](https://github.github.com/spec-kit/reference/overview.html). Это облегчает перенос практики между инструментами и командами.

Spec Kit явно отделяет продуктовую спецификацию от технического планирования. `/speckit.specify` фокусируется на what/why, `/speckit.plan` принимает технический стек и архитектурные решения [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html). Это снижает риск преждевременного выбора реализации.

Методика встроенно поддерживает контрольные точки качества: clarify, checklist, analyze, контрольные точки рабочих процессов, соответствие constitution [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html) [Workflows](https://github.github.com/spec-kit/reference/workflows.html).

Интеграционная модель делает Spec Kit переносимым: можно работать с Copilot, Codex, Claude, Gemini и другими агентами, а интеграция `generic` оставляет аварийный выход для инструментов вне списка [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html).

## Сбои и риски чрезмерного применения

Первый риск — документарная имитация прогресса. Если человек быстро прогоняет `/speckit.specify`, `/speckit.plan`, `/speckit.tasks` и не проверяет содержание, получается несколько убедительных файлов, но не обязательно более правильная разработка. Quick Start фактически предупреждает об этом через регулярные контрольные точки `/speckit.clarify`, `/speckit.checklist`, `/speckit.analyze` для задач с значимой неоднозначностью [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html).

Второй риск — слепота к контексту. Spec Kit Agents прямо говорит, что ИИ-агенты разработки в больших развивающихся репозиториях могут галлюцинировать API и нарушать архитектуру из-за отсутствия текущего контекста репозитория; структура SDD сама по себе не гарантирует привязку к контексту [Spec Kit Agents](https://arxiv.org/abs/2604.05278). Для будущей главы важно не обещать, что `spec.md` автоматически решает проблему знания репозитория.

Третий риск — дрейфа после реализации. Если после `/speckit.implement` код меняется вручную, а `spec.md`, `plan.md`, `tasks.md` не обновляются, метод возвращается к старой проблеме отставшей документации. Issue #1063 формулирует это как пробелы после реализации и дрейф артефактов [issue #1063](https://github.com/github/spec-kit/issues/1063).

Четвёртый риск — перегрузка проверки. Обзор Birgitta Böckeler на Martin Fowler/Thoughtworks подчёркивает, что термин SDD ещё семантически расплывчат, а развёрнутые подходы с множеством файлов могут усилить нагрузку проверки и дать ложное чувство контроля [Understanding Spec-Driven-Development](https://martinfowler.com/articles/exploring-gen-ai/sdd-3-tools.html). Для досье это не первоисточник Spec Kit, но полезный внешний критический источник.

Пятый риск — слепая вера в community-расширения, пресеты и рабочие процессы. Документация по рабочим процессам прямо говорит, что сопровождающие не проверяют, не аудитят, не одобряют и не поддерживают код рабочих процессов; перед установкой нужно проверять исходник и использовать его на свой риск [Workflows](https://github.github.com/spec-kit/reference/workflows.html). Документация по расширениям и пресетам показывает богатую экосистему, но это не равняется гарантированному качеству каждого пакета [Community Extensions](https://github.github.com/spec-kit/community/extensions.html) [Community Presets](https://github.github.com/spec-kit/community/presets.html).

У этого риска есть техническая форма: расширения и пресеты могут менять команды, шаблоны, скрипты и поведение хуков, а приоритет разрешения решает, какой файл победит [Extensions Reference](https://github.github.com/spec-kit/reference/extensions.html) [Presets Reference](https://github.github.com/spec-kit/reference/presets.html). Если команда не проверяет `specify preset resolve <name>`, `.specify/extensions.yml` и статус установленных расширений, она может думать, что работает по базовому Spec Kit, хотя фактически использует локальное переопределение или сторонний пакет.

Шестой риск — маленькие изменения. Thoughtworks-обзор делит SDD на `spec-first`, `spec-anchored` и `spec-as-source` и задаёт вопрос, не слишком ли тяжёлы некоторые рабочие процессы для большинства реальных задач [Understanding Spec-Driven-Development](https://martinfowler.com/articles/exploring-gen-ai/sdd-3-tools.html). Для Spec Kit это означает: метод особенно полезен там, где неоднозначность, цена ошибки и потребность в трассируемости достаточно велики; для маленького исправления вся цепочка может быть чрезмерной.

Седьмой риск — ложная уверенность в хуках. Базовые командные шаблоны умеют обнаруживать определения хуков и выводить `EXECUTE_COMMAND`, но не выполняют произвольные условия сами; для условий нужен HookExecutor [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md). Если конкретная агентская интеграция плохо выполняет этот протокол, обязательные хуки могут стать текстовой инструкцией, а не реальным барьером. Для больших репозиториев это особенно важно: контекстные проверки должны быть проверяемо выполнены, а не просто упомянуты в prompt.

Восьмой риск — скрытое расхождение между ожидаемым базовым рабочим процессом и локальной установкой. На это работают сразу несколько механизмов: `.specify/templates/overrides/`, пресеты, расширения, каталоги рабочих процессов, интеграция по умолчанию и версия CLI [github/spec-kit](https://github.com/github/spec-kit) [Core Commands](https://github.github.com/spec-kit/reference/core.html) [Workflows](https://github.github.com/spec-kit/reference/workflows.html). Если анализировать проект только по `spec.md` и `plan.md`, можно пропустить, что реальная команда `/speckit.plan` уже переопределена пресетом, рабочий процесс пришёл из пользовательского каталога, а локальный `specify` не поддерживает новую возможность фичи.

Девятый риск — ошибка установки из-за совпадающих имён пакетов. Installation Guide прямо предупреждает, что PyPI-пакеты с похожими или теми же именами, например `specify-cli`, не являются официальными пакетами Spec Kit [Installation Guide](https://github.github.com/spec-kit/installation.html). Для практического справочника это практическая проверка: перед разбором странного поведения нужно посмотреть `specify version`, путь исполняемого файла и команду установки, а не считать любой `specify-cli` на машине тем самым Spec Kit.

Десятый риск — тихое отсутствие слоя агентского контекста. Если расширение `agent-context` отключено или его `agent-context-config.yml` указывает не тот `context_file`, команды могут существовать, но агент не получит актуальные инструкции уровня проекта между маркерами `SPECKIT START` / `SPECKIT END` [AGENTS.md](https://github.com/github/spec-kit/blob/main/AGENTS.md). Это не ломает сами `spec.md` и `plan.md`, но ослабляет перенос методологии в конкретную поверхность агента.

Одиннадцатый риск — неправильная активная фича из-за сложного резервного порядка. Если в проекте есть `.specify/feature.json`, `SPECIFY_FEATURE_DIRECTORY`, текущая Git-ветка и несколько каталогов `specs/` с похожими числовыми префиксами, нужно понимать, какой слой победил [scripts/bash/common.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/common.sh). Особенно неприятен случай, когда вывод команды выглядит нормальным, но `FEATURE_DIR` указывает не на тот каталог. Для диагностики нужны `setup-plan.sh --json`, `setup-tasks.sh --json`, проверка `.specify/feature.json` и текущих переменных окружения.

Двенадцатый риск — агент неправильно вызывает сгенерированные скрипты. GitHub issue #987 описывает повторяющийся случай, когда разные агенты неверно вызывали `create-new-feature.sh` с `--short-name` и пытались исправляться после чтения синтаксиса скрипта; PR #1001 был смёржен 22 октября 2025 года как правка prompt/example для этой проблемы [issue #987](https://github.com/github/spec-kit/issues/987) [PR #1001](https://github.com/github/spec-kit/pull/1001). Это слабый источник и исторический баг, но он показывает важную методологическую деталь: если агент должен исполнять shell-скрипты, prompt-шаблоны должны давать не только смысл фазы, но и точную форму команды.

Тринадцатый риск — расхождение вариантов скриптов. GitHub issue #975 показывает, что PowerShell `create-new-feature.ps1` в версии `v0.0.20` мог выбирать номер фичи только по `specs/` текущей ветки и игнорировать существующие имена веток, что приводило к повторному `001-*` при работе с несмёрженными ветками фич; текущий скрипт уже содержит проверку веток и refs, но сам факт такого сбоя показывает, что совпадение поведения bash/PowerShell нужно проверять по текущим сгенерированным файлам, а не предполагать [issue #975](https://github.com/github/spec-kit/issues/975) [scripts/powershell/create-new-feature.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/create-new-feature.ps1).

Четырнадцатый риск — стек шаблонов зависит от локальной Python-среды. PowerShell `Resolve-TemplateContent` пытается прочитать манифесты пресетов через Python и PyYAML, чтобы применить `prepend`, `append` или `wrap`; при отсутствии Python 3 или PyYAML он предупреждает, что стратегии композиции могут быть проигнорированы [scripts/powershell/common.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/common.ps1). Для команды это неприятный класс сбоя: пресет вроде бы установлен, но фактическая команда/шаблон может быть ближе к поведению замены или резервного варианта (`replace`/`fallback`), чем к ожидаемой композиции.

Отдельный риск — переносимость интеграции может быть переоценена. Страница Supported Integrations перечисляет много агентов, но рядом с ними есть разные ограничения: Kiro CLI требует резервных инструкций вместо обычной подстановки аргументов, Pi требует отдельной ручной настройки MCP-сервера, а часть интеграций помечена как особый способ запуска или путь миграции [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Для будущей теории это граница обещания “use any coding agent”: Spec Kit действительно выносит процесс в общие артефакты, но поверхность агента остаётся частью процесса и может менять качество исполнения.

Пятнадцатый риск — рассинхронизация CLI и проектных файлов. Upgrade Guide подчёркивает, что CLI-инструмент отделён от проектных файлов: обновление `specify` даёт новые возможности CLI, но слеш-команды, скрипты и шаблоны в проекте обновляются отдельно через `specify init --here --force --integration <agent>` [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html). Если обновить только одно из двух, агент может читать старые командные шаблоны, а человек — смотреть новую документацию и ожидать поведения, которого в локальных файлах ещё нет.

Шестнадцатый риск — повреждённый или устаревший слой манифеста. Если `.specify/integrations/<key>.manifest.json` отсутствует, нечитаем или создан старой версией, Spec Kit может относиться к уже существующим общим файлам как к неотслеживаемым или кастомизированным (`untracked`/`customized`) и не обновлять их без `--force` [src/specify_cli/shared_infra.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/shared_infra.py). Это лучше, чем тихо перезаписать пользовательские правки, но для человека выглядит как “обновление ничего не поменяло”. Диагностика должна смотреть не только на версию CLI, но и на файлы манифестов, предупреждения о пропущенных, сохранённых и symlink-путях, а также фактический статус с отслеживанием хешей сгенерированных файлов.

Семнадцатый риск — `--force` безопасен не для всех слоёв одинаково. Руководство Upgrade Guide прямо говорит, что `specs/` и исходный код не входят в пакеты шаблонов и не перезаписываются, но `.specify/memory/constitution.md`, `.specify/scripts/` и `.specify/templates/` могут быть перезаписаны при обновлении проектных файлов [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html). Поэтому перед обновлением нужно сохранять кастомизированную constitution, шаблоны и скрипты; иначе Spec Kit может потерять именно те локальные правила, ради которых его внедряли.

Восемнадцатый риск — скрытый дрейф в CLI-dispatch конкретного агента. Если `SPECKIT_INTEGRATION_<KEY>_EXECUTABLE` или `SPECKIT_INTEGRATION_<KEY>_EXTRA_ARGS` заданы локально, `specify` может запускать не тот исполняемый файл или не тот режим агента, который ожидает команда; для ключей вроде `cursor-agent` и `kiro-cli` имя переменной ещё и нормализуется через подчёркивания [src/specify_cli/integrations/base.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/base.py). Это особенно опасно в CI и скриптовых рабочих процессах: внешне интеграция установлена, командные файлы свежие, но фактическое исполнение идёт через изменённую командную строку агента.

Девятнадцатый риск — смешение “поддерживаемой интеграции” с “готово к совместной установке”. Reference отдельно перечисляет интеграции, безопасные для совместной установки, и предупреждает, что не все поддерживаемые агенты безопасно ставить рядом без `--force` [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Если команда механически добавляет новый агент в уже настроенный проект, она может получить конфликт файлов контекста, путей команд или общих настроек; если применить `--force`, риск смещается в сторону перезаписи управляемых файлов. Для практики это значит, что перед экспериментом с новой интеграцией нужен короткий аудит: текущая интеграция по умолчанию, установленные интеграции, признак безопасности, файлы манифестов и локальные правки.

Двадцатый риск — глобальные или внешние командные поверхности хуже видны в обычном diff. Если интеграция вроде `Hermes` использует `~/.hermes/skills/`, а `generic` пишет команды в произвольный каталог, ревью репозитория может не показать полный набор инструкций, которые реально читает агент [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Это особенно важно для командной работы и воспроизводимости: два разработчика могут иметь одинаковые `specs/` и `.specify/`, но разные глобальные skills или разные `--commands-dir`.

Проход source-opening добавил отдельный риск обновления/переопределения. Руководство Upgrade Guide прямо предупреждает, что `specify init --here --force` сейчас может перезаписать `.specify/memory/constitution.md`; обходной путь — заранее сохранить constitution и восстановить после обновления. Тот же guide отдельно предупреждает, что кастомизированные скрипты/шаблоны в `.specify/scripts/` и `.specify/templates/` могут быть перезаписаны, а у IDE-агентов после обновления могут появиться дублирующиеся слеш-команды, которые нужно удалить вручную [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html). Для методологии это важно: Spec Kit делает процесс воспроизводимым через файлы, но эти же файлы становятся изменяемой инфраструктурой, которую можно случайно сбросить при обновлении.

Ещё один риск связан с пресетами/расширениями: чем больше поведение процесса переносится в каталоги, приоритеты и локальные переопределения, тем сложнее понять, какой именно процесс исполнил агент. Для аудита недостаточно сохранить `spec.md` и `plan.md`; нужно сохранять версию Spec Kit, интеграцию, активные пресеты/расширения, порядок приоритетов и локальные переопределения проекта.

## Аналитические заметки предыдущих проходов

### Что должно попасть в теорию

В теории Spec Kit стоит описывать не как “пример хорошего prompt”, а как спецификационный конвейер, где каждый Markdown-артефакт становится входом следующего шага: constitution, spec, clarify/checklist, plan, tasks, analyze, implement.

Нужно показать разделение контекста: `.specify/memory/constitution.md` для долговременных принципов, `specs/[branch-name]/` для артефактов фичи, файлы интеграций для конкретного агента, состояние запуска рабочего процесса для автоматизированной оркестрации.

Нужно уточнить, что в текущих шаблонах активная фича определяется не только Git-веткой. В теоретическом тексте стоит показать связку хука ветки `before_specify`, `SPECIFY_FEATURE_DIRECTORY`, `.specify/feature.json` и `specs/[feature]/` как несколько уровней адресации контекста фичи [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md).

Нужно сохранить различие между поставляемым Spec Kit и исследовательским Spec Kit Agents. Первый даёт набор инструментов и рабочий процесс; второй показывает усиление через роли PM/developer, хуки обнаружения и валидационные хуки. Их нельзя склеивать в один “официальный режим”, но вместе они хорошо показывают направление развития SDD.

Нужно честно написать про человеческие контрольные точки. Spec Kit не снимает необходимость проверки; он делает проверку предметной: сначала спецификация, потом план, потом задачи и согласованность, потом реализация.

Нужно включить ограничение: без привязки к репозиторию и согласование после реализации спецификационные документы могут стать красивой, но устаревшей поверхностью.

Нужно отдельно описать слой расширений/пресетов как силу и риск. Сила в том, что команды, хуки, шаблоны и поправки constitution можно упаковывать и переносить; риск в том, что базовый Spec Kit может быть переопределён сторонним или локальным пакетом, и тогда фактический процесс надо проверять через установленные расширения, `.specify/extensions.yml`, `.specify/preset.yml` и `specify preset resolve` [Extensions Reference](https://github.github.com/spec-kit/reference/extensions.html) [Presets Reference](https://github.github.com/spec-kit/reference/presets.html).

Нужно добавить жизненный цикл constitution как отдельный элемент теории. `/speckit.constitution` важен не только тем, что создаёт `.specify/memory/constitution.md`, но и тем, что требует semver-управления, `Sync Impact Report` и проверки зависимых шаблонов; это делает правила проекта частью исполняемой методологической инфраструктуры [constitution.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/constitution.md).

Нужно аккуратно показать операционную интроспекцию: `specify check`, `specify version --features --json`, рабочий процесс `--json`, статус рабочего процесса и `resume`. Spec Kit интересен тем, что часть процесса можно проверять и оркестрировать машинно, но это требует читать состояние CLI, состояние запуска рабочего процесса и разрешение каталогов, а не только содержательные Markdown-файлы [Core Commands](https://github.github.com/spec-kit/reference/core.html) [Workflows](https://github.github.com/spec-kit/reference/workflows.html).

Нужно показать жизненный цикл самого инструмента как часть методологии. Spec Kit-проект живёт не только через спецификации фич: CLI обновляется отдельно, проектные файлы обновляются отдельно, интеграции имеют `install`/`use`/`switch`/`upgrade`/`uninstall`, а управляемые файлы защищаются SHA-256-манифестами [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html) [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html). Без этого теория будет выглядеть как статичная цепочка документов, хотя на практике это сопровождаемая инфраструктура.

Нужно точнее сформулировать обещание “Use any coding agent”. Теоретически Spec Kit переносит spec/plan/tasks и командные шаблоны между агентами; практически каждая интеграция имеет свою поверхность: слеш-команду, skill, recipe, wrapper, настройку MCP или пользовательский каталог команд [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). В теории это лучше показать как слой адаптеров, а не как полную прозрачность агента.

Нужно добавить сгенерированные скрипты как отдельный слой методологии. `common.sh`, `create-new-feature.sh`, `setup-plan.sh` и `setup-tasks.sh` показывают, что Spec Kit — это не только prompt-файлы команд для агента: под ними есть разрешение исполняемых путей, проверка веток, порядок выбора шаблонов и JSON-выводы [scripts/bash/common.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/common.sh) [scripts/bash/create-new-feature.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/create-new-feature.sh) [scripts/bash/setup-plan.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/setup-plan.sh) [scripts/bash/setup-tasks.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/setup-tasks.sh). Для теории это важно как материальный мост между Markdown-методологией и реальным репозиторием.

Нужно уточнить, что этот слой работает на нескольких платформах. PowerShell-варианты сохраняют общий Spec Kit-порядок, но вводят отдельные условия: сравнение путей без учёта регистра на Windows, обработку gitflow-префикса ветки, `.ps1`-флаги и зависимость некоторых путей композиции пресетов от Python/PyYAML [scripts/powershell/common.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/common.ps1). В теории это помогает не сводить Spec Kit к Unix-ориентированной CLI-истории.

### Что лучше уйдёт в практический справочник / полевой справочник

В практический справочник подойдут практические рецепты:

- как инициализировать проект: `uvx --from git+https://github.com/github/spec-kit.git specify init <PROJECT_NAME>` или `specify init .`;
- когда выбирать `--script ps`, `--no-git`, `--branch-numbering timestamp`, `--preset <id>` и когда задавать `SPECIFY_FEATURE` для режима без Git;
- как выбрать интеграцию, в том числе Codex skills-режим через `--integration codex --integration-options="--skills"`;
- минимальная цепочка для эксперимента и расширенная цепочка для продуктовой фичи;
- как читать `Sync Impact Report` в `.specify/memory/constitution.md` после `/speckit.constitution`;
- как читать `spec.md`, `plan.md`, `tasks.md` перед переходом к следующей фазе;
- как использовать `/speckit.analyze` до реализации;
- как читать `FEATURE_DIR/checklists/` перед `/speckit.implement`, потому что незакрытые пункты создают отдельную контрольную точку;
- когда запускать `/speckit.taskstoissues`;
- как обновлять CLI и проектные файлы, не потеряв кастомизированную constitution.
- как проверять фактические шаблоны/команды после установки расширения или пресеты через `specify extension list`, `.specify/extensions.yml`, `.specify/preset.yml` и `specify preset resolve <name>`.
- как использовать `specify check`, `specify version --features --json`, `specify workflow status` и `specify workflow resume` для проверки установленной среды и продолжения рабочего процесса.
- как проверять слой интеграций через `specify integration list`, когда использовать `integration use`, `integration switch`, `integration upgrade`, и почему `uvx` не обновляет постоянный `specify` на `PATH`.
- как отличать поддерживаемую интеграцию от интеграции, безопасной для совместной установки и почему перед `install`/`switch` нужно смотреть интеграцию по умолчанию, установленные интеграции, манифесты и локальные изменения.
- как диагностировать нестандартные поверхности: `Hermes` глобальные skills, `Goose` recipes, `RovoDev` обёртки/MCP-конфигурацию, `Pi` настройку MCP и `generic --commands-dir`.
- как читать примечания по конкретной интеграции: формат команд, режим skills, резервные разделы, способ запуска в среде агента, предварительные требования MCP и migration notes для выбранного агента.
- как проверять метаданные уровня исходного кода интеграции: `key`, `context_file`, каталог вывода команд, `invoke_separator`, `multi_install_safe`, frontmatter, companion prompt-файлы и переопределения окружения `SPECKIT_INTEGRATION_<KEY>_EXECUTABLE` / `SPECKIT_INTEGRATION_<KEY>_EXTRA_ARGS`.
- как безопасно обновлять Spec Kit: сначала `specify self check` / `specify self upgrade --dry-run`, затем обновление CLI, затем `specify init --here --force --integration <agent>`, с отдельным сохранением кастомизированного `constitution.md`, `.specify/templates/` и `.specify/scripts/`.
- как читать `.specify/integrations/<key>.manifest.json` и `.specify/integrations/speckit.manifest.json`: отличать управляемые файлы от кастомизированных/неотслеживаемых файлов, понимать предупреждения о пропущенных, сохранённых и symlink-путях и решать, когда нужен `--force`.
- как проверять каталоги интеграций: `SPECKIT_INTEGRATION_CATALOG_URL`, `.specify/integration-catalogs.yml`, пользовательскую конфигурацию, `.specify/integrations/.cache/`, статус `install_allowed` и предупреждение о каталогах не по умолчанию.
- как запускать `create-new-feature.sh --dry-run --json`, `setup-plan.sh --json` и `setup-tasks.sh --json`, чтобы увидеть реальные `FEATURE_DIR`, `FEATURE_SPEC`, `IMPL_PLAN`, `TASKS_TEMPLATE` и сопроводительные документы до того, как агент начнёт менять содержательные файлы.
- как на Windows запускать `.specify/scripts/powershell/create-new-feature.ps1 -DryRun -Json`, `.specify/scripts/powershell/setup-plan.ps1 -Json` и `.specify/scripts/powershell/setup-tasks.ps1 -Json`, а также как проверять `specify version`, Python 3 и PyYAML, если в проекте используются пресеты с композиционными стратегиями.

В полевой справочник подойдут сценарии:

- фича с высокой неоднозначностью, где `/speckit.clarify` экономит переделку;
- большой репозиторий, где нужно пробное чтение репозитория без записи перед планом;
- команда с несколькими агентами, где Spec Kit интеграции помогают удержать общий процесс;
- случай дрейфа после реализации, где нужно вручную согласовать спецификацию, план и задачи;
- маленькая исправительная задача, где полный Spec Kit-процесс оказывается тяжёлым.
- проект, где пресет или расширение неожиданно меняет поведение командного шаблона, и команда должна восстановить, какой шаблон реально исполнялся.
- проект без Git, где активную фичу приходится задавать через `SPECIFY_FEATURE`, а не через ветку.
- обновление или переключение интеграции, где изменённые файлы агента должны сохраниться, а устаревшие управляемые файлы удалиться.
- обновление, где CLI обновили, но проектные командные шаблоны остались старыми, и агент продолжает работать по прежним файлам `/speckit.*`.
- обновление проектных файлов, где `specs/` сохранились, но кастомизированный `constitution.md` или шаблоны были перезаписаны из-за `--force`.
- проект внутри монорепозитория, где `.specify` должен быть корнем Spec Kit-процесса, а не родительский Git-корень.
- сбой, где агент неправильно вызывает сгенерированный скрипт; разбор должен начинаться со справки/вывода скрипта и текущего командного шаблона, а не с общей идеи SDD.
- Windows-проект, где PowerShell-скрипт выбирает не тот номер фичи, не тот `FEATURE_DIR` или не применяет ожидаемую композицию пресета; разбор должен сравнить вывод `.ps1`, `.specify/feature.json`, Git-ветки/удалённые refs, Python/PyYAML и текущие файлы в `.specify/presets/`.
- проект, где интеграция выглядит установленной, но фактическая поверхность агента требует особого поведения: режим Codex skills, резервные инструкции Kiro CLI, настройку Pi MCP или миграции с прежнего формата команд агента.
- проект, где интеграция запускается через переопределённый исполняемый файл или дополнительные аргументы, и команда должна проверить переменные окружения перед тем, как объяснять сбой содержанием `spec.md` или `plan.md`.

## Кандидаты на иллюстрации

- `https://github.github.com/spec-kit/` — первый экран документации и блоки `Spec-driven by default`, `Use any coding agent`, `Make it your own`. Может пригодиться для иллюстрации позиционирования Spec Kit как переносимого набора инструментов. Статус: кандидат; не использовать без проверки прав.
- `https://github.com/github/spec-kit` — дерево репозитория с `integrations`, `templates`, `extensions`, `presets`, `workflows`, `src/specify_cli`. Может показать материальную структуру методологии. Статус: кандидат; лучше пересобрать собственную схему вместо скриншота.
- `https://github.github.com/spec-kit/reference/workflows.html` — встроенная схема Mermaid/flowchart `Full SDD Cycle`: `specify -> review-spec -> plan -> review-plan -> tasks -> implement`. Может пригодиться для раздела о человеческих контрольных точках. Статус: кандидат; лучше перерисовать схему своими средствами.
- `https://github.github.com/spec-kit/reference/workflows.html` — пример JSON-статуса для приостановленного рабочего процесса (`run_id`, `workflow_id`, `status`, `current_step_id`). Может пригодиться как иллюстрация машинно-оркестрированного SDD, где человеческая контрольная точка выражена не только текстом, но и состоянием запуска. Статус: кандидат для собственной схемы.
- `templates/commands/*.md` из `github/spec-kit` — можно перерисовать собственную схему “командные шаблоны как контракт рабочего процесса”: хуки -> тело команды -> артефакты -> хуки после выполнения. Статус: кандидат для собственной схемы, не скриншот.
- `templates/commands/constitution.md` — можно перерисовать схему “обновление constitution -> решение по semver -> зависимые шаблоны -> Sync Impact Report”. Статус: кандидат для собственной схемы.
- `scripts/bash/common.sh` + `setup-plan.sh` + `setup-tasks.sh` — можно перерисовать схему “разрешение активной фичи”: корень `.specify` -> `SPECIFY_FEATURE_DIRECTORY` -> `.specify/feature.json` -> поиск по префиксу ветки -> сгенерированные пути. Статус: кандидат для собственной схемы.
- `scripts/powershell/common.ps1` — можно перерисовать вариант той же схемы для Windows: корень `.specify` -> проверка Git worktree -> `SPECIFY_FEATURE_DIRECTORY` -> `.specify/feature.json` -> поиск по префиксу ветки -> сравнение путей без учёта регистра на Windows -> форматированная команда `/speckit.*` или `/speckit-*`. Статус: кандидат для собственной схемы.
- `src/specify_cli/integrations/base.py` + `AGENTS.md` из `github/spec-kit` — можно перерисовать схему “контракт реестра интеграции”: `IntegrationBase.key/config/registrar_config/context_file` -> каталог вывода команд -> отслеживание манифеста -> управляемый файл контекста. Статус: кандидат для собственной схемы.
- `https://github.github.com/spec-kit/reference/integrations.html` — можно перерисовать схему жизненного цикла управления интеграциями: `install` -> SHA-256 manifest -> `use`/`switch` -> общие шаблоны согласованы с интеграцией по умолчанию -> `upgrade` -> устаревшие файлы удаляются / изменённые файлы сохраняются. Статус: кандидат для собственной схемы.
- `https://github.github.com/spec-kit/upgrade.html` — можно перерисовать схему “двухслойное обновление”: постоянный CLI (`specify self upgrade`) отдельно от проектных файлов (`specify init --here --force`), с безопасной зоной `specs/` и рискованной зоной `.specify/memory/constitution.md`. Статус: кандидат для собственной схемы.
- `AGENTS.md` / `agent-context-config.yml` из `github/spec-kit` — можно перерисовать схему “класс интеграции -> конфигурация агентского контекста -> управляемая секция в AGENTS.md/CLAUDE.md/etc.”. Статус: кандидат для собственной схемы, не скриншот.
- `src/specify_cli/integrations/__init__.py` + несколько `src/specify_cli/integrations/<agent>/__init__.py` — можно сделать таблицу “ключ интеграции -> каталог команд -> файл контекста -> стиль вызова -> специальная постобработка”. Это полезнее скриншота, потому что показывает, где именно Spec Kit перестаёт быть универсальным и становится специфичным для агента. Статус: кандидат для собственной схемы.
- `https://github.github.com/spec-kit/reference/integrations.html` — можно сделать отдельную схему “поддерживается != безопасно для совместной установки”: таблица поддерживаемых интеграций -> командная поверхность (`slash`, `skill`, `recipe`, `wrapper`, `generic dir`) -> флаг безопасности -> примечания по внешней настройке. Это поможет не смешивать маркетинговое обещание переносимости с фактической интеграционной топологией. Статус: кандидат для собственной схемы.
- `https://github.github.com/spec-kit/reference/presets.html` — схема или таблица стека разрешения из справочника пресетов. Может пригодиться для объяснения, почему фактический Spec Kit-процесс зависит от переопределений. Статус: кандидат; лучше пересобрать как свою диаграмму.
- `https://martinfowler.com/articles/exploring-gen-ai/sdd-3-tools.html` — диаграмма уровней SDD (`spec-first`, `spec-anchored`, `spec-as-source`) и обзор рабочего пространства spec-driven. Уже есть локальный ресурс `content/assets/theory-images/fowler-sdd-overview.png`; можно использовать для контекста SDD, если права и атрибуция сохранены. Статус: кандидат / уже локально присутствует в корпусе.
- `https://arxiv.org/abs/2604.05278` — статья со схемой по Spec Kit Agents слой привязки к контексту. Может пригодиться для различения базового Spec Kit и исследовательского усиления через хуки. Статус: кандидат; нужно проверить PDF и права перед использованием.

## Открытые вопросы

- Насколько текущий Spec Kit CLI действительно поддерживает все команды, перечисленные в README и документации, для каждой интеграции; GitHub issues показывают, что в отдельных интеграциях возможны расхождения.
- Как в реальных проектах поддерживается актуальность `spec.md`, `plan.md`, `tasks.md` после `/speckit.implement` и ручных правок кода.
- Насколько надёжно разные интеграции выполняют обязательные хуки из командных шаблонов: как реальная поверхность агента обрабатывает `EXECUTE_COMMAND`, `WAIT_FOR_COMPLETION` и условия HookExecutor.
- Является ли `constitution` в современном Spec Kit настолько неизменяемой (`immutable`), как описывает `spec-driven.md`, или это скорее историко-методологический пример из текста эпохи Specify.
- Какие расширения, пресеты и рабочие процессы можно считать достаточно зрелыми для теоретического текста, а какие лучше оставить как иллюстрации экосистемы.
- Как именно `.specify/feature.json`, текущая Git-ветка и `SPECIFY_FEATURE_DIRECTORY` согласуются в конфликтных случаях: какой механизм считать авторитетным для будущего практического описания.
- Как соотносится `SPECIFY_FEATURE` из справочника Core Commands с `SPECIFY_FEATURE_DIRECTORY` из командного шаблона: это разные уровни переопределения или следы разных версий механизма активной фичи.
- Как в будущей практической инструкции лучше проверять совпадение поведения bash и PowerShell-скриптов после обновления Spec Kit: сравнивать только документированные флаги недостаточно, потому что `common.ps1` содержит отдельные ветки для Windows-путей и композиции шаблонов.
- Должен ли будущий практический справочник рекомендовать обязательную предварительную проверку `--json` для `setup-plan.sh` и `setup-tasks.sh` перед запуском агента в больших репозиториях.
- Как часто в реальных проектах `Sync Impact Report` после `/speckit.constitution` действительно синхронизирует зависимые шаблоны, а не остаётся формальным комментарием.
- Насколько изменение поведения `git`-расширения по умолчанию в `v0.10.0` повлияет на описания быстрого старта, которые пока предполагают Git-ветку как стандартный носитель контекста фичи.
- Как в реальных проектах проверять, что `agent-context` расширение не отключена и что управляемая секция в `AGENTS.md`, `CLAUDE.md` или другом файле контекста действительно свежая.
- Насколько часто ошибки установки связаны с неофициальными PyPI-пакетами или временным `uvx` вместо постоянного `uv tool install`, и нужна ли для практического справочника отдельная диагностическая карточка по `specify version` / `specify self check`.
- Насколько глубоко будущему практическому справочнику нужно объяснять слой манифеста: показывать только `.specify/integrations/<key>.manifest.json` как диагностический файл или разбирать записи SHA-256, безопасность символических ссылок (`symlink`) и группы preserved/skipped.
- Как часто обновления проектных файлов реально перезаписывают кастомизированный `constitution.md`, и стоит ли в практическом справочнике сделать обязательным коммит перед обновлением именно для `.specify/memory/constitution.md`, `.specify/templates/` и `.specify/scripts/`.
- Как отличать Spec Kit как spec-first/spec-anchored рабочий процесс от более радикального spec-as-source подхода; внешний обзор Thoughtworks считает границы SDD расплывчатыми.
- Насколько будущий практический справочник должен углубляться в оговорки по конкретным агентам: достаточно общей команды `specify integration list` или нужны отдельные диагностические карточки для Codex skills, резервных инструкций Kiro, Pi MCP и других нестандартных поверхностей.
- Нужно ли в практической диагностике всегда выводить переопределения окружения `SPECKIT_INTEGRATION_*_EXECUTABLE` и `SPECKIT_INTEGRATION_*_EXTRA_ARGS`, или достаточно проверять их только в CI/скриптовых рабочих процессах и при странном поведении вызова агента.
- Нужно ли следующему проходу читать файлы уровня исходного кода для `devin`, `hermes`, `goose`, `rovodev`, `generic` и `pi`, или справочных сведений достаточно для методологического досье.
- Как документировать интеграции с глобальными или внешними командными поверхностями так, чтобы изменения репозитория оставались проверяемым артефактом процесса.
- Какие права и атрибуция нужны для использования диаграмм из официальной документации, Martin Fowler/Thoughtworks и статьи arXiv.
