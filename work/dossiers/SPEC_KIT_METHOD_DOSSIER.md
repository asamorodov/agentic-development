# Spec Kit

## Черновое назначение

Это рабочее методологическое досье по Spec Kit. Его задача — сохранить фактуру источников для будущего текста сайта: порядок работы, команды, файлы, артефакты, места человеческого решения, проверки, ограничения и неудобные детали. Досье не является финальной главой и не должно сглаживать Spec Kit до общей формулы “сначала спецификация, потом код”.

## Краткая рабочая карта

Spec Kit в этом досье лучше читать не как один CLI и не как набор slash-команд, а как несколько слоёв одного SDD-процесса. Нижний слой — стандартная цепочка `specify → plan → tasks → implement`, где каждый шаг производит Markdown-артефакт для следующего шага. Над ней находятся integrations, которые высаживают этот процесс в разные AI coding agents. Ещё выше появляются workflows, presets и extensions: workflows делают процесс возобновляемой многошаговой оркестрацией с gates; presets меняют шаблоны, команды и терминологию; extensions добавляют domain-specific commands, external tool integrations и quality gates [Workflows](https://github.github.com/spec-kit/reference/workflows.html), [Presets](https://github.github.com/spec-kit/reference/presets.html), [Extensions](https://github.github.com/spec-kit/reference/extensions.html).

Эта карта важна для будущей теории: Spec Kit уже не сводится к «спецификация перед кодом». Его текущая поверхность показывает, как spec-first процесс превращается в настраиваемую инфраструктуру разработки: с каталогами, приоритетами resolution, human gates, возобновляемыми runs и риском supply-chain / override confusion.

## Роль в AI-driven SDLC

Spec Kit позиционирует себя как toolkit для Spec-Driven Development: разработка начинается с описания того, что нужно построить, проходит через структурированные фазы и только затем передаётся AI coding agent для реализации [документация Spec Kit](https://github.github.com/spec-kit/). В терминах будущей теории это соседний спецификационный режим: он переносит центр тяжести из одноразового запроса в цепочку версионируемых Markdown-артефактов.

Официальная документация формулирует базовую цепочку как `Spec -> Plan -> Tasks -> Implement`: каждая фаза создаёт Markdown-артефакт, который становится входом для следующей фазы [документация Spec Kit](https://github.github.com/spec-kit/). Это важнее, чем сам факт наличия CLI. Методология заставляет намерение пройти через несколько документов: сначала поведение и границы, затем технические решения, затем исполнимые задачи, затем реализация.

Spec Kit не является отдельной IDE. Он устанавливает командные файлы, context rules и структуру каталогов для выбранного агента через `specify init` и интеграции [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Поэтому его лучше понимать как переносимый слой процесса поверх Copilot, Codex, Claude, Gemini, Kiro, Windsurf и других инструментов, а не как одного исполнителя.

## Проблема, которую решает

Главная проблема в источниках описана как разрыв между намерением, документацией и кодом. В `spec-driven.md` Spec Kit утверждает, что традиционная разработка делает код источником истины, а спецификации быстро отстают; SDD пытается перевернуть это отношение: спецификация и план реализации должны управлять кодом, а не обслуживать его задним числом [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).

В практическом AI-контексте проблема ещё острее: агент склонен быстро переходить к коду, подставлять правдоподобные детали и терять связь с исходным намерением. Spec Kit отвечает на это шаблонами, фазами и проверками. В `spec-driven.md` отдельно подчёркнуто, что шаблоны должны сдерживать модель: спецификация должна держаться на `WHAT` и `WHY`, а детали `HOW` уходят в план [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).

Статья Spec Kit Agents добавляет исследовательский угол: даже структурированный SDD не решает проблему “context blindness” в больших репозиториях. Авторы предлагают phase-level context-grounding hooks: перед фазами `Specify`, `Plan`, `Tasks`, `Implement` выполняются read-only probing hooks, а validation hooks проверяют промежуточные артефакты и среду [Spec Kit Agents](https://arxiv.org/abs/2604.05278). Для досье это важная оговорка: сама цепочка документов полезна, но без привязки к реальному репозиторию агент может всё равно строить план на устаревших или выдуманных предпосылках.

## Workflow

### Установка и инициализация

Быстрый старт показывает запуск `uvx --from git+https://github.com/github/spec-kit.git specify init <PROJECT_NAME>` или инициализацию текущего каталога через `specify init .` [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html). Для постоянной установки документация также указывает `pipx install git+https://github.com/github/spec-kit.git`; README репозитория показывает вариант `uv tool install specify-cli --from git+https://github.com/github/spec-kit.git@vX.Y.Z` [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html) [github/spec-kit](https://github.com/github/spec-kit).

Installation Guide добавляет важную оговорку по цепочке поставки: официальные поддерживаемые пакеты Spec Kit идут из репозитория `github/spec-kit`; одноимённые пакеты на PyPI, включая `specify-cli`, не аффилированы с проектом и не поддерживаются maintainers [Installation Guide](https://github.github.com/spec-kit/installation.html). Поэтому практическая инструкция по Spec Kit должна фиксировать источник установки, а не просто писать “установите `specify-cli`”. Там же указаны prerequisites: Linux/macOS или Windows с PowerShell scripts без WSL, `uv` или `pipx`, Python 3.11+ и Git [Installation Guide](https://github.github.com/spec-kit/installation.html).

При инициализации integration тоже есть вариант по умолчанию, который может быть неожиданным. В интерактивном терминале CLI предлагает выбрать coding agent integration, а в неинтерактивных запусках вроде CI или piped runs по умолчанию выбирается GitHub Copilot, если не передать `--integration` [Installation Guide](https://github.github.com/spec-kit/installation.html). Это маленькая, но важная деталь для автоматизации: запуск `specify init` без явного `--integration codex|claude|gemini|...` может поставить не тот набор команд и context rules.

Core reference добавляет параметры, которые меняют не содержание методологии, а её техническую посадку в проект: `--script sh|ps` выбирает тип служебных scripts, `--no-git` отключает инициализацию Git, `--branch-numbering sequential|timestamp` меняет стратегию нумерации feature branches, а `--preset <id>` позволяет установить preset уже на этапе `specify init` [Core Commands](https://github.github.com/spec-kit/reference/core.html). Там же есть неудобная деталь про default: `git` extension пока включён при `specify init` по умолчанию, но начиная с `v0.10.0` должен требовать явного opt-in через `specify extension add git` [Core Commands](https://github.github.com/spec-kit/reference/core.html). Для будущего практического текста это значит, что команды и branch-поведение нельзя описывать как вечную константу: локальная версия CLI и init-options имеют значение.

Инициализация выбирает integration. Например, документация даёт `specify init my-project --integration copilot`, а для Codex README показывает `specify init . --integration codex --integration-options="--skills"` [документация Spec Kit](https://github.github.com/spec-kit/) [github/spec-kit](https://github.com/github/spec-kit). Для Codex CLI интеграция skills-based: skills устанавливаются в `.agents/skills`, а команды вызываются как `$speckit-<command>`; большинство других агентов используют slash-команды вида `/speckit.*` [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html) [github/spec-kit](https://github.com/github/spec-kit).

Integration reference показывает, что “поддерживаемый агент” в Spec Kit не означает одинаковое поведение всех поверхностей. Codex CLI может ставиться в skills-based mode, Kiro IDE получает Markdown-команды, а Kiro CLI из-за отсутствия dynamic argument substitution получает fallback sections с прямыми вызовами scripts; Pi поддерживается через Markdown-команды, но документация отдельно предупреждает, что CLI не настраивает MCP server внутри Pi [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Поэтому практическая инструкция должна проверять не только имя integration, но и её тип, формат команд, способ запуска в среде агента и примечания по конкретному агенту.

Текущая таблица supported integrations делает это различие ещё шире. В ней рядом стоят IDE-based agents, CLI-based agents, skills-based integrations и специальные форматы: `goose` использует YAML recipes в `.goose/recipes/`, `devin` устанавливает skills в `.devin/skills/` и вызывает их как `/speckit-<command>`, `hermes` ставит skills глобально в `~/.hermes/skills/`, `rovodev` генерирует `.rovodev/skills/`, wrappers и `prompts.yml` с запуском через `acli rovodev`, а `generic` требует явный `--integration-options="--commands-dir <path>"` для неподдержанного агента [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Это важная фактура для методологии: Spec Kit обещает переносимый SDD-процесс, но переносимость достигается не одинаковой командной поверхностью, а набором адаптеров с разной степенью изоляции и разными способами обнаружения команд.

Реестр Python делает эту неоднородность ещё заметнее. В `src/specify_cli/integrations/__init__.py` зарегистрированы десятки built-in integrations, причём имена subpackages иногда отличаются от пользовательских ключей: например, `kiro_cli` регистрирует ключ `kiro-cli`, а `cursor_agent` — `cursor-agent` [src/specify_cli/integrations/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/__init__.py). Это важно для диагностики: каталог исходного кода, ключ integration, имя CLI-binary и имя папки, куда ставятся команды, могут быть разными сущностями.

Документация по интеграциям подчёркивает деталь, которую легко потерять: в проекте есть default integration, но возможна установка нескольких интеграций, если они multi-install safe или пользователь явно берёт риск через `--force` [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Это превращает Spec Kit в командный слой, где один и тот же набор спецификационных артефактов может обслуживаться разными агентами, но shared templates всё равно выравниваются под default integration.

При этом “supported” и “multi-install safe” — разные статусы. Reference считает безопасными только integrations с изолированными agent directories, отдельным context file, стабильными command invocation settings и отдельным install manifest; в safe-список входят, например, `codex`, `claude`, `cursor-agent`, `gemini`, `qwen`, `kimi`, `windsurf`, `trae`, `tabnine`, `junie`, `roo`, `shai`, `kilocode`, `qodercli`, `iflow`, `codebuddy` и `auggie`, но туда не попадают integrations с общими context files, динамическими путями установки или общими настройками инструмента [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Для будущего Handbook это означает, что команду `specify integration install <key>` нельзя трактовать как безусловно безопасное добавление “ещё одного агента”: сначала нужно смотреть multi-install safety и понимать, какие файлы будут видны какому агенту.

Reference по integrations показывает, что этот слой управляется отдельным lifecycle, а не только initial `specify init`: `specify integration list` показывает доступные integrations, текущую default integration, installed integrations, CLI/IDE-type и multi-install safety; `install`, `use`, `switch`, `upgrade` и `uninstall` меняют состояние проекта после инициализации [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Важная operational-деталь: controlled multi-install support появился в Spec Kit `0.8.5`, а одноразовый запуск через `uvx --from git+https://github.com/github/spec-kit.git specify ...` использует временную копию только для этой команды и не обновляет persistent `specify` на `PATH` [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Поэтому странное поведение integrations надо проверять не только по документации, но и по локальной версии CLI.

Исходный код Python уточняет, что у integration layer есть свой проверяемый журнал установки. `IntegrationManifest.manifest_path` пишет файл `.specify/integrations/<key>.manifest.json`, а для shared scripts/templates используется отдельный `speckit.manifest.json` [src/specify_cli/integrations/manifest.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/manifest.py) [src/specify_cli/shared_infra.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/shared_infra.py). В manifest попадают относительные пути и SHA-256-хеши установленных файлов; при uninstall файл удаляется только если текущий хеш совпадает с записанным, а изменённые файлы остаются на месте без `force` [src/specify_cli/integrations/manifest.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/manifest.py). Это делает integration lifecycle не просто удобной командой, а механизмом сохранения пользовательских правок.

`IntegrationBase` показывает отдельный слой запуска команд через CLI агента. Для CLI-based integrations бинарник можно переопределить переменной вида `SPECKIT_INTEGRATION_<KEY>_EXECUTABLE`, а дополнительные флаги передать через `SPECKIT_INTEGRATION_<KEY>_EXTRA_ARGS`; для ключей с дефисом дефис заменяется подчёркиванием, например `kiro-cli` превращается в `SPECKIT_INTEGRATION_KIRO_CLI_EXTRA_ARGS` [src/specify_cli/integrations/base.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/base.py). Поэтому “запустить `/speckit.plan`” в автоматизированном workflow означает не только наличие command file, но и конкретную командную строку агента, модель, формат вывода, режим streaming/capture и локальные env overrides.

При работе без Git есть отдельная переменная `SPECIFY_FEATURE`: core reference описывает её как override feature detection для non-Git repositories; её нужно задать в контексте агента до `/speckit.plan` и последующих команд [Core Commands](https://github.github.com/spec-kit/reference/core.html). Это уточняет границу Git-centric описания Spec Kit: Git branch остаётся стандартной опорой, но методология допускает явную адресацию feature directory без ветки.

Фактические generated bash scripts делают эту картину ещё более конкретной. `common.sh` ищет корень проекта сначала по `.specify`, а не по Git; Git используется как fallback только если `.specify` не найден [scripts/bash/common.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/common.sh). Это важная защита для монорепозиториев и вложенных проектов: Spec Kit пытается не “уплыть” в родительский Git repository, если сам toolkit инициализирован глубже.

В `common.sh` active feature определяется несколькими слоями: `SPECIFY_FEATURE_DIRECTORY` имеет приоритет как явный override, затем читается `.specify/feature.json` с ключом `feature_directory`, и только после этого используется branch-name-based prefix lookup в `specs/` [scripts/bash/common.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/common.sh). Отдельная переменная `SPECIFY_FEATURE` влияет на `get_current_branch`, особенно в non-Git режиме; если Git отсутствует, script пытается выбрать последний каталог feature по sequential или timestamp prefix и только затем падает к `main` [scripts/bash/common.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/common.sh). Поэтому в практическом описании нельзя сводить active feature к одной ветке: реальная адресация состоит из env overrides, сохранённого state и fallback-поиска по `specs/`.

`create-new-feature.sh` показывает, что `/speckit.specify` опирается на отдельный script с собственными гарантиями: он поддерживает `--json`, `--dry-run`, `--allow-existing-branch`, `--short-name`, `--number` и `--timestamp`; sequential numbering берёт максимум из Git branches, remote refs и каталогов `specs/`, а timestamp mode создаёт prefix вида `YYYYMMDD-HHMMSS` [scripts/bash/create-new-feature.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/create-new-feature.sh). В обычном режиме script создаёт branch, каталог feature и `spec.md` из `spec-template`, а при `--dry-run` вычисляет имена и пути без создания ветки, каталогов и файлов [scripts/bash/create-new-feature.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/create-new-feature.sh). Это место полезно для Handbook: перед реальным запуском можно dry-run проверить, какие branch/spec paths получатся.

PowerShell-вариант `create-new-feature.ps1` сохраняет те же пользовательские флаги: `-Json`, `-DryRun`, `-AllowExistingBranch`, `-ShortName`, `-Number`, `-Timestamp`, `-Help` [scripts/powershell/create-new-feature.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/create-new-feature.ps1). Для Windows-проектов это важно не как косметический порт bash-скрипта, а как отдельная исполнимая поверхность с собственными правилами путей, `Get-ChildItem`, `Resolve-Path` и вызова `git`. Текущий `create-new-feature.ps1` уже считает порядковый номер не только по `specs/`, но и по локальным веткам и remote refs, а в режиме без побочных изменений использует `git ls-remote` вместо fetch [scripts/powershell/create-new-feature.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/create-new-feature.ps1). Исторически именно здесь был сбой: issue #975 описывал, что старый PowerShell script видел только `specs/` на текущей ветке и мог повторно создать `001-*`, если существующая feature branch ещё не была влита в `main`; PR #986 позже включил исправление [issue #975](https://github.com/github/spec-kit/issues/975) [PR #986](https://github.com/github/spec-kit/pull/986).

### Рекомендуемая цепочка

Quick Start различает быстрый путь для экспериментов и более строгий путь для production features. Лёгкая цепочка: `/speckit.specify -> /speckit.plan -> /speckit.tasks -> /speckit.implement`. Для задач с существенной неоднозначностью документация рекомендует регулярные quality gates: `/speckit.constitution -> /speckit.specify -> /speckit.clarify -> /speckit.checklist -> /speckit.plan -> /speckit.tasks -> /speckit.analyze -> /speckit.implement` [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html).

`/speckit.constitution` создаёт или обновляет управляющие принципы проекта. Quick Start предлагает передать конкретные принципы прямо аргументом команды, например library-first, strict TDD и предпочтение functional programming patterns [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html). В README уточняется, что этот шаг создаёт или обновляет `.specify/memory/constitution.md`, к которому coding agent обращается на следующих фазах [github/spec-kit](https://github.com/github/spec-kit).

Текущий template `/speckit.constitution` показывает, что constitution command работает не как свободное редактирование одного файла. Он должен загрузить `.specify/memory/constitution.md`, заполнить placeholder tokens, определить `RATIFICATION_DATE`, `LAST_AMENDED_DATE` и bump `CONSTITUTION_VERSION` по semver: `MAJOR` для несовместимых governance changes, `MINOR` для новых или существенно расширенных принципов, `PATCH` для уточнений и несемантических правок [constitution.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/constitution.md). После этого команда обязана сверить dependent artifacts: `.specify/templates/plan-template.md`, `spec-template.md`, `tasks-template.md`, command files и runtime guidance docs, а в начало constitution добавить `Sync Impact Report` с version change, изменёнными принципами, updated/pending templates и TODO [constitution.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/constitution.md). Это сильная деталь для методологии: долгосрочные правила проекта должны проталкиваться в шаблоны и команды, а не просто лежать отдельным manifesto.

`/speckit.specify` описывает, что и зачем нужно построить, без выбора стека. В `spec-driven.md` команда описана как механизм, который автоматически нумерует feature, создаёт branch, копирует шаблон спецификации и создаёт структуру `specs/[branch-name]/` [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md). Это не просто генерация текста: команда связывает спецификацию с веткой и каталогом feature.

Текущий command template уточняет это устройство: создание или переключение Git branch вынесено в `before_specify` hook, а сам core command всегда создаёт feature directory и `spec.md` [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md). Каталог спецификации больше нельзя понимать только как производную от branch name: команда может брать `SPECIFY_FEATURE_DIRECTORY`, иначе генерирует каталог под `specs/` по стратегии из `.specify/init-options.json` (`sequential` или `timestamp`) и сохраняет результат в `.specify/feature.json` [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md). Это важная практическая деталь: downstream-команды должны находить активную feature через сохранённый feature directory, а не полагаться только на имя ветки.

`/speckit.specify` сам создаёт `checklists/requirements.md` и валидирует начальную спецификацию против критериев качества: отсутствие implementation details, фокус на user value, завершённые обязательные секции, testable requirements, measurable success criteria, edge cases, bounded scope, dependencies and assumptions [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md). Если остаются `[NEEDS CLARIFICATION]`, команда ограничивает их числом трёх, приоритизирует scope, security/privacy и UX и задаёт пользователю вопросы с вариантами [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md). Это показывает, что первая фаза уже содержит встроенный quality loop, а не только первичную генерацию `spec.md`.

`/speckit.clarify` предназначена для снятия неоднозначности до планирования. Quick Start рекомендует использовать её перед `/speckit.plan`; ответы записываются в Clarifications section, а если пользователь сознательно пропускает уточнение для spike или prototype, это нужно явно сказать, чтобы агент не блокировался на недостающих уточнениях [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html) [github/spec-kit](https://github.com/github/spec-kit).

Command template для `/speckit.clarify` делает эту фазу более строгой, чем обычный чат с вопросами. Команда сканирует текущий `spec.md` по таксономии функционального scope, доменных данных, UX flow, non-functional qualities, integrations, edge cases, constraints, terminology и completion signals; затем задаёт максимум пять вопросов, причём ровно по одному за раз [clarify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/clarify.md). Принятый ответ сразу записывается в `## Clarifications` / `### Session YYYY-MM-DD`, встраивается в подходящие разделы спецификации, после чего при наличии `checklists/requirements.md` команда пересчитывает чекбоксы требований [clarify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/clarify.md). Для будущего текста это важная механика: уточнение не остаётся в диалоге, а возвращается в `spec.md` как версионируемый источник.

`/speckit.checklist` создаёт quality checklist для проверки полноты, ясности и согласованности требований. Quick Start помещает эту команду перед техническим планом, а README называет её optional command для validation requirements quality [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html) [github/spec-kit](https://github.com/github/spec-kit).

В шаблоне `/speckit.checklist` чек-листы названы “unit tests for requirements writing”: они проверяют качество текста требований, а не работоспособность реализации [checklist.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/checklist.md). Команда создаёт или дописывает тематические файлы в `FEATURE_DIR/checklists/`, например `ux.md`, `api.md`, `security.md`; элементы получают идентификаторы `CHK###`, а минимум 80% пунктов должны иметь traceability reference на раздел spec или явную пометку `[Gap]`, `[Ambiguity]`, `[Conflict]`, `[Assumption]` [checklist.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/checklist.md). Это удобно для теории: Spec Kit проверяет не “кнопка работает”, а “требование о кнопке достаточно ясно, полно и измеримо”.

`/speckit.plan` переводит спецификацию в технический план. `spec-driven.md` перечисляет его работу: анализ требований, user stories и acceptance criteria; проверка constitutional compliance; перевод бизнес-требований в архитектуру и implementation details; создание supporting documents для data models, API contracts и test scenarios; quickstart validation [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).

Command template для `/speckit.plan` добавляет конкретный порядок: запустить `setup-plan` из repo root, прочитать `FEATURE_SPEC`, `/memory/constitution.md` и `IMPL_PLAN` template, заполнить Technical Context, выполнить Constitution Check, затем пройти Phase 0 research и Phase 1 design/contracts [plan.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/plan.md). В Phase 0 неизвестности превращаются в research tasks, а результат должен попасть в `research.md` в форме `Decision`, `Rationale`, `Alternatives considered`; в Phase 1 создаются `data-model.md`, `contracts/`, `quickstart.md`, после чего обновляется agent context file между специальными markers [plan.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/plan.md). Значит, планирование — это не один документ `plan.md`, а пакет проектных артефактов и обновление контекста агента.

Generated `setup-plan.sh` уточняет входной gate для этой фазы. Он получает пути через `get_feature_paths`, а затем проверяет имя ветки через `check_feature_branch`, если только `.specify/feature.json` не указывает на существующий активный feature directory [scripts/bash/setup-plan.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/setup-plan.sh). Если `plan.md` уже существует, script не перезаписывает его, а пропускает копирование template; если template не найден, создаёт пустой `plan.md` с warning [scripts/bash/setup-plan.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/setup-plan.sh). Это неудобная, но важная деталь: отсутствие template не обязательно останавливает фазу plan, поэтому качество `plan.md` нельзя проверять только по факту существования файла.

`/speckit.tasks` читает `plan.md` как обязательный вход и, если они есть, `data-model.md`, `contracts/`, `research.md`; затем выводит `tasks.md`, отмечает независимые задачи `[P]` и выделяет safe parallel groups [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md). Это место, где план становится исполнимым списком работ.

`setup-tasks.sh` ведёт себя строже, чем `setup-plan.sh`: он останавливается, если в активном feature directory нет `plan.md` или `spec.md`, и прямо говорит запустить сначала соответствующие команды [scripts/bash/setup-tasks.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/setup-tasks.sh). Он также собирает список доступных supporting docs (`research.md`, `data-model.md`, `contracts/`, `quickstart.md`) и резолвит `tasks-template` через общий template override stack; если template не найден ни в overrides, presets, extensions, ни в core templates, команда завершается ошибкой [scripts/bash/setup-tasks.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/setup-tasks.sh). Поэтому `tasks.md` в Spec Kit появляется после более жёсткой проверки входов, чем `plan.md`.

PowerShell `common.ps1` показывает, что Windows-ветка не сводится к переписанным именам файлов. Он ищет `.specify` через `Resolve-Path -LiteralPath`, чтобы корректно обрабатывать пути с wildcard-символами, проверяет Git именно на уровне найденного Spec Kit root, принимает имена веток с gitflow-префиксом вроде `feat/004-name` через `Get-SpecKitEffectiveBranchName`, и сравнивает `.specify/feature.json` с активным feature directory без учёта регистра только на Windows [scripts/powershell/common.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/common.ps1). Для методологии это добавляет ещё одну границу: если будущая инструкция пишется для Windows, нельзя просто дать bash-команды и считать, что поведение идентично; надо проверять `.ps1`-скрипты, их JSON-вывод и путь к установленным templates.

Шаблон `/speckit.tasks` требует организовывать задачи по user stories, чтобы каждая история была independently testable increment [tasks.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/tasks.md). Формат задачи строгий: `- [ ] T001 [P] [US1] Description with file path`; story label обязателен только внутри story phases, а setup/foundational/polish phases его не используют [tasks.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/tasks.md). Команда должна вывести dependency graph, parallel execution examples, MVP-first strategy и summary по числу задач на каждую user story [tasks.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/tasks.md). Для будущего текста это помогает показать, как Spec Kit превращает спецификацию в work breakdown, пригодный для агента, а не просто в human TODO.

`/speckit.analyze` проверяет согласованность и покрытие spec/plan/task перед реализацией. Quick Start рекомендует держать первый запуск анализа до `/speckit.implement`, потому что найденные gaps ещё можно исправить в плане и задачах; повторный анализ после реализации допускается как дополнительное review [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html).

Текущий `/speckit.analyze` принципиально read-only: он не меняет файлы и только выдаёт structured analysis report по `spec.md`, `plan.md`, `tasks.md` и constitution [analyze.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/analyze.md). Он строит inventories requirements, stories/actions, task coverage и constitution rules, затем ищет duplication, ambiguity, underspecification, constitution alignment issues, coverage gaps и inconsistencies; constitution conflicts автоматически получают `CRITICAL` severity [analyze.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/analyze.md). Команда ограничивает findings до 50 и в конце предлагает remediation edits только как отдельный следующий шаг с явным разрешением пользователя [analyze.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/analyze.md). Это хороший пример human gate: методика заставляет сначала увидеть карту несогласованностей, а не сразу править всё агентом.

`/speckit.implement` исполняет задачи и строит feature согласно плану [github/spec-kit](https://github.com/github/spec-kit). В досье важно не изображать эту команду как магическое завершение: она получает смысл только при наличии предыдущих артефактов и проверок.

Шаблон `/speckit.implement` перед чтением контекста сканирует `FEATURE_DIR/checklists/`: если хотя бы один checklist содержит незакрытые пункты, команда должна остановиться и спросить, продолжать ли implementation [implement.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/implement.md). Затем она читает `tasks.md`, `plan.md`, optional `data-model.md`, `contracts/`, `research.md`, `quickstart.md` и constitution, проверяет или создаёт ignore-файлы по фактической технологии проекта, выполняет tasks phase-by-phase, уважает `[P]`, требует tests before code при TDD и отмечает выполненные задачи как `[X]` в `tasks.md` [implement.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/implement.md). Так появляется проверяемый след выполнения: не только код изменён, но и task list отражает состояние работ.

`/speckit.taskstoissues` конвертирует generated task lists в GitHub issues для tracking and execution [github/spec-kit](https://github.com/github/spec-kit). Эта команда показывает “хвост” методологии: task-файл может покинуть локальный AI workspace и стать частью командного трекинга.

Template для `/speckit.taskstoissues` особенно осторожен: команда должна получить Git remote через `git config --get remote.origin.url` и продолжать только если remote является GitHub URL; при создании issues через GitHub MCP сервер нельзя создавать issue в репозитории, который не соответствует remote URL [taskstoissues.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/taskstoissues.md). Это маленькая, но важная governance-деталь: перенос задач в GitHub issues является внешним действием и требует проверки адресата.

### Workflows как оркестрация

Reference по workflows показывает, что Spec Kit умеет автоматизировать multi-step SDD sequences: workflows связывают commands, prompts, shell steps и human checkpoints, поддерживают conditional logic, loops, fan-out/fan-in, pause/resume [Workflows](https://github.github.com/spec-kit/reference/workflows.html). Встроенный пример `Full SDD Cycle` запускает `speckit.specify`, затем gate `review-spec`, затем `speckit.plan`, gate `review-plan`, затем `speckit.tasks` и `speckit.implement` [Workflows](https://github.github.com/spec-kit/reference/workflows.html).

Состояние workflow сохраняется в `.specify/workflows/runs/<run_id>/`: `state.json`, `inputs.json`, `log.jsonl` [Workflows](https://github.github.com/spec-kit/reference/workflows.html). Это важная фактура для будущего текста: контекст живёт не только в `specs/`, но и в workflow run state, если используется workflow engine.

Workflow reference добавляет машинно-ориентированную грань: `specify workflow run <source> --json` печатает на stdout только JSON object с `run_id`, `workflow_id`, `status`, `current_step_id` и `current_step_index`, а прогресс шага уходит в stderr [Workflows](https://github.github.com/spec-kit/reference/workflows.html). Это делает workflow engine пригодным не только для человека в терминале, но и для внешнего orchestrator, который должен надёжно понять, где процесс paused, failed или completed.

Catalog management у workflows тоже влияет на provenance процесса. Поиск workflow идёт через catalogs, а resolution order такой: `SPECKIT_WORKFLOW_CATALOG_URL`, затем `.specify/workflow-catalogs.yml`, затем `~/.specify/workflow-catalogs.yml`, затем built-in defaults [Workflows](https://github.github.com/spec-kit/reference/workflows.html). Если будущий текст будет разбирать “какой workflow реально исполнился”, нужно проверять не только YAML в проекте, но и catalog source, который его поставил.

Во всех core command templates есть общий механизм extension hooks: команда ищет `.specify/extensions.yml`, читает ключи вида `hooks.before_specify`, `hooks.after_plan`, `hooks.before_implement`, фильтрует `enabled: false`, различает optional и mandatory hooks и для mandatory hook выводит `EXECUTE_COMMAND` [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md) [plan.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/plan.md) [implement.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/implement.md). При этом сами templates не интерпретируют непустые `condition`; условие оставлено HookExecutor implementation [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md). Это важная неудобная деталь: расширяемость есть, но её фактическая надёжность зависит от того, как конкретная integration исполняет `EXECUTE_COMMAND` и hook protocol.

Source-opening уточнение: workflow layer делает Spec Kit ближе к маленькому orchestrator, а не только к командам. `specify workflow run <source>` может запускать workflow из catalog id, URL или локального YAML-файла; run state получает `run_id`, `workflow_id`, `status`, `current_step_id`, а `specify workflow resume <run_id>` продолжает paused/failed run с того места, где он остановился [Workflows](https://github.github.com/spec-kit/reference/workflows.html). Встроенный пример `Full SDD Cycle` прямо содержит review gates после `specify` и `plan`: `review-spec` и `review-plan` могут approve/reject, а reject aborts workflow. Это показывает, что human checkpoint в Spec Kit может быть не только дисциплиной пользователя, но и элементом YAML workflow.

Ещё одна важная техническая деталь: `--json` у `workflow run` / `resume` печатает machine-readable outcome на stdout, а прогресс шагов уходит в stderr. Для будущего Handbook это означает, что Spec Kit workflows можно встраивать в automation without parsing rich terminal prose, но нужно отдельно обрабатывать stderr как operational log [Workflows](https://github.github.com/spec-kit/reference/workflows.html).

## Артефакты и файлы

Основные артефакты:

- `.specify/memory/constitution.md` — проектные принципы и governance, созданные или обновлённые `/speckit.constitution`; последующие фазы должны учитывать этот файл [github/spec-kit](https://github.com/github/spec-kit).
- `.specify/feature.json` — текущая feature state, куда `/speckit.specify` записывает `feature_name`, `feature_number`, `spec_file` и `feature_dir`; downstream-команды могут использовать этот файл как опору вместо угадывания активной ветки [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md).
- `specs/[branch-name]/spec.md` — спецификация feature с user stories, requirements и acceptance criteria; создаётся `/speckit.specify` [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).
- `specs/[feature]/checklists/requirements.md` — стартовый requirements quality checklist, который `/speckit.specify` создаёт и обновляет при clarifications; пункты проверяют качество требований, а не реализацию [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md) [clarify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/clarify.md).
- `specs/[feature]/checklists/*.md` — тематические checklists, созданные `/speckit.checklist`; каждый файл содержит purpose, created date, feature name, items с `CHK###` и traceability references или явные пометки `[Gap]`, `[Ambiguity]`, `[Conflict]`, `[Assumption]` [checklist.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/checklist.md).
- `specs/[branch-name]/plan.md` — technical implementation plan, создаваемый `/speckit.plan`; содержит стек, архитектурные решения, rationale и связи с требованиями [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).
- `specs/[branch-name]/research.md` — исследовательские заметки, например сравнения библиотек; в примере chat system туда попадают WebSocket library comparisons [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).
- `specs/[branch-name]/data-model.md` — модели данных, например Message и User schemas в примере chat feature [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).
- `specs/[branch-name]/contracts/` — API contracts, WebSocket events или REST endpoints [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).
- `specs/[branch-name]/quickstart.md` — key validation scenarios, генерируемые планированием [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).
- `specs/[branch-name]/tasks.md` — executable task list, созданный `/speckit.tasks`; может включать `[P]` для parallelizable tasks [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).
- `.specify/templates/`, `.specify/scripts/`, `.specify/memory/` — проектные файлы Spec Kit, которые обновляются при `specify init --here --force`; upgrade guide отдельно говорит, что `specs/` и source code при обновлении не трогаются [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html).
- `.specify/templates/overrides/` — project-local overrides, которые README ставит выше presets, extensions и core templates в resolution stack; они позволяют изменить template или command для одного проекта без создания preset [github/spec-kit](https://github.com/github/spec-kit). Для анализа конкретного проекта это один из первых каталогов, где нужно искать расхождение между “core Spec Kit” и реальным поведением.
- `.specify/integration.json` — состояние integration layer: default integration, installed integrations, integration settings и schema [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html).
- `.specify/extensions.yml` — проектная конфигурация extensions и hook bindings. Core command templates ищут в ней `before_*` и `after_*` hooks, но execution зависит от HookExecutor и конкретной integration [Extensions Reference](https://github.github.com/spec-kit/reference/extensions.html) [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md).
- `.specify/extension-catalogs.yml` — проектный источник catalog для поиска и установки extensions. Порядок разрешения: `SPECKIT_CATALOG_URL`, project config, user config, built-in defaults [Extensions Reference](https://github.github.com/spec-kit/reference/extensions.html). Для provenance это отдельный файл: он объясняет, откуда в проект пришла extension, а не только что она установлена.
- `.specify/extensions/agent-context/agent-context-config.yml` — конфигурация bundled `agent-context` extension. В `AGENTS.md` репозитория Spec Kit описано, что сюда записываются `context_file` и `context_markers` вроде `<!-- SPECKIT START -->` / `<!-- SPECKIT END -->`; слой Python и bundled bash/PowerShell scripts читают эти markers из единого source of truth [AGENTS.md](https://github.com/github/spec-kit/blob/main/AGENTS.md).
- `.specify/preset.yml` и `.specify/manifest.lock` — локальные следы preset layer: presets могут принести templates, commands, hooks, constitution amendments, workflow definitions и dependency constraints; lock file фиксирует resolved catalog refs [Presets Reference](https://github.github.com/spec-kit/reference/presets.html).
- `.specify/workflow-catalogs.yml` — project-level workflow catalog source; он стоит ниже `SPECKIT_WORKFLOW_CATALOG_URL`, но выше user config и built-in defaults в порядке resolution [Workflows](https://github.github.com/spec-kit/reference/workflows.html).
- `.specify/workflows/runs/<run_id>/state.json`, `inputs.json`, `log.jsonl` — состояние workflow run для pause/resume [Workflows](https://github.github.com/spec-kit/reference/workflows.html).
- `.specify/integrations/<key>.manifest.json` — manifest конкретной integration: integration id, CLI version, install timestamp и соответствие относительных путей SHA-256-хешам [src/specify_cli/integrations/manifest.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/manifest.py). Это файл provenance для agent-specific commands и scripts.
- `.specify/integrations/speckit.manifest.json` — manifest shared infrastructure: `.specify/scripts/` и `.specify/templates/`, которые зависят от core pack/source checkout и от default integration invoke separator [src/specify_cli/shared_infra.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/shared_infra.py) [src/specify_cli/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/__init__.py).
- `.specify/integration-catalogs.yml` и `.specify/integrations/.cache/catalog-*.json` — конфигурация и cache для integration catalogs. Порядок источников: `SPECKIT_INTEGRATION_CATALOG_URL`, project config, user config, built-in default and community catalogs; community catalog по умолчанию discovery-only [src/specify_cli/integrations/catalog.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/catalog.py).
- `.devin/skills/`, `.goose/recipes/`, `.rovodev/skills/`, `.rovodev/prompts.yml`, `.acli/rovodev/mcp.json`, `~/.hermes/skills/` и пользовательский каталог `generic` integration — примеры agent-specific мест, куда Spec Kit может установить команды вне привычной схемы `.agents/skills` или `.claude/skills` [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Эти пути важны для аудита: отсутствие `/speckit.plan` в одном каталоге ещё не значит, что команда не установлена для другой поверхности агента.
- `.specify/scripts/bash/common.sh` и соответствующие PowerShell variants — generated helper layer, который определяет repo root, active feature, branch validation, command invocation separator и порядок выбора шаблонов [scripts/bash/common.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/common.sh).
- `.specify/scripts/bash/create-new-feature.sh` — script, который создаёт branch, feature directory и стартовый `spec.md`, либо в `--dry-run` только вычисляет эти значения [scripts/bash/create-new-feature.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/create-new-feature.sh).
- `.specify/scripts/bash/setup-plan.sh` — script подготовки `plan.md`; он резолвит active feature, проверяет branch naming, если не действует `.specify/feature.json`, и копирует `plan-template` только если `plan.md` ещё нет [scripts/bash/setup-plan.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/setup-plan.sh).
- `.specify/scripts/bash/setup-tasks.sh` — script подготовки tasks-фазы; он требует существующие `spec.md` и `plan.md`, перечисляет supporting docs и резолвит `tasks-template` через override/preset/extension/core stack [scripts/bash/setup-tasks.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/setup-tasks.sh).
- `.specify/scripts/powershell/common.ps1`, `create-new-feature.ps1`, `setup-plan.ps1`, `setup-tasks.ps1` — Windows/PowerShell-варианты, которые нужно читать отдельно при диагностике Windows-проектов. Важные детали: `-LiteralPath`, проверка имени ветки с удалением gitflow-префикса, сравнение путей без учёта регистра на Windows, `Get-InvokeSeparator` из `.specify/integration.json`, и разрешение шаблонов через overrides/presets/extensions/core [scripts/powershell/common.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/common.ps1) [scripts/powershell/create-new-feature.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/create-new-feature.ps1).
- `src/specify_cli/integrations/base.py` и `src/specify_cli/integrations/__init__.py` — Python-слой реестра integrations. `IntegrationBase` задаёт `key`, `config`, `registrar_config`, `context_file`, `invoke_separator`, `multi_install_safe`, построение нативного вызова команды и установку command files; `__init__.py` регистрирует встроенные integrations в `INTEGRATION_REGISTRY` [src/specify_cli/integrations/base.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/base.py) [AGENTS.md в github/spec-kit](https://raw.githubusercontent.com/github/spec-kit/main/AGENTS.md).

Репозиторий `github/spec-kit` физически показывает устройство инструмента: `integrations`, `templates`, `extensions`, `presets`, `workflows`, `src/specify_cli` [github/spec-kit](https://github.com/github/spec-kit). Для будущего текста это полезно как доказательство, что Spec Kit — не только методологическая статья, но CLI, шаблоны, интеграции, расширения и workflow engine.

Presets и extensions добавляют отдельный слой resolution, который может менять фактический файл, читаемый командой. В presets project-local overrides `.specify/templates/overrides/` имеют highest precedence, затем идут installed presets, installed extensions и core templates `.specify/templates/`; resolution делается per file, а `specify preset resolve <name>` показывает, какой файл победил [Presets](https://github.github.com/spec-kit/reference/presets.html). Это сильная, но потенциально опасная деталь: два проекта с одинаковой командой `/speckit.plan` могут фактически читать разные `plan-template.md`, если у них разные presets / overrides.

Extensions живут в `.specify/extensions/<ext>/`, могут иметь project config, local overrides и environment variables; config merge идёт от extension defaults к project config, затем local override и env vars [Extensions](https://github.github.com/spec-kit/reference/extensions.html). Поэтому extensions не стоит описывать просто как “плагины”: они могут менять команды, шаблоны и quality gates, а также вводить локальные конфигурационные состояния, которые не очевидны по одному Markdown-файлу спецификации.

## Где живёт контекст

Контекст разделён между несколькими слоями.

Первый слой — `specs/[branch-name]/`: спецификация, план, research, data model, contracts, quickstart и tasks. Quick Start отмечает, что команды автоматически определяют active feature по текущей Git branch, например `001-feature-name`; чтобы переключиться между спецификациями, достаточно переключить ветку [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html). Это делает Git branch частью адресации контекста.

Второй проход по templates уточняет эту картину: branch остаётся важной, но active feature также хранится в `.specify/feature.json`, а `SPECIFY_FEATURE_DIRECTORY` может явно указать каталог спецификации [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md). Поэтому будущему тексту лучше не описывать Spec Kit как систему, которая всегда “просто берёт текущую ветку”. Реальный механизм допускает branch hook, environment override и сохранённое состояние feature.

Generated `common.sh` добавляет ещё одну практическую тонкость: он считает `.specify` главным маркером Spec Kit project root, а Git root — запасным вариантом [scripts/bash/common.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/common.sh). Если `.specify` лежит внутри большого репозитория, команды должны работать относительно этого вложенного корня. Для будущего текста это хороший пример того, как методология защищает свой локальный контекст от случайного расширения до всего монорепозитория.

PowerShell `common.ps1` сохраняет тот же приоритет `.specify` над Git, но добавляет Windows-специфичные условия контекста. `Test-HasGit` проверяет наличие `.git` как directory или file, что важно для worktrees/submodules; `Test-FeatureJsonMatchesFeatureDir` нормализует пути и сравнивает их без учёта регистра только на Windows; `Find-FeatureDirByPrefix` останавливается с ошибкой, если один prefix совпал с несколькими каталогами `specs/` [scripts/powershell/common.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/common.ps1). В досье это закрывает часть прежнего открытого вопроса: PowerShell-вариант не просто “похож” на bash, он явно обрабатывает worktree, чувствительность к регистру и неоднозначные совпадения по prefix.

Второй слой — `.specify/memory/constitution.md`. Конституция задаёт долговременные правила проекта: стандарты качества, тестирования, UX consistency, performance requirements и governance для технических решений [github/spec-kit](https://github.com/github/spec-kit). В отличие от feature spec, это не одноразовый документ конкретной задачи.

После чтения `constitution.md` template этот слой лучше понимать как живой governance artifact. Команда `/speckit.constitution` должна не только переписать текст принципов, но и синхронизировать зависимые templates и guidance docs, а в `Sync Impact Report` зафиксировать, какие файлы обновлены и что осталось pending [constitution.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/constitution.md). Если этот отчёт устарел или отсутствует, downstream-команды могут ссылаться на старую версию правил даже при свежей constitution.

Третий слой — integration files и context rules, которые Spec Kit устанавливает под конкретного агента. Для Codex это `.agents/skills` и `AGENTS.md`, для Claude Code `.claude/skills` и `CLAUDE.md`, для некоторых других агентов собственные command directories и rules files [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Здесь живёт адаптация методологии к поверхности конкретного инструмента.

Source-level metadata показывает, что agent context — не один универсальный файл. `codex` и `claude` используют skills directories, но с разными context files: `AGENTS.md` и `CLAUDE.md`; `cursor-agent` тоже ставит skills в `.cursor/skills`, а managed context хранит в `.cursor/rules/specify-rules.mdc`; `kiro-cli` кладёт команды в `.kiro/commands/` и опирается на `AGENTS.md`; `windsurf` кладёт команды в `.windsurf/workflows`, а context file держит в `.windsurf/rules/specify-rules.md`; `gemini` смотрит на `GEMINI.md`. Для `pi` в reference важнее не path-деталь, а ограничение: Spec Kit не настраивает MCP server автоматически [src/specify_cli/integrations/codex/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/codex/__init__.py) [src/specify_cli/integrations/claude/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/claude/__init__.py) [src/specify_cli/integrations/cursor_agent/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/cursor_agent/__init__.py) [src/specify_cli/integrations/kiro_cli/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/kiro_cli/__init__.py) [src/specify_cli/integrations/windsurf/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/windsurf/__init__.py) [src/specify_cli/integrations/gemini/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/gemini/__init__.py) [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Практический вывод: при переносе проекта между агентами нужно проверять не только `.specify/integration.json`, но и то, какой файл реально получает managed section от `agent-context`.

Актуальный список supported integrations добавляет несколько краевых случаев для вопроса о том, где живёт контекст. `Hermes` хранит skills глобально в пользовательском домашнем каталоге, а в проекте оставляет локальный marker `.hermes/skills/`; `RovoDev` использует не только skills directory, но и wrappers/MCP config; `Goose` получает recipes, а не Markdown slash commands; `generic` вообще переносит command files в каталог, который пользователь сам передал через `--commands-dir` [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Поэтому будущий аудит Spec Kit-проекта должен искать команды не только внутри репозитория: часть рабочей поверхности может оказаться в домашнем каталоге, в конфигурации конкретного вендора или в произвольном каталоге.

В этом слое важно различать installed integration и default integration. Reference говорит, что `specify integration use <key>` меняет default integration без удаления остальных installed integrations и refreshes managed shared templates, чтобы command references соответствовали invocation style нового default; modified или untracked shared templates сохраняются, если не передан `--force` [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Значит, фактический prompt surface проекта определяется не только набором папок `.agents/skills`, `.claude/skills` и т.п., но и тем, какая integration сейчас default в `.specify/integration.json`.

`AGENTS.md` самого репозитория Spec Kit уточняет, что обновление context file теперь централизовано через bundled `agent-context` extension. Integration class задаёт `context_file`, а managed Spec Kit section создаётся, обновляется и удаляется с markers из `.specify/extensions/agent-context/agent-context-config.yml`; пользователь может полностью отключить этот слой командой `specify extension disable agent-context`, после чего Spec Kit пропускает создание, обновление и удаление context file [AGENTS.md](https://github.com/github/spec-kit/blob/main/AGENTS.md). Для будущей теории это важная грань “где живёт контекст”: часть контекста находится не в `specs/` и не в constitution, а в agent-specific instruction file, причём этот слой можно случайно или сознательно выключить.

Четвёртый слой — extension, preset and override configuration. `.specify/templates/overrides/` может перекрыть core template локально, `.specify/extensions.yml` может привязать hooks к фазам, а `.specify/preset.yml` описывает applied presets, config и overrides; reference по presets отдельно показывает dependency declarations и `preset.updated` hook как часть жизненного цикла preset [github/spec-kit](https://github.com/github/spec-kit) [Extensions Reference](https://github.github.com/spec-kit/reference/extensions.html) [Presets Reference](https://github.github.com/spec-kit/reference/presets.html). Этот слой влияет на команды, но не является содержательным feature-документом.

Пятый слой — installation manifests и catalogs. `.specify/integrations/*.manifest.json` хранит хеши файлов, созданных Spec Kit, а `.specify/integration-catalogs.yml` может менять источник integrations вплоть до custom catalog через переменную окружения [src/specify_cli/integrations/manifest.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/manifest.py) [src/specify_cli/integrations/catalog.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/catalog.py). Это контекст сопровождения, а не контекст feature: по нему видно, какие команды считаются managed, какие catalog sources использовались и какие файлы нельзя тихо перезаписать.

Шестой слой появляется при workflows: `.specify/workflows/runs/<run_id>/` хранит состояние выполнения, входы и лог шагов [Workflows](https://github.github.com/spec-kit/reference/workflows.html). Это важно для процессов с gate steps: workflow может остановиться на человеческом решении и быть продолжен командой `specify workflow resume <run_id>` [Workflows](https://github.github.com/spec-kit/reference/workflows.html).

## Роли, агенты и участники

Базовый Spec Kit не навязывает сложную систему ролей. Его “участники” — человек, выбранный AI coding agent, Specify CLI и набор команд/шаблонов. Человек задаёт намерение, принципы, уточнения, стек и принимает решения на фазовых переходах; агент генерирует и изменяет артефакты; CLI устанавливает инфраструктуру и управляет integrations, workflows, presets и extensions.

У выбранного агента есть ещё одна роль: он должен корректно исполнять контракт команд своей integration. Для integrations на Markdown-командах это чтение и выполнение файлов `/speckit.*`; для Codex skills mode — вызов `$speckit-*`; для Kiro CLI — работа с fallback-инструкциями, где аргументы не подставляются динамически, а команда получает явные примеры прямого вызова scripts [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html) [issue #1926](https://github.com/github/spec-kit/issues/1926). Это не продуктовая роль вроде PM/developer, но для методологии она критична: один и тот же `spec.md` может быть создан по-разному, если поверхность агента иначе понимает параметры команды.

Python-слой integrations делает роль “выбранного агента” технически конкретной. В `AGENTS.md` репозитория Spec Kit сказано, что каждая integration — отдельный subpackage под `src/specify_cli/integrations/<agent>/`, а global `INTEGRATION_REGISTRY` является единым источником сведений о каталогах, форматах, возможностях и context files [AGENTS.md в github/spec-kit](https://raw.githubusercontent.com/github/spec-kit/main/AGENTS.md). `IntegrationBase` дополнительно показывает, что CLI-based integration должна иметь `key`, совпадающий с исполняемым инструментом, потому что проверки опираются на `shutil.which`, а для нестандартной установки можно задать `SPECKIT_INTEGRATION_<KEY>_EXECUTABLE` [src/specify_cli/integrations/base.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/base.py). Это важная рабочая деталь: “поддержка агента” в Spec Kit — не только набор Markdown-команд, но и контракт реестра для установки, проверки и возможного dispatch.

Отдельные integration classes добавляют agent-specific упаковку команд. `CodexIntegration` в режиме skills устанавливает команды как skills и меняет invocation separator на `-`, из-за чего `speckit.plan` превращается в `$speckit-plan`; при scripted dispatch он строит `codex exec "<prompt>"` и добавляет `--json`, если нужен machine-readable output [src/specify_cli/integrations/codex/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/codex/__init__.py). `ClaudeIntegration` тоже ставит skills, но дополнительно injects `argument-hint`, `user-invocable` и `disable-model-invocation` в skill frontmatter; `CursorAgentIntegration` по умолчанию ставит skills в `.cursor/skills`, а не обычные `.md` slash commands [src/specify_cli/integrations/claude/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/claude/__init__.py) [src/specify_cli/integrations/cursor_agent/__init__.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/cursor_agent/__init__.py). Поэтому человек, проверяющий результат, должен смотреть не только на содержимое `templates/commands/*.md`, но и на post-processing конкретной integration: он может менять имя команды, место хранения, frontmatter и способ non-interactive dispatch.

Supported integrations reference показывает, что у некоторых агентов роль “исполнителя Spec Kit-команды” зависит от внешней среды сильнее, чем у классических slash commands. `Hermes` работает через глобальный каталог skills, `RovoDev` через `acli rovodev`, MCP configuration и generated wrapper scripts, `Pi` требует ручной настройки MCP server, а `generic` integration перекладывает ответственность за каталог команд на пользователя [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). В методологическом описании это нужно сохранить как границу ответственности: Spec Kit может поставить командные файлы, но не всегда полностью настраивает агентскую среду, которая должна эти файлы прочитать и исполнить.

Research-угол появляется в `spec-driven.md`: там сказано, что research agents могут собирать technical context, проверять library compatibility, performance benchmarks, security implications и организационные ограничения [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md). В официальной quickstart-цепочке это не оформлено как отдельная роль пользователя, но идея исследовательского контекста встроена в plan/research artifacts.

Spec Kit Agents делает роли явными: multi-agent SDD pipeline с PM and developer roles, state-machine orchestrator, discovery hooks и validation hooks [Spec Kit Agents](https://arxiv.org/abs/2604.05278). Этот источник нельзя автоматически переносить на весь Spec Kit как shipped default, но он показывает направление усиления методологии: отделить product/requirements роль от developer role и добавить проверяемые контекстные hooks.

## Человеческие подтверждения

В базовом порядке работы человеческие решения стоят между фазами, даже если не всегда оформлены как отдельные approval screens.

Перед `/speckit.plan` человек должен проверить, что `spec.md` действительно описывает нужное поведение, а неопределённости либо сняты через `/speckit.clarify`, либо сознательно оставлены для exploratory prototype [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html). Перед `/speckit.tasks` нужно проверить технический план: выбор стека, архитектуру, constraints, quickstart validation и связь с требованиями [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md). Перед `/speckit.implement` нужно проверить `tasks.md` и результат `/speckit.analyze`, потому что после начала реализации исправлять пробелы дороже [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html).

Command templates добавляют ещё один явно машинно-проверяемый gate. `/speckit.implement` перед стартом собирает статус всех файлов в `FEATURE_DIR/checklists/`; если есть незакрытые пункты, он выводит таблицу с counts и спрашивает пользователя, продолжать ли несмотря на incomplete checklists [implement.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/implement.md). Это не просто рекомендация “проверить требования”: незакрытый checklist становится blocking prompt перед кодом.

Workflow engine делает эти решения явными: встроенный `Full SDD Cycle` включает gate `review-spec` с сообщением “Review the generated spec before planning” и gate `review-plan` перед tasks; при reject workflow aborts [Workflows](https://github.github.com/spec-kit/reference/workflows.html). Это хороший материал для будущего текста: Spec Kit можно использовать как последовательность команд вручную, но workflows превращают фазовые решения в формальные остановки.

Upgrade guide добавляет ещё один вид человеческого решения: при `specify init --here --force` обновляются command files, scripts, templates и memory files, а документация отдельно предупреждает сохранить или закоммитить `.specify/memory/constitution.md`, если он был кастомизирован [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html). Значит, сопровождение Spec Kit-проекта требует review не только feature artifacts, но и самой методологической инфраструктуры.

`/speckit.constitution` добавляет governance decision в явной форме: если тип version bump неоднозначен, template требует предложить reasoning перед финализацией, а при неизвестной `RATIFICATION_DATE` разрешает оставить `TODO()` с объяснением и включить его в `Sync Impact Report` [constitution.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/constitution.md). Это показывает, что человек принимает решение не только о фиче, но и о статусе проектных правил: это уточнение, новое правило или несовместимая смена принципа.

Extension and preset layers расширяют список человеческих решений. Extension reference говорит, что extension commands регистрируются в текущей AI coding agent integration, но если команда не появилась, агент может требовать restart [Extensions Reference](https://github.github.com/spec-kit/reference/extensions.html). Preset reference даёт `specify preset resolve <name>` для просмотра, какой файл реально победил в resolution stack [Presets Reference](https://github.github.com/spec-kit/reference/presets.html). В рабочих терминах это значит: перед важным запуском человеку может понадобиться проверить не только `spec.md`, но и то, какие templates/commands реально используются после preset/extension overrides.

Integration lifecycle добавляет ещё один человеческий gate. При `specify integration install <key>` Spec Kit автоматически продолжает только если уже установленная и новая integration объявлены multi-install safe; иначе нужно сознательно выбрать `switch` или передать `--force` [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). При `switch` в уже установленную integration параметр `--integration-options` rejected, потому что изменение options требует reinstall managed files; recommended path — сначала `upgrade <key> --integration-options ...`, затем `use <key>` [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Это маленькое, но важное место решения: человек выбирает между переносимостью команды, чистотой agent-specific инструкций и риском перезаписи managed files.

## Проверка результата

Проверка в Spec Kit многоуровневая.

На уровне требований `/speckit.checklist` валидирует completeness, clarity и consistency требований; `spec-driven.md` описывает checklists как “unit tests” для спецификации: нет ли `[NEEDS CLARIFICATION]`, тестируемы ли требования, измеримы ли success criteria [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html) [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).

Command templates дают более конкретную лестницу проверки требований. `/speckit.specify` создаёт стартовый `requirements.md` и оценивает spec readiness; `/speckit.clarify` после каждого принятого ответа обновляет checklist item status; `/speckit.checklist` создаёт дополнительные доменные checklists с traceability references и issue markers [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md) [clarify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/clarify.md) [checklist.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/checklist.md). Поэтому checklists в Spec Kit — не один финальный отчёт, а серия файлов, которые сопровождают spec до реализации.

На уровне плана `spec-driven.md` описывает constitutional compliance: plan должен проверять соответствие project constitution and architectural principles [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md). В этом же файле есть более жёсткий пример constitutional foundation с gates вроде Simplicity Gate, Anti-Abstraction Gate и Integration-First Gate; важно воспринимать его как методологический пример из `spec-driven.md`, а не обязательно как неизменный набор статей для каждого современного проекта [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).

На уровне согласованности `/speckit.analyze` проверяет spec/plan/task consistency and coverage перед implementation [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html). Это главный официальный механизм против рассинхронизации артефактов до реализации.

Шаблон `/speckit.analyze` полезен именно как read-only audit: он строит coverage map, помечает требования без задач, задачи без требований, расплывчатые adjectives, missing acceptance criteria, placeholder markers и constitution conflicts; при этом он не исправляет документы без отдельного разрешения [analyze.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/analyze.md). В досье это стоит хранить как пример правильного разделения: сначала diagnosis, затем отдельная правка.

На уровне repository evidence Spec Kit Agents предлагает discovery hooks и validation hooks: discovery собирает relevant files, conventions, dependencies and history до фаз, validation проверяет intermediate artifacts and, after implementation, project checks such as tests and linters where applicable [Spec Kit Agents](https://arxiv.org/abs/2604.05278). Это не заменяет official quickstart, но хорошо показывает, где базовому Spec Kit может понадобиться усиление для больших репозиториев.

На уровне установленной среды `specify check` проверяет наличие `git` и CLI-based AI coding agents, остаётся offline и отдельно советует `specify self check`, если локальная команда ведёт себя как старая версия или нужная CLI feature отсутствует [Core Commands](https://github.github.com/spec-kit/reference/core.html). `specify version --features --json` предназначен для scripts и coding agents, которым нужно выбрать workflow по возможностям установленного CLI [Core Commands](https://github.github.com/spec-kit/reference/core.html). Это отдельный вид проверки: перед методологическим процессом нужно понять, что локальная CLI-поверхность действительно поддерживает ожидаемые команды.

На уровне integration layer проверка начинается с `specify integration list`: он показывает current/default integration, остальные installed integrations, необходимость внешнего CLI tool или IDE-based режим, а также multi-install safety [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). После обновления CLI сам факт новой версии недостаточен: `specify integration upgrade [<key>]` refreshes installed integration files, blocks on locally modified files без `--force`, removes stale files и держит shared templates aligned with the default integration [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Поэтому проверка установленной среды должна включать и CLI version, и состояние managed integration files.

Для современных supported integrations проверка должна учитывать тип поверхности. `specify integration list` полезен как общий вход, но для `generic` дополнительно нужен фактический `--commands-dir`, для `Hermes` нужно проверить глобальный `~/.hermes/skills/`, для `RovoDev` — generated wrappers, `.rovodev/prompts.yml` и MCP config, для `Goose` — recipes, а для `Pi` — ручную настройку MCP server, которую CLI не делает автоматически [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Это не косметическая деталь: команда может считаться установленной, но конкретная поверхность агента всё ещё не видеть свежие команды или исполнять их через другой механизм запуска.

На уровне generated scripts есть дополнительные технические проверки. `check_feature_branch` принимает sequential branch names `001-feature-name` и timestamp names `YYYYMMDD-HHMMSS-feature-name`, но отклоняет ветку без распознаваемого prefix, если проект находится в Git repository [scripts/bash/common.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/common.sh). `setup-tasks.sh` дополнительно останавливает фазу tasks, если отсутствуют `spec.md`, `plan.md` или обязательный `tasks-template` [scripts/bash/setup-tasks.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/setup-tasks.sh). Эти проверки полезны не как гарантия качества требований, а как защита от запуска фаз в неверном каталоге или в неполной инфраструктуре.

PowerShell-слой добавляет проверки, которые стоит отдельно учитывать в Windows-инструкции. `Get-FeaturePathsEnv` возвращает набор путей `FEATURE_DIR`, `FEATURE_SPEC`, `IMPL_PLAN`, `TASKS`, `RESEARCH`, `DATA_MODEL`, `QUICKSTART`, `CONTRACTS_DIR`; `Get-InvokeSeparator` читает `.specify/integration.json`, чтобы форматировать команду как `/speckit.plan` или `/speckit-plan` в зависимости от integration settings [scripts/powershell/common.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/common.ps1). Ещё одна проверка скрыта в композиции шаблонов: `Resolve-TemplateContent` пытается найти Python 3 и PyYAML для чтения preset manifests; если Python или PyYAML отсутствуют, composition strategies могут быть проигнорированы с warning [scripts/powershell/common.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/common.ps1). Поэтому “файлы на месте” не всегда означает “используется тот же стек шаблонов”, особенно в проектах с presets.

## Хвост жизненного цикла

Spec Kit не заканчивается на `/speckit.implement`.

Во-первых, `tasks.md` может быть превращён в GitHub issues через `/speckit.taskstoissues` [github/spec-kit](https://github.com/github/spec-kit). Это делает результат планирования пригодным для командного трекинга и распределения.

Во-вторых, upgrade guide показывает сопровождение самой методологической среды: CLI можно обновлять через `specify self upgrade`, project files — через `specify init --here --force`, а `specs/` при этом считается безопасной пользовательской зоной, которую template packages не трогают [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html).

Upgrade guide делает это сопровождение довольно конкретным. `specify self check` read-only проверяет, есть ли новая release; `specify self upgrade --dry-run` показывает, что будет запущено; bare `specify self upgrade` выполняется сразу и auto-detects `uv tool` или `pipx`, а для `uvx`, source checkout и unsupported runtime печатает guidance без изменения файлов [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html). Для pinned upgrade тег должен начинаться с `vMAJOR.MINOR.PATCH`; branch names, hash refs, `latest` и bare versions без `v` rejected [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html). Это полезная деталь для автоматизации: обновление Spec Kit имеет собственную проверяемую семантику, а не сводится к “переустановите пакет”.

Project-file upgrade имеет отдельный риск. `specify init --here --force` обновляет slash command files, `.specify/scripts/`, `.specify/templates/` и `.specify/memory/`, но документация прямо предупреждает, что `.specify/memory/constitution.md` сейчас может быть overwritten default template, если его заранее не сохранить или не восстановить из Git [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html). То есть `specs/` защищён, а project governance file — нет; это важная граница безопасности обновления.

В-третьих, Workflows поддерживают pause/resume и run state, поэтому долгий процесс можно продолжать с того места, где он остановился на gate или ошибке [Workflows](https://github.github.com/spec-kit/reference/workflows.html).

В-четвёртых, extensions and presets имеют собственный lifecycle. Extensions можно искать, ставить, обновлять, отключать, менять priority и каталог; catalog resolution идёт через `SPECKIT_CATALOG_URL`, `.specify/extension-catalogs.yml`, user config и built-in defaults, а configuration merge идёт от defaults к project config, local overrides и environment variables `SPECKIT_<EXT>_*` [Extensions Reference](https://github.github.com/spec-kit/reference/extensions.html). Presets можно складывать по priority, а file resolution идёт через project-local overrides, installed presets, installed extensions и core templates; `specify preset resolve <name>` показывает, какой файл фактически выбран [Presets Reference](https://github.github.com/spec-kit/reference/presets.html). Это делает lifecycle Spec Kit-проекта похожим на сопровождение маленькой методологической платформы, а не только набора документов.

В-пятых, integrations имеют безопасный uninstall/switch/upgrade path. Reference говорит, что Spec Kit tracks every file created during install with original SHA-256 hash: unmodified files удаляются автоматически, modified files сохраняются, а `--force` нужен для удаления или перезаписи изменённых файлов [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). При `integration upgrade` команда блокируется на locally modified files без `--force`, stale files удаляются, а shared templates остаются aligned with the default integration [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Это важный “хвост” метода: перенос между агентами и обновление команд не должны тихо уничтожать локальные правки.

Детали исходного кода делают этот хвост ещё строже. Shared infrastructure installer различает три режима: обычный install добавляет только отсутствующие файлы, `refresh_managed=True` перезаписывает только те файлы, чей текущий хеш на диске всё ещё совпадает с прежним manifest hash, а `force=True` перезаписывает обычные файлы независимо от хеша; symlinks и symlinked parents не перезаписываются, потому что могут уводить запись за пределы project root [src/specify_cli/shared_infra.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/shared_infra.py). При записи manifest используется temporary file и `os.replace`, а пути валидируются как относительные к project root [src/specify_cli/integrations/manifest.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/manifest.py). Для практической методологии это означает, что “обновить Spec Kit” — это операция с diff-like safety, а не простое копирование свежих шаблонов.

В-шестых, `agent-context` устанавливается как bundled extension при `specify init`, если пользователь не передал `--ignore-agent-tools`; upgrade/reference-слой отдельно показывает, что проблемы после обновления часто лечатся не повторным `specify`, а restart IDE/editor и проверкой agent-specific folders [Installation Guide](https://github.github.com/spec-kit/installation.html) [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html). Для lifecycle это означает: после обновления методологической инфраструктуры нужно проверить не только файлы на диске, но и то, что конкретный агент перечитал новые commands/skills/context.

Открытый хвост: официальная документация хорошо описывает путь до implementation, но слабее фиксирует обязательную post-implementation reconciliation. В issue #1063 пользователь формулирует отсутствие first-class way to reconcile post-implementation gaps после `specify -> plan -> tasks -> implement`, чтобы вносить изменения обратно в `spec.md`, `plan.md`, `tasks.md` [issue #1063](https://github.com/github/spec-kit/issues/1063). Это issue нельзя использовать как официальный факт о продуктовой roadmap, но оно точно показывает практический риск: код может уйти вперёд, а спецификационные артефакты снова начнут отставать.

## Сильные стороны

Spec Kit даёт явную цепочку передачи контекста. Вместо одного большого prompt появляются `spec.md`, `plan.md`, supporting documents и `tasks.md`; каждый следующий шаг обязан читать предыдущие артефакты [документация Spec Kit](https://github.github.com/spec-kit/) [spec-driven.md](https://raw.githubusercontent.com/github/spec-kit/main/spec-driven.md).

Методика хорошо материализована. Есть CLI, integrations, templates, scripts, presets, extensions и workflows, а не только статья о SDD [github/spec-kit](https://github.com/github/spec-kit) [CLI Reference](https://github.github.com/spec-kit/reference/overview.html). Это облегчает перенос практики между инструментами и командами.

Spec Kit явно отделяет продуктовую спецификацию от технического планирования. `/speckit.specify` фокусируется на what/why, `/speckit.plan` принимает tech stack и architecture choices [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html). Это снижает риск преждевременного выбора реализации.

Методика встроенно поддерживает quality gates: clarify, checklist, analyze, workflow gates, constitutional compliance [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html) [Workflows](https://github.github.com/spec-kit/reference/workflows.html).

Интеграционная модель делает Spec Kit переносимым: можно работать с Copilot, Codex, Claude, Gemini и другими агентами, а `generic` integration оставляет escape hatch для инструментов вне списка [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html).

## Failure modes / overuse risks

Первый риск — документарная имитация прогресса. Если человек быстро прогоняет `/speckit.specify`, `/speckit.plan`, `/speckit.tasks` и не проверяет содержание, получается несколько убедительных файлов, но не обязательно более правильная разработка. Quick Start фактически предупреждает об этом через регулярные gates `/speckit.clarify`, `/speckit.checklist`, `/speckit.analyze` для задач с meaningful ambiguity [Quick Start Guide](https://github.github.com/spec-kit/quickstart.html).

Второй риск — context blindness. Spec Kit Agents прямо говорит, что AI coding agents в больших evolving repositories могут галлюцинировать APIs и нарушать architecture из-за отсутствия текущего repo context; структура SDD сама по себе не гарантирует grounding [Spec Kit Agents](https://arxiv.org/abs/2604.05278). Для будущей главы важно не обещать, что `spec.md` автоматически решает проблему знания репозитория.

Третий риск — drift после реализации. Если после `/speckit.implement` код меняется вручную, а `spec.md`, `plan.md`, `tasks.md` не обновляются, метод возвращается к старой проблеме отставшей документации. Issue #1063 формулирует это как post-implementation gaps и artifact drift [issue #1063](https://github.com/github/spec-kit/issues/1063).

Четвёртый риск — review overload. Обзор Birgitta Böckeler на Martin Fowler/Thoughtworks подчёркивает, что термин SDD ещё семантически расплывчат, а elaborated approaches with many files могут усилить review burden и дать ложное чувство контроля [Understanding Spec-Driven-Development](https://martinfowler.com/articles/exploring-gen-ai/sdd-3-tools.html). Для досье это не первоисточник Spec Kit, но полезный внешний критический источник.

Пятый риск — слепая вера в community extensions, presets и workflows. Документация workflows прямо говорит, что maintainers не review, audit, endorse or support workflow code; перед установкой нужно проверять source и использовать at your own discretion [Workflows](https://github.github.com/spec-kit/reference/workflows.html). Документация по extensions и presets показывает богатую экосистему, но это не равняется гарантированному качеству каждого пакета [Community Extensions](https://github.github.com/spec-kit/community/extensions.html) [Community Presets](https://github.github.com/spec-kit/community/presets.html).

У этого риска есть техническая форма: extensions and presets могут менять commands, templates, scripts и hook behavior, а resolution priority решает, какой файл победит [Extensions Reference](https://github.github.com/spec-kit/reference/extensions.html) [Presets Reference](https://github.github.com/spec-kit/reference/presets.html). Если команда не проверяет `specify preset resolve <name>`, `.specify/extensions.yml` и installed extension status, она может думать, что работает по core Spec Kit, хотя фактически использует локальный override или сторонний пакет.

Шестой риск — маленькие изменения. Thoughtworks-обзор делит SDD на spec-first, spec-anchored и spec-as-source и задаёт вопрос, не слишком ли тяжёлы некоторые workflows для большинства реальных задач [Understanding Spec-Driven-Development](https://martinfowler.com/articles/exploring-gen-ai/sdd-3-tools.html). Для Spec Kit это означает: метод особенно полезен там, где неоднозначность, цена ошибки и потребность в traceability достаточно велики; для маленького исправления вся цепочка может быть чрезмерной.

Седьмой риск — ложная уверенность в hooks. Core command templates умеют обнаруживать hook definitions и выводить `EXECUTE_COMMAND`, но не выполняют произвольные условия сами; для conditions нужен HookExecutor [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md). Если конкретная agent integration плохо выполняет этот protocol, mandatory hooks могут стать текстовой инструкцией, а не реальным barrier. Для больших репозиториев это особенно важно: контекстные проверки должны быть проверяемо выполнены, а не просто упомянуты в prompt.

Восьмой риск — скрытое расхождение между ожидаемым core workflow и локальной установкой. На это работают сразу несколько механизмов: `.specify/templates/overrides/`, presets, extensions, workflow catalogs, default integration и версия CLI [github/spec-kit](https://github.com/github/spec-kit) [Core Commands](https://github.github.com/spec-kit/reference/core.html) [Workflows](https://github.github.com/spec-kit/reference/workflows.html). Если анализировать проект только по `spec.md` и `plan.md`, можно пропустить, что реальная команда `/speckit.plan` уже переопределена preset, workflow пришёл из custom catalog, а локальный `specify` не поддерживает новую feature.

Девятый риск — ошибка установки из-за совпадающих имён пакетов. Installation Guide прямо предупреждает, что PyPI-пакеты с похожими или теми же именами, например `specify-cli`, не являются официальными пакетами Spec Kit [Installation Guide](https://github.github.com/spec-kit/installation.html). Для Handbook это практическая проверка: перед разбором странного поведения нужно посмотреть `specify version`, путь исполняемого файла и команду установки, а не считать любой `specify-cli` на машине тем самым Spec Kit.

Десятый риск — тихое отсутствие слоя agent context. Если `agent-context` extension отключена или её `agent-context-config.yml` указывает не тот `context_file`, команды могут существовать, но агент не получит актуальные project-level instructions между markers `SPECKIT START` / `SPECKIT END` [AGENTS.md](https://github.com/github/spec-kit/blob/main/AGENTS.md). Это не ломает сами `spec.md` и `plan.md`, но ослабляет перенос методологии в конкретную поверхность агента.

Одиннадцатый риск — неправильная активная feature из-за сложного fallback-порядка. Если в проекте есть `.specify/feature.json`, `SPECIFY_FEATURE_DIRECTORY`, текущая Git branch и несколько каталогов `specs/` с похожими numeric prefixes, нужно понимать, какой слой победил [scripts/bash/common.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/common.sh). Особенно неприятен случай, когда command output выглядит нормальным, но `FEATURE_DIR` указывает не на тот каталог. Для диагностики нужны `setup-plan.sh --json`, `setup-tasks.sh --json`, проверка `.specify/feature.json` и текущих environment variables.

Двенадцатый риск — агент неправильно вызывает generated scripts. Issue #987 описывает повторяющийся случай, когда разные агенты неверно вызывали `create-new-feature.sh` с `--short-name` и пытались исправляться после чтения script syntax; PR #1001 был смёржен 22 октября 2025 года как правка prompt/example для этой проблемы [issue #987](https://github.com/github/spec-kit/issues/987) [PR #1001](https://github.com/github/spec-kit/pull/1001). Это слабый источник и исторический баг, но он показывает важную методологическую деталь: если агент должен исполнять shell scripts, prompt templates должны давать не только смысл фазы, но и точную форму команды.

Тринадцатый риск — расхождение вариантов скриптов. Issue #975 показывает, что PowerShell `create-new-feature.ps1` в версии `v0.0.20` мог выбирать номер feature только по `specs/` текущей ветки и игнорировать существующие имена веток, что приводило к повторному `001-*` при работе с несмёрженными feature branches; текущий script уже содержит проверку веток и refs, но сам факт такого бага показывает, что совпадение поведения bash/PowerShell нужно проверять по текущим generated files, а не предполагать [issue #975](https://github.com/github/spec-kit/issues/975) [scripts/powershell/create-new-feature.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/create-new-feature.ps1).

Четырнадцатый риск — стек шаблонов зависит от локальной Python-среды. PowerShell `Resolve-TemplateContent` пытается прочитать preset manifests через Python и PyYAML, чтобы применить `prepend`, `append` или `wrap`; при отсутствии Python 3 или PyYAML он предупреждает, что composition strategies могут быть проигнорированы [scripts/powershell/common.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/common.ps1). Для команды это неприятный класс сбоя: preset вроде бы установлен, но фактический command/template может быть ближе к replace/fallback behavior, чем к ожидаемой композиции.

Отдельный риск — переносимость integrations может быть переоценена. Страница Supported integrations перечисляет много агентов, но рядом с ними есть разные ограничения: Kiro CLI требует fallback-инструкций вместо обычной подстановки аргументов, Pi требует отдельной ручной настройки MCP server, а часть integrations помечена как особый способ запуска или путь миграции [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Для будущей теории это граница обещания “use any coding agent”: Spec Kit действительно выносит процесс в общие артефакты, но поверхность агента остаётся частью процесса и может менять качество исполнения.

Пятнадцатый риск — рассинхронизация CLI и project files. Upgrade guide подчёркивает, что CLI tool отделён от project files: обновление `specify` даёт новые возможности CLI, но slash commands, scripts и templates в проекте обновляются отдельно через `specify init --here --force --integration <agent>` [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html). Если обновить только одно из двух, агент может читать старые command templates, а человек — смотреть новую документацию и ожидать поведения, которого в локальных files ещё нет.

Шестнадцатый риск — повреждённый или устаревший manifest layer. Если `.specify/integrations/<key>.manifest.json` отсутствует, нечитаем или создан старой версией, Spec Kit может относиться к уже существующим shared files как к untracked/customized и не обновлять их без `--force` [src/specify_cli/shared_infra.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/shared_infra.py). Это лучше, чем тихо перезаписать пользовательские правки, но для человека выглядит как “upgrade ничего не поменял”. Диагностика должна смотреть не только на версию CLI, но и на manifest files, warnings about skipped/preserved/symlinked paths и фактический hash-tracked статус generated files.

Семнадцатый риск — `--force` безопасен не для всех слоёв одинаково. Upgrade guide прямо говорит, что `specs/` и source code не входят в template packages и не перезаписываются, но `.specify/memory/constitution.md`, `.specify/scripts/` и `.specify/templates/` могут быть overwritten при project-file update [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html). Поэтому перед обновлением нужно сохранять customized constitution, templates и scripts; иначе Spec Kit может потерять именно те локальные правила, ради которых его внедряли.

Восемнадцатый риск — скрытый drift в CLI-dispatch конкретного агента. Если `SPECKIT_INTEGRATION_<KEY>_EXECUTABLE` или `SPECKIT_INTEGRATION_<KEY>_EXTRA_ARGS` заданы локально, `specify` может запускать не тот binary или не тот режим агента, который ожидает команда; для ключей вроде `cursor-agent` и `kiro-cli` имя переменной ещё и нормализуется через подчёркивания [src/specify_cli/integrations/base.py](https://raw.githubusercontent.com/github/spec-kit/main/src/specify_cli/integrations/base.py). Это особенно опасно в CI и scripted workflows: внешне integration установлена, command files свежие, но фактическое исполнение идёт через изменённый agent command line.

Девятнадцатый риск — смешение “supported integration” с “готово к совместной установке”. Reference отдельно перечисляет multi-install safe integrations и предупреждает, что не все supported agents безопасно ставить рядом без `--force` [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Если команда механически добавляет новый агент в уже настроенный проект, она может получить конфликт context files, command paths или shared settings; если применить `--force`, риск смещается в сторону перезаписи managed files. Для практики это значит, что перед экспериментом с новой integration нужен короткий аудит: текущий default, installed integrations, признак безопасности, manifest files и локальные правки.

Двадцатый риск — глобальные или внешние командные поверхности хуже видны в обычном diff. Если integration вроде `Hermes` использует `~/.hermes/skills/`, а `generic` пишет команды в произвольный каталог, ревью репозитория может не показать полный набор инструкций, которые реально читает агент [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). Это особенно важно для командной работы и воспроизводимости: два разработчика могут иметь одинаковые `specs/` и `.specify/`, но разные глобальные skills или разные `--commands-dir`.

Source-opening pass добавил отдельный риск upgrade/override. Upgrade guide прямо предупреждает, что `specify init --here --force` сейчас может перезаписать `.specify/memory/constitution.md`; workaround — заранее сохранить constitution и восстановить после upgrade. Тот же guide отдельно предупреждает, что custom scripts/templates in `.specify/scripts/` and `.specify/templates/` can be overwritten, а у IDE-based agents после upgrade могут появиться duplicate slash commands, которые нужно удалить вручную [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html). Для методологии это важно: Spec Kit делает процесс воспроизводимым через файлы, но эти же файлы становятся mutable infrastructure, которую можно случайно сбросить при обновлении.

Ещё один риск связан с presets/extensions: чем больше process behavior переносится в catalogs, priorities и local overrides, тем сложнее понять, какой именно процесс исполнил агент. Для аудита недостаточно сохранить `spec.md` и `plan.md`; нужно сохранять версию Spec Kit, integration, активные presets/extensions, priority order и project-local overrides.

## Аналитические заметки предыдущих проходов

### Что должно попасть в теорию

В теории Spec Kit стоит описывать не как “пример хорошего prompt”, а как спецификационный конвейер, где каждый Markdown-артефакт становится входом следующего шага: constitution, spec, clarify/checklist, plan, tasks, analyze, implement.

Нужно показать разделение контекста: `.specify/memory/constitution.md` для долговременных принципов, `specs/[branch-name]/` для feature artifacts, integration files для конкретного агента, workflow run state для автоматизированной оркестрации.

Нужно уточнить, что в текущих templates активная feature определяется не только Git branch. В теоретическом тексте стоит показать связку `before_specify` branch hook, `SPECIFY_FEATURE_DIRECTORY`, `.specify/feature.json` и `specs/[feature]/` как несколько уровней адресации feature context [specify.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/specify.md).

Нужно сохранить различие между shipped Spec Kit и исследовательским Spec Kit Agents. Первый даёт toolkit и workflow; второй показывает усиление через PM/developer roles, discovery hooks и validation hooks. Их нельзя склеивать в один “официальный режим”, но вместе они хорошо показывают направление развития SDD.

Нужно честно написать про человеческие gates. Spec Kit не снимает необходимость review; он делает review предметным: сначала spec, потом plan, потом tasks and consistency, потом implementation.

Нужно включить ограничение: без repo grounding и post-implementation reconciliation спецификационные документы могут стать красивой, но устаревшей поверхностью.

Нужно отдельно описать extension/preset layer как силу и риск. Сила в том, что команды, hooks, templates и constitution amendments можно упаковывать и переносить; риск в том, что core Spec Kit может быть переопределён сторонним или локальным пакетом, и тогда фактический процесс надо проверять через installed extensions, `.specify/extensions.yml`, `.specify/preset.yml` и `specify preset resolve` [Extensions Reference](https://github.github.com/spec-kit/reference/extensions.html) [Presets Reference](https://github.github.com/spec-kit/reference/presets.html).

Нужно добавить constitution lifecycle как отдельный элемент теории. `/speckit.constitution` важен не только тем, что создаёт `.specify/memory/constitution.md`, но и тем, что требует semver governance, `Sync Impact Report` и проверку dependent templates; это делает правила проекта частью исполняемой методологической инфраструктуры [constitution.md](https://raw.githubusercontent.com/github/spec-kit/main/templates/commands/constitution.md).

Нужно аккуратно показать operational introspection: `specify check`, `specify version --features --json`, workflow `--json`, workflow status and resume. Spec Kit интересен тем, что часть процесса можно проверять и оркестрировать машинно, но это требует читать состояние CLI, workflow run и catalog resolution, а не только содержательные Markdown-файлы [Core Commands](https://github.github.com/spec-kit/reference/core.html) [Workflows](https://github.github.com/spec-kit/reference/workflows.html).

Нужно показать lifecycle самого инструмента как часть методологии. Spec Kit-проект живёт не только через feature specs: CLI обновляется отдельно, project files обновляются отдельно, integrations имеют install/use/switch/upgrade/uninstall, а managed files защищаются SHA-256 manifests [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html) [Upgrade Guide](https://github.github.com/spec-kit/upgrade.html). Без этого теория будет выглядеть как статичная цепочка документов, хотя на практике это сопровождаемая инфраструктура.

Нужно точнее сформулировать обещание “Use any coding agent”. Теоретически Spec Kit переносит spec/plan/tasks и command templates между агентами; практически каждая integration имеет свою поверхность: slash command, skill, recipe, wrapper, MCP setup или пользовательский commands directory [Supported AI Coding Agent Integrations](https://github.github.com/spec-kit/reference/integrations.html). В теории это лучше показать как adapter layer, а не как полную прозрачность агента.

Нужно добавить generated scripts как отдельный слой методологии. `common.sh`, `create-new-feature.sh`, `setup-plan.sh` и `setup-tasks.sh` показывают, что Spec Kit — это не только command prompts для агента: под ними есть executable path resolution, branch validation, порядок выбора шаблонов и JSON-выводы [scripts/bash/common.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/common.sh) [scripts/bash/create-new-feature.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/create-new-feature.sh) [scripts/bash/setup-plan.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/setup-plan.sh) [scripts/bash/setup-tasks.sh](https://raw.githubusercontent.com/github/spec-kit/main/scripts/bash/setup-tasks.sh). Для теории это важно как материальный мост между Markdown-методологией и реальным репозиторием.

Нужно уточнить, что этот слой работает на нескольких платформах. PowerShell-варианты сохраняют общий Spec Kit-порядок, но вводят отдельные условия: сравнение путей без учёта регистра на Windows, обработку gitflow-префикса ветки, `.ps1`-флаги и зависимость некоторых путей композиции presets от Python/PyYAML [scripts/powershell/common.ps1](https://raw.githubusercontent.com/github/spec-kit/main/scripts/powershell/common.ps1). В теории это помогает не сводить Spec Kit к Unix-ориентированной CLI-истории.

### Что лучше уйдёт в Handbook / Fieldbook

В Handbook подойдут практические рецепты:

- как инициализировать проект: `uvx --from git+https://github.com/github/spec-kit.git specify init <PROJECT_NAME>` или `specify init .`;
- когда выбирать `--script ps`, `--no-git`, `--branch-numbering timestamp`, `--preset <id>` и когда задавать `SPECIFY_FEATURE` для non-Git режима;
- как выбрать integration, в том числе Codex skills mode через `--integration codex --integration-options="--skills"`;
- минимальная цепочка для эксперимента и расширенная цепочка для production feature;
- как читать `Sync Impact Report` в `.specify/memory/constitution.md` после `/speckit.constitution`;
- как читать `spec.md`, `plan.md`, `tasks.md` перед переходом к следующей фазе;
- как использовать `/speckit.analyze` до реализации;
- как читать `FEATURE_DIR/checklists/` перед `/speckit.implement`, потому что незакрытые пункты создают отдельный gate;
- когда запускать `/speckit.taskstoissues`;
- как обновлять CLI и project files, не потеряв customized constitution.
- как проверять фактические templates/commands после установки extensions или presets через `specify extension list`, `.specify/extensions.yml`, `.specify/preset.yml` и `specify preset resolve <name>`.
- как использовать `specify check`, `specify version --features --json`, `specify workflow status` и `specify workflow resume` для проверки установленной среды и продолжения workflow.
- как проверять integration layer через `specify integration list`, когда использовать `integration use`, `integration switch`, `integration upgrade`, и почему `uvx` не обновляет persistent `specify` на `PATH`.
- как отличать supported integration от multi-install safe integration и почему перед `install`/`switch` нужно смотреть default integration, installed integrations, manifests и local modifications.
- как диагностировать нестандартные surfaces: `Hermes` global skills, `Goose` recipes, `RovoDev` wrappers/MCP config, `Pi` MCP setup и `generic --commands-dir`.
- как читать примечания по конкретной integration: формат команд, skills mode, fallback sections, способ запуска в среде агента, MCP prerequisites и migration notes для выбранного агента.
- как проверять source-level metadata integration: `key`, `context_file`, command output directory, `invoke_separator`, `multi_install_safe`, frontmatter/companion prompts и env overrides `SPECKIT_INTEGRATION_<KEY>_EXECUTABLE` / `SPECKIT_INTEGRATION_<KEY>_EXTRA_ARGS`.
- как безопасно обновлять Spec Kit: сначала `specify self check` / `specify self upgrade --dry-run`, затем CLI upgrade, затем `specify init --here --force --integration <agent>`, с отдельным сохранением customized `constitution.md`, `.specify/templates/` и `.specify/scripts/`.
- как читать `.specify/integrations/<key>.manifest.json` и `.specify/integrations/speckit.manifest.json`: отличать managed files от customized/untracked files, понимать warnings about skipped, preserved and symlinked paths, и решать, когда нужен `--force`.
- как проверять integration catalogs: `SPECKIT_INTEGRATION_CATALOG_URL`, `.specify/integration-catalogs.yml`, user-level config, `.specify/integrations/.cache/`, install-allowed status и предупреждение о non-default catalogs.
- как запускать `create-new-feature.sh --dry-run --json`, `setup-plan.sh --json` и `setup-tasks.sh --json`, чтобы увидеть реальные `FEATURE_DIR`, `FEATURE_SPEC`, `IMPL_PLAN`, `TASKS_TEMPLATE` и supporting docs до того, как агент начнёт менять содержательные файлы.
- как на Windows запускать `.specify/scripts/powershell/create-new-feature.ps1 -DryRun -Json`, `.specify/scripts/powershell/setup-plan.ps1 -Json` и `.specify/scripts/powershell/setup-tasks.ps1 -Json`, а также как проверять `specify version`, Python 3 и PyYAML, если в проекте используются presets с композиционными стратегиями.

В Fieldbook подойдут сценарии:

- feature с высокой неоднозначностью, где `/speckit.clarify` экономит переделку;
- большой репозиторий, где нужен read-only repo probe перед планом;
- команда с несколькими агентами, где Spec Kit integrations помогают удержать общий процесс;
- случай drift после реализации, где нужно вручную reconciliate spec/plan/tasks;
- маленький bugfix, где полный Spec Kit-процесс оказывается тяжёлым.
- проект, где preset или extension неожиданно меняет поведение command template, и команда должна восстановить, какой template реально исполнялся.
- проект без Git, где active feature приходится задавать через `SPECIFY_FEATURE`, а не через branch.
- upgrade или integration switch, где modified agent files должны сохраниться, а stale managed files удалиться.
- upgrade, где CLI обновили, но project command templates остались старыми, и агент продолжает работать по прежним `/speckit.*` files.
- project-file update, где `specs/` сохранились, но customized `constitution.md` или templates были overwritten из-за `--force`.
- проект внутри монорепозитория, где `.specify` должен быть корнем Spec Kit-процесса, а не родительский Git root.
- сбой, где агент неправильно вызывает generated script; разбор должен начинаться с help/output script и текущего command template, а не с общей идеи SDD.
- Windows-проект, где PowerShell script выбирает не тот feature number, не тот `FEATURE_DIR` или не применяет ожидаемую preset composition; разбор должен сравнить `.ps1` output, `.specify/feature.json`, Git branches/remote refs, Python/PyYAML и текущие files в `.specify/presets/`.
- проект, где integration выглядит установленной, но фактическая поверхность агента требует особого поведения: Codex skills mode, Kiro CLI fallback, Pi MCP setup или миграции с прежнего формата команд агента.
- проект, где integration запускается через переопределённый binary или extra args, и команда должна проверить env variables перед тем, как объяснять сбой содержанием `spec.md` или `plan.md`.

## Кандидаты на иллюстрации

- `https://github.github.com/spec-kit/` — первый экран документации и блоки `Spec-driven by default`, `Use any coding agent`, `Make it your own`. Может пригодиться для иллюстрации позиционирования Spec Kit как переносимого toolkit. Статус: кандидат; не использовать без проверки прав.
- `https://github.com/github/spec-kit` — дерево репозитория с `integrations`, `templates`, `extensions`, `presets`, `workflows`, `src/specify_cli`. Может показать материальную структуру методологии. Статус: кандидат; лучше пересобрать собственную схему вместо скриншота.
- `https://github.github.com/spec-kit/reference/workflows.html` — Mermaid/flowchart built-in `Full SDD Cycle`: `specify -> review-spec -> plan -> review-plan -> tasks -> implement`. Может пригодиться для раздела о human gates. Статус: кандидат; лучше перерисовать схему своими средствами.
- `https://github.github.com/spec-kit/reference/workflows.html` — JSON status example для paused workflow (`run_id`, `workflow_id`, `status`, `current_step_id`). Может пригодиться как иллюстрация machine-orchestrated SDD, где human gate выражен не только текстом, но и состоянием run. Статус: кандидат для собственной схемы.
- `templates/commands/*.md` из `github/spec-kit` — можно перерисовать собственную схему “command templates as workflow contract”: hooks -> command body -> artifacts -> after hooks. Статус: кандидат для собственной схемы, не скриншот.
- `templates/commands/constitution.md` — можно перерисовать схему “constitution update -> semver decision -> dependent templates -> Sync Impact Report”. Статус: кандидат для собственной схемы.
- `scripts/bash/common.sh` + `setup-plan.sh` + `setup-tasks.sh` — можно перерисовать схему “active feature resolution”: `.specify` root -> `SPECIFY_FEATURE_DIRECTORY` -> `.specify/feature.json` -> branch prefix lookup -> generated paths. Статус: кандидат для собственной схемы.
- `scripts/powershell/common.ps1` — можно перерисовать вариант той же схемы для Windows: `.specify` root -> Git worktree check -> `SPECIFY_FEATURE_DIRECTORY` -> `.specify/feature.json` -> branch prefix lookup -> сравнение путей без учёта регистра на Windows -> форматированная команда `/speckit.*` или `/speckit-*`. Статус: кандидат для собственной схемы.
- `src/specify_cli/integrations/base.py` + `AGENTS.md` из `github/spec-kit` — можно перерисовать схему “контракт реестра integrations”: `IntegrationBase.key/config/registrar_config/context_file` -> command output directory -> manifest tracking -> managed context file. Статус: кандидат для собственной схемы.
- `https://github.github.com/spec-kit/reference/integrations.html` — можно перерисовать lifecycle-схему integration management: `install` -> SHA-256 manifest -> `use`/`switch` -> shared templates aligned to default -> `upgrade` -> stale files removed / modified files preserved. Статус: кандидат для собственной схемы.
- `https://github.github.com/spec-kit/upgrade.html` — можно перерисовать схему “двухслойное обновление”: persistent CLI (`specify self upgrade`) отдельно от project files (`specify init --here --force`), с безопасной зоной `specs/` и рискованной зоной `.specify/memory/constitution.md`. Статус: кандидат для собственной схемы.
- `AGENTS.md` / `agent-context-config.yml` из `github/spec-kit` — можно перерисовать схему “integration class -> agent-context config -> managed section in AGENTS.md/CLAUDE.md/etc.”. Статус: кандидат для собственной схемы, не скриншот.
- `src/specify_cli/integrations/__init__.py` + несколько `src/specify_cli/integrations/<agent>/__init__.py` — можно сделать таблицу “integration key -> command directory -> context file -> invocation style -> special post-processing”. Это полезнее скриншота, потому что показывает, где именно Spec Kit перестаёт быть универсальным и становится agent-specific. Статус: кандидат для собственной схемы.
- `https://github.github.com/spec-kit/reference/integrations.html` — можно сделать отдельную схему “supported != multi-install safe”: supported table -> command surface (`slash`, `skill`, `recipe`, `wrapper`, `generic dir`) -> safe flag -> external setup notes. Это поможет не смешивать маркетинговое обещание переносимости с фактической интеграционной топологией. Статус: кандидат для собственной схемы.
- `https://github.github.com/spec-kit/reference/presets.html` — схема или таблица resolution stack из presets reference. Может пригодиться для объяснения, почему фактический Spec Kit-процесс зависит от overrides. Статус: кандидат; лучше пересобрать как свою диаграмму.
- `https://martinfowler.com/articles/exploring-gen-ai/sdd-3-tools.html` — диаграмма уровней SDD (`spec-first`, `spec-anchored`, `spec-as-source`) и overview spec-driven workspace. Уже есть локальный asset `content/assets/theory-images/fowler-sdd-overview.png`; можно использовать для контекста SDD, если права и attribution сохранены. Статус: кандидат / уже локально присутствует в корпусе.
- `https://arxiv.org/abs/2604.05278` — paper с figure по Spec Kit Agents context-grounding layer. Может пригодиться для различения базового Spec Kit и research-усиления через hooks. Статус: кандидат; нужно проверить PDF и права перед использованием.

## Открытые вопросы

- Насколько текущий Spec Kit CLI действительно поддерживает все команды, перечисленные в README и docs, для каждой integration; issues показывают, что в отдельных интеграциях возможны расхождения.
- Как в реальных проектах поддерживается актуальность `spec.md`, `plan.md`, `tasks.md` после `/speckit.implement` и ручных правок кода.
- Насколько надёжно разные integrations выполняют mandatory hooks из command templates: как реальная поверхность агента обрабатывает `EXECUTE_COMMAND`, `WAIT_FOR_COMPLETION` и HookExecutor conditions.
- Является ли constitution в современном Spec Kit настолько “immutable”, как описывает `spec-driven.md`, или это скорее историко-методологический пример из Specify-era текста.
- Какие extensions, presets и workflows можно считать достаточно зрелыми для теоретического текста, а какие лучше оставить как иллюстрации экосистемы.
- Как именно `.specify/feature.json`, текущая Git branch и `SPECIFY_FEATURE_DIRECTORY` согласуются в конфликтных случаях: какой механизм считать авторитетным для будущего практического описания.
- Как соотносится `SPECIFY_FEATURE` из core reference с `SPECIFY_FEATURE_DIRECTORY` из command template: это разные уровни override или следы разных версий механизма active feature.
- Как в будущей практической инструкции лучше проверять совпадение поведения bash и PowerShell scripts после обновления Spec Kit: сравнивать только documented flags недостаточно, потому что `common.ps1` содержит отдельные ветки для Windows-путей и композиции шаблонов.
- Должен ли будущий Handbook рекомендовать обязательный `--json` preflight для `setup-plan.sh` и `setup-tasks.sh` перед запуском агента в больших репозиториях.
- Как часто в реальных проектах `Sync Impact Report` после `/speckit.constitution` действительно синхронизирует dependent templates, а не остаётся формальным комментарием.
- Насколько изменение default поведения `git` extension в `v0.10.0` повлияет на quickstart-описания, которые пока предполагают Git branch как стандартный carrier feature context.
- Как в реальных проектах проверять, что `agent-context` extension не отключена и что managed section в `AGENTS.md`, `CLAUDE.md` или другом context file действительно свежая.
- Насколько часто ошибки установки связаны с неофициальными PyPI-пакетами или временным `uvx` вместо persistent `uv tool install`, и нужна ли для Handbook отдельная диагностическая карточка по `specify version` / `specify self check`.
- Насколько глубоко будущему Handbook нужно объяснять manifest layer: показывать только `.specify/integrations/<key>.manifest.json` как диагностический файл или разбирать SHA-256 records, symlink safety и preserved/skipped buckets.
- Как часто project-file upgrades реально перезаписывают customized `constitution.md`, и стоит ли в Handbook сделать обязательным pre-upgrade commit именно для `.specify/memory/constitution.md`, `.specify/templates/` и `.specify/scripts/`.
- Как отличать Spec Kit как spec-first/spec-anchored workflow от более радикального spec-as-source подхода; внешний обзор Thoughtworks считает границы SDD расплывчатыми.
- Насколько будущий Handbook должен углубляться в оговорки по конкретным агентам: достаточно общей команды `specify integration list` или нужны отдельные диагностические карточки для Codex skills, Kiro fallback, Pi MCP и других нестандартных поверхностей.
- Нужно ли в практической диагностике всегда выводить env overrides `SPECKIT_INTEGRATION_*_EXECUTABLE` и `SPECKIT_INTEGRATION_*_EXTRA_ARGS`, или достаточно проверять их только в CI/scripted workflows и при странном поведении agent invocation.
- Нужно ли следующему проходу читать source-level files для `devin`, `hermes`, `goose`, `rovodev`, `generic` и `pi`, или reference-level сведений достаточно для методологического досье.
- Как документировать integrations с глобальными или внешними командными поверхностями так, чтобы diff репозитория оставался проверяемым артефактом процесса.
- Какие права и attribution нужны для использования диаграмм из official docs, Martin Fowler/Thoughtworks и arXiv paper.
