# Kiro Specs

## Черновое назначение

Документ создан для накопления фактуры о Kiro Specs как методологическом режиме спецификационного управления разработкой. Это рабочее досье, а не готовая глава сайта. Его задача — сохранить устройство процесса, имена артефактов, места принятия решений человеком, ограничения и детали продуктовой реализации, чтобы позже можно было написать плотный раздел о Kiro в части про соседние спецификационные режимы.

В текущей архитектуре теории Kiro Specs нужен как пример продуктового спецификационного рабочего процесса внутри IDE: спецификация становится не внешним документом и не только набором команд, а частью поверхности разработки, выполнения задач и контекстной подстановки для агента. Это назначение зафиксировано в `work/specification-cluster-deep-plan.md` и `work/approved-ai-sdlc-plan.md`.

## Краткая рабочая карта

Kiro Specs лучше читать как продуктовую спецификационную дисциплину: поверхность IDE/Web/CLI пытается сделать спецификации не внешней практикой, а встроенным состоянием продукта. Ядро — Feature Specs с последовательностью `requirements.md → design.md → tasks.md → выполнение`; рядом находятся Bugfix Specs, Quick Plan и Analyze Requirements. Проход с открытием источников добавляет ещё два важных слоя: проверку требований через нейросимвольный анализ и проверку реализации через тесты на основе свойств. Поэтому Kiro находится не просто рядом со Spec Kit, а показывает другую траекторию: рабочий процесс с первичной спецификацией становится продуктовой средой, где требования, анализ, тесты, steering, hooks, веб-автономность и доставка PR постепенно собираются в одну платформу.

## Роль в SDLC с ИИ

Kiro помещает спецификацию между намерением человека и агентским исполнением внутри IDE. В базовом описании Specs называются способом разбить сложную фичу на требования, дизайн и задачи реализации, чтобы двигаться от идеи к работающему изменению через ревьюируемые промежуточные артефакты, а не через один длинный запрос к агенту. Источник: [Kiro Docs: Specs](https://kiro.dev/docs/specs/).

Для SDLC с ИИ это важно по двум причинам.

Во-первых, Kiro делает жизненный цикл видимым в продуктовой среде. Пользователь не просто просит агента “сделай фичу”; он получает набор файлов в `.kiro/specs/{spec-name}/`, где намерение, дизайн и задачи лежат отдельно и могут быть просмотрены до реализации. Источник: [Kiro Docs: Feature Specs](https://kiro.dev/docs/specs/feature-specs/).

Во-вторых, Kiro связывает спецификацию с исполнением. Задачи из `tasks.md` становятся единицами, которые агент может выполнять по одной; рядом с задачей меняется статус, а IDE показывает прогресс спецификации. Это превращает спецификацию в рабочий интерфейс для делегирования, а не только в справочный документ. Источник: [Kiro Docs: Feature Specs](https://kiro.dev/docs/specs/feature-specs/).

Kiro не нужно смешивать со Spec Kit. Spec Kit важен как переносимый набор инструментов с шаблонами, скриптами и рабочим процессом, управляемым командами. Kiro важен как встроенная IDE-поверхность: пользователь проходит требования, дизайн, задачи, выполнение задач, контекст `#spec`, Sync Files, hooks и steering внутри одного продукта. Это локальное различение нужно только для понимания Kiro; полноценное сравнение относится к отдельному проходу.

Внутри самого Kiro важно различать Vibe sessions и Spec sessions. Сессия Vibe рассчитана на быстрые вопросы, объяснения и исследовательскую работу в разговорном режиме; сессия Spec выбирается через переключатель сессий и ведёт сложную задачу через формализованный план, последовательное исполнение и отслеживание прогресса. Для методологии это уточняет границу применения: Kiro не заставляет каждую работу проходить через Specs, но выделяет Specs как режим для сложных фич, существенного рефакторинга, командной документации и задач, где требования и дизайн требуют итераций. Источник: [Vibe vs Spec sessions](https://kiro.dev/docs/chat/vibe/).

## Проблема, которую решает Kiro Specs

Kiro явно отталкивается от ограничения “vibe coding”: быстрый интерактивный агентский режим удобен для прототипов, но теряет устойчивость, когда фича становится сложной, требует нескольких файлов, архитектурных решений, пограничных случаев и последовательной реализации. В представлении Kiro Specs должны сохранить скорость разработки с AI-помощью, но добавить структуру: сначала выяснить требования, затем зафиксировать технический дизайн, затем разбить реализацию на задачи. Источники: [Introducing Kiro](https://kiro.dev/blog/introducing-kiro/) и [Kiro Docs: Specs](https://kiro.dev/docs/specs/).

Практическая проблема выглядит так: агент может быстро написать код, но плохо удерживает полный набор требований, скрытые зависимости, ограничения существующей архитектуры и последовательность безопасных изменений. Kiro переносит часть этой нагрузки в файлы спецификации. Требования становятся проверяемым списком поведения, дизайн — местом архитектурного решения, задачи — очередью реализации. Источник: [Kiro Docs: Feature Specs](https://kiro.dev/docs/specs/feature-specs/).

Отдельная проблема — уже существующий код. Документация Kiro допускает создание спецификации не только для новой фичи, но и через анализ существующих требований или текущей кодовой базы. В режиме Analyze Requirements Kiro должен изучить существующие документы или код, найти неясности, противоречия, неполные требования и предложить дополнительные вопросы или уточнения. Источник: [Kiro Docs: Analyze Requirements](https://kiro.dev/docs/specs/analyze-requirements/).

Блог Kiro про анализ требований уточняет, какие именно ошибки требований особенно опасны для агентской разработки: неверный уровень детализации, неоднозначность, противоречивость и неполнота. Важная для теории деталь: авторы связывают рост риска именно с инженерией с AI-помощью, потому что начальный запрос фактически становится требованием, а агент быстро превращает недосказанность в кодовые решения, которые человек может не успеть заметить. Источник: [Requirements analysis: catching requirement bugs before they become code](https://kiro.dev/blog/deep-spec-analysis/).

Там же приведены численные ориентиры из внешних исследований: при мутации ясных запросов в неоднозначные, неполные или противоречивые Pass@1 падал на 20-40%, а 60-90% синтаксически валидного кода оказывалось семантически неверным; отдельная работа по underspecification описывает, что недоопределённые запросы примерно вдвое чаще регрессируют при смене модели или запроса. Эти цифры не нужно превращать в универсальную метрику Kiro, но они полезны как обоснование, почему спецификационный слой нужен до генерации кода. Источник: [Deep Spec Analysis](https://kiro.dev/blog/deep-spec-analysis/).

## Базовый рабочий процесс Feature Specs

Feature Spec — основной сценарий Kiro для новой фичи. Он строится вокруг трёх файлов:

- `requirements.md`;
- `design.md`;
- `tasks.md`.

Все они создаются в `.kiro/specs/{spec-name}/`. Источник: [Kiro Docs: Feature Specs](https://kiro.dev/docs/specs/feature-specs/).

Feature Specs поддерживают два варианта рабочего процесса: Requirements-First и Design-First. В Requirements-First пользователь начинает с поведения системы, Kiro формализует его в требования, затем строит дизайн и задачи. В Design-First пользователь начинает с технической архитектуры, псевдокода, низкоуровневого дизайна или уже существующего документа дизайна; Kiro сначала создаёт `design.md`, затем выводит требования, которые должны быть технически осуществимы в рамках подтверждённой архитектуры, и после этого создаёт задачи. Источники: [Feature Specs](https://kiro.dev/docs/specs/feature-specs/), [Requirements-First Workflow](https://kiro.dev/docs/specs/feature-specs/requirements-first/) и [Design-First Workflow](https://kiro.dev/docs/specs/feature-specs/tech-design-first/).

### 1. Требования

Первый шаг в Requirements-First — формулирование требований. Kiro просит описать фичу в чате или через интерфейс Specs и затем генерирует `requirements.md`. В документации требования описаны через пользовательские истории, критерии приёмки и нотацию EARS. Базовая форма требования выглядит как `WHEN [condition/event] THE SYSTEM SHALL [expected behavior]`. Для будущей теории важно сохранить именно эту деталь: Kiro не ограничивается произвольным “описанием фичи”, а пытается получить поведенческую форму, которую можно ревьюить и затем трассировать к реализации. Источник: [Kiro Docs: Feature Specs](https://kiro.dev/docs/specs/feature-specs/).

У требований есть человеческая контрольная точка. Пользователь должен открыть `requirements.md`, проверить полноту и точность, при необходимости продолжить диалог с Kiro и подтвердить переход дальше. В нормальном Feature Spec Kiro не должен автоматически перескакивать к дизайну без проверки. Источник: [Kiro Docs: Feature Specs](https://kiro.dev/docs/specs/feature-specs/).

### 2. Дизайн

После подтверждения требований Kiro создаёт `design.md`. Этот файл должен описать архитектурный подход, компоненты, интерфейсы, модели данных, обработку ошибок и стратегию тестирования. Для досье важен сам переход: `requirements.md` становится входом для `design.md`, а дизайн должен объяснить, как требования будут реализованы технически. Источник: [Kiro Docs: Feature Specs](https://kiro.dev/docs/specs/feature-specs/).

Здесь возникает вторая человеческая контрольная точка. Пользователь просматривает дизайн, уточняет спорные решения и только затем разрешает перейти к задачам. Это место, где человек должен отловить архитектурно неверное направление до того, как агент начнёт массово менять код. Источник: [Kiro Docs: Feature Specs](https://kiro.dev/docs/specs/feature-specs/).

В Design-First этот порядок разворачивается: дизайн становится первым артефактом, а требования выводятся из подтверждённой архитектуры. Kiro различает высокоуровневый дизайн (High Level Design) и низкоуровневый дизайн (Low Level Design). Высокоуровневый дизайн подходит для системной архитектуры, компонентов, взаимодействий и нефункциональных свойств; низкоуровневый дизайн — для алгоритмических деталей, интерфейсов, контрактов и структур данных. Источник: [Design-First Workflow](https://kiro.dev/docs/specs/feature-specs/tech-design-first/).

### 3. Задачи

После подтверждения дизайна Kiro создаёт `tasks.md`: упорядоченный список задач реализации. Важная особенность — задачи должны быть достаточно малыми, чтобы их можно было выполнять последовательно и отслеживать статус. В Requirements-First документации для `tasks.md` отдельно названы дискретные отслеживаемые задачи, описания и ожидаемые результаты, зависимости между задачами и необязательные/обязательные задачи. Это делает этап задач не только списком шагов, но и местом, где человек может отделить обязательный MVP-путь от дополнительных задач. Источник: [Requirements-First Workflow](https://kiro.dev/docs/specs/feature-specs/requirements-first/).

Здесь есть третья контрольная точка: пользователь смотрит разбиение задач, меняет приоритеты, отмечает необязательные задачи и подтверждает, что порядок, детализация и охват подходят для реализации. После этого задачи можно запускать из интерфейса Kiro. Источник: [Requirements-First Workflow](https://kiro.dev/docs/specs/feature-specs/requirements-first/).

### 4. Выполнение задач

На этапе реализации пользователь выбирает задачу из `tasks.md`, а Kiro выполняет её с учётом контекста спецификации. Статус задачи меняется по мере выполнения. В будущей теории это нужно описывать как продуктовую единицу делегирования: единицей поручения становится не весь запрос и не вся фича, а конкретная задача из утверждённой спецификации. Источник: [Kiro Docs: Feature Specs](https://kiro.dev/docs/specs/feature-specs/).

Общая страница Specs уточняет интерфейсные статусы выполнения задач: для `tasks.md` Kiro показывает обновления статуса в реальном времени, а задачи обновляются как `in-progress` или `completed`. Официальная стенограмма на главной странице Kiro показывает ту же механику на примере спецификации `email-opt-in`: у задачи `Set up backend API foundation` действие `Start task` переходит в `Task in progress`, затем в зелёный `Task completed`; флажок становится отмеченным, рядом появляются `View changes` и `View execution`, а в панели чата видна карточка статуса `Status: Completed`. Это закрывает часть вопроса о видимых статусах, но не доказывает, что статус хранится прямо в `tasks.md`, а не в метаданных IDE. Источники: [Kiro Docs: Specs](https://kiro.dev/docs/specs/) и [Kiro home page transcript](https://kiro.dev/).

Kiro может запускать все задачи разом. В этом режиме он строит граф зависимостей из `tasks.md`, группирует независимые задачи в волны и выполняет задачи внутри одной волны параллельно, а волны — последовательно. Это важная деталь: список задач в Kiro не просто контрольный список, а потенциальный план параллельного исполнения с зависимостями. Источник: [Kiro Docs: Specs](https://kiro.dev/docs/specs/).

Пост `Specs just got faster (and smarter)` раскрывает правила параллельного исполнения подробнее. Kiro не запускает параллельно задачи, которые пишут в одни и те же файлы; настройка и инфраструктурные работы идут первыми; тестовые задачи ждут код, который они проверяют; каждая параллельная задача получает изолированный контекст, чтобы состояние одной задачи не протекало в другую. Если одна задача падает, остальные независимые задачи продолжают выполняться, а пользователь видит, какие задачи завершились и какие требуют внимания. Источник: [Specs just got faster (and smarter)](https://kiro.dev/blog/faster-smarter-specs/).

Более ранний пост про `Run all Tasks` важен как история продуктового ограничения. Kiro изначально не давал кнопку запуска всех задач, потому что команда хотела сохранить видимость и контроль: пользователь должен был видеть исполнение каждой задачи и ловить ошибки рано. Кнопку добавили только после появления дополнительных опор: тестов на основе свойств, серверов разработки, диагностик LSP, subagents и видимости в реальном времени. В том же посте Kiro ограничивает сильный сценарий: `Run all Tasks` особенно уместен для небольших Feature Specs, а предварительная проверка спецификации остаётся обязательной частью работы. Источник: [Run all tasks: the feature we refused to ship](https://kiro.dev/blog/run-all-tasks/).

По рекомендациям Specs Best Practices `Run all Tasks` запускает только незавершённые задачи, которые помечены как обязательные. Это важно для чтения `tasks.md`: необязательные задачи не должны автоматически становиться частью пакетного запуска, а уже реализованные задачи нужно предварительно синхронизировать или явно пометить. Источник: [Specs Best Practices](https://kiro.dev/docs/specs/best-practices/).

Kiro также поддерживает выполнение задач через hooks: автоматические действия до начала задачи или после её завершения. Это связывает Specs с более широкой автоматизацией IDE. Например, hook перед задачей может подготовить окружение или подтянуть контекст, а hook после задачи — запустить форматирование, тесты или другую проверку. Источник: [Kiro Docs: Hooks](https://kiro.dev/docs/hooks/) и [Kiro Docs: Hook Triggers](https://kiro.dev/docs/hooks/types/).

Отдельный слой контроля появляется уже во время исполнения задачи: Kiro различает режимы Autopilot и Supervised. В Autopilot агент может менять файлы и выполнять многошаговую работу без подтверждения каждого действия; пользователь сохраняет контроль через просмотр всех изменений, Revert и возможность остановить исполнение. В Supervised Kiro останавливается после каждого хода с файловыми изменениями и показывает правки для подтверждения; этот режим работает и в обычном чате, и в сессиях чата спецификации. Для Specs это важная деталь: контрольные точки проверки перед `requirements.md`, `design.md` и `tasks.md` не являются единственной точкой человеческого контроля; после старта реализации человек может дополнительно включить проверку изменений на уровне файлов или отдельных фрагментов правок. Источник: [Kiro Docs: Autopilot](https://kiro.dev/docs/chat/autopilot/).

## Bugfix Specs

Bugfix Specs используют похожую трёхфазную структуру, но первая фаза меняется. Вместо `requirements.md` появляется `bugfix.md`. В нём фиксируются текущее поведение, ожидаемое поведение, шаги воспроизведения, область действия и ограничения. Затем Kiro создаёт `design.md` и `tasks.md`. Источник: [Kiro Docs: Bugfix Specs](https://kiro.dev/docs/specs/bugfix-specs/).

Это важное отличие: Kiro признаёт, что исправление дефекта требует другого входного артефакта, чем новая фича. Для bugfix спецификация начинается с диагностики расхождения между текущим и ожидаемым поведением, а не с пользовательской истории. Это делает Kiro полезным не только для планирования новых возможностей, но и для рабочего процесса исправления.

В документации Bugfix Specs отдельно выделено неизменяемое поведение (`unchanged behavior`): спецификация должна фиксировать, что должно продолжить работать после исправления. В `bugfix.md` это выражается формой вроде `WHEN [condition] THEN the system SHALL CONTINUE TO [existing behavior]`. Затем `design.md` должен включить анализ первопричины, предлагаемый подход к исправлению и свойства, которые нужно проверить: дефект воспроизводится в текущей реализации, исправленная реализация даёт правильное поведение, неизменяемое поведение продолжает работать. Фаза задач генерирует задачи реализации с тестами на основе свойств для воспроизведения бага, проверки исправления и предотвращения регрессий. Источник: [Bugfix Specs](https://kiro.dev/docs/specs/bugfix-specs/).

В SDLC с ИИ это показывает, что спецификационный слой может обслуживать разные типы изменения. Для новой фичи нужен файл намерений и критериев приёмки; для бага — файл воспроизведения и границ исправления. Обе ветки сходятся в `design.md` и `tasks.md`, потому что реализация всё равно требует технического решения и последовательного выполнения.

## Quick Plan

Quick Plan — укороченный режим для понятных или малых задач. Он создаёт те же артефакты, что и стандартная Spec session, но делает это за один проход: Kiro задаёт уточняющие вопросы заранее, затем генерирует `requirements.md`, `design.md` и `tasks.md` без контрольных точек утверждения между фазами. Источник: [Kiro Docs: Quick Plan](https://kiro.dev/docs/specs/quick-plan/).

Для досье особенно важна граница применимости. Quick Plan не отменяет Specs; он показывает, что Kiro не требует полного `requirements.md` → `design.md` → `tasks.md` цикла для каждой мелкой операции. Если задача понятна, риск низкий, а пользователь не нуждается в отдельном ревью требований и дизайна, Quick Plan уменьшает накладные расходы. Источник: [Kiro Docs: Quick Plan](https://kiro.dev/docs/specs/quick-plan/).

Блог Kiro уточняет, что перед вопросами Quick Plan сканирует рабочее пространство, чтобы определить язык, фреймворки и структуру проекта, а затем задаёт вопросы по четырём областям: границы и ограничения, неоднозначности в описании, развилки реализации и направляющие решения о форме фичи. После генерации пользователь сразу попадает к списку задач, но артефакты всё равно сохраняются и используются агентом при исполнении. Если пользователь меняет задачу, Kiro регенерирует только задачи; если меняет дизайн, перестраивает дизайн и задачи; если меняет границы, запускает цепочку от требований заново. Источник: [Specs just got faster (and smarter)](https://kiro.dev/blog/faster-smarter-specs/).

С точки зрения режимов отказа это двойственная деталь. Quick Plan полезен как защита от чрезмерного использования полного рабочего процесса спецификации. Но он же может стать способом преждевременно обойти важные контрольные точки, если задача кажется простой только на поверхности. В будущей теории стоит показать, что Kiro не даёт универсального ответа “всегда писать спецификацию”; он вводит несколько режимов, и ответственность за выбор режима остаётся у человека.

## Analyze Requirements

Analyze Requirements — режим, где Kiro помогает не сразу создавать спецификацию, а сначала анализирует существующие материалы. В документации речь идёт о том, что Kiro может читать текущие документы требований или кодовую базу, выявлять пробелы, неоднозначности и противоречия, а затем предлагать уточнения. Источник: [Kiro Docs: Analyze Requirements](https://kiro.dev/docs/specs/analyze-requirements/).

Это важный слой перед обычным Feature Spec. В сложной организации требования редко приходят как чистый запрос. Они могут быть разбросаны по тикетам, документации, старому коду и разговорам. Analyze Requirements превращает Kiro в помощника по нормализации входа: до проектирования он должен помочь человеку увидеть, какие требования ещё не готовы для реализации.

Человеческая контрольная точка здесь находится в ответе на уточнения. Если Kiro нашёл неоднозначность, человек должен решить, какое поведение действительно нужно. Иначе спецификация будет выглядеть полной, но закрепит неверное понимание. Для будущего текста это хороший пример того, что спецификационный рабочий процесс не устраняет суждение; он делает места суждения более явными.

Документация подчёркивает, что этот анализ занимает минуты, а не секунды, потому что Kiro рассуждает по всему набору требований. Находки приходят как уточняющие вопросы: для каждого вопроса указываются затронутые требования, объяснение проблемы и предложенные исправления; человек может выбрать исправление, дать собственный ответ или отклонить вопрос, если неоднозначность намеренная. По мере разрешения вопросов Kiro обновляет `requirements.md`. Источник: [Analyze Requirements](https://kiro.dev/docs/specs/analyze-requirements/).

`Deep Spec Analysis` раскрывает внутреннюю модель этого шага как три стадии. Сначала уточнение доводит грубые критерии приёмки до уровня, где названы события, входы, состояния, выходы и условия между ними; при этом Kiro ищет неправильное использование EARS-паттернов, расплывчатые уточнители и язык уровня реализации. Затем автоформализация переводит требования в SMT-lib: схема с сущностями, атрибутами, событиями, входами и выходами, assertions как логическими импликациями и фоновыми утверждениями для неявных ограничений домена. Третья стадия — логический анализ, где автоматизированное рассуждение ищет противоречия, пробелы, принимаемые сценарии и отклоняемые сценарии. Источник: [Deep Spec Analysis](https://kiro.dev/blog/deep-spec-analysis/).

Для досье особенно важен интерфейс находки. Технические находки не показываются пользователю как формальная логика; Kiro переводит их в короткие вопросы с двумя вариантами ответа. В посте перечислены пять типов находки: вопросы о неоднозначности, вопросы о конфликте, вопросы о полноте, вопросы о принимаемых сценариях и вопросы об отклоняемых сценариях. Вариант A означает оставить требование как есть, вариант B — изменить его конкретным предложенным способом; после ответа LLM переписывает затронутые требования так, чтобы выбор человека стал явным в `requirements.md`. Источник: [Deep Spec Analysis](https://kiro.dev/blog/deep-spec-analysis/).

Ограничение этого механизма тоже важно сохранить. Автоформализация зависит от того, насколько хорошо явная спецификация и фоновые утверждения описывают домен. В посте прямо сказано, что самая перспективная зона для улучшения — построение хороших формальных доменных моделей: без явной модели возможного и невозможного механизм рассуждения может находить проблемы в сценариях, которые невозможны в реальном домене. Текущая реализация частично компенсирует это множественной выборкой фоновых утверждений о знаниях домена и частотным порогом, но не снимает проблему полностью. Источник: [Deep Spec Analysis](https://kiro.dev/blog/deep-spec-analysis/).

Уточнение после открытия источников: глубокий анализ требований у Kiro устроен не как “LLM перечитал требования и дал советы”. Публичный подробный разбор описывает три стадии: уточнение делает тезисные критерии приёмки проверяемыми и свободными от решений реализации, автоформализация переводит требования в формальную модель, а логический анализ ищет противоречия, пробелы, принимаемые и отклоняемые сценарии [Kiro deep requirements analysis](https://kiro.dev/blog/deep-spec-analysis/). Стадия refinement отдельно проверяет неверное использование EARS-паттернов, расплывчатые уточнители и язык уровня реализации. Это важная фактура: Analyze Requirements — не просто расширенное ревью, а попытка превратить требования в материал, пригодный для формального анализа.

Блог “Specs just got faster” формулирует пользовательскую поверхность проще: после генерации требований можно выбрать `Analyze Requirements`; Kiro ищет неоднозначность через несколько интерпретаций требований и использует автоматизированное рассуждение, чтобы проверить согласованность и пробелы [Specs just got faster](https://kiro.dev/blog/faster-smarter-specs/). Для досье это усиливает различие между Feature Specs как потоком создания и Analyze Requirements как слоем корректности и неоднозначности поверх требований.

## Контекст: где живёт знание

В Kiro контекст распределён по нескольким слоям.

Первый слой — файлы конкретной спецификации в `.kiro/specs/{spec-name}/`: `requirements.md` или `bugfix.md`, `design.md`, `tasks.md`. Это локальный контекст изменения. Источник: [Kiro Docs: Feature Specs](https://kiro.dev/docs/specs/feature-specs/) и [Kiro Docs: Bugfix Specs](https://kiro.dev/docs/specs/bugfix-specs/).

Второй слой — файлы steering в `.kiro/steering/`. Steering задаёт долгоживущий проектный контекст: обзор продукта, технологический стек, структура проекта, соглашения и другие правила, которые должны повторно использоваться в разных задачах. Kiro может сгенерировать три фундаментальных файла: `product.md`, `tech.md` и `structure.md`; они по умолчанию включаются в каждое взаимодействие. Steering бывает уровня рабочего пространства в `.kiro/steering/` и глобального уровня в `~/.kiro/steering/`; при конфликте steering рабочего пространства имеет приоритет. Источник: [Kiro Docs: Steering](https://kiro.dev/docs/steering/).

Руководство `Your first project` показывает, что генерация steering не является абстрактной анкетой: при `Generate Steering Docs` Kiro анализирует репозиторий и читает файлы вроде `package.json`, `README.md`, `vite.config.js`, `App.tsx` и `CartContext.tsx`, после чего складывает проектный steering в `.kiro/steering/`. Для методологии это значит, что долговременный контекст можно первично собрать из фактической структуры проекта, но его всё равно нужно ревьюить как сгенерированную документацию. Источник: [Your first project](https://kiro.dev/docs/getting-started/first-project/).

Файлы steering имеют режимы включения: `always`, `fileMatch`, `manual` и `auto`. Это важно для контекстной экономики: не вся проектная память должна грузиться всегда. Стандарты всего рабочего пространства можно держать всегда включёнными, правила для конкретных файлов включать по паттернам, тяжёлые справочные документы подключать вручную или через auto-описание. Источник: [Steering](https://kiro.dev/docs/steering/).

Третий слой — поставщики контекста чата. Для Specs важен `#spec`: пользователь может выбрать спецификацию из списка или сослаться на конкретную спецификацию вроде `#spec:user-authentication`. Kiro автоматически включает все файлы спецификации — `requirements.md`, `design.md` и `tasks.md` — в контекст разговора. Это позволяет обсуждать реализацию, уточнения, проверку соответствия и причины архитектурных решений, не пересказывая спецификацию вручную. Источник: [Kiro Docs: Chat](https://kiro.dev/docs/chat/).

Отдельно рядом со Specs стоит IDE MCP. MCP добавляет в Kiro внешние инструменты, шаблоны запросов и шаблоны ресурсов; они могут появляться в `#` списке упоминаний, а инструменты можно вызывать по смысловому запросу или явно в форме вроде `#[aws-docs] search_documentation ...`. Для Specs это важно в двух местах: на входе, когда требования или дизайн импортируются из внешних систем, и на исполнении, когда задача может опираться на внешнюю документацию, API или доменную базу знаний. Источники: [MCP](https://kiro.dev/docs/mcp/) и [MCP Tools](https://kiro.dev/docs/mcp/usage/).

У IDE MCP есть собственная модель управления риском. Конфигурация лежит на уровне рабочего пространства в `.kiro/settings/mcp.json` и на уровне пользователя в `~/.kiro/settings/mcp.json`; настройки рабочего пространства имеют приоритет. Для каждого сервера можно задавать `autoApprove` и `disabledTools`, временно отключать сервер или отдельные инструменты, а Kiro перед запуском MCP-инструмента показывает пользователю запрос подтверждения с деталями инструмента и параметрами. Если MCP-сервер просит дополнительные данные во время выполнения, Kiro показывает форму уточнения или URL-запрос, а пользователь может отправить ответ, отклонить запрос или закрыть его. Источники: [MCP Configuration](https://kiro.dev/docs/mcp/configuration/) и [MCP Tools](https://kiro.dev/docs/mcp/usage/).

Steering можно связывать с живыми файлами рабочего пространства через markdown-ссылки вида `#[[file:<relative_file_name>]]`. Это важная практическая деталь: steering не обязан быть только ручным пересказом стандартов, он может ссылаться на `api/openapi.yaml`, `.env.example`, UI-паттерны или другие текущие файлы проекта. Kiro также поддерживает `AGENTS.md` как директиву steering; такие файлы можно положить в корень рабочего пространства или в `~/.kiro/steering/`, но в отличие от файлов steering Kiro они всегда включаются и не поддерживают режимы включения. Источник: [Kiro Docs: Steering](https://kiro.dev/docs/steering/).

Четвёртый слой — Sync Files. Kiro может синхронизировать файлы спецификации с текущим состоянием проекта, в том числе учитывать уже выполненные задачи. Для досье это важный хвост жизненного цикла: спецификация не обязана умереть после первого шага генерации; её можно приводить в соответствие с изменившейся реальностью проекта. Источник: [Kiro Docs: Specs Best Practices](https://kiro.dev/docs/specs/best-practices/).

Specs Best Practices дают конкретный сценарий для уже реализованных задач: если коллега или сам пользователь выполнил часть работы в другой сессии, можно открыть `tasks.md` и нажать `Sync Files`, чтобы Kiro автоматически отметил завершённые задачи. Альтернатива — в чате спецификации попросить Kiro проверить, какие задачи уже завершены; тогда он анализирует кодовую базу, выявляет реализованную функциональность и обновляет спецификацию задач. Источник: [Specs Best Practices](https://kiro.dev/docs/specs/best-practices/).

Пятый слой — CLI-поверхность вокруг той же `.kiro` конфигурации, но с важной оговоркой. Официальный блог о Kiro CLI говорит, что конфигурация `.kiro` работает и в IDE, и в CLI: Kiro CLI может использовать те же файлы steering и MCP-серверы, а проектная документация и конфигурацию `.kiro` переносятся между средами. Это не означает, что CLI-документация описывает тот же визуальный рабочий процесс Specs с контрольными точками, но означает, что долговременный контекст Specs-соседних механизмов не ограничен IDE. Источник: [Bring Kiro agents to your terminal with Kiro CLI](https://kiro.dev/blog/introducing-kiro-cli/).

Для CLI особенно важна разница между обычными файлами steering и пользовательскими агентами. CLI steering в `.kiro/steering/` становится доступен в чат-сессиях CLI, но документация отдельно предупреждает: пользовательские агенты не получают файлы steering автоматически; их нужно явно добавить в `resources`, например `file://.kiro/steering/**/*.md`. Для методологии это существенная неудобная деталь: если команда создаёт специализированного агента для ревью, backend или тестов, проектная память может не попасть к нему сама по себе. Источник: [Kiro CLI: Steering](https://kiro.dev/docs/cli/steering/).

Шестой слой — Powers в IDE. Это не часть основного порядка Specs, но соседний механизм, который влияет на то, какие знания и инструменты агент получает при исполнении задач, связанных со спецификацией. Powers упаковывают `POWER.md`, конфигурацию MCP-сервера и, опционально, steering/hooks; Kiro активирует релевантные powers по ключевым словам в разговоре, вместо того чтобы заранее загружать все описания MCP-инструментов. В документации проблема описана через перегрузку контекста: несколько MCP-серверов могут загрузить сотни описаний инструментов и занять значительную часть окна контекста ещё до первого запроса. Для Kiro Specs это важно как ещё один контекстный слой рядом со steering и hooks: он может усилить реализацию и проверки, но его автоматическая активация не заменяет явное ревью файлов спецификации. Источники: [Kiro Docs: Powers](https://kiro.dev/docs/powers/) и [Create powers](https://kiro.dev/docs/powers/create/).

Практическая деталь Powers: если power включает MCP, при установке Kiro регистрирует MCP-сервер в `~/.kiro/settings/mcp.json` в секции Powers; custom power минимально требует `POWER.md`, а `mcp.json` и `steering/` добавляются по необходимости. Это делает Powers удобной упаковкой доменной экспертизы, но также создаёт ещё один источник изменения доступных инструментов и контекста, который команда должна понимать при разборе поведения агента. Источники: [Install powers](https://kiro.dev/docs/powers/installation/) и [Create powers](https://kiro.dev/docs/powers/create/).

Седьмой слой — корпоративное управление. Для методологии это не фаза Specs, а организационная рамка, которая заранее ограничивает, какие модели, MCP-серверы, ключи API и веб-инструменты вообще доступны агенту в IDE и CLI. В консоли Kiro администратор может включить управление доступом к моделям, собрать список разрешённых моделей, назначить модель по умолчанию, полностью выключить MCP или подключить реестр MCP с проверенными серверами, разрешить или запретить генерацию ключей API для автоматизации CLI и выключить `web_search`/`web_fetch`. Источник: [Governance](https://kiro.dev/docs/enterprise/governance/).

Управление MCP особенно важно для Specs, потому что внешние инструменты могут стать входом требований, источником доменного знания или частью исполнения задачи. Официальная документация описывает два уровня: переключатель MCP on/off и MCP registry URL в Kiro Profile. Реестр — это JSON-файл, который обслуживается по HTTPS; клиенты Kiro получают его при старте и затем синхронизируют примерно раз в 24 часа. Если сервер удалён из реестра, Kiro прекращает его использование и не даёт добавить обратно; если версия в реестре изменилась, клиент перезапускает сервер с версией, заданной реестром. У этого контроля есть граница: документация прямо предупреждает, что проверка работает на стороне клиента и пользователь с административным доступом к машине может её обойти. Источник: [MCP Governance](https://kiro.dev/docs/enterprise/governance/mcp/).

Документация CLI registry добавляет неудобную деталь: даже при активном реестре персональные MCP-серверы из `mcp.json` могут загружаться рядом с серверами из управляемого реестра, а выбор через `/mcp add` показывает только записи реестра. Пользователь также может задавать локальные переопределения реестра для `env`, `headers` и `timeout`; реестр задаёт основу сервера, но ключи аутентификации, командные заголовки и `timeout` могут оставаться локальной настройкой. Для будущего текста это важно как граница между централизованной политикой и фактической конфигурацией среды разработки. Источник: [Kiro CLI: MCP Registry](https://kiro.dev/docs/cli/mcp/registry/).

Восьмой слой — корпоративный мониторинг и отслеживание. Это не часть внутренней цепочки Specs, но важный слой наблюдаемости вокруг IDE/CLI-работы: администраторы могут включить дашборд агрегированных метрик, отчёты активности по пользователям, журналирование запросов в S3, а также опираться на AWS CloudTrail и Amazon CloudWatch для событий и метрик AWS-аккаунта. Для Kiro Specs это закрывает отдельный вопрос происхождения и наблюдаемости: часть работы становится видимой не только через `requirements.md`, `design.md`, `tasks.md`, diff и hooks, но и через административные журналы использования. При этом эти журналы не являются моделью трассировки спецификации: дашборд и отчёты активности по пользователям говорят о подписках, клиентах, сообщениях, credits, моделях и некоторых старых метриках принятия/отклонения кода, а журналы запросов фиксируют запросы, ответы и встроенные подсказки; они не заменяют чтение файлов спецификации и ревью истории выполнения. Источники: [Monitoring and tracking](https://kiro.dev/docs/enterprise/monitor-and-track/), [Viewing Kiro usage on the dashboard](https://kiro.dev/docs/enterprise/monitor-and-track/dashboard), [Viewing per-user activity](https://kiro.dev/docs/enterprise/monitor-and-track/user-activity/) и [Logging user prompts](https://kiro.dev/docs/enterprise/monitor-and-track/prompt-logging/).

Слой данных и приватности нужно держать рядом с мониторингом. Документация Kiro по защите данных говорит, что Kiro хранит вопросы, ответы и дополнительный контекст вроде кода для генерации ответов; для Free Tier и индивидуальных подписчиков контент хранится в `us-east-1`, а для корпоративных пользователей источник формулирует более жёсткую границу: `data is not stored`. Там же указано, что корпоративный контент не используется для улучшения сервиса, корпоративные пользователи автоматически исключены из сбора телеметрии и контента AWS, а настройки телеметрии для отчётов активности пользователей контролирует администратор в консоли Kiro. Для методологии это важная оговорка: если Specs используются в корпоративном контуре, контроль доступа, журналирование и хранение данных становятся частью процесса внедрения, а не только вопросом удобства IDE. Источник: [Data protection](https://kiro.dev/docs/privacy-and-security/data-protection/).

## Артефакты

Основные артефакты Kiro Specs:

- `.kiro/specs/{spec-name}/requirements.md` — требования для новой фичи: пользовательские истории, критерии приёмки, уточнённое поведение.
- `.kiro/specs/{spec-name}/bugfix.md` — входной файл для исправления дефекта: текущее поведение, ожидаемое поведение, воспроизведение, область действия.
- `.kiro/specs/{spec-name}/design.md` — технический дизайн: архитектура, компоненты, модели данных, интерфейсы, обработка ошибок, стратегия тестирования.
- `.kiro/specs/{spec-name}/tasks.md` — упорядоченный список задач реализации с ожидаемыми результатами, зависимостями и разделением на необязательные и обязательные задачи.
- импортированные источники требований — существующие PRD, JIRA-тикеты, страницы Confluence, GitHub issues, артефакты из подключённых через MCP систем или загруженные документы, из которых Kiro может создать требования в режиме Requirements-First.
- импортированные источники дизайна — существующий дизайн-документ, архитектурная диаграмма, псевдокод, низкоуровневый дизайн или технический документ, из которого Kiro может начать Design-First spec; Specs Best Practices отдельно допускают PNG/JPG-диаграммы, экспорт из draw.io или Lucidchart и фотографии эскизов на доске.
- `.kiro/steering/product.md`, `.kiro/steering/tech.md`, `.kiro/steering/structure.md` — фундаментальные файлы steering для продуктового, технологического и структурного контекста проекта.
- другие файлы в `.kiro/steering/` или `~/.kiro/steering/` — долгоживущие проектные правила и контекст, которые Kiro может использовать между спецификациями.
- `AGENTS.md` — дополнительный формат директив steering, который Kiro подхватывает из корня рабочего пространства или из `~/.kiro/steering/`, но включает всегда.
- ссылки `#[[file:<relative_file_name>]]` внутри файлов steering — способ привязать steering к живым файлам рабочего пространства.
- `.kiro/settings/mcp.json` и `~/.kiro/settings/mcp.json` для IDE MCP — конфигурация внешних инструментов, шаблонов запросов и ресурсов на уровне рабочего пространства и пользователя; через `autoApprove` и `disabledTools` команда управляет тем, что Kiro может вызывать без повторного подтверждения и какие инструменты вообще скрыты от агента.
- MCP prompts — внешние шаблоны запросов в `#` списке упоминаний, которые пользователь добавляет в чате спецификации как контекст.
- MCP elicitation requests — встроенные формы уточнения или URL-запросы в ленте выполнения, через которые внешний MCP-сервер просит у человека недостающие данные во время выполнения инструмента.
- Kiro Profile shared settings — общие настройки уровня предприятия вроде `model_config`, `mcpEnabled`, `mcpRegistryUrl`, `apiKeysEnabled`, `webSearchEnabled` и `webFetchEnabled`, которые администратор публикует из консоли Kiro, а клиенты Kiro получают при старте. Источник: [Governance](https://kiro.dev/docs/enterprise/governance/).
- MCP registry JSON — централизованный список разрешённых MCP-серверов, который клиенты Kiro читают по HTTPS; он может управлять версией сервера и конфигурацией, но остаётся проверкой на стороне клиента и требует сетевого доступа к URL реестра. Источник: [MCP Governance](https://kiro.dev/docs/enterprise/governance/mcp/).
- список разрешённых моделей и модель по умолчанию — артефакты корпоративного управления, которые ограничивают выбор моделей для пользователей группы, но требуют предварительно включённого управления доступом к моделям. Источник: [Model Governance](https://kiro.dev/docs/enterprise/governance/model/).
- переключатель генерации API key и переключатели веб-инструментов — настройки, которые разрешают или запрещают генерацию ключей API для автоматизации CLI и доступ к `web_search`/`web_fetch`. Источники: [API key Governance](https://kiro.dev/docs/enterprise/governance/api-keys/) и [Web Tools Governance](https://kiro.dev/docs/enterprise/governance/web-tools/).
- дашборд использования Kiro — административный дашборд с агрегированными метриками IDE/CLI: подписки, активные пользователи, потраченные credits, тип клиента, уровень подписки и фильтры по языкам программирования. Источник: [Viewing Kiro usage on the dashboard](https://kiro.dev/docs/enterprise/monitor-and-track/dashboard).
- CSV-отчёт активности пользователей — ежедневный отчёт по пользователям, типу клиента (`KIRO_IDE`, `KIRO_CLI`, `PLUGIN`), сообщениям, разговорам, credits, overage, счётчикам сообщений по моделям и старым метрикам принятых/отклонённых/скрытых сгенерированных строк. Отчёт пишется в S3 по пути вида `s3://bucketName/prefix/AWSLogs/accountId/KiroLogs/user_report/region/year/month/day/00/clientType_accountId_user_report_timestamp.csv`; при числе пользователей больше 1000 за день Kiro разбивает отчёт на несколько CSV. Источник: [Viewing per-user activity](https://kiro.dev/docs/enterprise/monitor-and-track/user-activity/).
- журналы запросов в S3 — журналы встроенных подсказок и разговоров в IDE chat, где при включённой настройке Kiro пишет запросы пользователя и ответы Kiro в S3 bucket в аккаунте клиента; примеры полей включают `prompt`, `assistantResponse`, `conversationId`, `utteranceId`, `requestId`, `leftContext`, `rightContext`, `fileName`, `userId` и `timeStamp`. Источник: [Logging user prompts](https://kiro.dev/docs/enterprise/monitor-and-track/prompt-logging/).
- AWS CloudTrail и Amazon CloudWatch — внешние AWS-механизмы наблюдаемости вокруг активности Kiro: CloudTrail фиксирует API-вызовы/события в S3, CloudWatch позволяет собирать метрики, дашборды и сигналы тревоги, например по количеству вызовов или ежедневно активных пользователей. Источник: [Monitoring and tracking](https://kiro.dev/docs/enterprise/monitor-and-track/).
- `.kiro/agents/` и `~/.kiro/agents/` — пользовательские агенты CLI; они задают инструменты, разрешения, ресурсы, MCP-серверы и инструкцию промпта для специализированных рабочих режимов.
- `.kiro/settings/mcp.json` и `~/.kiro/settings/mcp.json` — CLI MCP конфигурация на уровне проекта и пользователя; agent-level `mcpServers` и `includeMcpJson` могут менять, какие внешние инструменты доступны конкретному агенту.
- CLI resources вроде `file://README.md`, `file://.kiro/steering/**/*.md`, `skill://.kiro/skills/**/SKILL.md` и ресурсов базы знаний — явная упаковка контекста для пользовательских агентов.
- `POWER.md`, `mcp.json` и `steering/` внутри каталога power — упаковка динамически подключаемого контекста, MCP-инструменты и инструкции для конкретного рабочего режима IDE Powers; это соседний механизм Kiro, который может влиять на исполнение спецификации через доступные инструменты и steering, но не заменяет `requirements.md`, `design.md` и `tasks.md`.
- определения hooks — автоматизация, которая может срабатывать на события IDE, включая выполнение задач спецификации.
- ссылки чата вроде `#spec` — способ явно добавить контекст спецификации в разговор.

Источники: [Feature Specs](https://kiro.dev/docs/specs/feature-specs/), [Requirements-First Workflow](https://kiro.dev/docs/specs/feature-specs/requirements-first/), [Design-First Workflow](https://kiro.dev/docs/specs/feature-specs/tech-design-first/), [Bugfix Specs](https://kiro.dev/docs/specs/bugfix-specs/), [Steering](https://kiro.dev/docs/steering/), [Hooks](https://kiro.dev/docs/hooks/), [Context](https://kiro.dev/docs/chat/context/), [MCP Configuration](https://kiro.dev/docs/mcp/configuration/), [MCP Tools](https://kiro.dev/docs/mcp/usage/), [Kiro CLI: Custom agents](https://kiro.dev/docs/cli/custom-agents/), [Kiro CLI: MCP](https://kiro.dev/docs/cli/mcp/), [Kiro CLI: Agent configuration reference](https://kiro.dev/docs/cli/custom-agents/configuration-reference/), [Create powers](https://kiro.dev/docs/powers/create/), [Governance](https://kiro.dev/docs/enterprise/governance/), [Model Governance](https://kiro.dev/docs/enterprise/governance/model/), [MCP Governance](https://kiro.dev/docs/enterprise/governance/mcp/), [API key Governance](https://kiro.dev/docs/enterprise/governance/api-keys/), [Web Tools Governance](https://kiro.dev/docs/enterprise/governance/web-tools/), [Monitoring and tracking](https://kiro.dev/docs/enterprise/monitor-and-track/), [Viewing per-user activity](https://kiro.dev/docs/enterprise/monitor-and-track/user-activity/) и [Logging user prompts](https://kiro.dev/docs/enterprise/monitor-and-track/prompt-logging/).

## Как один артефакт становится входом для следующего

В Feature Specs цепочка выглядит так:

1. Описание фичи в чате или интерфейсе Specs становится `requirements.md`; входом также могут быть существующие требования из PRD, JIRA, Confluence, GitHub issues, MCP-подключённых систем или загруженных документов.
2. Подтверждённый `requirements.md` становится входом для `design.md`.
3. Подтверждённый `design.md` становится входом для `tasks.md`.
4. `tasks.md` становится интерфейсом последовательного агентского исполнения.
5. При запуске всех задач Kiro может построить граф зависимостей, выполнить независимые задачи параллельно по волнам и обновлять прогресс.
6. Результаты выполнения задач могут быть отражены в статус задачи и через Sync Files возвращены в файлы спецификации.

Источники: [Feature Specs](https://kiro.dev/docs/specs/feature-specs/), [Requirements-First Workflow](https://kiro.dev/docs/specs/feature-specs/requirements-first/) и [Specs Best Practices](https://kiro.dev/docs/specs/best-practices/).

Specs Best Practices дают две конкретные дорожки импорта существующих требований. Если источник требований доступен через MCP, его можно подключить напрямую к сессии Spec; если это обычный файл в репозитории, документация предлагает положить его рядом с кодом и в чате спецификации сказать примерно `#foo-prfaq.md Generate a spec from it`. Тогда внешний PRD, выгрузка задач или другой документ становится входом для генерации спецификаций требований и дизайна, а не остаётся ссылкой в голове пользователя. Источник: [Specs Best Practices](https://kiro.dev/docs/specs/best-practices/).

В Design-First цепочка другая:

1. Техническая идея, описание архитектуры, псевдокод, существующий дизайн-документ или архитектурная диаграмма становится `design.md`.
2. Подтверждённый `design.md` становится входом для вывода `requirements.md`.
3. Подтверждённые требования и дизайн становятся входом для `tasks.md`.
4. `tasks.md` переходит в выполнение задач.

Источник: [Design-First Workflow](https://kiro.dev/docs/specs/feature-specs/tech-design-first/).

Specs Best Practices добавляют ограничение, которое легко потерять при гладком пересказе: порядок работы выбирается при создании spec и не переключается внутри уже начатой спецификации. Если команда начала Requirements-First, а затем поняла, что нужен Design-First, нужно создать новую Feature Spec и при необходимости перенести релевантный материал. Источник: [Specs Best Practices](https://kiro.dev/docs/specs/best-practices/).

В Bugfix Specs цепочка похожа, но первый файл другой:

1. Описание дефекта становится `bugfix.md`.
2. Подтверждённый `bugfix.md` становится входом для `design.md`.
3. Подтверждённый `design.md` становится входом для `tasks.md`.
4. Задачи выполняются по одной и меняют код.

Источник: [Bugfix Specs](https://kiro.dev/docs/specs/bugfix-specs/).

Это делает Kiro хорошим примером поэтапной агентности. Агент не получает всё сразу как одно поручение; он участвует в цепочке, где каждый следующий шаг зависит от человечески подтверждённого предыдущего артефакта.

## Роли и участники

В Kiro Specs нет отдельной многоагентной социальной модели вроде Gas Town или BMAD. Основные участники проще:

- человек-заказчик или инженер формулирует намерение, отвечает на вопросы, утверждает требования, дизайн и задачи;
- Kiro Agent генерирует файлы спецификации, анализирует требования, предлагает дизайн, разбивает работу на задачи и выполняет выбранные задачи;
- IDE-поверхность показывает specs, статус задачи, hooks, контекст чата и связь с кодовой базой;
- файлы steering выступают как долговременная память проекта, но их всё равно создаёт и поддерживает человек или команда.
- корпоративный администратор задаёт организационную политику: какие модели доступны, включён ли MCP, используется ли реестр MCP, можно ли выпускать ключи API для CLI и доступны ли веб-инструменты. Это роль до начала конкретной сессии Spec, но она влияет на то, чем агент сможет пользоваться при создании и исполнении спецификации. Источник: [Governance](https://kiro.dev/docs/enterprise/governance/).
- в CLI пользовательские агенты могут задавать специализированные роли через `tools`, `allowedTools`, `resources`, `mcpServers`, `includeMcpJson` и инструкцию промпта; это уже не отдельная фаза Specs, а способ ограничить инструменты, заранее загрузить контекст и уменьшить количество запросов разрешения в повторяемом рабочем режиме. Источники: [Kiro CLI: Custom agents](https://kiro.dev/docs/cli/custom-agents/) и [Agent configuration reference](https://kiro.dev/docs/cli/custom-agents/configuration-reference/).
- CLI subagents позволяют вынести сфокусированную задачу в изолированный контекст, запускать до четырёх subagents одновременно, строить DAG с зависимостями и возвращать краткий итог в основной разговор. Это соседний агентский механизм Kiro, который может обслуживать сложную работу вокруг кода, но официальные источники не показывают его как обязательную часть рабочего процесса IDE Specs. Источник: [Kiro CLI: Subagents](https://kiro.dev/docs/cli/chat/subagents/).
- IDE subagents дают похожую идею изолированного контекста: Kiro может запускать встроенный subagent для сбора контекста или subagent общего назначения, а также пользовательские subagents из `~/.kiro/agents` или `.kiro/agents`. Но для Specs есть жёсткая граница: файлы steering и MCP-серверы работают в subagents, а Specs недоступны и Hooks внутри subagents не срабатывают. Поэтому subagent нельзя автоматически считать полноценным исполнителем задачи спецификации с теми же гарантиями видимости и автоматизации, что у основного порядка Specs. Источник: [Kiro Docs: Subagents](https://kiro.dev/docs/chat/subagents/).

Для теории важно не преувеличивать автономность Kiro. Это не полностью автономный разработчик. Сила Kiro Specs именно в том, что агентское исполнение вставлено в последовательность человеческих утверждений и файлов, которые можно читать.

## Человеческие подтверждения

Kiro Specs устроены вокруг контрольных точек проверки:

- подтверждение `requirements.md` перед генерацией `design.md`;
- подтверждение `design.md` перед генерацией `tasks.md`;
- подтверждение `tasks.md` перед выполнением задач;
- решение, какие задачи считать необязательными, а какие входят в обязательный путь реализации;
- выбор конкретной задачи для выполнения;
- решения по уточнениям, если Analyze Requirements нашёл пробелы, неоднозначности или противоречия.

Источники: [Feature Specs](https://kiro.dev/docs/specs/feature-specs/) и [Analyze Requirements](https://kiro.dev/docs/specs/analyze-requirements/).

Quick Plan намеренно ослабляет эти контрольные точки. Поэтому Kiro содержит не только механизм утверждений, но и продуктовый выбор между полным рабочим процессом спецификации и быстрым планированием. Источник: [Quick Plan](https://kiro.dev/docs/specs/quick-plan/).

После перехода к реализации есть ещё один тип подтверждения: режим Supervised. Он не меняет структуру файлов спецификации, но заставляет Kiro останавливаться после ходов с файловыми изменениями и ждать решения человека по diff на уровне файлов или отдельных фрагментов правок. Это полезно для критичных кодовых баз и незнакомого кода, где одних предварительных контрольных точек недостаточно: даже правильный `tasks.md` может привести к спорным конкретным изменениям. Источник: [Autopilot](https://kiro.dev/docs/chat/autopilot/).

Если сессия Spec использует MCP-инструменты, появляется ещё одна человеческая контрольная точка: подтверждение конкретного вызова инструмента. Kiro показывает, какой инструмент будет вызван и с какими параметрами; пользователь может разрешить или запретить вызов. Для часто используемых доверенных инструментов можно настроить `autoApprove`, но это уже сознательное ослабление остановки на каждом вызове. Elicitation requests добавляют обратный ход: внешний сервер может остановиться во время выполнения инструмента и попросить недостающие данные через форму или URL, а пользователь решает, отвечать, отклонить или закрыть запрос. Источник: [MCP Tools](https://kiro.dev/docs/mcp/usage/).

Корпоративное управление добавляет контрольную точку другого типа: решение принимает не исполнитель отдельной задачи, а администратор группы до начала работы. Управление моделями может ограничить пользователя только разрешёнными моделями и назначить модель по умолчанию; управление ключами API решает, сможет ли пользователь запускать автоматизацию CLI с личным ключом API; управление веб-инструментами включает или выключает `web_search` и `web_fetch`; управление MCP определяет, доступны ли MCP-серверы вообще и берутся ли они из централизованного реестра. Это не заменяет ревью `requirements.md`, `design.md` и `tasks.md`, но задаёт верхнюю границу того, чем агент может пользоваться во время работы со спецификацией. Источники: [Model Governance](https://kiro.dev/docs/enterprise/governance/model/), [API key Governance](https://kiro.dev/docs/enterprise/governance/api-keys/), [Web Tools Governance](https://kiro.dev/docs/enterprise/governance/web-tools/) и [MCP Governance](https://kiro.dev/docs/enterprise/governance/mcp/).

Мониторинг и отслеживание добавляют ещё одно административное решение: включать ли отчёты активности пользователей и журналирование запросов и в какой S3 bucket писать данные. Это не человеческая контрольная точка внутри сессии Spec, но важная политическая контрольная точка вокруг процесса: журналирование запросов может сохранять запросы, ответы и контекст встроенных подсказок, а значит, команда должна заранее решить, какие репозитории, данные и пользователи допустимы для такого режима наблюдаемости. Источники: [Viewing per-user activity](https://kiro.dev/docs/enterprise/monitor-and-track/user-activity/) и [Logging user prompts](https://kiro.dev/docs/enterprise/monitor-and-track/prompt-logging/).

## Проверка результата

Проверка в Kiro Specs распределена по нескольким местам.

На уровне требований проверяется полнота и точность поведения. Критерии приёмки в `requirements.md` дают человеку материал для ревью до дизайна. Источник: [Feature Specs](https://kiro.dev/docs/specs/feature-specs/).

На уровне дизайна проверяются архитектурные решения, компоненты, интерфейсы и стратегия тестирования. Здесь риск в том, что агент может предложить технически правдоподобный, но не подходящий проекту путь; поэтому ревью дизайна — обязательная контрольная точка. Источник: [Feature Specs](https://kiro.dev/docs/specs/feature-specs/).

На уровне задач проверяются детализация, зависимости, ожидаемые результаты и граница между необязательными и обязательными задачами. `tasks.md` должен быть достаточно конкретным, чтобы выполнение задач не превращалось в крупное неограниченное поручение, но при этом не заставляло агента выполнять второстепенные улучшения как обязательную часть MVP. Источник: [Requirements-First Workflow](https://kiro.dev/docs/specs/feature-specs/requirements-first/).

После выполнения задач hooks могут запускать дополнительные проверки. Документация Kiro описывает hooks как автоматизацию, которая реагирует на события, включая выполнение задач до и после. Более точная деталь: `Pre Task Execution` срабатывает до начала задачи спецификации, когда статус меняется на `in_progress`, а `Post Task Execution` — после завершения, когда статус меняется на `completed`; примеры сценариев использования включают подготовку окружения, проверку предварительных условий, запуск тестов, линтинг/форматирование и уведомление внешних систем. В теории это можно использовать как связку спецификация → исполнение → свидетельство, но нужно аккуратно: сами документы не утверждают, что hooks гарантируют качество; они дают механизм автоматизации проверок. Источник: [Hooks](https://kiro.dev/docs/hooks/) и [Hook Triggers](https://kiro.dev/docs/hooks/types/).

Hooks также могут вмешиваться на уровне инструментов. `Pre Tool Use` и `Post Tool Use` поддерживают категории встроенных инструментов, включая `read`, `write`, `shell`, `web`, `spec`, а также фильтры префиксов `@mcp`, `@powers`, `@builtin` и `*` для всех инструментов. Для Kiro Specs это практический механизм контроля вокруг spec-инструментов, MCP-инструментов и инструментов Powers: команда может блокировать часть вызовов, добавить инструкции перед вызовом инструмента или ревьюить файлы после `write`. Источник: [Hook Triggers](https://kiro.dev/docs/hooks/types/).

Режим Supervised добавляет проверку самих изменений, а не только результатов тестов. В этом режиме Kiro показывает правки после хода, позволяет принять или отклонить изменения по файлам, а для изменений файлов — по отдельным фрагментам правок. Это особенно важно для Specs, где задача может быть корректно выбрана, но конкретная реализация затрагивает неожиданные места. Источник: [Autopilot](https://kiro.dev/docs/chat/autopilot/).

В CLI пользовательские агенты могут усилить проверку результата через заранее заданные инструменты и разрешения, но это другой тип контроля. `allowedTools` определяет, какие инструменты агент может использовать без запроса разрешения; `toolsSettings` может ограничивать операции, например через разрешённые пути; MCP-инструменты можно добавлять на уровне конфигурации агента. Это полезно для повторяемых рабочих ролей, но одновременно расширяет поверхность риска: документация подчёркивает, что инструменты записи работают с правами пользователя, а MCP-инструменты с правом записи требуют осторожного включения. Источник: [Agent configuration reference](https://kiro.dev/docs/cli/custom-agents/configuration-reference/).

Отдельная линия проверки появилась в changelog 0.6 и отдельной странице `Correctness with Property-based tests`: Kiro извлекает свойства из требований, оформленных в EARS, определяет, какие из них можно логически тестировать, и генерирует большое число случайных случаев при запуске PBT. В фазе дизайна свойство можно связать с исходным требованием и связанной задачей; во время исполнения Kiro запускает PBT на реализации, а при сбое показывает конкретный сценарий сбоя. Источники: [Kiro IDE changelog 0.6](https://kiro.dev/changelog/ide/0-6) и [Correctness with Property-based tests](https://kiro.dev/docs/specs/correctness/).

Ограничение PBT нужно писать явно. Документация Kiro говорит, что PBT не является формальной верификацией и не доказывает отсутствие всех багов; это более сильное свидетельство, чем одни тесты на примерах, особенно для пограничных случаев и контрпримеров. В интерфейсной логике Kiro PBT необязателен по умолчанию, чтобы пользователь мог сначала сосредоточиться на основной реализации, а затем прогнать свойства и решить, что исправлять: реализацию, тест или требование. Источник: [Correctness with Property-based tests](https://kiro.dev/docs/specs/correctness/).

Корпоративный мониторинг даёт проверку другого уровня: не “эта задача корректно реализована”, а “кто и как использовал Kiro, какие модели и клиенты участвовали, сколько сообщений и credits было потрачено, какие запросы/ответы были сохранены”. Для методологии это полезно как слой аудита и происхождения вокруг SDLC с ИИ, но его нельзя подменять проверкой качества. Журналы запросов могут помочь восстановить ход разговора и контекст встроенных подсказок, а CloudTrail/CloudWatch — отследить события AWS-аккаунта и метрики сервиса; ни один из этих источников сам по себе не подтверждает, что `requirements.md`, `design.md`, `tasks.md` и итоговый diff согласованы. Источники: [Monitoring and tracking](https://kiro.dev/docs/enterprise/monitor-and-track/), [Viewing Kiro usage on the dashboard](https://kiro.dev/docs/enterprise/monitor-and-track/dashboard) и [Logging user prompts](https://kiro.dev/docs/enterprise/monitor-and-track/prompt-logging/).

Уточнение после открытия источников: Kiro также выносит корректность спецификации (`spec correctness`) в property-based testing. Документация описывает этот слой как перевод спецификаций на естественном языке в исполняемые свойства и генерацию полного набора тестовых случаев; цель — проверить универсальные свойства на пространствах входов, а не только отдельные модульные тесты на примерах [Correctness with Property-based tests](https://kiro.dev/docs/specs/correctness/). При этом Kiro честно ограничивает утверждение: PBT не гарантирует отсутствие всех багов, но даёт более сильное свидетельство корректности, чем отдельные примеры. Для будущей теории это важный мост между артефактами спецификации и проверочным каркасом.

## Хвост жизненного цикла

После реализации спецификация остаётся полезной как след намерения, обоснование дизайна и список выполненных задач. Sync Files особенно важен для хвоста: он позволяет привести файлы спецификации к текущему состоянию проекта и отметить уже реализованное. Источник: [Specs Best Practices](https://kiro.dev/docs/specs/best-practices/).

В документации Requirements-First есть практический путь исправления для случая, когда требования меняются после дизайна: пользователь редактирует `requirements.md`, затем нажимает `Refine` в `design.md`, и Kiro обновляет дизайн и задачи под новые требования. В Quick Plan похожая логика описана как частичная регенерация: изменение задачи меняет только задачи, изменение дизайна перестраивает дизайн и задачи, изменение границ запускает цепочку от требований. Источники: [Requirements-First Workflow](https://kiro.dev/docs/specs/feature-specs/requirements-first/) и [Specs just got faster (and smarter)](https://kiro.dev/blog/faster-smarter-specs/).

Checkpointing добавляет ещё один слой защиты вокруг реализации, но его нельзя считать заменой Git или ревью спецификации. В IDE каждый запрос создаёт checkpoint в истории чата; Restore откатывает изменения в кодовой базе и отбрасывает добавления контекста после выбранной точки. Механизм работает через снимки файлов, которые Kiro менял встроенными инструментами изменения файлов, и не отслеживает изменения, сделанные вручную, форматтерами, MCP-инструментами или bash-командами. Revert отличается от checkpoint тем, что откатывает только последний ход и только файловые изменения, тогда как checkpoint может откатить несколько ходов и разговорный контекст. Источник: [Kiro Docs: Checkpoints](https://kiro.dev/docs/chat/checkpoints/).

Kiro также поддерживает несколько specs в одном проекте. Это позволяет разделять крупную работу на независимые спецификации, но создаёт риск расхождения между specs, файлами steering и кодом. Источник: [Specs Best Practices](https://kiro.dev/docs/specs/best-practices/).

CLI добавляет ещё один хвост жизненного цикла: безинтерактивный режим для CI/CD. Он запускается через `kiro-cli chat --no-interactive`, требует `KIRO_API_KEY`, не позволяет ввод пользователя в середине сессии и поэтому требует заранее доверить инструменты через `--trust-all-tools` или `--trust-tools`; если конвейер зависит от MCP, можно включить `--require-mcp-startup`, чтобы быстро падать при недоступных MCP-серверах. Для Kiro Specs это не прямое продолжение визуальной работы со спецификацией, но важный соседний путь: часть проверки, ревью PR или генерации тестов можно вынести в неинтерактивный жизненный цикл после реализации, если команда принимает связанные риски разрешений. Источник: [Kiro CLI: Headless mode](https://kiro.dev/docs/cli/headless/).

По состоянию официальных источников на 9 июня 2026 года Kiro Web нужно держать отдельно от IDE Specs. Документация Web описывает preview-поверхность, где пользователь выбирает репозитории, работает интерактивно или включает Autonomous mode, агент задаёт уточняющие вопросы, строит план, использует специализированные sub-agents и открывает pull request; Web может работать по нескольким репозиториям и читает `.kiro/steering/`. Отдельный блог от 18 мая 2026 года говорит, что Specs планируется принести в Kiro Web, чтобы итерировать требования и дизайн и затем передавать работу автономному агенту, но текущая документация Web не подтверждает полноценный рабочий процесс IDE Specs с `requirements.md`, `design.md`, `tasks.md` и контрольными точками в Web. Источники: [Kiro Web](https://kiro.dev/docs/web/), [Autonomous mode](https://kiro.dev/docs/web/autonomous-mode/), [Creating tasks](https://kiro.dev/docs/web/using-the-agent/creating-tasks/) и [Introducing Kiro Web](https://kiro.dev/blog/introducing-kiro-web/).

У Web Preview есть отдельные ограничения корпоративного управления и MCP. Документация по AWS IAM Identity Center говорит, что Web Preview требует Identity Center, но общие настройки вроде MCP, доступности моделей, URL реестра MCP и Customer Managed Keys на Web Preview не применяются; пользовательские настройки Web хранятся отдельно, а функции корпоративного управления вроде выбора модели и механизмов контроля веб-инструментов для Web Preview ещё недоступны. Страница Web MCP уточняет другую границу: MCP-серверы загружаются только при создании песочницы, могут быть HTTP или локальными, а stdio MCP-серверы работают вне песочницы исполнения инструментов агента и могут читать код, secrets и переменные окружения. Поэтому Web MCP нельзя автоматически приравнивать к управлению MCP в IDE/CLI. Источники: [AWS Identity Center](https://kiro.dev/docs/web/identity-center/) и [Kiro Web: Powers and MCP](https://kiro.dev/docs/web/sandbox/mcp/).

Для будущего текста стоит показать, что Kiro Specs — это не только планирование перед кодом. В зрелом использовании specs становятся частью текущей проектной памяти: они объясняют, почему изменение было сделано именно так, какие задачи были пройдены и как текущая реализация связана с исходным намерением.

Отдельно нужно держать автономный режим Web. В Kiro Web Preview autonomous mode агент сначала задаёт уточняющие вопросы, потом строит план, делегирует работу специализированным sub-agents и открывает pull request после завершения [Autonomous mode](https://kiro.dev/docs/web/autonomous-mode/). Это не равно Feature Specs в IDE, но показывает направление жизненного цикла: спецификация и планирование становятся входом для более автономного цикла доставки. В будущей теории это может связать Kiro Specs с темой “от спецификации к PR”, но досье не должно подменять specs-раздел рассказом об автономности Web.

## Сильные стороны

Главная сила Kiro — продуктовая дисциплина. Многие подходы, управляемые спецификациями, требуют внешних скриптов, дисциплины или ручного переноса контекста. Kiro встраивает требования, дизайн, задачи, выполнение задач, steering и hooks в IDE. Для команды это снижает трение: спецификация лежит рядом с кодом и агентом. Источник: [Kiro Docs: Specs](https://kiro.dev/docs/specs/).

Вторая сильная сторона — явные контрольные точки. Требования, дизайн и задачи можно проверять до реализации. Это особенно полезно там, где ошибка в понимании дороже, чем время на проверку. Источник: [Feature Specs](https://kiro.dev/docs/specs/feature-specs/).

Третья сильная сторона — разные режимы. Feature Specs, Bugfix Specs, Analyze Requirements и Quick Plan закрывают разные ситуации. Kiro не требует одинакового процесса для новой фичи, дефекта, анализа существующих требований и маленькой понятной задачи. Источники: [Feature Specs](https://kiro.dev/docs/specs/feature-specs/), [Bugfix Specs](https://kiro.dev/docs/specs/bugfix-specs/), [Analyze Requirements](https://kiro.dev/docs/specs/analyze-requirements/), [Quick Plan](https://kiro.dev/docs/specs/quick-plan/).

Четвёртая сильная сторона — сочетание локального и долговременного контекста. `.kiro/specs/` хранит контекст конкретного изменения, `.kiro/steering/` — устойчивые правила проекта, `#spec` позволяет явно добавить спецификацию в чат. Источники: [Steering](https://kiro.dev/docs/steering/) и [Context](https://kiro.dev/docs/chat/context/).

Пятая сильная сторона — переносимость части контекста за пределы IDE. Kiro CLI официально использует ту же `.kiro`-конфигурацию для steering и MCP, а пользовательские агенты позволяют сделать повторяемые роли с собственными инструментами, ресурсами и моделью разрешений. Это не заменяет Specs, но расширяет методологическую рамку: спецификационные файлы и проектный steering могут жить рядом с терминальными агентами, безинтерактивными запусками и специализированными ролями. Источники: [Kiro CLI blog](https://kiro.dev/blog/introducing-kiro-cli/), [Kiro CLI: Custom agents](https://kiro.dev/docs/cli/custom-agents/) и [Kiro CLI: Configuration](https://kiro.dev/docs/cli/chat/configuration/).

Шестая сильная сторона — Kiro связывает пакетное исполнение с проверками, а не только с ускорением. История `Run all Tasks` показывает, что продуктовая команда сначала отказалась от полного автозапуска задач, а затем вернула его после PBT, диагностики IDE, dev-серверов, subagents и видимости в реальном времени. Для теории это полезная деталь: автономность в Kiro подаётся как результат накопления механизмов защиты, а не как самоценная кнопка. Источник: [Run all tasks: the feature we refused to ship](https://kiro.dev/blog/run-all-tasks/).

Седьмая сильная сторона — Specs-соседние механизмы можно вписать в организационную политику. Корпоративное управление не делает спецификацию качественной, но позволяет заранее сузить модельный выбор, отключить MCP, перевести MCP на серверы из управляемого реестра, запретить генерацию ключей API для автоматизации CLI или выключить веб-инструменты. Для командного SDLC с ИИ это полезно как слой над индивидуальными подтверждениями: ограничения не приходится заново объяснять каждому пользователю в каждой сессии Spec. Источник: [Governance](https://kiro.dev/docs/enterprise/governance/).

## Режимы отказа и риски чрезмерного использования

Первый риск — ложная полнота. `requirements.md`, `design.md` и `tasks.md` могут выглядеть аккуратно, даже если исходная постановка неверна. Kiro делает материал ревьюируемым, но не заменяет суждение человека. Особенно это видно в Analyze Requirements: если найденные пробелы или неоднозначности не разобрать, спецификация закрепит неопределённость вместо того, чтобы её снять. Источник: [Analyze Requirements](https://kiro.dev/docs/specs/analyze-requirements/).

Второй риск — чрезмерное использование полного рабочего процесса. Для малой задачи полный проход через требования, дизайн и задачи может быть лишним. Quick Plan существует именно как продуктовый ответ на этот риск, но выбор режима остаётся за человеком. Источник: [Quick Plan](https://kiro.dev/docs/specs/quick-plan/).

Третий риск — преждевременный Quick Plan. Если задача кажется простой, но затрагивает архитектуру, безопасность, совместимость или несколько подсистем, отказ от контрольных точек может вернуть пользователя к обычному агентскому “сделай быстро”. Это не ошибка Kiro, а риск неправильного выбора режима.

Четвёртый риск — спецификация отстаёт от кода. Sync Files снижает этот риск, но сам факт наличия механизма синхронизации показывает проблему: спецификации и реализация могут разойтись. Источник: [Specs Best Practices](https://kiro.dev/docs/specs/best-practices/).

Пятый риск — продуктовая поверхность скрывает стоимость инженерного суждения. Встроенная IDE-поверхность делает процесс удобным, но может скрыть, что важные решения всё равно требуют инженерного чтения: критерии приёмки, компромиссы дизайна, детализацию задач, стратегию тестирования и границы исправления ошибок. Это особенно важно для будущего раздела: Kiro нельзя подавать как автоматическое решение проблемы качества спецификации.

Отдельный риск хвоста жизненного цикла связан с checkpointing. Restore может вернуть файлы и контекст к прежнему состоянию, но документация Kiro прямо ограничивает покрытие: не отслеживаются ручные правки, инструменты форматирования, MCP-инструменты и bash-команды. Если пользователь смешивает агентские изменения с внешними правками между checkpoint и restore, откат может потерять человеческие изменения или не покрыть внешне сделанные изменения. Источник: [Kiro Docs: Checkpoints](https://kiro.dev/docs/chat/checkpoints/).

Ещё один риск — неправильный выбор входного порядка работы. Requirements-First и Design-First можно использовать в одном проекте через разные спецификации, но нельзя переключить уже начатую спецификацию на другой режим. Если команда поздно понимает, что сначала нужна техническая проверка осуществимости, ей придётся создавать новую Feature Spec и переносить материал. Источник: [Specs Best Practices](https://kiro.dev/docs/specs/best-practices/).

Шестой риск — ложные срабатывания в анализе требований из-за слабой доменной модели. Если фоновые утверждения неполные или неверные, автоматизированное рассуждение может показать конфликт, пробел или неожиданный сценарий там, где реальный домен уже запрещает соответствующее состояние. В `Deep Spec Analysis` это названо главным направлением дальнейшего улучшения: без хороших формальных доменных моделей требования можно формально анализировать, но часть выводов останется зависимой от качества скрытой доменной модели. Источник: [Deep Spec Analysis](https://kiro.dev/blog/deep-spec-analysis/).

Седьмой риск — потеря проектного контекста при переходе к пользовательским агентам. В обычных чат-сессиях CLI файлы steering автоматически доступны, но пользовательские агенты требуют явного добавления steering в `resources`. Если этого не сделать, специализированный агент может иметь правильные инструменты и разрешения, но не знать командных правил, архитектурных ограничений или актуальных проектных соглашений. Источник: [Kiro CLI: Steering](https://kiro.dev/docs/cli/steering/).

Восьмой риск — смешение удобства разрешений с качеством процесса. Пользовательские агенты и безинтерактивный режим позволяют заранее доверять инструментам, MCP-серверам и операциям записи, чтобы не прерывать работу разрешениями. Это удобно для CI/CD и повторяемых ролей, но документация Kiro отдельно предупреждает, что инструменты записи и MCP-инструменты с правом записи действуют с правами пользователя; значит, методология должна различать “меньше запросов разрешения” и “лучше проверенный результат”. Источники: [Agent configuration reference](https://kiro.dev/docs/cli/custom-agents/configuration-reference/) и [Headless mode](https://kiro.dev/docs/cli/headless/).

Девятый риск — ошибочно считать subagents продолжением Specs. В IDE subagents могут ускорить исследование или параллельную работу, но официальная документация прямо ограничивает их: они не имеют доступа к Specs, а Hooks в них не срабатывают. Если команда вынесет часть проверки, связанной со спецификацией или реализации в subagent, нужно отдельно передать ему нужные факты из `requirements.md`, `design.md` и `tasks.md` и не ожидать автоматических spec hooks. Источник: [Kiro Docs: Subagents](https://kiro.dev/docs/chat/subagents/).

Десятый риск — невидимая смена контекста через Powers. Динамическая загрузка полезна против перегрузки контекста, но при разборе результата важно понимать, какие powers активировались, какие MCP-инструменты они добавили и какие инструкции steering принесли. Иначе два похожих исполнения спецификации могут отличаться не из-за файлов спецификации, а из-за разного набора активированных powers. Источник: [Kiro Docs: Powers](https://kiro.dev/docs/powers/).

Одиннадцатый риск — слишком широкое доверие к MCP-инструментам. MCP полезен для импорта требований, чтения документации и обращения к внешним системам во время работы со спецификацией, но `autoApprove` и инструменты с правом записи меняют границу человеческого контроля. Если команда включает автоматическое подтверждение для удобства или не использует `disabledTools` для опасных операций, Specs могут выглядеть дисциплинированно на уровне `requirements.md`/`design.md`/`tasks.md`, но фактическое исполнение получит слишком широкий доступ к внешним действиям. Источники: [MCP Configuration](https://kiro.dev/docs/mcp/configuration/) и [MCP Tools](https://kiro.dev/docs/mcp/usage/).

Двенадцатый риск — преждевременно описывать Kiro Web как Specs-поверхность. Web уже добавляет автономные PR-ориентированные задачи, работу по нескольким репозиториям, песочницу и steering, а блог обещает перенос Specs в Web. Но до появления текущей документации по Web Specs это нужно держать как направление дорожной карты/preview, а не как подтверждённый рабочий процесс IDE Specs. Источники: [Kiro Web](https://kiro.dev/docs/web/) и [Introducing Kiro Web](https://kiro.dev/blog/introducing-kiro-web/).

Тринадцатый риск — переоценить корпоративное управление как техническую гарантию. Документация MCP Governance прямо называет проверку клиентской; серверы из управляемого реестра синхронизируются при старте и примерно раз в сутки, а пользователь с административным доступом к машине может обойти ограничения. CLI registry при этом не запрещает персональные MCP-серверы из `mcp.json`. Значит, корпоративное управление полезно как организационный контроль и политика по умолчанию, но не заменяет локальный аудит конфигурации и ревью фактического окружения агента. Источники: [MCP Governance](https://kiro.dev/docs/enterprise/governance/mcp/) и [Kiro CLI: MCP Registry](https://kiro.dev/docs/cli/mcp/registry/).

Четырнадцатый риск — смешать механизмы контроля Web Preview с механизмами контроля IDE/CLI. Для Kiro Web Preview общие настройки из Identity Center не применяются, функции корпоративного управления ещё недоступны, а Web MCP-сервер может иметь доступ к коду, secrets и переменным окружения вне песочницы инструментов агента. Это означает, что даже если IDE Specs и CLI ограничены через Kiro Profile и MCP registry, Web Preview нужно проверять отдельно. Источники: [AWS Identity Center](https://kiro.dev/docs/web/identity-center/) и [Kiro Web: Powers and MCP](https://kiro.dev/docs/web/sandbox/mcp/).

Пятнадцатый риск — принять мониторинг за трассировку спецификации. Дашборд, отчёты активности пользователей, журналы запросов, CloudTrail и CloudWatch полезны для административной видимости, расследований и FinOps, но они не строят автоматическую связь `требование → решение дизайна → задача → diff → свидетельство тестов`. Более того, журналирование запросов само становится чувствительным артефактом: в журналы могут попасть запросы, ответы и контекст кода для встроенных подсказок, поэтому включение такого режима требует отдельной политики хранения, доступа и очистки. Источники: [Monitoring and tracking](https://kiro.dev/docs/enterprise/monitor-and-track/), [Viewing per-user activity](https://kiro.dev/docs/enterprise/monitor-and-track/user-activity/), [Logging user prompts](https://kiro.dev/docs/enterprise/monitor-and-track/prompt-logging/) и [Data protection](https://kiro.dev/docs/privacy-and-security/data-protection/).

Проход с открытием источников уточнил риск “продуктовая уверенность”. Нейросимвольный анализ требований звучит сильнее обычной LLM-проверки, но его результат всё равно зависит от того, насколько требования можно формализовать. Если критерии приёмки остаются слишком тезисными, уточнение сначала должно довести их до проверяемых событий, входов, состояний и выходов; без этого сама формализация получает плохой вход [Kiro deep requirements analysis](https://kiro.dev/blog/deep-spec-analysis/). Поэтому Kiro Specs не отменяет ремесло формулирования требований, а частично автоматизирует обнаружение дыр после того, как требования уже достаточно структурированы.

Второй риск — смешать разные продуктовые поверхности. IDE Feature Specs, Analyze Requirements, Correctness/PBT и автономный режим Web относятся к одной Kiro-экосистеме, но не являются одним и тем же рабочим процессом. Для теории нужно использовать их как связанные слои, а не как доказательство, что любая спецификация Kiro автоматически проходит полный путь от требований до PR.

## Аналитические заметки предыдущих проходов

### Что должно попасть в теорию

Kiro должен войти в теорию как пример того, что спецификационный слой может стать продуктовой поверхностью SDLC. Он показывает не только “пишите спецификацию перед кодом”, а конкретную цепочку файлов и контрольных точек:

```text
идея фичи
→ requirements.md
→ design.md
→ tasks.md
→ выполнение задач
→ status / Sync Files / hooks
```

Для bugfix:

```text
отчёт об ошибке
→ bugfix.md
→ design.md
→ tasks.md
→ выполнение задач
```

В будущем тексте стоит раскрыть:

- почему `prompt` слишком слаб как единственная единица управления сложной фичей;
- как Kiro превращает requirements/design/tasks в наблюдаемые артефакты SDLC;
- как Kiro различает Vibe sessions и Spec sessions: быстрый разговорный режим не исчезает, но сложные задачи переводятся в формальную сессию Spec с отслеживанием прогресса;
- как контрольные точки проверки удерживают человека в контуре;
- почему Analyze Requirements нужно описывать не как “LLM перечитал требования”, а как связку уточнения, автоформализации и логического анализа с человеческими вопросами с двумя вариантами;
- почему Quick Plan важен как защита от бюрократизации работы со спецификацией;
- почему параллельное выполнение задач превращает `tasks.md` в план исполнения с учётом зависимостей, а не только список дел;
- как режимы Autopilot и Supervised добавляют слой контроля над фактическими файловыми изменениями после контрольных точек спецификации;
- почему IDE subagents нельзя описывать как полноценных исполнителей спецификации: Specs недоступны subagents, а Hooks в них не срабатывают;
- как Powers добавляют динамически подключаемый контекст и MCP-инструменты, но остаются соседним механизмом Kiro, а не частью основной цепочки `requirements.md` → `design.md` → `tasks.md`;
- как MCP становится внешним инструментальным слоем вокруг Specs: импорт требований и дизайна, шаблоны запросов и шаблоны ресурсов, подтверждения вызовов инструментов, `autoApprove`, `disabledTools` и запросы уточнения от MCP-сервера;
- как корпоративное управление ограничивает доступные модели, MCP-серверы, ключи API и веб-инструменты до начала сессии Spec, но остаётся организационным и клиентским контролем, а не доказательством качества результата;
- как корпоративный мониторинг добавляет административную наблюдаемость вокруг IDE/CLI, но не заменяет трассировку самой спецификации;
- почему Web/autonomous mode нужно описывать как соседнюю и развивающуюся поверхность Kiro: она использует steering и жизненный цикл PR, но официальная Web-документация пока не доказывает тот же рабочий процесс IDE Specs;
- как `.kiro/steering/`, `#spec`, hooks и Sync Files расширяют спецификацию за пределы одной чат-сессии;
- как CLI, пользовательские агенты, MCP-конфигурация и безинтерактивный режим расширяют Kiro за пределы IDE, но не должны описываться как тот же самый рабочий процесс Specs без отдельного подтверждения;
- где Kiro остаётся зависимым от человеческого суждения.

### Что лучше уйдёт в Handbook / Fieldbook

В Handbook / Fieldbook лучше вынести практические правила выбора режима:

- когда использовать полный Feature Spec;
- когда оставаться в Vibe session, а когда переходить в Spec session через `Generate spec`;
- когда Bugfix Spec лучше обычного запроса на исправление ошибки;
- когда достаточно Quick Plan;
- как читать `requirements.md` перед подтверждением;
- что проверять в `design.md`;
- как оценивать детализация задач в `tasks.md`;
- как читать находки Analyze Requirements: вопросы о неоднозначности, конфликте, полноте, принимаемых сценариях и отклоняемых сценариях;
- когда не доверять формальной находке без проверки доменной предпосылки;
- как использовать параллельный `Run all Tasks`, если задачи независимы, и когда лучше запускать их по одной;
- когда запускать Sync Files;
- какие hooks стоит привязать к выполнению задач спецификации;
- как поддерживать `.kiro/steering/`, чтобы она не стала устаревшим мусором.
- когда запускать задачу спецификации в режиме Autopilot, а когда включать Supervised и принимать фрагменты правок вручную;
- как использовать subagents для исследования вокруг спецификации, не предполагая, что им доступны Specs и Hooks;
- как проверять, какие Powers активировались и какие MCP-инструменты или инструкции steering они добавили к работе;
- какие MCP-серверы и инструменты разрешены в сессии Spec, какие инструменты надо добавить в `disabledTools`, а какие допустимо включить в `autoApprove`;
- как реагировать на MCP elicitation requests и какие данные нельзя отдавать внешнему серверу;
- как проверять Kiro Profile shared settings: список разрешённых моделей, MCP on/off, MCP registry URL, API key generation и веб-инструменты;
- как настраивать мониторинг/отслеживание: кто имеет доступ к дашборду использования, отчётам активности пользователей и журналам запросов, в какой S3 bucket пишутся данные и какие правила хранения и доступа применяются;
- как читать MCP registry как список разрешённых серверов: какие серверы управляются централизованно, какие персональные серверы всё ещё загружаются из `mcp.json`, какие локальные переопределения допустимы;
- как отдельно проверять Kiro Web Preview: какие настройки Identity Center реально применяются, какие механизмы корпоративного управления не действуют и какие MCP-серверы попадают в песочницу;
- как создавать пользовательских агентов CLI так, чтобы они явно получали нужные файлы steering через `resources`;
- когда для безинтерактивного режима допустимы `--trust-tools` или `--trust-all-tools`, а когда требуется более узкая модель разрешений;
- как проверять приоритет конфигурации MCP: Agent > Project > Global для MCP-серверов и Project > Global для steering.

Туда же можно вынести чек-лист ревью:

- требования содержат проверяемые критерии приёмки;
- дизайн явно говорит о компонентах, интерфейсах, данных, ошибках и тестах;
- задачи достаточно малы и упорядочены;
- Quick Plan не используется для задачи с высокой неопределённостью;
- после реализации спецификация синхронизирована с фактическим состоянием проекта.

### Дополнительный срез источников из чат-прохода

- `deep-spec-analysis` — раскрыт как источник об уточнении → автоформализации → логическом анализе и проверке неоднозначных, неполных или противоречивых требований.

## Кандидаты на иллюстрации

- Схема `requirements.md → design.md → tasks.md → выполнение задач`. Источник: Kiro Docs Feature Specs. Назначение: показать поэтапный SDLC внутри IDE. Статус: кандидат, лучше перерисовать как собственную схему.
- Разветвление `Feature Spec` vs `Bugfix Spec`: `requirements.md` или `bugfix.md` сходятся в `design.md` и `tasks.md`. Источник: Feature Specs и Bugfix Specs. Назначение: показать, что Kiro различает типы изменений. Статус: кандидат, лучше перерисовать.
- Диаграмма контекста: `.kiro/specs/`, `.kiro/steering/`, `#spec`, hooks, Sync Files. Источники: Specs, Steering, Context, Hooks, Best Practices. Назначение: показать, где живёт контекст и как он возвращается в исполнение. Статус: кандидат.
- Сравнительная мини-лента контрольных точек и Quick Plan. Источник: Feature Specs и Quick Plan. Назначение: показать, что Kiro содержит и полный рабочий процесс, и быстрый путь. Статус: кандидат.
- Диаграмма параллельного выполнения задач волнами: граф зависимостей, сначала настройка, параллельные волны, тесты после кода, сбой одной задачи не останавливает независимые задачи. Источник: [Specs just got faster (and smarter)](https://kiro.dev/blog/faster-smarter-specs/). Назначение: показать, что `tasks.md` может стать планом параллельного исполнения, а не только списком. Статус: официальный кандидат изображения, лучше перерисовать.
- Диаграмма конвейера анализа требований: уточнение → автоформализация → логический анализ → вопросы с двумя вариантами → переписанный `requirements.md`. Источник: [Deep Spec Analysis](https://kiro.dev/blog/deep-spec-analysis/). Назначение: показать, где автоматическая проверка заканчивается и где человек выбирает смысл. Статус: официальный кандидат изображения, лучше перерисовать.
- Диаграмма рабочего процесса property-based testing: EARS-требование → извлечённое свойство → связанная задача → сгенерированные случаи → сценарий сбоя → исправление реализации/теста/требования. Источник: [Correctness with Property-based tests](https://kiro.dev/docs/specs/correctness/). Назначение: показать, как спецификация становится проверяемым поведением, но не превращается в формальную гарантию. Статус: официальный кандидат изображения, лучше перерисовать.
- Мини-скрин или перерисованная последовательная лента из официальной стенограммы главной страницы: `Start task → Task in progress → Task completed → View changes / View execution`. Источник: [Kiro home page transcript](https://kiro.dev/). Назначение: показать видимый статус задачи как часть продуктового делегирования. Статус: кандидат, лучше перерисовать; стенограмма полезнее как фактическое подтверждение состояний UI, чем как готовый ассет.
- Диаграмма мостика контекста CLI: `.kiro/steering/` и MCP используются IDE и CLI, но пользовательские агенты требуют явных `resources`; subagents имеют изолированный контекст; безинтерактивный режим требует заранее доверенных инструментов. Источники: [Kiro CLI blog](https://kiro.dev/blog/introducing-kiro-cli/), [Kiro CLI: Steering](https://kiro.dev/docs/cli/steering/), [Custom agents](https://kiro.dev/docs/cli/custom-agents/) и [Headless mode](https://kiro.dev/docs/cli/headless/). Назначение: показать, что Kiro-методология расширяется за пределы Specs UI, но с другими контрактами контекста и безопасности. Статус: кандидат для Handbook/Fieldbook, не для основного сравнения Specs.
- Диаграмма защитного стека для `Run all Tasks`: валидация спецификации → только обязательные задачи → граф зависимостей → PBT/dev servers/LSP diagnostics/subagents → видимость в реальном времени. Источник: [Run all tasks](https://kiro.dev/blog/run-all-tasks/) и [Specs Best Practices](https://kiro.dev/docs/specs/best-practices/). Назначение: показать, что пакетное исполнение в Kiro опирается на дополнительные проверки и ограничения. Статус: кандидат, лучше перерисовать.
- Мини-схема покрытия checkpoint: checkpoint запроса → Restore/Revert → покрываются встроенные изменения файлов; не покрываются ручные изменения, инструменты форматирования, MCP-инструменты, bash-команды. Источник: [Checkpoints](https://kiro.dev/docs/chat/checkpoints/). Назначение: показать защитный механизм и его границы. Статус: кандидат для Handbook/Fieldbook.
- Мини-схема контроля исполнения: контрольные точки спецификации → выполнение задач → переключатель Autopilot/Supervised → подтверждение на уровне файлов или фрагментов правок → Revert/checkpoint. Источник: [Autopilot](https://kiro.dev/docs/chat/autopilot/). Назначение: показать, что контроль человека продолжается после утверждения `tasks.md`. Статус: кандидат для Handbook/Fieldbook.
- Схема границ subagents: основной рабочий процесс Specs имеет Specs/Hooks, subagents имеют steering и MCP, но не имеют Specs и не запускают Hooks. Источник: [Subagents](https://kiro.dev/docs/chat/subagents/). Назначение: не дать будущему тексту ошибочно описать subagents как полноценное окружение исполнения спецификации. Статус: кандидат.
- Схема Powers как динамически подключаемого контекстного слоя: ключевые слова запроса → активированный power → `POWER.md` + MCP + steering/hooks → исполнение агентом. Источник: [Powers](https://kiro.dev/docs/powers/). Назначение: показать, как Kiro снижает перегрузку контекста MCP рядом со Specs. Статус: кандидат, не для основной схемы Specs.
- Мини-схема выбора режима: Vibe session для быстрого разговора и исследования → `Generate spec` / переключатель сессий → Spec session для формального плана, последовательного исполнения и отслеживания прогресса. Источник: [Vibe vs Spec sessions](https://kiro.dev/docs/chat/vibe/). Назначение: показать, что Kiro не бюрократизирует все задачи, а предлагает переключатель режима. Статус: кандидат.
- Схема IDE MCP вокруг Specs: внешние системы / шаблоны запросов / шаблоны ресурсов → упоминание через `#` или явный вызов `#[server] tool` → подтверждение/уточнение → вход спецификации или выполнение задач. Источники: [MCP](https://kiro.dev/docs/mcp/) и [MCP Tools](https://kiro.dev/docs/mcp/usage/). Назначение: показать внешний контекст и человеческие подтверждения вызовов инструментов. Статус: кандидат для Handbook/Fieldbook.
- Схема контроля hooks/tools: статус задачи спецификации `in_progress`/`completed` → pre/post task hooks; категории инструментов `spec`, `@mcp`, `@powers`, `write`, `shell` → pre/post tool hooks. Источник: [Hook Triggers](https://kiro.dev/docs/hooks/types/). Назначение: показать, как проверки и ограничения можно привязать к исполнению спецификации и инструментам. Статус: кандидат.
- Схема разведения IDE Specs и Kiro Web: IDE Specs имеют `requirements.md`/`design.md`/`tasks.md` и контрольные точки; Web preview имеет автономный PR-цикл, песочницу и steering; блог от 18 мая 2026 обещает перенос Specs в Web. Источники: [Kiro Web](https://kiro.dev/docs/web/) и [Introducing Kiro Web](https://kiro.dev/blog/introducing-kiro-web/). Назначение: не дать будущему тексту преждевременно склеить текущий рабочий процесс IDE и дорожную карту Web. Статус: кандидат для открытых вопросов/сравнительной схемы.
- Схема корпоративного управления вокруг Specs: Kiro Profile shared settings → список разрешённых моделей / модель по умолчанию, MCP on/off / registry URL, API key generation, веб-инструменты → работа со спецификацией в IDE/CLI. Источники: [Governance](https://kiro.dev/docs/enterprise/governance/), [Model Governance](https://kiro.dev/docs/enterprise/governance/model/), [MCP Governance](https://kiro.dev/docs/enterprise/governance/mcp/), [API key Governance](https://kiro.dev/docs/enterprise/governance/api-keys/) и [Web Tools Governance](https://kiro.dev/docs/enterprise/governance/web-tools/). Назначение: показать, что часть контроля над agentic SDLC находится выше конкретной сессии Spec. Статус: кандидат для Handbook/Fieldbook.
- Мини-схема границ MCP registry: серверы из управляемого реестра синхронизируются централизованно, персональные MCP-серверы из `mcp.json` могут сосуществовать, локальные переопределения задают env/headers/timeout, проверка выполняется на стороне клиента. Источники: [MCP Governance](https://kiro.dev/docs/enterprise/governance/mcp/) и [Kiro CLI: MCP Registry](https://kiro.dev/docs/cli/mcp/registry/). Назначение: не дать будущему тексту описать registry как абсолютную техническую песочницу. Статус: кандидат для раздела рисков.
- Схема разведения корпоративного управления Web и IDE/CLI: Identity Center включает доступ к Web Preview, но общие настройки/корпоративное управление не применяются к Web; Web MCP загружается в песочницу, поддерживает HTTP/локальные серверы, а stdio servers имеют доступ вне песочницы инструментов. Источники: [AWS Identity Center](https://kiro.dev/docs/web/identity-center/) и [Kiro Web: Powers and MCP](https://kiro.dev/docs/web/sandbox/mcp/). Назначение: показать, почему Web Preview требует отдельной проверки поверхности политик. Статус: кандидат для открытых вопросов/Fieldbook.
- Схема мониторинга/происхождения вокруг Specs: работа со спецификацией в IDE/CLI → дашборд использования / CSV активности пользователей / журналы запросов в S3 / CloudTrail / CloudWatch; рядом отдельная стрелка “не заменяет трассировку requirement-task-diff-test”. Источники: [Monitoring and tracking](https://kiro.dev/docs/enterprise/monitor-and-track/), [Viewing per-user activity](https://kiro.dev/docs/enterprise/monitor-and-track/user-activity/) и [Logging user prompts](https://kiro.dev/docs/enterprise/monitor-and-track/prompt-logging/). Назначение: показать, что административная наблюдаемость и методологическая трассировка Specs — разные слои. Статус: кандидат для Handbook/Fieldbook.

## Открытые вопросы

- Закрыто в проходе 02: фундаментальные файлы steering по умолчанию в текущей документации названы как `product.md`, `tech.md` и `structure.md`; они включаются в каждое взаимодействие по умолчанию. Источник: [Steering](https://kiro.dev/docs/steering/).
- Закрыто в проходе 02: `Deep Spec Analysis` прочитан и использован для деталей уточнения, автоформализации, логического анализа, semantic entropy, пяти типов находок и ограничения доменной модели.
- Закрыто в проходе 02: `#spec` может ссылаться на конкретную спецификацию в форме `#spec:user-authentication` и включает requirements, design и tasks. Источник: [Chat](https://kiro.dev/docs/chat/).
- Частично закрыто в проходе 03: статусы выполнения задач видны как `in-progress` и `completed` в документации Specs, а официальная стенограмма главной страницы показывает `Start task`, `Task in progress`, `Task completed`, checkbox, `View changes`, `View execution` и карточка статуса. Остаётся неизвестным, записываются ли эти статусы прямо в `tasks.md` или хранятся отдельно в метаданных IDE.
- Частично закрыто в проходе 03: официальная демонстрация/стенограмма для выполнения задач найдена на главной странице Kiro, но отдельного официального материала, где одновременно хорошо видны контрольные точки проверки и Sync Files, пока нет. Иллюстрации лучше перерисовывать.
- Частично закрыто в проходе 02: официальные кандидаты изображений найдены для параллельного выполнения задач, Quick Plan, анализа требований и PBT, но их лучше перерисовывать и отдельно проверять права.
- Частично закрыто в проходе 03: Kiro CLI проверен по официальной документации и блогу. CLI использует конфигурацию `.kiro`, steering и MCP, поддерживает пользовательских агентов, subagents и безинтерактивный режим, но найденные источники не доказывают, что CLI полностью повторяет рабочий процесс IDE Specs с контрольными точками проверки. Для теории это нужно описывать как соседнюю поверхность Kiro, а не как прямую замену Specs UI.
- Частично закрыто в проходе 04: Sync Files уточнён по best practices как способ отметить уже реализованные задачи и привести файлы спецификации к текущему состоянию. Остаётся неизвестным, где именно технически хранится статус задачи: прямо в `tasks.md` или в метаданных IDE.
- Частично закрыто в проходе 04: `Run all Tasks` уточнён как пакетный запуск только для незавершённых обязательных задач и как продуктовый компромисс после добавления механизмов защиты. Остаётся проверить, есть ли отдельная текущая страница документации, где этот режим описан не только в блоге/Specs Best Practices.
- Частично закрыто в проходе 04: checkpointing добавлен как слой защиты, но с явным ограничением: Restore/Revert не покрывают ручные изменения, инструменты форматирования, MCP-инструменты и bash-команды.
- Поиск прохода 04 не нашёл официального подтверждения, что CLI умеет полноценно запускать или редактировать рабочий процесс IDE Specs с контрольными точками, `Sync Files` и выполнение задач из терминала. До появления такого источника CLI нужно оставлять соседней поверхностью Kiro, а не заменой Specs UI.
- Закрыто в проходе 05: для IDE subagents найдено официальное ограничение — файлы steering и MCP-серверы работают, но subagents не имеют доступа к Specs, а Hooks в них не срабатывают. Источник: [Subagents](https://kiro.dev/docs/chat/subagents/).
- Закрыто в проходе 05: режим Supervised работает в сессиях чата спецификации и даёт подтверждение на уровне файлов или фрагментов правок после ходов с файловыми изменениями. Источник: [Autopilot](https://kiro.dev/docs/chat/autopilot/).
- Частично закрыто в проходе 05: Powers уточнены как динамически подключаемая упаковка `POWER.md`, MCP конфигурация и steering/hooks. Остаётся не использовать Powers как доказательство возможностей Specs: найденные источники показывают контекстный и инструментальный слой IDE, но не отдельный рабочий процесс Specs.
- Закрыто в проходе 06: различие Vibe sessions и Spec sessions найдено в официальной документации. Для Kiro Specs это важно как граница применения: быстрый разговорный режим сохраняется, а сложная работа переводится в Spec session через переключатель сессий или `Generate spec`. Источник: [Vibe vs Spec sessions](https://kiro.dev/docs/chat/vibe/).
- Закрыто в проходе 06: IDE MCP уточнён как отдельный контекстно-инструментальный слой вокруг Specs: конфигурация workspace/user, инструменты/шаблоны запросов/ресурсы, `autoApprove`, `disabledTools`, подтверждения вызовов инструментов и elicitation requests. Источники: [MCP Configuration](https://kiro.dev/docs/mcp/configuration/) и [MCP Tools](https://kiro.dev/docs/mcp/usage/).
- Частично закрыто в проходе 06: hook triggers уточнены до pre/post выполнения задач спецификации и pre/post категорий использования инструментов, включая `spec`, `@mcp`, `@powers`, `write`, `shell`. Это подтверждает механизм автоматизации проверок, но не доказывает качество самих проверок. Источник: [Hook Triggers](https://kiro.dev/docs/hooks/types/).
- Частично закрыто в проходе 06: Kiro Web проверен по официальной документации и блогу от 18 мая 2026. Документация Web показывает автономный PR-ориентированный цикл, песочницу, steering и выполнение по нескольким репозиториям; блог говорит, что Specs будут перенесены в Web. Но текущая найденная документация Web не подтверждает полноценный рабочий процесс Specs с `requirements.md`, `design.md`, `tasks.md` и контрольными точками в Web.
- Закрыто в проходе 07: корпоративное управление проверено по официальной документации. Для IDE/CLI Kiro Profile может централизованно управлять списком разрешённых моделей / моделью по умолчанию, MCP on/off, MCP registry URL, API key generation и веб-инструментами. Это нужно описывать как политический слой вокруг Specs, а не как часть `requirements.md` → `design.md` → `tasks.md`.
- Частично закрыто в проходе 07: MCP registry уточнён как централизованный список разрешённых серверов с проверкой на стороне клиента, синхронизацией при старте и примерно раз в сутки, но персональные MCP-серверы и локальные переопределения в CLI остаются важной границей. Открытым остаётся вопрос, как это выглядит в реальных корпоративных установках и какие журналы аудита доступны администраторам.
- Частично закрыто в проходе 07: Kiro Web Preview уточнён по Identity Center и документации Web MCP. Общие настройки/корпоративное управление не применяются к Web Preview, а stdio MCP-серверы могут иметь доступ вне песочницы инструментов агента. Открытым остаётся вопрос, когда Web Preview получит полноценные механизмы корпоративного управления и как это изменит связь Web с будущими Web Specs.
- Частично закрыто в проходе 08: найден официальный слой корпоративного мониторинга — дашборд использования, CSV-отчёты активности пользователей, журналы запросов в S3, CloudTrail и CloudWatch. Это отвечает на вопрос о документированной административной видимости, но не показывает полноценную трассировку `requirement → design → task → diff → tests`; такую связку нужно считать отдельной методологической практикой или открытым вопросом продукта. Источники: [Monitoring and tracking](https://kiro.dev/docs/enterprise/monitor-and-track/), [Viewing per-user activity](https://kiro.dev/docs/enterprise/monitor-and-track/user-activity/) и [Logging user prompts](https://kiro.dev/docs/enterprise/monitor-and-track/prompt-logging/).
- Частично закрыто в проходе 08: документация Data protection уточняет, что Kiro может хранить вопросы, ответы и дополнительный контекст для Free Tier и индивидуальных пользователей, а для корпоративных пользователей формулирует `data is not stored` и исключение из сбора телеметрии/контента AWS. Остаётся проверить, как это соотносится с журналированием запросов в S3, включённым клиентом, и фактическими политиками хранения конкретной организации. Источник: [Data protection](https://kiro.dev/docs/privacy-and-security/data-protection/).
