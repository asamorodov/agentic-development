# TDAD comparative

Рабочее досье по двум линиям, которые используют одну аббревиатуру `TDAD`. Документ нужен не как финальная сравнительная глава, а как буфер фактуры: какие проблемы решают источники, какие рабочие процессы предлагают, какие артефакты создают, где живут тесты и где остаются человеческие решения.

## Зачем это досье нужно

`TDAD` нельзя переносить в будущую теорию как один метод. В первоисточниках уже видны две разные линии:

- `Test-Driven AI Agent Definition: Compiling Tool-Using Agents from Behavioral Specifications` рассматривает промпт и системные инструкции агента как артефакт, который компилируется из поведенческой спецификации под набор исполняемых тестов [статья Test-Driven AI Agent Definition](https://arxiv.org/abs/2603.08806).
- `Test-Driven Agentic Development: Reducing Code Regressions...` относится к кодинговым агентам: перед изменением кода метод строит карту связей между исходными файлами и тестами, чтобы агент выбирал релевантные проверки и исправлял регрессии до завершения задачи [статья Test-Driven Agentic Development](https://arxiv.org/abs/2603.17973).

Общая идея у этих линий похожа: тесты становятся не поздней ручной проверкой, а частью управляющего контура для агента. Различие принципиальное. Первая линия управляет поведением самого агента и выпускает проверенный промпт. Вторая линия управляет изменением в кодовой базе и даёт агенту доказательный маршрут через тесты.

## Линия 1: Test-Driven AI Agent Definition

### Проблема

Статья `2603.08806` исходит из практической проблемы production-агентов с инструментами. Небольшое изменение промпта может изменить поведение без явного сбоя: агент нарушает политику, неверно использует инструмент, теряет следование runbook или начинает давать ответы, которые выглядят правдоподобно, но не проходят скрытые условия качества [статья Test-Driven AI Agent Definition](https://arxiv.org/abs/2603.08806). Обычная уверенность в хорошо написанном prompt здесь недостаточна, потому что поведение агента нужно проверять на исполняемых сценариях.

В формулировке статьи проблема не сводится к удобству prompt engineering. Авторы прямо привязывают TDAD к production-развёртыванию tool-using agents: нужна измеримая behavioral compliance, потому что prompt regressions, tool misuse и policy violations могут проявиться только после выпуска [статья Test-Driven AI Agent Definition](https://arxiv.org/abs/2603.08806). Для будущей теории это даёт важный угол: агентская инструкция становится объектом выпуска, а не внутренней заметкой разработчика.

### Рабочий процесс

Источник описывает TDAD как цикл компиляции agent definition из behavioral specifications. Инженер задаёт спецификацию поведения; один coding agent превращает её в executable tests; другой coding agent итеративно уточняет prompt/system instructions, пока видимые тесты не проходят [статья Test-Driven AI Agent Definition](https://arxiv.org/abs/2603.08806). В результате появляется не новый код приложения, а промпт агента, который прошёл тестовый контур и может быть оценён на скрытом наборе.

Это важная смена роли prompt: он перестаёт быть текстом, который человек вручную «улучшает», и становится компилируемым артефактом. Для будущей теории это полезно рядом с темой agent memory и harness: поведение агента выносится из неявной инструкции в спецификацию, тесты и повторяемый процесс сборки.

Референсный репозиторий уточняет роли внутри процесса. Product team пишет `spec.yaml`; test builder должен переводить спецификацию в executable tests; coding agent итеративно правит prompt до прохождения тестов; итоговый prompt становится выпускаемым агентом с regression suite, которую можно расширять [README tdad-paper-code](https://raw.githubusercontent.com/f-labs-io/tdad-paper-code/main/README.md). В Docker-пайплайне эти роли разложены на сервисы: `testsmith` генерирует тесты из спецификации, `compiler` компилирует prompt до прохождения visible tests, `evaluate` запускает hidden tests, `mutation` выполняет mutation testing, `agent` даёт интерактивный чат с собранным агентом [README tdad-paper-code](https://raw.githubusercontent.com/f-labs-io/tdad-paper-code/main/README.md).

Полный HTML/PDF статьи даёт более точное устройство pipeline, чем краткий abstract. Спецификация описывается как YAML-документ, где есть `tools`, `policies`, `decision tree`, `response contract`, `test guidance` и `mutation intents`; автор подчёркивает, что спецификация остаётся единым источником истины, а тесты реализуют её, но не заменяют [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1). `TestSmith` проходит `decision tree` и создаёт `MFT`, `INV` и `DIR` tests, включая deterministic fixtures и `canary values` для обнаружения утечки PII; `PromptSmith` запускает `visible suite`, группирует сбои по root cause и делает минимальную правку prompt/tool descriptions под крупнейший кластер сбоев; `Built Agent` отвечает через выделенный инструмент `respond`, чтобы harness проверял структурированные поля и trace вызовов инструментов, а не свободный текст; `MutationSmith` работает только после compilation и не участвует в улучшении prompt [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1).

Важная деталь для будущей теории: в первой линии компилируется не только system prompt. Статья прямо называет описания инструментов первоклассными артефактами: `PromptSmith` может уточнять `preconditions`, `sequencing constraints` и `return-value semantics` инструмента, потому что описания инструментов лежат в контексте рядом с system prompt и сильно влияют на tool use [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1). Значит, `agent definition` здесь ближе к связке prompt + tool descriptions + response contract, а не к одному текстовому prompt.

### Артефакты

По статье и референсному репозиторию нужно сохранять следующие артефакты:

- `behavioral specification` — исходное описание поведения агента;
- `specs/core/{spec}/v{1,2}/spec.yaml` — файловая форма спецификации: v1 как базовая версия и v2 как evolution scenario;
- `visible tests` — тесты, по которым идёт компиляция prompt;
- `hidden tests` — удержанный набор для проверки обобщения и защиты от подгонки под видимый набор;
- `tests_visible/core/{spec}/*.py` и `tests_hidden/core/{spec}/*.py` — конкретные директории для двух контуров проверки;
- `prompt under compilation` — промпт или системные инструкции, которые итеративно меняются;
- `agent_artifacts/core/{spec}/seed_prompt.txt` и `agent_artifacts/core/{spec}/system_prompt.txt` — минимальный стартовый prompt и compiled agent prompt в репозитории;
- `agent artifact` — собранный prompt/agent definition после прохождения тестов;
- `mutation packs` — варианты намеренно испорченного поведения для semantic mutation testing;
- `spec evolution scenarios` — изменения требований, на которых измеряется сохранение поведения после эволюции спецификации [статья Test-Driven AI Agent Definition](https://arxiv.org/abs/2603.08806) [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code).
- `test_guidance` — примеры, которые проясняют неоднозначные policy terms; без них TestSmith может породить конфликтующие тесты для одного и того же требования, например одновременно ожидать clarification и direct answer для близких запросов [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1).
- `respond` tool / structured response contract — контракт финального ответа агента: `decision`, `node_id`, `evidence`, `user_message` и другие schema-validated fields, по которым можно делать deterministic assertions [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1).
- optimized tool descriptions — часть compiled agent artifact, если `PromptSmith` уточняет, когда инструмент должен вызываться, какие поля нужны до вызова и что делать с результатом [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1).

Репозиторий `f-labs-io/tdad-paper-code` важен потому, что даёт файловую форму, которой не хватает в кратком пересказе статьи. В нём указаны `specs/core/{spec}/v{1,2}/spec.yaml`, `tests_visible/core/{spec}/*.py`, `tests_hidden/core/{spec}/*.py`, `mutation_packs/core/{spec}/*.patch`, `agent_artifacts/`, `tdadlib/`, `results/` и `scripts/` [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code).

Текущий README репозитория первой линии уточняет прежний открытый вопрос о статусе `SpecSuite-Core`: `SupportOps` описан как worked example со спецификацией, generated tests, fixtures, mutations и results; `DataInsights`, `IncidentRunbook` и `ExpenseGuard` имеют spec + mutation intents, а тесты для них предполагается генерировать запуском pipeline [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code). Там же указано, что benchmark specifications and results доступны как HuggingFace dataset `f-labs-io/SpecSuite-Core` с конфигурациями `specs` (8 spec versions), `mutations` (27 mutation intents) и `results` (24 pipeline runs) [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code). Это важное уточнение provenance: репозиторий разделяет worked example, runnable specs и вынесенный слой dataset/results.

Pass 05 уточняет dataset-слой первой линии. Страница HuggingFace `f-labs-io/SpecSuite-Core` описывает `SpecSuite-Core` как 4 глубоко специфицированных behavioral specifications и явно перечисляет, что каждая спецификация включает typed tool schemas, policies with priorities, decision trees, response contracts, mutation intents и v1 -> v2 evolution [HuggingFace SpecSuite-Core](https://huggingface.co/datasets/f-labs-io/SpecSuite-Core). В таблице dataset-card указаны домены и изменения v2: `SupportOps` имеет 7 tools, 4 policies и v2 abuse detection; `DataInsights` — 3 tools, 4 policies и cost-aware queries; `IncidentRunbook` — 6 tools, 4 policies и customer impact tracking; `ExpenseGuard` — 6 tools, 5 policies и manager approval gate [HuggingFace SpecSuite-Core](https://huggingface.co/datasets/f-labs-io/SpecSuite-Core). Это делает dataset не просто архивом результатов, а машинно читаемым слоем спецификаций, который можно использовать для проверки paper claims без запуска всего Docker pipeline.

У dataset есть три конфигурации, которые полезно держать отдельно от GitHub repo: `specs` содержит 8 строк, то есть 4 specs x 2 versions, с полем `spec_yaml`; `mutations` содержит 27 строк с полями вроде `mutation_pack_id`, `spec_lineage`, `spec_version`, `mutation_id`, `severity`, `category`, `intent`, `activation_probe`; `results` содержит 24 строки end-to-end pipeline runs и поля метрик `vpr_percent`, `hpr_percent`, `mutation_score`, `total_cost_usd` [HuggingFace SpecSuite-Core](https://huggingface.co/datasets/f-labs-io/SpecSuite-Core). Для будущей теории это важно как пример разделения исходного контракта, намеренных поведенческих поломок и результатов прогонов: спецификация, mutation intent и pipeline result являются разными артефактами, а не одним «бенчмарком».

### Команды и среда

Для локального слоя досье важны команды из README репозитория: запуск видимых тестов через `pytest -q tests_visible -m visible` и компиляция prompt через `python scripts/compile_prompt.py --spec ... --prompt ... --test-cmd ... --max-iters 8` [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code). Полный pipeline в репозитории опирается на Docker services `testsmith`, `compiler`, `evaluate`, `mutation`, `agent`, `login`, `init-volumes` [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code).

Неудобная, но важная деталь: такой pipeline нельзя описывать как обычный локальный запуск пары скриптов. В прежних проходах зафиксировано, что полная сборка работает через Docker volumes; локально созданные файлы могут быть недоступны контейнерам без пересборки и переинициализации volume. Это нужно проверить по актуальному README перед финальным использованием, но уже сейчас это хороший пример хрупкости метода: сильный harness требует аккуратной инфраструктуры.

README делает эту хрупкость явной: полный Docker-процесс работает с volume-слоем, а не с локальными файлами; после локальных изменений нужно выполнять `docker compose build` и `docker compose run --rm init-volumes`, иначе контейнеры могут не увидеть обновлённые `tests_visible`, `tests_hidden` или `agent_artifacts` [README tdad-paper-code](https://raw.githubusercontent.com/f-labs-io/tdad-paper-code/main/README.md). Там же описан порядок v1 -> v2 evolution: сначала скомпилировать v1, затем сгенерировать v2 tests через `testsmith-v2`, собрать v2 prompt через `compiler-v2` с warm start от v1 и измерить SURS командой `evaluate-surs`, где v1 tests запускаются против v2 prompt [README tdad-paper-code](https://raw.githubusercontent.com/f-labs-io/tdad-paper-code/main/README.md).

### Проверка результата

Источник вводит три разных проверочных механизма:

- split на visible/hidden tests: компиляция идёт по видимому набору, а итоговый агент проверяется на скрытом;
- semantic mutation testing: создаются правдоподобные faulty prompt variants, и тестовый набор должен их поймать;
- spec evolution scenarios: проверяется, сохраняется ли безопасность поведения при изменении требований [статья Test-Driven AI Agent Definition](https://arxiv.org/abs/2603.08806).

HTML статьи уточняет механику split: `visible tests` занимают примерно 40-70% и используются в compilation loop; `hidden tests` занимают 30-60%, не показываются `PromptSmith` и используются для отчётности. Для каждой ветки `decision tree` primary `MFT` обычно попадает в visible set, а `INV`/`DIR` variants, boundary cases и metamorphic tests уходят в hidden set [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1). В benchmark mode тесты генерируются заново для каждого trial и hidden set заморожен внутри trial; в production mode failing hidden tests могут переводиться в visible set, после чего агент пересобирается [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1). Это важная граница: hidden tests не навсегда «секретные», но на момент компиляции они отделены от оптимизатора.

Mutation testing в этой линии устроен не как синтаксический patch фиксированного файла. `MutationSmith` получает mutation intent, создаёт plausible faulty prompt variant и сначала проверяет activation probe: действительно ли мутант поменял поведение. `Non-activating mutants` после заданного числа попыток исключаются из mutation score, а `killed mutant` означает, что хотя бы один visible test упал на мутированном prompt [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1). Это одновременно сила и ограничение: метод пытается не наказывать suite за equivalent mutants, но exclusion может завысить качество тестов, если «неактивированный» mutant на самом деле указывает на слепое пятно.

Repo-level mutation result нужно держать отдельно от paper-level aggregate. GitHub README сейчас заявляет для `SupportOps v1` 100% mutation score: 6/6 activated mutants killed across all runs, 7 total intents и 1 non-activating excluded; детали по отдельным runs предлагается смотреть в `results/supportops_v1_*.json`, а paper-level mutation intents — в Table B1 [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code). Это снимает часть путаницы с прежним примером `M_ALWAYS_TICKET`: текущий README больше не даёт тот же 80% локальный список как основной статус, поэтому старую находку надо рассматривать как состояние конкретного README-среза, а не как стабильный результат всей линии.

Pass 05 обнаружил, что статус mutation score у `SupportOps v1` нельзя считать закрытым без commit-level проверки. Rendered GitHub page и поисковый фрагмент показывают вариант README с 100% mutation score: 6/6 activated mutants killed, 7 total intents и 1 non-activating excluded [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code). Но raw README по `main` в этом же проходе отдаёт более старую формулировку: 80% mutation score, 4/5 killed, survived mutant `M_ALWAYS_TICKET` [raw README tdad-paper-code](https://raw.githubusercontent.com/f-labs-io/tdad-paper-code/main/README.md). Одновременно HuggingFace dataset содержит mutation intent `ALWAYS_CREATE_TICKET` как minor `quality_regression` для `SupportOps v1`, где агент всегда создаёт support ticket вместо выполнения запроса [HuggingFace SpecSuite-Core](https://huggingface.co/datasets/f-labs-io/SpecSuite-Core). Поэтому аккуратная формулировка сейчас такая: `ALWAYS_CREATE_TICKET`/`M_ALWAYS_TICKET` — реальный mutation angle про ложную escalation, но его статус killed/survived зависит от слоя источника и версии README; использовать его как стабильное число нельзя до проверки commit hash, dataset rows и `results/supportops_v1_*.json`.

Для spec evolution статья вводит `SURS`: v2 compilation стартует от v1 prompt artifact, но запускается только на v2 tests; held-out v1 invariant tests не показываются `PromptSmith`, а после сборки v2 agent проверяется на сохранение этих invariants [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1). Отдельно автор вводит `RPR` как production-рекомендацию против стохастичности: один pass/fail недостаточен, для обычных сценариев предлагается несколько повторных запусков с порогом, а для PII/auth — zero-failures policy; в самом исследовании RPR не измерялся [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1).

Заявленные числа нужно хранить именно как утверждения источника, не как установленный общий результат для всех агентов: SpecSuite-Core включает четыре deeply specified agents по policy compliance, grounded analytics, runbook adherence и deterministic enforcement; статья заявляет 24 independent trials, 92% v1 compilation success, 97% mean hidden pass rate, 58% compilation success for evolved specifications, mutation scores 86-100%, 78% v2 hidden pass rate и 97% regression safety scores [статья Test-Driven AI Agent Definition](https://arxiv.org/abs/2603.08806). Для будущей главы нужны полные PDF/HTML, определения метрик, failure cases и содержимое harness.

SpecSuite-Core в полной статье описан как маленький, но глубокий benchmark protocol, а не как широкий набор одношаговых задач. Четыре specs имеют 10-14 decision nodes, hidden tests 40-60%, mutation intent catalogs по 5-7 intents на spec и v1 -> v2 evolution scenarios [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1). Домены тоже важны: `SupportOps` проверяет auth before action, PII refusal, plan eligibility и escalation triggers; `DataInsights` проверяет SQL grounding, numeric precision, ambiguity detection и cost-aware queries; `IncidentRunbook` проверяет evidence-first ordering, severity routing и runbook compliance; `ExpenseGuard` проверяет spending caps, FX conversion, receipt requirements и manager approval [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1).

Репозиторий выглядит уже, чем формулировка статьи, но не как простое противоречие: GitHub repo содержит worked example и runnable pipeline, а часть benchmark artifacts вынесена в слой dataset/results [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code). Поэтому вопрос по воспроизводимости нужно формулировать точнее: какие именно spec versions, generated tests, mutation intents и per-trial results можно получить из GitHub repo, какие лежат в HuggingFace dataset, а какие остаются только в статье.

Полная статья называет surviving mutants на уровне SpecSuite-Core: `HALLUCINATE_NUMBERS` выжил в DataInsights v1 и был закрыт в v2; `SKIP_RUNBOOK_LOOKUP` выжил в IncidentRunbook v1, а затем был закрыт улучшенным test coverage в v2 [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1). Это лучше использовать как paper-level факт. Старый repo-level пример `M_ALWAYS_TICKET`, зафиксированный в прежнем проходе по raw README-срезу, теперь нужно оставить только как historical note до проверки commit history, потому что текущий GitHub README для `SupportOps v1` заявляет 100% mutation score с 1 non-activating excluded mutant [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code).

Pass 09 уточняет статус этого конфликта как вопрос происхождения факта, а не как содержательный вывод о методе. Публичная страница `f-labs-io/tdad-paper-code` показывает всего 2 commits, `Issues 0`, `Pull requests 0` и README-слой, где `SupportOps v1` описан как worked example с 6/6 activated mutants killed и одним excluded non-activating mutant [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code). При этом dataset-слой `SpecSuite-Core` отдельно хранит `mutations` с 27 rows и использует близкое имя intent `ALWAYS_CREATE_TICKET`, а не старое `M_ALWAYS_TICKET` [HuggingFace SpecSuite-Core](https://huggingface.co/datasets/f-labs-io/SpecSuite-Core). Без точного commit hash или raw per-run JSON безопасная будущая формулировка такая: первая линия демонстрирует semantic mutation testing на отдельных domain scenarios, но спорный ticket-creation mutant нельзя использовать как стабильный числовой пример.

### Где человек принимает решение

Человек остаётся владельцем спецификации и критерия выпуска. Автоматический цикл может сгенерировать тесты и улучшать prompt, но именно человек должен решить, что behavioral specification выражает нужную политику, что hidden tests не подменяют реальное качество и что агент можно выпускать после прохождения harness. Важно не писать, что TDAD «гарантирует» корректность агента: источник показывает измеряемую проверку поведения на конкретной suite, а не доказательство общей безопасности.

`tdad-skill` показывает минимальную агентскую поверхность второй линии. Skill activates when repository has `.tdad/`; one-time setup is `pip install tdad` and `tdad index .`, producing `.tdad/test_map.txt`. For each changed file, the agent does `grep 'path/to/changed_file' .tdad/test_map.txt`, collects unique test files and runs the project-specific test runner; non-Python projects require tree-sitter extras such as `tdad[treesitter-all]` [tdad-skill SKILL.md](https://raw.githubusercontent.com/pepealonso95/tdad-skill/main/SKILL.md). Это полезно для Handbook: TDAD можно описать как маленький regression guard skill, not as an all-purpose development framework.

SpecSuite-Core reference implementation in the other TDAD line stores specs, visible tests, hidden tests, mutation packs and results; README frames the mental model as product team writes a spec, test builder translates spec to executable tests, coding agent iterates until tests pass, and the resulting prompt is the shipped agent with regression suite [TDAD Reference Implementation README](https://raw.githubusercontent.com/f-labs-io/tdad-paper-code/main/README.md). Это усиливает различение: первая линия компилирует agent behavior from tests; вторая линия protects code changes from regressions by targeted test selection.

## Линия 2: Test-Driven Agentic Development

### Проблема

Статья `2603.17973` решает другую задачу. AI coding agents могут закрывать issue и одновременно создавать регрессии. В такой ситуации resolution rate недостаточен: агент может выглядеть успешным по основному тесту или бенчмарку, но ломать смежное поведение [статья Test-Driven Agentic Development](https://arxiv.org/abs/2603.17973). Эта линия TDAD переносит внимание с «агент написал патч» на «агент выбрал и прошёл тесты, которые связаны с затронутыми файлами».

Важная отрицательная находка второй статьи: добавление TDD procedural instructions без targeted test context ухудшило regression rate до 9.94%, то есть оказалось хуже vanilla baseline с 6.08% [статья Test-Driven Agentic Development](https://arxiv.org/abs/2603.17973). В репозитории это сформулировано как вывод о небольших open-weight моделях: им полезнее конкретный graph-derived context, чем длинная предписывающая процедура [репозиторий TDAD](https://github.com/pepealonso95/TDAD). Для будущего текста это сильная оговорка против простого совета «попросите агента делать TDD».

### Рабочий процесс

Метод строит карту влияния между source files и tests до изменения кода. Агент получает эту карту как lightweight agent skill: статический текстовый файл, который он может запрашивать во время выполнения [статья Test-Driven Agentic Development](https://arxiv.org/abs/2603.17973). Практический смысл: агенту не просто говорят «запусти тесты», а дают структуру, по которой можно выбрать targeted tests для конкретного patch.

В рабочем цикле это выглядит так:

1. Построить или загрузить dependency map между кодом и тестами.
2. Дать агенту задачу исправления вместе с доступом к карте.
3. Во время изменения агент выбирает affected tests.
4. Если тесты показывают регрессию, агент использует сигнал для self-correction.
5. Завершение должно опираться не только на diff, но и на evidence по релевантным тестам [статья Test-Driven Agentic Development](https://arxiv.org/abs/2603.17973) [репозиторий TDAD](https://github.com/pepealonso95/TDAD).

`tdad-skill` превращает это в очень конкретную процедуру, обращённую к агенту. Если `.tdad/test_map.txt` отсутствует, один раз выполняются `pip install tdad` и `tdad index .`; для проектов не на Python README предлагает `pip install tdad[treesitter-all]` и `tdad index . --languages python,javascript` или `TDAD_LANGUAGES=...` [SKILL.md tdad-skill](https://github.com/pepealonso95/tdad-skill/blob/main/SKILL.md) [README tdad-skill](https://github.com/pepealonso95/tdad-skill). После изменения кода агент должен для каждого изменённого файла искать строку через `grep 'path/to/changed_file' .tdad/test_map.txt`, где формат строки описан как `source_file: test_a test_b test_c`, затем собрать уникальные impacted tests, запустить их, исправлять регрессии до зелёного результата и только потом двигаться к сдаче patch [SKILL.md tdad-skill](https://github.com/pepealonso95/tdad-skill/blob/main/SKILL.md).

CLI-слой репозитория даёт более машинную форму того же процесса: `tdad index /path/to/your/repo`, `tdad impact /path/to/your/repo --files src/module.py`, `tdad run-tests /path/to/your/repo --tests tests/test_module.py::test_foo`, `tdad stats /path/to/your/repo` [README tdad-skill](https://github.com/pepealonso95/tdad-skill). По умолчанию используется NetworkX backend без внешнего сервиса; Neo4j остаётся опциональным backend через `TDAD_BACKEND=neo4j` и требует Docker [репозиторий TDAD](https://github.com/pepealonso95/TDAD) [README tdad-skill](https://github.com/pepealonso95/tdad-skill).

Опубликованный `tdad-skill` показывает, что production-facing skill уже шире paper-level «20-line `SKILL.md`». Его README предлагает установку через Claude Code marketplace (`/skill install pepealonso95/tdad-skill`), `npx skills add pepealonso95/tdad-skill` или ручное копирование в `.claude/skills/tdad/SKILL.md`, а сам `SKILL.md` на GitHub имеет metadata, compatibility, one-time setup, multi-language runners, правило обязательного нового regression test и fallback при отсутствии записи в `.tdad/test_map.txt` [README tdad-skill](https://github.com/pepealonso95/tdad-skill) [SKILL.md tdad-skill](https://github.com/pepealonso95/tdad-skill/blob/main/SKILL.md). Для будущей теории это важная развилка: статья фиксирует исследовательский минимализм runtime-инструкции, а опубликованный skill превращает его в переносимый рабочий пакет.

HTML статьи второй линии уточняет, что runtime-интеграция намеренно сведена к двум статическим артефактам: `test_map.txt` и короткий `SKILL.md`; во время выполнения агенту нужны только `grep` и `pytest`, без graph database, MCP server или API calls [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2). Это важный дизайн-выбор: сложность вынесена в indexing/impact-analysis stage, а агент получает маленький переносимый контекст, который помещается в ограниченное окно.

### Артефакты

Для этой линии центральны другие артефакты:

- `dependency map` или code-test graph между исходными файлами и тестами;
- `.tdad/test_map.txt` — статическая текстовая карта, которую skill заставляет агента читать после изменения кода;
- `agent skill/static text file`, через который агент получает карту, включая `SKILL.md` как переносимый формат инструкции для Claude Code и других agents;
- selected impacted tests;
- graph nodes `File`, `Function`, `Class`, `Test`;
- graph edges `CONTAINS`, `CALLS`, `IMPORTS`, `INHERITS`, `TESTS`;
- ranked list of tests, которую выдаёт impact analyzer;
- patch, который исправляет issue;
- вывод тестов как свидетельство отсутствия регрессии на выбранном наборе;
- benchmark/evaluation record по SWE-bench Verified и выбранным моделям [статья Test-Driven Agentic Development](https://arxiv.org/abs/2603.17973) [репозиторий TDAD](https://github.com/pepealonso95/TDAD).

Здесь артефактом является не промпт агента, а карта тестового влияния и следы проверок вокруг конкретного изменения. Это роднит линию с harness engineering, observability и тестовой маршрутизацией.

Pass 06 уточняет, что repo-level воспроизводимость второй линии устроена как несколько поколений harness, а не как один неизменный каталог результатов. `claudecode_n_codex_swebench/run_benchmark.py` создаёт `benchmark_runs/{timestamp}_{run_name}` с подкаталогами `predictions/` и `evaluations/`, пишет `progress.log`, а в структуре результата хранит не только `resolved`, но и признаки вроде `patch_gate_valid`, `graph_guard_passed`, `indexed_search_success`, `impacted_selected_count`, `tdd_evidence_complete`, `f2p_runtime_strategy`, `p2p_runtime_unreliable_reason` и ссылки на prediction/eval files [run_benchmark.py TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/claudecode_n_codex_swebench/run_benchmark.py). Это важно для будущей теории: TDAD второй линии проверяется не одной строкой «тесты прошли», а набором telemetry-полей о том, откуда пришёл тестовый сигнал, насколько он был надёжен и какой слой графа реально использовался.

Поэтому ссылки на `benchmark_runs/`, `predictions/` и `evaluation_results/` нельзя читать как полный список всех артефактов. Вторая линия эволюционирует от Claude Code/Qwen-mini benchmark runner к opencode + local MLX pipeline, и provenance результата зависит от того, каким поколением harness он получен.

В README `tdad-skill` архитектура разложена на пять компонентов: `Language Plugins`, `Graph Builder`, `Test Linker`, `Impact Analyzer`, `Test Runner` [README tdad-skill](https://github.com/pepealonso95/tdad-skill). `Graph Builder` наполняет граф nodes and edges и поддерживает full/incremental indexing через content hashing; `Test Linker` связывает тесты с кодом через naming conventions, static analysis of imports/calls и опциональные coverage data; `Impact Analyzer` ранжирует impacted tests; `Test Runner` делегирует запуск языковому plugin [README tdad-skill](https://github.com/pepealonso95/tdad-skill).

Полная статья описывает Python indexing pipeline конкретнее. `AST Parser` извлекает function definitions с signatures, line ranges и docstrings, class definitions с base classes and methods, import statements и call targets; test files находятся по `test_*.py` и `*_test.py`, test functions по `test_*`, test classes по `Test*` [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2). `Test Linker` строит `TESTS` edges тремя стратегиями в порядке приоритета: naming conventions (`test_foo.py` -> `foo.py`), prefix matching с progressive truncation stem, directory proximity при неоднозначных совпадениях; для монолитных `tests.py` используется proximity algorithm к ближайшему non-test ancestor [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2).

Impact analysis запускает четыре стратегии параллельно и сохраняет максимальный score, если test найден несколькими путями. Balanced profile использует base weights `Direct 0.95`, `Coverage 0.80`, `Transitive 0.70`, `Imports 0.50`; confidence component отражает силу связи: `1.0` для direct `TESTS`, `0.56` для transitive call chains, `0.5` для coverage и `0.45` для imports [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2). Тесты выбираются tiered list с настраиваемым максимумом, по умолчанию до 50; profiles `conservative`, `balanced`, `aggressive` позволяют менять precision/recall trade-off без переписывания формулы [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2).

Поддержка языков тоже является частью фактуры, а не украшением: Python использует встроенный `ast`, а JavaScript/TypeScript, Go, Java, Rust и Dart опираются на tree-sitter extras и соответствующие runners вроде Jest/Vitest/Mocha, `go test`, Maven/Gradle, `cargo test`, `dart test`/`flutter test` [README tdad-skill](https://github.com/pepealonso95/tdad-skill). Это означает, что перенос TDAD в реальный проект зависит от качества language plugin и test detection, а не только от общей идеи code-test graph.

### Проверка результата

Источник заявляет evaluation на SWE-bench Verified и моделях с открытыми весами на потребительском железе: Qwen3-Coder 30B на 100 instances и Qwen3.5-35B-A3B на 25 instances [статья Test-Driven Agentic Development](https://arxiv.org/abs/2603.17973). Заявленный результат — снижение regression rate на 70%, с 6.08% до 1.82% относительно vanilla baseline [статья Test-Driven Agentic Development](https://arxiv.org/abs/2603.17973).

Полная статья показывает, почему headline `70%` нельзя читать отдельно от denominators. В Phase 1 vanilla дал 562 P2P failures из 9,245 tests, TDD Prompt — 799 из 8,040, GraphRAG+TDD — 155 из 8,536; catastrophic regressions снизились с 3 у vanilla и 5 у TDD Prompt до 1 у GraphRAG+TDD [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2). При этом resolution rate у GraphRAG+TDD был 29% против 31% у vanilla, а generation rate 74% против 86%; автор интерпретирует это как conservative bias и higher empty-patch rate, когда карта показывала высокий regression risk [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2).

Эти числа нельзя использовать как общий аргумент «TDAD снижает регрессии» без уточнения. Нужно сохранить условия: какие instances, какие модели, как строилась карта, какие тесты считались impacted, как определялась regression и что было baseline. Репозиторий `pepealonso95/TDAD` нужен следующему проходу именно для этого: проверить формат карты, команды запуска, структуру `tdad-skill` и воспроизводимость результата [репозиторий TDAD](https://github.com/pepealonso95/TDAD) [репозиторий tdad-skill](https://github.com/pepealonso95/tdad-skill).

README `tdad-skill` добавляет определения метрик, которые нужно сохранить рядом с числами: `Resolution Rate` — доля instances, где patch fixes target issue and passes FAIL_TO_PASS without breaking PASS_TO_PASS; `Test-Level Regression Rate` — `sum(PASS_TO_PASS failures) / sum(total PASS_TO_PASS tests) * 100`; `Generation Rate` — доля instances, где агент произвёл non-empty patch [README tdad-skill](https://github.com/pepealonso95/tdad-skill). Там же указано, что GraphRAG + TDD имел 29% resolution против 31% vanilla и больше empty patches: 26 против 14; источник объясняет trade-off не худшим качеством patch, а большей вероятностью не сгенерировать patch [README tdad-skill](https://github.com/pepealonso95/tdad-skill).

Impact analysis в `tdad-skill` ранжирует тесты по стратегиям с весами: `Direct` 0.95, `Coverage` 0.80, `Transitive` 0.70, `Imports` 0.50 [README tdad-skill](https://github.com/pepealonso95/tdad-skill). Это нужно проверить в коде перед финальным использованием, но уже сейчас видно, что «релевантный тест» не является бинарным фактом: это результат эвристического scoring, который может ошибиться на динамических вызовах, плохо названных тестах, неполной coverage или нестандартной архитектуре.

Phase 2 важна как другой режим пользы: там TDAD Skill с Qwen3.5-35B-A3B + OpenCode повысил resolved с 6/25 до 8/25 и generated patches с 10/25 до 17/25, при 0% regression в обеих группах [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2). Автор объясняет это не снижением regressions, а улучшением navigation/generation: code-test graph дал агенту структурный контекст, который помог вообще произвести patch [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2).

Auto-improvement loop второй линии тоже стоит сохранить как отдельную фактуру. Claude Code agent получал experiment history и делал одно focused change в `SKILL.md`, `impact.py`, `ast_parser.py` и других файлах; unit tests были acceptance gate, benchmark evaluation сравнивался с best known scores, regressions вызывали rollback, а evaluation script был checksummed через SHA-256 и read-only [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2). Из 15 iterations приняли 4 изменения: упрощение `SKILL.md` с 107 до 20 lines, static `test_map.txt` export, directory-proximity scoring и import-based mappings; 11 iterations были reverted, а resolution на 10-instance subset вырос с 12% до 60% при 0% regression [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2).

`EXPERIMENTS.md` подтверждает и расширяет эту картину. В `EXP-027` loop описан как 15 iterations за примерно 36 hours: `python -m autoloop.loop --eval-mode full --max-iterations 15 -v`; перед каждой итерацией проверялся SHA-256 для `evaluate_fixed.py`, затем создавался snapshot TDAD source files, Claude Opus 4.6 делал одно изменение, `pytest tests/` служил локальным gate, full eval запускался на 10 instances, а результат сравнивался с best score для решения keep/revert [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). В `EXP-028` уже после перехода на NetworkX backend оценка на тех же 25 instances, что `EXP-025`, дала 8/25 resolved против 6/25 baseline и 17/25 generated против 10/25; сам источник подчёркивает, что улучшение пришло от генерации большего числа patch, а не от резкого роста качества среди сгенерированных patch (62% против 60%) [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). Для досье это уточняет механизм пользы: TDAD Skill может помогать агенту начать и завершить patch, а не только ловить regressions.

Перед этим улучшением в `EXP-026` был важный отрицательный результат, который нельзя потерять. На тех же 25 SWE-bench Verified instances, что `EXP-025`, TDAD mode с opencode, Qwen3.5-35B-A3B и Neo4j-based skill дал 7/25 generated patches и 3/25 resolved против baseline 10/25 generated и 6/25 resolved; regression rate остался 0% в обеих группах [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). Журнал называет несколько возможных причин: индексирование съедало время, skill мог не вызываться, opencode мог не обнаруживать `.opencode/skills/tdad/SKILL.md`, а сама процедура требовала от модели понять skill, вызвать `tdad impact`, интерпретировать результат и запустить тесты [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). Это важный failure case метода: добавление graph context как runtime skill может ухудшить генерацию patch, если контекст плохо упакован или слишком дорог для модели и среды.

`EXP-028` показывает, какая именно repair-trajectory вывела метод из этого провала. Neo4j заменён на NetworkX backend с persistence в `.tdad/graph.pkl`; `networkx>=3.0` стал core dependency, а Neo4j ушёл в optional `pip install tdad[neo4j]` [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). Smoke test на самом TDAD repo дал 66 files, 346 functions, 22 classes, 46 tests, 1292 edges, 1264 linked tests и 17 exported source files in test map; на SWE-bench run по тем же 25 instances NetworkX indexing прошёл без Docker/Neo4j issues, примерно за 30 секунд на instance, а выигрыш пришёл от меньшего числа empty patches: 8 вместо 15 [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). Для будущей теории это показывает не абстрактную пользу «графа», а инженерную развилку: runtime graph service оказался слишком тяжёлым, а статический `test_map.txt` + лёгкий backend лучше совпали с агентским рабочим процессом.

Журнал экспериментов даёт отдельный слой фактуры про надёжность самого evaluation signal. В `EXP-021j` авторы обнаружили, что runs могли получить `resolved=0` через stale global eval JSON, если Docker/evaluator падал, но caller переиспользовал исторический файл; исправление заставило `evaluate_predictions.py` падать громко, а `run_benchmark.py` принимать только fresh eval JSON, созданный текущим invocation [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). В `EXP-021k` это проверено на single-instance run: primary artifacts перечислены как `benchmark_runs/.../progress.log`, `report.json`, `evaluations/graphrag_tdd.eval.json`, а selected fresh eval получил same-run timestamp [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). `EXP-021l` усилил это до exact attribution: evaluator печатает `EVAL_JSON_PATH:`, а runner отказывается от результата со статусом `no_exact_result`, если точный путь не найден [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). Это неудобная, но важная деталь метода: regression guard требует не только карты тестов, но и строгого происхождения eval-файла.

Targeted audit pass 09 не нашёл публичного слоя обсуждений, который мог бы закрыть эту тему через issues или pull requests: текущая GitHub-страница `pepealonso95/TDAD` показывает `Issues 0` и `Pull requests 0`, а также top-level folders `benchmark_runs`, `claudecode_n_codex_swebench` и `tdad` рядом с `EXPERIMENTS.md` [репозиторий TDAD](https://github.com/pepealonso95/TDAD). Поэтому вопрос о provenance второй линии теперь лучше формулировать не как поиск issue/PR discussions, а как проверку содержимого закоммиченных artifact directories и конкретных путей из `EXPERIMENTS.md`. Если точные `report.json`, `progress.log` или `evaluations/*.eval.json` не открываются как raw-файлы, их нельзя использовать как подтверждение per-instance telemetry: в досье остаётся только то, что `EXPERIMENTS.md` называет эти пути и объясняет причину stricter eval attribution.

Переход к opencode + local MLX в `EXP-024` добавляет ещё один failure mode, который относится к агентскому workflow, а не к идее code-test graph. Для работы с `mlx_vlm.server` понадобился proxy `mlx_proxy.py`, потому что server не отдавал `index` в `tool_calls` и ставил `finish_reason: "stop"` даже при tool calls; opencode из-за этого считал, что модель закончила работу после 1-3 tool calls и не переходил к многошаговому редактированию [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). Там же зафиксирован practical bottleneck: при контексте около 36k prefill занимал примерно 54 секунды на раунд, а single instance требовал 10+ минут, поэтому 600s timeout становился частью результата [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). Для будущего текста это полезно как предупреждение: у TDAD второй линии качество может ограничиваться OpenAI-compatible API quirks, локальной скоростью inference и тайм-аутами не меньше, чем точностью dependency map.

### Где человек принимает решение

Во второй линии человек решает, можно ли доверять карте влияния и достаточно ли выбранного набора тестов. Если graph неполный, устаревший или построен только по части проекта, агент получит убедительный, но ложный маршрут проверки. Поэтому человеческое решение остаётся вокруг качества карты, границы запуска, release risk и случаев, где нужно расширить тестовый набор вручную.

`SKILL.md` добавляет ещё один human gate: каждое изменение должно включать хотя бы один новый тест, а если `.tdad/test_map.txt` не содержит запись, агент должен использовать fallback по naming convention или искать imports через `grep -r 'changed_module' tests/` [SKILL.md tdad-skill](https://github.com/pepealonso95/tdad-skill/blob/main/SKILL.md). Человек должен решить, когда такого fallback достаточно, а когда надо перестроить индекс, расширить карту или запускать полный suite.

Статья второй линии явно ограничивает применимость: эксперименты шли на 100 + 25 instances, без formal significance tests; оба model setups были меньшими локальными моделями; реализация в статье Python-only; static analysis не видит dynamic dispatch, monkey-patching и runtime-generated code; auto-loop проверялся на 10 instances per iteration [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2). Для человеческого решения это означает, что TDAD map нельзя считать oracle: владелец проекта должен знать, где static graph заведомо слаб.

## Сравнительная развилка внутри досье

Это досье допускает локальное сравнение, потому что без него аббревиатура `TDAD` становится опасной.

| Вопрос | Test-Driven AI Agent Definition | Test-Driven Agentic Development |
| --- | --- | --- |
| Что является объектом управления | Поведение tool-using agent | Изменение кода, которое делает coding agent |
| Главный артефакт | Behavioral spec, tests, compiled prompt/agent definition | Code-test dependency map, impacted tests, patch evidence |
| Где тесты включены в процесс | Тесты компилируют и проверяют prompt | Тесты направляют выбор проверок и self-correction при patch |
| Что выпускается | Проверенный prompt/agent artifact | Кодовое изменение с evidence по релевантным тестам |
| Основной риск | Подгонка prompt под видимые тесты, неполная спецификация, скрытые policy gaps, exclusion non-activating mutants | Устаревшая или неполная карта, неверный выбор тестов, ложная уверенность в coverage, blind spots static analysis |
| Ближайшая роль в AI-driven SDLC | Спецификация поведения агента и проверяемая сборка agent definition | Исполнение, проверка и regression guard для агентского изменения кода |
| Неудобная деталь источника | Полный pipeline завязан на Docker volumes; GitHub repo, `results/` и HuggingFace dataset разделяют разные слои SpecSuite-Core | Prompt-only TDD в эксперименте ухудшил regressions; paper-level 20-line skill, published `tdad-skill` и `EXPERIMENTS.md` описывают разные стадии метода |

Для будущей теории сильная формулировка такая: test-driven подходы в агентской разработке могут управлять разными объектами. Иногда тесты компилируют поведение агента. Иногда тесты ограничивают кодовое изменение. В обоих случаях тесты становятся внешним контуром реальности, но артефакты, риски и человеческие решения разные.

Табличная строка про неудобную деталь первой линии нуждается в более точной формулировке после pass 04: текущий README уже описывает `SupportOps` как worked example, а остальные specs как spec + mutation intents, причём specs/mutations/results также вынесены в HuggingFace dataset [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code). Поэтому риск здесь не «репозиторий противоречит статье», а «разные слои артефактов живут в разных местах»: GitHub repo, Docker volumes, `results/` и dataset. Для второй линии похожая оговорка: paper-level `20-line SKILL.md`, published `tdad-skill` и `EXPERIMENTS.md` описывают разные стадии одного метода, и их нельзя механически склеивать [SKILL.md tdad-skill](https://github.com/pepealonso95/tdad-skill/blob/main/SKILL.md) [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md).

## Связанные источники и соседние линии

`TDFlow` оставлен как соседний источник, а не как факт о двух основных TDAD. По arXiv abstract это test-driven agentic workflow для repository-scale software engineering, где human-written tests задают задачу, а sub-agents разделяют patch proposing, debugging, patch revision и optional test generation [TDFlow](https://arxiv.org/abs/2510.23761). Его нельзя использовать как подтверждение деталей TDAD, но он помогает увидеть соседний полюс: тесты могут быть не картой влияния и не компилятором prompt, а задачей, вокруг которой организован multi-agent repair. Для будущих проходов нужен полный PDF/HTML, потому что abstract заявляет 88.8% pass rate on SWE-Bench Lite, 94.3% on SWE-Bench Verified и только 7 случаев test hacking после ручной проверки 800 запусков [TDFlow](https://arxiv.org/abs/2510.23761).

Pass 07 прочитал HTML полной статьи `TDFlow` и уточняет статус этого соседнего источника. `TDFlow` начинается с human-written reproduction tests или, опционально, с генерации таких тестов; затем тесты запускаются, `Explore Files` предлагает repository-level patch только с правами чтения файлов, поиска и просмотра дерева, `Revise Patch` чинит плохо применимый patch без изменения вставляемого кода, `Debug One` разбирает один падающий тест через ограниченный debugger и отдаёт отчёт, а агрегированные debug reports возвращаются в следующий цикл `Explore Files` [HTML статьи TDFlow](https://ar5iv.org/html/2510.23761v2). Эта линия полезна для будущей теории как contrast case: тесты здесь являются не dependency map и не hidden evaluation, а явным входом задачи для workflow. При этом источник сам задаёт жёсткую границу применимости: при неверных reproduction tests у процесса нет механизма исправить их позже, а rigid multi-agent workflow требует более сложной среды и не имеет early stopping, если тесты нерешаемы [HTML статьи TDFlow](https://ar5iv.org/html/2510.23761v2).

Для TDAD-досье особенно важны две детали `TDFlow`, потому что они показывают риск слишком общего слова `test-driven`. Во-первых, `TDFlow` намеренно даёт обычно скрытые human-written SWE-Bench tests системе и модифицирует baseline-системы для честного сравнения; эти числа нельзя переносить на setting, где агент видит только issue description [HTML статьи TDFlow](https://ar5iv.org/html/2510.23761v2). Во-вторых, авторы считают test hacking отдельным failure mode: вручную проверяют 300 SWE-Bench Lite и 500 SWE-Bench Verified runs, отмечают прямое изменение тестов, skip/disable, ослабление assertions, test-only logic, hardcoded outputs и другие признаки, а найденные случаи засчитывают как failures [HTML статьи TDFlow](https://ar5iv.org/html/2510.23761v2). Это не факт о TDAD, но полезная будущая проверочная линза для любых test-driven agent workflows.

Страница AI Paper Digest по `2603.17973` остаётся вторичным навигационным источником, а не равным первоисточником [AI Paper Digest TDAD](https://ai-paper-delta.vercel.app/en/papers/2603.17973). Её можно использовать для поиска связанных ссылок, но не для переноса фактов, если те не проверены по статье или репозиторию.

## Failure modes / overuse risks

- Смешать две линии TDAD в один метод и получить ложный вывод: будто prompt compilation и regression impact analysis являются частями одного инструмента.
- Назвать обе линии TDD в привычном смысле и потерять специфику. Первая линия ближе к evaluation-driven prompt compiler; вторая — к graph-based test selection для agentic patching.
- Принять проход тестов за гарантию качества. В первой линии hidden tests защищают от части подгонки, но не покрывают всю политику поведения. Во второй линии targeted tests полезны только настолько, насколько хороша карта влияния.
- Не заметить, что первая линия сама вводит anti-gaming infrastructure. Если hidden tests доступны PromptSmith, visible tests можно переписывать из compilation container или MutationSmith работает в той же сессии, метод теряет ключевую защиту [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1).
- Переиспользовать headline metrics без условий эксперимента. Числа из статей нужно хранить рядом с моделью, набором задач, harness и baseline.
- Автоматизировать слишком рано. Обе линии требуют хорошей спецификации или хорошей карты. Если исходный артефакт слабый, агент получит формальную процедуру с плохим сигналом.
- Перепутать процедурную инструкцию с рабочим контекстом. Вторая линия показывает, что TDD instructions сами по себе могут ухудшить regression rate, если агент не получил конкретную карту связи кода и тестов.
- Перенести вывод второй линии на frontier models без проверки. Авторы сами ограничивают finding меньшими локальными моделями и допускают, что при больших context windows TDD Prompting Paradox может проявляться иначе [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2).
- Считать repository README равным полной статье. В первой линии кодовый репозиторий сейчас явно показывает не все paper-level артефакты; во второй линии нужно отделять production CLI, `tdad-skill` и экспериментальную оценку на SWE-bench.
- Считать published skill тем же самым артефактом, который оценивался в статье. Paper-level runtime skill был намеренно коротким и работал вместе со static `test_map.txt`; опубликованный `tdad-skill` уже включает metadata, multi-language runners, обязательный новый тест и fallback-процедуры [SKILL.md tdad-skill](https://github.com/pepealonso95/tdad-skill/blob/main/SKILL.md). Это полезно для практики, но при переносе чисел из статьи нужно указывать, какой вариант skill использовался.
- Не увидеть отрицательный `EXP-026` во второй линии. Если дать агенту Neo4j-backed skill как дополнительную процедуру, но не убедиться, что skill обнаружен, вызван и помещается в бюджет времени, результат может стать хуже baseline: меньше generated patches и меньше resolved instances при той же нулевой regression rate [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md).
- Использовать `ALWAYS_CREATE_TICKET`/`M_ALWAYS_TICKET` как простую иллюстрацию killed/survived mutant без версии. В текущих доступных слоях есть конфликт между raw README, rendered GitHub page и dataset naming; безопасно говорить только о самом mutation intent, а не о стабильном проценте, пока не проверены commit hash и per-run JSON [raw README tdad-paper-code](https://raw.githubusercontent.com/f-labs-io/tdad-paper-code/main/README.md) [HuggingFace SpecSuite-Core](https://huggingface.co/datasets/f-labs-io/SpecSuite-Core).
- Читать eval-файлы второй линии без provenance. `EXP-021j`/`EXP-021l` показывают, что stale `evaluation_results/*.json` мог исказить resolved status, поэтому для серьёзного вывода нужен exact eval path, fresh timestamp и связь с конкретным `benchmark_runs/.../report.json` [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md).
- Считать OpenAI-compatible local model server прозрачной заменой облачного API. В `EXP-024` opencode потребовал `mlx_proxy.py` из-за несовместимых `tool_calls` и `finish_reason`, а локальная скорость inference и 600s timeout стали частью observed generation/resolution rate [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md).
- Переносить `TDFlow`-числа в TDAD без смены постановки задачи. `TDFlow` в ключевом режиме получает human-written reproduction tests, то есть работает с другим contract: человек задаёт проверяемое поведение тестами, а workflow решает эти тесты. Это нельзя склеивать с TDAD code-test impact analysis, где карта выбирает impacted tests для уже заданного patching task [HTML статьи TDFlow](https://ar5iv.org/html/2510.23761v2).
- Считать test-driven workflow защищённым от test hacking только потому, что тесты запущены. `TDFlow` прямо вводит запреты на изменение test folders, ограничение видимости файловой системы, ручную проверку patch/logs и rubric признаков test hacking; значит, test-driven контур требует отдельной anti-gaming политики, а не только команды запуска тестов [HTML статьи TDFlow](https://ar5iv.org/html/2510.23761v2).
- Продолжать широкое накопление источников после того, как основные статьи, репозитории, HTML-версии, dataset-слой и экспериментальный журнал уже прочитаны. После pass 08 более полезная работа выглядит как targeted provenance audit: открыть конкретные `report.json`, `progress.log`, `evaluations/*.json`, issue/PR discussions и commit history для спорных мест, а не искать ещё один общий обзор TDAD. Иначе досье начнёт накапливать навигационный шум вместо новой фактуры.

Source-opening pass добавил более жёсткое ограничение по метрикам graph-based TDAD. В Phase 1 SWE-bench Verified на 100 instances TDAD / GraphRAG+TDD reduced test-level regression rate from 6.08% to 1.82%, но resolution rate slightly decreased and generation rate was lower; authors note denominator effects in instance-level regression rates [TDAD arXiv HTML](https://arxiv.org/html/2603.17973v2). Поэтому headline “70% regression reduction” нельзя читать как free improvement: метод может заставить agent abstain more often when risk is visible.

Experiment log in the older TDAD repository is also useful as warning material: one attempted Qwen agent loop hit 500 errors at around 28K context, sliding window dropped too much context, and the model kept re-reading files before the author rewrote the approach into a single-shot API call [TDAD EXPERIMENTS.md](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). Это не основной результат paper, но хороший источник для нашей темы: agentic test workflows can fail through context management and loop fragility, not only through bad tests.

## Аналитические заметки предыдущих проходов

### Что должно попасть в теорию

- Различение двух объектов test-driven управления: поведение агента и кодовое изменение.
- Идея prompt как compile artifact: agent definition можно собирать из behavioral spec под тесты, а не только писать вручную.
- Идея code-test graph как runtime aid для агента: не «запусти всё», а выбери проверки по карте влияния.
- Предупреждение о ложной уверенности: тестовый контур повышает наблюдаемость, но качество зависит от спецификации, карты, скрытого набора, harness и человеческого решения о выпуске.
- Различение тестов как компилятора поведения, тестов как маршрутизатора проверки и тестов как задачи для multi-agent repair. После чтения полной статьи `TDFlow` можно использовать как соседний contrast case: workflow не строит code-test graph и не компилирует prompt, а принимает reproduction tests как главный вход, разносит patch proposal, patch revision и per-test debugging по sub-agents и считает плохую генерацию тестов главным пределом автономности [HTML статьи TDFlow](https://ar5iv.org/html/2510.23761v2).
- Тезис `context over procedure` из второй линии: для coding agents полезнее дать короткий структурный контекст о том, какие tests at risk, чем длинно описывать процедуру TDD; но это утверждение нужно привязывать к меньшим локальным моделям и условиям эксперимента.
- Тезис первой линии о separation of artifact access: hidden tests, visible tests, compiler и evaluator должны быть разделены инфраструктурно, иначе test-driven prompt compilation легко превращается в optimization against exposed tests.

### Что лучше уйдёт в Handbook / Fieldbook

- Практический чек для agent prompt: есть ли behavioral spec, visible tests, hidden tests, mutation/evolution checks и release decision.
- Практический чек для coding agent: есть ли карта source/test, как она обновляется, какие impacted tests выбраны, где сохранён вывод тестов, кто расширяет набор при сомнении.
- Практический чек по `.tdad/test_map.txt`: индекс создан, изменённые файлы найдены через `grep`, fallback зафиксирован явно, каждый patch добавляет хотя бы один тест, а отсутствие записи в карте не считается молчаливым разрешением пропустить проверку.
- Fieldbook-паттерн: если агентский рабочий процесс даёт «успешный diff» без связанного тестового подтверждения, результат нельзя считать завершённым.
- Соседний Fieldbook-паттерн из `TDFlow`: если человек может написать точные reproduction tests, агентский workflow можно сузить до test-resolution loop, но тогда нужны отдельные запреты на test hacking, явный контроль качества самих reproduction tests и остановка, если тесты неверны или нерешаемы [HTML статьи TDFlow](https://ar5iv.org/html/2510.23761v2).


## Краткая рабочая карта

TDAD в этом досье остаётся сравнительным случаем двух разных линий, которые нельзя сливать в один метод. Первая линия — Test-Driven AI Agent Definition / SpecSuite-Core: спецификация, generated tests, hidden tests, mutations and prompt compilation. Вторая линия — Test-Driven Agentic Development: graph-based impact analysis для AI coding agents, где агент после изменения находит affected tests through `.tdad/test_map.txt` and runs them before submitting patch. Source-opening pass усиливает вторую линию: её ядро не “TDD instructions”, а конкретная карта связи source files and tests.


Source-opening уточнение по graph-based TDAD: paper прямо формулирует ключевой тезис — agents do not need to be told how to do TDD; they need to know which tests to check [TDAD arXiv HTML](https://arxiv.org/html/2603.17973v2). Pipeline имеет две стадии: indexing строит code–test graph, а export создаёт static `test_map.txt`; at runtime agent only needs grep and pytest, without graph database, MCP server or API calls. Это важно для теории: TDAD deliberately avoids adding a heavy runtime orchestrator; expensive graph work is pushed before the agentic coding loop.

Graph schema включает four node types — File, Function, Class, Test — and five edge types: CONTAINS, CALLS, IMPORTS, TESTS, INHERITS [TDAD arXiv HTML](https://arxiv.org/html/2603.17973v2). Indexing pipeline строится из AST Parser, Graph Builder and Test Linker. Test Linker uses naming conventions, prefix matching and directory proximity to connect tests to source code. Это даёт методологии конкретное failure surface: если linking heuristics плохи для проекта, impacted-test map даст агенту ложный verification target.
