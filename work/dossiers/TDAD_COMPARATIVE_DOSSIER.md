# Сравнительное досье TDAD

Рабочее досье по двум линиям, которые используют одну аббревиатуру `TDAD`. Документ нужен не как финальная сравнительная глава, а как буфер фактуры: какие проблемы решают источники, какие рабочие процессы предлагают, какие артефакты создают, где живут тесты и где остаются человеческие решения.

## Зачем это досье нужно

`TDAD` нельзя переносить в будущую теорию как один метод. В первоисточниках уже видны две разные линии:

- `Test-Driven AI Agent Definition: Compiling Tool-Using Agents from Behavioral Specifications` рассматривает промпт и системные инструкции агента как артефакт, который компилируется из поведенческой спецификации под набор исполняемых тестов [статья Test-Driven AI Agent Definition](https://arxiv.org/abs/2603.08806).
- `Test-Driven Agentic Development: Reducing Code Regressions...` относится к кодинговым агентам: перед изменением кода метод строит карту связей между исходными файлами и тестами, чтобы агент выбирал релевантные проверки и исправлял регрессии до завершения задачи [статья Test-Driven Agentic Development](https://arxiv.org/abs/2603.17973).

Общая идея у этих линий похожа: тесты становятся не поздней ручной проверкой, а частью управляющего контура для агента. Различие принципиальное. Первая линия управляет поведением самого агента и выпускает проверенный промпт. Вторая линия управляет изменением в кодовой базе и даёт агенту доказательный маршрут через тесты.

## Линия 1: Test-Driven AI Agent Definition

### Проблема

Статья `2603.08806` исходит из практической проблемы промышленных агентов с инструментами. Небольшое изменение промпта может изменить поведение без явного сбоя: агент нарушает политику, неверно использует инструмент, теряет следование рабочему регламенту или начинает давать ответы, которые выглядят правдоподобно, но не проходят скрытые условия качества [статья Test-Driven AI Agent Definition](https://arxiv.org/abs/2603.08806). Обычная уверенность в хорошо написанном промпте здесь недостаточна, потому что поведение агента нужно проверять на исполняемых сценариях.

В формулировке статьи проблема не сводится к удобству инженерии промптов. Авторы прямо привязывают TDAD к промышленному развёртыванию агентов с инструментами: нужна измеримая проверка соблюдения поведенческих требований, потому что регрессии промпта, неверное использование инструментов и нарушения политики могут проявиться только после выпуска [статья Test-Driven AI Agent Definition](https://arxiv.org/abs/2603.08806). Для будущей теории это даёт важный угол: агентская инструкция становится объектом выпуска, а не внутренней заметкой разработчика.

### Рабочий процесс

Источник описывает TDAD как цикл компиляции `agent definition` из поведенческих спецификаций. Инженер задаёт спецификацию поведения; один кодинговый агент превращает её в исполняемые тесты; другой кодинговый агент итеративно уточняет промпт и системные инструкции, пока видимые тесты не проходят [статья Test-Driven AI Agent Definition](https://arxiv.org/abs/2603.08806). В результате появляется не новый код приложения, а промпт агента, который прошёл тестовый контур и может быть оценён на скрытом наборе.

Это важная смена роли промпта: он перестаёт быть текстом, который человек вручную «улучшает», и становится компилируемым артефактом. Для будущей теории это полезно рядом с темой памяти агента и проверочного контура: поведение агента выносится из неявной инструкции в спецификацию, тесты и повторяемый процесс сборки.

Референсный репозиторий уточняет роли внутри процесса. Продуктовая команда пишет `spec.yaml`; построитель тестов должен переводить спецификацию в исполняемые тесты; кодинговый агент итеративно правит промпт до прохождения тестов; итоговый промпт становится выпускаемым агентом с регрессионным набором тестов, который можно расширять [README tdad-paper-code](https://raw.githubusercontent.com/f-labs-io/tdad-paper-code/main/README.md). В Docker-пайплайне эти роли разложены на сервисы: `testsmith` генерирует тесты из спецификации, `compiler` компилирует промпт до прохождения `visible tests`, `evaluate` запускает `hidden tests`, `mutation` выполняет мутационное тестирование, `agent` даёт интерактивный чат с собранным агентом [README tdad-paper-code](https://raw.githubusercontent.com/f-labs-io/tdad-paper-code/main/README.md).

Полный HTML/PDF статьи даёт более точное устройство пайплайна, чем краткая аннотация. Спецификация описывается как YAML-документ, где есть `tools`, `policies`, `decision tree`, `response contract`, `test guidance` и `mutation intents`; автор подчёркивает, что спецификация остаётся единым источником истины, а тесты реализуют её, но не заменяют [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1). `TestSmith` проходит `decision tree` и создаёт тесты `MFT`, `INV` и `DIR`, включая детерминированные фикстуры и `canary values` для обнаружения утечки PII; `PromptSmith` запускает `visible suite`, группирует сбои по первопричине и делает минимальную правку промпта и описаний инструментов под крупнейший кластер сбоев; `Built Agent` отвечает через выделенный инструмент `respond`, чтобы проверочный контур проверял структурированные поля и трассу вызовов инструментов, а не свободный текст; `MutationSmith` работает только после компиляции и не участвует в улучшении промпта [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1).

Важная деталь для будущей теории: в первой линии компилируется не только `system prompt`. Статья прямо называет описания инструментов первоклассными артефактами: `PromptSmith` может уточнять `preconditions`, `sequencing constraints` и `return-value semantics` инструмента, потому что описания инструментов лежат в контексте рядом с `system prompt` и сильно влияют на использование инструментов [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1). Значит, `agent definition` здесь ближе к связке промпт + описания инструментов + контракт ответа, а не к одному текстовому промпту.

### Артефакты

По статье и референсному репозиторию нужно сохранять следующие артефакты:

- `behavioral specification` — исходное описание поведения агента;
- `specs/core/{spec}/v{1,2}/spec.yaml` — файловая форма спецификации: v1 как базовая версия и v2 как сценарий эволюции;
- `visible tests` — тесты, по которым идёт компиляция промпта;
- `hidden tests` — удержанный набор для проверки обобщения и защиты от подгонки под видимый набор;
- `tests_visible/core/{spec}/*.py` и `tests_hidden/core/{spec}/*.py` — конкретные директории для двух контуров проверки;
- `prompt under compilation` — промпт или системные инструкции, которые итеративно меняются;
- `agent_artifacts/core/{spec}/seed_prompt.txt` и `agent_artifacts/core/{spec}/system_prompt.txt` — минимальный стартовый промпт и скомпилированный агентский промпт в репозитории;
- `agent artifact` — собранный промпт или `agent definition` после прохождения тестов;
- `mutation packs` — варианты намеренно испорченного поведения для семантического мутационного тестирования;
- `spec evolution scenarios` — изменения требований, на которых измеряется сохранение поведения после эволюции спецификации [статья Test-Driven AI Agent Definition](https://arxiv.org/abs/2603.08806) [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code).
- `test_guidance` — примеры, которые проясняют неоднозначные термины политики; без них TestSmith может породить конфликтующие тесты для одного и того же требования, например одновременно ожидать уточнение и прямой ответ для близких запросов [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1).
- инструмент `respond` / структурированный контракт ответа — контракт финального ответа агента: `decision`, `node_id`, `evidence`, `user_message` и другие поля, проверяемые по схеме, по которым можно делать детерминированные утверждения в тестах [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1).
- оптимизированные описания инструментов — часть скомпилированного агентского артефакта, если `PromptSmith` уточняет, когда инструмент должен вызываться, какие поля нужны до вызова и что делать с результатом [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1).

Репозиторий `f-labs-io/tdad-paper-code` важен потому, что даёт файловую форму, которой не хватает в кратком пересказе статьи. В нём указаны `specs/core/{spec}/v{1,2}/spec.yaml`, `tests_visible/core/{spec}/*.py`, `tests_hidden/core/{spec}/*.py`, `mutation_packs/core/{spec}/*.patch`, `agent_artifacts/`, `tdadlib/`, `results/` и `scripts/` [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code).

Текущий README репозитория первой линии уточняет прежний открытый вопрос о статусе `SpecSuite-Core`: `SupportOps` описан как рабочий пример со спецификацией, сгенерированными тестами, фикстурами, мутациями и результатами; `DataInsights`, `IncidentRunbook` и `ExpenseGuard` имеют спецификацию + `mutation intents`, а тесты для них предполагается генерировать запуском пайплайна [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code). Там же указано, что спецификации и результаты бенчмарка доступны как набор данных HuggingFace `f-labs-io/SpecSuite-Core` с конфигурациями `specs` (8 версий спецификаций), `mutations` (27 `mutation intents`) и `results` (24 запуска пайплайна) [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code). Это важное уточнение происхождения: репозиторий разделяет рабочий пример, исполняемые спецификации и вынесенный слой набора данных/результатов.

Проход 05 уточняет слой набора данных первой линии. Страница HuggingFace `f-labs-io/SpecSuite-Core` описывает `SpecSuite-Core` как 4 глубоко специфицированные поведенческие спецификации и явно перечисляет, что каждая спецификация включает типизированные схемы инструментов, политики с приоритетами, деревья решений, контракты ответа, `mutation intents` и эволюцию v1 -> v2 [HuggingFace SpecSuite-Core](https://huggingface.co/datasets/f-labs-io/SpecSuite-Core). В таблице карточки набора данных указаны домены и изменения v2: `SupportOps` имеет 7 инструментов, 4 политики и обнаружение злоупотреблений в v2; `DataInsights` — 3 инструмента, 4 политики и запросы с учётом стоимости; `IncidentRunbook` — 6 инструментов, 4 политики и отслеживание влияния на клиентов; `ExpenseGuard` — 6 инструментов, 5 политик и gate менеджерского согласования [HuggingFace SpecSuite-Core](https://huggingface.co/datasets/f-labs-io/SpecSuite-Core). Это делает набор данных не просто архивом результатов, а машинно читаемым слоем спецификаций, который можно использовать для проверки утверждений статьи без запуска всего Docker-пайплайна.

У набора данных есть три конфигурации, которые полезно держать отдельно от GitHub-репозитория: `specs` содержит 8 строк, то есть 4 спецификации x 2 версии, с полем `spec_yaml`; `mutations` содержит 27 строк с полями вроде `mutation_pack_id`, `spec_lineage`, `spec_version`, `mutation_id`, `severity`, `category`, `intent`, `activation_probe`; `results` содержит 24 строки сквозных запусков пайплайна и поля метрик `vpr_percent`, `hpr_percent`, `mutation_score`, `total_cost_usd` [HuggingFace SpecSuite-Core](https://huggingface.co/datasets/f-labs-io/SpecSuite-Core). Для будущей теории это важно как пример разделения исходного контракта, намеренных поведенческих поломок и результатов прогонов: спецификация, `mutation intent` и результат пайплайна являются разными артефактами, а не одним «бенчмарком».

### Команды и среда

Для локального слоя досье важны команды из README репозитория: запуск видимых тестов через `pytest -q tests_visible -m visible` и компиляция промпта через `python scripts/compile_prompt.py --spec ... --prompt ... --test-cmd ... --max-iters 8` [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code). Полный пайплайн в репозитории опирается на Docker-сервисы `testsmith`, `compiler`, `evaluate`, `mutation`, `agent`, `login`, `init-volumes` [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code).

Неудобная, но важная деталь: такой пайплайн нельзя описывать как обычный локальный запуск пары скриптов. В прежних проходах зафиксировано, что полная сборка работает через тома Docker; локально созданные файлы могут быть недоступны контейнерам без пересборки и переинициализации тома. Это нужно проверить по актуальному README перед финальным использованием, но уже сейчас это хороший пример хрупкости метода: сильный проверочный контур требует аккуратной инфраструктуры.

README делает эту хрупкость явной: полный Docker-процесс работает со слоем томов, а не с локальными файлами; после локальных изменений нужно выполнять `docker compose build` и `docker compose run --rm init-volumes`, иначе контейнеры могут не увидеть обновлённые `tests_visible`, `tests_hidden` или `agent_artifacts` [README tdad-paper-code](https://raw.githubusercontent.com/f-labs-io/tdad-paper-code/main/README.md). Там же описан порядок эволюции v1 -> v2: сначала скомпилировать v1, затем сгенерировать тесты v2 через `testsmith-v2`, собрать промпт v2 через `compiler-v2` с начальным состоянием от v1 (`warm start`) и измерить SURS командой `evaluate-surs`, где тесты v1 запускаются против промпта v2 [README tdad-paper-code](https://raw.githubusercontent.com/f-labs-io/tdad-paper-code/main/README.md).

### Проверка результата

Источник вводит три разных проверочных механизма:

- разделение на `visible tests` и `hidden tests`: компиляция идёт по видимому набору, а итоговый агент проверяется на скрытом;
- семантическое мутационное тестирование: создаются правдоподобные неисправные варианты промпта, и тестовый набор должен их поймать;
- сценарии эволюции спецификации: проверяется, сохраняется ли безопасность поведения при изменении требований [статья Test-Driven AI Agent Definition](https://arxiv.org/abs/2603.08806).

HTML статьи уточняет механику разделения: `visible tests` занимают примерно 40-70% и используются в цикле компиляции; `hidden tests` занимают 30-60%, не показываются `PromptSmith` и используются для отчётности. Для каждой ветки `decision tree` первичный `MFT` обычно попадает в видимый набор, а варианты `INV`/`DIR`, граничные случаи и метаморфные тесты уходят в скрытый набор [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1). В режиме бенчмарка тесты генерируются заново для каждой попытки, и скрытый набор заморожен внутри этой попытки; в промышленном режиме падающие скрытые тесты могут переводиться в видимый набор, после чего агент пересобирается [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1). Это важная граница: `hidden tests` не навсегда «секретные», но на момент компиляции они отделены от оптимизатора.

Мутационное тестирование в этой линии устроено не как синтаксический патч фиксированного файла. `MutationSmith` получает `mutation intent`, создаёт правдоподобный неисправный вариант промпта и сначала проверяет `activation probe`: действительно ли мутант поменял поведение. Неактивирующие мутанты после заданного числа попыток исключаются из `mutation score`, а убитый мутант означает, что хотя бы один видимый тест упал на мутированном промпте [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1). Это одновременно сила и ограничение: метод пытается не наказывать набор тестов за эквивалентных мутантов, но исключение может завысить качество тестов, если «неактивированный» мутант на самом деле указывает на слепое пятно.

Результат мутационного тестирования на уровне репозитория нужно держать отдельно от агрегата на уровне статьи. GitHub README сейчас заявляет для `SupportOps v1` 100% `mutation score`: 6/6 активированных мутантов убиты во всех запусках, 7 намерений всего и 1 неактивированный мутант исключён; детали по отдельным запускам предлагается смотреть в `results/supportops_v1_*.json`, а `mutation intents` на уровне статьи — в таблице B1 [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code). Это снимает часть путаницы с прежним примером `M_ALWAYS_TICKET`: текущий README больше не даёт тот же 80% локальный список как основной статус, поэтому старую находку надо рассматривать как состояние конкретного README-среза, а не как стабильный результат всей линии.

Проход 05 обнаружил, что статус `mutation score` у `SupportOps v1` нельзя считать закрытым без проверки на уровне коммита. Отрендеренная страница GitHub и поисковый фрагмент показывают вариант README со 100% `mutation score`: 6/6 активированных мутантов убиты, 7 намерений всего и 1 неактивированный мутант исключён [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code). Но сырой README по `main` в этом же проходе отдаёт более старую формулировку: 80% `mutation score`, 4/5 убиты, выживший мутант `M_ALWAYS_TICKET` [raw README tdad-paper-code](https://raw.githubusercontent.com/f-labs-io/tdad-paper-code/main/README.md). Одновременно набор данных HuggingFace содержит `mutation intent` `ALWAYS_CREATE_TICKET` с уровнем `minor` и категорией `quality_regression` для `SupportOps v1`, где агент всегда создаёт тикет поддержки вместо выполнения запроса [HuggingFace SpecSuite-Core](https://huggingface.co/datasets/f-labs-io/SpecSuite-Core). Поэтому аккуратная формулировка сейчас такая: `ALWAYS_CREATE_TICKET`/`M_ALWAYS_TICKET` — реальный мутационный угол про ложную эскалацию, но его статус «убит/выжил» зависит от слоя источника и версии README; использовать его как стабильное число нельзя до проверки хэша коммита, строк набора данных и `results/supportops_v1_*.json`.

Для эволюции спецификации статья вводит `SURS`: компиляция v2 стартует от артефакта промпта v1, но запускается только на тестах v2; удержанные инвариантные тесты v1 не показываются `PromptSmith`, а после сборки агент v2 проверяется на сохранение этих инвариантов [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1). Отдельно автор вводит `RPR` как промышленную рекомендацию против стохастичности: одного результата «проход/сбой» недостаточно, для обычных сценариев предлагается несколько повторных запусков с порогом, а для PII/auth — политика нулевых сбоев; в самом исследовании RPR не измерялся [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1).

Заявленные числа нужно хранить именно как утверждения источника, не как установленный общий результат для всех агентов: SpecSuite-Core включает четыре глубоко специфицированных агента по соблюдению политик, аналитике, привязанной к данным, следованию рабочему регламенту и детерминированному принуждению; статья заявляет 24 независимые попытки, 92% успешной компиляции v1, 97% среднего прохождения скрытых тестов, 58% успешной компиляции эволюционировавших спецификаций, `mutation scores` 86-100%, 78% прохождения скрытых тестов v2 и 97% `regression safety scores` [статья Test-Driven AI Agent Definition](https://arxiv.org/abs/2603.08806). Для будущей главы нужны полные PDF/HTML, определения метрик, случаи сбоя и содержимое проверочного контура.

SpecSuite-Core в полной статье описан как маленький, но глубокий протокол бенчмарка, а не как широкий набор одношаговых задач. Четыре спецификации имеют 10-14 узлов решений, 40-60% скрытых тестов, каталоги `mutation intents` по 5-7 намерений на спецификацию и сценарии эволюции v1 -> v2 [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1). Домены тоже важны: `SupportOps` проверяет авторизацию перед действием, отказ при PII, соответствие плану и триггеры эскалации; `DataInsights` проверяет SQL grounding, числовую точность, обнаружение неоднозначности и запросы с учётом стоимости; `IncidentRunbook` проверяет порядок «сначала свидетельства», маршрутизацию по серьёзности и соблюдение рабочего регламента; `ExpenseGuard` проверяет лимиты расходов, конвертацию FX, требования к чекам и менеджерское согласование [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1).

Репозиторий выглядит уже, чем формулировка статьи, но не как простое противоречие: GitHub-репозиторий содержит рабочий пример и исполняемый пайплайн, а часть артефактов бенчмарка вынесена в слой набора данных/результатов [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code). Поэтому вопрос по воспроизводимости нужно формулировать точнее: какие именно версии спецификаций, сгенерированные тесты, `mutation intents` и результаты отдельных попыток можно получить из GitHub-репозитория, какие лежат в наборе данных HuggingFace, а какие остаются только в статье.

Полная статья называет выживших мутантов на уровне SpecSuite-Core: `HALLUCINATE_NUMBERS` выжил в DataInsights v1 и был закрыт в v2; `SKIP_RUNBOOK_LOOKUP` выжил в IncidentRunbook v1, а затем был закрыт улучшенным тестовым покрытием в v2 [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1). Это лучше использовать как факт на уровне статьи. Старый пример `M_ALWAYS_TICKET` на уровне репозитория, зафиксированный в прежнем проходе по сырому README-срезу, теперь нужно оставить только как историческую заметку до проверки истории коммитов, потому что текущий GitHub README для `SupportOps v1` заявляет 100% `mutation score` с 1 исключённым неактивированным мутантом [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code).

Проход 09 уточняет статус этого конфликта как вопрос происхождения факта, а не как содержательный вывод о методе. Публичная страница `f-labs-io/tdad-paper-code` показывает всего 2 коммита, `Issues 0`, `Pull requests 0` и README-слой, где `SupportOps v1` описан как рабочий пример с 6/6 убитыми активированными мутантами и одним исключённым неактивированным мутантом [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code). При этом слой набора данных `SpecSuite-Core` отдельно хранит `mutations` с 27 строками и использует близкое имя намерения `ALWAYS_CREATE_TICKET`, а не старое `M_ALWAYS_TICKET` [HuggingFace SpecSuite-Core](https://huggingface.co/datasets/f-labs-io/SpecSuite-Core). Без точного хэша коммита или сырого JSON по отдельным запускам безопасная будущая формулировка такая: первая линия демонстрирует семантическое мутационное тестирование на отдельных доменных сценариях, но спорный мутант про создание тикета нельзя использовать как стабильный числовой пример.

### Где человек принимает решение

Человек остаётся владельцем спецификации и критерия выпуска. Автоматический цикл может сгенерировать тесты и улучшать промпт, но именно человек должен решить, что поведенческая спецификация выражает нужную политику, что `hidden tests` не подменяют реальное качество и что агент можно выпускать после прохождения проверочного контура. Важно не писать, что TDAD «гарантирует» корректность агента: источник показывает измеряемую проверку поведения на конкретном наборе тестов, а не доказательство общей безопасности.

`tdad-skill` показывает минимальную агентскую поверхность второй линии. Skill активируется, когда в репозитории есть `.tdad/`; разовая настройка выполняется командами `pip install tdad` и `tdad index .`, которые создают `.tdad/test_map.txt`. Для каждого изменённого файла агент выполняет `grep 'path/to/changed_file' .tdad/test_map.txt`, собирает уникальные тестовые файлы и запускает проектное средство запуска тестов; проекты не на Python требуют дополнительных зависимостей tree-sitter, например `tdad[treesitter-all]` [tdad-skill SKILL.md](https://raw.githubusercontent.com/pepealonso95/tdad-skill/main/SKILL.md). Это полезно для Handbook: TDAD можно описать как маленький skill для регрессионной защиты, а не как универсальный фреймворк разработки.

Референсная реализация SpecSuite-Core в другой линии TDAD хранит спецификации, `visible tests`, `hidden tests`, `mutation packs` и результаты; README описывает ментальную модель так: продуктовая команда пишет спецификацию, построитель тестов переводит её в исполняемые тесты, кодинговый агент итерирует до прохождения тестов, а итоговый промпт становится поставляемым агентом с регрессионным набором [TDAD Reference Implementation README](https://raw.githubusercontent.com/f-labs-io/tdad-paper-code/main/README.md). Это усиливает различение: первая линия компилирует поведение агента из тестов; вторая линия защищает изменения кода от регрессий через целевой выбор тестов.

## Линия 2: Test-Driven Agentic Development

### Проблема

Статья `2603.17973` решает другую задачу. Кодинговые ИИ-агенты могут закрывать задачи и одновременно создавать регрессии. В такой ситуации `resolution rate` недостаточен: агент может выглядеть успешным по основному тесту или бенчмарку, но ломать смежное поведение [статья Test-Driven Agentic Development](https://arxiv.org/abs/2603.17973). Эта линия TDAD переносит внимание с «агент написал патч» на «агент выбрал и прошёл тесты, которые связаны с затронутыми файлами».

Важная отрицательная находка второй статьи: добавление процедурных TDD-инструкций без контекста целевых тестов ухудшило `regression rate` до 9.94%, то есть оказалось хуже `vanilla baseline` с 6.08% [статья Test-Driven Agentic Development](https://arxiv.org/abs/2603.17973). В репозитории это сформулировано как вывод о небольших моделях с открытыми весами: им полезнее конкретный контекст, полученный из графа, чем длинная предписывающая процедура [репозиторий TDAD](https://github.com/pepealonso95/TDAD). Для будущего текста это сильная оговорка против простого совета «попросите агента делать TDD».

### Рабочий процесс

Метод строит карту влияния между исходными файлами и тестами до изменения кода. Агент получает эту карту как лёгкий агентский skill: статический текстовый файл, который он может запрашивать во время выполнения [статья Test-Driven Agentic Development](https://arxiv.org/abs/2603.17973). Практический смысл: агенту не просто говорят «запусти тесты», а дают структуру, по которой можно выбрать целевые тесты для конкретного патча.

В рабочем цикле это выглядит так:

1. Построить или загрузить карту зависимостей между кодом и тестами.
2. Дать агенту задачу исправления вместе с доступом к карте.
3. Во время изменения агент выбирает затронутые тесты.
4. Если тесты показывают регрессию, агент использует сигнал для самокоррекции.
5. Завершение должно опираться не только на патч, но и на свидетельства по релевантным тестам [статья Test-Driven Agentic Development](https://arxiv.org/abs/2603.17973) [репозиторий TDAD](https://github.com/pepealonso95/TDAD).

`tdad-skill` превращает это в очень конкретную процедуру, обращённую к агенту. Если `.tdad/test_map.txt` отсутствует, один раз выполняются `pip install tdad` и `tdad index .`; для проектов не на Python README предлагает `pip install tdad[treesitter-all]` и `tdad index . --languages python,javascript` или `TDAD_LANGUAGES=...` [SKILL.md tdad-skill](https://github.com/pepealonso95/tdad-skill/blob/main/SKILL.md) [README tdad-skill](https://github.com/pepealonso95/tdad-skill). После изменения кода агент должен для каждого изменённого файла искать строку через `grep 'path/to/changed_file' .tdad/test_map.txt`, где формат строки описан как `source_file: test_a test_b test_c`, затем собрать уникальные затронутые тесты, запустить их, исправлять регрессии до зелёного результата и только потом двигаться к сдаче патча [SKILL.md tdad-skill](https://github.com/pepealonso95/tdad-skill/blob/main/SKILL.md).

CLI-слой репозитория даёт более машинную форму того же процесса: `tdad index /path/to/your/repo`, `tdad impact /path/to/your/repo --files src/module.py`, `tdad run-tests /path/to/your/repo --tests tests/test_module.py::test_foo`, `tdad stats /path/to/your/repo` [README tdad-skill](https://github.com/pepealonso95/tdad-skill). По умолчанию используется бэкенд NetworkX без внешнего сервиса; Neo4j остаётся опциональным бэкендом через `TDAD_BACKEND=neo4j` и требует Docker [репозиторий TDAD](https://github.com/pepealonso95/TDAD) [README tdad-skill](https://github.com/pepealonso95/tdad-skill).

Опубликованный `tdad-skill` показывает, что skill для промышленного использования уже шире «`SKILL.md` на 20 строк» на уровне статьи. Его README предлагает установку через Claude Code marketplace (`/skill install pepealonso95/tdad-skill`), `npx skills add pepealonso95/tdad-skill` или ручное копирование в `.claude/skills/tdad/SKILL.md`, а сам `SKILL.md` на GitHub имеет метаданные, совместимость, одноразовую настройку, средства запуска для нескольких языков, правило обязательного нового регрессионного теста и резервную процедуру при отсутствии записи в `.tdad/test_map.txt` [README tdad-skill](https://github.com/pepealonso95/tdad-skill) [SKILL.md tdad-skill](https://github.com/pepealonso95/tdad-skill/blob/main/SKILL.md). Для будущей теории это важная развилка: статья фиксирует исследовательский минимализм инструкции времени выполнения, а опубликованный skill превращает его в переносимый рабочий пакет.

HTML статьи второй линии уточняет, что интеграция времени выполнения намеренно сведена к двум статическим артефактам: `test_map.txt` и короткий `SKILL.md`; во время выполнения агенту нужны только `grep` и `pytest`, без графовой базы данных, MCP-сервера или API-вызовов [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2). Это важный проектный выбор: сложность вынесена в стадию индексации и анализа влияния, а агент получает маленький переносимый контекст, который помещается в ограниченное окно.

### Артефакты

Для этой линии центральны другие артефакты:

- `dependency map` или граф связи кода и тестов между исходными файлами и тестами;
- `.tdad/test_map.txt` — статическая текстовая карта, которую skill заставляет агента читать после изменения кода;
- агентский skill / статический текстовый файл, через который агент получает карту, включая `SKILL.md` как переносимый формат инструкции для Claude Code и других агентов;
- выбранные затронутые тесты;
- узлы графа `File`, `Function`, `Class`, `Test`;
- рёбра графа `CONTAINS`, `CALLS`, `IMPORTS`, `INHERITS`, `TESTS`;
- ранжированный список тестов, который выдаёт анализатор влияния;
- патч, который исправляет задачу;
- вывод тестов как свидетельство отсутствия регрессии на выбранном наборе;
- запись бенчмарка/оценки по SWE-bench Verified и выбранным моделям [статья Test-Driven Agentic Development](https://arxiv.org/abs/2603.17973) [репозиторий TDAD](https://github.com/pepealonso95/TDAD).
- `EXPERIMENTS.md`, `benchmark_runs/`, `predictions/`, `evaluation_results/` и `claudecode_n_codex_swebench/` — слой воспроизводимости второй линии на уровне репозитория: README прямо называет `EXPERIMENTS.md` подробным журналом экспериментов `EXP-001`-`EXP-028`, а также указывает `run_benchmark.py`, `evaluate_predictions.py`, сгенерированные патчи и артефакты отдельных запусков [репозиторий TDAD](https://github.com/pepealonso95/TDAD).

Здесь артефактом является не промпт агента, а карта тестового влияния и следы проверок вокруг конкретного изменения. Это роднит линию с инженерией проверочного контура, наблюдаемостью и тестовой маршрутизацией.

Проход 06 уточняет, что воспроизводимость второй линии на уровне репозитория устроена как несколько поколений проверочного контура, а не как один неизменный каталог результатов. `claudecode_n_codex_swebench/run_benchmark.py` создаёт `benchmark_runs/{timestamp}_{run_name}` с подкаталогами `predictions/` и `evaluations/`, пишет `progress.log`, а в структуре результата хранит не только `resolved`, но и признаки вроде `patch_gate_valid`, `graph_guard_passed`, `indexed_search_success`, `impacted_selected_count`, `tdd_evidence_complete`, `f2p_runtime_strategy`, `p2p_runtime_unreliable_reason` и ссылки на файлы `prediction`/`eval` [run_benchmark.py TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/claudecode_n_codex_swebench/run_benchmark.py). Это важно для будущей теории: TDAD второй линии проверяется не одной строкой «тесты прошли», а набором телеметрических полей о том, откуда пришёл тестовый сигнал, насколько он был надёжен и какой слой графа реально использовался.

Начиная с `EXP-024`, в журнале появляется другой стек оценки: `tdad/eval/` с `run.py`, `instance.py`, `opencode_interface.py`, `dataset.py`, `evaluate.py`, `neo4j_lifecycle.py`, `mlx_proxy.py` и шаблоном `prompts/swe_bench.txt`; артефакты запусков уже лежат в `tdad/eval/runs/...`, например `tdad/eval/runs/20260310_223213_20260310_223213/` [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). Поэтому ссылки на `benchmark_runs/`, `predictions/` и `evaluation_results/` нельзя читать как полный список всех артефактов. Вторая линия эволюционирует от старого средства запуска для Claude Code/Qwen-mini к пайплайну opencode + локальный MLX, и происхождение результата зависит от поколения проверочного контура.

Проход 09 добавляет к этому происхождению на уровне репозитория слой истории коммитов. Публичная страница GitHub показывает у `pepealonso95/TDAD` 20 коммитов, а история фиксирует несколько полезных якорей коммитов: `d75152f` для `[EXP-001] Add experiment tracking system` от 28 октября 2025, `56e8f43` для `run on swe verified` от 29 октября 2025, `8cdbc92` для `graph process` от 15 января 2026, `18d1e04` для baseline с 30% resolution от 20 февраля 2026, `56d450b` для `All versions run` от 11 марта 2026, `1264c6d` для `[EXP-028] Add NetworkX backend, drop Neo4j Docker requirement` от 14 марта 2026 и `73d1234` для update thesis paper / TDAD skill / experiments log от 18 марта 2026 [commit history TDAD](https://github.com/pepealonso95/TDAD/commits/main/). Это не заменяет `EXPERIMENTS.md`, но помогает не смешивать этапы метода: бэкенд NetworkX, опубликованный skill и обновлённая упаковка thesis paper попадают в репозиторий поздно, после ранних SWE-bench запусков и после нескольких поколений проверочного контура.

В README `tdad-skill` архитектура разложена на пять компонентов: `Language Plugins`, `Graph Builder`, `Test Linker`, `Impact Analyzer`, `Test Runner` [README tdad-skill](https://github.com/pepealonso95/tdad-skill). `Graph Builder` наполняет граф узлами и рёбрами и поддерживает полную/инкрементальную индексацию через хеширование содержимого; `Test Linker` связывает тесты с кодом через соглашения об именовании, статический анализ импортов/вызовов и опциональные данные покрытия; `Impact Analyzer` ранжирует затронутые тесты; `Test Runner` делегирует запуск языковому плагину [README tdad-skill](https://github.com/pepealonso95/tdad-skill).

Полная статья описывает Python-пайплайн индексации конкретнее. `AST Parser` извлекает определения функций с сигнатурами, диапазонами строк и docstrings, определения классов с базовыми классами и методами, `import`-инструкции и цели вызовов; тестовые файлы находятся по `test_*.py` и `*_test.py`, тестовые функции по `test_*`, тестовые классы по `Test*` [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2). `Test Linker` строит рёбра `TESTS` тремя стратегиями в порядке приоритета: соглашения об именовании (`test_foo.py` -> `foo.py`), сопоставление по префиксу с прогрессивным усечением основы имени, близость директорий при неоднозначных совпадениях; для монолитных `tests.py` используется алгоритм близости к ближайшему нетестовому предку [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2).

Анализ влияния запускает четыре стратегии параллельно и сохраняет максимальную оценку, если тест найден несколькими путями. Профиль `balanced` использует базовые веса `Direct 0.95`, `Coverage 0.80`, `Transitive 0.70`, `Imports 0.50`; компонент уверенности отражает силу связи: `1.0` для прямого `TESTS`, `0.56` для транзитивных цепочек вызовов, `0.5` для покрытия и `0.45` для импортов [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2). Тесты выбираются ступенчатым списком с настраиваемым максимумом, по умолчанию до 50; профили `conservative`, `balanced`, `aggressive` позволяют менять компромисс точности и полноты без переписывания формулы [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2).

Поддержка языков тоже является частью фактуры, а не украшением: Python использует встроенный `ast`, а JavaScript/TypeScript, Go, Java, Rust и Dart опираются на дополнительные зависимости tree-sitter и соответствующие средства запуска вроде Jest/Vitest/Mocha, `go test`, Maven/Gradle, `cargo test`, `dart test`/`flutter test` [README tdad-skill](https://github.com/pepealonso95/tdad-skill). Это означает, что перенос TDAD в реальный проект зависит от качества языкового плагина и обнаружения тестов, а не только от общей идеи графа связи кода и тестов.

### Проверка результата

Источник заявляет оценку на SWE-bench Verified и моделях с открытыми весами на потребительском железе: Qwen3-Coder 30B на 100 экземплярах задач и Qwen3.5-35B-A3B на 25 экземплярах задач [статья Test-Driven Agentic Development](https://arxiv.org/abs/2603.17973). Заявленный результат — снижение `regression rate` на 70%, с 6.08% до 1.82% относительно `vanilla baseline` [статья Test-Driven Agentic Development](https://arxiv.org/abs/2603.17973).

Полная статья показывает, почему заголовочное `70%` нельзя читать отдельно от знаменателей. В Phase 1 vanilla дал 562 сбоя P2P-тестов (`P2P failures`) из 9,245 тестов, TDD Prompt — 799 из 8,040, GraphRAG+TDD — 155 из 8,536; катастрофические регрессии (`catastrophic regressions`) снизились с 3 у vanilla и 5 у TDD Prompt до 1 у GraphRAG+TDD [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2). При этом `resolution rate` у GraphRAG+TDD был 29% против 31% у vanilla, а `generation rate` 74% против 86%; автор интерпретирует это как консервативное смещение и более высокую долю пустых патчей, когда карта показывала высокий риск регрессии [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2).

Эти числа нельзя использовать как общий аргумент «TDAD снижает регрессии» без уточнения. Нужно сохранить условия: какие экземпляры задач, какие модели, как строилась карта, какие тесты считались затронутыми, как определялась регрессия и что было базовым вариантом. Репозиторий `pepealonso95/TDAD` нужен следующему проходу именно для этого: проверить формат карты, команды запуска, структуру `tdad-skill` и воспроизводимость результата [репозиторий TDAD](https://github.com/pepealonso95/TDAD) [репозиторий tdad-skill](https://github.com/pepealonso95/tdad-skill).

README `tdad-skill` добавляет определения метрик, которые нужно сохранить рядом с числами: `Resolution Rate` — доля экземпляров задач, где патч исправляет целевую задачу и проходит FAIL_TO_PASS, не ломая PASS_TO_PASS; `Test-Level Regression Rate` — `sum(PASS_TO_PASS failures) / sum(total PASS_TO_PASS tests) * 100`; `Generation Rate` — доля экземпляров задач, где агент произвёл непустой патч [README tdad-skill](https://github.com/pepealonso95/tdad-skill). Там же указано, что GraphRAG + TDD имел 29% resolution против 31% vanilla и больше пустых патчей: 26 против 14; источник объясняет компромисс не худшим качеством патча, а большей вероятностью не сгенерировать патч [README tdad-skill](https://github.com/pepealonso95/tdad-skill).

Анализ влияния в `tdad-skill` ранжирует тесты по стратегиям с весами: `Direct` 0.95, `Coverage` 0.80, `Transitive` 0.70, `Imports` 0.50 [README tdad-skill](https://github.com/pepealonso95/tdad-skill). Это нужно проверить в коде перед финальным использованием, но уже сейчас видно, что «релевантный тест» не является бинарным фактом: это результат эвристической оценки, которая может ошибиться на динамических вызовах, плохо названных тестах, неполном покрытии или нестандартной архитектуре.

Phase 2 важна как другой режим пользы: там TDAD Skill с Qwen3.5-35B-A3B + OpenCode повысил `resolved` с 6/25 до 8/25 и количество сгенерированных патчей с 10/25 до 17/25, при 0% регрессии в обеих группах [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2). Автор объясняет это не снижением регрессий, а улучшением навигации/генерации: граф связи кода и тестов дал агенту структурный контекст, который помог вообще создать патч [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2).

Цикл автоулучшения второй линии тоже стоит сохранить как отдельную фактуру. Агент Claude Code получал историю экспериментов и делал одно сфокусированное изменение в `SKILL.md`, `impact.py`, `ast_parser.py` и других файлах; модульные тесты были приёмочной контрольной точкой, оценка бенчмарка сравнивалась с лучшими известными значениями, регрессии вызывали откат, а скрипт оценки проверялся по SHA-256 и был доступен только для чтения [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2). Из 15 итераций приняли 4 изменения: упрощение `SKILL.md` с 107 до 20 строк, статический экспорт `test_map.txt`, скоринг по близости директорий и связи на основе импортов; 11 итераций были откатаны, а resolution на подмножестве из 10 экземпляров задач вырос с 12% до 60% при 0% регрессии [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2).

`EXPERIMENTS.md` подтверждает и расширяет эту картину. В `EXP-027` цикл описан как 15 итераций за примерно 36 часов: `python -m autoloop.loop --eval-mode full --max-iterations 15 -v`; перед каждой итерацией проверялся SHA-256 для `evaluate_fixed.py`, затем создавался снимок исходных файлов TDAD, Claude Opus 4.6 делал одно изменение, `pytest tests/` служил локальной контрольной точкой, полная оценка запускалась на 10 экземплярах задач, а результат сравнивался с лучшим значением для решения сохранить/откатить (`keep/revert`) [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). В `EXP-028` уже после перехода на бэкенд NetworkX оценка на тех же 25 экземплярах задач, что `EXP-025`, дала 8/25 `resolved` против 6/25 базового варианта и 17/25 сгенерированных патчей против 10/25; сам источник подчёркивает, что улучшение пришло от генерации большего числа патчей, а не от резкого роста качества среди сгенерированных патчей (62% против 60%) [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). Для досье это уточняет механизм пользы: TDAD Skill может помогать агенту начать и завершить патч, а не только ловить регрессии.

Перед этим улучшением в `EXP-026` был важный отрицательный результат, который нельзя потерять. На тех же 25 экземплярах SWE-bench Verified, что `EXP-025`, режим TDAD с opencode, Qwen3.5-35B-A3B и skill на основе Neo4j дал 7/25 сгенерированных патчей и 3/25 resolved против базового варианта: 10/25 сгенерированных патчей и 6/25 resolved; `regression rate` остался 0% в обеих группах [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). Журнал называет несколько возможных причин: индексирование съедало время, skill мог не вызываться, opencode мог не обнаруживать `.opencode/skills/tdad/SKILL.md`, а сама процедура требовала от модели понять skill, вызвать `tdad impact`, интерпретировать результат и запустить тесты [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). Это важный случай сбоя метода: добавление графового контекста как skill времени выполнения может ухудшить генерацию патча, если контекст плохо упакован или слишком дорог для модели и среды.

`EXP-028` показывает, какая именно траектория исправления вывела метод из этого провала. Neo4j заменён на бэкенд NetworkX с сохранением состояния в `.tdad/graph.pkl`; `networkx>=3.0` стал основной зависимостью, а Neo4j ушёл в опциональный `pip install tdad[neo4j]` [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). Smoke-проверка на самом репозитории TDAD дала 66 файлов, 346 функций, 22 класса, 46 тестов, 1292 ребра, 1264 связанных теста и 17 экспортированных исходных файлов в тестовой карте; на запуске SWE-bench по тем же 25 экземплярам задач индексация NetworkX прошла без проблем Docker/Neo4j, примерно за 30 секунд на экземпляр задачи, а выигрыш пришёл от меньшего числа пустых патчей: 8 вместо 15 [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). Для будущей теории это показывает не абстрактную пользу «графа», а инженерную развилку: графовый сервис времени выполнения оказался слишком тяжёлым, а статический `test_map.txt` + лёгкий бэкенд лучше совпали с агентским рабочим процессом.

Журнал экспериментов даёт отдельный слой фактуры про надёжность самого оценочного сигнала. В `EXP-021j` авторы обнаружили, что запуски могли получить `resolved=0` через устаревший глобальный eval JSON, если Docker/evaluator падал, но вызывающая сторона переиспользовала исторический файл; исправление заставило `evaluate_predictions.py` падать громко, а `run_benchmark.py` принимать только свежий eval JSON, созданный текущим вызовом [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). В `EXP-021k` это проверено на запуске одного экземпляра задачи: основные артефакты перечислены как `benchmark_runs/.../progress.log`, `report.json`, `evaluations/graphrag_tdd.eval.json`, а выбранная свежая оценка получила временную метку того же запуска [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). `EXP-021l` усилил это до точной атрибуции: evaluator печатает `EVAL_JSON_PATH:`, а средство запуска отказывается от результата со статусом `no_exact_result`, если точный путь не найден [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). Это неудобная, но важная деталь метода: регрессионная защита требует не только карты тестов, но и строгого происхождения eval-файла.

Целевой аудит прохода 09 не нашёл публичного слоя обсуждений, который мог бы закрыть эту тему через issues или pull requests: текущая GitHub-страница `pepealonso95/TDAD` показывает `Issues 0` и `Pull requests 0`, а также каталоги верхнего уровня `benchmark_runs`, `claudecode_n_codex_swebench` и `tdad` рядом с `EXPERIMENTS.md` [репозиторий TDAD](https://github.com/pepealonso95/TDAD). Поэтому вопрос о происхождении второй линии теперь лучше формулировать не как поиск обсуждений в issue/PR, а как проверку содержимого закоммиченных каталогов артефактов и конкретных путей из `EXPERIMENTS.md`. Если точные `report.json`, `progress.log` или `evaluations/*.eval.json` не открываются как raw-файлы, их нельзя использовать как подтверждение телеметрии по отдельным экземплярам задач: в досье остаётся только то, что `EXPERIMENTS.md` называет эти пути и объясняет причину более строгой атрибуции оценки.

Переход к opencode + локальному MLX в `EXP-024` добавляет ещё один режим сбоя, который относится к агентскому рабочему процессу, а не к идее графа связи кода и тестов. Для работы с `mlx_vlm.server` понадобился прокси `mlx_proxy.py`, потому что сервер не отдавал `index` в `tool_calls` и ставил `finish_reason: "stop"` даже при вызовах инструментов; opencode из-за этого считал, что модель закончила работу после 1-3 вызовов инструментов и не переходил к многошаговому редактированию [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). Там же зафиксировано практическое узкое место: при контексте около 36k этап prefill занимал примерно 54 секунды на раунд, а один экземпляр задачи требовал 10+ минут, поэтому тайм-аут 600s становился частью результата [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). Для будущего текста это полезно как предупреждение: у TDAD второй линии качество может ограничиваться особенностями API, совместимого с OpenAI, локальной скоростью инференса и тайм-аутами не меньше, чем точностью карты зависимостей.

### Где человек принимает решение

Во второй линии человек решает, можно ли доверять карте влияния и достаточно ли выбранного набора тестов. Если граф неполный, устаревший или построен только по части проекта, агент получит убедительный, но ложный маршрут проверки. Поэтому человеческое решение остаётся вокруг качества карты, границы запуска, риска выпуска и случаев, где нужно расширить тестовый набор вручную.

`SKILL.md` добавляет ещё одну человеческую контрольную точку: каждое изменение должно включать хотя бы один новый тест, а если `.tdad/test_map.txt` не содержит запись, агент должен использовать резервную процедуру по соглашению об именовании или искать imports через `grep -r 'changed_module' tests/` [SKILL.md tdad-skill](https://github.com/pepealonso95/tdad-skill/blob/main/SKILL.md). Человек должен решить, когда такой резервной процедуры достаточно, а когда надо перестроить индекс, расширить карту или запускать полный набор тестов.

Статья второй линии явно ограничивает применимость: эксперименты шли на 100 + 25 экземплярах задач, без формальных тестов значимости; обе настройки моделей были меньшими локальными моделями; реализация в статье только для Python; статический анализ не видит динамическую диспетчеризацию, monkey-patching и код, сгенерированный во время выполнения; auto-loop проверялся на 10 экземплярах задач на итерацию [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2). Для человеческого решения это означает, что карту TDAD нельзя считать оракулом: владелец проекта должен знать, где статический граф заведомо слаб.

## Сравнительная развилка внутри досье

Это досье допускает локальное сравнение, потому что без него аббревиатура `TDAD` становится опасной.

| Вопрос | Test-Driven AI Agent Definition | Test-Driven Agentic Development |
| --- | --- | --- |
| Что является объектом управления | Поведение агента с инструментами | Изменение кода, которое делает кодинговый агент |
| Главный артефакт | Поведенческая спецификация, тесты, скомпилированный промпт/`agent definition` | Карта зависимости кода и тестов, затронутые тесты, свидетельства по патчу |
| Где тесты включены в процесс | Тесты компилируют и проверяют промпт | Тесты направляют выбор проверок и самокоррекцию при патче |
| Что выпускается | Проверенный промпт/агентский артефакт | Кодовое изменение со свидетельствами по релевантным тестам |
| Основной риск | Подгонка промпта под видимые тесты, неполная спецификация, скрытые разрывы политики, исключение неактивированных мутантов | Устаревшая или неполная карта, неверный выбор тестов, ложная уверенность в покрытии, слепые зоны статического анализа |
| Ближайшая роль в SDLC с ИИ-агентами | Спецификация поведения агента и проверяемая сборка `agent definition` | Исполнение, проверка и регрессионная защита для агентского изменения кода |
| Неудобная деталь источника | Полный пайплайн завязан на тома Docker; GitHub-репозиторий, `results/` и набор данных HuggingFace разделяют разные слои SpecSuite-Core | TDD только через промпт в эксперименте ухудшил регрессии; skill на 20 строк на уровне статьи, опубликованный `tdad-skill` и `EXPERIMENTS.md` описывают разные стадии метода |

Для будущей теории сильная формулировка такая: подходы, управляемые тестами, в агентской разработке могут управлять разными объектами. Иногда тесты компилируют поведение агента. Иногда тесты ограничивают кодовое изменение. В обоих случаях тесты становятся внешним контуром реальности, но артефакты, риски и человеческие решения разные.

Табличная строка про неудобную деталь первой линии нуждается в более точной формулировке после прохода 04: текущий README уже описывает `SupportOps` как рабочий пример, а остальные спецификации как спецификация + `mutation intents`, причём specs/mutations/results также вынесены в набор данных HuggingFace [репозиторий tdad-paper-code](https://github.com/f-labs-io/tdad-paper-code). Поэтому риск здесь не «репозиторий противоречит статье», а «разные слои артефактов живут в разных местах»: GitHub-репозиторий, тома Docker, `results/` и набор данных. Для второй линии похожая оговорка: `20-line SKILL.md` на уровне статьи, опубликованный `tdad-skill` и `EXPERIMENTS.md` описывают разные стадии одного метода, и их нельзя механически склеивать [SKILL.md tdad-skill](https://github.com/pepealonso95/tdad-skill/blob/main/SKILL.md) [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md).

## Связанные источники и соседние линии

`TDFlow` оставлен как соседний источник, а не как факт о двух основных TDAD. По аннотации arXiv это агентский рабочий процесс, управляемый тестами, для разработки на уровне репозитория, где написанные человеком тесты задают задачу, а подагенты разделяют предложение патча, отладку, пересмотр патча и опциональную генерацию тестов [TDFlow](https://arxiv.org/abs/2510.23761). Его нельзя использовать как подтверждение деталей TDAD, но он помогает увидеть соседний полюс: тесты могут быть не картой влияния и не компилятором промпта, а задачей, вокруг которой организован многоагентный ремонт. Для будущих проходов нужен полный PDF/HTML, потому что аннотация заявляет `pass rate` 88.8% на SWE-Bench Lite, 94.3% на SWE-Bench Verified и только 7 случаев подыгрывания тестам (`test hacking`) после ручной проверки 800 запусков [TDFlow](https://arxiv.org/abs/2510.23761).

Проход 07 прочитал HTML полной статьи `TDFlow` и уточняет статус этого соседнего источника. `TDFlow` начинается с написанных человеком воспроизводящих тестов (`reproduction tests`) или, опционально, с генерации таких тестов; затем тесты запускаются, `Explore Files` предлагает патч на уровне репозитория только с правами чтения файлов, поиска и просмотра дерева, `Revise Patch` чинит плохо применимый патч без изменения вставляемого кода, `Debug One` разбирает один падающий тест через ограниченный отладчик и отдаёт отчёт, а агрегированные отчёты отладки возвращаются в следующий цикл `Explore Files` [HTML статьи TDFlow](https://ar5iv.org/html/2510.23761v2). Эта линия полезна для будущей теории как контрастный случай: тесты здесь являются не картой зависимостей и не скрытой оценкой, а явным входом задачи для рабочего процесса. При этом источник сам задаёт жёсткую границу применимости: при неверных воспроизводящих тестах у процесса нет механизма исправить их позже, а жёсткий многоагентный рабочий процесс требует более сложной среды и не имеет ранней остановки, если тесты нерешаемы [HTML статьи TDFlow](https://ar5iv.org/html/2510.23761v2).

Для TDAD-досье особенно важны две детали `TDFlow`, потому что они показывают риск слишком общего слова `test-driven`. Во-первых, `TDFlow` намеренно даёт системе обычно скрытые написанные человеком тесты SWE-Bench и модифицирует базовые системы для честного сравнения; эти числа нельзя переносить на постановку, где агент видит только описание задачи [HTML статьи TDFlow](https://ar5iv.org/html/2510.23761v2). Во-вторых, авторы считают подыгрывание тестам (`test hacking`) отдельным режимом сбоя: вручную проверяют 300 запусков SWE-Bench Lite и 500 запусков SWE-Bench Verified, отмечают прямое изменение тестов, пропуск/отключение проверок (`skip`/`disable`), ослабление проверочных утверждений, логику только для тестов (`test-only logic`), жёстко заданные ответы (`hardcoded outputs`) и другие признаки, а найденные случаи засчитывают как сбои (`failures`) [HTML статьи TDFlow](https://ar5iv.org/html/2510.23761v2). Это не факт о TDAD, но полезная будущая проверочная линза для любых агентских рабочих процессов, управляемых тестами.

Страница AI Paper Digest по `2603.17973` остаётся вторичным навигационным источником, а не равным первоисточником [AI Paper Digest TDAD](https://ai-paper-delta.vercel.app/en/papers/2603.17973). Её можно использовать для поиска связанных ссылок, но не для переноса фактов, если те не проверены по статье или репозиторию.

## Режимы сбоя / риски чрезмерного применения

- Смешать две линии TDAD в один метод и получить ложный вывод: будто компиляция промпта и анализ регрессионного влияния являются частями одного инструмента.
- Назвать обе линии TDD в привычном смысле и потерять специфику. Первая линия ближе к компилятору промптов, управляемому оценкой; вторая — к графовому выбору тестов для агентского исправления кода.
- Принять проход тестов за гарантию качества. В первой линии `hidden tests` защищают от части подгонки, но не покрывают всю политику поведения. Во второй линии целевые тесты полезны только настолько, насколько хороша карта влияния.
- Не заметить, что первая линия сама вводит инфраструктуру защиты от подыгрывания тестам. Если `hidden tests` доступны PromptSmith, `visible tests` можно переписывать из контейнера компиляции или MutationSmith работает в той же сессии, метод теряет ключевую защиту [HTML статьи Test-Driven AI Agent Definition](https://arxiv.org/html/2603.08806v1).
- Переиспользовать заголовочные метрики без условий эксперимента. Числа из статей нужно хранить рядом с моделью, набором задач, проверочным контуром и базовым вариантом.
- Автоматизировать слишком рано. Обе линии требуют хорошей спецификации или хорошей карты. Если исходный артефакт слабый, агент получит формальную процедуру с плохим сигналом.
- Перепутать процедурную инструкцию с рабочим контекстом. Вторая линия показывает, что TDD-инструкции сами по себе могут ухудшить `regression rate`, если агент не получил конкретную карту связи кода и тестов.
- Перенести вывод второй линии на передовые модели без проверки. Авторы сами ограничивают вывод меньшими локальными моделями и допускают, что при больших контекстных окнах TDD Prompting Paradox может проявляться иначе [HTML статьи Test-Driven Agentic Development](https://arxiv.org/html/2603.17973v2).
- Считать README репозитория равным полной статье. В первой линии кодовый репозиторий сейчас явно показывает не все артефакты на уровне статьи; во второй линии нужно отделять промышленный CLI, `tdad-skill` и экспериментальную оценку на SWE-bench.
- Считать опубликованный skill тем же самым артефактом, который оценивался в статье. Skill времени выполнения на уровне статьи был намеренно коротким и работал вместе со статическим `test_map.txt`; опубликованный `tdad-skill` уже включает метаданные, средства запуска для нескольких языков, обязательный новый тест и резервные процедуры [SKILL.md tdad-skill](https://github.com/pepealonso95/tdad-skill/blob/main/SKILL.md). Это полезно для практики, но при переносе чисел из статьи нужно указывать, какой вариант skill использовался.
- Не увидеть отрицательный `EXP-026` во второй линии. Если дать агенту skill на основе Neo4j как дополнительную процедуру, но не убедиться, что skill обнаружен, вызван и помещается в бюджет времени, результат может стать хуже базового варианта: меньше сгенерированных патчей и меньше экземпляров задач со статусом `resolved` при той же нулевой `regression rate` [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md).
- Использовать `ALWAYS_CREATE_TICKET`/`M_ALWAYS_TICKET` как простую иллюстрацию убитого/выжившего мутанта без версии. В текущих доступных слоях есть конфликт между сырым README, отрендеренной страницей GitHub и именованием в наборе данных; безопасно говорить только о самом `mutation intent`, а не о стабильном проценте, пока не проверены хэш коммита и JSON отдельных запусков [raw README tdad-paper-code](https://raw.githubusercontent.com/f-labs-io/tdad-paper-code/main/README.md) [HuggingFace SpecSuite-Core](https://huggingface.co/datasets/f-labs-io/SpecSuite-Core).
- Читать eval-файлы второй линии без происхождения. `EXP-021j`/`EXP-021l` показывают, что устаревший `evaluation_results/*.json` мог исказить статус `resolved`, поэтому для серьёзного вывода нужны точный путь eval-файла, свежая временная метка и связь с конкретным `benchmark_runs/.../report.json` [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md).
- Считать локальный сервер модели с API, совместимым с OpenAI, прозрачной заменой облачного API. В `EXP-024` opencode потребовал `mlx_proxy.py` из-за несовместимых `tool_calls` и `finish_reason`, а локальная скорость инференса и тайм-аут 600s стали частью наблюдаемых показателей `generation rate` / `resolution rate` [EXPERIMENTS.md TDAD](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md).
- Переносить `TDFlow`-числа в TDAD без смены постановки задачи. `TDFlow` в ключевом режиме получает написанные человеком воспроизводящие тесты (`reproduction tests`), то есть работает с другим контрактом: человек задаёт проверяемое поведение тестами, а рабочий процесс решает эти тесты. Это нельзя склеивать с TDAD-анализом влияния между кодом и тестами, где карта выбирает затронутые тесты для уже заданной задачи исправления [HTML статьи TDFlow](https://ar5iv.org/html/2510.23761v2).
- Считать рабочий процесс, управляемый тестами, защищённым от подыгрывания тестам (`test hacking`) только потому, что тесты запущены. `TDFlow` прямо вводит запреты на изменение тестовых каталогов, ограничение видимости файловой системы, ручную проверку патчей/логов и рубрику признаков подыгрывания тестам (`test hacking`); значит, контур, управляемый тестами, требует отдельной политики против подыгрывания проверкам, а не только команды запуска тестов [HTML статьи TDFlow](https://ar5iv.org/html/2510.23761v2).
- Продолжать широкое накопление источников после того, как основные статьи, репозитории, HTML-версии, слой набора данных и экспериментальный журнал уже прочитаны. После прохода 08 более полезная работа выглядит как целевой аудит происхождения: открыть конкретные `report.json`, `progress.log`, `evaluations/*.json`, обсуждения issue/PR и историю коммитов для спорных мест, а не искать ещё один общий обзор TDAD. Иначе досье начнёт накапливать навигационный шум вместо новой фактуры.

Проход по открытию источников добавил более жёсткое ограничение по метрикам графового TDAD. В Phase 1 SWE-bench Verified на 100 экземплярах задач TDAD / GraphRAG+TDD снизил `test-level regression rate` с 6.08% до 1.82%, но `resolution rate` немного снизился, а `generation rate` был ниже; авторы отмечают эффекты знаменателя в `instance-level regression rates` [TDAD arXiv HTML](https://arxiv.org/html/2603.17973v2). Поэтому заголовочную формулу про снижение регрессий на 70% нельзя читать как бесплатное улучшение: метод может заставить агента чаще воздерживаться, когда риск видим.

Журнал экспериментов в более старом TDAD-репозитории также полезен как предупреждающий материал: одна попытка агентского цикла Qwen упёрлась в ошибки 500 примерно на контексте 28K, скользящее окно отбрасывало слишком много контекста, а модель постоянно перечитывала файлы, прежде чем автор переписал подход в одиночный API-вызов [TDAD EXPERIMENTS.md](https://raw.githubusercontent.com/pepealonso95/TDAD/main/EXPERIMENTS.md). Это не основной результат статьи, но хороший источник для нашей темы: агентские тестовые рабочие процессы могут ломаться из-за управления контекстом и хрупкости цикла, а не только из-за плохих тестов.

## Аналитические заметки предыдущих проходов

### Что должно попасть в теорию

- Различение двух объектов управления через тесты: поведение агента и кодовое изменение.
- Идея промпта как компилируемого артефакта: `agent definition` можно собирать из поведенческой спецификации под тесты, а не только писать вручную.
- Идея графа связи кода и тестов как помощи агенту во время выполнения: не «запусти всё», а выбери проверки по карте влияния.
- Предупреждение о ложной уверенности: тестовый контур повышает наблюдаемость, но качество зависит от спецификации, карты, скрытого набора, проверочного контура и человеческого решения о выпуске.
- Различение тестов как компилятора поведения, тестов как маршрутизатора проверки и тестов как задачи для многоагентного ремонта. После чтения полной статьи `TDFlow` можно использовать как соседний контрастный случай: рабочий процесс не строит граф связи кода и тестов и не компилирует промпт, а принимает воспроизводящие тесты (`reproduction tests`) как главный вход, разносит предложение патча, пересмотр патча и отладку отдельных тестов по подагентам и считает некачественную генерацию тестов главным пределом автономности [HTML статьи TDFlow](https://ar5iv.org/html/2510.23761v2).
- Тезис «контекст важнее процедуры» из второй линии: для кодинговых агентов полезнее дать короткий структурный контекст о том, какие тесты под риском, чем подробно описывать процедуру TDD; но это утверждение нужно привязывать к меньшим локальным моделям и условиям эксперимента.
- Тезис первой линии о разделении доступа к артефактам: `hidden tests`, `visible tests`, компилятор и оценщик должны быть разделены инфраструктурно, иначе управляемая тестами компиляция промпта легко превращается в оптимизацию под раскрытые тесты.

### Что лучше уйдёт в Handbook / Fieldbook

- Практический чек для агентского промпта: есть ли поведенческая спецификация, `visible tests`, `hidden tests`, mutation/evolution checks и решение о выпуске.
- Практический чек для кодингового агента: есть ли карта связей исходных файлов и тестов, как она обновляется, какие затронутые тесты выбраны, где сохранён вывод тестов, кто расширяет набор при сомнении.
- Практический чек по `.tdad/test_map.txt`: индекс создан, изменённые файлы найдены через `grep`, резервная процедура зафиксирована явно, каждый патч добавляет хотя бы один тест, а отсутствие записи в карте не считается молчаливым разрешением пропустить проверку.
- Fieldbook-паттерн: если агентский рабочий процесс даёт «успешный патч» без связанного тестового подтверждения, результат нельзя считать завершённым.
- Соседний Fieldbook-паттерн из `TDFlow`: если человек может написать точные воспроизводящие тесты (`reproduction tests`), агентский рабочий процесс можно сузить до цикла решения тестов, но тогда нужны отдельные запреты на подыгрывание тестам (`test hacking`), явный контроль качества самих воспроизводящих тестов и остановка, если тесты неверны или нерешаемы [HTML статьи TDFlow](https://ar5iv.org/html/2510.23761v2).


## Краткая рабочая карта

TDAD в этом досье остаётся сравнительным случаем двух разных линий, которые нельзя сливать в один метод. Первая линия — Test-Driven AI Agent Definition / SpecSuite-Core: спецификация, сгенерированные тесты, `hidden tests`, мутации и компиляция промпта. Вторая линия — Test-Driven Agentic Development: графовый анализ влияния для кодинговых ИИ-агентов, где агент после изменения находит затронутые тесты через `.tdad/test_map.txt` и запускает их перед сдачей патча. Проход по открытию источников усиливает вторую линию: её ядро не TDD-инструкции, а конкретная карта связи исходных файлов и тестов.


Уточнение прохода по открытию источников для графового TDAD: статья прямо формулирует ключевой тезис — агентам не нужно рассказывать, как делать TDD; им нужно знать, какие тесты проверять [TDAD arXiv HTML](https://arxiv.org/html/2603.17973v2). Пайплайн имеет две стадии: индексация строит граф связи кода и тестов, а экспорт создаёт статический `test_map.txt`; во время выполнения агенту нужны только `grep` и `pytest`, без графовой базы данных, MCP-сервера или API-вызовов. Это важно для теории: TDAD намеренно избегает добавления тяжёлого оркестратора времени выполнения; дорогая работа с графом вынесена до агентского цикла кодирования.

Схема графа включает четыре типа узлов — File, Function, Class, Test — и пять типов рёбер: CONTAINS, CALLS, IMPORTS, TESTS, INHERITS [TDAD arXiv HTML](https://arxiv.org/html/2603.17973v2). Пайплайн индексации строится из AST Parser, Graph Builder и Test Linker. Test Linker использует соглашения об именовании, сопоставление по префиксу и близость директорий, чтобы связать тесты с исходным кодом. Это даёт методологии конкретную поверхность сбоя: если эвристики связывания плохи для проекта, карта затронутых тестов даст агенту ложную цель проверки.
