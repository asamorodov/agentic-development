# Template: план целевой группы для пакета-задания

Этот файл задаёт форму build-time плана целевой группы. По такому плану затем собирается исполнительный пакет. План должен быть достаточно точным для сборки пакета, но не должен превращаться в методологический трактат.

Не бюрократизируйте. Не добавляйте общие рассуждения, длинные объяснения и лишние контрольные списки. Но все пути файлов и тексты prompt-ов должны быть указаны точно.


## Конвенция имён

Файл плана хранится в `work/theory-writing/target-group-plans/` и заканчивается на `_TARGET_GROUP_PLAN.md`.

Если целевая группа создаёт или меняет один документ, имя плана начинается с имени этого документа без расширения. Если целевых документов несколько, имя начинается с краткого имени группы. Собранный пакет из одного плана хранится в `work/theory-writing/packages/` и называется как план без `_TARGET_GROUP_PLAN`, но с расширением `.zip`.

## 1. Обрабатываемые файлы

Укажите полный список файлов, которые пакет должен создать или изменить.

Для каждого файла укажите:

```yaml
- path: path/to/file.md
  status: existing            # existing | future
  role: primary               # primary | secondary | supporting-output | diagnostic-output
  write_policy: replace-full-file
  result_policy: overlay-path # overlay-path | result-only | diagnostic-only
  notes: "Зачем файл входит в целевую группу."
```

Если файл уже существует, укажите его реальный путь. Если файл ещё не существует, укажите будущий путь и отметьте, что пакет должен создать его.

По умолчанию существующие обрабатываемые файлы будут упакованы внутрь исполнительного архива. Future-файлы будут созданы в пакете с пустым или явно заданным начальным содержимым.

Если целевая группа содержит несколько связанных документов, явно пометьте группу как `linked-target-edit`. Это значит, что один рабочий проход может редактировать несколько целевых файлов вместе и обязан проверять согласованность между ними.

## 2. Файлы для чтения

Укажите полный список внутренних файлов, которые prompt-ы этой целевой группы могут читать. Разделяйте их по смыслу, но без лишней детализации:

- управляющие документы;
- скелетон, планы и карты;
- `work/theory-writing/asset-catalog/*`, если целевая группа может использовать иллюстрации, screenshots, diagrams или figure candidates;
- досье;
- истории и story-dossiers;
- source maps и comparative reports;
- языковые, стилевые, источниковые протоколы и `protocols/rules/visual-assets-and-figures.md`, если есть визуальные материалы;
- иные материалы, если они действительно нужны.

Не включайте документы “на всякий случай”. Каждый файл должен быть нужен именно для этой целевой группы.

По умолчанию все файлы из этой секции будут упакованы внутрь исполнительного архива как read-only источники. Repo-snapshot-bound режим допустим только если он явно указан при постановке задачи.

Внешние источники задайте в двух формах:

1. явно заданные URL или источники, если они уже известны;
2. выводимые источники, которые исполнитель обязан найти из самих материалов: inline-ссылки, source registers, названия авторов, инструментов, статей, репозиториев, методологий, figure-кандидатов.

Внешние источники нужны не для широты, а для проверки и насыщения конкретного текста. Если источник используется в тексте, ссылка должна ставиться сразу там, где вводится факт.

## 3. Очередь рабочих prompt-ов

Опишите очередь prompt-ов, которые будут использованы для работы над целевой группой.

Для каждого prompt-а дайте готовый финальный текст, пригодный для помещения в исполнительный пакет. Не описывайте prompt пересказом.

Каждый prompt в очереди должен начинаться с блока:

```text
Прочитай сначала:
- ...
- ...
```

В этот блок включайте только источники, нужные именно для данного прохода. Языковой, стилевой и источниковый протоколы включайте во все prompt-ы, где есть написание, редактура, перенос фактуры, ссылки или правка текста.

В каждом prompt-е должны быть указаны:

- что прочитать;
- что сделать;
- какие файлы записать;
- какие ограничения соблюдать.

Prompt-ы должны быть короткими и рабочими. Не превращайте их в длинные чек-листы. Формулируйте направления и критерии качества, а не подробную механику каждой фразы.

Обычная очередь для writing-задачи:

```text
первичное написание или основная правка → 2–4 прохода добавления фактуры и деталей → системное выравнивание → 1–3 адресных прохода усиления основной функции при необходимости → 2–3 общих редакторских прохода оценки и repair → языковой проход → стилевой проход → финальная проверка и упаковка
```

Если задача не writing-задача, адаптируйте очередь под её природу. Не навязывайте главам, досье, repair-pass-ам, техническим атласам и тестовым пакетам одну и ту же форму.

Если план относится к technical atlas / concept atlas, не формулируйте атлас как простое место для «технических деталей». Атласный раздел должен быть самостоятельной концептуально-технической статьёй: связно объяснять конкретную концепцию или методологию на основе досье, сохранять источниковую фактуру, иллюстрации, команды, оговорки и ограничения, а также показывать связь с общей теорией. Повтор с теорией допустим, когда он нужен для самостоятельного понимания; механический копипаст и сброс неразобранной сложности в атлас запрещены.

Если план относится к большой статье концептуально-технического атласа, используй отдельный blueprint `work/theory-writing/ATLAS_ARTICLE_PACKAGE_BLUEPRINT.md`. Базовая очередь такой статьи — 25 рабочих проходов плюс финальная проверка: source-depth, два свободных доборных прохода, visual asset passes, concept reinforcement, три общих редакторских прохода, public/article structure, companion sync, два языковых и два стилевых прохода. Не вводи внутренних ограничений объёма. Разрешай внешние первоисточники, особенно для поиска реальных изображений. Релевантные локальные assets вставляй как `<figure><img ...></figure>`; внешние реальные изображения ставь как `external-real-candidate` placeholders и заноси в нижний раздел для asset-pass.

Ссылки на внутренние и внешние источники должны ставиться сразу в основной текст результата. Для визуального материала план обязан вводить asset-classification gate до любой inline-вставки или переписывания. Каждый кандидат должен быть отнесён к одному из классов: `synthetic_figure`, `local_image_asset`, `external_real_image_candidate` или `editorial_visual_idea`.

План не должен формулировать правило так, будто любой figure/image candidate обязан стать текстовой схемой или автоматически попасть в основной текст. Inline-вставка означает размещение **уже классифицированной** выбранной фигуры по месту применения: синтетическая схема вставляется как авторский `<figure>` только после строгой проверки полезности и нетривиальности, локальный готовый asset — как `<figure><img src="...">...</figure>`, внешний реальный screenshot/source diagram без asset-pass остаётся в `*_figure_candidates.md` и общем asset catalog со статусом и причиной. Готовые иллюстрации нельзя пересказывать, перерисовывать или ухудшать ради закрытия пункта «добавить figure».

Собственные синтетические схемы должны быть редкими: создавать их можно только там, где они действительно проясняют нетривиальную связь, процесс, границу или сравнение, которые плохо удерживаются одной прозой. Не добавляйте схемы ради визуального слоя, декоративного разнообразия, повторения уже сказанного или компенсации несделанного asset-pass.

Если в ходе планирования видно, что фактуры не хватает, очередь должна содержать отдельный `source-pass`. Исполнитель не должен закрывать фактологическую дыру гладкой прозой.

Финальный prompt очереди должен проверять результат, собрать выходные файлы, manifest, verification report и resume note. Эта проверка должна быть механической, а не только повествовательной: каждый файл из раздела `Обрабатываемые файлы` должен существовать или получить явный `blocked` status; target outputs нельзя заменять companion-обещанием `создать позже`; mandatory upstream inputs должны быть реально прочитаны, если они требуются для готовности. Если пакет поддерживает аварийное завершение, это должно быть отражено в финальном prompt-е или в отдельной аварийной инструкции.

## Дополнительные требования к writing-планам

Для writing-задач очередь prompt-ов должна иметь готовые тексты prompt-ов, а не пересказ. Каждый prompt начинается с `Прочитай сначала:` и читает только нужные для текущего прохода материалы.

После направленных проходов добавления фактуры добавляй один свободный source/depth pass: он раскрывает источники, выводимые из текста, source usage, inline-ссылок, названий инструментов, авторов, статей, репозиториев и figure-кандидатов. Новые источники используются только после открытия и фиксируются в source usage.

В системном выравнивании отдельно проверяй, не стал ли текст оглавлением будущих глав вместо самостоятельного аргумента.

Если у будущего фрагмента есть заранее понятная особая опасность, после системного выравнивания можно добавить 1–3 адресных прохода усиления основной функции. Эти проходы могут иметь специальную тему, но только если она прямо следует из роли фрагмента: терминологический контракт, критерии выбора режима, переход между двумя слоями, граница процессов и состояния работы, привязка свидетельств к state transition, сценарий восстановления и т.п. Они не заменяют последующие общие редакторские проходы и не должны становиться визуальным, источниковым или стилевым проходом под другим именем.

После системного выравнивания и до языковых/стилевых проходов добавляй 2–3 общих редакторских прохода оценки и repair. Для поздних синтетических фрагментов также планируй post-import validation после выполнения пакета: уже в развёрнутом репозитории надо независимо проверить существование всех target outputs, фактическое чтение upstream-файлов и синхронизацию companion-долгов. Они должны сохранять общность: каждый проход повторяет простую рабочую формулу «оцени текст, насколько он хорошо выполняет поставленную задачу; после формулирования проблем исправь их», а не получает заранее специальную тему вроде визуального слоя, источников или стиля. Каждый такой проход начинается с диагностики функции фрагмента против target-group plan, скелетона, карты документов, соседних узлов и companion-файлов, затем исправляет реальные дефекты минимально достаточной правкой, завершает regression audit и readiness status (`ready_for_next_fragment`, `ready_with_known_debts`, `blocked_by_source_gap`, `blocked_by_asset_pass` или `needs_human_review`).

Для обычных фрагментов минимум — два общих редакторских прохода. Для несущих, сложных или поздних композиционных фрагментов лучше закладывать три; третий остаётся общим и не должен переписывать текст ради активности, если существенных проблем уже нет. Anti-meta и asset-classification checks выполняются внутри этих общих проходов: служебный мета-текст убирается, но перед любой правкой figure сначала выполняется asset-классификация, чтобы cleanup не превратил реальное изображение в текстовый суррогат. Только после этого идут два языковых и два стилевых прохода.

Не ставь в теоретическом тексте ссылки на внутренние досье, планы и отчёты. Если материал найден через внутренний документ, найди первоисточник; если его нельзя уверенно восстановить, не подменяй его внутренней ссылкой.
