# spdd_method_open_questions

Статус: Final sync. Устаревшие вопросы P01–P09 закрыты, сняты как не-блокеры или переведены в конкретные долги статьи. Шаблонного C5/A10-долга нет: эти фрагменты доступны как read-only context, а текущий долг должен быть сформулирован предметно.

## Закрыто или снято как не-блокер

| Старый вопрос | Текущий статус |
| --- | --- |
| Делать ли отдельный раздел про `/spdd-reverse`? | Нет. В статье он оставлен только как ограниченный путь OpenSPDD от существующего кода к Canvas. Настоящий долг — проверить актуальный источник команды перед расширением или визуализацией. |
| Нужен ли полный разбор установки OpenSPDD по средам? | Нет. Статья не является руководством по установке. Список сред перенесён в раздел про инструментальную поверхность и служит только границе доверия. |
| Достаточно ли примера с биллингом? | Да для текущей статьи. Пример показывает, как один запрос раскладывается по семи частям Canvas и почему Canvas не сводится к обычному плану. |
| Нужно ли превращать `daily-spdd-spec-planner` в отдельный режим метода? | Нет. Он оставлен как внешний сигнал планового сопровождения спецификаций, а не как центральная часть SPDD. |
| Нужна ли отдельная теория формализации намерения? | Нет. `Intent Formalization` используется только для статуса SPDD между свободным промптом и формальными спецификациями. |
| Встраивать ли внешние реальные кандидаты как заглушки? | Да для трёх полезных кандидатов в package-версии: они имеют inline placeholders, нижний раздел `Внешние изображения для asset-pass` и записи в `spdd_method_external_image_queue.md`. |
| Нужна ли отдельная таблица «когда применять / не применять»? | Нет. Это уже раскрыто в разделах «Где SPDD оправдан», «Типичные неверные чтения» и «Сценарии сбоев». |
| Нужен ли шаблонный C5/A10-долг? | Нет. C5 и A10 — read-only context, а не автоматический долг статьи. В текущем тексте нет конкретной несинхронизированной проблемы, требующей отдельного C5/A10-блокера. |

## Настоящие блокеры перед публикацией

- Проверить актуальное состояние OpenSPDD README и шаблонов перед публикацией: статья опирается на v0.4.9 templates и на README/design philosophy как на точечные источники, а содержимое репозитория может измениться.
- Проверить источник по `/spdd-reverse`, если редакция решит расширять его роль: в текущем тексте он не является самостоятельной механикой, но более сильное утверждение потребует актуального template-level подтверждения.
- Проверить Codex skills и `allow_implicit_invocation: false` по текущему OpenSPDD README, если публикационная версия будет подробнее говорить о доверии к навыкам и автозапуске.
- Провести asset-pass для трёх inline external-real candidates: скачать/зафиксировать локальные ассеты или отказаться от них, проверить права, качество и соответствие статье; после этого заменить placeholders и нижний временный раздел локальными `<img>` или удалить кандидаты.
- Перед финальной публикацией выполнить обычную проверку ссылок на внешние источники и относительные ссылки на внутренние фрагменты теории.

## Не делать без нового задания

- Не расширять статью до install manual OpenSPDD.
- Не превращать ledger в coverage-матрицу всех прочитанных источников.
- Не считать inline placeholders публикационными изображениями: они допустимы только как package asset-pass markers; перед публикацией заменить их проверенными реальными изображениями или удалить после решения по кандидатам.
- Не расширять раздел связи с общей рамкой SDLC с ИИ-агентами до отдельной статьи о TDAD, ADR, Spec Kit/Kiro или persistent work graph.
