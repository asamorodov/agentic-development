# Gas Town / Beads restore protocol

Дата: 2026-06-05  
Статус: обязательный протокол перед написанием Gas Town deep part.

## 1. Исходная точка

Gas Town нельзя восстанавливать из последней expanded/synthesis-версии. В ней старый Gas Town уже был сжат до одного подраздела, хотя новая версия добавила отдельные полезные детали.

Правильный seed — полный Gas Town-раздел из старого сайта / документа baseline.

## 2. Обязательные outputs

```text
baseline_extracts/GAS_TOWN_FROM_SITE_BASELINE.md
drafts/part_ix_gas_town_seed_unchanged.md
reports/GAS_TOWN_BASELINE_STRUCTURE_MAP.md
reports/GAS_TOWN_BASELINE_DETAIL_INVENTORY.md
reports/GAS_TOWN_ADAPTATION_PLAN.md
reports/GAS_TOWN_ANTI_DEGRADATION_CHECK.md
```

## 3. Что сохранить из старого сайта

Минимально должны быть проверены и сохранены или явно перенесены:

- исходная логика, почему Gas Town важен;
- роли и организационные функции;
- Mayor как поверхность управляемой видимости;
- Beads как память задачи / причина / блокеры / связи / состояние;
- различие агента, сессии, рабочей идентичности и роли;
- hooks / GUPP / molecules / wisps / supporting agents;
- Witness / Deacon / Refinery / Dogs если присутствуют в baseline;
- стоимость orchestration;
- связь с платформенными примитивами и limits.

## 4. Что добавить поверх baseline

Из expanded quarry и новых источников можно добавить:

- `gt nudge`;
- `gt seance`;
- DoltHub negative evaluation;
- details on worktrees/mailboxes/identities/handoffs if absent or thin;
- comparison with AI-driven SDLC operational lifecycle;
- explicit contrast with SPDD: specification lifecycle vs organizational/operational lifecycle.

## 5. Anti-degradation rule

Новая Gas Town part может быть переструктурирована под SDLC, но не должна стать короче по смыслу, беднее по ролям или общим пересказом. Если деталь из старого сайта исчезает из main text, отчёт должен указать, куда она перенесена и почему.
