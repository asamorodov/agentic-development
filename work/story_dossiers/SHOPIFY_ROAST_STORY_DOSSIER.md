# Shopify Roast

## Черновое назначение

Документ создан режимом fresh для накопления материала из первоисточников. Это досье для будущей истории, а не обзор инструмента. В первом проходе важно не выбрать один красивый смысл, а удержать несколько несовместимых углов: Roast как внутренняя инфраструктура Shopify для массового применения LLM к коду; Roast как Ruby-оркестратор для воспроизводимых AI-workflow; Roast как слой дисциплины вокруг Claude Code; Roast как пример того, как быстро меняется публичный API инструмента и как документация, issues и packaging показывают трение вокруг раннего open source.

## Рабочая карта истории

- Угол 1: **массовая внутренняя инженерная операция**. Shopify Engineering в открытой странице статьи Obie Fernandez от 18 июня 2025 года связывает Roast с Boba: внутренним workflow для добавления Sorbet type annotations в Ruby test suite Shopify, где часть шагов детерминированная, а CodingAgent используется там, где автоматические исправления не закрыли ошибки ([Shopify Engineering, 2025-06-18](https://shopify.engineering/introducing-roast)). Этот угол поддерживает историю про industrial-scale AI-refactoring, но мешает ему то, что публичный репозиторий показывает скорее фреймворк и примеры, чем полную внутреннюю операционную машину Shopify. Числа «около 30 млн строк Ruby» и «250 000 запросов в секунду», занесённые в pass 01, не подтверждены раскрытой официальной страницей Shopify; пока их нужно держать как слабый внешний след, а не как факт из инженерного блога.
- Угол 2: **дисциплина вокруг AI, а не «чат с моделью»**. README текущего `Shopify/roast` описывает Roast как Ruby-фреймворк для structured AI workflows, где workflow задаётся DSL-блоком `execute do ... end`, а шаги могут быть prompt-файлами, Ruby blocks, API-вызовами, tool integrations и ручными проверками ([README](https://github.com/Shopify/roast)). Этот угол поддерживает историю про превращение непредсказуемой работы LLM в воспроизводимые последовательности, но требует осторожности: источник пишет о возможностях фреймворка, а не доказывает качество конкретных изменений.
- Угол 3: **Roast как слой между scripted engineering и агентами**. Текущий README и config-файлы фиксируют cogs `chat`, `agent`, `ruby`, `cmd`, `map`, `repeat`, `call`; `chat` работает с OpenAI, Anthropic, Perplexity и Gemini, а `agent` по умолчанию использует Claude Code CLI и также поддерживает `:pi` ([README](https://github.com/Shopify/roast/blob/main/README.md), [chat config](https://github.com/Shopify/roast/blob/main/lib/roast/cogs/chat/config.rb), [agent config](https://github.com/Shopify/roast/blob/main/lib/roast/cogs/agent/config.rb)). Это даёт материал для истории про агентскую разработку как управляемую среду с явными шагами. Контрфакт: сам репозиторий не показывает внутренние метрики успешности и не раскрывает полный pipeline Shopify.
- Угол 4: **публичная версия как ранний, движущийся open source**. RubyDoc показывает старую модель запуска через `workflow.yml` и `bundle exec roast execute`, а текущий README и tutorial уже ведут к Ruby DSL с `execute do ... end`, `bin/roast execute`, `cmd`, `agent`, `chat`, `ruby`, `map`, `repeat`, `call` ([RubyDoc snapshot](https://rubydoc.info/github/Shopify/roast), [README](https://github.com/Shopify/roast/blob/main/README.md), [tutorial README](https://github.com/Shopify/roast/blob/main/tutorial/README.md)). PR `#519`, `#521` и `#524` показывают переходный слой: preview-ветка, попытка `init` для DSL examples и новый README для 1.0 ([Shopify/roast#519](https://github.com/Shopify/roast/pull/519), [Shopify/roast#521](https://github.com/Shopify/roast/pull/521), [Shopify/roast#524](https://github.com/Shopify/roast/pull/524)). Это отдельная история о том, что практики вокруг AI-инструментов меняются быстрее, чем вторичные обзоры успевают стабилизироваться.

## Контекст

Официальная страница Shopify Engineering, раскрытая во втором проходе, датирует материал 18 июня 2025 года и указывает автором Obie Fernandez; прежняя запись pass 01 про 8 мая 2025 года, Ivan Stanojevic и Sam Rawal выглядит ошибочной или относится к другому, не раскрытому источнику ([Shopify Engineering, 2025-06-18](https://shopify.engineering/introducing-roast)). В тексте Roast определяется как open-source Ruby gem и Ruby-based DSL для создания структурированных AI-workflows, который вырос из внутренней работы Shopify. В статье Roast не выглядит как универсальный «автокодер»: акцент на workflow как на наборе шагов, где обычный код, shell-команды и агентские вызовы соединяются в повторяемый сценарий.

Ближайший внутренний кейс в статье — **Boba**. Это workflow для добавления Sorbet type annotations в Ruby tests. По описанию Shopify, Boba начинает с детерминированных шагов: убирает `# typed: false`, повышает strictness до `true`, запускает Sorbet autocorrect, а затем передаёт оставшиеся ошибки CodingAgent, который получает конкретный файл, ошибки типизации и инструкции исправить проблему без изменения поведения теста ([Shopify Engineering, 2025-06-18](https://shopify.engineering/introducing-roast)). Это важная деталь: AI-часть встроена не в свободный чат, а в цепочку, где часть работы заранее сведена к механическим преобразованиям, а агент получает суженное поле действия.

Внешний пост Daniel Doubrovkine от 10 мая 2025 года показывает более приземлённую сторону: автор устанавливает `roast-ai` локально, получает `Unable to find gem 'roast-ai'` с корпоративным gem source, исправляет это через явное указание `source "https://rubygems.org"` и затем запускает минимальный workflow над вопросом про `rand(1000)` ([Doubrovkine, 2025-05-10](https://code.dblock.org/2025/05/10/executing-structured-ai-workflows-with-shopify-roast.html)). Этот источник полезен именно потому, что в нём видны не только возможности, но и первая шероховатость установки.

## Ход событий и эволюция публичной формы

Этот раздел теперь раскрывает не лозунг `structured AI workflows`, а конкретные рабочие сцены: какие входы собирались, какие команды запускались, где вызывалась модель, где подключался `coding agent`, что оставалось обычным Ruby-кодом, какие артефакты получались на выходе и какие трения видны в публичных PR. Новый материал встроен внутрь самих эпизодов; это не добавка сверху к прежнему пересказу, а замена конспективного слоя прямым описанием событий и механизмов.

### Эпизод 1. Внутренняя мотивация: Shopify сначала ограничивает агента, а не расширяет ему свободу

Официальная страница Shopify Engineering связывает Roast с работой Augmented Engineering Developer Experience team над задачами вроде flaky tests и низкого test coverage. Команда искала способ использовать AI agents для массовой оценки и оптимизации unit tests, но столкнулась с базовой проблемой: агент, которому дают свободно «бродить» по миллионам строк кода, оказывается ненадёжным. В источнике это важно не как общая философия, а как исходная инженерная развилка: если AI-шаг непредсказуем, то управление процессом нужно вынести из модели в исполняемую workflow-структуру ([Shopify Engineering](https://shopify.engineering/introducing-roast)).

Ранняя публичная форма Roast в статье ещё описана как `workflow.yml` плюс markdown-prompt-ы. Workflow собирается из шагов нескольких типов. Директория шага может содержать `prompt.md`; prompt может использовать ERB-переменные; shell-команда пишется как `$()` и её результат передаётся дальше; inline prompt можно написать прямо в YAML; префикс `^` отправляет prompt во встроенный `CodingAgent`, работающий через Claude Code; отдельный шаг можно реализовать пользовательский Ruby-класс, наследуемый от `BaseStep`; вложенные массивы шагов запускаются параллельно. Это не «чат с Claude», а исполняемая процедура, где Roast решает, какие части остаются shell/Ruby, а какие передаются LLM или coding agent ([Shopify Engineering](https://shopify.engineering/introducing-roast), [RubyDoc snapshot](https://rubydoc.info/github/Shopify/roast)).

Набор встроенных tools в раннем описании также показывает, как именно ограничивался агентский труд: `ReadFile`, `WriteFile`, `UpdateFiles`, `Grep`, `SearchFile`, `Cmd`, `Bash`, `CodingAgent`. Модель получает не весь мир, а набор операций внутри заранее заданной workflow-сцены. Поэтому правильная формулировка для истории: Shopify не просто выпустил AI framework; Shopify попытался превратить повторяемую AI-работу в workflow-as-code, где недетерминированный AI-шаг окружён детерминированной подготовкой, проверкой и маршрутизацией.

### Эпизод 2. Внешний запуск старого YAML-примера: grading `resources_test.rb` до Ruby DSL

Пост Daniel Doubrovkine полезен тем, что показывает не презентационную схему, а ручной запуск раннего Roast. Автор получает OpenAI API key, проверяет его обычным `curl` к `gpt-4.1-mini`, клонирует `Shopify/roast` и пытается запустить пример grading workflow. В исходном примере разные шаги были привязаны к разным моделям: coverage analysis использовал `gpt-4.1-mini`, recommendations — `o3`. У автора free tier не позволял пользоваться этими моделями так, как было задумано, поэтому он редактирует `workflow.yml`: добавляет `api_token: $(echo $OPENAI_API_KEY)`, переводит model на `gpt-4.1-mini` и убирает локальные overrides моделей у отдельных шагов `analyze_coverage`, `generate_grades`, `generate_recommendations`. Это конкретный эпизод адаптации на краю применения: чтобы workflow вообще исполнился у внешнего пользователя, модельную конфигурацию пришлось упростить ([Doubrovkine, 2025-05-10](https://code.dblock.org/2025/05/10/executing-structured-ai-workflows-with-shopify-roast.html)).

После этого автор запускает:

```bash
./exe/roast execute examples/grading/workflow.yml test/roast/resources_test.rb
```

Roast начинает выполнение как обычный инструмент командной строки: печатает, какой workflow и какой целевой файл используется, затем выполняет шаги. В логе видны `read_dependencies`, поиск `resources.rb`, чтение `lib/roast/resources.rb`, затем `run_coverage`, который запускает Ruby/Minitest над `resources_test.rb`. Тестовый прогон проходит с seed, показывает 13 runs, 16 assertions, 0 failures, 0 errors, 0 skips. После этого идут AI/analysis шаги: `analyze_coverage`, `verify_test_helpers`, `verify_mocks_and_stubs`, `generate_grades`, `calculate_final_grade`, `format_result`, `generate_recommendations`. Финальный результат пишется в `.roast/sessions/.../final_output.txt` ([Doubrovkine, 2025-05-10](https://code.dblock.org/2025/05/10/executing-structured-ai-workflows-with-shopify-roast.html)).

Сам report тоже важен как проверочный артефакт. Он выставляет overall score 80/100 и grade B, разбивает оценку на line coverage, method coverage, branch coverage, отсутствие helpers/mocks/stubs, readability, maintainability и test effectiveness. Когда автор запускает тот же пример с более дорогими моделями, статический coverage остаётся тем же, но оценка test effectiveness меняется: более сильная модель находит дополнительные untested edge cases и снижает соответствующий subscore. Это хороший прямой материал для будущей истории: Roast не просто «использовал LLM для тестов», а собрал pipeline, где обычный test run и coverage дают факты, LLM оценивает качество тестов по rubric, а финальный output становится сравнимым report. Одновременно источник показывает чувствительность результата к выбранной модели.

### Эпизод 3. Boba: Sorbet typing как смешанный deterministic/agentic workflow

Boba — главный внутренний кейс, но его нужно писать осторожно: публичный источник не даёт полный workflow file, diff, метрики и внутренний масштаб операции. Тем не менее он раскрывает саму форму работы достаточно прямо. Boba занимается добавлением Sorbet type annotations в Ruby tests. Последовательность начинается не с агента, а с механических шагов: cleanup через `sed`, удаление или замена `# typed: false`, повышение strictness, запуск Sorbet autocorrect. Только остаточные ошибки, которые не закрываются детерминированными преобразованиями и autocorrect, передаются `CodingAgent` ([Shopify Engineering](https://shopify.engineering/introducing-roast)).

В этой сцене `CodingAgent` получает уже суженную задачу: конкретный файл, оставшиеся ошибки типизации и instruction исправить код так, чтобы поведение теста не изменилось. Он итеративно правит ошибки типизации, запускает проверки, возвращается к коду и добивается прохождения tests/type checking. Поэтому смысл Boba не в том, что Claude «сам типизирует Ruby test suite». Смысл в том, что Roast ставит Claude Code в конец цепочки после детерминированной подготовки и обратной связи typechecker. Агенту достаётся та часть, где нужно понимание кода; всё, что можно сделать механически, сделано до него.

Это особенно важно для теории агентской разработки: Boba демонстрирует паттерн «сначала сужение пространства, потом агент». Если переписать этот кейс конспективно как «внутренний workflow добавляет Sorbet annotations», пропадает почти всё содержательное. Нужная версия: `sed` и Sorbet autocorrect сначала уменьшают шум; workflow собирает ошибки; `CodingAgent` работает по остаточному набору; tests/type checking закрывают контур проверки. Ограничение сохраняется: источник не позволяет утверждать, как именно был устроен полный Boba pipeline внутри Shopify, сколько файлов он обработал и каков был процент успешных автоматических изменений.

### Эпизод 4. Старый YAML surface: Roast сначала выглядел как convention-oriented orchestration framework

RubyDoc snapshot фиксирует старую публичную модель: workflow описывался YAML-файлом, рядом лежали prompt files, custom Ruby steps и shell script steps. В YAML можно было задавать `name`, глобальную модель, `tools`, `steps`, `each`, `repeat`, `case/when/else`, `json`, `params`, `coerce_to`, `exit_on_error: false`. Шаги могли быть directory-based (`prompt.md`), custom Ruby classes, raw prompt steps или shell scripts; command execution мог включать exit status downstream, если `exit_on_error: false`. Agent step с префиксом `^` отправлял prompt в `CodingAgent` без дополнительной LLM-трансляции, а session continuity поддерживалась через `continue: true` или `resume: step_name` ([RubyDoc snapshot](https://rubydoc.info/github/Shopify/roast)).

В старой модели уже были сильные идеи: shared conversation context, session replay, parallel execution, function caching, instrumentation, human input steps, environment variables для shell steps (`ROAST_WORKFLOW_RESOURCE`, `ROAST_STEP_NAME`, `ROAST_WORKFLOW_OUTPUT`). Но поверхность была data/config-first: автор workflow пишет YAML и prompt-файлы, а не полноценный Ruby DSL. Поэтому переход к Ruby DSL был не косметикой, а сменой объектного уровня: вместо описания структуры процесса в YAML workflow становится обычным Ruby-файлом, где можно пользоваться кодом языка, локальными функциями, scopes, blocks и typed outputs.

### Эпизод 5. Переход к Ruby DSL: PR `#519`, `#521`, `#524` и релизы `v0.5.2` → `v1.0.0`

Переход виден в публичных PR и заметках к релизам. PR `#519` был 1.0 preview branch и прямо формулировал крупное изменение как удаление YAML-based prototype. PR `#521` двигал `init` для DSL workflow examples: идея была копировать примеры из `dsl/<example>.rb` в `.roast`, то есть сделать Ruby DSL examples стартовой точкой для пользователя. PR `#524` переписывал README под 1.0. Эти PR закрыты и находятся в контексте Graphite stack, поэтому их нельзя механически читать как merged history, но они показывают направление работы и язык команды в момент перехода ([Shopify/roast#519](https://github.com/Shopify/roast/pull/519), [Shopify/roast#521](https://github.com/Shopify/roast/pull/521), [Shopify/roast#524](https://github.com/Shopify/roast/pull/524)).

Более устойчивое основание дают releases. `v0.5.2` назван последним planned release before v1.0; в notes сказано, что legacy functionality будет удаляться, `main` пойдёт к release candidate, а v0.x development уйдёт в `develop-v0`. `v1.0.0` затем описан как полная переработка вокруг нового Ruby DSL: workflows are pure Ruby instead of YAML; основные cogs — `chat`, `cmd`, `agent`, `ruby`, `map`, `repeat`; outputs доступны by name; reusable scopes вызываются через `call`; collections обрабатываются через `map`; multiple providers включают OpenAI, Anthropic и Gemini; `agent` cog даёт локальный доступ к файловой системе через Claude Code. Это и есть публичный turning point Roast как инструмента ([GitHub releases](https://github.com/Shopify/roast/releases)).

### Эпизод 6. Минимальный Ruby DSL пример из README: `git diff` → coding agent → stakeholder summary

Текущий README даёт маленький, но очень плотный пример `analyze_codebase.rb`. Workflow начинается с `cmd(:recent_changes)`, который выполняет:

```bash
git diff --name-only HEAD~5..HEAD
```

Затем `agent(:review)` берёт `cmd!(:recent_changes).lines` и просит code-review recently changed files with focus on security, performance, maintainability. Затем `chat(:summary)` читает `agent!(:review).response` и делает summary for non-technical stakeholders. Запуск — `bin/roast execute analyze_codebase.rb` ([Roast README raw](https://raw.githubusercontent.com/Shopify/roast/main/README.md)).

Этот пример нужно писать не как «Roast chains cogs», а как плотную миниатюру управления процессом. Входной набор файлов не придумывает модель: его возвращает `git diff`. Кодовое ревью делает `agent`, то есть локальный `coding agent` с доступом к файловой системе. Пересборку результата для нетехнических заинтересованных лиц делает `chat`, то есть обычный LLM turn. Управление порядком шагов, именами, зависимостью от предыдущего output и командой запуска принадлежит Ruby-файлу. Поэтому workflow file становится reviewable artifact: можно увидеть, какие файлы выбираются, какой агент вызывается, как результат пересказывается и где потенциально поставить проверки.

### Эпизод 7. Tutorial `02_chaining_cogs`: security review, приоритизация, executive summary и Ruby report

`tutorial/02_chaining_cogs/code_review.rb` показывает тот же принцип в более полном виде. В `config` блоке `agent` настроен на provider `:claude` и model `claude-3-5-haiku-20241022`; `chat` настроен на provider `:openai` и model `gpt-4o-mini`. Затем `execute` блок объявляет несколько named cogs. `agent(:review_security)` получает prompt как security-focused code reviewer: проанализировать files changed by latest commit, найти vulnerabilities, объяснить, что это за уязвимость, почему она опасна и как её исправить. `chat(:prioritize)` берёт `agent!(:review_security).text` и просит ranked action list with Critical/High/Medium/Low. `chat(:summarize_for_executive)` тоже берёт agent response и делает 2-3 sentence summary for non-technical stakeholder, с фокусом на бизнес-эффект и срочность ([code review workflow](https://github.com/Shopify/roast/blob/main/tutorial/02_chaining_cogs/code_review.rb), [raw code](https://raw.githubusercontent.com/Shopify/roast/main/tutorial/02_chaining_cogs/code_review.rb)).

После model steps workflow возвращается в Ruby. `ruby(:display_report)` печатает разделители на 80 символов, заголовок `SECURITY REVIEW REPORT`, секции `Executive Summary`, `Prioritized Action Items`, `Detailed Findings`, а затем возвращает hash с `report_generated_at: Time.now.iso8601`, списком sections и `total_length`, взятым из длины `agent!(:review_security).response`. Последний Ruby-шаг `ruby(:check_report)` читает `ruby!(:display_report).value`: если `total_length > 0`, печатает success; иначе warning. В этом примере роли разделены буквально по строкам workflow: Claude-agent ищет vulnerability details, OpenAI-chat перепаковывает результаты для разных читателей, Ruby форматирует report и проверяет, что он не пустой.

### Эпизод 8. Tutorial `02_chaining_cogs`: session resumption и развилки контекста

`session_resumption.rb` раскрывает механизм, который легко потерять, если писать только «Roast supports sessions». В примере `chat(:introduce_topic)` сообщает модели secret code word `thunderbolt` и ожидает, что модель подтвердит запоминание. Ruby-шаг печатает first turn. Затем `chat(:recall_code)` выставляет `my.session = chat!(:introduce_topic).session` и спрашивает secret code word, то есть продолжает conversation state от предыдущего chat cog. Затем `chat(:update_code)` продолжает session от `recall_code` и меняет code word to `mermaid` ([session resumption tutorial](https://raw.githubusercontent.com/Shopify/roast/main/tutorial/02_chaining_cogs/session_resumption.rb)).

Далее пример показывает fork semantics. `chat(:resume_from_beginning)` снова берёт session от `introduce_topic`, а не от `update_code`, и спрашивает current code word. Комментарий в source объясняет смысл: возобновление из сохранённой session создаёт развилку разговора; из одной и той же сохранённой точки можно создавать новую ветку без добавления нового контекста. После chat example идёт agent example: `agent(:analyze_file)` просит agent перечислить файлы в текущей директории, а `agent(:followup_question)` выставляет `my.session = agent!(:analyze_file).session` и просит выбрать один file и объяснить, что он делает. Это не просто convenience; это способ сделать transcript/session first-class output между cogs, чтобы следующий шаг мог продолжить или fork previous model context.

### Эпизод 9. Tutorial `07_processing_collections`: массовая обработка и parallel map как переход от одного prompt к batch workflow

Раздел `07_processing_collections` важен, потому что Roast нужен не только для одного красивого agent call. `map` выполняет named scope для каждого item: автор workflow объявляет scope через `execute(:name)`, а `map` вызывает его на наборе элементов. `collect` затем собирает outputs; block может видеть output, item и index. `reduce` объединяет outputs с required initial accumulator и может skip item, если block возвращает `nil`. Для parallel execution есть `config do map do parallel(3) end` или `map(:unlimited) do parallel! end`; реализация использует fibers, но `collect` и `reduce` возвращают original order regardless completion order ([tutorial chapter 7](https://github.com/Shopify/roast/blob/main/tutorial/07_processing_collections/README.md)).

Практически это означает, что Roast пытается закрыть gap between one-off prompt и массовой инженерной операцией. Workflow может обработать collection serially или параллельно, затем собрать outputs, обратиться к `.iteration(index)`, `.first`, `.last`, задать `initial_index = 1`, передать index в `call`. Это маленькие DSL-детали, но для истории они критичны: если Boba или test grading нужно масштабировать на множество файлов, нужен именно такой слой итерации, а не ручное повторение chat prompts.

### Эпизод 10. Provider configuration PR `#888`: provider choice остаётся в workflow, а не в глобальном переключателе

PR `#888` был merged README-only change, сделанный в ветке `codex/provider-setup-docs`. Его summary уточняет provider settings: `chat` steps получают `OPENAI_API_KEY`, optional `OPENAI_API_BASE` и provider config вроде `provider :openai`; `agent` providers включают `:claude` и `:pi`, причём соответствующие локальные CLI должны быть отдельно установлены и авторизованы. Важное отрицательное знание: PR фиксирует, что нет CLI flag или env var для глобального переключения default provider; provider choice задаётся in workflow config blocks ([Shopify/roast#888](https://github.com/Shopify/roast/pull/888)).

Это operational episode, а не просто docs polish. В Roast один workflow может вызывать `agent` через Claude Code или Pi, а `chat` через OpenAI, Anthropic, Perplexity или Gemini, но эта маршрутизация должна быть видна в самом workflow/config. Поэтому этот PR хорош как источник о provider-документации и Codex-assisted сопровождения docs, но не как доказательство runtime behavior.

### Эпизод 11. Formatter/block-events PR stack: agent evidence нужно не только получить, но и показать читаемо

Серия PR `#893`, `#898`, `#899`, `#900`, `#901`, `#902`, `#903` раскрывает другой тип реальной работы: когда workflow начинает печатать prompt/response, stats и tool output, простое логирование быстро становится частью продукта. `#893` закрывает issue `#876` и меняет terminal/log output так, чтобы каждая строка показывала, к какому action она относится. Автор выбирает dots on continuation lines вместо whitespace, потому что plain whitespace выглядит случайной ошибкой и может быть stripped by `msg2str`. Reviewer принимает dots as intentional и robust, но PR одновременно раскрывает хрупкость прежнего подхода: formatter угадывал stdout/stderr coloring по символам внутри строки ([Shopify/roast#893](https://github.com/Shopify/roast/pull/893)).

Дальше `#898` вводит новый тип block event для более аккуратного форматирования prompt/response сообщений из `chat` и `agent` cogs. В PR body обсуждаются варианты и выбирается block representation: вместо того чтобы multi-line prompt/response размазывались по plain log lines, событие получает header/content structure. `#899` переводит Claude agent cogs на block events; reviewer замечает, что это позволяет избавиться от части `io` capture in tests. `#900` делает то же для chat cogs. `#901` распространяет подход на Pi agent cogs. `#902` переносит stats output в ту же модель: `AGENT STATS` включает turns, duration, cost, tokens и session id; `LLM STATS` включает model, optional temperature, input tokens и output tokens. Наконец `#903` вводит `Roast::Log::Message(type:, text:)`, меняет `LogFormatter#colourize`, чтобы он смотрел на `type`, а не угадывал смысл по glyphs вроде `❯`, `❙`, `❯❯`, `❙❙`, и добавляет `.to_s` normalization, потому что messages могут быть string или `Roast::Log::Message` ([Shopify/roast#898](https://github.com/Shopify/roast/pull/898), [Shopify/roast#899](https://github.com/Shopify/roast/pull/899), [Shopify/roast#900](https://github.com/Shopify/roast/pull/900), [Shopify/roast#901](https://github.com/Shopify/roast/pull/901), [Shopify/roast#902](https://github.com/Shopify/roast/pull/902), [Shopify/roast#903](https://github.com/Shopify/roast/pull/903)).

Для будущей истории это сильный материальный слой. Roast продаётся как structured AI workflows, но структура нужна не только внутри исполнения. Она нужна в output, потому что пользователь, reviewer или agent должны видеть prompt, response, stats, session id и границы action. Иначе workflow evidence превращается в плохо читаемый stream. PR stack показывает, как проект постепенно делает transcript/log output more typed: сначала line prefixes, потом block events, потом typed log messages.

### Эпизод 13. Побочные operational use cases: tests, SRE, competitive intelligence, Chesterton's Fence

Официальная статья перечисляет ещё несколько применений: test quality at scale, SRE workflow for scanning Slack, competitive intelligence from news/market analysis/customer/CRM data, и Chesterton's Fence code research tool, который исследует commit history и related PRs, чтобы объяснить, почему существует код. В будущей истории эти use cases лучше использовать не как доказанные подробные эпизоды, а как evidence, что Shopify видел Roast как внутренний orchestration substrate, а публичный репозиторий раскрывает только часть этого substrate.

## Практика в действии

После прямого раскрытия источников практика Roast выглядит не как один метод, а как несколько повторяющихся рабочих форм.

**1. Workflow сначала собирает и сужает вход, а модель работает уже по подготовленному материалу.** В `analyze_codebase.rb` список файлов берётся из `git diff`, а не из догадки агента. В grading example сначала запускается Minitest/coverage и только потом LLM оценивает качество покрытия и рекомендации. В Boba сначала идут `sed` и Sorbet autocorrect, затем `CodingAgent` получает остаточные ошибки типизации. Во всех трёх случаях модель не является первым и единственным исполнительным механизмом.

**2. Разные модели и agent runtimes используются как разные инструменты workflow.** В tutorial security review `agent` через Claude делает code-level analysis, `chat` через OpenAI делает prioritization и executive summary, Ruby собирает report. В provider docs `chat` и `agent` имеют разные provider surfaces и credentials. Следовательно, Roast не равен «обёртке над Claude Code»: Claude Code является одним мощным cog/provider внутри workflow, но chat-cogs и command/Ruby steps тоже формируют execution.

**3. Выходы становятся именованными артефактами, а не просто прозой в чате.** `cmd!(:recent_changes).lines`, `agent!(:review).response`, `agent!(:review_security).text`, `ruby!(:display_report).value`, `chat!(:introduce_topic).session`, `map` iteration outputs — это всё ссылки на named outputs. Благодаря этому следующий шаг может явно читать результат предыдущего, strict/tolerant access can be chosen with bang/non-bang methods, а session can be resumed or forked. Это превращает conversation state into data carried by workflow.

**4. Человеко-читаемая evidence становится отдельной инженерной задачей.** Formatter stack показывает, что для agentic workflow мало получить корректный response. Нужно показывать prompts, responses, stats, session ids и action boundaries так, чтобы человек мог review, а tests могли проверить output. Поэтому block events и typed log messages — не вторичный UI-polish, а часть harness: без них workflow становится трудным для аудита.

**5. Roast сам становится объектом agent-readable maintenance.** Architecture docs PR `#868` пытается дать future contributors и AI agents not only overview, but known bugs/gotchas. Draft workflow `#892` пытается сделать validation/update of those docs частью Roast workflow. Эта линия пока незавершённая, но она важна: tool for structured AI workflows начинает строить structured documentation workflow for itself.

## Повороты взгляда

Первый очевидный framing — «Shopify открыла инструмент, которым чинит свою кодовую базу AI». Источники заставляют уточнить его. Статья действительно даёт масштаб Boba и Shopify-контекст, но публичный README в первую очередь показывает фреймворк для workflow, providers и cogs. Между внутренней Boba и внешним `roast-ai` есть разрыв: будущей истории нужно не обещать, что любой пользователь получает всю внутреннюю машину Shopify, а показать, какие принципы вынесены наружу.

Второй поворот связан с формой инструмента. Старый публичный материал и внешний пост держатся за `workflow.yml`, а текущий README показывает Ruby DSL с `execute do ... end` и cogs. Это не мелочь документации: если писать историю по одному найденному источнику, можно ошибиться в текущей рабочей форме Roast. Досье должно хранить обе версии как слои времени.

Третий поворот — роль Claude и агентских провайдеров. Внешние обзоры могут быстро свести Roast к «оркестрации Claude Code», но текущие источники шире и точнее: `agent` действительно по умолчанию использует Claude Code CLI, однако `chat` имеет отдельный provider-слой для OpenAI, Anthropic, Perplexity и Gemini, а workflow может соединять оба типа cog в одном сценарии ([README](https://github.com/Shopify/roast/blob/main/README.md), [chat config](https://github.com/Shopify/roast/blob/main/lib/roast/cogs/chat/config.rb), [agent config](https://github.com/Shopify/roast/blob/main/lib/roast/cogs/agent/config.rb), [code_review.rb](https://github.com/Shopify/roast/blob/main/tutorial/02_chaining_cogs/code_review.rb)). Поэтому история про Claude возможна, но пока не должна подменять собой весь Roast.

## Ограничения и неудобные детали

- Публичная статья Shopify сильна как первичный source для намерения и Boba-рамки, но она не даёт полного воспроизводимого набора артефактов Boba: нет внутреннего workflow-файла, набора PR, метрик качества или примеров конкретных diff ([Shopify Engineering, 2025-06-18](https://shopify.engineering/introducing-roast)).
- Внешний пост Doubrovkine показывает рабочий запуск, но это минимальный учебный пример про `rand(1000)`, а не production-кейс. Его нельзя использовать как доказательство масштаба или качества Boba; он полезен для деталей установки, файлов и вывода ([Doubrovkine, 2025-05-10](https://code.dblock.org/2025/05/10/executing-structured-ai-workflows-with-shopify-roast.html)).
- Документация расходится по API: RubyDoc и ранние статьи показывают YAML workflow, текущий README показывает Ruby DSL. PR `#519`, `#521` и `#524` раскрыты во втором проходе и подтверждают, что переход был не косметическим, но их статус закрыт без merge, поэтому нужна отдельная проверка git history/tags, чтобы точно датировать попадание DSL в `main` ([RubyDoc snapshot](https://rubydoc.info/github/Shopify/roast), [README](https://github.com/Shopify/roast/blob/main/README.md), [Shopify/roast#519](https://github.com/Shopify/roast/pull/519), [Shopify/roast#521](https://github.com/Shopify/roast/pull/521), [Shopify/roast#524](https://github.com/Shopify/roast/pull/524)).
- PR `#468` «Avoiding session file names that are too long» показывает не методологическую, а файловую проблему: state-файлы раньше назывались через полный step name, а исправление вводит `MAX_FILENAME_LENGTH = 255`, `sanitize_step_name`, MD5-суффикс и тест с очень длинным prompt-подобным названием шага про coverage MCP tool ([Shopify/roast#468](https://github.com/Shopify/roast/pull/468)). PR тоже закрыт без merge, но это хороший след реального трения вокруг session storage и длинных агентских инструкций.
- Packaging-данные из внешних индексов всё ещё требуют проверки, но repo-состояние стало яснее: текущий `main` задаёт `VERSION = "1.1.0"` и gem name `roast-ai` через `roast-ai.gemspec` ([version.rb](https://github.com/Shopify/roast/blob/main/lib/roast/version.rb), [roast-ai.gemspec](https://github.com/Shopify/roast/blob/main/roast-ai.gemspec)). RubyGems и The Ruby Toolbox остаются отдельными индексами с собственными задержками и download-count, их нельзя смешивать с repo version как один источник.
- Packaging-индексы теперь выглядят не просто «требуют проверки», а явно расходятся. RubyGems-страница `roast-ai`, раскрытая в pass 03, показывает текущей версией `0.4.3`, дату релиза 8 августа 2025 года, 17 total versions, total downloads `10,920`, runtime dependencies старого слоя (`cli-kit`, `cli-ui`, `json-schema`, `open_router`, `raix-openai-eight`, `sqlite3`, `thor` и др.) и required Ruby `>= 0` ([RubyGems](https://rubygems.org/gems/roast-ai)). GitHub Releases при этом показывают `v1.0.0` 23 февраля 2026 года и `1.1.0` 2 апреля 2026 года, а The Ruby Toolbox показывает current version `1.1.0`, 34 releases и downloads `256,250`, но одновременно странные даты first/latest release `1980-01-02` ([GitHub releases](https://github.com/Shopify/roast/releases), [The Ruby Toolbox](https://www.ruby-toolbox.com/projects/roast-ai)). Вывод для будущей истории: packaging-слой нельзя датировать одним числом из индекса; нужно явно различать repo release, RubyGems page и агрегатор.
- PR `#893`-`#903` раскрывают output/formatter слой, но не раскрывают bodies исходных issues `#876`, `#904`-`#909` отдельно: связь с issues видна через `fixes/closes` в PR bodies, однако фактура прохода основана на PR bodies, diffs и review comments, а не на issue bodies ([Shopify/roast#893](https://github.com/Shopify/roast/pull/893), [Shopify/roast#898](https://github.com/Shopify/roast/pull/898), [Shopify/roast#903](https://github.com/Shopify/roast/pull/903)).
- PR `#888` раскрыт как merged README-only change, но он сам содержит оговорку по verification: `bundle exec rake` не запускался из-за отсутствия Ruby/Bundler в workspace. Поэтому это хороший source для provider-документации и Codex-assisted docs workflow, но не для утверждения, что provider behavior был протестирован в этом PR ([Shopify/roast#888](https://github.com/Shopify/roast/pull/888)).
- PR `#868` и `#892` нельзя писать как завершённую линию внедрения architecture docs: `#868` открыт, автор называет материал draft LLM-generated documentation и прямо сомневается в полной корректности; `#892` открыт как draft, не mergeable через GitHub из-за открытого downstack PR и содержит только минимальный `cmd(:files)` workflow с опечаткой `disdlay!` ([Shopify/roast#868](https://github.com/Shopify/roast/pull/868), [Shopify/roast#892](https://github.com/Shopify/roast/pull/892)).

## Неровная фактура

- Внешний запуск начался с банального сбоя установки gem из-за приватного source: `Unable to find gem 'roast-ai' in rubygems repository https://rubygems.pkg.github.com/dblock/`. Исправление — явно добавить `source "https://rubygems.org"` в Gemfile. Это маленькая деталь, но она показывает, что adoption начинается не с философии AI, а с resolver и настроек RubyGems ([Doubrovkine, 2025-05-10](https://code.dblock.org/2025/05/10/executing-structured-ai-workflows-with-shopify-roast.html)).
- В примере Doubrovkine workflow вызывает `gpt-4o`, а текущие config-файлы показывают provider-слой для OpenAI, Anthropic, Perplexity и Gemini. Это временной слой и одновременно предупреждение: Roast нельзя автоматически читать как Claude-only ([Doubrovkine, 2025-05-10](https://code.dblock.org/2025/05/10/executing-structured-ai-workflows-with-shopify-roast.html), [chat config](https://github.com/Shopify/roast/blob/main/lib/roast/cogs/chat/config.rb)).
- В README рядом стоят `chat`, `agent`, `ruby` и `cmd`. Это неровная, но важная деталь: AI-шаги не изолированы от обычной инженерной автоматизации, а включены в тот же исполняемый workflow, где можно вызвать `git diff`, передать список файлов агенту и затем пересказать результат отдельным LLM-запросом ([README](https://github.com/Shopify/roast/blob/main/README.md)).
- `map` с `parallel(3)` или `parallel!` в tutorial выглядит как маленькая DSL-деталь, но именно она показывает, как Roast пытается перенести задачу от «одного prompt на один файл» к массовой обработке входного набора ([tutorial chapter 7](https://github.com/Shopify/roast/blob/main/tutorial/07_processing_collections/README.md)).
- В тесте PR `#468` длинный step name составлен как человеческая агентская инструкция: «use the coding agent and its coverage mcp tool...» повторяется пять раз. Это почти комическая, но полезная деталь: в agentic workflow имя шага легко становится длинным prompt-описанием, а затем ломает файловую модель state storage ([Shopify/roast#468](https://github.com/Shopify/roast/pull/468)).
- Открытый issues list на GitHub к началу июня 2026 года показывает, что после `1.1.0` трение не исчезло, а сместилось в форматирование, provider boundaries и tool-output слой: открыты `#875` «Replace deprecated models in docs and tutorial workflows» от 13 мая 2026, `#882` «Clarify providers and env variables in docs» от 14 мая 2026, `#895` «Agent and chat cogs' prompts and responses are not formatted correctly» и discussion `#896` «Potential approaches to prompt and response formatting» от 26 мая 2026; 4 июня 2026 появилась серия открытых задач `format_read`, `format_grep`, `format_edit`, `format_task`, `format_agent`, `format_taskupdate` и близких formatter-issues ([GitHub issues list](https://github.com/Shopify/roast/issues)). Пока это не раскрытые issue bodies, но уже проверяемый сигнал: публичный developer experience вокруг agent/chat output остаётся активной рабочей зоной, а не завершённой polish-фазой.
- В pass 04 добавлен осторожный след про block events: поисковые результаты и GitHub activity указывают на работы `Add a block type event`, `Use block events in claude`, `Use block events in chat cogs`, `Use block events in LLM STATS` ([GitHub activity](https://github.com/Shopify/roast/activity)). Это нельзя ещё писать как завершённый эпизод, но это полезный кандидат: возможно, ответом на проблемы форматирования стал не один formatter, а переход к более структурированному event-представлению вывода.
- PR `#893` оставляет полезную review-фактуру: попытка решить многострочный вывод через особые символы сразу породила разговор о хрупкости `LogFormatter`, который распознаёт stdout/stderr по символам в строке. Reviewer прямо указала, что нужен first-class способ обозначать раскраску, а follow-up `#903` сделал это через `Roast::Log::Message(type:, text:)` ([Shopify/roast#893](https://github.com/Shopify/roast/pull/893), [Shopify/roast#903](https://github.com/Shopify/roast/pull/903)).
- В `#902` спор дошёл не до алгоритма, а до визуальной формы header. Reviewer заметила, что после появления нового formatter `AGENT STATS` больше не обязан быть в all caps и square brackets; автор ответил, что all caps сохраняет единообразие с prompt/response headers, но признал, что формат headers можно обсуждать отдельно ([Shopify/roast#902](https://github.com/Shopify/roast/pull/902)). Это маленькая, но живая деталь: когда workflow выводит agent evidence, интерфейсные решения попадают прямо в code review.
- В `#888` есть маленькая, но важная строка provenance: правка была сделана с Codex assistance, а финальный diff сверили с provider configuration source. То есть AI применён не только как объект Roast, но и как участник сопровождения документации Roast; человеческая проверка источника при этом остаётся частью процесса ([Shopify/roast#888](https://github.com/Shopify/roast/pull/888)).
- В `#868` generated architecture docs включают файл для «плохих новостей», а не только описание счастливого пути: hardcoded path для Claude dump, расхождения `next!` в sync/async, edge cases вокруг `outputs!`, empty scope и falsy values. Для истории это сильная деталь: если такие docs действительно предназначены для AI agents and new contributors, проект пытается заранее дать агенту карту мест, где легко ошибиться ([Shopify/roast#868](https://github.com/Shopify/roast/pull/868)).
- В `#892` сам workflow для проверки architecture docs содержит `disdlay!`. Это не стоит высмеивать, но стоит сохранить: ранняя автоматизация проверки AI-generated docs началась с draft-файла, который ещё не выглядит исполняемым без правки ([Shopify/roast#892](https://github.com/Shopify/roast/pull/892)).

## Детали, восстановленные после сверки с последней загруженной версией

- The package-level mechanics should remain: `Gemfile`, `lib/roast.rb`, `roast.gemspec`, `spec.name = "roast-ai"`, `spec.version = Roast::VERSION`, `required_ruby_version >= 3.3.0`, and dependencies such as `activesupport`, `ruby_llm`, `type_toolkit`, `zeitwerk` show Roast as a Ruby library and not just a blogged workflow idea ([Shopify Roast repository](https://github.com/Shopify/roast)).
- The historical YAML surface should not disappear: early examples included `roast execute workflow.yml`, `tools:` blocks and `steps/echo_response.rb`; this matters because the story changed from YAML / step files toward `Roast.workflow` / `Roast::Workflow` and then Ruby DSL/cogs ([Shopify Roast repository](https://github.com/Shopify/roast)).
- Keep the transition markers: `Roast.workflow`, `Roast::Workflow`, `NotImplementedError`, `v0.x`, `async`, `DEBUG` and the old prompt-like example `{"question":"does rand(1000) include 1000?"}` help show that Roast evolved through real public API surfaces, not only through a cleaned-up launch narrative ([Shopify Roast repository](https://github.com/Shopify/roast)).

## Синтез

Shopify Roast is a strong story because it turns AI workflow from instruction into executable code. The key is not “Ruby DSL” as a technology preference, but the separation of responsibilities: `cmd` and `ruby` handle deterministic work, `agent` handles local code changes, `chat` handles model summaries, `map` and `repeat` express control flow. The agent becomes one cog in a process, not the process owner.

The Boba/Sorbet example gives the best narrative core: mechanical cleanup and Sorbet autocorrect reduce the problem; the coding agent handles the remaining ambiguous ошибки типизации; tests and typecheck close the loop. The public evolution from YAML prototype to Ruby DSL, followed by provider/session/logging friction and draft agent-readable architecture docs, makes the story less smooth and more valuable. Roast shows what happens when a company starts treating AI-assisted work as something that should be versioned, reviewed, run and debugged like code.
