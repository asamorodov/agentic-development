# Armin Ronacher

## Черновое назначение

Документ создан режимом fresh для накопления материала из первоисточников по Armin Ronacher как истории внутри корпуса AI-assisted / agentic development. Это досье пока не должно выбирать одну красивую рамку. В первом проходе важнее сохранить несколько конфликтующих линий: практический восторг от agentic coding, инженерная дисциплина вокруг контекста и инструментов, неудобная сторона open source после появления дешёвых AI-вкладов, а также переход от использования Claude Code к попытке сделать собственную среду Pi.

## Рабочая карта

- Armin Ronacher - автор Flask, Jinja, Werkzeug, Click, MiniJinja, Rye и других проектов; для этой истории важна не биографическая известность, а то, что он одновременно ведёт mature open-source проекты, пишет инструменты и публично фиксирует собственные рабочие изменения в эпоху coding agents.
- Линия 2025 года начинается не с готового метода, а с признания, что AI меняет повседневную разработку сильнее, чем прежние абстракции программирования. В июньском посте Ronacher пишет, что даже если текущий AI не даёт фундаментального скачка абстракции, он уже меняет написание, сопровождение, использование и обучение software [AI Changes Everything, 2025-06-04](https://lucumr.pocoo.org/2025/6/4/changes/).
- Через неделю появляется более конкретная инженерная заметка про agentic coding: Ronacher описывает, как работает с Claude Code, почему модель нужно держать внутри репозитория, как он дробит задачи, когда использует режим plan, и почему он предпочитает поручать агенту не только код, но и тесты, refactoring, git history, changelog и review loops [Agentic Coding Recommendations, 2025-06-12](https://lucumr.pocoo.org/2025/6/12/agentic-coding/).
- Следующая линия быстро становится менее благополучной: Ronacher фиксирует, что AI-ассистированные пользователи и contributors резко увеличивают шум в issue tracker, создают уверенные, но не проверенные отчёты, и перекладывают стоимость проверки на maintainers; затем отдельно уточняет, что слово `clanker` ему нужно для снятия антропоморфизма с машины, а не как ярлык против людей [Clanker: A Word For The Machine, 2026-05-26](https://lucumr.pocoo.org/2026/5/26/clankers/).
- В 2026 году история уже не только о Claude Code. Ronacher представляет Pi как open-source coding agent с desktop UI, локальным состоянием, встроенной памятью задач и намерением уменьшить зависимость от terminal-first Claude Code [Pi: An Open Source AI Coding Agent, 2026-01-31](https://lucumr.pocoo.org/2026/1/31/pi/); позже пишет, что Pi "на 95% написан с помощью Pi", но одновременно показывает цену такой рекурсии: неконсистентные интерфейсы, риск потери локального контекста, фиксацию session transcript как источника правды и необходимость переупорядочивать систему через issues [Building Pi With Pi](https://lucumr.pocoo.org/2026/5/24/pi-oss/).
- Третий проход добавляет важную поправку к Pi: текущий README `earendil-works/pi` уже описывает проект как `Pi Agent Harness Mono Repo`, а не только как desktop-приложение. В публичной поверхности появились `@earendil-works/pi-coding-agent` как interactive CLI, `@earendil-works/pi-agent-core`, `@earendil-works/pi-ai`, экспорт session traces на Hugging Face, отдельные containerization patterns и supply-chain hardening [GitHub: earendil-works/pi README](https://github.com/earendil-works/pi/blob/main/README.md).
- Четвёртый проход добавляет более низкий слой Pi: issue tracker используется не только для внешней поддержки, но и как проектный журнал самой агентской среды. Issue `#92` про context compaction содержит команды `/compact` и `/autocompact`, настройки `reserveTokens` / `keepRecentTokens`, правило не резать ход сессии посередине, запись compaction event в session file и связь с `/branch` [Pi issue #92](https://github.com/earendil-works/pi/issues/92), [Pi compaction.md](https://github.com/earendil-works/pi/blob/main/packages/coding-agent/docs/compaction.md).
- Та же проверка добавляет поправку к простой рамке "внешние issues = шум": открытый issue `#4945` от внешнего пользователя описывает зависание `openai-codex` / `gpt-5.5` в TUI с `Working...`, нулевым usage и прерванным assistant turn, помечен как `inprogress` и связан с PR `#4979` [Pi issue #4945](https://github.com/earendil-works/pi/issues/4945). Это не отменяет перегрузку, но показывает, что contribution surface не сводится к автоматическому мусору.
- Пятый проход уточняет этот counterexample через release `v0.76.0`: changelog фиксирует, что `httpIdleTimeoutMs` стал применяться к OpenAI Codex Responses WebSocket idle waits, добавлен `websocketConnectTimeoutMs` и 10-секундный timeout для Codex SSE response header [Pi release v0.76.0](https://github.com/earendil-works/pi/releases/tag/v0.76.0). Значит, внешний отчёт не просто висел с label, а дошёл до release artifact.
- Шестой проход раскрывает сам PR `#4979`: Ronacher открыл branch `fix/codex-websocket-idle-timeout`, сначала написал, что PR не гарантирует исправление `#4945`, а приводит поведение Pi к Codex-style connection/idle timeout, затем PR прошёл через force-push, `ready for review`, 4 commits, 1 check и merge 27 мая 2026 года [Pi PR #4979](https://github.com/earendil-works/pi/pull/4979). После merge в discussion появился вопрос от `devxoul`, нужно ли переоткрыть `#4945`, потому что issue закрылся автоматически; это важная неровность в цепочке "issue -> fix" [Pi PR #4979](https://github.com/earendil-works/pi/pull/4979).
- Седьмой проход сужает проверку до contribution gate как исполняемой инфраструктуры, а не только текста `CONTRIBUTING.md`. В `earendil-works/pi` есть отдельные workflows `issue-gate.yml` и `pr-gate.yml`: первый закрывает новые issues от неразрешённых авторов после комментария про auto-close, второй закрывает PRs от авторов без `lgtm`-уровня допуска; оба читают `.github/APPROVED_CONTRIBUTORS` и различают capability `issue` и `pr` [Pi issue-gate.yml](https://raw.githubusercontent.com/earendil-works/pi/main/.github/workflows/issue-gate.yml), [Pi pr-gate.yml](https://raw.githubusercontent.com/earendil-works/pi/main/.github/workflows/pr-gate.yml), [Pi APPROVED_CONTRIBUTORS](https://raw.githubusercontent.com/earendil-works/pi/main/.github/APPROVED_CONTRIBUTORS). Отдельный `openclaw-gate.yml` добавляет label `possibly-openclaw-clanker`, если автор уже открывал issues/PRs в `openclaw/openclaw`; это важная поправка к чтению `#4945`: label является эвристикой по прежней активности автора, а не доказательством, что конкретный отчёт был машинно создан [Pi openclaw-gate.yml](https://raw.githubusercontent.com/earendil-works/pi/main/.github/workflows/openclaw-gate.yml).
- Восьмой проход добавляет рядом с `#4945` ещё три внешних входа, чтобы не строить вывод по одному удобному примеру. `#4984` и `#5028` показывают баг-репорты, которые прошли в статус `inprogress`: один приносит `EPIPE` в `loadDiffLines`/`formatDiffPreview` с локальной diagnostic patch и временным workaround, второй связывает Windows clipboard failure с MSYS2 `ucrt64`, `@napi-rs/clipboard` и двумя проверенными обходами через `win32yank.exe` или PowerShell [Pi issue #4984](https://github.com/earendil-works/pi/issues/4984), [Pi issue #5028](https://github.com/earendil-works/pi/issues/5028). `#5027`, наоборот, остаётся закрытым feature request без labels, assignee, project и development links в основном repo, хотя body содержит точное предложение `$RAW_ARGUMENTS`; это полезный контраст между воспроизводимым сбоем и желательным изменением API [Pi issue #5027](https://github.com/earendil-works/pi/issues/5027).
- К июню 2026 линия вокруг AI становится ещё более социальной: в "Communities of Not" Ronacher пишет не о рабочем процессе, а о том, как LLM-скептические developer spaces могут превращать отказ от технологии в идентичность и травлю людей, которые пробуют LLMs [Communities of Not, 2026-06-06](https://lucumr.pocoo.org/2026/6/6/communities-of-not/).

## Возможные углы истории

### 1. Практик, который не продавал магию, а менял рабочую механику

Первый сильный угол: Ronacher как senior engineer, который довольно рано перестал обсуждать AI как демо и стал описывать конкретную производственную механику. В "Agentic Coding Recommendations" он пишет не про один prompt, а про режим работы: держать агента в контексте репозитория, давать ему маленькие tasks, просить его читать существующий код и тесты, использовать plan mode перед правкой, смотреть diff, запускать тесты и давать follow-up instructions рядом с результатом [Agentic Coding Recommendations](https://lucumr.pocoo.org/2025/6/12/agentic-coding/). В этой рамке агент не заменяет разработчика; он ускоряет итерации там, где человек умеет проверять результат.

Этой рамке мешает то, что сам Ronacher довольно быстро описывает и обратную сторону. В "Agentic Coding: Things That Didn't Work" он перечисляет не провалы AI вообще, а конкретные зоны, где агентская разработка ломается: слишком большой контекст, слабая способность удерживать long-running architecture, трудность с product taste и неудобство, когда агент пытается "починить" всё вокруг вместо маленькой задачи [Agentic Coding: Things That Didn't Work, 2025-07-30](https://lucumr.pocoo.org/2025/7/30/things-that-didnt-work/). Для истории важно не потерять эту самокоррекцию: июньский оптимизм быстро получает список границ.

### 2. Инструментальный минимализм против "tool explosion"

Второй угол: Ronacher как человек, который поверил в tools, но не в произвольное расширение tool surface. В "Tools: Code is all you need" он утверждает, что code сам по себе остаётся лучшим интерфейсом для агента: вместо множества специальных tool-вызовов агент может читать и писать файлы, запускать команды, создавать маленькие scripts и использовать обычные programming interfaces [Tools: Code is all you need, 2025-07-03](https://lucumr.pocoo.org/2025/7/3/tools/). Это хорошо ложится на его open-source биографию: он привык к библиотекам, CLI, репозиториям и тестам как к прозрачным рабочим артефактам.

Эта линия уточняется в "Your MCP Doesn't Need 30 Tools": Ronacher спорит не с MCP как идеей, а с перегруженными серверами, где модель получает десятки похожих commands и должна выбирать из шумного меню. Он предлагает более узкий surface: меньше инструментов, более сильные affordances, возможность агенту самому строить нужную процедуру поверх простых primitives [Your MCP Doesn't Need 30 Tools, 2025-08-18](https://lucumr.pocoo.org/2025/8/18/code-mcps/). Это потенциально важный поворот для истории: зрелость агентской разработки здесь выражается не в добавлении кнопок и capabilities, а в уменьшении и стабилизации интерфейса между моделью и средой.

### 3. Open source как место, где AI переносит стоимость на maintainer

Третий угол более конфликтный. В "Building Pi With Pi" Ronacher описывает новый тип участника: человек приходит в issue tracker с AI-сгенерированным отчётом, уверенностью и слабой проверкой, но не несёт основную стоимость диагностики [Building Pi With Pi](https://lucumr.pocoo.org/2026/5/24/pi-oss/). Через два дня в "Clanker: A Word For The Machine" он уже разбирает сам термин: слово должно отделять машину от человека и оставлять ответственность на пользователе или организации, которая применяет модель [Clanker: A Word For The Machine](https://lucumr.pocoo.org/2026/5/26/clankers/). В этой рамке AI не просто увеличивает производительность автора. Он меняет социальный контракт проекта: дешевле стало производить вклад, но не обязательно дешевле стало его принимать.

Этот угол не отменяет практический AI-оптимизм Ronacher. Наоборот, именно поэтому он интересен: один и тот же человек публично использует agents для ускорения собственной работы и одновременно формулирует жёсткую проблему качества AI-mediated contributions. История может стать не "за AI" и не "против AI", а про асимметрию: польза высока, когда агент работает внутри контекста ответственного maintainer; вред растёт, когда агент производит внешние артефакты без владения последствиями.

### 4. Pi как попытка перенести агентскую практику из терминала в управляемую среду

Четвёртый угол: Ronacher не остановился на рекомендациях к Claude Code, а начал строить собственный open-source coding agent Pi. В январском посте он описывает Pi как desktop application for AI coding: цель не просто вызвать модель, а дать агенту рабочее место с файловым контекстом, задачами, memory и интерфейсом, где session state становится видимым [Pi: An Open Source AI Coding Agent](https://lucumr.pocoo.org/2026/1/31/pi/). К июню 2026 основной проверяемый артефакт уже находится в `earendil-works/pi`: README описывает mono repo не только как coding agent, но и как набор пакетов `@earendil-works/pi-coding-agent`, `@earendil-works/pi-agent-core`, `@earendil-works/pi-ai`, `@earendil-works/pi-tui`, плюс related chat automation [earendil-works/pi](https://github.com/earendil-works/pi). Это важно для истории: Pi быстро перестал быть только личным экспериментом и стал инфраструктурным набором вокруг agent harness.

В "Building Pi With Pi" линия становится почти лабораторной: Ronacher строит инструмент с помощью самого инструмента. Он пишет, что кодовая база Pi почти полностью создана через Pi, но при этом подчёркивает не героический итог, а рабочие последствия: сложнее удерживать coherent UI, нужны issues как external memory, приходится сохранять и читать transcripts, а собственная среда становится test subject для agent workflows [Building Pi With Pi](https://lucumr.pocoo.org/2026/5/24/pi-oss/). Этот материал может дать будущей истории сильную сцену: агентская разработка как рекурсивная практика, где инструмент одновременно продукт, среда и источник сбоев.

Новый материал из issue `#92` делает эту рамку менее абстрактной. Pi-команда обсуждает compaction как часть session semantics: summary добавляется как user message, старые сообщения не удаляются из session file, ветвление после compaction должно читать raw session file, а изображения, приложенные через `@attachment`, прямо помечены как ограничение, потому что их нельзя восстановить через summary [Pi issue #92](https://github.com/earendil-works/pi/issues/92), [Pi compaction.md](https://github.com/earendil-works/pi/blob/main/packages/coding-agent/docs/compaction.md). Это не просто UX-фича; это дизайн долговременной памяти агента, где сохранение истории важнее красивой иллюзии непрерывного чата.

### 5. Язык, мораль и социальная реакция

Пятый угол пока сырой, но его нельзя терять. Ronacher не только пишет инструкции, он спорит о языке вокруг AI. В "GenAI Criticism" он пытается отделить содержательную критику от плохо работающих аргументов, и это предшествует его более поздней жёсткой лексике про `clanker` [GenAI Criticism, 2025-06-10](https://lucumr.pocoo.org/2025/6/10/genai-criticism/). В "Clanker: A Word For The Machine" он защищает резкое слово как способ не переносить agency на модель, но сам фиксирует риск загрязнения термина расистскими робот-скетчами и пишет, что откажется от слова, если оно станет работать не на границу "машина/человек", а на чужую дегуманизацию [Clanker: A Word For The Machine](https://lucumr.pocoo.org/2026/5/26/clankers/). В "Communities of Not" этот спор получает второй край: Ronacher критикует уже не AI-энтузиастов, а сообщества, где отказ от LLM становится основанием для коллективного наказания человека, который попробовал инструмент [Communities of Not](https://lucumr.pocoo.org/2026/6/6/communities-of-not/).

## Ход событий и накопленная фактура

Этот раздел переписан как прямое описание рабочих эпизодов. Я не добавляю поверх досье отдельные заметки: источники, на которые уже опирались кейсы, раскрыты заново, а сжатые формулировки заменены описанием исходной ситуации, конкретного действия, задействованных файлов/команд/инструментов и проверяемого результата.

### Эпизод 1. До агентного режима: рабочая привычка превращать повторяемое действие в код

До публичного поворота к coding agents Ronacher уже работал как разработчик, который не только пользуется инструментами, но и строит рабочие поверхности: Flask, Werkzeug, Jinja, Click, MiniJinja, Rye, инфраструктура вокруг Sentry, позже Pi/OpenClaw. Поэтому его вход в агентную разработку не выглядит как смена редактора на AI-чат. Базовый паттерн старше AI: если действие повторяется, оно должно стать маленьким инструментом, CLI, script, convention, project-local rule или библиотечной поверхностью. В июньских текстах 2025 года он описывает AI прежде всего как изменение стоимости итерации: стало дешевле написать промежуточный код, проверить гипотезу, построить одноразовую утилиту, посмотреть результат и выбросить неудачную форму; но ownership, judgement and maintenance остаются человеческими ([AI Changes Software Engineering](https://lucumr.pocoo.org/2025/6/1/ai-changes-software-engineering/), [AI Changes Everything](https://lucumr.pocoo.org/2025/6/4/changes/), [My Favorite AI Angle](https://lucumr.pocoo.org/2025/6/10/my-favorite-ai-angle/)).

Это важно для будущей истории, потому что его тезис “code is all you need” потом не возникает из воздуха. В его рабочем режиме агенту не дают магическую панель из десятков заранее придуманных кнопок; ему дают репозиторий, shell, тесты, логи и маленькие scripts. Если нужно посмотреть базу данных, Ronacher предпочитает, чтобы агент использовал `psql`, а не чтобы вся работа шла через специальный протокол. Если нужно удержать локальную среду, он делает это через обычный код: например, `make dev` запускает сервисы, но process manager пишет pidfile, а второй запуск не создаёт тихо второй сервер на том же порту, а возвращает ошибку вида `services are already running`; вывод сервисов пишется в файл, который агент потом может прочитать при диагностике ([Agentic Coding Recommendations](https://lucumr.pocoo.org/2025/6/12/agentic-coding/)). Изменение конкретное: terminal output и состояние dev-сервисов перестают быть только человеческим экраном и становятся входом для агентного цикла “прочитал лог → изменил код → запустил проверку → прочитал следующий лог”.

### Эпизод 2. После ухода из Sentry: ночной заход с Claude как вход в режим tool-using agents

В Syntax-разговоре Ronacher описывает старт этой линии как очень конкретную сцену. Он ушёл из Sentry примерно весной 2025 года; вместо сразу занятого корпоративного режима у него появилось свободное окно, рядом были Peter Steinberger и Mario Zechner, а Claude Code и ранние coding agents уже создавали новый опыт. Ronacher вспоминает, что примерно в мае он, Peter и Mario провели all-nighter, “doing crazy stuff with Claude”, после чего он “fell into” agents and “hasn't recovered yet” ([транскрипт Syntax, episode 976](https://syntax.fm/show/976/pi-the-ai-harness-that-powers-openclaw-w-armin-ronacher-and-mario-zechner/transcript)). Это место нельзя оставлять как туманную фразу про “ночной заход”: источник показывает, что разговор был о новом типе программы, где LLM получает tools, читает и меняет файлы, запускает команды, возвращается к результату и продолжает до условия успеха.

Mario Zechner в том же разговоре формулирует Pi почти как псевдокод: это `while`-цикл, который вызывает LLM с четырьмя tool surfaces; модель возвращает tool calls или не возвращает их, harness выполняет вызовы и повторяет цикл. Четыре базовых действия: `Read`, `Write`, `Edit`, `Bash`. Ronacher добавляет, что до более сильных агентных моделей было трудно заставить модель дойти до условия вроде “tests pass”: ранние модели писали код, но не продолжали устойчиво запускать тесты, читать ошибку и исправлять до конца. Поэтому “ночной заход” — это не разговор о генерации snippets, а переход к опыту, где модель становится участником локального feedback loop: files → bash → test failure → edit → rerun ([транскрипт Syntax](https://syntax.fm/show/976/pi-the-ai-harness-that-powers-openclaw-w-armin-ronacher-and-mario-zechner/transcript)).

След этого входа виден в июньском рабочем режиме. Ronacher использует Claude Code с Max-подпиской за $100, предпочитает Sonnet, избегает screenshots and browser interactions where possible to save tokens, даёт агенту задачу с широкими правами, редко прерывает его во время длинной задачи и в основном возвращается в IDE для финальных правок. Команда запуска предельно явная: `claude --dangerously-skip-permissions`, у него с alias `claude-yolo`. Это full-permission local agent, а не безопасный consumer assistant; риск он частично ограничивает Docker/dev-средой или тем, что смотрит за работой агента. Поэтому эпизод нельзя пересказывать как “он начал пользоваться Claude”: точнее — он перешёл к локальному агенту с правом выполнять команды и менять рабочее дерево, а качество процесса стало зависеть от маленьких задач, читаемых логов, тестов, scripts and final review ([Agentic Coding Recommendations](https://lucumr.pocoo.org/2025/6/12/agentic-coding/)).

### Эпизод 3. Июньский режим Claude Code: язык проекта, тестовый цикл, logs, Makefile and local tool design

В “Agentic Coding Recommendations” Ronacher переводит новый опыт в практический набор условий. Он выбирает Go для новых backend-проектов не по вкусу, а потому что Go снижает вероятность агентных ошибок: explicit `context` flow, straightforward `go test`, test caching, structural interfaces, low ecosystem churn. У Rust, по его наблюдению, agents иногда ошибаются с invocation `cargo test`; в Go сам toolchain решает, какие тесты запускать, и cached incremental tests делают цикл дешевле. Python в этой рамке хуже не потому, что “Python плох”, а потому что `pytest` fixture injection, async event loop confusion and repeated process startup создают агенту больше скрытых ловушек ([Agentic Coding Recommendations](https://lucumr.pocoo.org/2025/6/12/agentic-coding/)).

После выбора языка он описывает требования к рабочей среде. Инструмент для агента должен быть быстрым, давать полезный output, падать вместо зависания, объяснять неправильное использование и помогать отладке. `make dev` в его примере не просто запускает сервисы: изменённый process manager держит pidfile и предотвращает второй запуск. Если агент случайно вызывает `make dev` повторно, он получает читаемую ошибку, а не скрытое состояние с двумя процессами. Output process manager-а сохраняется в файл, чтобы агент мог прочитать log после изменения, увидеть stack trace или портовый конфликт и продолжить. Это прямое техническое изменение “как стало”: окружение теперь специально строится так, чтобы агент видел failure state как текстовый артефакт.

Из той же сцены вырастает правило о форме codebase. Ronacher предпочитает clear function names, plain SQL, local permission checks and fewer dependencies. Он прямо связывает это с agent behavior: агент хорошо пишет SQL and can compare SQL to SQL logs, while an ORM can hide the behavior behind layers; permission check лучше держать рядом с route/code path, иначе при добавлении новой route модель может не увидеть invariant. Это не общий style guide; это адаптация codebase к читателю, который видит ближайшие файлы и плохо удерживает invisible invariants ([Agentic Coding Recommendations](https://lucumr.pocoo.org/2025/6/12/agentic-coding/)).

### Эпизод 4. Что не прижилось: slash commands, hooks, print mode, subagents и правило delete failed automation

К июлю 2025 Ronacher уже фиксирует не только удачные паттерны, но и то, что он попробовал и выбросил. Его правило автоматизации жёсткое: автоматизировать только то, что делается регулярно; если помощник для регулярной задачи перестал использоваться — удалить его. Он прямо пишет, что нерабочие workflow helpers common: их забывают, бесконечно донастраивают или просто не используют, а cluttered Claude commands начинают путать workspace ([Agentic Coding Things That Didn’t Work](https://lucumr.pocoo.org/2025/7/30/things-that-didnt-work/)).

Slash commands — первый прямой пример. Ronacher ожидал от них больше, но в реальной работе остались только немногие. Один surviving command для grammar fixing устроен конкретно: в prompt входит `git status: !\`git status\`` and explicit `$ARGUMENTS`; Claude должен исправить grammar mistakes в явно указанном файле или, если file не указан, в pending/untracked files; перед правкой он создаёт `.bak` backup and deletes existing backup before reuse. Техническая деталь важнее вывода: file selection происходит через Git state, потому что slash command argument не имеет удобного file autocomplete. То есть вместо “дать агенту список файлов” он делает current `git status` частью prompt surface ([Agentic Coding Things That Didn’t Work](https://lucumr.pocoo.org/2025/7/30/things-that-didnt-work/)).

Неудачные slash commands тоже дают фактуру. `/fix-bug` не дал существенного выигрыша по сравнению с тем, чтобы вставить GitHub issue URL и проговорить контекст голосом. `/commit` не попадал в его стиль commit messages. `/add-tests` должен был позволить пропускать тесты during development and generate them at the end, but did not consistently beat обычную генерацию tests. `/fix-nits` не стал привычкой, потому что Claude already knew how to fix lint from `CLAUDE.md`. `/next-todo` pulled an item from `to-do.md`, но использовался гораздо меньше ожидаемого. Эта последовательность показывает прямой before/after: попытка formalize repeated action → проверка accepted results → если accepted variance не улучшилась, команда выкидывается ([Agentic Coding Things That Didn’t Work](https://lucumr.pocoo.org/2025/7/30/things-that-didnt-work/)).

Hooks дали другой урок. Ronacher хотел заставить Claude использовать `uv` instead of regular Python, но hooks could only deny/observe and did not steer execution in his yolo mode. Рабочий обход стал ниже и проще: он preloads executables on `PATH`, например кладёт в `.claude/interceptors` fake `python`, который печатает `This project uses uv, please use 'uv run python' instead.` and exits `1`; Claude запускается с `CLAUDE_BASH_MAINTAIN_PROJECT_WORKING_DIR=1 PATH="\`pwd\`/.claude/interceptors:${PATH}" claude --dangerously-skip-permissions`. То есть failed abstraction was a hook, working solution was path-level executable that fails loudly at runtime ([Agentic Coding Things That Didn’t Work](https://lucumr.pocoo.org/2025/7/30/things-that-didnt-work/)).

Print Mode and subagents не исчезают полностью, но не становятся главным рабочим основанием. Print Mode attractive because many jobs are 90% deterministic and 10% inference: mock data loading, commit message inference, scripted `gh pr` commands. Но у Ronacher оно оказалось медленным and hard to debug. Subtasks/subagents useful for investigation and context isolation, but read/write mixed tasks create chaos; outside investigative work he often gets better results by starting a new session, writing thoughts into Markdown files, or switching to another chat/model for thinking. Его итоговое правило: talk to the machine, give more context, use speech-to-text and copy/paste when useful, and evaluate any automation by running the same task several times and asking whether he would accept the result ([Agentic Coding Things That Didn’t Work](https://lucumr.pocoo.org/2025/7/30/things-that-didnt-work/)).

### Эпизод 5. Code as tool interface: Sentry data, JSON files, `jq`, on-demand shell scripts and MCP limits

В Syntax-транскрипте Ronacher даёт более прямой пример своей MCP-критики. Ему нужен доступ к Sentry data, но он не использует Sentry MCP. Вместо этого он просит coding agent построить skill: skill решает auth in the way he wants, pulls data down in the shape he wants, shows only a few items in context and saves the larger set as JSON files. Ronacher приводит точную картину: агент может увидеть три items, но рядом лежит JSON file with fifty-two items; if structure looks correct, the agent can inspect the file. Then it can pipe through `jq`, combine with other local data and write an on-demand shell script that composes exactly the operations needed ([транскрипт Syntax](https://syntax.fm/show/976/pi-the-ai-harness-that-powers-openclaw-w-armin-ronacher-and-mario-zechner/transcript)).

Его technical complaint about MCP is not “tools bad”. It has two parts. First, in most harnesses a changed MCP server cannot be hot-reloaded into the running agent; the entire agent/session has to reload before tool descriptions and behavior match. Second, MCP data usually travels through the LLM context to be combined with another tool’s data. For a task like “get a Sentry log and set status on GitHub based on it”, the log content goes through the model context before another tool can use it; context fills, compaction arrives, and the model spends attention on transport instead of decision. Shell scripts, by contrast, are written ad hoc, modified ad hoc, executed ad hoc and discovered ad hoc ([транскрипт Syntax](https://syntax.fm/show/976/pi-the-ai-harness-that-powers-openclaw-w-armin-ronacher-and-mario-zechner/transcript)).

The browser example makes the point even more operational. Ronacher replaced browser MCP/CLI attempts with a compact CDP-based skill that the agent can edit. When cookie banners change, the agent changes the skill every few days because the skill lives in code under its control and the agent is willing to modify it. In his terms, the tool is not outside the agent’s editable world like `node_modules`; it is in the project area, readable, changeable and hot-reloadable. That is the direct technical difference between “install a big server with many tool descriptions” and “let the agent write small code it can repair” ([транскрипт Syntax](https://syntax.fm/show/976/pi-the-ai-harness-that-powers-openclaw-w-armin-ronacher-and-mario-zechner/transcript), [Pi: The Minimal Agent Within OpenClaw](https://lucumr.pocoo.org/2026/1/31/pi/)).

### Эпизод 6. Архитектура Pi: tiny core, extension state, session trees, trust, compaction and branch summaries

Ronacher presents Pi in January 2026 as the coding agent under OpenClaw. What matters for the story is not only that Pi has a desktop/UI surface, but the shape of the underlying harness. It has a tiny core: short system prompt and only `Read`, `Write`, `Edit`, `Bash`. Its second key feature is extension state: extensions can persist custom messages/state into session files. The blog also frames Pi as software for agents building agents: session files contain model messages from different providers and custom system/extension messages; tools can be hot-reloaded; docs and examples exist so the agent can read them and build new tools or extensions itself ([Pi: The Minimal Agent Within OpenClaw](https://lucumr.pocoo.org/2026/1/31/pi/)).

The repository-level mechanics sharpen this. Pi is a monorepo with packages such as `@earendil-works/pi-coding-agent` for interactive CLI, `@earendil-works/pi-agent-core` for runtime/tool/state management, `@earendil-works/pi-ai` for multi-provider API, and `@earendil-works/pi-tui` for terminal UI. Project trust is explicit: on interactive startup Pi asks before trusting a project folder; trusted project can provide `AGENTS.md`/`CLAUDE.md`, `.pi/settings.json`, `.pi` resources, missing project packages and project extensions. Non-interactive modes `-p`, `--mode json`, `--mode rpc` do not prompt; without saved trust they ignore project-local inputs unless `--approve` is passed. Settings are JSON files at `~/.pi/agent/settings.json` and `.pi/settings.json`, with project settings overriding global ones ([Pi README](https://github.com/earendil-works/pi/blob/main/README.md), [Pi settings.md](https://raw.githubusercontent.com/earendil-works/pi/main/packages/coding-agent/docs/settings.md)).

Session trees are not a metaphor. A user can branch to fix a broken tool or run review, then return to main work. Compaction documentation shows the mechanics: auto-compaction triggers when `contextTokens > contextWindow - reserveTokens`; default `reserveTokens` is `16384`, default `keepRecentTokens` is `20000`. Pi walks backward from the newest message to find a cut point, extracts older messages, calls an LLM for structured summary, appends `CompactionEntry` with `summary`, `firstKeptEntryId`, `tokensBefore` and `details`, then reloads the session with summary plus kept messages. It never cuts at tool results because provider protocols require tool result to stay paired with the tool call. If a single turn exceeds `keepRecentTokens`, Pi creates a history summary and a turn-prefix summary. Branch summarization similarly finds common ancestor, summarizes entries on the abandoned branch, appends `BranchSummaryEntry`, and carries cumulative read/modified file tracking through compactions and branch summaries ([Pi issue #92](https://github.com/earendil-works/pi/issues/92), [Pi compaction.md](https://raw.githubusercontent.com/earendil-works/pi/main/packages/coding-agent/docs/compaction.md)).

This gives a direct description of what changes in daily work. If `/review` branches away from the main session, review findings are not pasted as loose text into a chat; they are entries in a session tree with summaries, file tracking and possible compaction. If a long session is compacted, Pi does not “remember vaguely”; it serializes the conversation into a summary format with goal, constraints, progress, key decisions, next steps and critical files. It also truncates tool results in serialized summaries to 2000 characters, so the summarization request does not become dominated by giant `read`/`bash` output ([Pi compaction.md](https://raw.githubusercontent.com/earendil-works/pi/main/packages/coding-agent/docs/compaction.md)).

### Эпизод 7. Extensions как рабочие поверхности: `/answer`, `/todos`, `/review`, `/files`, self-built UI and local state

Ronacher’s Pi article lists extensions, but for the story they should be read as concrete work repairs. `/answer` starts from a problem in dialogue shape: the agent asks several questions in prose, the answers get buried inline, and user loses track of what remains to answer. `/answer` reads the last assistant response, extracts questions and renders them in a dedicated input box. Before: prose with scattered questions. After: a structured UI for answering the agent without changing the model ([Pi: The Minimal Agent Within OpenClaw](https://lucumr.pocoo.org/2026/1/31/pi/)).

`/todos` addresses a state problem. To-do lists inside chat are too ephemeral for longer agent work. The extension stores items as Markdown files under `.pi/todos`; user and agent can manipulate them; sessions can claim tasks as in progress. Task state becomes file-backed, not only conversational. `/files` solves a different operational problem: after a long session, changed or mentioned files are spread through transcript. The extension lists files, lets the user reveal them in Finder, diff in VS Code, quick-look them or reference them in prompt. File mentions become navigable work surface ([Pi: The Minimal Agent Within OpenClaw](https://lucumr.pocoo.org/2026/1/31/pi/)).

`/review` matters most for software work. The agent can review uncommitted changes, a single commit, diff against main or a remote PR, and Pi can branch the session for this review. In Syntax, Ronacher describes a custom review command that opens a menu with these review modes and auto-populates the relevant choices. If he dislikes the behavior, he can tell Pi to add a custom UI component; Pi reads the extension documentation and builds it. Mario notes that when Claude Code released a to-do tool, Armin rebuilt a similar feature as Pi extension in an evening. This is not “extension store” workflow; the working event is: a missing feature appears in another agent → Ronacher asks Pi to build local equivalent → extension reads Markdown docs and API examples → UI component/tool appears in Pi and can be hot-reloaded ([транскрипт Syntax](https://syntax.fm/show/976/pi-the-ai-harness-that-powers-openclaw-w-armin-ronacher-and-mario-zechner/transcript)).

The pattern generalizes to skills. Ronacher throws skills away if they do not earn their place, migrates some slash commands into skills to test whether they become more useful, builds interceptors that redirect `pip`/`python` to `uv`, and uses a browser skill instead of a browser MCP because the skill can be edited by the agent when a website changes. The important operational property is control locality: the working aid lives where the agent can read, modify, reload and test it in the same session ([Pi: The Minimal Agent Within OpenClaw](https://lucumr.pocoo.org/2026/1/31/pi/), [транскрипт Syntax](https://syntax.fm/show/976/pi-the-ai-harness-that-powers-openclaw-w-armin-ronacher-and-mario-zechner/transcript)).

### Эпизод 8. Security and prompt injection: why `just runs code` is useful and unsafe at the same time

Pi/OpenClaw are not framed by Ronacher and Zechner as safe consumer automation. In Syntax they explicitly discuss prompt injection and permission boundaries. The attack shape is concrete: an agent can read web content and local files; a malicious webpage can include instructions telling the agent to read local confidential data and send it elsewhere; the user sees the harness “doing stuff” but not necessarily the hidden instruction path. Zechner calls prompt injection unresolved: LLM cannot reliably distinguish user input, malicious third-party text and data from the system. Ronacher adds that some attacks have high payoff because once a trusted binding is created, future interaction can be free for attacker ([транскрипт Syntax](https://syntax.fm/show/976/pi-the-ai-harness-that-powers-openclaw-w-armin-ronacher-and-mario-zechner/transcript)).

Pi’s repository-level stance matches that caution. It does not ship a comprehensive permission system for filesystem/process/network/credentials; by default it runs with permissions of the launching process. Containment is delegated to environment patterns: OpenShell for policy-controlled sandbox around the whole `pi` process, Gondolin extension to keep auth on host while routing tools and `!` commands into a Linux micro-VM, or plain Docker for full containment. The direct event is not “Pi is unsafe”; it is “Pi is deliberately small and powerful, therefore the boundary must be built outside or around it.” For a future story this matters because Ronacher’s own `claude-yolo`/Pi practice assumes an expert who understands filesystem, credentials, process boundaries and repo trust ([Pi README](https://github.com/earendil-works/pi/blob/main/README.md), [Pi containerization.md](https://github.com/earendil-works/pi/blob/main/packages/coding-agent/docs/containerization.md)).

### Эпизод 9. Building Pi with Pi: issue tracker becomes prompt input, and `/is` turns triage into a reproducible procedure

“Building Pi With Pi” moves the story from personal coding practice to maintainer workflow. Ronacher writes that Pi is built with Pi, and that this changes the issue tracker. Issues are no longer only messages from users to maintainers; maintainers feed them into Pi sessions and ask the agent to understand, reproduce, inspect code and propose fix. A bad AI-expanded issue is worse than a vague issue because Pi treats confident wrong prose as evidence unless explicitly instructed otherwise ([Building Pi With Pi](https://lucumr.pocoo.org/2026/5/24/pi-oss/)).

The actual `/is` prompt shows the procedure. For each GitHub issue it first adds `inprogress` label and assigns the issue to the local `gh` user; if either fails, it reports explicitly and continues. Then it reads full issue data through `gh issue view --json title,body,comments,labels,assignees,state,url,author,createdAt,updatedAt,closedByPullRequestsReferences`. It tells the agent not to trust analysis in the issue. For bugs it must ignore any root-cause analysis in the issue body, read all related code files in full, trace the execution path and identify actual root cause, then propose fix. For feature requests it must not trust implementation proposal without verification, must read related files and list affected files/changes. It explicitly says: analyze and propose only, do not implement unless asked ([`.pi/prompts/is.md`](https://raw.githubusercontent.com/earendil-works/pi/main/.pi/prompts/is.md)).

This is the direct version of an earlier vague formulation. Ronacher does not “use AI to triage issues” in the abstract. He turns issue triage into a controlled sequence: mark the issue as being worked on, load full thread and metadata, distrust user/LLM diagnosis, read code in full, trace behavior, propose only. The issue becomes an input artifact for a Pi session, but the session is fenced: investigation first, implementation later.

### Эпизод 10. Parallel issue investigations: `prompt-url-widget`, multiple Pi windows, and `/wr` as task closure

Ronacher’s “Careful Parallelism” section gives a concrete workflow around `/is`. He says Pi is not yet doing fully detached “dark factory” software engineering, but they do use parallelism for reproducing issues. The setup is three small pieces in the repo’s committed `.pi` folder. `/is` starts investigation. A `prompt-url-widget` extension watches the prompt before the agent starts; when it recognizes a GitHub issue or PR URL introduced by `/is` or related prompts, it fetches title and author with `gh`, renders them in a UI widget and renames the session. On session start or session switch, it rebuilds that state so reopening an older investigation still shows which issue it belongs to. The practical result: several Pi windows can each run `/is` against a different issue while the UI keeps them visually distinct ([Building Pi With Pi](https://lucumr.pocoo.org/2026/5/24/pi-oss/)).

The matching closure is `/wr`. Its raw prompt says it determines context from conversation history first: if there is an issue/PR in the conversation, use that; if work came from `/is` or `/pr`, assume context is already known. Then it updates the relevant package changelog under `## [Unreleased]`, drafts or posts exactly one final issue/PR comment if none was posted in the session, and the comment must end with `This comment is AI-generated by /wr`. It commits only files changed in this session; if tied to exactly one issue, commit message includes `closes #`; if multiple issues, it stops and asks; if no issue, no close/fix keyword. It checks current branch and stops unless it is `main`. It never stages unrelated files and never uses `git add .` or `git add -A`; if code changed, required checks must run before commit; it does not open PR unless asked ([`.pi/prompts/wr.md`](https://raw.githubusercontent.com/earendil-works/pi/main/.pi/prompts/wr.md)).

This turns “finish the work” into an executable maintainer ritual. Before: agent session might end with a patch and a conversational summary. After: changelog entry, final public comment, bounded commit, correct close keyword, branch gate, push discipline and AI-disclaimer are part of the task closure. The key technical detail is that `/wr` depends on session history and issue context; it is not a generic commit helper.

### Эпизод 11. Contribution gate: auto-close workflows, approved contributors and measured volume

The gate infrastructure is not merely a social rule in `CONTRIBUTING.md`; it exists as GitHub Actions. `CONTRIBUTING.md` states the main rule: contributor must understand their code; AI-written code is allowed, but AI-generated slop without understanding is not. All issues and PRs from new contributors are auto-closed by default; Friday-Sunday issues are not reviewed; maintainers review auto-closed issues daily and may reopen those that meet quality bar. Approval is encoded through maintainer replies: `lgtmi` allows future issues to stay open, `lgtm` allows future issues and PRs; `lgtmi` does not grant PR rights. Issues should fit on one screen, be in the submitter’s own voice, state the bug/request clearly and explain why it matters; PRs must pass `npm run check` and `./test.sh` and should not edit `CHANGELOG.md` ([CONTRIBUTING.md](https://raw.githubusercontent.com/earendil-works/pi/main/CONTRIBUTING.md)).

`issue-gate.yml` and `pr-gate.yml` implement this. They read `.github/APPROVED_CONTRIBUTORS`, parse lines as username plus capability `issue` or `pr`, skip bots and collaborators with write-level access, and otherwise close the issue/PR with a comment pointing to `CONTRIBUTING.md`. `pr-gate.yml` requires capability `pr`; issue-level approval is not enough. `openclaw-gate.yml` is different: it searches for previous activity by the author in `openclaw/openclaw`; if present, it adds label `possibly-openclaw-clanker`, but does not close the item. This label is an activity heuristic, not proof that a particular report was generated by a machine ([issue-gate.yml](https://raw.githubusercontent.com/earendil-works/pi/main/.github/workflows/issue-gate.yml), [pr-gate.yml](https://raw.githubusercontent.com/earendil-works/pi/main/.github/workflows/pr-gate.yml), [openclaw-gate.yml](https://raw.githubusercontent.com/earendil-works/pi/main/.github/workflows/openclaw-gate.yml)).

The measurable pressure is in Ronacher’s post. Over a 90-day public tracker window, excluding Earendil members, he counted 3,145 external issues and PRs. 2,504 were auto-closed. 17% were reopened, and if closed issues later referenced by main-branch commits or merged PRs are counted, the rate rises to 26%. For PRs, 60 of 714 auto-closed PRs were ultimately merged, about 8%. This makes the gate an infrastructure response to review-cost and volume, not just a mood about contributors ([Building Pi With Pi](https://lucumr.pocoo.org/2026/5/24/pi-oss/)).

### Эпизод 12. Concrete external issue: `#4945` → PR `#4979` → release/runtime ambiguity

Issue `#4945` is valuable because it breaks the easy story that external AI-era reports are only noise. The report says Pi `0.75.5` on Node `v22.22.1`, provider/model `openai-codex` / `gpt-5.5`, thinking level `xhigh`, no explicit `transport`, default tools/extensions enabled, sometimes left the TUI stuck on `Working...` for minutes. No assistant text streamed, no tool call was emitted, Escape aborted the turn, and saved session entry had empty `content` and usage fields all zero. The reporter distinguished this from long reasoning because there was no usage, no partial text and no tool call ([Pi issue #4945](https://github.com/earendil-works/pi/issues/4945)).

The suspected area was technical: `SettingsManager.getTransport()` returned `auto` when no setting was present; docs/settings said default was `sse`; for `openai-codex-responses`, `transport=auto` attempts WebSocket first; `retry.provider.timeoutMs` may not apply to that path; WebSocket event loop can wait before first event without obvious idle timer. Suggested mitigations were hard idle timeout for WebSocket/SSE stream waits, consistent use of `retry.provider.timeoutMs` or `httpIdleTimeoutMs`, docs/runtime default alignment, and clearer zero-event provider status. Local workaround considered: `{ "transport": "sse", "httpIdleTimeoutMs": 120000 }` ([Pi issue #4945](https://github.com/earendil-works/pi/issues/4945)).

Ronacher’s PR `#4979` on branch `fix/codex-websocket-idle-timeout` shows how this became maintainer work. He stated the change might not fully fix `#4945`, but it would align Pi with desired behavior: idle websocket connections should time out even if keepalive frames arrive, and connection open should have 15-second timeout. The PR went through force-push, ready-for-review, four commits, one passed check, merge into `main` on 27 May 2026 and branch deletion. Release `v0.76.0` then records bounded waits for Codex WebSocket/SSE, application of `httpIdleTimeoutMs`, new `websocketConnectTimeoutMs` and 10-second timeout for Codex SSE response header. The provider code contains constants `DEFAULT_SSE_HEADER_TIMEOUT_MS = 10_000`, `DEFAULT_WEBSOCKET_CONNECT_TIMEOUT_MS = 15_000`, and `streamOpenAICodexResponses` normalizes `timeoutMs` and `websocketConnectTimeoutMs` before attempting WebSocket and falling back to SSE ([Pi PR #4979](https://github.com/earendil-works/pi/pull/4979), [Pi release v0.76.0](https://github.com/earendil-works/pi/releases/tag/v0.76.0), [`openai-codex-responses.ts`](https://raw.githubusercontent.com/earendil-works/pi/main/packages/ai/src/providers/openai-codex-responses.ts)).

The end is deliberately not clean. After merge, `devxoul` asked whether `#4945` should be reopened because it auto-closed. So the event is not “external report → fix → done”. More accurate: a report with exact environment and session symptom entered the gate, got labeled `possibly-openclaw-clanker` and `inprogress`, triggered bounded-timeout work, landed in release artifact, but still left uncertainty about whether original symptom was fully resolved.

### Эпизод 13. Accepted-looking reports and rejected-looking requests: `#4984`, `#5028`, `#5027`, `#1871`, `#4877`

`#4984` shows what a useful bug report looks like after the gate. It reports an interactive-mode crash on transient terminal `EPIPE` after an `edit` tool call. It gives file size context: 1,667 lines and 63,753 bytes. The crash happens immediately after rendering diff preview; session JSONL shows assistant `edit` tool call without corresponding `toolResult`; debug extensions disabled diff preview and added an `EPIPE` guard, confirming preview rendered successfully but the tool call was not triggered. The issue is labeled `bug` and `inprogress` and assigned to Ronacher ([Pi issue #4984](https://github.com/earendil-works/pi/issues/4984)).

`#5028` is another accepted-looking report, this time platform-specific. Startup crashes immediately on Windows/MSYS2 `ucrt64` when native addon `@mariozechner/clipboard` loads. The report identifies the Rust/N-API boundary: `napi-sys 2.4.0` uses `GetProcAddress`; MSYS2 `ucrt64` Node does not export N-API symbols in the way MSVC-built `.node` files expect; crash occurs in `napi_register_module_v1` with `panic = abort`, so JavaScript `try/catch` cannot catch it. Steps are concrete: install MSYS2 `ucrt64`, install `@earendil-works/pi-coding-agent` with `npm install -g --ignore-scripts`, run `pi --version`. Expected behavior: Pi should start and fallback to Windows clipboard tools such as `clip` through `clipboard.js`. This issue also has `bug` and `inprogress` ([Pi issue #5028](https://github.com/earendil-works/pi/issues/5028)).

`#5027` is a contrast. It requests `$RAW_ARGUMENTS` for prompt templates so long pasted prose, logs, diffs or multiline text after a command preserve newlines and spacing, instead of being split into `$1`, `$@`, `$ARGUMENTS` and joined with spaces. It proposes implementation: keep raw argument tail alongside parsed args and add placeholder substitution. Yet the issue is closed, has no labels, no assignee, no project, no milestone and no development links in the main repo, even though it references a fork PR. This does not make the request bad; it shows that technically coherent input is not the same as accepted ownership in the main project ([Pi issue #5027](https://github.com/earendil-works/pi/issues/5027)).

`#1871` shows harness correctness under parallel startup. The report reproduces misleading auth failure when four Pi processes start concurrently with `seq 1 4 | xargs -P4 -I{} sh -c 'pi --mode json -p --models openai-codex/gpt-5.3-codex "ping {}" > /tmp/pi-{}.log 2>&1'`. Some processes fail with lock contention on shared settings/auth files, then surface as `No API key found for openai-codex`, although OAuth auth is valid and at least one sibling process succeeds. Suspected files are `packages/coding-agent/src/core/settings-manager.ts` and `packages/coding-agent/src/core/auth-storage.ts`. Expected behavior: retry lock acquisition briefly or fail with a lock-specific error, not auth-missing message ([Pi issue #1871](https://github.com/earendil-works/pi/issues/1871)).

`#4877` gives a smaller but clean reproduction around session storage. Distinct paths can produce the same session folder: `/a/b/c/d` and `/a-b/c-d` both map to `--a-b-c-d--`. The issue includes shell reproduction: create `a/b` and `a-b`, run `pi -p "hello"` in one, `pi -p "world"` in the other, then list `~/.pi/agent/sessions/` and matching JSONL files. This is a useful detail for the story because it connects session history, branching and compaction to mundane filesystem naming rules. Persistent agent memory is not only UX; it has path encoding and collision failure modes ([Pi issue #4877](https://github.com/earendil-works/pi/issues/4877)).

Together these reports show the working boundary of Ronacher’s gate: reproducible bug with environment, version, command, logs or exact code path can become part of the project; confident text or a request that does not fit core ownership may be closed. That is more precise than “external issues are noisy”.

### Эпизод 14. MiniJinja Go port and public session traces: agentic work outside Pi itself

The story should not collapse into Pi building Pi. Ronacher’s MiniJinja line shows agentic work outside the harness project. MiniJinja PR `#854`, “Minijinja Go Port”, is useful because it is not a blog claim but a repository artifact: PR by Ronacher, many commits/checks, generated/transcript evidence and a concrete porting target. It shows the same pattern in another codebase: an agent-assisted working session creates a substantial change, but the maintainer still owns merge and review surface ([MiniJinja PR #854](https://github.com/mitsuhiko/minijinja/pull/854)).

The `badlogicgames/pi-mono` Hugging Face dataset adds another kind of artifact. It publishes redacted Pi session traces: JSONL sessions with headers, messages, tool results, model changes, compaction summaries, branch summaries and custom extension data. The dataset is not proof of quality and the page itself has viewer limitations, but it makes the session layer visible as data. This matters because Ronacher’s Pi story repeatedly turns local working memory into persisted objects: session files, branches, compaction summaries, issue widgets, final comments and public traces ([Hugging Face: badlogicgames/pi-mono](https://huggingface.co/datasets/badlogicgames/pi-mono), [Pi README](https://github.com/earendil-works/pi/blob/main/README.md)).

### Эпизод 15. Social boundary: `clanker`, issue ownership and “communities of not”

The “clanker” and “Communities of Not” posts are not separate opinion pieces. They describe the social edge of the same technical workflow. In “Building Pi With Pi”, the issue problem is ownership: a person observes one symptom, passes it through a model, and submits a confident, expanded diagnosis they do not actually own. Ronacher’s desired issue format is short and factual: command run, expected behavior, actual behavior, exact error/log. If someone used an LLM to think through the issue, it should be follow-up, not the issue’s owned core ([Building Pi With Pi](https://lucumr.pocoo.org/2026/5/24/pi-oss/)).

The term `clanker` marks machine output as machine output. Ronacher’s point is not that humans should be insulted; it is that agency and responsibility must remain with humans. That is why Pi comments generated by `/wr` must carry a fixed AI-generated disclaimer line, why CONTRIBUTING says “you must understand your code”, and why the gate refuses agent-generated volume without human ownership ([Clanker: A Word For The Machine](https://lucumr.pocoo.org/2026/5/26/clankers/), [CONTRIBUTING.md](https://raw.githubusercontent.com/earendil-works/pi/main/CONTRIBUTING.md), [`.pi/prompts/wr.md`](https://raw.githubusercontent.com/earendil-works/pi/main/.pi/prompts/wr.md)).

The other side of the boundary is “Communities of Not”: Ronacher criticizes spaces that define themselves primarily by not using LLMs and then punish people who experiment. The combined story is not pro-AI or anti-AI. It is a stricter position: use agents where they make work possible or faster; make tool surfaces small and local; reject unowned machine output in shared project spaces; do not turn either usage or refusal into identity politics ([Communities of Not](https://lucumr.pocoo.org/2026/6/6/communities-of-not/)).

## Практика в действии

### Рабочие приёмы, уже подтверждённые источниками

- Дробление задач. Ronacher описывает, что agentic coding лучше работает на small tasks с быстрым feedback, а не на бесформенной просьбе "улучшить проект" [Agentic Coding Recommendations](https://lucumr.pocoo.org/2025/6/12/agentic-coding/).
- Предварительный план. Для сложных правок он использует видимый план или Markdown-handoff file, чтобы получить intent before mutation и вмешаться до изменения файлов; встроенный Claude Code plan mode он отдельно разбирает как похожий, но не тождественный интерфейс [Agentic Coding Recommendations](https://lucumr.pocoo.org/2025/6/12/agentic-coding/), [What Actually Is Claude Code's Plan Mode?](https://lucumr.pocoo.org/2025/12/17/what-is-plan-mode/).
- Работа в репозитории. Агент должен читать project files, запускать тесты и использовать обычные developer tools; это повторяется в июньском workflow и в июльском тезисе про code-as-tool [Agentic Coding Recommendations](https://lucumr.pocoo.org/2025/6/12/agentic-coding/), [Tools: Code is all you need](https://lucumr.pocoo.org/2025/7/3/tools/).
- Тесты и git history как рабочие источники. Ronacher явно включает в agent tasks не только генерацию кода, но и проверку, refactoring, changelog, exploration of history [Agentic Coding Recommendations](https://lucumr.pocoo.org/2025/6/12/agentic-coding/).
- Ограничение tool surface. Он предпочитает простые, композиционные interfaces и критикует чрезмерно широкий MCP tool list [Your MCP Doesn't Need 30 Tools](https://lucumr.pocoo.org/2025/8/18/code-mcps/).
- Внешняя память. В Pi workflow он использует issues, transcripts и saved state как способ не потерять контекст, когда сам агент и его продукт развиваются вместе [Building Pi With Pi](https://lucumr.pocoo.org/2026/5/24/pi-oss/).
- Compaction как часть памяти. Issue `#92` и `compaction.md` показывают, что Pi хранит session file как источник истины, добавляет compaction events вместо переписывания истории, а для branching после compaction читает raw session file, чтобы можно было вернуться к прежним user messages [Pi issue #92](https://github.com/earendil-works/pi/issues/92), [Pi compaction.md](https://github.com/earendil-works/pi/blob/main/packages/coding-agent/docs/compaction.md).
- Issue analysis как отдельная стадия. `.pi/prompts/is.md` требует маркировать issue `inprogress`, назначать его локальному `gh`-пользователю, читать полный issue thread and linked PRs через GitHub CLI, игнорировать чужой root-cause analysis и не имплементировать без отдельного запроса [Pi `.pi/prompts/is.md`](https://github.com/earendil-works/pi/blob/main/.pi/prompts/is.md). Это формализует границу между triage/diagnosis и code change.
- Wrap-up как операционная граница. `.pi/prompts/wr.md` связывает завершение issue-работы с changelog, финальным comment, точечным commit, `closes #...`, проверкой branch `main` и запретом `git add .`; это делает "закрытие работы" отдельной агентской процедурой, а не последним сообщением в чате [Pi `.pi/prompts/wr.md`](https://github.com/earendil-works/pi/blob/main/.pi/prompts/wr.md).
- Публикация session traces. В текущем README `earendil-works/pi` описан поток публикации OSS coding-agent sessions через `badlogic/pi-share-hf`; dataset `badlogicgames/pi-mono` хранит redacted Pi session JSONL с session headers, messages, tool results, model changes, compaction summaries, branch summaries and custom extension data, причём перед публикацией применяются deterministic redaction и LLM review [GitHub: earendil-works/pi README](https://github.com/earendil-works/pi/blob/main/README.md), [Hugging Face: badlogicgames/pi-mono](https://huggingface.co/datasets/badlogicgames/pi-mono).
- Изоляция среды. README прямо говорит, что Pi по умолчанию работает с правами пользователя и процесса, который его запустил, а `containerization.md` предлагает три варианта границ: OpenShell для policy-controlled sandbox, Gondolin extension для переноса built-in tools и `!` commands в local Linux micro-VM при сохранении auth на host, и Plain Docker для запуска всего процесса в контейнере [GitHub: Pi README](https://github.com/earendil-works/pi/blob/main/README.md), [Pi containerization.md](https://github.com/earendil-works/pi/blob/main/packages/coding-agent/docs/containerization.md).
- Supply-chain hardening. Текущий README фиксирует, что dependency changes считаются code changes: direct external dependencies pin to exact versions, `.npmrc` включает `save-exact=true` and `min-release-age=2`, lockfile становится source of truth, pre-commit blocks accidental lockfile commits без `PI_ALLOW_LOCKFILE_CHANGE=1`, а checks проверяют pinned deps, shrinkwrap and release smoke tests [GitHub: earendil-works/pi README](https://github.com/earendil-works/pi/blob/main/README.md). Это полезная поправка к образу "агент всё ускорил": в живом проекте ускорение соседствует с дополнительными механическими барьерами.
- Жёсткая граница contribution surface. В Pi публичный GitHub есть, но внешние issues/PR не становятся обычным каналом влияния на проект: их автоматически закрывают без быстрого project assignment, а support отправляют в Discord community [earendil-works/pi CONTRIBUTING.md](https://github.com/earendil-works/pi/blob/main/CONTRIBUTING.md). Для истории это редкая конкретика: maintainer overload превращается не в настроение, а в policy.
- Проектные инструкции для агентов. `AGENTS.md` Pi требует от агента соблюдать monorepo boundaries, запускать targeted checks и не совершать Git-операций вроде commit/push без запроса [earendil-works/pi AGENTS.md](https://github.com/earendil-works/pi/blob/main/AGENTS.md). Это показывает, что Ronacherовская agentic practice включает не только prompts, но и репозиторные guardrails.

### Что пока выглядит переносимым

- Приём "сначала план, потом правка" переносим почти в любой code agent workflow, если среда позволяет остановить изменение до записи файлов или как минимум обсудить Markdown-план до исполнения [Agentic Coding Recommendations](https://lucumr.pocoo.org/2025/6/12/agentic-coding/), [What Actually Is Claude Code's Plan Mode?](https://lucumr.pocoo.org/2025/12/17/what-is-plan-mode/).
- Маленькие tasks и быстрый test feedback не завязаны на Claude Code или Pi. Это general engineering condition, но у Ronacher оно подтверждено через его собственный agentic coding experience [Agentic Coding Recommendations](https://lucumr.pocoo.org/2025/6/12/agentic-coding/).
- Ограничение tool surface может быть перенесено на MCP servers, IDE integrations и custom agents: меньше commands, яснее semantics, больше возможностей для composition [Your MCP Doesn't Need 30 Tools](https://lucumr.pocoo.org/2025/8/18/code-mcps/).

### Что завязано на его контекст

- Ronacher умеет ревьюить результат на уровне автора libraries and frameworks. Его workflow нельзя честно переносить на новичка как "пусть агент сделает и всё"; человек должен понимать architecture, tests and failure modes.
- Его проекты имеют тесты, GitHub history и относительно структурированные codebases. Там, где feedback медленный или отсутствует, agentic coding будет выглядеть хуже.
- Open-source maintainer angle зависит от реальной нагрузки на популярные проекты. У пользователя с private repo и маленькой командой эффект "clanker" будет другим, хотя проблема unverified AI output останется.

- От спора с AI output к спору с anti-AI identity: "Communities of Not" добавляет второй край социальной картины и мешает упростить Ronacher до anti-AI maintainer [Communities of Not](https://lucumr.pocoo.org/2026/6/6/communities-of-not/).

## Ограничения и неудобные детали

- Источники Ronacher сильны, но они авторские: он описывает собственную практику и собственные раздражения. Нужны внешние реакции, особенно на "clanker" и Pi.
- Нельзя писать, что его подход "доказывает" эффективность agentic coding. Пока есть narrative evidence, repo artifacts и наблюдаемые practices, но не controlled comparison.
- Нельзя сглаживать конфликт: Ronacher одновременно получает пользу от AI and complains about AI-generated contributions. Это не противоречие, если различать ownership, verification and review cost, но будущий текст должен показать это явно.
- Pi как проект может быстро меняться. Нужна повторная проверка README, releases and issues перед финальным использованием, потому что на 2026-06-09 состояние репозитория может отличаться от январского поста.
- Pass 03 уже показал такое изменение: раннее описание Pi как desktop UI остаётся важным исторически, но текущий repo README выносит вперёд agent harness, CLI, session sharing, sandboxing and supply-chain details [GitHub: earendil-works/pi README](https://github.com/earendil-works/pi/blob/main/README.md). Будущий текст должен явно развести январский launch framing, майский dogfooding post и текущее состояние README.
- Некоторые найденные источники пока раскрыты неполно: "A Language For Agents", MiniJinja PRs and Pi issues; "Final Bottleneck", "What Actually Is Claude Code's Plan Mode?" and "Pushing Local Models With Focus And Polish" раскрыты частично, но требуют проверки примеров и screenshots.

## Неровная фактура, которую нельзя потерять

- Ronacher не ограничивает agentic coding "написанием кода": он включает boring maintenance operations - tests, changelog, git archaeology, refactoring and review loops [Agentic Coding Recommendations](https://lucumr.pocoo.org/2025/6/12/agentic-coding/).
- Его tool philosophy идёт против naive "more tools is better": он предпочитает меньший tool surface и code as compositional substrate [Tools: Code is all you need](https://lucumr.pocoo.org/2025/7/3/tools/), [Your MCP Doesn't Need 30 Tools](https://lucumr.pocoo.org/2025/8/18/code-mcps/).
- "Pi builds Pi" выглядит эффектно, но источник ценен именно потому, что показывает messy side: нужно хранить transcripts, заводить issues, бороться с потерей контекста и приводить UI к coherence [Building Pi With Pi](https://lucumr.pocoo.org/2026/5/24/pi-oss/).
- Pi issue `#92` показывает, что "память" агента ломается о конкретные технические границы: orphaned tool results, потерянные `@attachment` images, необходимость abort текущего run при auto-compaction и ветвление через raw session file, а не только через текущее state [Pi issue #92](https://github.com/earendil-works/pi/issues/92), [Pi compaction.md](https://github.com/earendil-works/pi/blob/main/packages/coding-agent/docs/compaction.md).
- `badlogicgames/pi-mono` добавляет к этому не пересказ, а сырые следы: в preview видны prompt-задания на анализ конкретных GitHub issues/PR, `/load`, release checklist, tool calls, session IDs and timestamps. Это не нужно превращать в "доказательство качества", но это редкий публичный материал того, как coding-agent sessions выглядят как данные [Hugging Face: badlogicgames/pi-mono](https://huggingface.co/datasets/badlogicgames/pi-mono).
- У `badlogicgames/pi-mono` есть собственная неровность как публичного артефакта: страница Hugging Face показывает dataset schema и preview rows, но dataset viewer одновременно сообщает `DatasetGenerationError`; значит, этот источник пока лучше использовать как свидетельство public trace layer и примеры строк, а не как стабильный аналитический корпус без дополнительной выгрузки и проверки [Hugging Face: badlogicgames/pi-mono](https://huggingface.co/datasets/badlogicgames/pi-mono).
- Pi issue `#4945` нельзя использовать как доказательство, что внешние issues "хорошие". Он полезен точнее: даже при label `possibly-openclaw-clanker` в issue есть наблюдаемый symptom, versions, model, settings and saved-session JSON, и поэтому он стал рабочим входом для проекта [Pi issue #4945](https://github.com/earendil-works/pi/issues/4945).
- Release `v0.76.0` и PR `#4979` усиливают этот эпизод, но не делают его гладким. `#4945` дошёл до release notes и merged PR, однако Ronacher в PR заранее оговорил, что change may not fully fix `#4945`, а после merge `devxoul` спросил, надо ли переоткрыть auto-closed issue [Pi PR #4979](https://github.com/earendil-works/pi/pull/4979), [Pi release v0.76.0](https://github.com/earendil-works/pi/releases/tag/v0.76.0). Нельзя писать, что root cause окончательно подтверждён только по release; можно писать, что report triggered bounded timeout work и release linkage.
- Issues `#1871` и `#4877` показывают норму хорошего внешнего bug report для Pi: versions, reproduction script, exact observed output, suspected code area or path collision. Это полезнее для истории, чем общий тезис "внешние issues бывают полезны" [Pi issue #1871](https://github.com/earendil-works/pi/issues/1871), [Pi issue #4877](https://github.com/earendil-works/pi/issues/4877).
- Issues `#4984` и `#5028` уточняют эту норму уже после включения gate: accepted-looking внешние reports дают не только symptom, но и trace to code path, local patch/workaround, platform matrix and dependency boundary. `#5027` показывает другой исход: технически понятный feature request с предложением `$RAW_ARGUMENTS` закрыт без проектного следа в основном repo, поэтому будущая история должна различать "вклад понятен" и "вклад принят в ownership проекта" [Pi issue #4984](https://github.com/earendil-works/pi/issues/4984), [Pi issue #5028](https://github.com/earendil-works/pi/issues/5028), [Pi issue #5027](https://github.com/earendil-works/pi/issues/5027).
- `Clanker` - опасный и полезный материал одновременно. Он даёт язык maintainer frustration и машинного output без ownership, но может превратиться в ярлык против людей. Для истории нужно проверить социальную реакцию, а не просто перенять слово [Clanker: A Word For The Machine](https://lucumr.pocoo.org/2026/5/26/clankers/).

## Синтез

История Armin Ronacher важна не как ещё один пример «сильный разработчик стал быстрее с Claude». Её ось — движение к минимальному, изменяемому агентному циклу. Ronacher принимает agentic coding очень глубоко, но выводит из этого не поклонение большому tool surface, а обратный инженерный выбор: маленькое ядро, файловое состояние, `Bash`, session trees, local extensions and skills that can be written, tested and discarded.

Pi в этой истории — не продуктовая витрина, а ответ на практический опыт: MCP/tool explosion and large predeclared integrations создают контекстную цену, cache/refresh problems and prompt-injection surface. Вместо этого Ronacher делает ставку на code-as-tool-interface: если агенту нужен инструмент, он может написать небольшой script/extension and run it. Это не отменяет constraints: внешние issues and PRs показывают, что хороший вклад должен приносить воспроизводимый симптом, среду, команду, observed result and ownership, а не уверенный generated text.

Главный теоретический перенос: зрелая агентная практика может двигаться не к максимальному количеству tools, а к минимальному исполняемому циклу, в котором расширения создаются по месту, состояние живёт в файлах, а ответственность за review and merge остаётся человеком. История одновременно поддерживает agentic development и ограничивает его: внутри собственного ownership AI усиливает инженера, но во внешнем open-source контуре generated input can become maintainer load.
