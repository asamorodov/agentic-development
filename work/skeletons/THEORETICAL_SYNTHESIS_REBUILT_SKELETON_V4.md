# Скелетон пересборки теоретического синтеза — v4.3-PWG

Статус: рабочий скелетон перед написанием теоретической главы.  
Дата: 2026-06-10.  
Основание: `work/approved-ai-sdlc-plan.md` v4, ADR-0007, ADR-0008, ADR-0009, `work/reports/METHODOLOGY_DEPTH_CONTRACT.md`, текущие методологические досье, сравнительные аудиты, расширенное досье `Persistent Work Graph` и решение усилить этот механизм как отдельный глубокий опорный узел.

Этот скелетон заменяет v4.2 как рабочую структуру. Он сохраняет верхнюю архитектуру AI-driven SDLC, сравнительные главы v4.2, границу между историями, теорией и техническим атласом, но вносит одно существенное изменение: `Persistent Work Graph` больше не является коротким подпунктом части о контексте. Он становится отдельным глубоким опорным механизмом рабочего состояния.

## Рабочий принцип

Верхняя рамка остаётся прежней:

```text
AI-driven SDLC = жизненный цикл программного изменения
```

Текст должен идти по жизненным напряжениям, а не по спискам инструментов. Методологии получают защищённую глубину, но теория не должна превращаться в каталог практик, источников или историй.

Истории уже находятся в первом разделе сайта. Теория может ссылаться на них как на фактические якоря, сравнивать их и извлекать из них механизмы, но не должна заново пересказывать сами истории.

Теория держит две оси одновременно.

Первая ось — жизненный цикл изменения:

```text
намерение → архитектурное решение → рабочее состояние → процесс → исполнение → свидетельства → завершение → сопровождение
```

Вторая ось — рабочие субстраты, через которые изменение переносится между стадиями:

```text
спецификация → память решения → устойчивый граф работы → среда исполнения → свидетельства → governance → ремонт жизненного цикла
```

Сильная агентская разработка возникает не потому, что модель лучше пишет код сама по себе, а потому что изменение удерживается в подходящих внешних субстратах: спецификациях, решениях, графе работы, среде исполнения, свидетельствах и человеческих подтверждениях.

## Методологический контракт

### Глубокие опорные узлы

```text
SPDD
Persistent Work Graph
Gas Town / Beads
```

Эти узлы не одинакового типа, поэтому их нужно различать.

- `SPDD` — глубокий опорный случай спецификационного жизненного цикла.
- `Persistent Work Graph` — глубокий опорный механизм жизненного цикла рабочего состояния.
- `Gas Town / Beads` — глубокий опорный случай организационно-операционного жизненного цикла.

`Persistent Work Graph` не нужно сводить к `Beads`, `GitHub Issues`, `Taskmaster`, `LangGraph`, `Temporal`, CRDT-редактору или файловому рабочему процессу. Это переносимый механизм, который объясняет, как работа становится долговечным объектом: с владельцем, зависимостями, контрольными барьерами, готовностью, передачей, компактным восстановлением контекста, пакетом свидетельств и последующей очисткой незакрытого состояния.

### Защищённые методологические профили

```text
Spec Kit
Kiro Specs
Constitutional SDD
TDAD: Test-Driven AI Agent Definition
TDAD: Test-Driven Agentic Development
GSD / Open GSD
BMAD Method
```

Каждый защищённый методологический профиль должен раскрывать:

```text
проблема → рабочий процесс → артефакты → контекст → роли → человеческие подтверждения → проверка → последующий ремонт жизненного цикла → сильные стороны → сбои и риски → сравнение с соседними подходами → граница theory / Handbook / Fieldbook
```

### Защищённый профиль архитектурного решения

```text
ADR / Architecture Decision Records
```

ADR не становится отдельным глубоким опорным случаем наравне с SPDD, Persistent Work Graph и Gas Town, но получает защищённую глубину внутри Части III. В теории нужно раскрыть Nygard, MADR, статусы ADR, reconstructed ADR, операционную проекцию ADR для агента и контуры подтверждения решения.

### Технический атлас / концептуально-технический атлас

Технический атлас больше не является только приложением для технической фактуры. Он работает как концептуально-технический слой: набор самостоятельных статей с опорой на источники по тяжёлым концепциям, методологиям и инструментальным формам:

```text
SPDD
Spec Kit
Kiro Specs
Constitutional SDD
TDAD
GSD / Open GSD
BMAD Method
Persistent Work Graph
Gas Town / Beads
ADR
```

В атласе должны жить не только источниковая база, файлы, команды, структуры, интерфейсы, проверочные контуры, ограничения источников, кандидаты на схемы и скриншоты, но и связное объяснение самой концепции для читателя, который пришёл именно за ней. Теоретическая глава при этом остаётся самодостаточной как поперечный SDLC-синтез; контролируемое повторение между теорией и атласом допустим, если оно нужно для самостоятельного чтения статьи атласа.

# Введение. Почему речь не о генерации кода

## Задача части

Снять три ложных чтения:

1. это не “как лучше промптить”;
2. это не “обычный SDLC, где в каждую фазу вставили AI”;
3. это не “чем больше агент сгенерировал кода, тем лучше”.

## Ход

1. Код подешевел, но смысл, проверка, рабочее состояние и ответственность не подешевели автоматически.
2. Агент усиливает существующую систему только там, где есть внешние рабочие субстраты.
3. Объект анализа — жизненный цикл программного изменения.
4. Артефакт важен только тогда, когда переносит изменение через границу жизненного цикла.
5. Истории сайта уже дали фактический корпус; теория извлекает из них механизмы и не пересказывает их заново.
6. Переход: сначала нужно сменить единицу анализа.

# I. Единица анализа: программное изменение, а не prompt

## Управляющий тезис

Агентская разработка становится понятной только если смотреть на программное изменение как на контур: намерение, архитектурное решение, рабочее состояние, процесс, исполнение, свидетельства, ревью, право завершения и сопровождение.

## Что раскрыть

1. Почему `prompt → code` слишком бедная модель.
2. Программное изменение как жизненный цикл.
3. Где возникает рабочий запрос до спецификации:
   - issue;
   - Slack thread;
   - PR feedback;
   - support ticket;
   - browser observation;
   - workflow run;
   - source shard;
   - архитектурное решение;
   - документный пакет.
4. Артефакт входит в теорию только если несёт функцию в жизненном цикле изменения.
5. Основные рабочие субстраты:
   - намерение / спецификация;
   - решение / архитектурная память;
   - устойчивый граф работы;
   - среда исполнения;
   - свидетельства;
   - governance;
   - ремонт жизненного цикла.
6. Переход к реальной агентской сессии.

# II. Реальная агентская сессия: трасса, вмешательство, выживание результата

## Управляющий тезис

Реальная агентская разработка видна в трассе, но трасса не удерживает всю инженерную работу. Часть смысла, решений, рабочего состояния и ответственности должна пережить сессию.

## Сравнительный узел: SWE-chat / Programming by Chat / How Coding Agents Fail

- SWE-chat показывает трассу действий и вызовы инструментов.
- Programming by Chat показывает постепенную спецификацию и типы сессий.
- How Coding Agents Fail показывает рассогласования между намерением, действиями агента и результатом.

Синтез:

```text
трасса сессии необходима, но она не является долговечным рабочим состоянием.
```

## Что раскрыть

1. Трасса как свидетельство, но не вся правда.
2. Постепенная спецификация как нормальная форма работы.
3. Коррекция и сопротивление как сигнал качества.
4. Что исчезает после сессии.
5. Что должно выжить после сессии:
   - намерение;
   - решение;
   - рабочее состояние;
   - свидетельства;
   - нерешённые противоречия;
   - границы ответственности.
6. Переход к артефактам намерения и решения.

# III. Намерение, спецификация, контракт и архитектурное решение

## Управляющий тезис

AI-driven SDLC начинается с превращения намерения, решений и ограничений во внешние артефакты, которые можно ревьюить, версионировать, связывать с кодом и возвращать в следующий цикл.

## Что раскрыть

1. Prompt как локальная инструкция.
2. Долг намерения: что ломается, когда намерение остаётся только в чате.
3. Разные артефакты намерения и решения.

## Сравнительный узел: specification / ADR / contract

### Specification

Спецификация фиксирует ожидаемое изменение, границы и проверяемые требования. В сильном режиме она становится рабочей поверхностью агента.

### Contract

Контракт задаёт проверяемое ожидание: поведение, API, ожидание потребителя, проверочный oracle или условие совместимости. Он помогает доказать, что изменение не разрушило договорённость, но сам по себе не объясняет, почему архитектурное решение было принято.

### ADR

ADR хранит память архитектурного решения.

Обязательные элементы теоретического объяснения:

1. Nygard: ADR как короткая запись решения, контекста и последствий.
2. MADR: более структурированная форма ADR с шаблоном, статусами, связями и последствиями.
3. ADR не является задачей, спецификацией или комментарием к коду.
4. ADR фиксирует не только “что сделали”, но и почему это решение было разумным в данных обстоятельствах.

### Статус и происхождение записи

Нужно развести статус решения и происхождение записи.

Статусы решения:

```text
proposed / accepted / deprecated / superseded
```

Происхождение записи:

```text
generated / reconstructed
```

Generated или reconstructed не дают права считать решение принятым.

### Reconstructed ADR

Reconstructed ADR — не автоматически принятый ADR, а гипотеза о решении:

```text
evidence
confidence
known gaps
review owner
confirmation plan
```

### Операционная проекция ADR

Полный ADR не должен целиком превращаться в бесформенный контекст для агента. Для агента нужна короткая операционная проекция:

```text
scope
rule
status
backlink to full ADR
applicability check
```

### Подтверждение ADR

Подтверждение ADR шире ревью владельца. Возможные контуры подтверждения:

```text
owner review
code structure
API compatibility
consumer contract
performance / SLO
rollout / runtime evidence
```

## Переход

SPDD показывает, как намерение становится полноценным спецификационным жизненным циклом. Но даже сильная спецификация не решает вопрос долговечного рабочего состояния; для этого позже нужен Persistent Work Graph.

# IV. Глубокий опорный случай: SPDD как спецификационный жизненный цикл

## Управляющий тезис

SPDD показывает, как намерение превращается в управляемый спецификационный жизненный цикл: prompt становится полноценным артефактом поставки, связанным с анализом, генерацией, тестами, ревью, обновлением и синхронизацией.

## Правило baseline

До написания взять old-site SPDD seed без изменений. Адаптация допустима только после anti-degradation inventory.

## Что раскрыть

1. Почему SPDD возник.
2. Prompt как полноценный артефакт поставки.
3. Story shaping.
4. Analysis context.
5. REASONS Canvas.
6. OpenSPDD command layer.
7. Пример с billing engine.
8. Генерация как downstream operation.
9. API tests и функциональные свидетельства.
10. Итеративное ревью.
11. Обновление prompt и дисциплина синхронизации.
12. Human learning и границы ревью.
13. Fit / ROI.
14. Что SPDD даёт AI-driven SDLC.
15. Чего SPDD не решает: долговечная очередь работы, ownership, контрольные барьеры, handoff, recovery.

## Граница с техническим атласом

В теории остаётся механизм SPDD и его значение для жизненного цикла. В атласе SPDD должен раскрываться как самостоятельная чтение от конкретной концепции статья: читатель должен понять саму концепцию, затем команды, шаблоны, файлы, примеры, заметки по конкретным источникам и кандидаты на схемы. Повтор основных тезисов теории допустим, если он нужен для самостоятельного чтения раздела.

# V. Защищённые спецификационные методологические профили

## Управляющий тезис

SPDD не исчерпывает слой спецификации. Соседние методологии показывают разные способы сделать намерение управляемым: через инструментарий, поверхность IDE, тесты, constitution, управление и ответственность или разбиение работы на задачи.

## Правило глубины

Для каждой методологии раскрыть:

```text
проблема → рабочий процесс → артефакты → контекст → роли → человеческие подтверждения → проверка → последующий ремонт жизненного цикла → сильные стороны → сбои и риски → сравнение с соседними подходами → граница theory / Handbook / Fieldbook
```

## V.1. Spec Kit как переносимый инструментарий спецификаций

1. Проблема: превращение спецификаций в переносимый рабочий процесс агентной разработки.
2. Рабочий процесс: constitution → specify → plan → tasks → implement.
3. Артефакты: constitution, spec, plan, tasks, templates, scripts, `.specify`, extensions / presets.
4. Контекст: project memory and principles.
5. Человеческие подтверждения: constitution, ревью плана, подтверждение задач.
6. Проверка: analyze / checklist / hooks / extensions.
7. Хвост жизненного цикла: specs and constitution evolve.
8. Сильные стороны: переносимость, экосистема, организационная настройка.
9. Сбои и риски: ритуальная спецификация, template cargo cult, stale constitution.
10. Сравнение: закрытый цикл SPDD vs инструментарий Spec Kit.

## V.2. Kiro Specs как продуктовая поверхность спецификационного рабочего процесса

1. Проблема: сделать спецификационный рабочий процесс частью IDE и продуктовой поверхности.
2. Рабочий процесс: requirements / bug analysis → design → tasks → выполнение задач.
3. Артефакты: `requirements.md` / `bugfix.md`, `design.md`, `tasks.md`, task status.
4. Контекст: `#spec`, Sync Files, repo scanning.
5. Человеческие подтверждения: контрольные ревью-точки vs Quick Plan.
6. Проверка: выполнение задач, sync, ревью-контрольные точки.
7. Хвост жизненного цикла: specs split, updated, synced with implementation.
8. Сильные стороны: продуктовая поверхность совместной работы.
9. Сбои и риски: shallow specs, Quick Plan overuse, spec drift.
10. Сравнение: IDE-поверхность Kiro vs переносимый инструментарий Spec Kit.

## V.3. TDAD comparative protected profile

### TDAD A: Test-Driven AI Agent Definition

1. Проблема: задать поведение агента, который пользуется инструментами.
2. Рабочий процесс: behavioral specs → tests → компиляция / уточнение prompt.
3. Артефакты: behavioral spec, visible / hidden tests, compiled prompt.
4. Проверка: semantic mutation testing, hidden tests.
5. Сбои и риски: gaming тестов, узкое поведение, переобучение prompt.

### TDAD B: Test-Driven Agentic Development

1. Проблема: снизить регрессии coding-agent через целевой тестовый контекст.
2. Рабочий процесс: code-test graph → impact analysis → agent skill → targeted tests.
3. Артефакты: graph, impact weights, skill file, test set.
4. Проверка: regression reduction.
5. Сбои и риски: устаревание графа, ложное наведение, процедурные TDD-инструкции без контекста.

### Сравнение двух TDAD

Развести две разные идеи под одним сокращением:

```text
tests as behavioral specification
тесты как поверхность наведения против регрессий
```

## V.4. Constitutional SDD как управление и ответственность внутри спецификации

1. Проблема: security и governance должны формировать генерацию до появления кода.
2. Рабочий процесс: Constitution → constraints → generation → traceability → review.
3. Артефакты: machine-readable Constitution, constraint mappings, traceability reports.
4. Контекст: CWE / MITRE / regulatory frameworks.
5. Человеческие подтверждения: подтверждение constitution, security review.
6. Проверка: заявления о снижении security defects, traceability.
7. Хвост жизненного цикла: Constitution evolves after incidents / regulatory change.
8. Сильные стороны: security и governance сдвигаются в слой спецификации.
9. Сбои и риски: узкая эмпирическая база, ложная безопасность, чрезмерные ограничения.
10. Сравнение: Constitutional SDD vs policy-as-code vs security review.

## V.5. Сравнительный синтез: пять способов сделать намерение управляемым

Сравнить:

- SPDD — метод с закрытым циклом;
- Spec Kit — инструментарий / экосистема;
- Kiro — продуктовая IDE-поверхность;
- TDAD — тесты как спецификация поведения или поверхность наведения для регрессий;
- Constitutional SDD — security и governance как часть спецификации.

Вопрос:

```text
какой артефакт становится центральным,
какое человеческое суждение остаётся необходимым,
и какой порядок обновления нужен после появления кода?
```

# VI. Контекст и рабочее состояние: проект как интерфейс агента

## Управляющий тезис

Проект становится интерфейсом агента только тогда, когда состояние работы вынесено из головы и чата в поддерживаемые, проверяемые и ограниченные поверхности. Но больше контекста не значит лучше.

## Что раскрыть

1. Что агент должен знать, чтобы не действовать вслепую.
2. Где живёт состояние задачи до Persistent Work Graph:
   - context files;
   - `AGENTS.md`;
   - rules;
   - skills;
   - project memory;
   - issue comments;
   - PR thread;
   - local notes;
   - handoff.
3. Почему больше контекста не значит лучше.
4. Как кодовая база становится рабочей поверхностью.
5. Как проект показывает владельцев, правила и runnable path.
6. Stale context как отдельный вид долга.
7. Переход: markdown-память, стенограмма и handoff-файл не являются устойчивым графом работы.

Эта часть больше не должна “вмещать” Persistent Work Graph целиком. Она готовит проблему, которую решает следующая часть.

# VII. Глубокий опорный механизм: Persistent Work Graph как устойчивый граф работы

## Управляющий тезис

Persistent Work Graph — это переносимый механизм, в котором работа становится долговечным объектом: её можно продолжить после обрыва сессии, передать другому агенту, заблокировать контрольным барьером, восстановить через compact prime, проверить по артефактам и очистить после завершения.

Это не обычный issue tracker, не execution graph и не весь Gas Town. Это рабочий субстрат между context files и организационной средой.

## VII.1. Почему сессии и markdown-памяти недостаточно

1. Сессия обрывается.
2. Контекст сжимается.
3. Агент теряет открытые противоречия.
4. Несколько агентов начинают дублировать работу.
5. Источник открыт, но не перенесён в канонический текст.
6. Задача “готова” только потому, что агент произвёл текст.

## VII.2. Work item как долговечный объект

Минимальный объект:

```text
id
title
status
scope
owner / claimant
dependencies
gates
artifacts
evidence
handoff
history
```

## VII.3. Зависимости и очередь готовности

Не просто список задач, а граф готовности:

```text
blocked
ready
claimed
in review
waiting for gate
completed
stale
```

## VII.4. Ownership, claim, pin, lock

Зачем нужны:

- владелец;
- claim;
- pin;
- lock;
- handoff;
- conflict detection.

Главный тезис:

```text
без ownership граф превращается в общий список пожеланий.
```

## VII.5. Gates как долговечные ожидания

Gate — это не комментарий “проверить потом”, а объект, который блокирует движение:

```text
human approval
source audit
language gate
test gate
compatibility gate
security gate
merge / synthesis gate
```

## VII.6. Prime / rehydration

Compact prime возвращает агенту рабочую правду без полного перечитывания проекта:

```text
что делаем
почему
что уже решено
что заблокировано
какие артефакты обязательны
какие противоречия открыты
где человек должен принять решение
```

## VII.7. Recovery

После обрыва должно быть понятно:

```text
что было последним валидным состоянием
какие артефакты записаны
какие действия не завершены
что можно продолжить
что требует human review
```

## VII.8. Состояние источников и промежуточные артефакты

Для документной работы Persistent Work Graph особенно важен.

Состояния источника:

```text
found
opened
read
used_in_main_text
candidate_image_only
rejected_with_reason
needs_reopen
```

Состояния прохода:

```text
job
pass
delta
audit
repair
result
manifest
```

## VII.9. Безопасная параллельная работа с источниками

Параллелить можно поиск и раскрытие источников, но не каноническую запись.

Минимальные элементы:

```text
source shard
read snapshot
source claim
worker evidence package
citation audit
canonical write lock
image-candidate gate
synthesis pass
```

## VII.10. Persistent Work Graph и соседние механизмы

Развести:

```text
Persistent Work Graph ≠ GitHub Issues / Linear
Persistent Work Graph ≠ Taskmaster
Persistent Work Graph ≠ LangGraph / Temporal / durable execution
Persistent Work Graph ≠ CRDT editor
Persistent Work Graph ≠ STORM-like file mediator
Persistent Work Graph ≠ Gas Town целиком
```

## VII.11. Сбои и риски

1. Graph theater.
2. Stale graph.
3. False completion.
4. Wrong authority.
5. Storage complexity.
6. Auto-approval collapse.
7. Metadata leaks.
8. Semantic convergence trap.
9. Read-snapshot staleness.
10. Coordination overconfidence.

## VII.12. Что Persistent Work Graph даёт общей теории

Persistent Work Graph меняет модель агентной разработки:

```text
работа живёт не в prompt,
не в стенограмме,
не только в issue,
а в долговечном графе состояний, контрольных барьеров и артефактов.
```

Это делает возможными:

- продолжение после обрыва;
- безопасную параллельность;
- передачу между агентами;
- доказуемое завершение;
- cleanup / последующую очистку незакрытого состояния.

## Граница с техническим атласом

В теории остаётся механизм PWG и его роль в AI-driven SDLC. В атласе PWG должен быть самостоятельной концептуально-технической статьёй: он может повторить нужное объяснение механизма, а затем дать конкретные структуры объектов, команды, source registers, схемы, протоколы параллельной работы, Beads/Taskmaster/LangGraph/STORM/CodeCRDT источниковые заметки и кандидаты на иллюстрации.

# VIII. Защищённые процессные методологические профили

## Управляющий тезис

Иногда локальных документов и ручного выбора режима недостаточно. Тогда процесс сам становится артефактом: он задаёт роли, состояние, рабочий процесс, проверку и handoff.

GSD и BMAD — более лёгкие формы внешнего состояния и процессной дисциплины. Gas Town — более тяжёлая организационная среда. Persistent Work Graph объясняет, почему этим формам нужен долговечный рабочий субстрат.

## Правило глубины

GSD и BMAD должны быть реконструированы как защищённые методологические профили, а не как короткий обзор.

## VIII.1. Когда достаточно разговора

Малые обратимые задачи. Без методологического overkill.

## VIII.2. Когда сначала нужен research or plan

Boris Tane / HumanLayer / Mark Erikson как практические якоря.

## VIII.3. Когда нужен spec-first

Мост к Частям III–V.

## VIII.4. GSD / Open GSD как лёгкий контур внешнего состояния

Обязательные элементы:

1. Проблема: распад контекста и длительная непрерывность задачи.
2. Рабочий процесс: discuss → plan → execute → verify → ship.
3. Артефакты: state / context files, планы, заметки о проверке.
4. Контекст: вынесенное состояние, исполнение со свежим контекстом.
5. Роли: human, agent, possible subagents.
6. Человеческие подтверждения: plan approval, verify / ship decision.
7. Проверка: verify step, small checkable phases.
8. Хвост жизненного цикла: очистка состояния, обновление плана, ремонт контекста.
9. Сильные стороны: lightweight process discipline.
10. Сбои и риски: устаревшее состояние, ложная свежесть, чрезмерная фрагментация.
11. Сравнение: GSD vs Spec Kit, BMAD, Persistent Work Graph, Gas Town.

Граница:

```text
GSD не является полноценным Persistent Work Graph,
но показывает лёгкий контур состояние / контрольные барьеры / восстановление.
```

## VIII.5. BMAD как role-based guided process

Обязательные элементы:

1. Проблема: идеация, планирование, архитектура и реализация требуют ролевой структуры.
2. Рабочий процесс: analysis → planning → architecture → implementation, adapted by project scale.
3. Артефакты: role outputs, plans, architecture docs, implementation tasks.
4. Контекст: направляемый рабочий процесс и контекст роли.
5. Роли: specialized agents / domain experts.
6. Человеческие подтверждения: подтверждение планирования / архитектуры, границы исполнения задач.
7. Проверка: контрольные барьеры рабочего процесса, ревью, проверки реализации.
8. Хвост жизненного цикла: состояние процесса, handoff, результаты ролей.
9. Сильные стороны: role-based organization and guided flow.
10. Сбои и риски: role theater, overprocess, shallow expert simulation.
11. Сравнение: BMAD vs GSD, Spec Kit, Persistent Work Graph, Gas Town.

Граница:

```text
BMAD организует роли и фазы,
но требует долговечного состояния, чтобы role outputs не стали театром.
```

## VIII.6. Reversa / OpenSpec / AgentSPEX как соседние формы

Короткое рассмотрение:

- Reversa: recovering operational specs from legacy;
- OpenSpec / AgentSPEX: спецификация агентного рабочего процесса и переносимость.

Роль: поддержать спектр process-as-artifact, но не становиться глубоким опорным узлом.

## VIII.7. Сравнительный синтез: когда процесс становится артефактом

Сравнить:

- Spec Kit — инструментарий спецификаций;
- GSD — лёгкий контекстный / процессный цикл;
- BMAD — role-based guided process;
- Reversa / OpenSpec / AgentSPEX — восстановление или спецификация рабочего процесса;
- Persistent Work Graph — долговечный рабочий субстрат;
- Gas Town — полноценная организационная среда.

Вопрос:

```text
когда процесс становится артефактом,
а когда нужен отдельный устойчивый граф работы?
```

## VIII.8. Когда агенту надо отказать

Риск, необратимость, недостающий контекст, слабые свидетельства, неясный владелец.

# IX. Исполнение: среда агента, harness, tools, permissions

## Управляющий тезис

Среда исполнения должна не только дать агенту способность действовать, но и сделать действие ограниченным, воспроизводимым, наблюдаемым и обратимым.

## Что раскрыть

1. Возможность действовать.
2. Граница действия.
3. Execution envelope:
   - sandbox;
   - worktree;
   - devbox;
   - permissions;
   - approvals.
4. Поверхность инструментов и наблюдения:
   - CLI;
   - logs;
   - browser;
   - database;
   - MCP;
   - local scripts.
5. Сравнительный узел: Harness / Sandvault / platform agents.
6. Среда запуска и движок рабочего процесса:
   - structured run;
   - replay;
   - resumability;
   - observability;
   - failure recovery.
7. Platform agent:
   - internal developer platform;
   - blueprints;
   - CI limits;
   - human review.
8. Истории как короткие якоря, не встроенные кейсы:
   - Shopify Roast;
   - Quix / Klaus Kode;
   - Stripe Minions;
   - Armin Ronacher.
9. Где среда исполнения сама становится долгом.
10. Связь с Persistent Work Graph:

```text
runtime исполнения может выполнить шаг,
но Persistent Work Graph хранит существование работы, контрольные барьеры и право продолжения.
```

# X. Глубокий опорный случай: Gas Town / Beads как организационно-операционный жизненный цикл

## Управляющий тезис

Gas Town показывает, как агентская разработка становится организационной средой: роли, память задачи, управляющие поверхности, сервисные агенты, handoff и стоимость оркестрации.

Gas Town теперь читается не как единственный носитель Persistent Work Graph, а как сильнейший случай, где этот механизм соединён с организационной системой.

## Правило baseline

До написания взять old-site Gas Town seed без изменений. Адаптация допустима только после anti-degradation inventory.

## Что раскрыть

1. Почему Gas Town нельзя оставлять обзорным разделом.
2. Роли как организационные функции.
3. Mayor: видимость и управление.
4. Агент не равен сессии.
5. Beads как слой данных, управления и объяснения почему.
6. Persistent Work Graph на живом материале:
   - bead;
   - dependency;
   - gate;
   - ready queue;
   - prime;
   - recovery;
   - multi-agent coordination.
7. GUPP, hooks, molecules, wisps.
8. Refinery, Witness, Deacon.
9. Продолжение сессии и handoff.
10. DoltHub contrast.
11. Стоимость: пропускная способность против понимания.
12. Что Gas Town даёт AI-driven SDLC сверх Persistent Work Graph:
    - роли;
    - организационная видимость;
    - сервисные агенты;
    - операционная культура;
    - стоимость оркестрации.

## Граница с Частью VII

Часть VII объясняет переносимый механизм Persistent Work Graph. Часть X показывает, как этот механизм становится частью живой организационно-операционной среды.

## Граница с техническим атласом

В теории остаётся смысл и механизм Gas Town как организационно-операционного случая. В атласе Gas Town / Beads раскрывается как самостоятельная чтение от конкретной концепции статья: роли, Mayor, Beads, GUPP, hooks, molecules, wisps, service agents, diagrams и заметки по конкретным источникам должны быть связаны с объяснением концепции, а не просто перечислены как технический справочник.

# XI. Свидетельства, тесты, ревью и качество доказательства

## Управляющий тезис

Когда генерация дешевеет, главным дефицитом становится право считать изменение доказанным. Свидетельство зависит от типа изменения.

## Что раскрыть

1. Почему summary агента не является свидетельством.
2. Наблюдение и свидетельство:

```text
наблюдение помогает агенту продолжить работу;
свидетельство позволяет человеку принять или отклонить изменение.
```

3. Поведенческая правильность.
4. Граничная правильность.
5. Сравнительный узел: architecture fitness / contract tests / test data.
6. Архитектурная сохранность.
7. API compatibility:
   - breaking changes;
   - consumer contracts;
   - deployability.
8. Performance / SLO / error budget.
9. Rollout / runtime confirmation.
10. Replay / transcript / trace.
11. Надёжность доказательства: данные, среда, oracle.
12. Социальная проверка:
   - owner review;
   - CODEOWNERS;
   - maintainer acceptance.
13. Пакет свидетельств как поддержка решения.
14. Связь с ADR Confirmation.
15. Связь с Persistent Work Graph:

```text
Persistent Work Graph должен хранить не только статус задачи,
но и указатель на evidence package,
иначе completed превращается в false completion.
```

# XII. Завершение, управление ответственностью и внешний контур

## Управляющий тезис

Право действовать и право завершать изменение принадлежат разным контурам. Агент может выполнить работу, но не владеть её принятием.

## Что раскрыть

1. Агент может исполнить, но не владеть завершением.
2. Кто обязан смотреть.
3. Кто может принять.
4. Кто отвечает за выпуск.
5. Сравнительный узел: SASE / open-source policies / CODEOWNERS.
6. Contribution boundary:
   - AI-generated code;
   - AI-generated reasoning;
   - maintainer attention;
   - trust cost.
7. No-AI / limited-AI policies как сигнал управления и ответственности.
8. Zig как короткий якорь, не встроенная история.
9. Восстановимость завершения.
10. Связь с Persistent Work Graph:

```text
агент может обновить рабочий статус,
но не может сам присвоить себе authority финального принятия.
```

## Что раскрыть

1. Release feedback.
2. Incident feedback.
3. Drift feedback.
4. Сравнительный узел: incident / stale ADR / context cleanup.
5. Stale specification.
6. Stale ADR.
7. Устаревшая операционная проекция.
8. Stale `AGENTS.md` / rules / skills.
9. Устаревший граф работы.
10. Work graph cleanup:
    - closed work;
    - stale dependencies;
    - obsolete gates;

# Порядок написания и готовность материала

Этот раздел не требует заранее создать новые “документы будущей теории”. Теоретическая глава и технический атлас будут написаны по скелетону. Досье, карты источников, старые версии разделов, истории и отчёты являются материалом для письма, а не готовыми частями будущего сайта.

Перед написанием каждой части нужно проверить, что для неё достаточно фактуры в уже собранных историях, досье, картах источников и рабочих заметках. Если материала не хватает, сначала проводится дополнительный проход по досье или источникам. Нельзя закрывать слабую часть гладкой теоретической прозой поверх недостаточной фактуры.

Ниже указаны материалы, которыми нужно пользоваться прежде всего. Это не запрет на дополнительные источники, а приоритет чтения: сначала собранный корпус, затем точечное раскрытие внешних первоисточников, если при письме не хватает проверки или точной ссылки.

# Проверки качества скелетона

## Для каждой методологической секции

1. Раскрыта ли структура защищённого методологического профиля?
2. Есть ли сравнение с соседними методами?
3. Сохраняется ли линия жизненного цикла изменения?
4. Не превращается ли раздел в поверхностный обзор?
5. Понятно ли, что идёт в теорию, а что в Handbook / Fieldbook?

## Для Persistent Work Graph

1. Не сводится ли Persistent Work Graph к Beads?
2. Не сводится ли Persistent Work Graph к issue tracker?
3. Разведены ли work graph и execution graph?
4. Есть ли owner / claim / gate / readiness / recovery?
5. Есть ли source state и evidence package для документной работы?
6. Есть ли режимы сбоя: graph theater, stale graph, false completion, wrong authority?
7. Не стал ли раздел технической схемой реализации вместо теоретического механизма?
8. Есть ли отдельная глава в техническом атласе?

## Для Gas Town

1. Не сводится ли Gas Town к Persistent Work Graph?
2. Видны ли роли, Mayor, service agents, handoff и организационная среда?
3. Показано ли, что Gas Town — глубокий опорный случай, а не просто источник для Persistent Work Graph?

## Для ADR

1. Объяснены ли Nygard и MADR внутри теории?
2. Разведены ли статус решения и происхождение записи?
3. Разведены ли полный ADR и операционная проекция ADR для агента?
4. Есть ли подтверждение шире ревью владельца?
5. Не превращается ли ADR в задачу, спецификацию или бесформенный контекст?

## Для историй

1. Ссылается ли теория на истории как на якоря?
2. Не пересказывает ли теория истории заново?
3. Сохраняется ли первый раздел сайта как фактический корпус историй?

## Для технического атласа

1. Даёт ли он путь чтения от конкретной концепции: можно ли понять конкретную концепцию без сборки материала из разных глав теории?
2. Ограничен ли он выбранными тяжёлыми досье и концепциями или требует отдельного human gate для расширения?
3. Не стал ли он механическим дублёром общей теории?
4. Не деградировал ли он до склада команд, скриншоты и источниковые заметки без связного объяснения?
5. Сохраняет ли он технические детали, которые перегрузили бы теорию, но связывает ли их с концепцией?
6. Есть ли отдельная глава Persistent Work Graph?
