
# BMAD Method

Рабочее досье по BMAD Method после отдельного русского языковой ремонт. Это не финальная глава и не профиль; цель — сохранить механику BMAD как цепочку фаз, агентов, skills, файлов, контрольные точки, режимы поломки и вопросов для следующего source pass.

BMAD Getting Started описывает метод как управляемые рабочие маршруты со специализированными AI-агентами, которые проводят через планирование, архитектуру и реализацию [BMAD Getting Started](https://docs.bmad-method.org/tutorials/getting-started/). Минимальный вход: Node.js 20.12+, Git, AI-IDE и идея проекта [BMAD Getting Started](https://docs.bmad-method.org/tutorials/getting-started/). Установка: `npx bmad-method install`, для предварительная версия `npx bmad-method@next install`; установщик создаёт `_bmad/` и `_bmad-output/` [BMAD Getting Started](https://docs.bmad-method.org/tutorials/getting-started/). `_bmad/` содержит агенты, рабочий маршрутs, задачи и configuration; `_bmad-output/` предназначен для сохраняемых artifacts [BMAD Getting Started](https://docs.bmad-method.org/tutorials/getting-started/).

BMAD-Help — не декоративный помощник, а навигационный механизм. Источник говорит, что после установки нужно вызвать `bmad-help`; он определяет установленные модули, проверяет прогресс проекта, рекомендует следующий рабочий маршрут и первую нужную задачу [BMAD Getting Started](https://docs.bmad-method.org/tutorials/getting-started/). Дополнительная важная деталь: BMad-Help автоматически запускается в конце каждого рабочего маршрута и подсказывает следующий шаг [BMAD Getting Started](https://docs.bmad-method.org/tutorials/getting-started/). Для будущей главы это нужно раскрыть как слой против потери ориентира: BMAD пытается решать проблему “что делать дальше” внутри длинного процесса.

Четыре фазы BMAD: Анализ, Планирование, Проектирование решения, Реализация [BMAD Getting Started](https://docs.bmad-method.org/tutorials/getting-started/). Фаза анализа необязательна: brainstorming, market/domain/technical research, product brief, PRFAQ [BMAD Getting Started](https://docs.bmad-method.org/tutorials/getting-started/). Фаза планирования обязательна: для BMad Method и Enterprise tracks run `bmad-prd`; на выходе появляются `prd.md`, `addendum.md`, `decision-log.md`; режимы включают Create, Update, Validate [BMAD Getting Started](https://docs.bmad-method.org/tutorials/getting-started/). Проектирование решения создаёт архитектура и epics/stories; Реализация идёт эпик за эпиком, story за story [BMAD Getting Started](https://docs.bmad-method.org/tutorials/getting-started/).

Треки важны. Quick Flow подходит для исправлений ошибок, простых функций и ясного объёма, roughly 1-15 stories; BMad Method подходит для продуктов, платформ и сложных функций, roughly 10-50+ stories; Enterprise подходит для систем с требованиями соответствия и multi-tenant архитектурой, roughly 30+ stories [BMAD Getting Started](https://docs.bmad-method.org/tutorials/getting-started/). Источник explicitly warns that число story — ориентир, а не определение; трек выбирают по потребностям планирования [BMAD Getting Started](https://docs.bmad-method.org/tutorials/getting-started/). Это важный оговорка: нельзя механически выбирать процесс по числу story, нужно смотреть на степень неопределённости и требуемые документы.

Планирование и solutioning chain: `bmad-prd` creates или validates requirements artifacts; опциональный UX-проход использует `bmad-agent-ux-designer` и `bmad-ux`; архитектура uses `bmad-agent-architect` и `bmad-create-architecture`; epics и stories now come после архитектура so database, API patterns и tech stack can inform story breakdown; PM runs `bmad-create-epics-and-stories` using both PRD и Architecture [BMAD Getting Started](https://docs.bmad-method.org/tutorials/getting-started/). Здесь важно сохранить передачу контекста: PRD не заканчивает работу, архитектура уточняет реализацию, а stories создаются после архитектура, чтобы не быть product-only нарезкой.

Реализация Readiness Check — особенно важный контрольная точка. Источник рекомендует invoke Architect in a свежий чат и run `bmad-check-implementation-readiness`; он validates cohesion между all planning documents [BMAD Getting Started](https://docs.bmad-method.org/tutorials/getting-started/). Пока exact output формата PASS/CONCERNS/FAIL нужно проверить в deeper source pass; но уже сейчас понятно, что контрольная точка стоит между planning artifacts и реализация. Для будущей главы это можно описать как BMAD’s anti-false-передача контекста mechanism: нельзя честно передать story в dev, если PRD, архитектура и stories расходятся.

Build cycle: после planning, invoke Developer агент и run `bmad-sprint-planning`, which creates `sprint-status.yaml`; для each story repeat fresh-chat cycle: `bmad-create-story`, `bmad-dev-story`, `bmad-code-review`; после all stories in an epic run `bmad-retrospective` [BMAD Getting Started](https://docs.bmad-method.org/tutorials/getting-started/). Project output tree includes `_bmad-output/planning-artifacts/PRD.md`, `architecture.md`, `epics/`, `_bmad-output/implementation-artifacts/sprint-status.yaml` и optional `project-context.md` [BMAD Getting Started](https://docs.bmad-method.org/tutorials/getting-started/). Кандидат на изображение: project tree from Getting Started; причина — показывает, где физически живёт контекст.

Свежие чаты are a real methodological rule. Documentation says always start a свежий чат для each рабочий маршрут to prevent context limitations [BMAD Getting Started](https://docs.bmad-method.org/tutorials/getting-started/). Это непривычно, но важно: BMAD не полагается на один огромный чат; it выносит состояние в артефакты и заново загружает нужную роль и маршрут в новом разговоре. `project-context.md` can document technical preferences и реализация rules; it can be created manually в `_bmad-output/project-context.md` или generated после архитектура с `bmad-generate-project-context` [BMAD Getting Started](https://docs.bmad-method.org/tutorials/getting-started/).

Режимы поломки: role theater, where агент names create confidence without evidence; stale PRD после архитектура changes; stale архитектура после реализация discoveries; story breakdown created too early; readiness check run как ritual; свежий чатs without enough artifact context; `project-context.md` missing или obsolete; Quick Flow used для a complex platform; code review performed на same context that made the mistake; BMad-Help recommendation trusted without reading artifacts. Phase-output laundering is the most dangerous pattern: every phase produces a plausible file, but the files do не actually constrain the next phase.

Сравнительная заметка внутри темы: BMAD is role/рабочий маршрут/artifact heavy. Он отличается от Spec Kit на более сильной системой именованных агентов и более богатой фазовой таксономией; from Kiro на being installable между AI IDEs и using skills/рабочий маршрутs rather than one IDE задача surface; from GSD на deeper planning и role system, but weaker emphasis on minimal отгрузка передача контекста; from TDAD на relying on document cohesion и code review rather than скрытые тесты/mutation или impact map; from Constitutional SDD на не making security constitution the central artifact.
