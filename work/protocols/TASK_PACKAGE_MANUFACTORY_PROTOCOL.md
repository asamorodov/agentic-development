# Протокол пакетной мануфактуры

Статус: рабочая спецификация для механической сборки исполнительных пакетов из уже подготовленных планов целевых групп.

Назначение: отделить смысловое проектирование пакетов от их механической сборки. Этот протокол применяется, когда планы целевых групп уже написаны, проверены и приняты, а следующая задача — собрать по ним self-contained Python-gated пакеты и положить их в файловую систему.

## Основной принцип

Пакетная мануфактура не пишет теорию, не выбирает источники заново и не улучшает prompt-очереди. Она выполняет сборочную работу: берёт принятый `*_TARGET_GROUP_PLAN.md`, берёт перечисленные в нём источники и создаёт отдельный исполнительный пакет.

Если нужно улучшить план целевой группы, это делается до мануфактуры. Сборщик не должен чинить смысл плана по ходу сборки.

## Входы

Обычные входы мануфактуры:

- принятые планы целевых групп из `work/theory-writing/target-group-plans/`;
- протокол создания пакетов-заданий;
- протокол очередей writing-prompt-ов;
- шаблон плана целевой группы;
- все внутренние файлы, на которые ссылаются выбранные планы;
- общий read-only source bundle, если пакет мануфактуры self-contained.

Для одного собранного пакета имя архива берётся из имени плана без суффикса `_TARGET_GROUP_PLAN`:

```text
A2_SPECIFICATION_ADR_CONTRACT_TARGET_GROUP_PLAN.md
→ work/theory-writing/packages/A2_SPECIFICATION_ADR_CONTRACT.zip
```

Если пакет собирается из нескольких планов, имя должно описывать объединённую группу, а manifest обязан перечислить входящие планы.

## Механическая схема

Для мануфактуры с несколькими пакетами предпочтительна простая последовательность:

1. материализовать общий read-only source bundle;
2. по одному шагу собрать каждый пакет из одного плана;
3. после каждого пакета выполнить smoke-test первого и второго перехода runner-а;
4. в финале проверить все zip-архивы, записать manifest, verification report и resume note.

Не надо вести отдельный сложный реестр “источник → пакет”, если исходные планы уже содержат точные списки файлов для чтения. Безопаснее материализовать общий source bundle и дать каждому writing-пакету взять из него те файлы, которые указаны в его плане.

## Требования к собранному пакету

Собранный пакет должен быть self-contained, если явно не указано обратное. Он должен содержать `START.md`, generic Python-runner, opaque payload, нужные источники внутри payload, уникальные имена материализованных рабочих листов и финальную запись проверки/упаковки.

Видимый runner не должен содержать списки документов, названия планов, имена досье, `docs × passes`, `map_task`, финальную предметную логику или тексты внутренних prompt-ов.

## Smoke-test

Для каждого собранного пакета минимум: `unzip -t`, первый запуск runner-а создаёт рабочий лист, после фиктивных обязательных выходов второй запуск runner-а создаёт другой рабочий лист, видимый Python не раскрывает предметную структуру задания.

Если smoke-test не проходит, пакет считается failed. Нельзя скрывать это в финальном manifest.

## Аварийная упаковка

Пакет мануфактуры должен поддерживать аварийную упаковку текущего состояния: все уже собранные zip-архивы, текущие рабочие файлы, отчёты, state, последний материализованный рабочий лист, `INTERRUPTED.md` и `RESUME.md`. Это не нормальное завершение, а способ вернуть пользователю всё, что уже есть в файловой системе.

## Что не делает мануфактура

Мануфактура не заменяет target-group planning, source-pass, writing-pass, language-pass или repair-pass. Она не должна улучшать главы, дописывать источники, менять композиционную функцию узлов или переписывать prompt-очереди. Если такие изменения нужны, надо вернуться к планам целевых групп, а не чинить их внутри сборочного пакета.
