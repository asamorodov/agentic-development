# Протокол накопления материала из первоисточников в документ

Дата: 2026-06-08  
Статус: общий локальный протокол одного запуска над документом.  
Область применения: любой рабочий документ, который накапливает материал из первоисточников.

Этот протокол не описывает устройство досье, истории, главы или карты источников. Он задаёт общий ход одного запуска над явным `{имя документа}`.

Повторные запуски делает внешний оркестратор. Внутри этого протокола нет логики цикла.

## 0. Вход

Перед запуском должны быть заданы:

```text
{имя документа}
{номер прохода} — опционально
```

`{имя документа}` — конкретный файл, который нужно читать и изменять.

Если `{имя документа}` не задано, остановиться и запросить его.

## 1. Прочитать правила

Перед изменением документа прочитать релевантные правила:

```text
protocols/rules/language-style-rules.md
protocols/rules/russian-language.md
protocols/rules/terminology-and-translation.md
protocols/rules/human-technical-style.md
protocols/rules/english-source-handling.md
protocols/rules/source-and-provenance.md
```

Если есть жанровый prompt для конкретного типа документа, он уточняет этот протокол, но не заменяет его.

## 2. Понять документ

Прочитать `{имя документа}` целиком.

Определить:

- что документ уже знает;
- какие первоисточники уже указаны;
- какие формулировки поиска уже использовались;
- где в документе находятся разделы для источников, поисковых формулировок и иллюстраций;
- какую задачу документ выполняет в текущей работе.

Если таких разделов нет, добавить их аккуратно и без перестройки всего документа.

## 3. Обновить поиск первоисточников

Извлечь из документа уже известные первоисточники.

Понять, как они были найдены. Если в документе есть история поисковых формулировок, не повторять её механически.

При необходимости изменить критерий поиска: другим углом, другим термином, другой группой источников, другой формулировкой проблемы. Если дальнейшее изменение критерия почти наверняка даст шум, этот шаг можно пропустить, но надо кратко указать почему.

Новые поисковые формулировки записать в соответствующий раздел документа, чтобы будущие проходы не начинали поиск с нуля.

## 4. Найти и обновить список первоисточников

Провести поиск новых первоисточников по обновлённым формулировкам.

Собрать полный актуальный список:

```text
уже были в документе
+
найдены сейчас
+
оставлены как кандидаты / слабые источники
```

Обновить раздел источников в документе.

Раздел источников должен показывать не только ссылки, но и роль источника: зачем он нужен документу, насколько он первичный, какие ограничения у него есть.

Если источник найден, но не прочитан, нельзя делать по нему содержательные выводы. Записать его как `непрочитанный / требует проверки` и не использовать как подтверждение деталей в основном тексте документа.

## 5. Перенести детали из источников

Полностью прочитать релевантные первоисточники, насколько это возможно в текущем запуске.

Переносить в документ не пересказ “в среднем”, а детали, которые делают документ сильнее:

- конкретные механизмы;
- названия файлов, команд, фаз, ролей, артефактов;
- ограничения и оговорки;
- странные или неудобные детали;
- утверждения источника вместе с границами применимости;
- различия, привязанные к конкретному источнику;
- фактуру, которую легко потерять при гладком пересказе.

Не надо следовать длинному чек-листу извлечения. Модель должна использовать здравый смысл и жанровый prompt. Главное направление: **сохранять угловатость первоисточника и не сглаживать его до общего вывода**.

Детали добавлять структурно, в те места документа, где они усиливают смысл. Не сваливать всё в конец.

Если новая деталь противоречит прежнему тексту документа, не замазывать конфликт: явно поправить, уточнить или вынести как открытый вопрос.

## 6. Ссылки по месту

Источник должен появляться не только в общем списке источников.

Когда в документ добавляется конкретная деталь, утверждение или различие из источника, ссылка на источник должна стоять рядом с этой деталью.

Список источников всё равно должен быть обновлён полностью.

## 7. Иллюстрации

Проверить, есть ли в первоисточниках иллюстрации, диаграммы, схемы, screenshots or tables, которые могут быть полезны будущему тексту.

Добавить кандидатов в раздел иллюстраций:

```text
источник
что изображено
зачем может пригодиться
где в будущем тексте может использоваться
статус: кандидат / нужно скачать / не использовать без проверки прав
```

Не скачивать и не вставлять изображения, если текущая задача этого явно не требует.

## 8. Русский язык и стиль

Провести полноценный проход по языку и стилю.

Нужно:

- писать объяснения по-русски;
- убрать английский “клей”;
- оставить английский только для точных имён источников, инструментов, файлов, команд и устойчивых терминов;
- не полировать документ до пустого гладкого пересказа;
- не удалять фактуру источников;
- не заменять конкретику общими словами.

Если есть сомнение между красивой фразой и точной угловатой деталью, выбирать деталь.

## 9. Артефакты прохода

Если задан `{номер прохода}`, сохранить:

```text
passes/{безопасное_имя_документа}/{безопасное_имя_документа}_after_pass_{номер прохода}.md
passes/{безопасное_имя_документа}/{безопасное_имя_документа}_delta_{номер прохода}.md
passes/{безопасное_имя_документа}/{безопасное_имя_документа}_should_stop_{номер прохода}.txt
```

`{безопасное_имя_документа}` — имя без директорий и расширения.

Пример:

```text
work/dossiers/BMAD_METHOD_DOSSIER.md
→
passes/BMAD_METHOD_DOSSIER/BMAD_METHOD_DOSSIER_after_pass_03.md
passes/BMAD_METHOD_DOSSIER/BMAD_METHOD_DOSSIER_delta_03.md
passes/BMAD_METHOD_DOSSIER/BMAD_METHOD_DOSSIER_should_stop_03.txt
```

Такой namespace нужен, чтобы разные документы можно было пополнять параллельно без записи в одни и те же pass-файлы.

`after_pass` — полный текущий документ после прохода.

`delta` — отчёт о том, что изменено:

- какие источники добавлены;
- какие поисковые формулировки добавлены;
- какие детали перенесены;
- какие ссылки добавлены по месту;
- какие иллюстрации-кандидаты найдены;
- какие языковые и стилевые исправления сделаны;
- какие вопросы остались.

`should_stop`:

- `yes`, если новых содержательных деталей почти не найдено, поисковые углы исчерпаны и дальнейшие проходы скорее принесут шум;
- `no`, если есть смысл продолжать.

Нельзя писать `yes` только потому, что “документ выглядит достаточно хорошим”. Основание — именно исчерпание источников, поисковых углов и отсутствие новых содержательных деталей.

## 10. Что нельзя делать

- Не превращать документ в гладкий пересказ.
- Не удалять детали источников ради красоты.
- Не писать без чтения источников, что источник “подтверждает” мысль.
- Не делать выводы по источникам, которые не удалось прочитать.
- Не переносить источник только в общий список без ссылок по месту.
- Не запускать широкий поиск ради расширения корпуса, если он принесёт шум.
- Не менять назначение документа без отдельного человеческого подтверждения.
