# Протокол создания пакетов-заданий

Статус: рабочая спецификация для подготовки исполнительных архивов-заданий.

Назначение: описать, как из рабочих целей, документов и промптов собирать пакет-задание, который можно запускать короткой командой через `START.md`, не раскрывая исполнителю общий план работы как обычную таблицу шагов.

Этот документ описывает не конкретный пакет, а процесс его подготовки. Он применяется к документным задачам: русификации, repair-pass, написанию фрагментов теории, обработке глав, переносу фактуры, аудиту деградации и сборке результата. Для задач с реальной сборкой, тестами, Git-операциями или сложной кодовой интеграцией предпочтителен Codex / рабочая копия репозитория.

## 1. Базовые понятия

### Пакет-задание

Пакет-задание — архив, который содержит `START.md`, generic Python-скрипт, opaque payload и входные материалы, необходимые для выполнения выбранных целевых групп. Пользователь запускает его коротко:

```text
Открой START.md и выполни инструкцию.
```

Внутри пакета исполнитель должен видеть только текущий материализованный рабочий лист и явно разрешённые входы текущего листа. Общая последовательность работы не должна быть представлена как открытый `plan.md`, `queue.md`, список шагов, таблица документов или структура `docs × passes`.

### Целевая группа

Целевая группа — набор документов, которые обрабатываются пакетом вместе.

Чаще всего целевая группа содержит один целевой документ. Но группа может содержать несколько документов, если один проход должен править их согласованно: например, скелетон и план, глава и карта источников, несколько взаимосвязанных файлов технического атласа.

Один пакет может содержать несколько целевых групп. Группы обрабатываются последовательно. У каждой группы своя очередь промптов и свои требования к результатам.

### План целевой группы

План целевой группы — build-time документ, который полностью специфицирует часть будущего пакета. Он используется при сборке пакета, но не обязан попадать в исполнительный архив. Если он попадает в архив для диагностики, это должен быть отдельный debug-режим, а не обычный исполнительный режим.

План целевой группы состоит из трёх секций:

1. обрабатываемые файлы;
2. файлы для чтения во время работы;
3. очередь промптов.

Для очереди prompt-ов план должен содержать готовые финальные тексты, а не пересказ их намерения. Каждый prompt начинается с блока `Прочитай сначала:` и перечисляет только те источники, которые нужны именно для этого прохода. План должен быть точным для сборки пакета, но не должен раздуваться в методологический трактат.

По умолчанию все внутренние файлы, перечисленные в секциях «Обрабатываемые файлы» и «Файлы для чтения», упаковываются внутрь исполнительного архива: либо как исходное содержимое в opaque payload, либо как создаваемые будущие файлы с заданным начальным содержимым. Поэтому обычный пакет должен быть самодостаточным и запускаться в новом чате без дополнительной загрузки repo snapshot.

Repo-snapshot-bound режим допустим только как явное исключение: это должно быть прямо указано при постановке задачи и отражено в `START.md`.

### Конвенция имён и cache-папка

Планы целевых групп хранятся в каталоге:

```text
work/theory-writing/target-group-plans/
```

Имя каждого плана заканчивается на:

```text
_TARGET_GROUP_PLAN.md
```

Если план относится к одному будущему или существующему документу, имя плана начинается с имени этого документа без расширения. Например:

```text
work/theory-writing/fragments/A1_change_not_prompt.md
→ work/theory-writing/target-group-plans/A1_CHANGE_NOT_PROMPT_TARGET_GROUP_PLAN.md
```

Если план создаёт или изменяет несколько документов как одну целевую группу, начало имени описывает эту группу, но всё равно заканчивается на `_TARGET_GROUP_PLAN.md`. Пример:

```text
CORE_NODES_TARGET_GROUP_PLANS_GENERATION_TARGET_GROUP_PLAN.md
```

Собранные исполнительные пакеты хранятся рядом:

```text
work/theory-writing/packages/
```

Если пакет собран из одного плана целевой группы, имя архива равно имени плана без суффикса `_TARGET_GROUP_PLAN` и с расширением `.zip`:

```text
A1_CHANGE_NOT_PROMPT_TARGET_GROUP_PLAN.md
→ work/theory-writing/packages/A1_CHANGE_NOT_PROMPT.zip

CORE_NODES_TARGET_GROUP_PLANS_GENERATION_TARGET_GROUP_PLAN.md
→ work/theory-writing/packages/CORE_NODES_TARGET_GROUP_PLANS_GENERATION.zip
```

Если пакет собран из нескольких планов, его имя должно отражать объединённую группу, а рядом или внутри архива должен быть manifest, который перечисляет включённые планы целевых групп.


### Шаг / запись

Шаг пакета — один локальный рабочий лист, материализованный из opaque payload. На уровне реализации это запись payload: она содержит текущий текст инструкции, разрешённые входы, требуемые выходы и указатель на следующую запись.

Не финальный рабочий лист в конце предписывает снова запустить Python-скрипт. Финальный рабочий лист не предписывает запускать скрипт; он проверяет результат, собирает архив и завершает работу.


### Пакетная мануфактура

Если задача состоит не в написании текста, а в механической сборке нескольких исполнительных пакетов из уже принятых планов целевых групп, применяется отдельный протокол:

```text
work/protocols/TASK_PACKAGE_MANUFACTORY_PROTOCOL.md
```

Он фиксирует правило: сначала материализовать общий source bundle, затем собирать пакеты по одному плану за шаг, без повторного выбора источников и без изменения prompt-очередей.

### Специальный blueprint для статей атласа

Если пакет создаёт большую concept-first статью концептуально-технического атласа, перед подготовкой target-group plan нужно читать:

```text
work/theory-writing/ATLAS_ARTICLE_PACKAGE_BLUEPRINT.md
```

Такие пакеты не используют укороченную стандартную writing-очередь. По умолчанию они строятся как 25 рабочих проходов плюс финальная проверка, с несколькими source-depth pass-ами, двумя свободными проходами добора материала, отдельным visual asset layer, concept reinforcement, тремя общими редакторскими проходами и последующей языково-стилевой доводкой.

Для atlas article package нельзя задавать внутренний лимит объёма. Внешние источники разрешены, особенно для восстановления первоисточников и реальных image candidates. Локальные релевантные assets должны вставляться в статью; внешние реальные изображения фиксируются как candidates для будущего asset-pass, а не заменяются синтетическими схемами.

## 2. Двухфазная подготовка пакета

Подготовка сложного пакета идёт в две фазы.

### Фаза 1. Подготовка планов целевых групп

На этой фазе создаются планы целевых групп. Для простых пакетов, где один и тот же промпт повторяется над списком независимых файлов, планы могут быть сгенерированы почти механически. Для сложных пакетов планы пишутся явно.

Цель фазы 1 — не собрать архив, а зафиксировать состав работы:

- какие файлы будут редактироваться;
- какие файлы будут только читаться;
- какие готовые prompt-ы будут применены;
- какие источники каждый prompt читает в начале;
- где один проход должен править несколько файлов согласованно;
- какие артефакты должны доказывать выполнение каждого шага;
- где нормальная граница группы и что должно быть упаковано при обрыве.

Фаза 1 не должна бюрократизировать задачу. Список файлов и тексты prompt-ов должны быть точными, но сами prompt-ы должны оставаться короткими рабочими инструкциями, а не длинными чек-листами.

### Фаза 2. Сборка исполнительного пакета

На этой фазе выбранные планы целевых групп превращаются в opaque record chain.

Сборщик пакета:

1. читает планы целевых групп;
2. собирает содержимое всех существующих target-файлов и read-only файлов, перечисленных в планах;
3. создаёт пустые начальные версии для future target-файлов;
4. разворачивает очереди промптов в последовательность записей;
5. размещает записи групп строго последовательно: сначала вся первая группа, затем вся вторая и так далее;
6. для каждой записи задаёт материализуемые входы, текст локальной инструкции и обязательные выходы;
7. добавляет запись завершения группы, если группе нужен checkpoint;
8. добавляет финальную запись, которая проверяет и упаковывает результат;
9. кладёт структуру и входные материалы в opaque payload;
10. создаёт generic Python-скрипт-исполнитель, который не содержит логики конкретной задачи.


После разворачивания записей сборщик должен выполнить mechanical target-output audit по исходному target-group plan: каждый файл из раздела «Обрабатываемые файлы» должен быть либо создан как initial/future target, либо включён как existing target, либо явно помечен как blocked/deferred в final verification. Финальная запись пакета должна проверять существование всех target outputs и не должна позволять `ready_with_known_debts` при отсутствующем обязательном файле. Для self-contained пакетов сборщик также проверяет, что все required read-only inputs реально упакованы; для repo-snapshot-bound пакетов это исключение должно быть явно написано в `START.md`.

## 3. Формат плана целевой группы

План целевой группы — Markdown-документ с тремя обязательными секциями.

### 3.1. Обрабатываемые файлы

Секция фиксирует файлы, которые пакет должен создать или изменить.

Шаблон:

```markdown
## 1. Обрабатываемые файлы

- path: work/dossiers/PERSISTENT_WORK_GRAPH_MECHANISM_DOSSIER.md
  status: existing
  role: primary
  write_policy: replace-full-file
  result_policy: overlay-path
  notes: Основной документ группы.

- path: work/theory-writing/example.md
  status: future
  role: secondary
  write_policy: create-empty-then-edit
  result_policy: overlay-path
  initial_content: ""
  notes: Файл ещё не существует; пакет должен создать его.
```

Поля:

- `path` — путь в файловой системе репозитория или будущий путь;
- `status` — `existing` или `future`;
- `role` — `primary`, `secondary`, `supporting-output`;
- `write_policy` — `replace-full-file`, `append-section`, `create-empty-then-edit`, `patch-only`;
- `result_policy` — как результат попадает в итоговый архив: `overlay-path`, `result-only`, `diagnostic-only`;
- `initial_content` — только для будущих файлов;
- `notes` — зачем файл входит в группу.

Если целевых файлов несколько и они должны правиться согласованно, они остаются в одной целевой группе. Не надо разбивать их на независимые группы только ради удобства.

### 3.2. Файлы для чтения

Секция фиксирует файлы, к которым промпты могут обращаться во время работы, но которые не должны изменяться.

Шаблон:

```markdown
## 2. Файлы для чтения

- path: protocols/rules/russian-language.md
  required: true
  use: Языковые правила.

- path: work/skeletons/THEORETICAL_SYNTHESIS_REBUILT_SKELETON_V4.md
  required: true
  use: Архитектура будущей теоретической главы.

- path: work/reports/SPECIFICATION_METHODS_COMPARATIVE_SYNTHESIS.md
  required: false
  use: Дополнительная проверка сравнительного узла.
```

Поля:

- `path` — путь к файлу;
- `required` — можно ли продолжать, если файл отсутствует;
- `use` — зачем этот файл нужен.

Файлы из этой секции не должны попадать в выходной overlay как изменённые, если отдельный prompt явно не переводит их в секцию обрабатываемых файлов.

### 3.3. Очередь промптов

Секция фиксирует локальные задания для целевой группы. Здесь нужны готовые финальные тексты prompt-ов. Нельзя ограничиваться формулировками вроде “добавить фактуру” или “сделать repair-pass” без текста, который потом попадёт в исполнительный пакет.

Каждый prompt должен начинаться с блока `Прочитай сначала:`. В этот блок включаются только материалы, нужные для данного прохода. Для writing, редактуры, ссылок и переноса фактуры всегда включаются применимые языковые, стилевые и источниковые протоколы.

Шаблон для одного целевого файла:

```markdown
## 3. Очередь промптов

### P01

mode: single-target-edit
inputs:
  targets:
    - work/dossiers/PERSISTENT_WORK_GRAPH_MECHANISM_DOSSIER.md
  references:
    - protocols/rules/russian-language.md
outputs:
  targets:
    - work/dossiers/PERSISTENT_WORK_GRAPH_MECHANISM_DOSSIER.md
  artifacts:
    - local_report
    - checked_fragments
prompt: |
  Выполни редакторскую языковую правку текущего документа...
```

Шаблон для связанной правки нескольких целевых файлов:

```markdown
### P03

mode: linked-target-edit
inputs:
  targets:
    - work/skeletons/THEORETICAL_SYNTHESIS_REBUILT_SKELETON_V4.md
    - work/approved-ai-sdlc-plan.md
  references:
    - work/discourse.md
outputs:
  targets:
    - work/skeletons/THEORETICAL_SYNTHESIS_REBUILT_SKELETON_V4.md
    - work/approved-ai-sdlc-plan.md
  artifacts:
    - group_report
    - cross_file_consistency_check
prompt: |
  Обнови оба целевых файла как связанный комплект...
```

Обязательные поля:

- `mode` — `single-target-edit`, `linked-target-edit`, `audit`, `repair`, `checkpoint`, `finalize-group`;
- `inputs.targets` — целевые файлы текущего шага;
- `inputs.references` — read-only файлы текущего шага;
- `outputs.targets` — какие целевые файлы должны быть записаны после шага;
- `outputs.artifacts` — какие отчёты и проверочные артефакты обязательны;
- `prompt` — локальная инструкция.

Очередь промптов не должна содержать общий рассказ о будущем пакете. Каждый prompt должен описывать только локальное действие.

## 4. Один проход по нескольким целевым документам

Если несколько документов редактируются связанно, их нельзя имитировать как независимые шаги. Для этого используется `linked-target-edit`.

Такой шаг должен:

1. материализовать все целевые документы группы или их текущие версии;
2. материализовать только те read-only файлы, которые нужны текущему prompt;
3. требовать записи результата по каждому целевому документу;
4. требовать `cross_file_consistency_check`;
5. фиксировать, какие связи между файлами проверены;
6. не считать шаг завершённым, если хотя бы один целевой файл не записан;
7. если один из файлов не должен меняться, записать его без изменений и явно указать причину в отчёте.

Пример требований к отчёту linked-pass:

```text
## Связанные файлы
- <path A>
- <path B>

## Что должно быть согласовано
- термины;
- ссылки между файлами;
- нумерация глав;
- статус решений;
- список источников;
- future-path references.

## Что изменено по каждому файлу
...

## Что проверено между файлами
...

## Что оставлено без изменения и почему
...
```

Главное правило: связанный проход должен оставлять не только изменённые файлы, но и артефакт, доказывающий, что проверялись связи между ними.

## 5. Несколько целевых групп в одном пакете

Если пакет содержит несколько целевых групп, они обрабатываются последовательно. Нельзя перемешивать шаги разных групп, если план явно не задаёт общую группу.

Порядок сборки:

```text
group A: P01 → P02 → ... → group checkpoint
group B: P01 → P02 → ... → group checkpoint
group C: P01 → P02 → ... → group checkpoint
final package record
```

Групповой checkpoint должен создать:

- текущие итоговые версии целевых файлов группы;
- отчёт группы;
- список нерешённых мест;
- список файлов, которые должны попасть в итоговый overlay;
- при необходимости partial zip группы.

Групповой checkpoint может быть нормальной точкой диагностики, но не должен подменять финальное завершение всего пакета, если в пакете есть следующие группы.

## 6. Executor package: Python-gate и opaque record chain

Исполнительный пакет должен быть плоским и не раскрывать структуру задания через имена папок или файлов.

Минимальный состав:

```text
START.md
<opaque_runner>.py
<opaque_payload>.dat
```

Допускаются дополнительные непрозрачные файлы, если они не раскрывают структуру работы.

Видимый Python-скрипт должен быть generic materializer/state-machine. В нём не должно быть:

```text
списка документов;
числа проходов;
формулы docs × passes;
map_task;
group-by-document;
текста внутренних заданий;
текста финальной упаковки;
семантических имён групп;
логики конкретной предметной задачи.
```

Скрипт должен уметь только:

1. загрузить state;
2. декодировать текущую запись payload;
3. проверить, что обязательные выходы предыдущей записи существуют;
4. материализовать текущий рабочий лист и текущие входы;
5. записать state с указателем на следующую запись;
6. по emergency-команде упаковать текущее рабочее состояние.

Payload должен быть цепочкой записей, а не открытой таблицей `documents × passes`. Каждая запись содержит только локальную инструкцию, локальные входы, требования к выходам и указатель на следующую запись.

Имена материализованных рабочих листов должны быть уникальными для каждой записи. Нельзя много раз перезаписывать один и тот же instruction-файл: это выглядит как зависание даже при корректном продвижении state.

## 7. START.md и рабочие листы

`START.md` должен быть коротким:

```text
Выполни рабочее задание ниже.

Запусти:
python <runner>.py

Открой созданный .md-файл и выполни инструкцию.
```

Не надо писать в `START.md` про цепочку, очередь, будущие шаги, скрытую структуру или общий план.

Обычный не финальный рабочий лист должен содержать:

- разрешённые входы;
- запрет на открытие неразрешённых файлов;
- локальную задачу;
- требуемые выходы;
- требования к отчёту;
- команду запуска Python-скрипта в конце.

Финальный рабочий лист не содержит команды повторного запуска. Он проверяет, собирает итоговый архив и завершает работу.

## 8. Обрыв работы и аварийная упаковка

Каждый пакет должен предусматривать аварийную упаковку. Если исполнитель не может продолжить из-за лимита времени, лимита сообщений, системного сбоя, зависания инструмента или явного прерывания пользователем, он должен упаковать всё доступное рабочее состояние и отдать архив пользователю.

Generic runner должен поддерживать emergency-команду:

```text
python <runner>.py emergency
```

или эквивалентную команду, указанную в `START.md`.

Аварийный архив должен включать:

- все файлы, созданные в рабочем каталоге пакета;
- текущий state;
- текущий материализованный рабочий лист;
- все промежуточные результаты;
- все локальные отчёты;
- payload и runner, если это нужно для диагностики;
- `INTERRUPTED.md` с описанием, на чём работа остановилась;
- `RESUME.md` с инструкцией, как продолжить или как разобрать partial result.

Если пакет repo-snapshot-bound и работает с путями репозитория, аварийный архив должен дополнительно включить partial overlay по уже записанным целевым файлам. Если собрать overlay невозможно, нужно прямо указать это в `INTERRUPTED.md`, а не выдавать частичный результат за финальный.

Аварийная упаковка не считается нормальным завершением. Нормальное завершение определяется только финальной записью, которая создаёт финальный result zip и completion marker.

## 9. Нормальное завершение и result zip

Финальная запись должна собрать result zip.

Для repo-snapshot-bound пакета result zip должен содержать:

```text
overlay/
  <repository paths of changed target files>
reports/
  local and group reports
manifest/
  MANIFEST.md
  VERIFY.md
  RESUME.md
```

Для self-contained пакета result zip должен содержать:

```text
final/
  final target files
reports/
manifest/
  MANIFEST.md
  VERIFY.md
  RESUME.md
```

`VERIFY.md` должен явно проверить:

- какие целевые группы завершены;
- какие целевые файлы записаны;
- какие отчёты существуют;
- есть ли missing outputs;
- есть ли suspicious repeated outputs;
- есть ли emergency/interrupted state;
- какие файлы требуют ручного review.

Финальная инструкция должна создать completion marker, например:

```text
COMPLETE.txt
```

Если `COMPLETE.txt` отсутствует, запуск считается незавершённым.

## 10. Контроль честности и качества

Пакет не должен полагаться на фразу “работай честно”. Честность обеспечивается артефактами.

Каждый содержательный шаг должен требовать:

- результат;
- локальный отчёт;
- список конкретных проверенных или изменённых зон;
- список сохранённых деталей;
- список того, что оставлено без изменения и почему;
- при связанной правке нескольких файлов — cross-file consistency check.

Runner или финальная проверка должны помечать suspicious state, если:

- несколько pass-результатов побайтно одинаковы, хотя отчёты заявляют изменения;
- отчёты повторяются почти дословно;
- отчёт не содержит конкретных фрагментов;
- отсутствует результат по одному из target files;
- есть intermediate state без completion marker;
- группа завершена без group report;
- linked-target шаг не имеет cross-file consistency check.

Suspicious state не всегда означает плохой результат, но результат нельзя считать чисто принятым без ручного review.

## 11. Типы пакетов

### Self-contained package

Это режим по умолчанию. Все существующие target-файлы и все read-only файлы, перечисленные в планах целевых групп, находятся внутри opaque payload или внутри архива. Такой пакет можно запускать в новом чате без repo snapshot.

Плюсы:

- переносимость;
- простая стартовая команда;
- исполнитель работает с точно теми источниками, которые были утверждены в плане;
- не нужен доступ к репозиторию;
- меньше риск, что целевой чат возьмёт не тот baseline.

Минусы:

- целевые документы становятся копиями;
- результат нужно затем переносить в репозиторий или собирать overlay;
- если после сборки пакета baseline изменился, пакет не знает об этих изменениях.

### Repo-snapshot-bound package

Пакет предполагает, что пользователь загрузил в целевой чат срез репозитория. В payload лежат инструкции и рабочая логика, но не копии целевых документов и источников. Рабочие листы обращаются к путям внутри загруженного snapshot.

Это не режим по умолчанию. Его используют только по явному решению, когда важнее работать с живым срезом репозитория, чем получить полностью независимый исполнительный архив.

Плюсы:

- нет копий исходников внутри пакета;
- результат естественно собирается как overlay;
- пути совпадают с реальным репозиторием.

Минусы:

- пакет не работает без repo snapshot;
- START.md должен явно требовать наличие snapshot, но не раскрывать полный план;
- выше риск несовпадения между тем, что планировалось при сборке пакета, и тем, что реально оказалось в загруженном snapshot;
- аварийная упаковка должна уметь вернуть partial overlay.

### Current-chat-bound package

Пакет опирается на файлы, уже загруженные в текущий чат. Этот режим допустим для экспериментов, но плохо переносится и не должен быть режимом по умолчанию.

## 12. Как выбирать размер пакета

Длинные однообразные цепочки повышают риск:

- имитации проходов;
- самовольной остановки на красивой промежуточной границе;
- потери состояния;
- ошибочного вывода, что пакет завис.

Для реальной работы предпочтительно:

```text
один пакет = одна целевая группа
```

или максимум несколько небольших целевых групп, если каждая группа имеет явный checkpoint.

Для длинных задач лучше собрать серию пакетов, чем один пакет на десятки однотипных шагов. Если всё же нужен большой пакет, он должен иметь group checkpoints, emergency packaging и финальный completion marker.

## 13. Минимальный чеклист сборщика

Перед выдачей пакета сборщик должен проверить:

1. В executor archive нет `plan.md`, `queue.md`, `steps.md`, `instructions/`, `passes/`, `workflow/` или других раскрывающих структур.
2. Видимый Python не содержит списка документов, количества проходов, текста промптов, финальной упаковочной логики или subject-specific code.
3. Payload построен как opaque record chain.
4. Каждый non-final record создаёт рабочий лист с командой повторного запуска Python.
5. Final record не содержит команды повторного запуска.
6. У каждого record есть required outputs.
7. Для linked-target steps есть cross-file consistency report.
8. Есть emergency packaging command.
9. Есть финальный `COMPLETE.txt` или эквивалентный completion marker.
10. Smoke-test проверяет минимум два перехода и уникальные имена материализованных рабочих листов.
11. По умолчанию все target/read-only файлы из планов целевых групп включены в payload или архив.
12. Если пакет repo-snapshot-bound, в `START.md` сказано, что repo snapshot должен быть загружен, но не раскрывается общий план.
13. Пользователю выдаётся только архив и короткая стартовая команда.

## 14. Очереди промптов для написания глав и подглав

Для writing-пакетов, которые создают несущие фрагменты, подглавы или главы теоретического раздела, формат целевых групп и сборка opaque record chain остаются такими же, но содержательная очередь prompt-ов задаётся отдельным протоколом:

```text
work/protocols/THEORY_WRITING_PROMPT_QUEUE_PROTOCOL.md
```

Этот протокол определяет первичный writing-prompt, проходы добавления фактуры, системное выравнивание, repair-pass, языковые и стилевые проходы, работу с внутренними и внешними источниками, figure-кандидатами и linked-target-edit для нескольких целевых документов.
