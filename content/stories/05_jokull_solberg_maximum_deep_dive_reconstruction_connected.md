## Jökull Sólberg: PR как управляемый объект, `skills` как рабочая память и агентская автоматизация ожидания

История Jökull Sólberg показывает участок агентской разработки, который часто начинается уже после написания кода. Многие истории корпуса сосредоточены на входе в задачу: исследование, план, контекст, рабочее дерево, навык, браузер, права. Jökull переносит внимание на PR как объект сопровождения. После `push` остаётся цепочка ожиданий, проверок, замечаний, исправлений и повторных прогонов. Его `/babysit-pr` превращает эту цепочку в процедуру для Claude Code.

Общая проблема здесь близка Jesse Vincent: замечание проверяющего не должно автоматически становиться задачей. У Vincent агент сначала оценивает внешнего проверяющего: понимает ли он кодовую базу, какие предложенные исправления верны, что нужно отклонить или вынести человеку. У Jökull это превращается в рабочую классификацию [Fix / Dismiss / Escalate](#fieldbook--pr-triage-fix-dismiss-escalate). Автоматическая проверка полезна, но между сигналом и изменением должен стоять разбор.

Эта история также связана с Calvin French-Owen и Mae Capozzi. У Calvin Bugbot, Codex review, [CI](#handbook--pr-review-loop) и preview deploys должны замыкаться обратно в рабочий цикл, иначе человек остаётся сопровождающим очереди PR. У Mae похожая проблема выходит на платформенный уровень: CI, наблюдаемость, зависимости и командное внедрение становятся ограничителями скорости. Jökull даёт узкий, но очень конкретный ответ: агент сопровождает PR через CI, [Greptile](#handbook--pr-review-loop), `codex review`, повторные исправления и критерий готовности к слиянию.

Различие Jökull в том, что он показывает навыки не как подготовку к разработке, а как автоматизацию ожидания и переноса состояния между системами. PR, GitHub issue, gist, `.ics`, ридер или результат теста удобства становятся рабочими объектами, которые агент проводит через внешние сервисы и возвращает человеку в проверяемой форме.

### 1. Рабочая среда: TripToJapan, Next.js, tRPC, Drizzle, monorepo

Jökull Sólberg — сооснователь и технический директор TripToJapan.com. В “How I Use Claude Code” он описывает работу с Claude Code на платформе бронирования путешествий: Next.js, tRPC, Drizzle, monorepo, фичи, баги, CI, PR, GitHub issues и автоматическая проверка кода.

Это важно не как справочная деталь, а как условие всего процесса. Jökull работает с живой кодовой базой, где изменение проходит через несколько систем и оставляет след:

- код меняется в monorepo;
- GitHub issue фиксирует проблему или будущую работу;
- PR становится единицей поставки;
- CI проверяет формальные условия;
- [Greptile](#handbook--review-comments-as-signals) и Codex дают разные виды автоматической проверки;
- человек принимает решения там, где нужен контекст продукта или архитектуры.

Поэтому его история не сводится к генерации кода. Код — только часть работы. Практическая проблема часто возникает позже: PR нужно провести через ожидания, проверки, замечания, повторные правки и состояние готовности к слиянию.

### 2. Базовый режим: несколько сессий Claude Code и ограниченное внимание человека

Обычно у Jökull одновременно работают две-три сессии Claude Code в отдельных вкладках терминала. Дополнительно он держит постоянную сессию Claude Code на домашнем Mac mini через SSH и `tmux`: ноутбук можно закрыть, а процесс продолжит работу.

Одна сессия может делать запланированную фичу или исправление. Другая — создавать или уточнять GitHub issue с учётом кода. Третья — ждать CI или автоматическую проверку PR. Иногда два агента работают вокруг одной фичи: один исследует подход, другой движется к реализации.

Этот режим не предполагает бесконечного параллелизма. Две-три сессии — примерно тот объём переключения, который Jökull считает управляемым. Каждый агент живёт в отдельном контексте. Jökull не ведёт один огромный разговор, где смешаны фичи, баги, PR, переводы и проверки. Он держит несколько изолированных рабочих потоков и периодически возвращается к каждому.

Это задаёт важное ограничение. Агентская автоматизация должна уменьшать ручное сопровождение, а не создавать столько результатов, что человек перестаёт успевать их понимать.

### 3. Основная проблема: разработчик становится диспетчером ожидания

В PR-цикле Jökull замечает типичный дефект агентской разработки. Модель ускоряет создание изменения, но после `push` остаётся цепочка ожиданий.

Ручной PR-процесс выглядит так:

1. отправить код;
2. дождаться CI;
3. дождаться Greptile;
4. прочитать замечания;
5. исправить валидные пункты;
6. снова отправить изменения;
7. снова ждать CI и проверку;
8. повторить.

Каждый шаг небольшой. Проблема в форме процесса: несколько минут ожидания, короткое действие, снова ожидание. Человек либо держит PR в голове, либо переключается и забывает вернуться.

Многие агентские процессы улучшают генерацию кода, но оставляют человека живым таймером между CI, GitHub, автоматическим проверяющим и локальной проверкой. `/babysit-pr` появляется как ответ на эту проблему. Он не заменяет разработчика в архитектурном решении. Он убирает роль человека как диспетчера повторяющегося ожидания.

### 4. Перед PR: GitHub issue как короткое исследование

До PR у Jökull есть ещё один повторяемый процесс: создание GitHub issue через Claude Code и `gh` CLI.

Когда нужно завести issue, он не просит агента просто написать текст. Агент сначала должен:

- найти возможные дубликаты;
- прочитать релевантные пути в коде;
- проверить, что баг находится именно там, где кажется;
- уточнить терминологию по реальным моделям, компонентам и сервисам;
- добавить шаги воспроизведения с конкретными файлами и строками.

В результате issue становится коротким исследовательским артефактом, а не записью по памяти. Claude часто находит детали, которые человек мог пропустить: функция вызывается ещё из другого места, рядом есть связанный TODO, проблема затрагивает не один компонент, а поток данных.

Эта стадия показывает общий принцип Jökull. Если рабочий объект будет жить дальше, его стоит создавать из проверенного контекста. Issue уже должно содержать немного исследования. PR позже должен быть объектом сопровождения, а не просто дифф. Gist должен быть читаемым артефактом, а не куском чата.

### 5. `/pr-feedback`: промежуточная форма автоматизации

До `/babysit-pr` у Jökull был `/pr-feedback` — slash command для работы с замечаниями после PR.

Он использует Greptile как автоматического проверяющего. Greptile оставляет inline comments в PR и, по словам Jökull, ловит не только косметику: circular зависимости, race conditions, missing error handling и очевидные ошибки. Большая часть комментариев Greptile приводит к реальным изменениям. Когда Greptile ошибается, Claude часто способен распознать ложное замечание и оставить короткое объяснение, почему пункт пропущен.

`/pr-feedback` уже закрывал важную часть цикла:

- ждал завершения CI через `gh pr checks --watch`;
- получал проверку Greptile через MCP;
- обрабатывал замечания;
- отправлял исправления;
- запускал повторную проверку;
- повторял цикл до чистого состояния.

Но `/pr-feedback` ещё не задавал полный путь от `push` до готовности к слиянию. Он автоматизировал полезный фрагмент, а `/babysit-pr` стал следующей стадией: агент сопровождает PR как самостоятельный объект.

### 6. `/babysit-pr`: PR как объект, который агент ведёт до проверяемого выхода


`/babysit-pr` — Claude Code `skill`: markdown-файл с процедурой. В нём около 170 строк. Сильная часть здесь не в объёме кода, а в форме: процесс выражен как инструкция, которую агент может прочитать и выполнить.

В сжатом виде `/babysit-pr` делает следующее:

1. определяет текущий PR через `gh pr view`;
2. параллельно ждёт CI и Greptile;
3. обрабатывает тот источник, который завершился раньше, пока второй ещё работает;
4. запускает локальную проверку `codex review --base main`;
5. классифицирует все замечания как [Fix / Dismiss / Escalate](#fieldbook--pr-triage-fix-dismiss-escalate);
6. исправляет валидные пункты;
7. проверяет изменения через линтер;
8. делает `commit` и `push`;
9. запускает повторную проверку;
10. повторяет цикл до чистого состояния или до лимита итераций.

У процедуры есть критерий выхода: CI зелёный, новых неразобранных issues нет, PR готов к слиянию. Есть и ограничитель: максимум три итерации. Это защищает процесс от бесконечного ремонта. Если PR не очищается после нескольких проходов, задачу, вероятно, должен посмотреть человек.

Главное решение в этой процедуре — агент оценивает обратную связь перед изменением кода. Автоматическая проверка не становится приказом. Результат `codex review` тоже требует оценки. CI ближе к объективному сигналу, но и он может падать из-за нестабильного теста или инфраструктуры. Поэтому между сигналом и изменением стоит разбор.

### 7. Fix / Dismiss / Escalate: защита от механического послушания


В skill-файле есть простая таблица разбора замечаний: валидное и локально исправимое замечание чинится; ошибочное или основанное на непонимании отклоняется с причиной; требующее человеческого решения выносится пользователю.

Эта маленькая классификация меняет характер автономности. Без неё агент легко становится исполнительным продолжением автоматического проверяющего: комментарий появился, значит нужно менять код. Тогда одна модель начинает обслуживать другую, а человек теряет смысловой контроль над изменением.

У Jökull каждый пункт должен получить решение:

- **Fix** — замечание действительно указывает на проблему, исправление понятно, область изменения допустима;
- **Dismiss** — замечание неверно, основано на неправильном предположении или не относится к текущему изменению;
- **Escalate** — нужен человек: продуктовая граница, архитектурный выбор, риск выхода за scope.

Правила намеренно асимметричны. Ошибки CI чинятся всегда. Замечания Codex по умолчанию склоняются к Fix, потому что локальная проверка часто ловит логические разрывы. Комментарии Greptile оцениваются отдельно. Малозначимые пункты отклоняются, если их нельзя исправить совсем тривиально.

Такой режим даёт управляемую автономность. Агент может двигать PR, но обязан фиксировать, какие замечания принял, какие отклонил и где нужен человек.


[Fix / Dismiss / Escalate](#handbook--review-comments-as-signals) — близкий родственник приёма Jesse Vincent с оценкой внешнего проверяющего. В обоих случаях проверочный комментарий рассматривается как сигнал, а не приказ. Это защищает от режима, где одна модель механически обслуживает другую, а человек видит только растущий список “исправлений”.

### 8. Три источника проверки: CI, Greptile и Codex

Jökull запускает три источника проверки, потому что они ловят разные классы проблем.

**CI** ловит объективные сбои: ошибки типов, нарушения линтинга и регрессии тестов. Это самый жёсткий слой, но он видит только проверяемые формальные ошибки.

**Greptile** читает PR в контексте кодовой базы и может находить архитектурные проблемы, но иногда шумит. Его замечания полезны, однако их нельзя принимать механически.

**Codex review** через `codex review --base main` даёт независимый локальный взгляд. Jökull описывает Codex в этом режиме как почти педантичный инструмент. При написании кода такая педантичность может раздражать, а при проверке она полезна: Codex ловит несогласованность названий, пропущенные edge cases и тонкие логические ошибки.

Здесь нет одного главного проверяющего. Ценность возникает из низкого пересечения ошибок. Greptile может одобрить PR, а Codex найдёт реальный баг. В другой день может быть наоборот. Несколько фильтров с разными профилями ошибок дают больше пользы, чем один универсальный проверяющий.

У такого подхода есть цена: больше источников проверки дают больше шума. Поэтому разбор замечаний — обязательная часть схемы. Без него несколько проверяющих просто увеличили бы количество пунктов, которые кто-то должен механически разгребать.

### 9. Реальный пример: баг в синхронизации публикации тура

В “Babysitting PRs With Claude Code” Jökull приводит пример, который хорошо показывает смысл всей процедуры.

Он отправил исправление бага в синхронизации публикации тура. `/babysit-pr` сделал два прохода.

Первый проход:

- CI был зелёным: build, lint, typecheck, тесты;
- Greptile дал высокую уверенность и одно observation про best-effort webhook delivery;
- это замечание было отклонено как эксплуатационный вопрос, а не ошибка кода;
- Codex нашёл два пункта, оба оказались реальными.

Первый пункт от Codex: текст в admin UI говорил, что unpublished tours “will still be accessible”, но изменение делало такие туры 404. Значит, текст интерфейса больше не соответствовал поведению.

Второй пункт был серьёзнее. Webhook синхронизировал `isPublished=false` при unpublish в DatoCMS, но не восстанавливал `isPublished=true` при republish. В цикле unpublish → republish тур мог остаться в неправильном состоянии.

Оба пункта были исправлены, изменения отправлены, повторная проверка запущена. На втором проходе CI остался зелёным, Greptile новых issues не дал, auto-merge был поставлен в очередь. Весь путь от `push` до состояния готовности к слиянию занял около десяти минут, большую часть времени занял CI.

Скорость здесь не главный вывод. Формальные проверки и Greptile пропустили две смысловые проблемы: несоответствие текста интерфейса новому поведению и разрыв в жизненном цикле publish/unpublish. Codex как второй проверяющий увидел логическую дыру. Разбор замечаний позволил принять реальные пункты и не превратить процесс в слепое исправление всего подряд.

### 10. Почему `/babysit-pr` — не автоматический исправитель

Наивная версия процесса выглядела бы так: проверяющий бот оставил комментарии, агент исправил всё, CI прошёл, PR готов. У Jökull процесс устроен иначе.

`/babysit-pr` разделяет три уровня:

- **сбой проверки** — CI упал, typecheck не прошёл, тест сломан;
- **замечание проверяющего** — Greptile или Codex считает, что есть проблема;
- **решение о действии** — агент принимает пункт, отклоняет его или выносит человеку.

Это разделение защищает процесс от механического послушания. Проверка может быть полезной и ошибаться. Агент может двигать работу вперёд и одновременно слишком охотно чинить лишнее. Человеку не нужно сопровождать каждую минуту ожидания, но он должен оставаться владельцем решений, где требуется оценка смысла.

Зрелость `/babysit-pr` состоит именно в этом: автономность выдаётся на повторяемую цепочку с критериями выхода, ограничением итераций и классификацией спорных сигналов.


Три источника проверки у Jökull хорошо дополняют Calvin French-Owen и Mae Capozzi. Calvin пытается замкнуть Bugbot, Codex review, CI и preview deploys обратно в цикл реализации. Mae смотрит на похожую проблему на уровне платформы: чем больше агентских PR, тем важнее быстрые проверки, наблюдаемость и автоматическое разделение механического от человеческого решения.

### 11. `skill` как переносимая форма процесса

`/babysit-pr` показывает более общий приём: повторяемая процедура становится `skill`. Это файл, который агент может открыть и выполнить, а не одноразовая подсказка в текущем чате.

У Jökull `skill` обычно содержит:

- условие применения;
- входные данные;
- последовательность действий;
- нужные инструменты;
- формат результата;
- правила проверки;
- примеры команд;
- ограничения, когда нужно остановиться или спросить человека.

Такой `skill` отличается от длинного запроса. Запрос живёт в текущей сессии. Skill становится частью рабочей среды. Новая сессия или другой агент может использовать его позже.

В “How I Use Claude Code” Jökull отдельно описывает закрытие цикла после задачи. Когда агент в ходе работы понял новую процедуру, крайний случай или доменное правило, это знание нужно перенести в устойчивое место. Он использует два характерных запроса:

```text
Summarize everything you learned and create a new Claude skill
```

```text
Add the context I explained to the right CLAUDE.md
```

Смысл в том, что рабочая память пополняется не только до задачи, но и после неё. Если агенту пришлось объяснять, как устроен поток публикации, как проверять i18n или как правильно сопровождать PR, это объяснение не должно умереть вместе с сессией. Оно должно попасть в `skill`, локальный `CLAUDE.md` или другой устойчивый носитель. Иначе следующий агент снова начнёт с нуля.

Поэтому связанные статьи важны для основной истории. `/user-testing`, `icelandic-data`, отправка статьи на ридер, публикация gist и Code Mode для MCP — разные применения одной формы. Jökull переносит ручную цепочку в процедуру, которую агент может выполнить повторно.


`/babysit-pr` показывает навык не как подготовку к коду, а как сопровождение рабочего объекта. Это роднит Jökull с Matt Pocock и Jesse Vincent, но отличается назначением. У Pocock навыки чаще чинят вход в задачу или форму исполнения. У Vincent Superpowers удерживают крупный процесс. У Jökull навык ведёт PR через внешние системы и ожидания.

### 12. Управление контекстом: изоляция вместо бесконечного накопления

В “How I Use Claude Code” Jökull отдельно пишет про управление контекстом. Когда длинная сессия подходит к лимиту — остаётся примерно 5–10% окна, — он не пытается выжать из неё ещё один кусок работы. По его опыту, лучше перейти в свежий контекст.

Схема такая:

1. попросить Claude сделать конкретный план с проверяемыми шагами;
2. разбить крупную работу на куски, которые можно выполнять параллельно;
3. запустить эти куски как отдельные подагенты;
4. вернуть в родительский поток сжатые результаты.

В источнике есть более точная форма выхода из длинной сессии: не продолжать в этом же потоке, а сначала сделать план на 3–7 шагов. Для каждой подзадачи агент должен указать цель, входные данные — файлы, пути, уже найденные факты — и ожидаемый результат: patch, notes или commands to run. Это превращает почти исчерпанный контекст в диспетчерский документ для свежих подагентов.

Jökull также упоминает более агрессивное управление забыванием. Контекст нужно не только переносить и сжимать, но иногда прямо говорить агенту, что забыть: какие ранние гипотезы больше неактуальны, какие направления закрыты, какие промежуточные детали не нужно тащить дальше. Это отдельный приём. Без него новая сессия может унаследовать не только полезный план, но и шум старого поиска.

Он считает это лучше, чем “сбросить всё в файл и потом восстановить”. Каждый sub-task стартует в чистом контексте, остаётся сфокусированным и не тащит с собой шум длинной отладочной сессии.

Эта логика повторяется по всей истории:

- отдельные сессии Claude Code для параллельных задач;
- подагенты для чистых контекстов;
- i18n-agent вместо сотен строк переводческих правил в `CLAUDE.md`;
- `skills` вместо постоянной загрузки всего знания;
- `devcontainer` как изоляция прав и файловой системы;
- `рабочее дерево` как будущая изоляция параллельной записи.

Jökull не пытается решить всё большим контекстным окном. Он раскладывает знание и работу по разным носителям: постоянные соглашения, специализированные навыки, отдельные агенты, отдельные контейнеры, отдельные рабочие деревья.

### 13. Специализированные агенты: профиль задачи важнее роли

Jökull настраивает custom agents под конкретные повторяемые задачи.

`code-validator` на Haiku запускает `pnpm lint:fix` и `pnpm turbo typecheck`, затем сообщает проблемы. Больше ничего. Haiku выбран потому, что проверка не требует большой интеллектуальности. Здесь важны скорость, низкая стоимость и отсутствие лишней инициативы.

`i18n-translator` знает шесть locales TripToJapan, ICU message format, next-intl, правила пространств имён, `jq`-паттерны для массовых операций, случаи, когда `eslint-disable` допустим, и когда текст, видимый пользователю, должен быть вынесен в переводы. Этот агент появился из конкретной проблемы: переводческие инструкции в `next/app/CLAUDE.md` хорошо работали, но занимали сотни строк и загрязняли контекст при задачах, не связанных с переводами.

`software-architect-analyzer` на Opus нужен для планирования сложных фич. У него есть запрос в стиле “ultrathink”, который заставляет провести тщательный анализ до предложений. Для маленьких задач это избыточно, для архитектурных решений полезно.

Смысл специализированных агентов состоит в выборе правильного профиля для задачи:

- дешёвый и ограниченный агент для механической проверки;
- специализированный агент для домена с большим количеством правил;
- дорогая модель для архитектурного анализа.

### 14. i18n-specialist: доменное знание вынесено из общего контекста

Gist с `i18n-translator.md` показывает, насколько конкретным может быть специализированный агент.

В нём зафиксированы:

- шесть locales: `en`, `ja-JP`, `ko-KR`, `th-TH`, `zh-CN`, `zh-TW`;
- структура файлов `apps/next/messages/`;
- проверка через `pnpm i18n:check`;
- запрет на текст, видимый пользователю и зашитый в код, в `apps/next/app/[locale]`;
- пространства имён вроде `Common`, `Validation`, `Tours`, `Checkout`, `Cart`, `Accessibility`;
- соглашения по именованию ключей;
- ICU pluralization, select, rich text и параметры;
- культурные различия для японского, корейского, китайского и тайского;
- массовые операции через `jq`;
- аварийное восстановление JSON-файлов;
- критерии качества переводов.

Такое знание плохо держать в общем `CLAUDE.md`. Оно слишком длинное, слишком специализированное и редко нужно для обычной задачи. Когда задача касается переводов, оно должно быть доступно полностью.

Эта секция дополняет историю `/babysit-pr` как второй пример той же архитектуры. Повторяемое специализированное знание выносится из одноразового разговора и из постоянного общего контекста в вызываемый рабочий блок.

### 15. `CLAUDE.md` / `AGENTS.md`: соглашения как контракт для агента

В “The Missing Next.js AGENTS.md File” Jökull формулирует другой слой: ИИ-ассистенты хорошо пишут код, но плохо знают именно ваши проектные соглашения. Без явного файла они будут использовать Server Actions, когда команда предпочитает API routes; `useEffect`, когда лучше React Server Components; auth checks в middleware, когда проверка должна быть в page components; loading states для данных, которые должны грузиться server-side.

Его ответ — `CLAUDE.md`, `AGENTS.md` или похожий файл, который фиксирует решения команды.

Это не общая справка по Next.js. Это файл с явно заданными проектными соглашениями. В нём зафиксированы решения:

- pages — Server Components;
- данные и session validation выполняются server-side;
- Client Components получают props, а не промисы;
- `useEffect` не используется для data fetching, auth checks или derived state;
- Server Actions не используются, вместо них Hono RPC;
- состояние, которое должно переживать refresh или быть доступным по ссылке, хранится в URL;
- для изменений данных в Server Components вызывается `router.refresh()`;
- конкретные библиотеки вроде `nuqs`, `iron-session`, `gql.tada`, `@t3-oss/env-nextjs`, `better-result`, `next-intl`, `tailwind-variants`, `intentui`, `oxlint`, `oxfmt`.

Эти мнения не универсальны. Server Actions или middleware auth могут быть нормальным выбором в другом проекте. Агенту нужен контракт локальных соглашений. Без такого файла каждый PR, созданный с помощью ИИ, становится броском монеты: совпадёт ли типовое решение модели с правилами конкретной команды.

Это важный слой рядом со skills. `CLAUDE.md` / `AGENTS.md` держат устойчивые проектные решения. Skills держат процедуры и специализированные домены. Вместе они образуют рабочую память агента.

### 16. Типичные ошибки агентов для разработки как материал для правил и проверки

В “Mistakes Coding Agents Make Frequently” Jökull фиксирует повторяющиеся ошибки моделей. Это не просто список раздражающих случаев. Такой список можно использовать при настройке `CLAUDE.md`, при написании `skills` и при проверке PR: он показывает, какие короткие пути модель выбирает слишком охотно.

Основные случаи у него такие.

**Злоупотребление `useEffect`.** Агент видит требование “обновить X при изменении Y” и переводит его в императивную форму “сигнал → действие”. В React это часто приводит к `useEffect`, хотя значение можно вычислить во время прохода рендера или обработать в обработчике события. Такой код может работать, но он усложняет состояние и добавляет лишние места для ошибок.

**`as any`.** Агент стремится убрать ошибку TypeScript самым коротким способом. Вместо того чтобы проследить контракт данных до источника, он добавляет `as any` и делает ошибку невидимой для компилятора. Видимая ошибка сборки превращается в скрытый риск во время выполнения.

**Преждевременная логика обратной совместимости.** Агент добавляет ветки миграции, резервный разбор данных и поддержку старого формата даже там, где фича ещё не запущена, PR не слит и внешних потребителей нет. Он применяет общий инженерный принцип “не ломать существующих пользователей” к ситуации, где таких пользователей ещё не существует.

**Комментарии с ходом рассуждения.** Агент оставляет в коде комментарии, которые полезны во время проверки, но бессмысленны после слияния. Такие комментарии объясняют временное решение или ход работы над PR, а не устойчивое поведение кода.

**Широкие `try/catch`.** Агент оборачивает функции или обработчики маршрутов в общий `try/catch`, логирует ошибку и возвращает обобщённый ответ 500. Это ломает более высокий уровень обработки ошибок, ухудшает наблюдаемость и стирает различия между ошибкой валидации, проблемой авторизации, отсутствующей сущностью и настоящим падением сервера.

Эта негативная карта объясняет, почему Jökull так ценит локальную проверку Codex, проектные соглашения и специализированные инструкции. Модель часто чинит видимый симптом кратчайшим синтаксическим путём. Хороший процесс должен подталкивать её к правильному уровню решения: проверить источник типа, посмотреть существующий поток обработки ошибок, понять стадию фичи, отличить временный комментарий от устойчивого пояснения.

### 17. Безопасность и изоляция: `devcontainer`, `рабочее дерево`, YOLO mode

Автономия агента упирается в область возможного ущерба. Jökull не пытается решить эту проблему только текстовыми запретами. Там, где агенту нужно дать больше свободы, он ищет способ уменьшить последствия ошибки.

В “Running Claude Code Safely in Devcontainers” он описывает запуск Claude Code внутри `devcontainer`. Так можно чаще использовать `--dangerously-skip-permissions`, не отдавая агенту полную свободу на основной машине. Docker-изоляция позволяет Claude менять проект, запускать команды из `package.json` и работать с базой из `docker-compose`, но не трогать систему хоста, глобальную среду и файлы вне явно примонтированных путей.

Практические детали важны:

- Docker Compose поднимает приложение и зависимости вроде Postgres;
- `.claude` и `.claude.json` монтируются внутрь контейнера, чтобы не входить заново после пересборки;
- `node_modules` можно держать в Docker volume для скорости;
- Claude Code запускается как пользователь без прав `root`;
- GitHub CLI устанавливается как отдельная возможность среды;
- обычный худший сценарий — агент портит контейнер, который можно пересобрать.

Эта конфигурация не снимает все риски. `.claude` и `.claude.json` нужно добавить в `.gitignore`, потому что они появляются в корне проекта и не должны случайно попасть в репозиторий. Смонтированная аутентификация всё ещё может давать доступ к GitHub, а значит агент может выполнить разрушительное действие уже не только внутри контейнера, но и в удалённом репозитории. Доступ к интернету оставляет риск инъекции инструкций через внешний текст, который прочитает агент. Интеграция с IDE хуже, чем в обычной локальной среде. Но компромисс понятен: агент получает больше скорости внутри ограниченной области, а ущерб от ошибки становится меньше.

С `рабочее дерево` проблема другая. Jökull прямо пишет, что пока ограничен одним PR за раз. Агенты могут параллельно ждать, исследовать или выполнять локальные задачи, но безопасная параллельная запись требует изоляции. Git рабочих деревьев должны дать каждому агенту своё рабочее дерево и отдельный PR. Для монорепозитория с pnpm это исторически неудобно из-за `node_modules`, поэтому он ждёт лучшей поддержки рабочих деревьев и смотрит на Superset.sh.

В “YOLO Mode Doesn’t Scale” та же мысль перенесена на организационный уровень. В режиме одного пользователя агент действует в пространстве конкретного человека, с его файлами и его учётными данными. В компании так нельзя: роли в CMS, делегирование календаря, безопасность на уровне строк, утверждение расходов и другие системы имеют разные модели прав. Jökull предлагает схему с личными агентами, учётными данными с ограниченными правами, служебными конвертами запросов, следом происхождения запроса и заранее подготовленными решениями для владельца области.

В этой статье важна модель из двух независимых осей. Первая ось — организационные полномочия человека: что этот сотрудник имеет право делать. Вторая — доверие к агенту: насколько длинный поводок можно дать именно этому агенту в данной области. Новый CEO может иметь широкие полномочия, но его агент всё равно должен работать осторожно. Опытный офис-менеджер может иметь меньше формальных прав, но агенту можно дать больше самостоятельности в повторяемых задачах своей области. Это точнее, чем простое правило “роль человека равна правам агента”.

Jökull также описывает условный поводок. Разрешение может зависеть от времени, намерения или отправителя: например, если агент менеджера просит недельные метрики, можно отправить их автоматически; если тот же агент просит экспорт всех данных, нужен человек. Ещё один слой — самосогласование прав. Если агент упирается в запрет, он не должен просто падать. Он может собрать case: что пытался сделать, какие действия уже проверил, какого доступа не хватает, почему это нужно, и передать владельцу системы уже подготовленное решение.

Отдельный полезный сценарий — конфликтующие запросы агентов. Если два агента присылают разные запросы по часам работы или другому состоянию, это не обязательно два независимых тикета. Ответственный агент может сжать конфликт в один рабочий объект, найти источник расхождения и обнаружить, что устарела другая страница или справочник. Так конфликт становится не только ошибкой, но и сигналом о рассинхронизации контекста.

Общий принцип один: свобода агента должна зависеть от границы действия. Локальный `devcontainer`, будущий `рабочее дерево` и организационный конверт запроса — разные реализации этого принципа.


Безопасность Jökull ближе к практической изоляции, чем к полной песочнице Mike McQuaid. `devcontainer`, будущие рабочие деревья и ограниченные учётные данные уменьшают риск, но не заменяют жёсткую границу пользователя и файловых прав. Поэтому его подход хорошо переносится на PR-сопровождение, но для чувствительных репозиториев может потребовать слоя Mike или Arvid.

### 18. Практические шероховатости: `pbcopy`, `pbpaste` и маленькие переходы между системами

В коротких практических советах Jökull упоминает `pbcopy` и `pbpaste`: он просит Claude отправлять результат в буфер обмена, а не печатать его в интерфейс терминала. Копирование многострочного текста из TUI часто добавляет лишние переносы и пробелы; буфер обмена обходит эту проблему.

Эта деталь хорошо показывает его стиль. Он улучшает конкретные переходы между агентом и внешним миром:

- результат агента → буфер обмена;
- чат → gist;
- запись теста удобства → GitHub issues;
- комментарии к PR → исправления;
- вызовы MCP → JavaScript-блок;
- статья из web → EPUB;
- запрос календарного события → `.ics`-ссылка.

Каждый такой переход уменьшает ручную работу. В сумме агентская сессия превращается из разговора в набор рабочих действий, после которых остаются полезные артефакты.

### 19. Вариация паттерна: тест удобства → GitHub issues


Дополнительные статьи лучше читать как проверку того же приёма на других объектах.

В “From Usability Test Video to GitHub Issues” рабочий объект — не PR, а запись теста удобства. Пользователь записывает экран, планирует поездку и вслух объясняет, где застревает. Запись ценная, но после просмотра начинается ручная работа: найти точную секунду, где кнопка не сработала, сделать снимок экрана, написать issue, приложить подтверждение, повторить для следующего найденного пункта.

Jökull пишет Claude Code `skill`, который принимает путь к видео и возвращает набор GitHub issues со снимками экрана, GIF и итоговым gist.

Цепочка такая:

1. модель с пониманием видео строит линию времени с примерными временными отметками;
2. найденные пункты классифицируются на баги, улучшения и положительные наблюдения;
3. `ffmpeg` извлекает снимок экрана или GIF;
4. изображения загружаются в объектное хранилище;
5. создаётся GitHub issue на каждый найденный пункт;
6. создаётся итоговый gist по всей сессии.

В `SKILL.md` этой процедуры есть несколько рабочих шероховатостей. Для первичного разбора видео используется CLI `summarize`. Находки делятся не только на ошибки и улучшения, но и на положительные наблюдения. GitHub issue создаётся через `gh issue create`, а изображение добавляется отдельным комментарием через `gh issue comment`. Для итогового gist важно не передавать markdown через stdin без имени файла: GitHub в таком случае может создать `.txt`. Поэтому лучше сначала записать временный `.md`-файл и уже его отдавать в `gh gist create`.

Эти детали выглядят мелкими, но именно они делают skill рабочей процедурой, а не красивым описанием. Агент должен знать не только “создай issues”, но и как обойти конкретные особенности `gh`, gist и обработки кадров.

Главная деталь — проверка визуального подтверждения. Временные отметки приблизительные. Если взять кадр по указанному времени механически, можно получить неправильное место: на секунду раньше, на секунду позже, в середине прокрутки. Поэтому `skill` заставляет Claude прочитать каждый извлечённый кадр перед загрузкой. Если кадр не показывает нужный момент, временная отметка сдвигается, а кадр извлекается заново.

Второе правило касается формы подтверждения. Статические проблемы получают снимок экрана, динамические проблемы получают GIF. Если кнопка нажимается и ничего не происходит, снимок “ничего не произошло” почти ничего не доказывает. Нужен GIF, где видно действие и отсутствие реакции.

Эта статья усиливает центральную историю. Агентская процедура должна собрать проверяемое свидетельство. В `/babysit-pr` такими свидетельствами являются зелёный CI, результаты Greptile, проверка Codex и `commit`. В usability-flow — issue с правильным кадром или GIF.

### 20. Вариация паттерна: gist, ридер и `.ics` как внешние артефакты

В нескольких текстах Jökull показывает короткие процедуры, где результат агента превращается в объект, пригодный для внешней системы.

В “Turning Claude Code Output Into Shareable Links” он описывает привычку: когда нужно поделиться результатом агентской сессии, он просит агента создать gist. Обычно не нужен весь стенограмма. Нужен артефакт: markdown-документ, проверочный список, план, фрагмент кода или инструкция.

Он добавляет в глобальный `~/.claude/CLAUDE.md` правила:

- использовать `gh gist create`;
- по умолчанию создавать secret gists;
- давать понятное описание;
- выбирать правильные имена файлов и расширения;
- возвращать аккуратный viewer URL `md.solberg.is/<gist_id>`, а не стандартный GitHub gist URL;
- для чувствительного содержимого использовать локальное шифрование с ключом во фрагменте URL.

Смысл не в самой команде `gh gist create`. Фраза “make this shareable” становится устойчивым действием. Агент уже знает, как упаковать результат, опубликовать его и вернуть ссылку.

В “Piping the Web to My E-Reader” объектом становится статья из web. Skill выполняет цепочку: извлечь чистый текст через Mozilla Readability, конвертировать HTML в EPUB через Pandoc, найти CrossPoint Reader в локальной сети через mDNS, загрузить EPUB на устройство или положить его в очередь для отправки позже. Ручная операция “прочитать длинную статью на ридере” превращается в команду `send-article.sh`.

В “Calendar Invites the Hard Way” объектом становится календарное событие. Программно добавлять события в общий iCloud-календарь неудобно, но iOS умеет открывать `.ics` и показывать нативный экран “Add to Calendar”. Поэтому quick-ics, Cloudflare Worker, генерирует приглашение в календарь из параметров URL. Clawd, личный ассистент Jökull, может создать ссылку и отправить её пользователю; пользователь одним нажатием добавляет событие.

Эти примеры показывают общий принцип. Иногда агенту не нужен полный доступ к системе. Достаточно создать артефакт, который система уже умеет принимать: gist URL, EPUB, `.ics` file. Человек остаётся в точке подтверждения, но ручная подготовка исчезает.

### 21. Вариация паттерна: `icelandic-data` и skills для трудных источников данных

В “A data toolkit for Icelandic public data, built on Claude Code skills” Jökull переносит ту же форму в исследование данных.

Он строит репозиторий `icelandic-data`, который “учит Claude Code” работать с исландскими публичными источниками данных. Это набор файлов навыков, Python-скриптов и накопленных знаний о том, как извлекать данные из официальных API, которые не проектировались как удобные.

Каждый источник получает `.claude/skills/{source}.md`: конечные точки API, коды временных рядов, особенности кодировок, проблемы классификации. Рядом лежит Python-скрипт, который извлекает и чистит данные. После `setup.sh` Claude Code может работать с более чем шестнадцатью источниками.

Jökull противопоставляет два режима. Jupyter notebooks и R scripts последовательны, ручные и плохо переиспользуются. Подход через `skills` фиксирует знание об источнике один раз, а дальше с ним можно взаимодействовать разговорно.

Примеры результатов:

- анализ рынка топлива через ежедневные выгрузки цен, PDF-годовые отчёты и сравнения со скандинавскими странами;
- сравнение страховых компаний через показатели IFRS 17 из PDF-отчётов;
- анализ PM10 в Reykjavík через данные мониторинга воздуха, регистрационные данные и городские опросы;
- анализ закупок зимних служб через муниципальные расходы.

Здесь снова работает тот же механизм. Агенту дали рабочую среду: `skills`, скрипты, Docling, DuckDB, `jq`, знания об API, особенностях PDF и отчётных шаблонах. Сложность источника вынесена в повторяемую процедуру.

Для процесса разработки это важная аналогия. Проектная память должна хранить не только решения, но и способы входа в неудобные области: API, отчёты, CI, локальные соглашения, источники ошибок, процедуры проверки.

### 22. Вариация паттерна: Code Mode и длинные MCP-цепочки

В “Code Mode Changed How I Think About MCP” Jökull описывает `agent-cms`: CMS без пользовательского интерфейса, где ИИ-агенты управляют содержимым через MCP.

У CMS есть 33 MCP-инструмента. Создание структурированного поста — это не один вызов.

В источнике важна ещё и граница прав. У `agent-cms` есть не одна одинаковая MCP-поверхность, а два входа. `/mcp` даёт административный набор примерно из 31 метода, включая изменение схемы. `/mcp/editor` даёт более узкий набор примерно из 20 методов для работы с содержимым и не позволяет делать мутации схемы вроде `create_model`. Поэтому Code Mode здесь решает не только задачу скорости. Он помогает разделить роль администратора и роль редактора: агент получает тот набор действий, который соответствует задаче.

Создание структурированного поста — это не один вызов. Нужно посмотреть схему, создать модель, добавить поля с валидаторами и перекрёстными ссылками, создать записи со связанными данными, опубликовать их и проверить результат. Типичная редакторская задача требует 10–20 последовательных вызовов инструмента, и каждый зависит от результата предыдущего.

Проблема обычного MCP в том, что каждый вызов инструмента проходит через модель. Модель вызывает `create_model`, читает ответ, формирует `create_field`, ждёт ответ, формирует следующий вызов, и каждый раз в контекст возвращается всё больше промежуточных данных.

Cloudflare Code Mode меняет форму интерфейса. MCP-инструменты становятся типизированными методами внутри V8-песочницы. Модель видит один инструмент `code` и пишет JavaScript. Несколько операций с зависимыми данными выполняются за один вызов песочницы. Промежуточные результаты, которые не нужны модели, не попадают обратно в контекст. Учётные данные остаются на стороне хоста, у песочницы нет доступа к сети.

Тест, который убедил Jökull: создать несколько связанных моделей, поля с валидаторами перекрёстных ссылок, записи со связанными данными, опубликовать их и затем запросить обратно для проверки. Это больше двадцати операций с условиями, циклами и передачей ID между шагами. Code Mode сделал это одним вызовом `code`.

Связь с `/babysit-pr` прямая. Там длинная ручная цепочка сопровождения PR превращается в `skill`. Здесь длинная цепочка вызовов MCP превращается в исполняемый JavaScript-блок в контролируемой среде. В обоих случаях цель одна: убрать лишние переходы туда-обратно, сохранить проверяемость и не засорять контекст промежуточным шумом.

### 23. Вариация паттерна: Sentry Seer и правильный пакет контекста

В “The Context Problem: Notes on Sentry’s Seer” Jökull формулирует проблему шире: главный вызов для агентов разработки часто состоит не в сырой интеллектуальной мощности модели, а в ручной работе по подаче нужной информации.

Нельзя просто бросить всю кодовую базу в модель и попросить “исправление the баг”. Нужно подбирать контекст, отрезать лишнее, уводить агента от тупиковых направлений, приносить детали, когда он застрял, и давать цикл обратной связи. Jökull называет эту роль контекст wrangler; по смыслу это человек или система, которые собирают и подают агенту правильный контекст. Иногда такая работа занимает почти столько же времени, сколько ручное исправление бага.

Sentry Seer впечатляет его тем, что Sentry уже много лет собирает почти готовый пакет контекста: стек ошибки, код с sourcemap для конкретного коммита, цепочку действий перед сбоем, затронутого пользователя, параметры окружения и трассировку между сервисами. Это не просто набор данных, а связное описание ошибки.

Если Sentry настроен правильно — есть sourcemaps, интеграция с GitHub и доступ к исходному коду, — Seer получает контекст, который обычной модели пришлось бы мучительно собирать вручную. Тогда автоматическое исправление становится реалистичнее.

В заметке есть маленький, но показательный пример со Slack webhook. Код делал HTTP `fetch`, а затем пытался вызвать `response.json()`. Slack в этом месте возвращал обычный текст, поэтому разбор JSON падал. Seer предложил заменить это на `response.text()`. Это было приемлемое исправление, но не обязательно лучшее: в таком случае, возможно, вообще не нужно разбирать тело ответа, достаточно проверить статус. Деталь важна тем, что даже при хорошем пакете контекста агент может предложить рабочую, но не идеальную правку. Поэтому Seer снижает стоимость исправления, но не отменяет человеческую оценку качества решения.

Эта заметка связывает раннюю интуицию Jökull с `/babysit-pr`. Хорошая агентская система начинается с правильно собранного контекста и цикла обратной связи. `/babysit-pr` делает это для PR: собирает CI, Greptile, проверку Codex, линтер и `git push` в одну процедуру, где агент знает, что делать с каждым сигналом.

### 24. От сопровождения PR к бизнес-оркестрации

В конце “How I Use Claude Code” Jökull пишет, что команда исследует оркестрацию на уровне бизнеса: автоматические триггеры, передачи между агентами и определённые зоны ответственности для агентов, которые делают фоновое исследование, конкурентный анализ и похожие задачи.

Это не обязательно задачи разработки. Слой оркестрации может передавать результаты исследования агенту, который пишет код, если нужна реализация. Но ценность шире: агенты с ясными обязанностями могут передавать работу друг другу без ручного вмешательства.

`/babysit-pr` здесь выглядит как частный случай. Есть рабочий объект — PR. Он проходит через GitHub, CI, Greptile, Codex, линтер, `git push`, auto-merge. Агент сопровождает объект, но не стирает человеческое решение.

В бизнес-оркестрации объектом может быть исследование рынка, запрос поддержки, обновление контента, запрос прав, анализ цен или внутренний отчёт. Структура похожа:

- объект имеет состояние;
- разные системы дают сигналы;
- часть действий можно выполнить автоматически;
- часть нужно подготовить как решение для человека;
- нужны след запроса, происхождение запроса и критерии передачи человеку.

Так эта история выходит за пределы PR и сохраняет порядок. `/babysit-pr` остаётся базовым образцом, а остальные кейсы расширяют его: другой объект, другие инструменты, та же форма процедуры.

### 25. Ментальная модель Jökull

Jökull прямо пишет, что его модель — не “AI writes my code”. Эту формулировку стоит оставить близко к источнику, потому что она показывает, от чего он отталкивается. В русском пересказе смысл такой: Claude Code для него — медленный и обстоятельный инструмент, которым нужно управлять. Несколько агентов позволяют распараллелить ожидание вокруг CI, проверок и долгих операций. Контекстом лучше управлять явно. Специализированные агенты лучше подходят для повторяемых задач, чем общие подсказки. Цикл Greptile → исправление → повторная проверка действительно ловит баги.

Агентская разработка у него держится на внешних опорах:

- `skills`;
- `CLAUDE.md` / `AGENTS.md`;
- `gh` CLI;
- MCP;
- Code Mode;
- devcontainers;
- рабочих деревьев;
- Greptile;
- Codex review;
- `pbcopy` / `pbpaste`;
- объектное хранилище;
- gist;
- `ffmpeg`;
- скрипты;
- небольшие размещённые сервисы.

Модель делает интеллектуальную работу внутри этой среды, но среда задаёт форму движения.

### 26. Что в этой истории происходит на самом деле

Если восстановить историю Jökull как последовательность практических ходов, получается такая дуга.

Сначала есть обычная проблема агентов разработки: нужно вручную управлять контекстом, ждать CI, читать проверку, возвращаться к PR, повторно объяснять знания свежим агентам.

Потом появляются маленькие процедуры:

- создание issue через `gh` с предварительным чтением кода;
- `/pr-feedback` для комментариев Greptile;
- `pbcopy` / `pbpaste` для аккуратной передачи текста;
- `codex review` перед PR.

Затем процедуры оформляются как `skills`:

- `/babysit-pr` для PR;
- `/user-testing` для видео пользовательских тестов;
- i18n-specialist для переводов;
- навыки для публичных источников данных.

Параллельно появляется слой постоянной памяти:

- `CLAUDE.md` для общепроектного контекста;
- локальные `CLAUDE.md` внутри отдельных пакетов;
- `AGENTS.md` как файл проектных соглашений для Next.js;
- глобальный `CLAUDE.md` для публикации gist;
- скрипты и небольшие размещённые сервисы для повторяющихся переходов.

Дальше линия расширяется:

- devcontainers для безопасной автономии;
- рабочих деревьев для параллельной записи;
- Code Mode для длинных MCP-цепочек;
- личные агенты и служебные конверты запросов для автономности в организации;
- бизнес-оркестрация как следующий уровень передач между агентами.

`/babysit-pr` остаётся самым чистым примером. Вокруг него видна общая манера: найти ручную цепочку, выделить повторяемые решения, упаковать их в процедуру для агента, добавить проверяемый выход и оставить человека там, где нужен смысловой выбор.

### 27. Что переносимо в Codex / CU

Из истории Jökull переносится не конкретный `skill`, а несколько процессных форм.

**1. Жизненный цикл PR нужно считать отдельным объектом автоматизации.**  
Написать код — только часть работы. CI, проверка, исправления, повторная проверка, готовность к слиянию и auto-merge тоже требуют сопровождения. Если этот слой остаётся ручным, человек всё равно сопровождает агента как диспетчер.

**2. Между проверкой и изменением нужен разбор замечаний.**  
Несколько источников проверки полезны только при наличии классификации. Иначе агент начинает слепо обслуживать CI, Greptile, Codex или любого другого проверяющего. Минимальная форма: Fix / Dismiss / Escalate с причиной.

**3. Источники проверки должны различаться по профилю ошибок.**  
CI ловит формальные сбои. Greptile ловит часть архитектурных и контекстных проблем. Codex review ловит логические разрывы. Польза возникает из низкого пересечения, а не из веры в одного универсального проверяющего.

**4. `skills` — хороший формат для повторяющейся процедуры.**  
Если есть повторяемый процесс с понятными входными данными, инструментами, критериями выхода и правилами остановки, его стоит вынести в `skill` или аналогичный документ. Для этого не обязательно писать большую программу.

**5. Специализированное знание лучше выносить из общего контекста.**  
Сотни строк i18n-правил полезны только для i18n-задач. Сведения о публичных источниках данных полезны только для работы с данными. Их нужно раскрывать по ситуации, а не держать постоянно.

**6. `CLAUDE.md` / `AGENTS.md` должны фиксировать проектные решения, а не заменять энциклопедию.**  
Агентам нужны локальные соглашения: где проверять авторизацию, как получать данные, какие библиотеки использовать, какие анти-паттерны не принимать. Без этого они будут писать типовой код, который выглядит правдоподобно, но не соответствует проекту.

**7. Результат агента должен быть артефактом.**  
PR, issue, gist, снимок экрана, GIF, HTML-отчёт, JSON, ссылка на подтверждение — всё это лучше, чем “я сделал”. Артефакт можно проверить, переслать, использовать повторно.

**8. Автономия требует ограничения радиуса ущерба.**  
`devcontainer`, учётные данные с ограниченными правами, `рабочее дерево`, граница разрешений, нативный экран подтверждения, конверт запроса с указанием происхождения — разные формы одного принципа: агенту можно дать больше свободы там, где область действия ограничена и понятна.

**9. Длинные цепочки вызовов инструментов иногда нужно превращать в исполняемый блок.**  
Если задача требует 10–20 зависимых вызовов MCP, модель не должна гонять всё через чат по одному вызову. Code Mode показывает другой вариант: типизированные методы внутри песочницы и один контролируемый вызов `code`.

**10. Конец работы должен сохранять знание.**  
Если в ходе задачи человек объяснил доменную цель, крайний случай, процедуру тестирования или приём отладки, это должно попасть в `skill`, `CLAUDE.md`, gist или другой устойчивый носитель. Иначе свежий агент снова попросит то же объяснение.

### 28. Ограничения подхода

Подход Jökull силён там, где процесс уже достаточно понятен. `/babysit-pr` работает, потому что жизненный цикл PR имеет повторяемую форму: CI, проверка, исправления, `push`, повторная проверка, готовность к слиянию. `/user-testing` работает, потому что видео пользовательского теста можно превратить в линию времени, найденные пункты, визуальные подтверждения, issues и gist. `icelandic-data` работает, потому что источники данных можно описать через файлы навыков и скрипты.

Он хуже применим там, где ещё не ясно, какая процедура должна существовать. Нельзя заранее написать хороший `skill` для хаоса, который ещё не осмыслен. Сначала нужно несколько раз пройти процесс вручную, увидеть повторяющиеся решения, ложные старты, опасные места и критерии выхода.

Есть и зависимость от внешних источников:

- если CI медленный, `/babysit-pr` всё равно ждёт;
- если Greptile слишком шумит, разбор замечаний станет дорогим;
- если Codex review даёт много ложных замечаний, второй проверочный слой начнёт мешать;
- если `CLAUDE.md` разрастается и стареет, агент начнёт следовать мусору;
- если `devcontainer` монтирует слишком много чувствительного состояния, изоляция ослабнет;
- если рабочих деревьев плохо поддерживаются инструментами, параллельность станет дорогой.

Командный перенос требует ещё большей осторожности. Одиночный разработчик может быстро решить, что отклонить, что исправить, что слить автоматически. В команде нужны явные правила: кто имеет право отклонять замечания, какие проверки обязательны, какие изменения требуют человека, как фиксируется происхождение решения, где проходит граница учётных данных.

### 29. Практическая формула истории

Историю Jökull можно сжать до одной рабочей формулы:

> Повторяющуюся ручную цепочку нужно превратить в процедуру для агента, но каждое внешнее замечание должно проходить через классификацию, а каждый результат должен выходить как проверяемый артефакт.

`/babysit-pr` — самая чистая реализация этой формулы. Он берёт PR, который раньше требовал ручного сопровождения, и проводит его через CI, Greptile, Codex review, исправления, линтер, `commit`, `push` и повторную проверку. Но он не говорит агенту “исправляй всё”. Он задаёт другую последовательность: классифицировать, принять решение, зафиксировать причину, вынести человеку спорные пункты.

Другие посты показывают ту же форму на других объектах:

- видео пользовательского теста → линия времени → найденные пункты → снимки экрана/GIF → issues → gist;
- URL статьи → очищенный текст → EPUB → загрузка на ридер или очередь;
- запрос календарного события → `.ics`-ссылка → нативный экран добавления в календарь;
- результат агента → gist → аккуратная ссылка для просмотра;
- исландский источник данных → `skill` → скрипт → запрос или отчёт;
- цепочка MCP → JavaScript в песочнице → проверенное состояние CMS;
- запрос от внешнего агента → конверт запроса → заранее подготовленное решение.

Это не набор разрозненных лайфхаков. Это повторяющийся инженерный ход: агент получает рабочую поверхность, процедуру, инструменты, проверку и границы.

### 30. Почему эта история важна для Handbook

Для вводного Handbook по агентской разработке история Jökull полезна как пример среднего слоя между прямым разговором с моделью и полноценной платформой.

Слишком лёгкий подход оставляет человека диспетчером: он ждёт CI, читает проверку, копирует результаты, снова объясняет контекст, вручную создаёт issues, вручную прикладывает подтверждения.

Слишком тяжёлый подход рискует построить большую систему до того, как повторяющаяся проблема понята.

Jökull показывает промежуточную дисциплину:

- начать с реальной шероховатости;
- пройти процесс вручную;
- выделить повторяющиеся шаги;
- упаковать их в маленький `skill`, CLI, секцию `CLAUDE.md` или небольшой размещённый сервис;
- оставить оценку качества сигнала внутри процедуры;
- возвращать наружу проверяемый артефакт;
- обновлять рабочую память после завершения.

Для CU/dev-process это особенно ценно. Здесь есть почти готовые строительные блоки будущего процесса:

- `impact frontier` можно искать через исследовательские шаги агента;
- `propagation ledger` должен быть похож не на стенограмму, а на сжатый проверяемый артефакт;
- `distortion review` должен работать как разбор замечаний, а не как слепое принятие комментариев от проверяющего;
- `repair loop` должен иметь критерии выхода и ограничение итераций;
- `skills` и `CLAUDE.md` должны накапливать устойчивые процедуры и соглашения, а не всю память проекта;
- автономия должна зависеть от области действия, доверия и обратимости.

Главная ценность истории Jökull в том, что она показывает практическую форму агентского ритуала. Ритуал не обязан быть большим. Иногда это markdown-файл на 170 строк. Если он знает, какие источники читать, как ждать, как классифицировать замечания, какие проверки запускать, когда повторить цикл и когда позвать человека, он уже меняет процесс разработки.
