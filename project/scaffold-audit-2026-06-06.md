# Scaffold audit — 2026-06-06

## v6

Проверена новая модель scaffold после перехода к плоской рабочей зоне `/work`.

Исправления:

- рабочая зона текущей задачи теперь находится прямо в `/work`;
- главный файл задачи — `work/discourse.md`;
- `work/discourse.md` создан из пользовательского вставленного дискурса, исходный H1-заголовок удалён;
- основные документы handoff и source materials перенесены в корень `/work` как сопутствующие документы задачи;
- добавлены два протокола: `protocols/rules/codex-task-work-protocol.md` и `protocols/rules/chat-codex-transfer-protocol.md`;
- `AGENTS.md`, `/project`, `/protocols` и рабочие prompt-файлы обновлены под новую модель;
- обязательная подпапка задачи внутри `/work` убрана;
- `site/index.html` по-прежнему не содержит относительных ссылок на отсутствующие Markdown-файлы историй;
- структура остаётся KISS: `/content`, `/site`, `/site-spec`, `/protocols`, `/project`, `/work`.

Главный риск, который закрывает v6: Codex больше не должен искать активную задачу в единственной подпапке внутри `/work`; текущий дискурс лежит в фиксированном месте `work/discourse.md`.


## Дополнительная проверка v7

После просмотра `/work` было зафиксировано, что в рабочей зоне уже много документов, и дискурс может потерять связь с ними, если будет сохранять только смысловые выводы. Поэтому протоколы обновлены: `work/discourse.md` должен называть значимые файлы прямо в ходе рассказа и объяснять их роль в поворотах задачи.

Это правило не заменяет дискурс manifest-таблицей. Manifest может быть вспомогательным документом, но смысловая связь должна оставаться внутри `work/discourse.md`.


## Дополнительная проверка v9

После обсуждения skills принято не сжимать их в один общий каталог `skills.md`. Добавлена папка `protocols/skills/`, где один skill хранится в одном Markdown-файле. Отдельные подпапки под skills пока не создаются.

Также добавлены протоколы работы ChatGPT с GitHub-репозиторием и интеллектуального управления ветками/merge. Модель задач уточнена: Git branch является границей задачи, `/work` является рабочей зоной текущей ветки, а `work/discourse.md` остаётся главным носителем дискурса.

## Дополнительная проверка v10

Проверены пересечения между правилами, skills, prompts и проектной моделью веток. Зафиксировано различие ролей: `rules/` — инварианты, `skills/` — повторяемые режимы, `prompts/` — вызовы конкретных сессий, `work/discourse.md` — живая память текущей задачи.

Исправлены два практических недочёта: старое упоминание `/work/<stage>/` в `codex-task-contract.md` и опечатка `protocol/rules` в `apply-skill.md`. Добавлен отдельный skill для отката правок, потому что модель прямых commit'ов в рабочую ветку должна иметь безопасный путь восстановления.

## Дополнительная проверка v11

Объединены `codex-task-contract.md` и `codex-task-work-protocol.md`: теперь контракт задачи и порядок выполнения живут в одном файле `protocols/rules/codex-task-work-protocol.md`. Уточнена Git-модель: чистый `main` не содержит `/work`, а архив с `/work` является образом рабочей ветки. Также зафиксировано, что мелкие низкорисковые правки можно коммитить напрямую в `main` по явному разрешению пользователя, без искусственного создания ветки.
