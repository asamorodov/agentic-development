# Решения проекта

## 2026-06-06 — KISS-структура репозитория

Принята минимальная структура:

```text
/content
/site
/site-spec
/protocols
/project
/work
```

`/content` хранит содержательные Markdown-документы. `/site` хранит текущий построенный сайт, а не одноразовый `dist`. `/site-spec` описывает устройство сайта. `/protocols` хранит правила и рабочие prompt-файлы. `/project` хранит документацию о репозитории и решениях. `/work` хранит текущую активную задачу текущей ветки.

## 2026-06-06 — Один корневой AGENTS.md

Используется только один корневой `AGENTS.md`. Он должен быть короткой картой проекта и ссылаться на `/project`, `/protocols`, `/site-spec` и `/work`. Вложенные `AGENTS.md` не создаются по умолчанию.

## 2026-06-06 — Protocols и prompts

`/protocols` содержит две основные подпапки:

```text
protocols/rules/
protocols/prompts/
```

`rules/` хранит устойчивые правила работы. `prompts/` хранит исполняемые запросы для Codex или новых чатов.

## 2026-06-06 — Плоская модель `/work`

Принята модель: `/work` содержит текущую активную задачу текущей ветки, без обязательной подпапки задачи. Главный файл — `work/discourse.md`.

Параллельные подзадачи разводятся через Git branches/worktrees. Подпапки внутри `/work` создаются только при практической необходимости, например для большой серии однотипных pass-артефактов.

## 2026-06-06 — Дискурс вместо краткого state

`work/discourse.md` является главным носителем смысловой непрерывности задачи. Это не summary, не handoff, не список next actions и не status-log. Он должен сохранять ход обсуждения, причины решений, отвергнутые варианты, поздние уточнения и детали, которые нельзя потерять.

Codex должен читать `work/discourse.md` перед нетривиальной работой с текущей задачей и обновлять его после нетривиальной thread-сессии.

## 2026-06-06 — AI-driven SDLC как рамка перестройки теории

Перестройка теоретической части должна идти вокруг AI-driven SDLC как жизненного цикла программного изменения: намерение → контекст → делегирование → исполнение → свидетельства → ревью → право завершения → сопровождение / обучение среды.

Expanded theory является quarry. Старый сайт и старые вычитанные версии являются baseline для сохранения композиционной силы и деталей. SPDD и Gas Town являются deep anchor cases и не должны сжиматься без human gate.


## 2026-06-06 — Привязка дискурса к файлам

Принято правило: `work/discourse.md` должен сохранять связь смысловых поворотов с конкретными файлами и артефактами. Если решение возникло через audit, source map, approved plan, prompt-файл, отчёт, архив, seed-текст или другую рабочую заготовку, имя файла должно быть указано прямо в соответствующем месте дискурса.

Причина: текущая задача уже породила много рабочих документов в `/work`. Если дискурс сохранит только итоговые выводы, но потеряет имена файлов и их роль, следующий Codex-thread не сможет восстановить, какие документы были промежуточными, какие стали утверждёнными, какие показали деградацию, а какие нужно читать как источник локальных правил.


## 2026-06-06 — однородный discourse.md без служебных блоков

Для текущей задачи в `/work` главный файл — `work/discourse.md`. Он ведётся как однородный дискурс: повороты разговора, причины решений, поздние уточнения, связь с файлами и рабочая траектория. Не используется отдельный `start-here`, `state`, `progress` или `next-actions` слой. Текущая позиция должна быть понятна из последней части дискурса, а не из отдельного служебного раздела.

Заголовки в `discourse.md` допустимы только как названия смысловых поворотов. Дискурс не должен превращаться в status report, handoff, план, manifest или разговор агента с самим собой.


## 2026-06-06 — Skills как отдельные файлы

Принята модель `protocols/skills/`: один skill — один Markdown-файл. Все skills не сводятся в один общий `skills.md`, но и отдельные папки под каждый skill пока не создаются. Папка под конкретный skill появится только если skill потребует собственных шаблонов, скриптов, примеров или больших вспомогательных материалов.

## 2026-06-06 — ChatGPT может работать с GitHub-репозиторием точечно

Если GitHub-интеграция даёт write-доступ, ChatGPT может выполнять небольшие текстовые commit-операции в указанную ветку: обновлять `work/discourse.md` в рабочей ветке, а также `/project`, `/protocols`, prompt/skill-файлы. Мелкие низкорисковые правки можно писать прямо в `main` по явному разрешению пользователя. Тяжёлые изменения, rebuild, массовые правки, проверки и большие PR остаются зоной Codex.

## 2026-06-06 — Ветка как граница задачи

Принята модель: рабочая ветка соответствует активной задаче или подзадаче. `/work` хранит рабочий дискурс и сопутствующие документы текущей ветки. Параллельные подзадачи разводятся Git branches/worktrees, а не вложенными папками `/work`. Перед переносом в `main` временные материалы `/work` по умолчанию исключаются из итогового diff, если они не повышены до постоянных документов.

## 2026-06-06 — Протокольная карта и откат правок

Зафиксировано различие между четырьмя слоями: `rules/` задают устойчивые инварианты, `skills/` задают повторяемые режимы работы, `prompts/` запускают конкретные сессии с параметрами, а `work/discourse.md` хранит дискурс активной задачи. Добавлен skill `protocols/skills/rollback-working-change.md`, потому что рабочая модель допускает прямые commit'ы в рабочую ветку, но требует безопасного способа отката отдельных неудачных правок без ручной Git-рутины.

## 2026-06-06 — объединение task contract и task work protocol

`protocols/rules/codex-task-contract.md` удалён как отдельный файл. Его содержание объединено с `protocols/rules/codex-task-work-protocol.md`, потому что форма постановки задачи и порядок выполнения в этом процессе не должны расходиться. Один файл теперь описывает и контракт задачи, и порядок работы Codex.

## 2026-06-06 — прямые мелкие commit'ы в main

Принято, что мелкие низкорисковые поправки можно коммитить напрямую в `main`, если пользователь явно разрешил запись. Не нужно создавать ветку ради опечатки, малого уточнения README, простого уточнения протокола или похожей малой правки. Крупные изменения, `/content`, source precedence, baseline rules, архитектура теории и финализация стадий всё ещё идут через рабочую ветку или PR.

## 2026-06-06 — clean main без /work

Чистый стартовый `main` не содержит `/work`. Архив scaffold с `/work` рассматривается как образ рабочей ветки, а не как образ `main`. Модель такая: сначала создаётся чистый `main` из постоянных папок, затем от него создаётся рабочая ветка, в которой появляется `/work/discourse.md` и сопутствующие документы задачи.

## 2026-06-06 — Git history вместо копий рабочих документов

Для работы ChatGPT через GitHub-интеграцию принято правило: новые рабочие документы создаются в репозитории, по умолчанию в `/work`, а не внутри ответа чата. Существующие документы по умолчанию не копируются в `*_v2.md`/`*_new.md`, а обновляются как новая версия того же файла через commit.

Для существенных документов стандарт — один изменяемый документ = один commit. Commit message должен содержать понятный anchor, чтобы позже можно было восстановить старую версию файла по commit SHA/ref и чтобы модель могла сопоставить фразу пользователя “версия до такого-то действия” с конкретным commit через `work/discourse.md` and commit history.

`work/discourse.md` должен фиксировать такие anchors рядом с соответствующим смысловым поворотом. Если anchor неоднозначен, агент обязан остановиться и запросить уточнение, а не угадывать.

## 2026-06-06 — Разделение протоколов Codex и ChatGPT/GitHub

Принято решение развести два похожих, но разных операционных режима. `protocols/rules/codex-task-work-protocol.md` описывает Codex task/thread/session внутри репозитория, worktree or cloud environment. `protocols/rules/chat-github-repo-work-protocol.md` описывает работу ChatGPT через GitHub-интеграцию: чтение repo перед ответом, запись файлов через GitHub API, commit anchors and финальный ответ пользователю. Общие правила ведения `work/discourse.md` вынесены в `protocols/rules/discourse-maintenance-rules.md`, чтобы Codex и ChatGPT сохраняли один формат дискурса без смешения своих операционных протоколов.
