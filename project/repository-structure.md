# Структура репозитория

Репозиторий устроен по принципу KISS. Папки добавляются только тогда, когда у них есть устойчивая роль.

```text
/content
/site
/site-spec
/protocols
/project
/work
```

## `/content`

Главный источник содержательного текста. Сюда входят Markdown-документы корпуса и ассеты, связанные с содержанием.

Ожидаемая структура:

```text
content/
  Orientation.md
  Story_introduction.md
  Theoretical_synthesis.md
  Cross_story_synthesis.md
  Handbook.md
  Fieldbook.md
  stories/
  assets/
```

`/content` не должен содержать служебные отчёты, рабочие pass-файлы, временные заметки или инструкции для агента. Локальные ссылки между Markdown-файлами должны оставаться рабочими после переноса историй в `content/stories/`.

## `/site`

Текущий построенный сайт. Это не чистый одноразовый `dist`, а рабочий сайт-артефакт. Его можно открыть локально через `site/index.html`.

Codex может делать малые технические правки прямо в `/site`. Если такая правка меняет устойчивое поведение сайта, соответствующее правило нужно перенести в `/site-spec`.

## `/site-spec`

Спецификация сайта. Здесь описаны структура, навигация, сохранение текста, ссылки, якоря, `scrollspy`, последовательное чтение и проверки перед сдачей.

`/site-spec` не является пользовательским контентом сайта.

## `/protocols`

Правила работы и готовые запросы.

```text
protocols/
  rules/
  skills/
  prompts/
```

`rules/` содержит устойчивые инструкции, на которые ссылается `AGENTS.md`. `skills/` содержит повторяемые рабочие режимы: один skill — один Markdown-файл. `prompts/` содержит активные запросы, которые можно копировать в Codex или новый чат; prompt обычно вызывает skill и задаёт параметры.

## `/project`

Документация о самом репозитории: структура, решения, планы, открытые вопросы и карта приоритета источников.

## `/work`

Рабочая зона текущей активной задачи текущей ветки.

Главный файл:

```text
work/discourse.md
```

`discourse.md` хранит ход задачи как дискурс: повороты обсуждения, причины решений, отвергнутые варианты, поздние уточнения, ограничения и детали, которые нельзя потерять. Это не краткий status-файл и не handoff. Важные повороты должны быть связаны с конкретными файлами и артефактами: если решение возникло из audit, approved plan, source map, seed-текста, prompt-файла или отчёта, имя этого файла должно быть названо прямо в дискурсе.

Сопутствующие документы задачи по умолчанию лежат в корне `/work`: планы, карты источников, правила восстановления baseline, seed-тексты, аудиты, prompt-файлы и отчёты запусков. Эти документы не должны становиться невидимым складом рядом с дискурсом: когда они влияют на ход задачи, `work/discourse.md` должен называть их и объяснять их роль.

`/work` не является архивом всех задач. История задач фиксируется Git-ветками, коммитами, PR и при необходимости итоговыми отчётами. Параллельные подзадачи разводятся через Git branches/worktrees, а не через вложенные папки задач внутри `/work`.

Подпапки внутри `/work` можно создавать только при практической необходимости: много однотипных файлов, серия pass-артефактов или технически неудобный объём материалов.

## Служебные файлы

- `.gitignore` хранит минимальные исключения: системный шум, возможные будущие зависимости сборщика и локальный scratch.
- `.gitkeep` используется только если пустая директория уже является частью рабочего контракта. В текущей модели `/work` не требует пустых подпапок.


## Правило `/work`

`/work` — рабочая зона текущей задачи текущей ветки. Главный файл — `work/discourse.md`. Он ведётся как однородный дискурс по поворотам разговора и сохраняет связь с файлами, которые участвовали в этих поворотах.

В корне `/work` лежат сопутствующие документы текущей задачи. Подпапки не являются нормой; они создаются только при практической необходимости. Параллельные подзадачи разводятся через Git branches/worktrees, а не через вложенные папки задач внутри `/work`.


## `/work` и ветки

`/work` — рабочая зона текущей активной задачи текущей ветки. Это не архив всех задач. Главный файл — `work/discourse.md`.

Параллельные задачи и подзадачи по умолчанию разводятся через Git branches/worktrees, а не через вложенные папки внутри `/work`. Перед merge в `main` папка `/work` по умолчанию не переносится. Если в `/work` возникло устойчивое правило, решение, prompt или skill, его нужно перенести в постоянную папку: `/project`, `/protocols`, `/site-spec` или `/content`, в зависимости от роли.

Подробная модель описана в `project/branching-and-task-model.md`.
