# Branching and task model

Этот репозиторий использует Git branches/worktrees как основную границу задач.

## Main

`main` хранит принятое устойчивое состояние проекта. В обычном `main` нет папки `/work`.

`main` может получать мелкие низкорисковые правки напрямую: опечатки, простые уточнения README, малые изменения протоколов или project docs, если пользователь явно разрешил запись в `main`. Для таких правок не нужно создавать ветку только ради формы.

Крупные изменения, работа с `/content`, пересборка сайта, изменение source precedence, baseline restore rules, архитектуры теории и финализация больших стадий должны идти через рабочую ветку.

## Рабочая ветка

Рабочая ветка соответствует активной задаче или подзадаче. Внутри рабочей ветки папка `/work` является рабочей зоной этой задачи.

Главный файл:

```text
work/discourse.md
```

Он хранит не status и не summary, а дискурс: ход разговора, причины решений, поздние уточнения, отвергнутые варианты и связь поворотов с конкретными файлами.

## Подзадачи

Подзадачи не создают обязательные вложенные папки внутри `/work`. Если появляется параллельная подзадача, она по умолчанию получает отдельную Git branch/worktree и продолжает основной дискурс этой задачи. Конфликты в `work/discourse.md` при merge считаются содержательными конфликтами траектории, а не техническим шумом.

## /work

`/work` не является архивом всех задач. Это рабочее место текущей ветки.

Сопутствующие документы задачи лежат в корне `/work`: утверждённые планы, карты источников, baseline rules, seed-тексты, audit protocols, отчёты, prompt-файлы или run notes. Подпапки в `/work` создаются только при реальной необходимости, например для большой серии однотипных pass-артефактов.

Архив scaffold с `/work` является образом рабочей ветки, а не чистого `main`. Чистый `main` создаётся без `/work`, затем от него создаётся рабочая ветка, куда добавляется `/work`.

## Перенос в main

Перед merge в `main` нужно отделить итоговые изменения от рабочей зоны. `/work` по умолчанию не переносится в `main`. Если в `/work` родилось устойчивое правило, решение или protocol, его нужно перенести в постоянную папку before merge.
