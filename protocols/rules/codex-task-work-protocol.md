# Протокол работы Codex с задачей

Этот протокол описывает, как Codex работает с задачей внутри репозитория, локального проекта, worktree или cloud environment. Он не является протоколом работы ChatGPT через GitHub-интеграцию.

Codex выполняет задачу как thread/session: читает контекст, определяет область действия, меняет файлы, запускает проверки, готовит diff/commit/PR и обновляет дискурс задачи. Если нужно работать именно из ChatGPT через GitHub API, используется `protocols/rules/chat-github-repo-work-protocol.md`.

## Роль Codex

Codex используется для исполнительной работы с репозиторием:

- staged-задачи;
- изменения файлов;
- проверки и отчёты;
- подготовка audits;
- работа с сайтом;
- восстановление anchor cases;
- крупные правки контента после human gate;
- branch/worktree/PR-cycle.

Codex не должен заменять дискурс задачи кратким summary. Для текущей рабочей ветки главным носителем смысловой непрерывности является `work/discourse.md`.

## Начало Codex-задачи

Перед любой нетривиальной задачей Codex должен прочитать:

```text
AGENTS.md
project/repository-structure.md
project/source-precedence.md
protocols/rules/codex-task-work-protocol.md
```

Если работа идёт в рабочей ветке с `/work`, дополнительно прочитать:

```text
work/discourse.md
```

Затем Codex должен оценить сопутствующие документы в корне `/work` и прочитать те, которые нужны для текущей операции. Если задача относится к теории, audit, source precedence, baseline restore, SPDD, Gas Town, specification cluster или Stage 0.5, Codex обязан прочитать соответствующие документы из `/work`, а не угадывать их содержание по именам.

Если непонятно, какие документы из `/work` нужны, Codex должен сначала перечислить видимые кандидаты и прочитать релевантные текстовые документы, достаточные для безопасного понимания задачи. Если объём явно слишком большой для одной сессии, Codex должен остановиться и предложить порядок чтения, а не выполнять задачу на неполном контексте молча.

## Восстановление контекста

Перед правками Codex должен восстановить из `work/discourse.md`:

- последний смысловой поворот задачи;
- какие решения уже нельзя случайно откатить;
- какие документы являются авторитетными для текущей операции;
- что нельзя делать без human gate;
- какие файлы, seed sections, source maps or audits участвуют в задаче.

Дискурс нельзя заменять commit messages, prompt-файлами, summaries или памятью thread-сессии.

## Область действия

Перед изменениями Codex должен кратко зафиксировать:

- что он понял как задачу;
- какие файлы собирается менять;
- какие файлы не собирается трогать;
- какие проверки или артефакты подтвердят результат;
- где может потребоваться human gate.

Для рискованных операций сначала выполняется planning-only pass. Рискованными считаются изменения:

- архитектуры теоретического документа;
- source precedence;
- baseline restore rules;
- структуры репозитория;
- протоколов и skills;
- site-spec;
- пользовательского content;
- SPDD, Gas Town and other deep anchor cases;
- merge/finalization logic.

## Выполнение

Codex должен делать минимальный связный шаг. Не смешивать unrelated изменения. Не начинать content writing, если задача была про scaffold, аудит, протоколы, структуру или подготовку workspace.

Если задача создаёт новые рабочие документы текущей ветки, они создаются в `/work`. Если результат должен стать устойчивой частью проекта, его нужно перенести в `/project`, `/protocols`, `/site-spec` или `/content` только после явного решения или human gate.

Codex может группировать изменения в логические commits, если это уместно для задачи, но commit messages должны быть достаточно якорными, чтобы позже можно было восстановить, какая версия файла соответствует какому действию. Для точечных документных изменений можно использовать режим “один изменяемый документ — один commit”, но это не обязательное правило для всех Codex-задач.

## Дискурс

После нетривиальной Codex-сессии нужно обновить `work/discourse.md` по правилам:

```text
protocols/rules/discourse-maintenance-rules.md
```

Если сессия изменила, создала или переоценила файлы, их точные имена должны быть встроены в соответствующий поворот дискурса. Если для последующего сравнения важна версия файла, в дискурс нужно вписать commit anchor или commit SHA.

## Проверки и свидетельства

Codex должен выполнить проверки, которые соответствуют задаче, или честно указать, почему они не выполнялись. В отчёте нужно различать:

- проверки, которые выполнены;
- проверки, которые не выполнялись;
- проверки, которые невозможно выполнить в текущей среде;
- human gates, которые остаются.

## Human gate

Codex должен остановиться и спросить пользователя, если:

- обнаружен конфликт между `work/discourse.md`, `source-precedence.md` and current files;
- требуется изменить source precedence;
- требуется сжать или удалить old-site baseline seed;
- требуется менять SPDD/Gas Town без anti-degradation check;
- audit обнаружил структурный gap, который меняет architecture;
- merge в `main` может перенести `/work` or task-local files;
- итоговый diff содержит неожиданные файлы;
- задача требует решения, а не механического выполнения.

## Итоговый отчёт Codex

В конце задачи Codex указывает:

1. Какие файлы изменены.
2. Какие правила, skills and source documents были прочитаны.
3. Какие проверки выполнены.
4. Какие проверки не выполнялись.
5. Как обновлён `work/discourse.md`, если задача была нетривиальной.
6. Какие commits/PR/diff artifacts созданы.
7. Какие вопросы требуют решения человека.
