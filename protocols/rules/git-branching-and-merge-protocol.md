# Протокол веток и merge

Этот протокол описывает Git-модель репозитория и порядок интеллектуального управления ветками через ChatGPT, Codex и GitHub.

## Базовая модель

`main` — принятое устойчивое состояние проекта. В обычном `main` нет папки `/work`.

Рабочая ветка — граница активной задачи или подзадачи. В рабочей ветке может появляться `/work`: рабочий дискурс, локальные планы, аудиты, seed-файлы, source maps и другие task-local документы.

Параллельные подзадачи разводятся через Git branches/worktrees, а не через вложенные папки внутри `/work`.

Архив scaffold с `/work` является образом рабочей ветки, а не образом чистого `main`. Если нужно создать чистый `main`, его следует собрать из тех же постоянных папок без `/work`; затем от него создаётся рабочая ветка с `/work`.

## Коммиты в main

Мелкие и низкорисковые поправки можно коммитить напрямую в `main`, если пользователь явно это разрешил. Это нормальный режим для исправления опечаток в проектной документации, малых правок README, простого уточнения протокола, если оно не меняет процесс существенно, или иных очевидных текстовых исправлений.

Прямой commit в `main` не должен использоваться для:

- изменений `/content` как пользовательского текста;
- массовых изменений сайта;
- изменения source precedence;
- изменения baseline restore rules;
- изменения архитектуры теории;
- удаления или сжатия deep anchor material;
- финализации большой рабочей ветки;
- правок, где требуется human gate.

Такие изменения идут через рабочую ветку и, при необходимости, PR.

## Коммиты в рабочую ветку

Небольшие изменения можно коммитить прямо в рабочую ветку: обновление `work/discourse.md`, протоколов, skills, project docs или малых prompts. Для таких изменений PR не обязателен.

Крупные изменения, изменения `/content`, пересборка сайта, массовые переносы и финализация этапов должны идти через PR или явный merge-протокол.

## Откат и восстановление

Если отдельная правка оказалась неудачной, предпочтительный порядок такой:

1. Для отдельного commit — создать revert commit, чтобы история оставалась читаемой.
2. Для отдельного файла — восстановить файл из известного commit, `main` или предыдущей версии и сделать новый commit.
3. Для ошибочной серии commit'ов — остановиться и запросить human gate, прежде чем делать reset или force-update ветки.

Force update ветки допустим только по явной команде пользователя и только после отчёта о том, какие commit'ы будут потеряны.

## Финализация рабочей ветки

Перед переносом в `main` агент должен:

1. Сравнить рабочую ветку с `main`.
2. Определить итоговые изменения, которые должны попасть в `main`.
3. Проверить изменения в `/work`.
4. Не переносить `/work` в `main` по умолчанию.
5. Если из `/work` возникло устойчивое решение, правило, prompt, skill, source-precedence update или protocol material, перенести его в `/project`, `/protocols`, `/site-spec` или `/content` по роли документа.
6. Подготовить финальный diff.
7. Остановиться, если есть конфликт, неоднозначное удаление, изменение source precedence, baseline rules, deep anchor seeds или неожиданно большой diff.

## Merge в main

Merge в `main` может выполняться агентом только по явной команде пользователя или по заранее утверждённому prompt/skill. Предпочтительный путь для крупной ветки — PR, потому что PR даёт review surface, историю обсуждения и возможность остановиться до merge.

Если пользователь разрешил автоматическую финализацию, агент может выполнить merge только когда:

- diff понятен;
- `/work` исключён из переноса в `main`, а устойчивые материалы из `/work` перенесены в постоянные папки;
- нет значимых конфликтов;
- проверки, оговорённые в задаче, выполнены или явно указаны как невыполненные;
- head SHA не изменился неожиданно перед merge.

## Отчёт

После финализации агент сообщает:

- source branch;
- target branch / целевая ветка;
- PR или merge commit;
- какие файлы попали в `main`;
- какие файлы из `/work` исключены;
- какие проверки выполнены;
- какие вопросы остались у human gate.
